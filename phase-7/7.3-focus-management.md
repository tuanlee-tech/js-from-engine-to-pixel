# Phase 7.3 — Focus Management: Cơ chế thật từ Browser đến Screen Reader

> **Độ ưu tiên:** 🔴 PHẢI BIẾT (tabindex, focus visible, modal trap, restoration) + 🟡 NÊN BIẾT (roving tabindex) + 🟢 BIẾT THÊM (inert attribute)

## 1. Cơ chế thật — Focus trong browser engine

### Browser maintain focus state như thế nào

Browser có một global focus system. Tại bất kỳ thời điểm nào, chỉ có **một element** được focused trong toàn bộ document:

```
Browser Focus System
        │
        ├─ document.activeElement  → Element đang focused (hoặc <body> nếu không có gì)
        ├─ Focus Ring (internal)   → Pointer tới focused element
        └─ Focus Stack             → History khi focus di chuyển vào/ra nested contexts
                                     (iframe, shadow DOM)

Khi user nhấn Tab:
  1. Browser compute "next focusable element" theo tab order
  2. Dispatch "blur" event trên element cũ
  3. Move focus ring tới element mới
  4. Dispatch "focus" event trên element mới
  5. Dispatch "focusin" event (bubbles, khác "focus" không bubble)
  6. Browser paint focus indicator (outline)
  7. Accessibility event → screen reader announce element mới
```

### Tab order algorithm — Browser tính thứ tự như thế nào

```
Bước 1: Collect tất cả focusable elements trong document
        ↓
Bước 2: Partition thành 2 groups:
        Group A: tabindex > 0  (positive tabindex)
        Group B: tabindex = 0 hoặc implicit focusable (natural order)

        [tabindex < 0 → NOT included]
        ↓
Bước 3: Sort Group A theo tabindex value tăng dần
        tabindex=1 → tabindex=2 → tabindex=3 → ...
        Ties: DOM order
        ↓
Bước 4: Group B theo DOM order (document order)
        ↓
Bước 5: Final tab order = Group A + Group B
```

```
DOM:
  <input tabindex="2" />    ← Group A, value=2
  <button tabindex="1" />   ← Group A, value=1
  <a href="/">Home</a>      ← Group B, DOM position 3
  <input type="text" />     ← Group B, DOM position 4
  <button tabindex="3" />   ← Group A, value=3

Tab order:
  button[tabindex=1] → input[tabindex=2] → button[tabindex=3]
  → a[href] → input[type=text]

Positive tabindex nhảy lên trước toàn bộ DOM order!
→ Đây là lý do positive tabindex là anti-pattern
```

### Implicit focusable elements

Browser build-in focusability, không cần tabindex:

```
Focusable by default (tabindex=0 implicit):
  <a href="...">                    ← href phải có! <a> không có href thì không focusable
  <button>                          ← trừ khi disabled
  <input>                           ← trừ khi disabled hoặc type="hidden"
  <select>                          ← trừ khi disabled
  <textarea>                        ← trừ khi disabled
  <details>                         ← summary element bên trong
  <summary>                         ← first child của <details>
  Elements với tabindex="0"         ← explicit

NOT focusable by default:
  <div>, <span>, <p>, <h1>-<h6>
  <a> không có href
  <button disabled>, <input disabled>
  Elements với display:none hoặc visibility:hidden
  Elements với tabindex="-1"        ← removed từ tab order
```

## 2. tabindex — Ba giá trị, ba behaviors khác nhau

### tabindex="0" — Thêm vào natural tab order

```html
<!-- Thêm element vào tab order theo DOM position -->
<!-- Dùng khi: custom interactive element không có native focusability -->

<div
  role="button"
  tabindex="0"
  onclick="handleClick()"
  onkeydown="handleKeydown(event)"
>
  Custom Button
</div>

<!-- Vị trí trong tab order = vị trí của div trong DOM
     Đây là behavior đúng — tab order follow visual order -->
```

### tabindex="-1" — Focusable chỉ bằng code, không bằng Tab

```html
<!-- Element KHÔNG trong tab order
     Nhưng CÓ THỂ focused bằng element.focus() 
     
     Use cases:
     - Modal containers (focus vào modal khi mở)
     - Skip link targets
     - Error message containers (focus để scroll user tới error)
     - Elements cần nhận focus programmatically nhưng không phải interactive
-->

<main
  id="main-content"
  tabindex="-1"
>
  <!-- Skip link target — focus đây sau khi user click skip -->
</main>

<div
  role="dialog"
  tabindex="-1"
  id="modal"
>
  <!-- Modal container — focus đây khi modal mở -->
  <!-- User sau đó Tab vào interactive elements bên trong -->
</div>
```

### tabindex > 0 — Anti-pattern

```html
<!-- ❌ ĐỪNG DÙNG positive tabindex -->
<button tabindex="3">Third</button>
<button tabindex="1">First</button>
<!-- Nhảy lên trước trong tab order! -->
<button tabindex="2">Second</button>

<!-- Vấn đề:
     1. Phá vỡ natural tab order
     2. Khi thêm elements mới, phải re-number toàn bộ
     3. Rất khó maintain trong large codebase
     4. Gây confusion cho keyboard users
     
     Exception duy nhất: legacy codebase với fixed layout mà
     không thể refactor DOM order — last resort only -->
```

## 3. Focus Visible — Đừng bao giờ xóa outline

### Cơ chế browser paint focus indicator

```
Element receives focus
        │
        ├─ Browser check :focus CSS rules
        ├─ Browser check :focus-visible CSS rules
        └─ Browser paint focus ring theo UA stylesheet

User Agent (browser default) stylesheet:
  :focus {
    outline: 2px solid Highlight; /* Windows high contrast */
    /* hoặc */
    outline: 5px auto -webkit-focus-ring-color; /* Chrome/Safari */
  }
```

### :focus vs :focus-visible — Khác nhau quan trọng

`:focus` fire cho **tất cả** cách focus — mouse click, keyboard, programmatic.
`:focus-visible` chỉ fire khi browser heuristic xác định focus indicator **nên visible** — typically keyboard navigation.

```css
/* Browser heuristics cho :focus-visible:
   - Keyboard navigation (Tab, arrow keys) → :focus-visible = true
   - Mouse click vào button → thường :focus-visible = false
   - Mouse click vào input → :focus-visible = true (inputs luôn cần indicator)
   - Programmatic focus() → phụ thuộc vào context
*/

/* ❌ WORST PRACTICE — remove focus indicator hoàn toàn */
* {
  outline: none; /* Keyboard users mù hoàn toàn */
}

/* ❌ Chỉ hide khi focus, không có alternative */
:focus {
  outline: none;
}

/* ✅ Hide default chỉ khi không dùng keyboard, show custom khi dùng keyboard */
/* Modern approach: chỉ style :focus-visible */
:focus {
  outline: none; /* Remove UA default */
}

:focus-visible {
  /* Custom focus indicator — rõ ràng và đẹp */
  outline: 3px solid #005fcc;
  outline-offset: 2px;
  border-radius: 2px;
}

/* Approach khác: override UA style với custom style cho cả :focus */
:focus {
  outline: 3px solid #005fcc;
  outline-offset: 2px;
}
```

### Focus indicator phải đạt WCAG 2.2 criteria

WCAG 2.2 thêm **Success Criterion 2.4.11 Focus Appearance (AA)**:

```
Focus indicator phải có:
1. Perimeter ≥ perimeter của bounding box của element
   (outline phải bao quanh toàn bộ element)

2. Contrast ratio ≥ 3:1 giữa focused và unfocused state
   (outline color phải contrast đủ với background)

3. Contrast ratio ≥ 3:1 giữa focused pixels và adjacent colors
```

```css
/* ✅ Đáp ứng WCAG 2.2 SC 2.4.11 */
:focus-visible {
  /* outline bao quanh element — đủ perimeter */
  outline: 3px solid #005fcc; /* Blue — contrast tốt với white/light bg */
  outline-offset: 2px; /* Không overlap với element border */
}

/* ✅ Double-ring technique — visible trên mọi background */
:focus-visible {
  outline: 3px solid #ffffff; /* Inner ring: white */
  box-shadow: 0 0 0 5px #005fcc; /* Outer ring: blue */
  /* Work trên cả dark và light backgrounds */
}

/* ❌ Fail — outline quá nhỏ, low contrast */
:focus-visible {
  outline: 1px dotted #999; /* Quá mờ, không đủ contrast */
}
```

## 4. Modal Focus Trap — Implementation đầy đủ

### Tại sao focus trap phức tạp hơn bạn nghĩ

```
Các edge cases phải handle:

1. Focusable elements có thể bị hidden bằng CSS (display:none, visibility:hidden)
   → Phải filter ra khỏi focusable list

2. Focusable elements có thể có tabindex="-1"
   → Không nằm trong tab order — phải exclude

3. Disabled elements (button[disabled], input[disabled])
   → Không focusable — phải exclude

4. Elements bên trong aria-hidden="true"
   → Không nên receive focus — phải exclude

5. Shadow DOM — focusable elements bên trong shadow root
   → querySelectorAll không traverse shadow DOM

6. Shift+Tab — wrap ngược lại
   → Handle cả forward và backward tab

7. Dynamically added elements trong modal
   → Phải re-query khi DOM thay đổi, không cache list

8. Iframe bên trong modal
   → Focus có thể di chuyển vào iframe — handle khác nhau
```

```javascript
class FocusTrap {
  constructor(container) {
    this.container = container;
    this.handleKeydown = this.handleKeydown.bind(this);
    this.handleFocusOut = this.handleFocusOut.bind(this);
  }

  // Selector đầy đủ — không thiếu edge cases
  static FOCUSABLE_SELECTORS = [
    'a[href]',
    'area[href]',
    'input:not([disabled]):not([type="hidden"])',
    'select:not([disabled])',
    'textarea:not([disabled])',
    'button:not([disabled])',
    'iframe',
    'object',
    'embed',
    '[contenteditable]',
    '[tabindex]:not([tabindex="-1"])',
    'details > summary:first-of-type',
    'details',
  ].join(', ');

  getFocusableElements() {
    return Array.from(
      this.container.querySelectorAll(FocusTrap.FOCUSABLE_SELECTORS),
    ).filter((el) => {
      // Filter elements không thật sự focusable

      // 1. Bị hidden bởi CSS
      if (el.offsetParent === null && el.style.position !== 'fixed') {
        return false;
      }

      // 2. Bên trong aria-hidden subtree
      if (el.closest('[aria-hidden="true"]')) return false;

      // 3. Bên trong hidden element
      if (el.closest('[hidden]')) return false;

      // 4. Invisible (display:none, visibility:hidden)
      const style = window.getComputedStyle(el);
      if (style.display === 'none' || style.visibility === 'hidden') {
        return false;
      }

      return true;
    });
  }

  activate() {
    // Save previous focus để restore sau
    this.previouslyFocused = document.activeElement;

    // Focus vào container trước
    // Container có tabindex="-1" để nhận focus
    this.container.focus();

    // Sau đó move vào first focusable element
    // Dùng setTimeout để đảm bảo container focus xảy ra trước
    requestAnimationFrame(() => {
      const focusable = this.getFocusableElements();
      if (focusable.length > 0) {
        focusable[0].focus();
      }
    });

    this.container.addEventListener('keydown', this.handleKeydown);

    // Backup: nếu focus thoát ra ngoài container, pull nó về
    document.addEventListener('focusout', this.handleFocusOut);
  }

  deactivate() {
    this.container.removeEventListener('keydown', this.handleKeydown);
    document.removeEventListener('focusout', this.handleFocusOut);

    // Restore focus về element trước khi modal mở
    // Quan trọng: phải check element vẫn còn trong DOM và focusable
    if (
      this.previouslyFocused &&
      document.contains(this.previouslyFocused) &&
      this.previouslyFocused !== document.body
    ) {
      this.previouslyFocused.focus();
    }
  }

  handleKeydown(event) {
    if (event.key !== 'Tab') return;

    // Re-query mỗi lần — dynamic content có thể đã thay đổi
    const focusable = this.getFocusableElements();
    if (focusable.length === 0) {
      event.preventDefault();
      return;
    }

    const firstEl = focusable[0];
    const lastEl = focusable[focusable.length - 1];
    const currentIndex = focusable.indexOf(document.activeElement);

    if (event.shiftKey) {
      // Shift+Tab: backward
      if (document.activeElement === firstEl || currentIndex === -1) {
        event.preventDefault();
        lastEl.focus();
      }
    } else {
      // Tab: forward
      if (document.activeElement === lastEl) {
        event.preventDefault();
        firstEl.focus();
      }
    }
  }

  handleFocusOut(event) {
    // focusout bubbles — check nếu focus đi ra ngoài container
    requestAnimationFrame(() => {
      // Sau khi focus move xong, check activeElement
      if (
        !this.container.contains(document.activeElement) &&
        document.activeElement !== document.body
      ) {
        // Focus thoát ra — pull về first focusable
        const focusable = this.getFocusableElements();
        if (focusable.length > 0) {
          focusable[0].focus();
        }
      }
    });
  }
}
```

## 5. Focus Restoration — Pattern hay bị bỏ quên

### Tại sao restoration quan trọng với screen reader users

```
Scenario không có focus restoration:

1. User navigate tới "Delete item" button (screen reader reads: "Delete item, button")
2. Confirmation dialog mở
3. User confirm → dialog đóng → item bị xóa
4. Focus rơi về... <body>
5. Screen reader announces: "Document" hoặc im lặng
6. User không biết mình đang ở đâu trong page
7. User phải Tab từ đầu để tìm lại context

Scenario có focus restoration:

1. User navigate tới "Delete item" button
2. Dialog mở
3. User confirm → dialog đóng
4. Focus về element gần nhất meaningful
5. Screen reader continues từ đó
```

### Focus restoration patterns

```javascript
// Pattern 1: Restore về trigger element (phổ biến nhất)
class Dialog {
  open(triggerElement) {
    // Lưu trigger — element đã mở dialog
    this.trigger = triggerElement;

    this.element.removeAttribute('hidden');
    this.focusTrap.activate();
  }

  close() {
    this.element.setAttribute('hidden', '');
    this.focusTrap.deactivate(); // Internally restore focus về this.trigger
  }
}

// Pattern 2: Restore về element "meaningful" khi trigger bị xóa
// Ví dụ: xóa một list item → item đó biến mất → focus về đâu?
function deleteItem(itemElement) {
  // Tìm "next" element trước khi xóa
  const nextFocus = findNextFocus(itemElement);

  itemElement.remove();

  // Focus về element tiếp theo
  nextFocus?.focus();
}

function findNextFocus(element) {
  // Priority: next sibling → previous sibling → parent
  return (
    element.nextElementSibling?.querySelector('[tabindex], button, a[href]') ||
    element.previousElementSibling?.querySelector(
      '[tabindex], button, a[href]',
    ) ||
    element.parentElement
  );
}

// Pattern 3: Restore sau async operation
async function submitForm() {
  const submitButton = document.getElementById('submit-btn');

  // Loading state — button bị disabled
  submitButton.disabled = true;
  submitButton.setAttribute('aria-busy', 'true');

  try {
    await saveData();

    // Success: có thể navigate sang page khác
    // hoặc focus về success message
    const successMsg = document.getElementById('success-message');
    successMsg.textContent = 'Changes saved successfully';
    successMsg.focus(); // tabindex="-1" để nhận focus
  } catch (error) {
    // Error: restore focus về button, announce error
    submitButton.disabled = false;
    submitButton.removeAttribute('aria-busy');
    submitButton.focus(); // Focus về button

    // Announce error qua live region
    document.getElementById('error-region').textContent = error.message;
  }
}
```

## 6. Roving tabindex — Pattern cho Composite Widgets

### Vấn đề roving tabindex giải quyết

Widgets như toolbar, tab list, menu, radio group — có nhiều items nhưng chỉ **một item** nằm trong tab order tại một thời điểm. User Tab vào widget, dùng **arrow keys** để navigate bên trong, Tab thoát ra ngoài.

```
Tab order của page:
  [Skip link] → [Logo] → [Search] → [TOOLBAR] → [Main content]
                                          ↑
                             Chỉ 1 item trong tab order
                             Arrow keys navigate bên trong toolbar

Nếu không có roving tabindex:
  Toolbar có 10 buttons → 10 Tab stops → UX kinh khủng cho keyboard users
  User phải Tab qua 10 buttons chỉ để thoát ra khỏi toolbar
```

### Cơ chế roving tabindex

```
Initial state:
  button[Bold]:   tabindex="0"   ← Tab stop, first item focused by default
  button[Italic]: tabindex="-1"  ← Removed từ tab order
  button[Underline]: tabindex="-1"

User Tab vào toolbar → Bold button focused

User nhấn ArrowRight:
  button[Bold]:   tabindex="-1"  ← Removed từ tab order
  button[Italic]: tabindex="0"   ← Added to tab order, focused
  button[Underline]: tabindex="-1"

User nhấn Tab → focus thoát khỏi toolbar (next element sau toolbar)
User nhấn Shift+Tab → focus quay lại toolbar tại Italic button (tabindex="0")
```

```javascript
class RovingTabindex {
  constructor(container, itemSelector) {
    this.container = container;
    this.itemSelector = itemSelector;
    this.currentIndex = 0;

    this.init();
    this.container.addEventListener('keydown', this.handleKeydown.bind(this));
  }

  get items() {
    // Re-query để handle dynamic content
    return Array.from(
      this.container.querySelectorAll(this.itemSelector),
    ).filter((el) => !el.disabled && !el.hidden);
  }

  init() {
    const items = this.items;
    items.forEach((item, index) => {
      // Chỉ item đầu tiên có tabindex="0"
      // Các items còn lại: tabindex="-1"
      item.setAttribute('tabindex', index === 0 ? '0' : '-1');
    });
  }

  moveFocus(newIndex) {
    const items = this.items;
    if (items.length === 0) return;

    // Wrap around: nếu vượt quá cuối → về đầu, và ngược lại
    const wrappedIndex = (newIndex + items.length) % items.length;

    // Remove current item từ tab order
    items[this.currentIndex]?.setAttribute('tabindex', '-1');

    // Add new item vào tab order và focus
    items[wrappedIndex].setAttribute('tabindex', '0');
    items[wrappedIndex].focus();

    this.currentIndex = wrappedIndex;
  }

  handleKeydown(event) {
    const isVertical =
      this.container.getAttribute('aria-orientation') === 'vertical';

    switch (event.key) {
      case 'ArrowRight':
        if (!isVertical) {
          event.preventDefault();
          this.moveFocus(this.currentIndex + 1);
        }
        break;

      case 'ArrowLeft':
        if (!isVertical) {
          event.preventDefault();
          this.moveFocus(this.currentIndex - 1);
        }
        break;

      case 'ArrowDown':
        if (isVertical) {
          event.preventDefault();
          this.moveFocus(this.currentIndex + 1);
        }
        break;

      case 'ArrowUp':
        if (isVertical) {
          event.preventDefault();
          this.moveFocus(this.currentIndex - 1);
        }
        break;

      case 'Home':
        event.preventDefault();
        this.moveFocus(0);
        break;

      case 'End':
        event.preventDefault();
        this.moveFocus(this.items.length - 1);
        break;
    }
  }
}

// Usage — Toolbar
const toolbar = document.querySelector('[role="toolbar"]');
const rovingTabindex = new RovingTabindex(toolbar, 'button:not([disabled])');
```

## 7. inert attribute — Modern solution cho background content

### Cơ chế inert

`inert` là HTML attribute (không phải ARIA) — làm cho element và toàn bộ subtree:

1. Không focusable (removed khỏi tab order)
2. Không clickable (pointer events disabled)
3. Không searchable (browser find-in-page skip)
4. Hidden khỏi accessibility tree (như `aria-hidden="true"` trên toàn bộ subtree)

```
Trước inert — Modal management thủ công:
  - Phải set tabindex="-1" trên mọi focusable element trong background
  - Phải set aria-hidden="true" trên background containers
  - Phải remove event listeners
  - Khi modal đóng, phải undo tất cả → error-prone

Với inert:
  document.querySelector('#background-content').inert = true;
  // Done. Browser handle tất cả.
```

```javascript
// Modal với inert — modern pattern
class Modal {
  constructor(modalEl, mainContentEl) {
    this.modal = modalEl;
    this.mainContent = mainContentEl;
    this.previousFocus = null;
  }

  open() {
    this.previousFocus = document.activeElement;

    // Make background inert — browser handle tất cả
    // Không cần traverse DOM để set tabindex="-1" thủ công
    this.mainContent.inert = true;

    // Show modal
    this.modal.removeAttribute('hidden');
    this.modal.removeAttribute('inert');

    // Focus vào first focusable trong modal
    const firstFocusable = this.modal.querySelector(
      'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])',
    );
    firstFocusable?.focus();
  }

  close() {
    // Remove inert từ background
    this.mainContent.inert = false;

    // Hide modal, make it inert (không strictly necessary khi hidden, nhưng defensive)
    this.modal.setAttribute('hidden', '');

    // Restore focus
    this.previousFocus?.focus();
  }
}

// Usage
const modal = new Modal(
  document.getElementById('dialog'),
  document.getElementById('main-content'),
);

document
  .getElementById('open-btn')
  .addEventListener('click', () => modal.open());
document
  .getElementById('close-btn')
  .addEventListener('click', () => modal.close());
```

### Browser support và polyfill


## 8. Visualize — Focus flow trong SPA navigation

```
SPA: User click link để navigate sang page mới
        │
        ▼
Traditional web:
  Browser load new page → focus về <body> → screen reader đọc page title
  → User biết đã navigate xong

SPA (React Router, Next.js, v.v.):
  JS update DOM → URL change → visual content thay đổi
  Focus: còn ở link element đã clicked (hoặc mất nếu link component unmount)
  Screen reader: không có announcement → User không biết navigation đã xảy ra

┌─────────────────────────────────────────────────────────────┐
│              SPA Focus Management Strategies                 │
├──────────────────────────┬──────────────────────────────────┤
│ Strategy 1               │ Focus <h1> của page mới          │
│ (Most common)            │ Screen reader reads page title    │
│                          │ User biết navigation xong         │
├──────────────────────────┼──────────────────────────────────┤
│ Strategy 2               │ Focus <main> với tabindex="-1"   │
│                          │ Generic — không specific về page  │
├──────────────────────────┼──────────────────────────────────┤
│ Strategy 3               │ Announce qua aria-live region     │
│                          │ "Navigated to Dashboard"          │
│                          │ Focus stays where it is           │
│                          │ Less disruptive                   │
├──────────────────────────┼──────────────────────────────────┤
│ Strategy 4               │ Focus top of page (<body>)        │
│ (Least good)             │ User phải re-navigate từ đầu     │
└──────────────────────────┴──────────────────────────────────┘
```

```jsx
// React Router + Focus Management
import { useEffect, useRef } from 'react';
import { useLocation } from 'react-router-dom';

// Component này wrap mỗi page
function PageWrapper({ title, children }) {
  const headingRef = useRef(null);
  const location = useLocation();

  useEffect(() => {
    // Mỗi khi route thay đổi, focus vào h1 của page mới
    // tabIndex="-1" trên h1 để nhận programmatic focus
    headingRef.current?.focus();

    // Update document title → tab title + screen reader page context
    document.title = `${title} — My App`;
  }, [location.pathname, title]);

  return (
    <main>
      {/* tabIndex="-1": focusable bằng code, không trong tab order */}
      <h1
        ref={headingRef}
        tabIndex={-1}
        style={{ outline: 'none' }}
      >
        {title}
      </h1>
      {children}
    </main>
  );
}

// Usage
function DashboardPage() {
  return (
    <PageWrapper title='Dashboard'>
      <p>Welcome to your dashboard.</p>
    </PageWrapper>
  );
}
```

```javascript
function TodoList({ todos, onDelete }) {
  const listRef = useRef(null);

  const handleDelete = (id, index) => {
    // Tìm button tiếp theo trước khi xóa
    const buttons = listRef.current.querySelectorAll('button');

    // Focus button của item tiếp theo, hoặc item trước nếu là item cuối
    const nextButton = buttons[index + 1] || buttons[index - 1];
    nextButton?.focus();

    onDelete(id);
  };

  return (
    <ul ref={listRef}>
      {todos.map((todo, index) => (
        <li key={todo.id}>
          {todo.text}
          <button onClick={() => handleDelete(todo.id, index)}>
            Delete {todo.text}
          </button>
        </li>
      ))}
```

```javascript
// ❌ Bug: Focus trap không deactivate khi modal đóng bằng Escape

class BrokenFocusTrap {

    document.addEventListener('keydown', (e) => {
      if (e.key === 'Tab') this.handleTab(e);
      // Arrow function không thể removeEventListener!
    });
  }
  // Không có deactivate!
}

// ✅ Fix: Bind method để có thể remove, clean up properly
class WorkingFocusTrap {
  constructor() {
    // Bind once — same reference để removeEventListener work
    this.boundHandleKeydown = this.handleKeydown.bind(this);
  }

  activate() {
    document.addEventListener('keydown', this.boundHandleKeydown);
  }

  deactivate() {
    document.removeEventListener('keydown', this.boundHandleKeydown);
  }
}
```

## 10. Câu hỏi kiểm tra

**Q1: Tại sao positive tabindex (tabindex="1", "2", "3") là anti-pattern?**

> Positive tabindex tạo một "priority group" riêng biệt — tất cả elements với positive tabindex đều được tab tới TRƯỚC mọi elements có tabindex=0 trong tab order. Điều này phá vỡ natural DOM order: user nhấn Tab, focus nhảy đến element ở cuối page (tabindex=1), rồi về đầu page, rồi mới Tab qua phần còn lại — hoàn toàn counterintuitive. Ngoài ra, khi codebase grow, phải liên tục re-number tabindex values để maintain correct order. Fix đúng là sắp xếp DOM order theo visual/logical order, rồi dùng tabindex="0" hoặc không cần tabindex.

**Q2: Điều gì xảy ra khi bạn call `element.focus()` trên element không có `tabindex="-1"` và không phải native focusable?**

> Browser silently ignore call. Không có error thrown. `document.activeElement` không thay đổi (hoặc về `<body>`). Screen reader không announce gì. Đây là silent failure — rất khó debug vì không có warning. Fix: thêm `tabindex="-1"` trên element cần nhận programmatic focus. Hay gặp nhất với `<div role="dialog">`, `<main id="content">`, và error message containers — những elements này cần nhận focus programmatically nhưng không phải native focusable.

**Q3: Đoạn code này có vấn đề gì?**

```javascript
function openModal() {
  modal.style.display = 'flex';
  const focusable = modal.querySelectorAll('button, input, a[href]');
  focusable[0].focus();
}

function closeModal() {
  modal.style.display = 'none';
  document.querySelector('.open-modal-btn').focus();
}
```

> Ba vấn đề: (1) Không có focus trap — user có thể Tab ra khỏi modal và interact với background. (2) `querySelectorAll` có thể return elements bị hidden bởi CSS — `focusable[0]` có thể là element không thật sự focusable, focus sẽ silent fail. (3) `closeModal` hardcode selector `.open-modal-btn` — nếu modal có thể được mở từ nhiều triggers khác nhau, sẽ restore focus sai. Fix: save `document.activeElement` trước khi open, implement focus trap, restore về saved element khi close.

## Câu hỏi ôn tập (NotebookLM)

### Q1: Roving tabindex và aria-activedescendant giải quyết cùng một vấn đề nhưng theo cách khác nhau như thế nào? Khi nào dùng cái nào?

**Đáp án:**

Cả hai pattern đều giải quyết vấn đề: composite widget có nhiều items có thể navigate, nhưng chỉ nên có một Tab stop trong tab order của page.

**Roving tabindex:**

- DOM focus thật sự move từ item này sang item khác khi user nhấn arrow keys
- Tại mỗi thời điểm, item đang active có `tabindex="0"`, tất cả items khác có `tabindex="-1"`
- Khi user Tab lại vào widget sau khi thoát ra, focus đến item cuối cùng được selected (nhờ item đó vẫn có `tabindex="0"`)
- Screen reader announce dựa trên DOM focus thật sự
- **Dùng khi:** Items có thể receive DOM focus và phản ứng với focus events (toolbar buttons, tab list, radio group)

**aria-activedescendant:**

- DOM focus ở lại trên container element (e.g., combobox input, listbox)
- Container có `aria-activedescendant` attribute trỏ tới ID của item đang "virtually focused"
- Khi user nhấn arrow keys, JS chỉ update attribute — DOM focus không move
- Screen reader announce item được trỏ tới bởi `aria-activedescendant`
- **Dùng khi:** Container cần maintain DOM focus (combobox input cần nhận keyboard input), hoặc items quá nhiều và moving focus sẽ gây performance issues

**Trade-offs:**

- Roving tabindex: đơn giản hơn implement đúng, browser handle announcement tự nhiên, nhưng focus movement có thể trigger scroll unwanted
- aria-activedescendant: phức tạp hơn (phải manually scroll item vào view, phải manually style "focused" state), nhưng cho phép container giữ DOM focus

Thực tế: ARIA APG recommend roving tabindex cho hầu hết widgets. aria-activedescendant chủ yếu dùng cho combobox và listbox.

### Q2: Giải thích tại sao `:focus-visible` ra đời và nó giải quyết vấn đề gì mà `:focus` không làm được?

**Đáp án:**

Trước `:focus-visible`, developer gặp một conflict không thể giải quyết:

**Vấn đề:** Focus ring (outline) cần thiết cho keyboard users để biết focus đang ở đâu. Nhưng khi mouse user click vào button, focus ring xuất hiện — đây là visual noise không cần thiết vì mouse user có thể thấy cursor. Design thường muốn remove focus ring khi click chuột.

**Giải pháp sai phổ biến trước đây:**

```css
/* Remove focus ring cho tất cả → keyboard users bị mù */
:focus {
  outline: none;
}

/* Remove chỉ khi click, dùng JS để track mouse vs keyboard */
/* Phức tạp, fragile, không reliable */
```

**`:focus-visible` giải quyết:** Browser tự quyết định khi nào focus indicator NÊN visible dựa trên heuristics:

- User dùng keyboard (Tab, arrow keys) → `:focus-visible` true → show indicator
- User click chuột vào button → `:focus-visible` false → không show (cho most elements)
- User click vào input/textarea → `:focus-visible` true → show (vì user cần biết đang type vào đâu)
- Programmatic `element.focus()` → phụ thuộc context và browser

Điều này cho phép:

```css
/* Remove UA default cho tất cả */
:focus {
  outline: none;
}

/* Chỉ show custom indicator khi keyboard navigation */
:focus-visible {
  outline: 3px solid blue;
  outline-offset: 2px;
}
```

Mouse users thấy clean UI. Keyboard users thấy focus indicator rõ ràng. Cả hai đều happy. Browser logic quyết định — không cần JS để track input modality.

**Lưu ý quan trọng:** Không nên dùng `:focus-visible` để hoàn toàn remove focus ring mà không có replacement. Mục đích là replace UA default với custom indicator, không phải xóa hoàn toàn.

### Q3: Mô tả đầy đủ những gì cần xảy ra khi một modal dialog mở và đóng để đạt keyboard accessibility hoàn chỉnh.

**Đáp án:**

**Khi modal MỞ:**

1. **Save current focus:** `previousFocus = document.activeElement` — lưu trigger element để restore sau

2. **Show modal:** Remove `hidden` attribute (không chỉ toggle CSS `display`) để AT nhận biết element xuất hiện

3. **Set ARIA attributes:**
   - `role="dialog"` trên container
   - `aria-modal="true"` để hint cho AT rằng background inert
   - `aria-labelledby` trỏ tới heading của dialog

4. **Inert background** (modern): `mainContent.inert = true` — browser tự handle tabindex và aria-hidden cho toàn bộ background. Hoặc manual approach: set `aria-hidden="true"` và `tabindex="-1"` trên tất cả focusable elements bên ngoài modal.

5. **Focus vào modal:** Container có `tabindex="-1"` để nhận programmatic focus, sau đó focus move tới first meaningful focusable element bên trong (thường là heading hoặc first input, không nhất thiết phải là first focusable element)

6. **Activate focus trap:** Intercept Tab và Shift+Tab để wrap trong modal boundaries. Intercept Escape để close.

**Khi modal ĐÓNG:**

1. **Deactivate focus trap:** Remove event listeners

2. **Remove inert / restore aria-hidden:** `mainContent.inert = false`

3. **Hide modal:** Add `hidden` attribute

4. **Restore focus:** `previousFocus.focus()` — phải check element vẫn còn trong DOM và không disabled trước khi focus. Nếu trigger element bị xóa trong quá trình modal open (ví dụ: delete operation), tìm element fallback meaningful gần nhất.

5. **Announce nếu cần:** Nếu action trong modal thay đổi state của page (delete, save), announce kết quả qua aria-live region SAU KHI focus được restore — vì screen reader sẽ đọc focus location trước, rồi announce live region sau.

**Edge cases thường bị miss:**

- Modal mở bằng nhiều triggers khác nhau → phải save trigger, không hardcode selector
- Modal content thay đổi dynamically trong khi mở → phải re-query focusable elements trong focus trap, không cache
- Nested modals → cần stack của previousFocus, không phải single variable
- Modal dismiss bằng click overlay → cũng phải restore focus, không chỉ khi click X button
