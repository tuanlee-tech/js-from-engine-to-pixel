> **Độ ưu tiên:** 🔴 PHẢI BIẾT (toàn bộ phần core) + 🟡 NÊN BIẾT (advanced patterns) + 🟢 BIẾT THÊM (APG patterns)

## 1. Cơ chế thật — ARIA hoạt động như thế nào trong browser

### ARIA không thay đổi behavior — chỉ thay đổi accessibility tree

Đây là misconception phổ biến nhất về ARIA. Khi bạn thêm `role="button"` vào `<div>`:

```
TRƯỚC khi thêm ARIA:
DOM:              <div class="btn">Submit</div>
Accessibility Tree: role=generic, name="Submit"
Keyboard:         KHÔNG focusable, KHÔNG respond Enter/Space
Screen Reader:    Reads "Submit" như plain text

SAU khi thêm role="button":
DOM:              <div class="btn" role="button" tabindex="0">Submit</div>
Accessibility Tree: role=button, name="Submit"  ← thay đổi
Keyboard:         Focusable (nhờ tabindex="0"), NHƯNG...
                  Enter/Space KHÔNG hoạt động ← ARIA không thêm behavior!
Screen Reader:    Reads "Submit, button" ← screen reader biết đây là button
```

**Kết luận:** ARIA chỉ modify accessibility tree. Nó không thêm keyboard handling, không thêm click behavior, không thêm bất cứ JS behavior nào. Bạn phải tự implement.

### ARIA được process như thế nào trong browser engine

```
Browser Engine (Blink/Gecko/WebKit)
        │
        │  Parse HTML + CSS
        ▼
   DOM Tree + Style Tree
        │
        │  AXObjectCache (Blink) / AccessibilityObjectWrapper (WebKit)
        │  Traverse DOM nodes
        │  Compute ARIA properties cho mỗi node:
        │    1. Check element's implicit ARIA role (từ HTML tag)
        │    2. Check explicit role attribute → override implicit
        │    3. Collect tất cả aria-* attributes
        │    4. Compute accessible name (thuật toán riêng)
        │    5. Compute states (disabled, expanded, checked, v.v.)
        ▼
   Accessibility Tree (AOM)
        │
        │  Platform Accessibility API layer
        │  Windows: IAccessible2 / UI Automation
        │  macOS: NSAccessibility Protocol
        │  Linux: AT-SPI2
        ▼
   Screen Reader / AT
```

**Key insight:** Browser recompute accessibility tree mỗi khi DOM thay đổi. React re-render, classList toggle, attribute change — tất cả trigger accessibility tree update. Đây là lý do tại sao ARIA attribute changes được screen reader pick up ngay lập tức mà không cần reload.

## 2. ARIA Roles — Semantic layer trên HTML

### Role taxonomy

ARIA roles chia thành 6 categories:

```
┌─────────────────────────────────────────────────────────┐
│                  ARIA Role Categories                    │
├──────────────────────┬──────────────────────────────────┤
│ Abstract Roles       │ Không dùng trực tiếp — base      │
│                      │ classes cho ARIA taxonomy         │
│                      │ (roletype, widget, structure)     │
├──────────────────────┼──────────────────────────────────┤
│ Widget Roles         │ Interactive UI components         │
│                      │ button, checkbox, combobox,       │
│                      │ dialog, listbox, menu, slider,    │
│                      │ switch, tab, tabpanel, tooltip    │
├──────────────────────┼──────────────────────────────────┤
│ Document Structure   │ Cấu trúc nội dung                │
│ Roles                │ article, cell, columnheader,      │
│                      │ definition, figure, heading,      │
│                      │ list, listitem, row, table        │
├──────────────────────┼──────────────────────────────────┤
│ Landmark Roles       │ Navigation regions                │
│                      │ banner, complementary,            │
│                      │ contentinfo, form, main,          │
│                      │ navigation, region, search        │
├──────────────────────┼──────────────────────────────────┤
│ Live Region Roles    │ Dynamic content                   │
│                      │ alert, log, marquee, status,      │
│                      │ timer                             │
├──────────────────────┼──────────────────────────────────┤
│ Window Roles         │ Sub-windows                       │
│                      │ alertdialog, dialog               │
└──────────────────────┴──────────────────────────────────┘
```

### Implicit roles của HTML elements

Browser assign implicit role mà không cần bạn khai báo:

```html
<!-- Browser tự assign role, không cần aria attribute -->
<nav>        → role="navigation"
<main>       → role="main"
<header>     → role="banner"       (chỉ khi top-level, không trong <main>)
<footer>     → role="contentinfo"  (chỉ khi top-level)
<aside>      → role="complementary"
<section aria-label="..."> → role="region"  (chỉ khi có accessible name!)
<form aria-label="...">    → role="form"    (chỉ khi có accessible name!)
<article>    → role="article"
<h1>-<h6>   → role="heading" (level=1-6)
<ul>, <ol>   → role="list"
<li>         → role="listitem"
<table>      → role="table"
<tr>         → role="row"
<th>         → role="columnheader" hoặc "rowheader"
<td>         → role="cell"
<button>     → role="button"
<a href=""> → role="link"
<input type="text">     → role="textbox"
<input type="checkbox"> → role="checkbox"
<input type="radio">    → role="radio"
<select>     → role="listbox" hoặc "combobox"
<textarea>   → role="textbox"
<img alt="text"> → role="img"
<img alt="">     → role="presentation" (decorative — excluded)
```

**Quan trọng:** `<section>` và `<form>` **chỉ có landmark role khi có accessible name** (`aria-label` hoặc `aria-labelledby`). Không có tên → không có landmark role. Đây là gotcha nhiều người không biết.

### Role conflict — explicit override implicit

```html
<!-- Browser compute role theo thứ tự: explicit > implicit -->

<!-- explicit role="presentation" xóa bỏ implicit role của <table> -->
<!-- Dùng khi table layout, không phải data table -->
<table role="presentation">
  <tr>
    <td>Column 1</td>
    <td>Column 2</td>
  </tr>
</table>

<!-- explicit role="button" thêm button semantics cho link -->
<!-- Nhưng link mất "link" role — không phải best practice -->
<a
  href="#"
  role="button"
  >Open dialog</a
>
<!-- Better: dùng <button> thật sự -->

<!-- role="none" = role="presentation" — loại bỏ semantic -->
<ul role="none">
  <!-- Remove list semantics khi list dùng cho layout -->
  <li><a href="/">Home</a></li>
</ul>
```

## 3. ARIA States & Properties — Communication với AT

### Phân biệt States vs Properties

```
Properties: Ít thay đổi, describe characteristics của element
  aria-label, aria-labelledby, aria-describedby,
  aria-haspopup, aria-controls, aria-owns, aria-required

States: Thay đổi động theo user interaction hoặc app state
  aria-expanded, aria-checked, aria-selected,
  aria-disabled, aria-hidden, aria-invalid, aria-busy,
  aria-pressed, aria-current
```

### aria-label vs aria-labelledby vs aria-describedby

Ba attributes này hay bị nhầm lẫn. Mechanism khác nhau:

```
┌──────────────────┬──────────────────────────────────────────────┐
│ Attribute        │ Cơ chế                                       │
├──────────────────┼──────────────────────────────────────────────┤
│ aria-label       │ String trực tiếp — override text content     │
│                  │ Screen reader đọc value của attribute         │
│                  │ Không visible trên screen                     │
├──────────────────┼──────────────────────────────────────────────┤
│ aria-labelledby  │ Pointer tới element(s) khác bằng ID          │
│                  │ Browser lấy text của element được trỏ tới    │
│                  │ Ưu tiên CAO NHẤT trong accessible name algo  │
│                  │ Override cả aria-label                        │
│                  │ Có thể point tới multiple IDs                 │
├──────────────────┼──────────────────────────────────────────────┤
│ aria-describedby │ Pointer tới element chứa description          │
│                  │ Description = ADDITIONAL info, không phải name│
│                  │ Screen reader đọc sau khi đọc name + role    │
│                  │ Ưu tiên THẤP — supplementary information     │
└──────────────────┴──────────────────────────────────────────────┘
```

```html
<!-- aria-label: name là string trực tiếp -->
<!-- Dùng khi không có visible text để reference -->
<button aria-label="Close navigation menu">
  <svg aria-hidden="true">...</svg>
</button>
<!-- Screen reader: "Close navigation menu, button" -->

<!-- aria-labelledby: name lấy từ element khác -->
<!-- Dùng khi visible text đã tồn tại trên page -->
<h2 id="billing-heading">Billing Information</h2>
<form aria-labelledby="billing-heading">
  <!-- Form name = text của h2 = "Billing Information" -->
</form>

<!-- aria-labelledby với nhiều IDs — concatenate -->
<span id="qty-label">Quantity</span>
<span id="product-name">Blue Shirt (Large)</span>
<input
  type="number"
  aria-labelledby="qty-label product-name"
  <!-- Accessible name = "Quantity Blue Shirt (Large)" -->
/>

<!-- aria-describedby: extra information -->
<label for="password">Password</label>
<input
  type="password"
  id="password"
  aria-describedby="password-requirements"
/>
<ul id="password-requirements">
  <li>At least 8 characters</li>
  <li>One uppercase letter</li>
</ul>
<!-- Screen reader: "Password, edit text. At least 8 characters. One uppercase letter." -->
<!-- Name: "Password" | Description: requirements text -->
```

### aria-hidden — Deep dive

```html
<!-- Cách đúng: hide decorative content -->
<button>
  <!-- Icon: aria-hidden removes từ AT, focusable="false" ngăn SVG nhận focus trong IE/Edge -->
  <svg
    aria-hidden="true"
    focusable="false"
    width="16"
    height="16"
  >
    <path d="..." />
  </svg>
  Download report
  <!-- Screen reader: "Download report, button" — icon bị skip -->
</button>

<!-- Cách đúng: hide duplicate content -->
<a href="/home">
  <img
    src="logo.png"
    alt="Acme Corp"
  />
  <span aria-hidden="true">Acme Corp</span>
  <!-- Text duplicate của alt text → hide text, AT đọc từ alt -->
</a>

<!-- ❌ SAI: aria-hidden trên element có focusable descendant -->
<div aria-hidden="true">
  <button>This button is still in tab order!</button>
  <!-- Keyboard user Tab vào button, screen reader im lặng → confusion -->
</div>

<!-- ✅ ĐÚNG: phải xử lý focusability trước khi aria-hidden -->
<div aria-hidden="true">
  <button
    tabindex="-1"
    disabled
  >
    Now not focusable
  </button>
</div>
```

### Interactive States

```javascript
// aria-expanded — Collapsible sections, dropdowns, accordions
class Accordion {
  constructor(trigger, panel) {
    this.trigger = trigger;
    this.panel = panel;
    this.isOpen = false;

    // Initial state
    // aria-expanded="false" ngay từ đầu — AT biết đây là collapsible
    trigger.setAttribute('aria-expanded', 'false');
    panel.hidden = true;

    trigger.addEventListener('click', () => this.toggle());
  }

  toggle() {
    this.isOpen = !this.isOpen;

    // Sync visual state với ARIA state
    this.trigger.setAttribute('aria-expanded', String(this.isOpen));
    this.panel.hidden = !this.isOpen;
    // Screen reader announce: "Section title, button, expanded" hoặc "collapsed"
  }
}

// aria-pressed — Toggle buttons (trạng thái on/off)
// Khác checkbox: button có thể được "pressed" mà không cần checked state
const muteButton = document.getElementById('mute-btn');
muteButton.setAttribute('aria-pressed', 'false');

muteButton.addEventListener('click', () => {
  const isPressed = muteButton.getAttribute('aria-pressed') === 'true';
  muteButton.setAttribute('aria-pressed', String(!isPressed));
  toggleMute(!isPressed);
  // Screen reader: "Mute, button, pressed" hoặc "not pressed"
});

// aria-selected — Items trong listbox, tabs, treeitem
// Khác checked: selected dùng cho selection trong collection
function selectTab(tabEl, panelEl, allTabs, allPanels) {
  // Deselect tất cả
  allTabs.forEach((t) => {
    t.setAttribute('aria-selected', 'false');
    t.setAttribute('tabindex', '-1'); // Remove from tab order
  });
  allPanels.forEach((p) => (p.hidden = true));

  // Select tab được chọn
  tabEl.setAttribute('aria-selected', 'true');
  tabEl.setAttribute('tabindex', '0'); // Add back to tab order
  panelEl.hidden = false;
  // Screen reader: "Dashboard, tab, selected, 1 of 4"
}

// aria-invalid + aria-errormessage — Form validation
function setFieldError(input, errorMessage) {
  const errorEl = document.getElementById(input.id + '-error');

  input.setAttribute('aria-invalid', 'true');

  // aria-errormessage (ES2021+) — specific cho error messages
  // Khác aria-describedby: chỉ dùng khi aria-invalid=true
  input.setAttribute('aria-errormessage', errorEl.id);

  errorEl.textContent = errorMessage;
  // Screen reader sau khi user blur: "Email, edit text, invalid entry. Must be valid email format."
}

// aria-busy — Loading states
function startLoading(region) {
  region.setAttribute('aria-busy', 'true');
  // AT biết content đang cập nhật — tạm thời không announce changes
}

function finishLoading(region) {
  region.setAttribute('aria-busy', 'false');
  // AT biết update xong, đọc content mới
}
```

## 4. ARIA Live Regions — Dynamic content announcement

### Cơ chế hoạt động bên trong browser

Live regions là một trong những phần phức tạp nhất của ARIA vì timing:

```
User action
    │
    │  DOM mutation xảy ra
    ▼
Browser's Mutation Observer (internal)
    │
    │  Check: node nằm trong aria-live region không?
    │  Check: aria-live value là gì? (off/polite/assertive)
    ▼
Accessibility Event Queue
    │
    │  "polite": wait cho screen reader idle, rồi announce
    │  "assertive": interrupt ngay lập tức, announce ngay
    ▼
Platform Accessibility API
    │  IAccessible2::EVENT_LIVE_REGION_CHANGED (Windows)
    │  AXValueChanged notification (macOS)
    ▼
Screen Reader
    │  Đọc content của live region
```

**Timing gotcha — tại sao phải clear trước khi set:**

```javascript
const liveRegion = document.getElementById('status');

// ❌ SAI: nếu content giống nhau, không có mutation → không announce
liveRegion.textContent = 'Item saved';
// ... user click save lần 2
liveRegion.textContent = 'Item saved'; // Same content → no mutation event → no announcement!

// ✅ ĐÚNG: clear trước, set sau — đảm bảo có mutation event
function announce(message, politeness = 'polite') {
  liveRegion.textContent = '';

  // requestAnimationFrame đảm bảo clear được paint trước khi set
  requestAnimationFrame(() => {
    liveRegion.textContent = message;
  });
}
```

### aria-live values — Khi nào dùng cái nào

```
aria-live="off"       → Mặc định. Không announce changes.
aria-live="polite"    → Announce khi screen reader idle (sau khi đọc xong điều đang đọc)
aria-live="assertive" → Interrupt ngay lập tức. Dùng RẤT ÍT — disruptive.
```

```html
<!-- polite: notifications, status messages, search results -->
<div
  aria-live="polite"
  aria-atomic="true"
  class="sr-only"
  id="status-region"
>
  <!-- Inject text vào đây bằng JS khi cần announce -->
</div>

<!-- assertive: critical errors, time-sensitive alerts -->
<div
  role="alert"           <!-- role="alert" implies aria-live="assertive" + aria-atomic="true" -->
  class="sr-only"
  id="error-region"
>
</div>

<!-- role="status" implies aria-live="polite" + aria-atomic="true" -->
<div role="status" id="cart-count">
  3 items in cart
</div>
```

### aria-atomic và aria-relevant

```
aria-atomic="true"  → Đọc toàn bộ region mỗi khi bất kỳ phần nào thay đổi
aria-atomic="false" → Chỉ đọc phần thay đổi (mặc định)

aria-relevant="additions removals text all"
  → Chỉ announce loại changes nào
  → Mặc định: "additions text"
  → Ít khi cần override
```

```javascript
// Ví dụ thực tế: Shopping cart counter
// atomic="true" đảm bảo đọc "3 items in cart" không phải chỉ "3"
document.getElementById('cart-count').textContent = `${count} items in cart`;

// Ví dụ: Chat application
// atomic="false" vì chỉ muốn announce message mới, không đọc lại toàn bộ history
// additions text → announce khi message được thêm vào
const chatLog = document.querySelector('[role="log"]');
// role="log" → aria-live="polite" + aria-atomic="false"
// Phù hợp cho chat, feed, log entries
```

## 5. ARIA Relationships — aria-controls, aria-owns

### aria-controls

Dùng để establish relationship giữa control và element nó controls:

```html
<!-- Button controls một panel -->
<button
  aria-expanded="false"
  aria-controls="filters-panel"
>
  Show Filters
</button>

<div
  id="filters-panel"
  hidden
>
  <!-- Filter options -->
</div>

<!-- Tab controls tabpanel -->
<div role="tablist">
  <button
    role="tab"
    aria-selected="true"
    aria-controls="panel-1"
    id="tab-1"
  >
    Overview
  </button>
  <button
    role="tab"
    aria-selected="false"
    aria-controls="panel-2"
    id="tab-2"
    tabindex="-1"
  >
    Details
  </button>
</div>

<div
  role="tabpanel"
  id="panel-1"
  aria-labelledby="tab-1"
>
  Overview content
</div>
<div
  role="tabpanel"
  id="panel-2"
  aria-labelledby="tab-2"
  hidden
>
  Details content
</div>
```

**Lưu ý:** `aria-controls` support trong screen readers không đồng đều. NVDA support tốt, JAWS support, nhưng VoiceOver (macOS) historically inconsistent. Vẫn nên dùng vì best practice, nhưng đừng rely solely vào nó cho core UX.

### aria-owns

`aria-owns` rearranges accessibility tree mà không thay đổi DOM. Dùng khi DOM order khác với logical order:

```html
<!-- DOM structure: dropdown là sibling của button -->
<div class="combobox-wrapper">
  <input
    type="text"
    role="combobox"
    aria-expanded="true"
    aria-owns="suggestions-list"
    <!-- accessibility tree: input "owns" list, dù list không phải child trong DOM -->
  />
</div>

<!-- List ở cuối body để tránh overflow clipping issues -->
<ul
  id="suggestions-list"
  role="listbox"
  <!-- Trong accessibility tree, list này là "child" của input nhờ aria-owns -->
>
  <li role="option">Apple</li>
  <li role="option">Banana</li>
</ul>
```

**Rule 1 of ARIA — Quan trọng nhất:**

> **Nếu có HTML element native làm được điều bạn cần — dùng nó. Đừng tạo custom element với ARIA.**

```html
<!-- ❌ Custom checkbox với ARIA — nhiều việc, dễ sai -->
<div
  role="checkbox"
  aria-checked="false"
  tabindex="0"
  onkeydown="handleCheckboxKey(event)"
  onclick="toggleCheckbox(this)"
>
  Subscribe to newsletter
</div>

<!-- ✅ Native checkbox — browser làm hết -->
<label>
  <input
    type="checkbox"
    name="newsletter"
  />
  Subscribe to newsletter
</label>
```

## 7. Ví dụ production-level — Custom Combobox (Autocomplete)

Combobox là một trong những patterns phức tạp nhất. ARIA spec có full pattern cho nó.

```html
<div class="combobox-container">
  <label for="fruit-search" id="fruit-label">
    Select a fruit
  </label>

  <div class="combobox-wrapper">
    <input
      id="fruit-search"
      type="text"
      role="combobox"
      autocomplete="off"
      aria-expanded="false"
      aria-autocomplete="list"        <!-- Suggests từ list, không inline -->
      aria-controls="fruit-listbox"
      aria-activedescendant=""        <!-- ID của currently highlighted option -->
    />

    <button
      tabindex="-1"                   <!-- Không trong tab order -->
      aria-label="Show fruit options"
      aria-expanded="false"           <!-- Mirror input's expanded state -->
    >
      ▼
    </button>
  </div>

  <ul
    id="fruit-listbox"
    role="listbox"
    aria-label="Fruits"
    hidden
  >
    <li role="option" id="opt-apple" aria-selected="false">Apple</li>
    <li role="option" id="opt-banana" aria-selected="false">Banana</li>
    <li role="option" id="opt-cherry" aria-selected="false">Cherry</li>
  </ul>
</div>
```

```javascript
class AccessibleCombobox {
  constructor(container) {
    this.input = container.querySelector('[role="combobox"]');
    this.listbox = container.querySelector('[role="listbox"]');
    this.options = Array.from(container.querySelectorAll('[role="option"]'));
    this.activeIndex = -1;

    this.input.addEventListener('keydown', this.handleKeydown.bind(this));
    this.input.addEventListener('input', this.handleInput.bind(this));
    this.listbox.addEventListener('click', this.handleOptionClick.bind(this));
  }

  open() {
    this.listbox.hidden = false;
    this.input.setAttribute('aria-expanded', 'true');
  }

  close() {
    this.listbox.hidden = true;
    this.input.setAttribute('aria-expanded', 'false');
    this.input.setAttribute('aria-activedescendant', '');
    this.clearHighlight();
    this.activeIndex = -1;
  }

  highlightOption(index) {
    this.clearHighlight();
    this.activeIndex = index;

    if (index >= 0 && index < this.options.length) {
      const option = this.options[index];
      option.classList.add('highlighted');

      // aria-activedescendant: focus stays on input
      // nhưng screen reader announce option được highlight
      // Đây là pattern "virtual focus" — không move DOM focus
      this.input.setAttribute('aria-activedescendant', option.id);

      // Scroll option vào view nếu listbox có scroll
      option.scrollIntoView({ block: 'nearest' });
    }
  }

  clearHighlight() {
    this.options.forEach((opt) => opt.classList.remove('highlighted'));
  }

  selectOption(option) {
    // Deselect all
    this.options.forEach((opt) => opt.setAttribute('aria-selected', 'false'));

    // Select chosen
    option.setAttribute('aria-selected', 'true');
    this.input.value = option.textContent;
    this.close();
    this.input.focus();
  }

  handleKeydown(event) {
    const isOpen = this.input.getAttribute('aria-expanded') === 'true';

    switch (event.key) {
      case 'ArrowDown':
        event.preventDefault();
        if (!isOpen) {
          this.open();
          this.highlightOption(0);
        } else {
          this.highlightOption(
            Math.min(this.activeIndex + 1, this.options.length - 1),
          );
        }
        break;

      case 'ArrowUp':
        event.preventDefault();
        if (isOpen) {
          this.highlightOption(Math.max(this.activeIndex - 1, 0));
        }
        break;

      case 'Enter':
        event.preventDefault();
        if (isOpen && this.activeIndex >= 0) {
          this.selectOption(this.options[this.activeIndex]);
        }
        break;

      case 'Escape':
        if (isOpen) {
          this.close();
        }
        break;

      case 'Tab':
        // Tab closes listbox, focus moves naturally
        this.close();
        break;
    }
  }

  handleInput() {
    const query = this.input.value.toLowerCase();
    let hasVisible = false;

    this.options.forEach((option) => {
      const match = option.textContent.toLowerCase().includes(query);
      option.hidden = !match;
      if (match) hasVisible = true;
    });

    if (hasVisible && query.length > 0) {
      this.open();
      this.activeIndex = -1;
      this.input.setAttribute('aria-activedescendant', '');
    } else {
      this.close();
    }
  }

  handleOptionClick(event) {
    const option = event.target.closest('[role="option"]');
    if (option) this.selectOption(option);
  }
}
```

## 8. Ứng dụng thực tế

### React và ARIA — Những vấn đề hay gặp

**Problem 1: Stale ARIA states sau re-render**

```jsx
// ❌ ARIA state không sync với visual state
function Dropdown({ isOpen, children }) {
  return (
    <div>
      {/* isOpen thay đổi nhưng nếu button không re-render → stale aria-expanded */}
      <button>Menu</button>
      {isOpen && <ul>{children}</ul>}
    </div>
  );
}

// ✅ ARIA state luôn phản ánh state
function Dropdown({ isOpen, onToggle, children }) {
  return (
    <div>
      <button
        aria-expanded={isOpen} // Sync với state
        aria-controls='dropdown-menu'
        onClick={onToggle}
      >
        Menu
      </button>
      <ul
        id='dropdown-menu'
        role='menu'
        hidden={!isOpen} // hidden attribute thay vì conditional render
        // Lý do: conditional render → element bị destroy → screen reader không track state
        // hidden attribute → element tồn tại trong DOM nhưng ẩn → better state continuity
      >
        {children}
      </ul>
    </div>
  );
}
```

**Problem 2: aria-live trong React**

```jsx
// ❌ aria-live region bị unmount/remount → screen reader mất track
function Notifications({ messages }) {
  return (
    <div>
      {messages.length > 0 && (
        // Conditional render: khi messages empty → unmount → mất live region
        <div aria-live='polite'>{messages[messages.length - 1]}</div>
      )}
    </div>
  );
}

// ✅ Live region phải luôn tồn tại trong DOM
// Chỉ thay đổi content, không unmount
function Notifications({ messages }) {
  return (
    <div>
      {/* Always in DOM — chỉ content thay đổi */}
      <div
        aria-live='polite'
        aria-atomic='true'
      >
        {messages.length > 0 ? messages[messages.length - 1] : ''}
      </div>
    </div>
  );
}
```

## 9. Câu hỏi kiểm tra

**Q1: Tại sao `aria-live="assertive"` nên tránh dùng cho hầu hết trường hợp?**

> `assertive` interrupt screen reader ngay lập tức, bất kể đang đọc gì. Nếu user đang đọc một paragraph dài, announcement assertive sẽ cắt ngang giữa chừng, mất context. User phải navigate lại để đọc tiếp. Cho các status messages, notifications, search results — dùng `polite`. Chỉ dùng `assertive` cho critical errors không thể bỏ qua, ví dụ "Session expired, you will be logged out in 60 seconds."

**Q2: Điều gì xảy ra khi `aria-labelledby` và `aria-label` cùng có mặt trên một element?**

> `aria-labelledby` thắng — nó có priority cao hơn trong Accessible Name Computation Algorithm. Browser lấy text của element được trỏ tới bởi `aria-labelledby` và dùng làm accessible name, ignore `aria-label` hoàn toàn. Đây là behavior đúng với spec. Nếu muốn override, phải remove `aria-labelledby`.

**Q3: Đoạn code này có vấn đề gì?**

```html
<div
  role="button"
  tabindex="0"
  onclick="save()"
>
  Save
</div>
```

> Thiếu keyboard handler. `role="button"` và `tabindex="0"` làm cho element xuất hiện trong tab order và accessible như một button trong screen reader. Nhưng ARIA không thêm behavior — click handler chỉ work với mouse click. Keyboard user nhấn Enter hoặc Space → không có gì xảy ra. Fix: thêm keydown handler, hoặc tốt hơn là dùng `<button>` thật sự mà không cần ARIA.

## Câu hỏi ôn tập (NotebookLM)

### Q1: ARIA thay đổi gì và không thay đổi gì trong browser?

**Đáp án:**

ARIA (Accessible Rich Internet Applications) chỉ modify **Accessibility Tree** — cấu trúc dữ liệu riêng biệt mà browser build song song với DOM, và expose cho assistive technology qua platform accessibility API. ARIA không thay đổi visual rendering, không thay đổi CSS behavior, và **không thêm bất kỳ JavaScript behavior nào.**

Cụ thể, ARIA có thể:

- Thay đổi role của element trong accessibility tree (`role="button"` trên `<div>`)
- Thêm name, description cho element (`aria-label`, `aria-labelledby`, `aria-describedby`)
- Communicate states (`aria-expanded`, `aria-checked`, `aria-selected`, `aria-invalid`)
- Establish relationships (`aria-controls`, `aria-owns`, `aria-labelledby` cross-element)
- Trigger live region announcements (`aria-live`, `role="alert"`, `role="status"`)

ARIA không thể:

- Làm `<div>` focusable (cần `tabindex`)
- Làm `<div role="button">` respond với Enter/Space (cần keydown handler)
- Làm element tự động announce (cần content change trong live region)
- Override visual styles
- Fix fundamentally broken DOM structure

Nguyên tắc: ARIA là communication layer với AT. Nếu interaction behavior không work với keyboard, ARIA không fix được — bạn phải fix JavaScript. Nếu element không focusable, ARIA không fix được — bạn phải thêm `tabindex`. ARIA chỉ là annotation, không phải implementation.

### Q2: Virtual focus pattern trong ARIA là gì và tại sao cần thiết?

**Đáp án:**

Virtual focus là pattern where DOM focus ở lại trên một element (thường là container như combobox input), nhưng screen reader announces một element khác như thể nó đang focused — bằng cách dùng `aria-activedescendant`.

**Cơ chế:**

1. DOM focus ở trên element có `aria-activedescendant` attribute (e.g., combobox input)
2. Attribute trỏ tới ID của element đang được "virtually focused" (e.g., một option trong dropdown)
3. Screen reader đọc element được trỏ tới thay vì element thật sự có DOM focus
4. Khi user nhấn arrow keys, JS update `aria-activedescendant` để trỏ tới option tiếp theo
5. Screen reader announce option mới mỗi khi attribute thay đổi

**Tại sao cần:** Nếu move DOM focus vào từng option trong listbox khi user nhấn arrow keys, có nhiều vấn đề:

- Input field mất focus → không thể tiếp tục type để filter
- Screen reader phải announce toàn bộ context mỗi khi focus move
- Scroll behavior và visual focus indicator phức tạp hơn

Virtual focus giải quyết: focus ở input (user vẫn type được), screen reader vẫn announce options khi user navigate. Đây là pattern chuẩn cho combobox, listbox, tree, grid, toolbar — bất kỳ composite widget nào có nhiều items có thể navigate bằng arrow keys.

**Implementation:** Phải manually scroll option vào viewport (vì không có native scroll-into-view khi không phải real focus), phải manually handle `aria-selected` update, và phải clear `aria-activedescendant` khi widget close.

### Q3: Tại sao aria-live region phải luôn tồn tại trong DOM, không được conditional render?

**Đáp án:**

Screen reader thiết lập "watch" trên elements có `aria-live` attribute ngay khi nó mount vào DOM. Khi element bị unmount và remount (conditional render), screen reader phải re-establish watch — và trong khoảng thời gian đó, mutations có thể bị miss. Quan trọng hơn, nhiều screen reader implementations cần live region ổn định trong một khoảng thời gian trước khi bắt đầu announce changes từ nó.

Có hai vấn đề cụ thể với conditional render:

**Vấn đề 1 — Race condition:** Content được inject vào live region ngay khi nó mount → screen reader chưa kịp "subscribe" → miss announcement. Timing không reliable.

**Vấn đề 2 — Screen reader buffer:** JAWS và NVDA maintain internal buffer của live regions. Khi region bị destroy, buffer bị clear. Khi remount, screen reader treat nó như region hoàn toàn mới — và có thể không announce initial content.

**Solution đúng:** Live region luôn tồn tại trong DOM từ đầu. Chỉ thay đổi text content bên trong — mutation này mới trigger announcement. Nếu cần "empty" state, để content là empty string `""`. Screen reader sẽ announce empty string như silence — không gây vấn đề.

Ngoài ra, phải clear content rồi mới set lại khi muốn re-announce cùng một message, vì mutation observer chỉ fire khi có actual change — nếu set cùng text vào element đã có text đó, không có mutation event, không có announcement. Pattern: `el.textContent = ''` → `requestAnimationFrame` → `el.textContent = message`.
