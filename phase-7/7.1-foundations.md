# Phase 7.1 — Accessibility Foundations: Cơ chế thật từ Browser đến Screen Reader

> **Tất cả nội dung Phase 7.1 đều là 🔴 PHẢI BIẾT.** Đây không phải "nice to have" — ở nhiều thị trường là legal requirement (ADA ở Mỹ, EN 301 549 ở EU). Senior engineer không biết a11y là thiếu sót nghiêm trọng.

## 1. Cơ chế thật — Accessibility Tree & Screen Reader Pipeline

### Browser không chỉ build một cây — nó build hai cây

Khi browser parse HTML, nó build **DOM Tree** mà bạn đã biết. Nhưng song song, nó còn build **Accessibility Tree** — một cây riêng biệt, được expose cho các assistive technology (AT) như screen reader.

```
HTML Parsing
     │
     ▼
┌─────────────────────────────────────────────┐
│              DOM Tree                        │
│  <html>                                      │
│    <body>                                    │
│      <nav>                                   │
│        <button class="close">X</button>      │
│      </nav>                                  │
└─────────────────────────────────────────────┘
     │
     │  Browser Engine (Blink/WebKit/Gecko)
     │  builds AOM in parallel
     ▼
┌─────────────────────────────────────────────┐
│           Accessibility Tree (AOM)           │
│  role: WebArea                               │
│    role: navigation                          │
│      role: button                            │
│        name: "X"          ← accessible name  │
│        state: focusable                      │
│        state: enabled                        │
└─────────────────────────────────────────────┘
     │
     │  Platform Accessibility API
     │  (Windows: MSAA/UIA, macOS: NSAccessibility, Linux: AT-SPI)
     ▼
┌─────────────────────────────────────────────┐
│           Screen Reader                      │
│  (NVDA, JAWS, VoiceOver, TalkBack)          │
│  Reads: "X, button"                         │
└─────────────────────────────────────────────┘
```

**Điểm quan trọng:** Screen reader KHÔNG đọc DOM trực tiếp. Nó đọc **platform accessibility API** — là abstraction layer giữa browser và OS. Browser engine là người dịch từ DOM sang accessibility tree sang platform API.

### Accessibility Tree được tính toán như thế nào

Browser xác định node nào xuất hiện trong accessibility tree theo thuật toán:

```
1. Node có display: none hoặc visibility: hidden → EXCLUDED
2. Node có aria-hidden="true" → EXCLUDED (và toàn bộ subtree)
3. Node là presentational (role="presentation" hoặc role="none") → EXCLUDED
4. Còn lại → INCLUDED, với role được tính từ:
   - Implicit ARIA role của HTML element
   - Explicit role attribute (nếu có, override implicit)
```

**Ví dụ implicit role:**

| HTML Element              | Implicit ARIA Role    |
| ------------------------- | --------------------- |
| `<button>`                | `button`              |
| `<nav>`                   | `navigation`          |
| `<main>`                  | `main`                |
| `<h1>` - `<h6>`           | `heading` (level 1-6) |
| `<a href="...">`          | `link`                |
| `<a>` (no href)           | `generic`             |
| `<input type="checkbox">` | `checkbox`            |
| `<div>`, `<span>`         | `generic` (no role)   |

**`<div>` và `<span>` không có role** — đây là lý do tại sao dùng `<div>` thay `<button>` làm click target là vấn đề: screen reader không biết đây là interactive element.

### Accessible Name Computation — Browser tính tên của element như thế nào

Khi screen reader đọc "Close button", chữ "Close" là **accessible name** của element. Browser tính theo thứ tự ưu tiên (Accessible Name Computation Algorithm):

```
Thứ tự ưu tiên (cao → thấp):

1. aria-labelledby → trỏ tới element khác, lấy text của element đó
2. aria-label → string trực tiếp
3. Native label element (<label for="...">)
4. title attribute
5. Placeholder (cho input)
6. Text content của chính element (text nodes)
7. Alt attribute (cho images)
```

```html
<!-- Accessible name = "Username" (từ aria-labelledby) -->
<span id="user-label">Username</span>
<input
  aria-labelledby="user-label"
  type="text"
/>

<!-- Accessible name = "Close dialog" (từ aria-label, override text content) -->
<button aria-label="Close dialog">X</button>

<!-- Accessible name = "X" (từ text content) -->
<button>X</button>
<!-- Screen reader đọc: "X, button" — không helpful! -->
```

## 2. WCAG 2.1/2.2 — Không phải checklist, là mental model

WCAG (Web Content Accessibility Guidelines) được tổ chức theo 4 principles — **POUR**:

```
┌─────────────────────────────────────────────────────────┐
│                    WCAG POUR Model                       │
├──────────────┬──────────────────────────────────────────┤
│ Perceivable  │ Thông tin phải có thể nhận biết được     │
│              │ → Alt text, captions, color contrast      │
│              │ → Không chỉ dùng màu để conveyor info    │
├──────────────┼──────────────────────────────────────────┤
│ Operable     │ UI phải có thể thao tác được             │
│              │ → Keyboard accessible hoàn toàn           │
│              │ → Đủ thời gian, không flashing content   │
├──────────────┼──────────────────────────────────────────┤
│ Understandable│ Content và UI phải hiểu được            │
│              │ → Language của page được xác định        │
│              │ → Error messages rõ ràng, có gợi ý       │
│              │ → Consistent navigation                   │
├──────────────┼──────────────────────────────────────────┤
│ Robust       │ Content phải tương thích với AT hiện tại │
│              │ và tương lai                              │
│              │ → Valid HTML, ARIA đúng spec              │
│              │ → Không dùng deprecated features         │
└──────────────┴──────────────────────────────────────────┘
```

**Conformance levels:**

- **Level A** — Minimum. Không đạt là broken.
- **Level AA** — Target thực tế. Hầu hết legal requirements require AA.
- **Level AAA** — Aspirational. Không expected cho toàn bộ site.

> Khi ai hỏi "site này có accessible không?" — câu trả lời chuẩn là "WCAG 2.2 Level AA compliant." Đây là ngôn ngữ industry.

## 3. Semantic HTML — Tại sao nó không phải chỉ là "best practice"

### Mỗi semantic element = free accessibility

Khi bạn dùng `<button>`, browser tự động cấp cho nó:

- Role: `button` trong accessibility tree
- Keyboard: focusable bằng Tab, activatable bằng Enter và Space
- Screen reader: announce "button" sau label
- State: disabled, pressed, v.v. được tự động track

Khi bạn dùng `<div>` + click handler:

- Role: `generic` — screen reader không biết đây là interactive
- Keyboard: **không focusable** (không có tabindex)
- Screen reader: announce như plain text, không có "button"
- State: **bạn phải tự implement toàn bộ**

```html
<!-- ❌ DIV button — bạn phải implement MỌI THỨ thủ công -->
<div
  class="btn"
  onclick="submitForm()"
  tabindex="0"
  role="button"
  aria-pressed="false"
  onkeydown="if(event.key==='Enter'||event.key===' ') submitForm()"
>
  Submit
</div>
<!-- Vẫn thiếu: disabled state, focus management, v.v. -->

<!-- ✅ Native button — browser làm tất cả cho bạn -->
<button
  type="button"
  onclick="submitForm()"
>
  Submit
</button>
```

### Document Landmark Structure

Screen reader users thường navigate bằng landmarks — họ nhảy từ section này sang section khác, không đọc toàn bộ trang từ đầu.

```
┌──────────────────────────────────────────┐
│  <header> → role: banner                 │ ← skip tới đây bằng "B"
│    <nav> → role: navigation              │ ← skip tới đây bằng "N"
│      <ul><li><a href="/home">Home</a>    │
│      ...                                 │
├──────────────────────────────────────────┤
│  <main> → role: main                     │ ← skip tới đây bằng "M"
│    <article> → role: article             │
│      <h1>Page Title</h1>                 │ ← headings navigable bằng "H"
│      <h2>Section</h2>                    │
│      <p>Content...</p>                   │
│    <aside> → role: complementary         │
├──────────────────────────────────────────┤
│  <footer> → role: contentinfo           │
└──────────────────────────────────────────┘
```

**Quy tắc:**

- Chỉ có **một `<main>`** per page
- `<header>` và `<footer>` trong `<main>` không có landmark role — chỉ top-level mới có
- Nhiều `<nav>` phải có `aria-label` phân biệt: `<nav aria-label="Primary">`, `<nav aria-label="Breadcrumb">`

## 4. Keyboard Navigation — Cơ chế focus trong browser

### Focus Management hoạt động như thế nào

Browser maintain một **focus ring** — pointer tới element đang focused. Khi user nhấn Tab, browser traverse **tab order**:

```
Tab Order Algorithm:
1. Collect tất cả focusable elements
2. Sort theo:
   a. tabindex > 0: sort theo giá trị tăng dần (1, 2, 3...)
   b. tabindex = 0 hoặc không có tabindex: DOM order
   c. tabindex < 0: KHÔNG xuất hiện trong tab order
3. Cycle qua list
```

**Focusable elements mặc định (tabindex=0 implicit):**

- `<a href="...">`
- `<button>` (không disabled)
- `<input>`, `<select>`, `<textarea>` (không disabled)
- `<details>`
- Bất kỳ element nào có `tabindex="0"` explicit

```javascript
// Programmatic focus — element phải focusable
const dialog = document.getElementById('modal-dialog');
dialog.focus(); // Chỉ work nếu dialog có tabindex="-1" hoặc là focusable element

// tabindex="-1": focusable programmatically, KHÔNG nằm trong tab order
// Dùng cho: dialog containers, elements cần receive focus nhưng không trong natural tab flow
```

### Skip Links — Pattern quan trọng nhất nhưng hay bị bỏ quên

```html
<!-- Đặt là element ĐẦU TIÊN trong <body> -->
<a
  href="#main-content"
  class="skip-link"
  >Skip to main content</a
>

<nav><!-- navigation với nhiều links --></nav>

<main
  id="main-content"
  tabindex="-1"
>
  <!-- tabindex="-1" để #main-content nhận focus sau khi skip link clicked -->
  <h1>Page Content</h1>
</main>
```

```css
/* Skip link visible chỉ khi focused — không ảnh hưởng visual design */
.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
  background: #000;
  color: #fff;
  padding: 8px;
  z-index: 100;
  transition: top 0.1s;
}

.skip-link:focus {
  top: 0; /* Hiện ra khi user Tab tới nó */
}
```

**Tại sao cần:** Keyboard user và screen reader user phải Tab qua toàn bộ navigation menu ở mỗi page nếu không có skip link. Với nav 20 items, đó là 20 Tab presses trước khi tới content. Skip link cho phép nhảy thẳng vào `<main>`.

### Focus Trap — Pattern cho Modal

Khi modal mở, focus PHẢI ở trong modal. User không được Tab ra ngoài — nếu ra ngoài, screen reader user không biết modal đang mở, và có thể tương tác với nội dung phía sau.

```javascript
class FocusTrap {
  constructor(container) {
    this.container = container;
    this.handleKeydown = this.handleKeydown.bind(this);
  }

  activate() {
    // Lưu element đang focused trước khi mở modal
    this.previousFocus = document.activeElement;

    // Focus vào element đầu tiên trong modal
    const firstFocusable = this.getFocusableElements()[0];
    firstFocusable?.focus();

    // Bắt Tab event
    this.container.addEventListener('keydown', this.handleKeydown);
  }

  deactivate() {
    this.container.removeEventListener('keydown', this.handleKeydown);

    // TRẢ focus về element đã trigger modal — critical!
    this.previousFocus?.focus();
  }

  getFocusableElements() {
    return Array.from(
      this.container.querySelectorAll(
        'a[href], button:not([disabled]), input:not([disabled]), ' +
          'select:not([disabled]), textarea:not([disabled]), ' +
          '[tabindex]:not([tabindex="-1"])',
      ),
    ).filter((el) => !el.closest('[hidden]') && !el.closest('[aria-hidden]'));
  }

  handleKeydown(event) {
    if (event.key !== 'Tab') return;

    const focusable = this.getFocusableElements();
    const firstEl = focusable[0];
    const lastEl = focusable[focusable.length - 1];

    if (event.shiftKey) {
      // Shift+Tab: nếu đang ở element đầu tiên, wrap về cuối
      if (document.activeElement === firstEl) {
        event.preventDefault();
        lastEl.focus();
      }
    } else {
      // Tab: nếu đang ở element cuối, wrap về đầu
      if (document.activeElement === lastEl) {
        event.preventDefault();
        firstEl.focus();
      }
    }
  }
}

// Usage
const modal = document.getElementById('dialog');
const trap = new FocusTrap(modal);

function openModal() {
  modal.removeAttribute('hidden');
  modal.setAttribute('aria-modal', 'true');
  trap.activate();
}

function closeModal() {
  modal.setAttribute('hidden', '');
  trap.deactivate(); // Focus tự động trả về trigger element
}
```

## 5. Screen Reader Testing — Cơ chế thật

### Screen Reader đọc content như thế nào

Screen reader intercept keyboard events **trước** khi browser xử lý. Chúng có hai modes:

```
Virtual/Browse Mode:
├── Screen reader tạo "virtual buffer" — copy của accessibility tree
├── User navigate bằng arrow keys trong buffer này
├── Screen reader intercept arrow keys, KHÔNG pass tới browser
└── Dùng cho: đọc static content, navigate headings/landmarks

Application/Forms Mode:
├── Screen reader PASS keyboard events tới browser/application
├── User interact trực tiếp với controls
└── Switch sang mode này khi focus vào form controls, widgets
```

**Đây là lý do tại sao custom widgets khó:** Bạn phải implement keyboard behavior sao cho work trong **cả hai modes**.

### VoiceOver (macOS) — Cách dùng cơ bản để test

```
Bật/Tắt:  Cmd + F5
VO key:   Caps Lock (hoặc Ctrl + Option)

Navigate:
  VO + Right/Left  → Next/Previous element
  VO + U           → Rotor (list headings, links, landmarks)
  Tab              → Next focusable element

Đọc:
  VO + A           → Đọc từ cursor position
  VO + B           → Đọc từ đầu trang
```

### NVDA (Windows, free) — Shortcuts cơ bản

```
Bật/Tắt:  Ctrl + Alt + N (hoặc desktop shortcut)
NVDA key: Insert (hoặc CapsLock)

Navigate:
  Tab              → Next focusable
  H                → Next heading (in Browse mode)
  D                → Next landmark
  1-6              → Heading level 1-6
  NVDA + Space     → Toggle Browse/Forms mode
```

### Thứ tự test thực tế

```
1. Tab through toàn bộ page — mọi interactive element có focus không?
2. Check focus visible — bạn có thể THẤY focus indicator không?
3. Dùng screen reader navigate bằng headings — cấu trúc có logical không?
4. Test tất cả interactive elements chỉ với keyboard
5. Check form labels — screen reader announce đúng không?
6. Test dynamic content — announcements có đúng thời điểm không?
```

## 6. Color Contrast — Tại sao browser không check tự động

### Cơ chế tính contrast ratio

```
Contrast Ratio = (L1 + 0.05) / (L2 + 0.05)

L1 = relative luminance của màu sáng hơn
L2 = relative luminance của màu tối hơn

Relative luminance = f(R, G, B) theo công thức sRGB
Range: 0 (black) đến 1 (white)
```

**WCAG AA requirements:**

```
Normal text (< 18pt / < 14pt bold):  contrast ratio ≥ 4.5:1
Large text (≥ 18pt / ≥ 14pt bold):  contrast ratio ≥ 3:1
UI components, icons:                contrast ratio ≥ 3:1
```

**Common fails:**

```
Light gray text on white:  #999 on #fff = 2.85:1  ❌ FAIL AA
Blue links on white:       #0066cc on #fff = 5.74:1 ✅ PASS
Placeholder text:          #aaa on #fff = 2.32:1  ❌ FAIL (common mistake!)
```

```javascript
// Tính contrast ratio trong JS
function getLuminance(r, g, b) {
  // Normalize to 0-1
  const [rs, gs, bs] = [r, g, b].map((c) => {
    c /= 255;
    return c <= 0.04045 ? c / 12.92 : Math.pow((c + 0.055) / 1.055, 2.4);
  });
  return 0.2126 * rs + 0.7152 * gs + 0.0722 * bs;
}

function getContrastRatio(hex1, hex2) {
  const toRGB = (hex) => {
    const r = parseInt(hex.slice(1, 3), 16);
    const g = parseInt(hex.slice(3, 5), 16);
    const b = parseInt(hex.slice(5, 7), 16);
    return [r, g, b];
  };

  const L1 = getLuminance(...toRGB(hex1));
  const L2 = getLuminance(...toRGB(hex2));

  const lighter = Math.max(L1, L2);
  const darker = Math.min(L1, L2);

  return (lighter + 0.05) / (darker + 0.05);
}

getContrastRatio('#767676', '#ffffff'); // 4.54 — barely passes AA
getContrastRatio('#999999', '#ffffff'); // 2.85 — fails
```

**Quan trọng:** Contrast không phải chỉ về màu sắc — còn về **không dùng màu là thông tin duy nhất.** Error states không được thể hiện chỉ bằng màu đỏ mà không có icon hoặc text.

```html
<!-- ❌ Chỉ dùng màu để thể hiện error -->
<input
  type="email"
  style="border-color: red;"
/>

<!-- ✅ Màu + text + icon -->
<div class="field-error">
  <input
    type="email"
    aria-describedby="email-error"
    aria-invalid="true"
    style="border-color: red;"
  />
  <span
    id="email-error"
    role="alert"
  >
    ⚠ Invalid email format
    <!-- Icon + text -->
  </span>
</div>
```

## 7. Alt Text — Không phải SEO, là communication

### Browser xử lý alt text như thế nào

```
<img src="chart.png" alt="..." />
         │
         ▼
Accessibility Tree:
  role: img
  name: [alt text value]  ← Screen reader reads this
```

**Quyết định alt text:**

```
Image có convey information không?
├── Có → Descriptive alt text
│   ├── Complex (chart, diagram) → Alt text + long description
│   └── Simple → Ngắn gọn, mô tả purpose không phải appearance
└── Không (decorative) → alt=""
    └── Screen reader SKIP image hoàn toàn
```

**Ví dụ thực tế:**

```html
<!-- ❌ Alt text mô tả appearance, không phải meaning -->
<img
  src="chart.png"
  alt="A bar chart with blue bars"
/>

<!-- ✅ Alt text convey information -->
<img
  src="chart.png"
  alt="Q3 revenue by region: North $2.3M, South $1.8M, West $3.1M"
/>

<!-- ❌ Redundant — screen reader đọc "image of logo" -->
<img
  src="logo.png"
  alt="Image of company logo"
/>

<!-- ✅ Meaningful -->
<img
  src="logo.png"
  alt="Acme Corp"
/>

<!-- ✅ Decorative — screen reader skip -->
<img
  src="decorative-divider.png"
  alt=""
/>

<!-- ❌ Filename là alt text — worst practice -->
<img
  src="photo.jpg"
  alt="photo.jpg"
/>

<!-- Icon với text label — alt="" vì text label đã describe -->
<button>
  <img
    src="search-icon.svg"
    alt=""
  />
  <!-- alt="" vì button text đã nói -->
  Search
</button>

<!-- Icon button không có text — cần alt text -->
<button aria-label="Search">
  <img
    src="search-icon.svg"
    alt=""
  />
  <!-- alt="" vì aria-label đã describe button -->
</button>
```

## 8. Ví dụ Production-Level

### Component có full accessibility

```jsx
// React component — Accessible Modal
import { useEffect, useRef } from 'react';

function Modal({ isOpen, onClose, title, children }) {
  const modalRef = useRef(null);
  const previousFocusRef = useRef(null);

  useEffect(() => {
    if (isOpen) {
      // Lưu focus trước khi mở
      previousFocusRef.current = document.activeElement;

      // Focus vào modal (tabindex="-1" trên div)
      // Sau đó user Tab vào first focusable element bên trong
      modalRef.current?.focus();

      // Prevent body scroll (UX + prevent AT từ reading background)
      document.body.style.overflow = 'hidden';
    } else {
      // Restore focus — CRITICAL cho screen reader UX
      previousFocusRef.current?.focus();
      document.body.style.overflow = '';
    }
  }, [isOpen]);

  // Escape key để close
  useEffect(() => {
    const handleKeydown = (e) => {
      if (e.key === 'Escape' && isOpen) onClose();
    };
    document.addEventListener('keydown', handleKeydown);
    return () => document.removeEventListener('keydown', handleKeydown);
  }, [isOpen, onClose]);

  if (!isOpen) return null;

  return (
    <>
      {/* Overlay — click để close */}
      <div
        className='modal-overlay'
        onClick={onClose}
        aria-hidden='true' // Overlay không phải content, skip trong AT
      />

      <div
        ref={modalRef}
        role='dialog' // Tell screen reader: đây là dialog
        aria-modal='true' // Tell screen reader: content behind is inert
        aria-labelledby='modal-title' // Modal name = heading text
        tabIndex={-1} // Focusable programmatically, không trong tab order
        className='modal'
      >
        <div className='modal-header'>
          <h2 id='modal-title'>{title}</h2>

          <button
            onClick={onClose}
            aria-label='Close dialog' // Clear name cho icon-only button
          >
            <svg
              aria-hidden='true'
              focusable='false'
            >
              {' '}
              {/* aria-hidden + focusable=false cho SVG */}
              <use href='#icon-close' />
            </svg>
          </button>
        </div>

        <div className='modal-body'>{children}</div>
      </div>
    </>
  );
}
```

      type="email"

      aria-invalid="false"

      id="email-hint"
      class="field-hint"
    >
      We'll never share your email with anyone.
    </span>

    <!-- 
      role="alert": screen reader tự announce khi nội dung thay đổi
      aria-live="assertive" (implicit với role="alert")
      Bắt đầu là hidden, hiện ra khi có error
    -->
    <span
      id="email-error"
      class="field-error"
      role="alert"
    >
      <!-- JS điền vào đây khi validate -->
    </span>
  </div>

  <button type="submit">
    Create account
    <span class="sr-only">(required fields marked with *)</span>
  </button>
</form>

<style>
  /* Utility class — visually hidden nhưng vẫn accessible cho AT */
  .sr-only {
    position: absolute;
    width: 1px;
    height: 1px;
    padding: 0;
    margin: -1px;
    overflow: hidden;
    clip: rect(0, 0, 0, 0);

```javascript
function validateEmail(input) {
  const errorEl = document.getElementById('email-error');
  const isValid = /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(input.value);

  if (!isValid) {
    // Set aria-invalid — screen reader announce khi refocus
    input.setAttribute('aria-invalid', 'true');

    // Inject error message vào role="alert" element
    // role="alert" trigger screen reader announcement ngay lập tức
    errorEl.textContent =
      'Please enter a valid email address, like name@example.com';
  } else {
    input.setAttribute('aria-invalid', 'false');
    errorEl.textContent = '';
  }
}
```

## 9. Ứng dụng thực tế

### Khi nào a11y issues làm app crash với screen reader users

**Pattern 1: Dynamic content không announce**

```javascript
// ❌ Content update không notify AT
function showNotification(message) {
  document.getElementById('notification').textContent = message;
  // Screen reader user: không biết gì thay đổi
}

// ✅ Dùng aria-live region
// HTML: <div id="notification" aria-live="polite" aria-atomic="true"></div>
// "polite" = đọc khi user idle
// "assertive" = ngắt ngang đọc ngay (dùng cho errors critical)
// "atomic" = đọc toàn bộ content, không chỉ phần thay đổi
function showNotification(message) {
  const el = document.getElementById('notification');
  el.textContent = ''; // Clear trước
  requestAnimationFrame(() => {
    el.textContent = message; // Set sau — đảm bảo AT detect change
  });
}
```

**Pattern 2: SPA navigation không announce page change**

```javascript
// Khi SPA navigate, DOM thay đổi nhưng screen reader vẫn ở vị trí cũ
// Focus không reset, không có announcement
router.on('routeChange', (newRoute) => {
  // Option 1: Focus vào <main> element sau navigation
  document.querySelector('main').focus();

  // Option 2: Update page title + announce
  document.title = newRoute.title;

  // Option 3: Dùng aria-live region để announce navigation
  announcer.textContent = `Navigated to ${newRoute.title}`;
});
```

**Pattern 3: Infinite scroll — screen reader bị mắc kẹt**

Screen reader user navigate bằng heading/landmark — khi content append vào cuối, họ không biết có content mới.

```html
<!-- Pattern cho infinite scroll -->
<div
  role="feed"
  aria-label="News articles"
  aria-busy="false"
>
  <!-- role="feed" là role cho infinite scroll content -->
  <!-- aria-busy="true" khi đang load thêm -->

  <article
    aria-posinset="1"
    aria-setsize="-1"
  >
    <!-- aria-posinset: vị trí trong set -->
    <!-- aria-setsize="-1": tổng số không biết (infinite) -->
  </article>
</div>
```

## 10. Câu hỏi kiểm tra

**Q1: Tại sao `<div onclick="...">` không accessible, dù có mousedown và click handler hoạt động bình thường?**

> Browser không add `<div>` vào accessibility tree với interactive role. Screen reader không biết đây là clickable. Keyboard user không thể Tab tới nó (vì không phải focusable element). Ngay cả khi thêm `tabindex="0"`, bạn vẫn phải thêm keydown handler (Enter/Space), role="button", và tất cả ARIA states thủ công. Native `<button>` làm hết tất cả miễn phí.

**Q2: Điều gì xảy ra khi bạn set `aria-hidden="true"` trên một element đang có focus?**

> Element bị loại khỏi accessibility tree, nhưng nó vẫn có thể focused bằng keyboard — một trạng thái không consistent. Screen reader sẽ không announce element đó dù focus đang ở đó, gây confusion hoàn toàn. Đây là anti-pattern nghiêm trọng. Rule: không bao giờ `aria-hidden="true"` trên element có thể focused hoặc chứa focusable elements.

**Q3: Đoạn code này có vấn đề gì?**

```javascript
function openModal() {
  modal.style.display = 'block';
  modal.querySelector('button').focus();
}

function closeModal() {
  modal.style.display = 'none';
}
```

> Hai vấn đề:
>
> 1. `closeModal()` không trả focus về element đã trigger modal. Screen reader user sau khi đóng modal sẽ không biết focus đang ở đâu — thường rơi về top of page.
> 2. Không có focus trap — user có thể Tab ra khỏi modal, đọc background content trong khi modal vẫn mở.
>    Fix: Lưu `document.activeElement` trước khi mở, trả về khi đóng. Implement focus trap.

## Câu hỏi ôn tập (NotebookLM)

### Q1: Browser build accessibility tree như thế nào và screen reader tương tác với nó ra sao?

**Đáp án:**

Browser engine (Blink trong Chrome, Gecko trong Firefox) build **Accessibility Tree** song song với DOM Tree. Đây là cấu trúc dữ liệu riêng biệt, không phải là DOM — nó là abstraction được thiết kế để expose thông tin theo cách AT có thể hiểu.

Quá trình:

1. HTML parser tạo DOM nodes
2. Browser style engine compute styles
3. Accessibility engine traverse DOM và tạo **Accessibility Objects** tương ứng — mỗi object có: role, name, state, properties, relationships
4. Nodes bị loại khỏi accessibility tree nếu: `display:none`, `visibility:hidden`, `aria-hidden="true"`, hoặc role="presentation/none"
5. Accessibility tree được expose qua **Platform Accessibility API** của OS: MSAA/UIA (Windows), NSAccessibility (macOS), AT-SPI (Linux)

Screen reader không đọc DOM, không đọc CSS, không chạy JS — nó query platform accessibility API. Khi browser update DOM (React re-render, AJAX content, v.v.), browser phải update accessibility tree và notify AT qua platform API. Đây là lý do tại sao `aria-live` regions hoạt động: browser watch changes trong subtree đó và broadcast notification.

Accessible name của mỗi element được tính theo thuật toát **Accessible Name Computation** — ưu tiên theo thứ tự: aria-labelledby → aria-label → native label → title → text content → alt attribute. Browser chạy thuật toán này mỗi khi element được focused hoặc AT query.

### Q2: Tại sao focus trap là bắt buộc khi modal mở, và cách implement đúng?

**Đáp án:**

Khi modal mở mà không có focus trap, screen reader user có thể Tab ra khỏi modal dialog và interact với content phía sau — trong khi modal vẫn đang mở. Đây là trải nghiệm broken hoàn toàn: user đang "bên trong" một dialog nhưng screen reader announce content của background page. Keyboard-only user cũng bị ảnh hưởng tương tự.

`aria-modal="true"` nói với screen reader "đây là modal, content phía sau bị inert" — nhưng không phải tất cả screen readers implement đúng. Focus trap là giải pháp reliable hơn.

Implementation đúng phải handle:

1. **Collect focusable elements** trong modal: `a[href], button:not([disabled]), input, select, textarea, [tabindex]:not([tabindex="-1"])` — loại trừ elements trong `[hidden]` hoặc `[aria-hidden]`
2. **Intercept Tab key**: nếu focus đang ở element cuối cùng, Tab wrap về element đầu tiên; nếu đang ở đầu tiên, Shift+Tab wrap về cuối cùng
3. **Intercept Escape key**: close modal
4. **Lưu previous focus** (`document.activeElement`) trước khi mở modal
5. **Restore focus** khi đóng — đây là bước hay bị bỏ quên nhất. Screen reader user cần biết mình đang ở đâu sau khi close dialog

Ngoài ra, element container của modal nên có `role="dialog"`, `aria-modal="true"`, và `aria-labelledby` trỏ tới heading của dialog. Container nên có `tabindex="-1"` để có thể programmatically focused nhưng không nằm trong tab order.

### Q3: Giải thích sự khác biệt giữa `display:none`, `visibility:hidden`, và `aria-hidden="true"` đối với accessibility tree và screen reader

**Đáp án:**

Ba cách này ảnh hưởng khác nhau đến accessibility tree:

**`display: none`:**

- Element hoàn toàn bị loại khỏi rendering tree VÀ accessibility tree
- Screen reader không thể access element
- Element không focusable
- Con cháu cũng bị loại (subtree exclusion)
- Dùng khi: content thực sự không tồn tại (dropdown đóng, tab không active)

**`visibility: hidden`:**

- Element không render (invisible) nhưng vẫn chiếm layout space
- Bị loại khỏi accessibility tree — screen reader không đọc
- Element không focusable
- KHÁC `display:none`: `visibility: visible` trên child có thể override — child visible trong khi parent invisible
- Ít dùng hơn trong modern a11y patterns

**`aria-hidden="true"`:**

- Element vẫn render visually — người sáng mắt nhìn thấy bình thường
- Bị loại khỏi accessibility tree — screen reader không đọc
- Element vẫn có thể focused bằng keyboard!
- Đây là điểm nguy hiểm: nếu element có thể focused mà `aria-hidden="true"`, keyboard user có thể Tab vào nó nhưng screen reader không announce gì — confusion hoàn toàn
- Dùng đúng cho: decorative icons bên cạnh text label, decorative images, duplicate content (icon + text label — hide icon), overlay backdrop

**Anti-pattern cần tránh:**

```html
<!-- ❌ NGUY HIỂM: button có thể focused nhưng aria-hidden=true -->
<button aria-hidden="true">Submit</button>

<!-- ❌ NGUY HIỂM: focusable element trong aria-hidden container -->
<div aria-hidden="true">
  <a href="/home">Home</a>
  <!-- keyboard user có thể Tab vào đây -->
</div>

<!-- ✅ ĐÚNG: aria-hidden cho decorative icon -->
<button>
  <svg
    aria-hidden="true"
    focusable="false"
  >
    <use href="#icon-save" />
  </svg>
  Save document
</button>
```

Rule cốt lõi: **không bao giờ** đặt `aria-hidden="true"` trên element đang có focus hoặc element chứa bất kỳ focusable element nào.
