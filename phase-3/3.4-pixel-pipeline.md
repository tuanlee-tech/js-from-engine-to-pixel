### A — Pipeline tổng quan: 5 bước từ JS đến pixel

```
JavaScript → Style → Layout → Paint → Composite
```

Không phải mọi frame đều chạy đủ 5 bước. Browser **skip** những bước không cần thiết — đây chính là chìa khóa để optimize performance.

```
Thay đổi geometry (width, height, top, margin...):
JS → Style → Layout → Paint → Composite   (full pipeline — đắt nhất)

Thay đổi visual (color, background, shadow...):
JS → Style →          Paint → Composite   (skip Layout)

Thay đổi transform/opacity:
JS → Style →                   Composite  (skip Layout + Paint — rẻ nhất)
```

**Mục tiêu của mọi animation:** Chỉ trigger Composite step. Đây là lý do `transform` và `opacity` là hai properties được dùng cho animation trong production.

### B — Style Calculation: Bước đầu tiên sau JS

Trước khi Layout, browser phải biết **computed style** của mỗi element — giá trị cuối cùng sau khi apply tất cả CSS rules (cascade, specificity, inheritance).

**Cơ chế:**

```
1. Collect tất cả stylesheets (author, user agent, user)
2. Build selector index (một số browsers dùng bloom filter để lookup nhanh)
3. Với mỗi element trong Render Tree:
   a. Tìm tất cả matching rules
   b. Sort theo specificity + order
   c. Apply cascade → computed value
   d. Resolve relative values (em → px, % → px)
   e. Apply inheritance từ parent
```

**Style invalidation:** Khi DOM thay đổi hoặc class thay đổi, browser không recalculate toàn bộ — nó **invalidate** elements bị ảnh hưởng và recalculate chỉ chúng. Nhưng một số selectors gây cascade rộng hơn:

```css
/* Selector này khi match → có thể invalidate nhiều elements */
.parent .child {
} /* Thay đổi .parent → check tất cả .child */
:nth-child(odd) {
} /* Thay đổi bất kỳ sibling → recheck tất cả */

/* Selector này scope hẹp hơn → ít invalidation hơn */
.specific-class {
} /* Chỉ invalidate elements có class này */
#unique-id {
} /* Chỉ invalidate 1 element */
```

### C — Layout (Reflow): Đắt nhất trong pipeline

Layout là quá trình browser tính toán **geometry** của mỗi element — vị trí (x, y) và kích thước (width, height) trong document.

#### C.1 — Layout là global by default

Thay đổi một element có thể trigger layout của nhiều elements khác:

```
┌─────────────────────────────────────────────────┐
│ document                                        │
│  ┌──────────────────────┐                       │
│  │ .container           │                       │
│  │  width: 50%          │ ← resize window       │
│  │  ┌──────┐ ┌────────┐ │   → .container width thay đổi
│  │  │ .col │ │ .col   │ │   → cả 2 columns reflow
│  │  │      │ │        │ │   → content bên trong reflow
│  │  └──────┘ └────────┘ │   → toàn bộ document reflow
│  └──────────────────────┘                       │
└─────────────────────────────────────────────────┘
```

**Layout dependency tree:** Phần lớn layout là **block formatting context** — children phụ thuộc vào parent size, siblings phụ thuộc lẫn nhau. Một thay đổi có thể cascade xuống toàn bộ subtree và lên các ancestors.

#### C.2 — Forced Synchronous Layout (Layout Thrashing)

Browser **batch** layout — sau khi JS execution xong, layout được tính một lần. Nhưng khi bạn **đọc layout properties từ JS**, browser buộc phải tính layout ngay lập tức (synchronously) để trả về giá trị đúng.

**Layout-triggering READ properties:**

```javascript
// Tất cả các property/method này trigger synchronous layout:
element.offsetWidth     element.offsetHeight
element.offsetTop       element.offsetLeft
element.offsetParent
element.clientWidth     element.clientHeight
element.clientTop       element.clientLeft
element.scrollWidth     element.scrollHeight
element.scrollTop       element.scrollLeft
element.getBoundingClientRect()
element.getClientRects()
window.getComputedStyle(element)  // Một số properties
window.innerWidth       window.innerHeight
window.scrollX          window.scrollY
document.scrollingElement.scrollTop
```

**Cơ chế của thrashing:**

```
JS: element.style.width = '200px'    → Layout dirty (cần recalculate)
JS: element.offsetWidth              → READ! Browser: "layout dirty, must recalculate NOW"
                                        → Synchronous reflow #1
JS: element.style.height = '100px'  → Layout dirty again
JS: element.offsetHeight             → READ! → Synchronous reflow #2
...N lần → N reflows trong 1 JS task
```

```javascript
// ❌ LAYOUT THRASHING — pattern kinh điển
function updateCards(cards) {
  cards.forEach((card) => {
    // READ: trigger sync reflow
    const width = card.offsetWidth;

    // WRITE: invalidate layout
    card.style.height = width * 0.75 + 'px'; // 4:3 ratio

    // Vòng lặp tiếp: READ lại → reflow lại
    // N cards → N reflows
  });
}

// ✅ FIX: Tách read và write phases
function updateCards(cards) {
  // READ phase — 1 reflow (nếu layout dirty)
  const widths = cards.map((card) => card.offsetWidth);

  // WRITE phase — 0 reflows thêm (chỉ invalidate, không read)
  cards.forEach((card, i) => {
    card.style.height = widths[i] * 0.75 + 'px';
  });
  // Browser sẽ reflow 1 lần sau khi JS task xong
}
```

### D — Paint: Vẽ pixels

Sau Layout, browser biết geometry của mọi element. Paint bước tiếp theo: **record drawing commands** cho mỗi visual property.

**Paint không thật sự vẽ pixel ngay** — nó tạo **Display List** (danh sách drawing commands) cho Skia (2D graphics library của Chrome). Skia sau đó execute commands này.

**Paint order (theo spec):**

```
1. Background color
2. Background image
3. Border
4. Children (theo stacking order)
5. Outline
```

**Stacking context** ảnh hưởng paint order — một element với `z-index`, `position`, `opacity < 1`, `transform`, `filter` tạo stacking context mới, được paint như một unit.

#### Properties trigger Paint (skip Layout):

```css
/* Thay đổi những cái này → Paint nhưng không Layout */
color               background-color    background-image
border-color        outline-color       box-shadow
text-shadow         border-radius       visibility
```

**Paint là expensive vì:** CPU phải rasterize (convert drawing commands thành actual pixels) cho mỗi affected layer. Với large elements hoặc complex effects (blur, shadows), rasterization tốn nhiều CPU time.

### E — Composite: Rẻ nhất — chỉ GPU

Composite là bước cuối — browser lấy các **painted layers**, upload lên GPU, và GPU combine chúng thành final frame.

**Cơ chế:**

```
Painted layers (CPU textures)
         ↓
Upload to GPU memory
         ↓
GPU: Apply transforms, opacity, blend modes
     Composite layers thành final frame
         ↓
Display
```

**Tại sao Composite rẻ?**

1. **Xảy ra trên GPU** — song song, không block main thread
2. **Không cần rasterize lại** — layer đã được painted, chỉ move/blend
3. **Sub-pixel precision** — GPU có thể translate với floating point precision
4. **60fps khả thi** — 16.67ms per frame, GPU composite trong microseconds

**Properties chỉ trigger Composite:**

```css
transform   /* translate, scale, rotate, skew, matrix */
opacity     /* 0 đến 1 */
```

**Đây là lý do mọi animation production dùng transform và opacity:**

```css
/* ❌ Animation trigger Layout mỗi frame — jank */
@keyframes move-bad {
  from {
    left: 0;
  }
  to {
    left: 500px;
  }
}

/* ✅ Animation chỉ trigger Composite — smooth */
@keyframes move-good {
  from {
    transform: translateX(0);
  }
  to {
    transform: translateX(500px);
  }
}
```

### F — Compositing Layers: Cơ chế tạo và trade-offs

#### F.1 — Layer là gì

Browser chia page thành nhiều **compositing layers** — mỗi layer là một texture được paint riêng biệt và uploaded lên GPU.

```
Page (nhìn từ side):

Layer 3: Fixed header          ████████████████  ← riêng 1 layer
Layer 2: Animated element      ░░░░░░            ← riêng 1 layer
Layer 1: Main page content     ████████████████  ← main layer
                                ████████████████
                                ████████████████
```

Khi scroll: Layer 1 và 2 move, Layer 3 (fixed) stay put. GPU chỉ cần recomposite — không repaint.

#### F.2 — Layer Promotion: Khi nào browser tạo layer mới

Browser tự động tạo compositing layer khi element có:

```javascript
// Triggers layer creation (guaranteed):
'will-change: transform'; // Explicit hint
'will-change: opacity';
'position: fixed'; // Fixed elements
'position: sticky'; // Sticky elements
'<video>'; // Video elements
'<canvas>'; // Canvas
'<iframe>'; // Iframes (cross-origin)

// Triggers layer creation (implementation-dependent):
'transform: translateZ(0)'; // GPU layer hack (cũ)
'transform: translate3d(0,0,0)'; // Variant của hack
'opacity < 1'; // Có thể tạo layer
'filter: blur(...)'; // CSS filters
'mix-blend-mode'; // Blend modes
'isolation: isolate';
'overflow: scroll'; // Có thể tạo layer
```

#### F.3 — `will-change`: Dùng đúng cách

```css
/* will-change là hint cho browser: element sẽ sớm thay đổi property này */
/* Browser có thể tạo compositing layer TRƯỚC KHI animation bắt đầu */

/* ✅ Đúng cách: add trước animation, remove sau */
.card {
  transition: transform 300ms ease;
}

.card:hover {
  will-change: transform; /* Chỉ khi hover — chuẩn bị cho animation */
  transform: translateY(-4px);
}

/* Hoặc qua JavaScript: */
element.addEventListener('mouseenter', () => {
  element.style.willChange = 'transform';
  // Browser bắt đầu tạo compositing layer
});

element.addEventListener('animationend', () => {
  element.style.willChange = 'auto'; // Release layer
});
```

**`will-change` KHÔNG nên dùng như thế này:**

```css
/* ❌ Đặt will-change trên tất cả elements */
* {
  will-change: transform; /* Mọi element đều có compositing layer → hàng GB GPU memory */
}

/* ❌ Đặt will-change cố định trên elements không animate thường xuyên */
.static-card {
  will-change: transform; /* Card không animate → layer vẫn tồn tại → waste */
}

/* ❌ Dùng will-change để fix z-index issues */
.z-index-fix {
  will-change: transform; /* Hack — tạo stacking context, không phải optimize */
}
```

#### F.4 — Layer Squashing và Layer Explosion

**Layer Squashing:** Browser cố gắng gộp nhiều layers vào 1 khi có thể (giảm memory). Nhưng overlapping elements với different compositing triggers có thể không được squash → **layer explosion**.

```
❌ Layer Explosion scenario:
.animated-list có will-change: transform  → Layer A
  .item-1 overlap với animated-list       → Phải tạo Layer B
  .item-2 overlap với animated-list       → Phải tạo Layer C
  .item-3 overlap với animated-list       → Phải tạo Layer D
  ...100 items → 101 layers → hàng trăm MB GPU memory

✅ Fix: Cấu trúc DOM tránh overlap
  Hoặc promote cả list thay vì individual items
  Hoặc dùng contain: layout paint
```

### G — CSS Containment: `contain` property 🟡

CSS Containment cho phép developer **nói với browser**: "Element này độc lập — những gì xảy ra bên trong không ảnh hưởng bên ngoài."

Browser có thể optimize layout/paint/style calculations khi biết scope bị giới hạn.

```css
.card {
  contain: layout; /* Layout bên trong không ảnh hưởng layout bên ngoài */
  contain: paint; /* Content không paint ra ngoài border */
  contain: style; /* CSS counters không leak ra ngoài */
  contain: size; /* Element không phụ thuộc content để xác định size */

  /* Shortcuts: */
  contain: content; /* layout + paint + style */
  contain: strict; /* layout + paint + style + size */
}
```

**`contain: layout` — quan trọng nhất:**

```css
.widget {
  contain: layout;
}
/*
  Khi element bên trong .widget thay đổi size:
  → Browser KHÔNG cần recalculate layout bên ngoài .widget
  → Layout scope bị giới hạn trong .widget
  → Reflow chỉ xảy ra bên trong container, không cascade ra ngoài
*/
```

**`contain: paint`:**

```css
.scroll-container {
  contain: paint;
  overflow: auto;
}
/*
  Content bên trong KHÔNG thể paint ra ngoài border của container
  → Browser KHÔNG cần paint elements bên trong khi chúng nằm ngoài viewport của container
  → Efficient scrolling containers
*/
```

### H — `content-visibility: auto` 🟡

`content-visibility: auto` là một trong những CSS properties có impact lớn nhất với minimal effort.

**Cơ chế:**

```css
.article {
  content-visibility: auto;
  contain-intrinsic-size: 0 500px; /* Placeholder size khi skipped */
}
```

Khi element nằm **ngoài viewport**:

1. Browser **skip rendering** hoàn toàn — không layout, không paint, không style calculation
2. Element vẫn chiếm space (dùng `contain-intrinsic-size` làm placeholder)
3. Scrollbar vẫn đúng size

Khi element **sắp vào viewport**:

1. Browser tự động render element
2. Element hiển thị đúng

**Impact thực tế:**

```
Trang tin tức với 50 articles:
Không có content-visibility:
  Initial load: Style + Layout + Paint cho TẤT CẢ 50 articles
  → Slow first render

Với content-visibility: auto:
  Initial load: Chỉ render articles trong/gần viewport (~3-5 articles)
  → Fast first render
  → Remaining articles rendered khi scroll đến

Google case study: content-visibility giảm rendering time 7x cho long pages
```

**Caveats:**

```css
/* content-visibility: auto ẩn elements khỏi accessibility tree khi off-screen */
/* Dùng hidden-matchable để vẫn searchable */
.article {
  content-visibility: auto;
}

/* Search (Ctrl+F) vẫn tìm được text trong content-visibility: auto elements */
/* Vì browser render element khi text match tìm thấy */
```

### I — requestAnimationFrame: Timing chính xác

`requestAnimationFrame` (rAF) là API để sync code với browser's render cycle.

**Cơ chế thật:**

```
Browser render loop (60fps = mỗi 16.67ms):

┌─────────────────────────────────────────────────────┐
│ Frame N                                             │
│                                                     │
│  Input events processing                            │
│  rAF callbacks ← your code runs HERE               │
│  Style calculation                                  │
│  Layout                                             │
│  Paint                                              │
│  Composite                                          │
│  → Display frame N                                  │
└─────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────┐
│ Frame N+1                                           │
│  ...                                                │
```

**rAF callback nhận timestamp:**

```javascript
let lastTime = 0;

function animate(currentTime) {
  // currentTime: DOMHighResTimeStamp — milliseconds từ page load
  // Precision: sub-millisecond (performance.now() precision)

  const deltaTime = currentTime - lastTime; // ms kể từ frame trước
  lastTime = currentTime;

  // Frame-rate independent animation:
  // Thay vì move 5px mỗi frame (phụ thuộc frame rate)
  // Move theo deltaTime (consistent dù frame rate thay đổi)
  const speed = 300; // pixels per second
  position += speed * (deltaTime / 1000); // pixels per ms * ms = pixels

  element.style.transform = `translateX(${position}px)`;

  if (position < 500) {
    requestAnimationFrame(animate); // Schedule frame tiếp theo
  }
}

requestAnimationFrame(animate); // Bắt đầu
```

**rAF vs setTimeout:**

```javascript
// setTimeout(fn, 0): Enqueue vào macrotask queue
//   - Chạy SAU current task và microtasks
//   - Không sync với display refresh
//   - Có thể chạy nhiều lần giữa 2 frames hoặc bị skip frame
//   - Timer resolution: ~1ms (bị clamped lên 4ms trong inactive tabs)

// requestAnimationFrame(fn):
//   - Chạy MỘT LẦN mỗi frame, ngay trước browser paint
//   - Sync với display refresh rate (60fps, 90fps, 120fps, v.v.)
//   - Tự động pause khi tab không visible (tiết kiệm battery)
//   - Timestamp chính xác

// ❌ setTimeout cho animation — tệ:
function animateBad() {
  position += 5; // 5px mỗi "tick" — không phụ thuộc thời gian thật
  element.style.left = position + 'px'; // trigger layout!
  setTimeout(animateBad, 16); // ~60fps attempt nhưng không chính xác
}

// ✅ rAF cho animation — đúng:
function animateGood(time) {
  const delta = time - lastTime;
  position += (speed * delta) / 1000; // Phụ thuộc thời gian thật
  element.style.transform = `translateX(${position}px)`; // chỉ composite!
  lastTime = time;
  rafId = requestAnimationFrame(animateGood);
}
```

**rAF và Layout Thrashing — Double rAF trick:**

```javascript
// Đôi khi cần read layout, rồi write, trong cùng một animation
// Double rAF đảm bảo write xảy ra sau layout của frame hiện tại

function measureThenAnimate(element) {
  // Frame N — Read
  const height = element.offsetHeight; // Đọc layout

  requestAnimationFrame(() => {
    // Frame N+1 — Write (layout đã được compute cho frame N)
    element.style.transform = `translateY(${height}px)`;
    // Không trigger thrashing vì chúng ta đã read ở frame trước
  });
}
```

### J — Web Animations API (WAAPI)

WAAPI cho phép tạo animations bằng JavaScript với **browser-native performance** — animations chạy off main thread.

```javascript
// WAAPI — native performance, không cần rAF loop
const animation = element.animate(
  // Keyframes
  [
    { transform: 'translateX(0)', opacity: 1 },
    { transform: 'translateX(500px)', opacity: 0.5 },
  ],
  // Options
  {
    duration: 1000, // ms
    easing: 'ease-in-out',
    delay: 200,
    iterations: Infinity, // hoặc số lần lặp
    direction: 'alternate',
    fill: 'forwards', // Giữ end state sau khi xong
  },
);

// Control
animation.pause();
animation.play();
animation.reverse();
animation.cancel();

// Timing
animation.currentTime = 500; // Jump đến 500ms
animation.playbackRate = 2; // 2x speed

// Events
animation.addEventListener('finish', () => {
  console.log('Animation done');
});

// Promises
await animation.finished; // Chờ animation xong
await animation.ready; // Chờ animation sẵn sàng play

// Get all animations trên element:
element.getAnimations();
document.getAnimations(); // Tất cả animations trên page
```

**WAAPI vs CSS Animations:**

```
CSS Animations:
✅ Declarative, dễ đọc
✅ Browser optimize tốt
✅ Chạy off main thread (transform, opacity)
❌ Khó control dynamic (pause tại arbitrary point, reverse mid-animation)
❌ JavaScript phải compute keyframes phức tạp

WAAPI:
✅ Programmatic control đầy đủ
✅ Chạy off main thread (transform, opacity) — như CSS
✅ Dynamic keyframes
✅ Promises-based
✅ Can sync multiple animations
❌ Verbose hơn CSS
```

## PHẦN 2 — VISUAL DIAGRAMS

### Pipeline: Mỗi property trigger gì

```
Property thay đổi              Pipeline steps
─────────────────────────────────────────────────────────────────
width, height, margin,
padding, top, left,
font-size, display          → [Style] [Layout] [Paint] [Composite]
                                                  ↑ Đắt nhất

color, background-color,
box-shadow, border-radius,
outline, visibility         → [Style]          [Paint] [Composite]
                                                  ↑ Trung bình

transform, opacity          → [Style]                  [Composite]
                                                  ↑ Rẻ nhất

will-change                 → Promotes to own layer (1 lần)
                              Không trigger pipeline
```

### Layout Thrashing — Timeline

```
JS Task (16ms budget):
─────────────────────────────────────────────────────────

❌ THRASHING:
│ write style    │ READ (force reflow) │ write │ READ (force reflow) │...
│ el.style.width │ el.offsetWidth      │ style │ el.offsetWidth      │
  0ms              2ms (reflow)          4ms    6ms (reflow)

  → Dùng hết budget trong 1 task, frame drop

✅ BATCHED:
│ READ all       │ WRITE all     │          (browser reflow)          │
│ offsetWidth x3 │ style x3      │ → chỉ 1 reflow sau JS task         │
  0ms              2ms             end of task → browser reflows 1 lần

  → Còn budget cho logic khác, không frame drop
```

### Compositing Layers — Memory Impact

```
Page với nhiều layers:

                GPU Memory
                ┌──────────────────────────────────┐
Layer: header   │ ████ (100KB)                     │
Layer: hero img │ ████████████████████ (2MB)       │
Layer: card 1   │ ████ (50KB)                      │
Layer: card 2   │ ████ (50KB)                      │
Layer: card 3   │ ████ (50KB)                      │
...100 cards... │ ...5MB...                        │
Layer: modal    │ ████████ (200KB)                 │
Layer: tooltip  │ ██ (20KB)                        │
                └──────────────────────────────────┘
                  Total: 7-8MB GPU memory

Low-end mobile: 256MB total GPU memory budget
→ Quá nhiều layers → GPU memory pressure → jank, crash

✅ Chỉ promote layers thật sự cần thiết:
  - Animated elements
  - Fixed/sticky elements
  - Video, canvas
  - Modal overlays
```

## PHẦN 3 — CODE VÍ DỤ PRODUCTION

### Animation System — Production Grade

```javascript
// Animation utility tận dụng đúng properties và timing
class Animator {
  #rafId = null;
  #animations = new Map();

  // Animate với WAAPI — off main thread
  animateElement(element, keyframes, options = {}) {
    // Chỉ animate transform và opacity để tránh layout/paint
    const safeKeyframes = keyframes.map((frame) => {
      const safe = {};
      // Whitelist chỉ composite-only properties
      if ('transform' in frame) safe.transform = frame.transform;
      if ('opacity' in frame) safe.opacity = frame.opacity;
      // Convert layout-triggering properties sang transform:
      if ('x' in frame) safe.transform = `translateX(${frame.x}px)`;
      if ('y' in frame) safe.transform = `translateY(${frame.y}px)`;
      return safe;
    });

    const animation = element.animate(safeKeyframes, {
      duration: 300,
      easing: 'cubic-bezier(0.4, 0, 0.2, 1)', // Material Design easing
      fill: 'forwards',
      ...options,
    });

    // will-change khi chuẩn bị animate
    element.style.willChange = 'transform, opacity';

    animation.addEventListener(
      'finish',
      () => {
        // Release compositing layer sau khi animation xong
        element.style.willChange = 'auto';
      },
      { once: true },
    );

    return animation;
  }

  // Slide in từ dưới — pattern phổ biến cho cards, modals
  slideIn(element, { from = 'bottom', distance = 20 } = {}) {
    const transforms = {
      bottom: [`translateY(${distance}px)`, 'translateY(0)'],
      top: [`translateY(-${distance}px)`, 'translateY(0)'],
      left: [`translateX(-${distance}px)`, 'translateX(0)'],
      right: [`translateX(${distance}px)`, 'translateX(0)'],
    };

    const [fromTransform, toTransform] = transforms[from];

    return this.animateElement(
      element,
      [
        { transform: fromTransform, opacity: 0 },
        { transform: toTransform, opacity: 1 },
      ],
      { duration: 400, easing: 'cubic-bezier(0.0, 0, 0.2, 1)' },
    );
  }

  // Stagger animation cho list items
  async staggerIn(elements, delay = 50) {
    // will-change cho tất cả trước khi bắt đầu
    // (browser tạo layers đồng loạt, tránh jank từng cái)
    elements.forEach((el) => {
      el.style.willChange = 'transform, opacity';
    });

    const animations = elements.map((el, i) =>
      this.animateElement(
        el,
        [
          { transform: 'translateY(16px)', opacity: 0 },
          { transform: 'translateY(0)', opacity: 1 },
        ],
        { duration: 300, delay: i * delay },
      ),
    );

    // Chờ tất cả xong, release layers
    await Promise.all(animations.map((a) => a.finished));
    elements.forEach((el) => {
      el.style.willChange = 'auto';
    });
  }
}

// Sử dụng:
const animator = new Animator();

// Khi modal mở:
animator.slideIn(modal, { from: 'bottom', distance: 40 });

// Khi load product list:
const cards = document.querySelectorAll('.product-card');
animator.staggerIn(Array.from(cards), 60);
```

### Layout Thrashing Detector — Debug Tool

```javascript
// Development-only tool: detect và warn về layout thrashing
// Monkey-patch layout-reading properties để track calls

if (process.env.NODE_ENV === 'development') {
  const layoutProperties = [
    'offsetWidth',
    'offsetHeight',
    'offsetTop',
    'offsetLeft',
    'clientWidth',
    'clientHeight',
    'scrollTop',
    'scrollLeft',
  ];

  let isWriting = false;
  let writeStackTrace = null;
  let thrashCount = 0;

  // Intercept style writes
  const originalSetProperty = CSSStyleDeclaration.prototype.setProperty;
  CSSStyleDeclaration.prototype.setProperty = function (...args) {
    isWriting = true;
    writeStackTrace = new Error().stack;
    return originalSetProperty.apply(this, args);
  };

  // Intercept layout reads
  layoutProperties.forEach((prop) => {
    const descriptor = Object.getOwnPropertyDescriptor(
      HTMLElement.prototype,
      prop,
    );
    if (!descriptor) return;

    Object.defineProperty(HTMLElement.prototype, prop, {
      get() {
        if (isWriting) {
          thrashCount++;
          console.warn(
            `⚠️ Layout Thrashing detected!\n` +
              `Reading "${prop}" after style write.\n` +
              `Write occurred at:\n${writeStackTrace}\n` +
              `Read occurred at:\n${new Error().stack}`,
          );
        }
        isWriting = false; // Reset sau read
        return descriptor.get.call(this);
      },
      configurable: true,
    });
  });

  // Report cuối mỗi frame
  function reportThrashing(time) {
    if (thrashCount > 0) {
      console.error(
        `🚨 ${thrashCount} layout thrashing operations this frame!`,
      );
      thrashCount = 0;
    }
    isWriting = false;
    requestAnimationFrame(reportThrashing);
  }
  requestAnimationFrame(reportThrashing);
}
```

```javascript
// Long article list với content-visibility
// + IntersectionObserver để load real content khi gần viewport

class EfficientArticleList {
  constructor(container) {
    this.container = container;
    this.setupContentVisibility();
    this.setupLazyContent();
  }

  setupContentVisibility() {
    // CSS được set qua JS để có thể feature-detect
    const supportsContentVisibility = CSS.supports(
      'content-visibility',
      'auto',
    );

    if (supportsContentVisibility) {
      // Thêm styles cho tất cả articles
      const style = document.createElement('style');
      style.textContent = `
        .article-item {
          /*
           * content-visibility: auto:
           * - Elements ngoài viewport: skip rendering hoàn toàn
           * - Elements trong viewport: render bình thường
           */
          content-visibility: auto;
          
          /*
           * contain-intrinsic-size: placeholder size khi skipped
           * Browser dùng size này để calculate scrollbar
           * Estimate height của 1 article — adjust theo content thật
           */
          contain-intrinsic-size: 0 300px;
        }
      `;
      document.head.appendChild(style);
    }
  }

  setupLazyContent() {
    // Lazy load heavy content (images, charts) khi article vào viewport
    const observer = new IntersectionObserver(
      (entries) => {
        entries.forEach((entry) => {
          if (!entry.isIntersecting) return;

          const article = entry.target;

          // Load actual image thay cho placeholder
          const img = article.querySelector('img[data-src]');
          if (img) {
            img.src = img.dataset.src;
            img.removeAttribute('data-src');
          }

          // Load heavy components (chart, interactive widget)
          const chartContainer = article.querySelector('[data-chart]');
          if (chartContainer && !chartContainer.dataset.loaded) {
            chartContainer.dataset.loaded = 'true';
            import('./chart.js').then(({ renderChart }) => {
              renderChart(
                chartContainer,
                JSON.parse(chartContainer.dataset.chart),
              );
            });
          }

          observer.unobserve(article);
        });
      },
      {
        rootMargin: '200px', // Load 200px trước khi visible
      },
    );

    this.container.querySelectorAll('.article-item').forEach((article) => {
      observer.observe(article);
    });
  }
}
```

### Performance Monitor — Frame Budget

```javascript
// Monitor frame budget trong production
// Alert khi frame vượt quá 16ms (60fps budget)

class FrameMonitor {
  #samples = [];
  #maxSamples = 60; // Track 60 frames (~1 giây)
  #threshold = 16.67; // 60fps
  #rafId = null;
  #lastTime = 0;
  #onLongFrame;

  constructor({ threshold = 16.67, onLongFrame } = {}) {
    this.#threshold = threshold;
    this.#onLongFrame = onLongFrame || this.#defaultLongFrameHandler;
  }

  start() {
    this.#lastTime = performance.now();
    this.#rafId = requestAnimationFrame(this.#tick.bind(this));
  }

  stop() {
    if (this.#rafId) {
      cancelAnimationFrame(this.#rafId);
      this.#rafId = null;
    }
  }

  #tick(currentTime) {
    const frameDuration = currentTime - this.#lastTime;
    this.#lastTime = currentTime;

    this.#samples.push(frameDuration);
    if (this.#samples.length > this.#maxSamples) {
      this.#samples.shift();
    }

    if (frameDuration > this.#threshold) {
      this.#onLongFrame(frameDuration, this.#getStats());
    }

    this.#rafId = requestAnimationFrame(this.#tick.bind(this));
  }

  #getStats() {
    const sorted = [...this.#samples].sort((a, b) => a - b);
    return {
      avg: this.#samples.reduce((a, b) => a + b, 0) / this.#samples.length,
      p50: sorted[Math.floor(sorted.length * 0.5)],
      p95: sorted[Math.floor(sorted.length * 0.95)],
      p99: sorted[Math.floor(sorted.length * 0.99)],
      max: sorted[sorted.length - 1],
    };
  }

  #defaultLongFrameHandler(duration, stats) {
    console.warn(
      `🐢 Long frame: ${duration.toFixed(2)}ms\n` +
        `Stats (last ${this.#maxSamples} frames):\n` +
        `  avg: ${stats.avg.toFixed(2)}ms\n` +
        `  p95: ${stats.p95.toFixed(2)}ms\n` +
        `  p99: ${stats.p99.toFixed(2)}ms`,
    );
  }

  getStats() {
    return this.#getStats();
  }
}

// Dùng trong development:
if (process.env.NODE_ENV === 'development') {
  const monitor = new FrameMonitor({
    threshold: 16.67,
    onLongFrame: (duration, stats) => {
      // Send to analytics hoặc log
      analytics.track('long_frame', { duration, ...stats });
    },
  });
  monitor.start();
}
```

## CÂU HỎI ÔN TẬP — (Dành cho NotebookLM)

### Câu 1

**Hỏi:** Tại sao `transform: translateX(100px)` nhanh hơn `left: 100px` khi dùng cho animation, dù cả hai đều di chuyển element cùng khoảng cách? Giải thích theo pixel pipeline.

**Đáp án chi tiết:** `left: 100px` trigger full pixel pipeline: Style → Layout → Paint → Composite. Layout xảy ra vì `left` là geometry property — browser phải recalculate vị trí của element và có thể cả các elements khác bị ảnh hưởng (siblings, parent). Sau layout, element phải được repainted ở vị trí mới. Trong animation chạy 60fps, toàn bộ pipeline này phải hoàn thành trong 16.67ms mỗi frame — với complex pages, thường không kịp → frame drops → jank.

`transform: translateX(100px)` chỉ trigger Style → Composite. Browser không cần recalculate layout vì `transform` không ảnh hưởng đến document flow — element vẫn chiếm space ở vị trí gốc, chỉ được "painted" ở chỗ mới về mặt visual. Quan trọng hơn, với compositing layers (element có `will-change: transform` hoặc đang animate), transform được thực hiện hoàn toàn trên GPU: browser đã paint layer texture lên GPU, animation chỉ cần GPU move texture đó — không cần CPU paint lại, không block main thread. Ngay cả khi main thread đang chạy JavaScript nặng, GPU vẫn composite đúng frame.

### Câu 2

**Hỏi:** Giải thích cơ chế của Layout Thrashing. Đoạn code dưới đây thực hiện bao nhiêu synchronous reflows? Tại sao?

```javascript
const boxes = document.querySelectorAll('.box');
boxes.forEach((box) => {
  box.style.width = box.parentElement.offsetWidth / 2 + 'px';
});
```

**Đáp án chi tiết:** Đoạn code này thực hiện N synchronous reflows với N là số `.box` elements. Cơ chế: Khi `box.style.width = ...` execute, browser mark layout là "dirty" — geometry cần được recalculate. Trong iteration tiếp theo, `box.parentElement.offsetWidth` là một READ của layout property. Browser phải trả về giá trị chính xác của `offsetWidth` ngay lập tức — nhưng layout đang "dirty" từ write ở iteration trước. Browser buộc phải perform synchronous reflow toàn bộ document để compute giá trị đúng trước khi trả về. Sau đó `box.style.width = ...` write lại → layout dirty → iteration tiếp theo read → reflow lại. Pattern này tạo N reflows trong 1 JS task.

Fix: Tách read và write phases hoàn toàn.

```javascript
// Read phase — tối đa 1 reflow
const widths = Array.from(boxes).map((box) => box.parentElement.offsetWidth);
// Write phase — 0 reflow thêm
boxes.forEach((box, i) => {
  box.style.width = widths[i] / 2 + 'px';
});
// Browser reflow 1 lần sau JS task
```

Hoặc tốt hơn — dùng CSS: `.box { width: 50%; }` — zero JavaScript, zero reflow.

### Câu 3

**Hỏi:** `will-change: transform` làm gì chính xác trong browser? Tại sao đặt nó trên tất cả elements lại là anti-pattern? Khi nào nên add và remove nó?

**Đáp án chi tiết:** `will-change: transform` là hint cho browser: "Element này sắp có transform thay đổi." Browser response bằng cách tạo một compositing layer riêng cho element đó — paint element's content thành một GPU texture tách biệt với main page layer. GPU texture này được uploaded lên GPU memory. Khi transform thay đổi (via animation), GPU chỉ cần move/rotate/scale texture — không cần CPU repaint, không cần main thread.

Anti-pattern khi đặt trên tất cả elements: Mỗi compositing layer là một GPU texture. Một page với 500 elements đều có `will-change: transform` sẽ tạo 500 GPU textures. GPU memory (VRAM) rất giới hạn, đặc biệt trên mobile (64-256MB total). Một texture cho một div 1920x1080 tốn ~8MB (RGBA, không compressed). 500 layers có thể dùng hàng GB GPU memory — vượt budget → browser phải evict textures, re-upload → jank, hoặc crash trên low-end devices. Ngoài ra, tạo và destroy compositing layers cũng có cost — làm liên tục trong scroll, hover là overhead không cần thiết.

Pattern đúng: Add `will-change` dynamically chỉ ngay trước khi animation bắt đầu (hover in, modal open), remove ngay sau khi animation kết thúc (animationend, transitionend). Điều này tạo layer chỉ khi cần, release memory ngay khi xong.

```javascript
el.addEventListener('mouseenter', () => {
  el.style.willChange = 'transform'; // Tạo layer
});
el.addEventListener('mouseleave', () => {
  el.addEventListener(
    'transitionend',
    () => {
      el.style.willChange = 'auto'; // Release layer
    },
    { once: true },
  );
});
```
