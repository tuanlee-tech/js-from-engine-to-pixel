## PHẦN 1 — CƠ CHẾ THẬT

### 1.1 — Cái gì xảy ra khi bạn gõ URL và nhấn Enter?

Đây không phải câu hỏi mang tính "interview cliché". Đây là chuỗi cơ chế thật mà mỗi bước đều ảnh hưởng trực tiếp đến performance.

#### Bước 1: URL Parsing & Cache Check

Browser không gửi request ngay. Nó làm những việc này trước:

```
URL → Parse scheme (https?) → Check HSTS preload list
    → Check browser cache (disk cache, memory cache)
    → Nếu cache valid → DONE, không cần network
    → Nếu không → tiếp tục
```

**HSTS (HTTP Strict Transport Security):** Browser giữ một list domain đã khai báo "tôi chỉ nhận HTTPS". Khi gặp `http://bank.com`, browser tự redirect thành `https://bank.com` **trước khi gửi bất kỳ request nào ra network**. Đây là tại sao HSTS quan trọng hơn redirect phía server — redirect phía server vẫn gửi 1 request HTTP ra ngoài.

#### Bước 2: DNS Resolution

```
Browser DNS cache → OS DNS cache → Router cache
→ Recursive Resolver (ISP)
→ Root Name Server → TLD Name Server (.com)
→ Authoritative Name Server của domain
→ IP Address trả về
```

DNS lookup là **blocking** — browser phải có IP mới tiến hành TCP handshake. Đây là lý do `dns-prefetch` tồn tại.

**DNS TTL:** Record DNS có Time-To-Live. CDN thường set TTL thấp (60s) để có thể chuyển traffic nhanh. Site thường set TTL cao hơn (3600s+) để giảm lookup latency cho người dùng quay lại.

#### Bước 3: TCP Handshake

```
Client → [SYN] → Server
Client ← [SYN-ACK] ← Server
Client → [ACK] → Server
          (kết nối established)
```

**Một Round-Trip Time (RTT).** Với server ở Mỹ, RTT từ Việt Nam ~200ms. Chỉ để bắt tay TCP đã mất 200ms chưa gửi được byte nào.

**TCP Slow Start:** TCP không gửi full bandwidth ngay. Nó bắt đầu nhỏ (initial congestion window ~10 segments ≈ 14KB), rồi tăng dần. Đây là lý do tại sao **14KB đầu tiên của response cực kỳ quan trọng** — nếu HTML critical rendering path nằm gọn trong 14KB đầu, browser có thể bắt đầu render mà không cần round-trip thêm.

#### Bước 4: TLS Handshake (với HTTPS)

```
TLS 1.2: 2 RTT thêm sau TCP handshake
TLS 1.3: 1 RTT (hoặc 0-RTT với session resumption)
```

TLS 1.3 (2018) giảm handshake từ 2 RTT xuống 1 RTT. Với latency 200ms, đây là 200ms tiết kiệm cho mỗi new connection. `preconnect` resource hint giúp làm TCP + TLS handshake sớm, trước khi browser thực sự cần resource.

#### Bước 5: HTTP Request/Response

Browser gửi HTTP request. Server trả response.

**HTTP/1.1:** Mỗi request trên 1 connection. Browser mở 6 connection song song tối đa per domain (hard limit). Đây là nguồn gốc của "domain sharding" — trick cũ tạo nhiều subdomain để bypass limit.

**HTTP/2:** Multiplexing — nhiều request trên 1 connection, không cần domain sharding. Server Push — server gửi resource trước khi client hỏi (nhưng bị deprecated vì khó implement đúng). Header compression với HPACK.

**HTTP/3:** Chạy trên QUIC (UDP thay vì TCP). Giải quyết "head-of-line blocking" của TCP — với TCP, 1 packet mất → tất cả stream phải chờ. QUIC xử lý packet loss per-stream.

#### Bước 6: HTML Parsing bắt đầu — Streaming

Đây là điểm quan trọng: **browser không chờ toàn bộ HTML về mới parse**. Nó parse streaming — nhận bao nhiêu bytes thì parse bấy nhiêu.

```
Bytes đến → Tokenizer → Tree Construction Algorithm → DOM Node
```

**Tokenizer** chuyển raw bytes thành tokens: DOCTYPE, start tag, end tag, character data, comment.

**Tree Construction Algorithm** (state machine 80+ states) nhận tokens và build DOM tree. Nó xử lý malformed HTML (thiếu closing tag, v.v.) theo spec để đảm bảo tất cả browsers render giống nhau.

### 1.2 — Critical Rendering Path (CRP)

Đây là pipeline mà browser phải hoàn thành trước khi user thấy bất kỳ pixel nào.

```
HTML bytes → DOM
CSS bytes  → CSSOM
           DOM + CSSOM → Render Tree → Layout → Paint → Composite
```

#### DOM Construction

```javascript
// HTML này...
<div class="container">
  <p>Hello <em>world</em></p>
</div>

// ...thành DOM tree này:
Document
└── html
    ├── head
    └── body
        └── div.container
            └── p
                ├── "Hello "
                └── em
                    └── "world"
```

DOM construction là **incremental** (streaming). Mỗi token mới được add vào tree ngay.

#### CSSOM Construction

CSSOM construction **không incremental** — browser phải có toàn bộ CSS của một stylesheet trước khi build CSSOM. Tại sao? Vì CSS cascade — rule sau có thể override rule trước. Browser không thể áp dụng style một nửa.

```css
/* Rule sau override rule trước */
p {
  color: black;
}
p {
  color: red;
} /* Phải biết cái này tồn tại trước khi apply style */
```

**CSS là render-blocking:** Browser dừng render (không phải parse) khi gặp `<link rel="stylesheet">`. Nó phải download và parse CSS xong mới tiếp tục render. Tại sao? Vì render với CSSOM chưa đầy đủ sẽ tạo "Flash of Unstyled Content" (FOUC).

#### Render Tree Construction

Render Tree ≠ DOM Tree. Render Tree chỉ chứa nodes sẽ được render.

```
DOM Node có style display:none → KHÔNG có trong Render Tree
DOM Node ::before/::after     → CÓ trong Render Tree (dù không có trong DOM)
<head>, <script>              → KHÔNG có trong Render Tree
```

```
DOM Tree          CSSOM              Render Tree
div.container  +  .container{...} → div.container (với computed styles)
  p            +  p{color:red}    → p (color: red)
  em           +  em{font:italic} → em (font-style: italic)
  <!-- comment -->                → (bỏ qua)
```

#### Layout (Reflow)

Browser tính toán **chính xác** vị trí và kích thước của mỗi element trong Render Tree. Output là "box model" của mỗi element — x, y, width, height.

Layout phải chạy lại khi:

- DOM node được thêm/xóa
- Element thay đổi size (width, height, padding, margin, font-size)
- Window resize
- Bạn đọc layout properties từ JS (offsetWidth, getBoundingClientRect...)

**Layout là expensive nhất** vì một thay đổi nhỏ có thể trigger reflow của toàn bộ document (layout của child ảnh hưởng parent, parent ảnh hưởng siblings...).

#### Paint

Browser "vẽ" pixels cho mỗi visual layer. Paint record các draw calls — không phải pixel thực sự. Thứ được paint:

- Text
- Colors, backgrounds
- Borders
- Shadows
- Images

Paint output là **Display List** — danh sách các drawing commands.

#### Composite

Browser chia page thành các **compositing layers**, paint mỗi layer trên GPU, rồi composite (gộp) chúng lại thành final frame.

**Tại sao layers tồn tại?** Để một số animation không cần repaint. Nếu một element animate `transform`, và nó có layer riêng, chỉ cần di chuyển layer trên GPU — không repaint, không relayout.

**Cái gì trigger layer creation:**

- `will-change: transform` hoặc `will-change: opacity`
- `transform: translateZ(0)` hoặc `translate3d(0,0,0)` (hack cũ)
- `position: fixed`
- `<video>`, `<canvas>`, `<iframe>`
- CSS filter, opacity < 1

### 1.3 — Script Loading: Parser Blocking vs Async vs Defer

Đây là một trong những concepts quan trọng nhất — sai ở đây có thể làm site load chậm hàng giây.

#### Parser Blocking (không có attribute)

```html
<script src="analytics.js"></script>
```

**Cơ chế:**

1. HTML parser gặp `<script>` tag
2. Parser **dừng lại hoàn toàn** (blocked)
3. Browser download `analytics.js`
4. Browser execute script (vì script có thể gọi `document.write()`, modify DOM)
5. Sau khi execute xong, parser tiếp tục

**Tại sao lại block?** Vì script có thể gọi `document.write()` để inject thêm HTML vào stream. Browser không thể biết trước script có làm vậy không, nên phải dừng parser và chờ.

```
HTML parsing: ====|         |============================
Script download:  |---------|
Script execute:             |=|
                  BLOCKED HERE
```

**CSS cũng blocking script execution:** Nếu có `<link rel="stylesheet">` trước `<script>`, script execution bị block cho đến khi CSS download + parse xong. Lý do: script có thể đọc computed styles (`getComputedStyle`), nên CSSOM phải sẵn sàng.

#### async

```html
<script
  async
  src="analytics.js"
></script>
```

**Cơ chế:**

1. HTML parser gặp `async` script
2. Browser bắt đầu download **song song** với HTML parsing (không block)
3. Khi download xong → parser **dừng lại**, execute script ngay lập tức
4. Execute xong → parser tiếp tục

```
HTML parsing: ============================|   |====
Script download:  |---------|
Script execute:             NGAY KHI XONG |=|
                                           BLOCK NGẮN
```

**Đặc điểm quan trọng:**

- Không đảm bảo thứ tự execution (script nào download xong trước chạy trước)
- Không chờ DOM ready
- Không chờ scripts khác

**Dùng khi:** Script độc lập, không phụ thuộc DOM hay script khác. Ví dụ: analytics pixel, A/B testing.

#### defer

```html
<script
  defer
  src="app.js"
></script>
```

**Cơ chế:**

1. HTML parser gặp `defer` script
2. Browser download song song với HTML parsing (không block)
3. Parser tiếp tục cho đến khi HTML **parse xong hoàn toàn**
4. Sau khi HTML parse xong, **trước** `DOMContentLoaded` event → execute tất cả defer scripts **theo đúng thứ tự**

```
HTML parsing: ====================================|
Script download:  |---------|
Script execute:                                   |=| (sau HTML parse xong)
                                                  ↑ Đúng thứ tự
```

**Đặc điểm quan trọng:**

- Đảm bảo thứ tự: multiple defer scripts execute theo thứ tự xuất hiện trong HTML
- DOM đã sẵn sàng khi script chạy
- Execute trước `DOMContentLoaded`

**Dùng khi:** Script cần DOM, hoặc có dependency với scripts khác. Đây là lựa chọn tốt nhất cho hầu hết scripts.

#### type="module"

```html
<script
  type="module"
  src="app.js"
></script>
```

Module scripts **có behavior như `defer` by default**. Thêm vào đó:

- Luôn strict mode
- Scope riêng (không leak vào global)
- `import` được hỗ trợ

#### Preload Scanner — "Speculative Parsing"

Browser có một trick quan trọng: **preload scanner**. Đây là một parser phụ chạy song song với main HTML parser. Khi main parser bị block bởi một script, preload scanner vẫn tiếp tục scan HTML phía sau để tìm resources (`<img>`, `<link>`, `<script>`) và bắt đầu download chúng trước.

Đây là lý do tại sao putting scripts ở `<head>` không terrible như người ta nghĩ — preload scanner vẫn tìm thấy images phía dưới và download chúng.

### 1.4 — Resource Hints

Resource hints là cách bạn "nói chuyện" với browser về intent của mình, để browser có thể optimize việc loading resources.

#### dns-prefetch 🔴

```html
<link
  rel="dns-prefetch"
  href="//fonts.googleapis.com"
/>
```

**Cơ chế:** Browser thực hiện DNS lookup cho domain này **ngay lập tức**, trong background, mà không cần user click hay bạn request resource.

**Chi phí:** DNS lookup thường 20-120ms. `dns-prefetch` làm điều này sớm để khi cần resource từ domain đó, DNS đã được resolve.

**Dùng khi:** Third-party domains bạn biết sẽ dùng (CDN, fonts, analytics, payment gateways).

**Lưu ý:** `dns-prefetch` rẻ về resource, có thể dùng nhiều. Nhưng không nên abuse — mỗi lookup vẫn có overhead nhỏ.

#### preconnect 🔴

```html
<link
  rel="preconnect"
  href="https://fonts.googleapis.com"
/>
<link
  rel="preconnect"
  href="https://fonts.gstatic.com"
  crossorigin
/>
```

**Cơ chế:** Browser thực hiện **full connection setup** — DNS lookup + TCP handshake + TLS handshake — cho domain này trong background. Khi bạn thực sự cần resource, connection đã sẵn sàng.

**Chi phí:** Cao hơn `dns-prefetch` đáng kể — mỗi connection tốn CPU và network. Browser thường giữ connection idle tối đa 10 giây.

**Dùng khi:** Domains quan trọng mà bạn chắc chắn sẽ request resource trong vài giây tới. Tối đa 2-3 domains.

**Vì sao cần `crossorigin` với fonts?** Font requests là CORS requests. Nếu preconnect không có `crossorigin`, browser sẽ tạo anonymous connection. Khi font request thực sự xảy ra (là CORS), browser cần tạo connection mới với credentials — preconnect bị lãng phí.

#### preload 🔴

```html
<link
  rel="preload"
  href="/fonts/custom.woff2"
  as="font"
  type="font/woff2"
  crossorigin
/>
<link
  rel="preload"
  href="/css/critical.css"
  as="style"
/>
<link
  rel="preload"
  href="/js/app.js"
  as="script"
/>
```

**Cơ chế:** Browser download resource này **ngay lập tức, với high priority**, bất kể resource đó được reference ở đâu trong HTML/CSS.

**Quan trọng:** `preload` chỉ download, không execute. Bạn vẫn phải có `<script src>` hoặc `@font-face` để sử dụng resource đó.

**`as` attribute bắt buộc:** Để browser biết priority và set đúng Accept header. Nếu thiếu `as`, browser sẽ load với low priority và có thể load lần 2 khi gặp actual reference.

**Dùng khi:**

- Fonts (không bị discover cho đến khi CSSOM build xong và browser biết font nào cần)
- CSS critical path
- Script quan trọng bị defer hoặc nằm sâu trong HTML
- Hero image (LCP element)

**Nguy hiểm:** `preload` với resource không được dùng sẽ trigger console warning và waste bandwidth. Đừng preload mọi thứ.

#### prefetch 🟡

```html
<link
  rel="prefetch"
  href="/next-page.html"
/>
<link
  rel="prefetch"
  href="/images/big-image.jpg"
/>
```

**Cơ chế:** Browser download resource này với **lowest priority**, trong idle time, dự đoán user sẽ cần nó sau. Được cache vào disk cache.

**Khác preload:** `preload` = cần ngay, high priority. `prefetch` = có thể cần sau, lowest priority.

**Dùng khi:** Next page trong multi-step flow (checkout step 2 khi user đang ở step 1), hover-triggered content.

### 1.5 — Back/Forward Cache (bfcache) 🟡

bfcache là một trong những performance features ít được biết đến nhất, nhưng có impact lớn.

**Cơ chế:** Khi user navigate away từ một page, thay vì unload page đó, browser **freeze toàn bộ page** — bao gồm JavaScript execution state, DOM state, form values — và lưu vào memory. Khi user nhấn Back/Forward, browser **restore page từ memory**, không cần reparse HTML, reexecute JS, refetch resources. Gần như instant.

**Khi nào page KHÔNG được cache vào bfcache:**

- Page set `Cache-Control: no-store`
- Page dùng `unload` event listener (unload event không fire khi page vào bfcache)
- Page có `<object>` hay `<embed>` plugins
- Page có pending IndexedDB transactions
- OpenedWindow references (window.opener)

**`unload` event là kẻ thù của bfcache:**

```javascript
// ĐỪNG làm thế này — block bfcache
window.addEventListener('unload', () => {
  // cleanup code
});

// Dùng pagehide thay thế
window.addEventListener('pagehide', (event) => {
  if (event.persisted) {
    // Page đang vào bfcache — đừng cleanup mạnh tay
  } else {
    // Page thật sự bị unload — cleanup bình thường
  }
});

// Khi page restore từ bfcache
window.addEventListener('pageshow', (event) => {
  if (event.persisted) {
    // Restored từ bfcache — refresh data nếu cần
    refreshUserData();
  }
});
```

**Tại sao quan trọng:** Theo Chrome data, navigation với bfcache nhanh hơn **10x** so với regular navigation. Đây là một trong những single biggest performance wins.

**Kiểm tra trong DevTools:** Application tab → Back/forward cache → Test button.

### 1.6 — Browser Process Architecture 🟢

Modern browsers là **multi-process architecture**, không phải single process như thời IE6.

```
┌─────────────────────────────────────────────────────────────┐
│                     Browser Process                          │
│  (UI, address bar, tabs management, network coordinator)     │
└─────────────────────┬───────────────────────────────────────┘
                      │ IPC (Inter-Process Communication)
          ┌───────────┼──────────────────┐
          │           │                  │
┌─────────▼──┐ ┌──────▼─────┐ ┌────────▼───────┐
│ Renderer   │ │ Renderer   │ │ GPU Process    │
│ Process    │ │ Process    │ │ (compositing,  │
│ (Tab A)    │ │ (Tab B)    │ │  WebGL, video) │
│            │ │            │ └────────────────┘
│ - HTML parse│ │ - HTML parse│
│ - JS engine │ │ - JS engine │ ┌────────────────┐
│ - Layout   │ │ - Layout   │ │ Network Process│
│ - Paint    │ │ - Paint    │ │ (HTTP, cache,  │
└────────────┘ └────────────┘ │  DNS, TLS)     │
                               └────────────────┘
```

**Renderer Process (quan trọng nhất):**

- Mỗi tab (thường) có Renderer Process riêng
- Chứa: Blink rendering engine (layout, DOM, CSS) + V8 engine
- Chạy trong **sandbox** — không có direct access vào OS resources
- Nếu tab crash → chỉ tab đó crash, không crash cả browser

**Site Isolation (🟢):**
Trước 2018, multiple tabs của cùng một window có thể share Renderer Process (để tiết kiệm memory). Sau Spectre/Meltdown vulnerabilities, Chrome implement **Site Isolation**: mỗi **site** (scheme + eTLD+1) có Renderer Process riêng.

```
https://bank.com       → Renderer Process A
https://evil.com       → Renderer Process B
https://sub.bank.com   → Renderer Process A (cùng site)
https://other-bank.com → Renderer Process C (khác site)
```

**Tại sao Site Isolation quan trọng:** Spectre exploit cho phép một website đọc memory của process khác. Nếu bank.com và evil.com share process, evil.com có thể đọc data của bank.com. Site Isolation ngăn chặn điều này.

**Trade-off:** Nhiều processes hơn = nhiều memory hơn. Mỗi Renderer Process tốn ~50MB overhead.

### Critical Rendering Path — Blocking vs Non-blocking

```
Trường hợp XẤU (script blocking, CSS blocking):
─────────────────────────────────────────────────
[HTML Parse] ──► BLOCK ◄── [CSS Download + Parse]
                   ↓
              [Script Download + Execute]
                   ↓
          [CSSOM ready] ──► [Continue HTML Parse]
                                    ↓
                              [DOM + CSSOM] ──► [Render Tree] ──► [Layout] ──► [Paint]

Trường hợp TỐT (async/defer, preload):
────────────────────────────────────────
[HTML Parse] ────────────────────────────────────► Done
[CSS Download] ──────────────► [CSSOM Build]
[Script Download] ─────────────────────────────────► [Execute after DOM]
[Preloaded Font] ──────────►
                              [Render Tree] ──► [Layout] ──► [Paint] ◄── FIRST PAINT SỚM HƠN
```

### Script Loading Comparison

```
               |── HTML Parse ──────────────────────────────────────►|
               |                                                      |
<script>:      |══BLOCK══|── Download ──|═Execute═|── Continue Parse ──►|

<script async>:|── Parse ──────────────────|     |── Continue ──►|
               |          |── Download ──|       |
               |                         |=Exec=|

<script defer>:|── Parse ──────────────────────────────────►|
               |   |── Download ──|                         |═Exec═|
               |                                                    |→DOMContentLoaded

type="module": Như defer, nhưng:
               - Strict mode tự động
               - Scope riêng
               - import/export support
```

### Resource Hint Priority

```
High Priority ──────────────────────────────────── Low Priority

[preload]  →  [preconnect]  →  [dns-prefetch]  →  [prefetch]
  ↑               ↑               ↑                  ↑
Cần ngay       Sẽ cần         DNS thôi          Có thể cần
Full download  TCP+TLS ready  (20-120ms)        khi idle
Blocking slot  Connection     Lookup only       Background
```

## PHẦN 3 — CODE VÍ DỤ PRODUCTION-LEVEL

### 3.1 — Resource Hints đúng cách cho Google Fonts

```html
<!DOCTYPE html>
<html>
  <head>
    <!-- 
    Bước 1: preconnect với fonts.googleapis.com (để lấy CSS)
    Bước 2: preconnect với fonts.gstatic.com (để lấy font files)
    crossorigin PHẢI có vì font request là CORS request
    Nếu thiếu crossorigin → browser tạo anonymous connection,
    sau đó font request cần credentialed connection → connection mới = lãng phí preconnect
  -->
    <link
      rel="preconnect"
      href="https://fonts.googleapis.com"
    />
    <link
      rel="preconnect"
      href="https://fonts.gstatic.com"
      crossorigin
    />

    <!-- 
    Load font CSS bình thường
```

### 3.2 — Layout Thrashing Bug (production pattern)

```javascript
// ❌ BUG PHỔ BIẾN: DOM read/write xen kẽ
// Mỗi lần đọc layout property sau khi đã write → force synchronous reflow
function resizeElementsBroken(elements) {
  elements.forEach((el) => {
    // READ: el.offsetHeight → browser phải tính layout để trả về giá trị
    // Nhưng trước đó ta đã write style ở iteration trước → layout dirty
    // → Browser PHẢI reflow để trả về giá trị đúng
    // Với 100 elements → 100 synchronous reflows
    const currentHeight = el.offsetHeight; // FORCE REFLOW

    // WRITE: invalidate layout
    el.style.height = currentHeight * 1.5 + 'px'; // layout dirty
  });
}

// ✅ FIX 1: Tách read và write phase
function resizeElementsBetter(elements) {
  // READ PHASE: đọc tất cả giá trị (layout clean, chỉ 1 lần reflow nếu cần)
  const heights = elements.map((el) => el.offsetHeight);

  // WRITE PHASE: write tất cả (chỉ invalidate layout 1 lần)
  elements.forEach((el, i) => {
    el.style.height = heights[i] * 1.5 + 'px';
  });
  // Layout sẽ được tính toán 1 lần duy nhất ở frame tiếp theo
}

// ✅ FIX 2: FastDOM library pattern (dùng trong large codebase)
// Đảm bảo không bao giờ read sau write trong cùng frame
const fastdom = {
  reads: [],
  writes: [],
  scheduled: false,

  measure(fn) {
    // đặt lịch read
    this.reads.push(fn);
    this._scheduleFlush();
  },

  mutate(fn) {
    // đặt lịch write
    this.writes.push(fn);
    this._scheduleFlush();
  },

  _scheduleFlush() {
    if (!this.scheduled) {
      this.scheduled = true;
      requestAnimationFrame(() => this._flush());
    }
  },

  _flush() {
    // Reads trước
    this.reads.forEach((fn) => fn());
    this.reads = [];
    // Writes sau
    this.writes.forEach((fn) => fn());
    this.writes = [];
    this.scheduled = false;
  },
};

// Dùng:
elements.forEach((el) => {
  fastdom.measure(() => {
    const height = el.offsetHeight; // Sẽ chạy trong read phase
    fastdom.mutate(() => {
      el.style.height = height * 1.5 + 'px'; // Sẽ chạy trong write phase sau
    });
  });
});

// ✅ FIX 3: ResizeObserver thay vì đọc size manually
// ResizeObserver callback chạy sau layout, trước paint
// → Không cần đọc offsetHeight, không trigger reflow thêm
const resizeObserver = new ResizeObserver((entries) => {
  // entries chứa sẵn new size → không cần đọc từ DOM
  entries.forEach((entry) => {
    const { width, height } = entry.contentRect;
    // Có thể safely write ở đây
  });
});
elements.forEach((el) => resizeObserver.observe(el));
```

## PHẦN 4 — ỨNG DỤNG THỰC TẾ

### 4.1 — Performance Impact theo từng bước

**DNS → TCP → TLS là "hidden latency":**
Với user ở Việt Nam, kết nối đến server US:

- DNS: 20-120ms
- TCP handshake: ~200ms (1 RTT)
- TLS 1.3: ~200ms thêm (1 RTT)
- → **~500ms trước khi gửi được byte đầu tiên**

`preconnect` loại bỏ 400ms (TCP + TLS) cho third-party domains quan trọng.

**CRP bottlenecks thường gặp trong React apps:**

```
1. Large JS bundle blocking parse:
   → Code splitting (dynamic import)
   → Server-side rendering

2. Render-blocking CSS:
   → Extract critical CSS + inline
   → Load non-critical CSS async

3. Fonts gây FOIT (Flash of Invisible Text):
   → font-display: swap trong @font-face
   → Preload font files

4. LCP image không được preload:
   → Add preload hint cho hero image
```

### 4.2 — Hay gặp ở đâu

**React:**

- `React.lazy()` + `Suspense` = dynamic import + loading state
- Next.js `next/image` = automatic preload cho LCP images + responsive sizes
- Next.js `next/script` = wrapper cho async/defer/idle loading strategies
- `React.Suspense` triggers khi component đang load async resource

**DevTools — Kiểm tra CRP:**

1. **Network tab:** Xem request waterfall. Màu tím = DNS lookup, màu cam = SSL, màu xanh = waiting (TTFB).
2. **Performance tab:** Record page load. Xem "Parse HTML", "Evaluate Script", "Recalculate Style", "Layout".
3. **Lighthouse:** "Eliminate render-blocking resources", "Preload key requests", "Avoid enormous network payloads".
4. **Coverage tab:** Xem bao nhiêu % JS/CSS thực sự được execute.

**Signals quan trọng trong Performance tab:**

- **Long tasks** (>50ms): Main thread bị block, user thấy janky
- **Layout Shift:** CLS metric — elements nhảy vị trí sau load
- **First Contentful Paint (FCP):** Khi có content đầu tiên trên screen
- **Largest Contentful Paint (LCP):** Khi element lớn nhất (hero image/heading) render xong

### 4.3 — Bug phổ biến và cách nhận ra

**Bug 1: `unload` listener phá bfcache**

```
Nhận ra: Chrome DevTools → Application → Back/forward cache → "Not restored" → lý do
Fix: Đổi sang pagehide
```

**Bug 2: Preload không có `as` attribute**

```
Nhận ra: Console warning "A preload for ... was found, but is not used"
         Resource bị load 2 lần (một lần preload, một lần actual use)
Fix: Luôn thêm as="image|script|style|font"
```

**Bug 3: CSS blocking render vì đặt sai vị trí**

```
Nhận ra: Performance tab → "Recalculate Style" sau "Parse HTML" rất dài
         Waterfall: nhiều render-blocking resources
Fix: Critical CSS inline, non-critical async load
```

**Bug 4: Script không defer/async trong `<head>>`**

```
Nhận ra: Performance tab → "Evaluate Script" sớm, HTML parse bị interrupt
         Network waterfall: HTML download dừng, script download, HTML tiếp tục
Fix: defer cho app scripts, async cho analytics
```

## PHẦN 5 — CÂU HỎI KIỂM TRA

**Câu 1 (Tại sao): Tại sao HTTPS site vẫn có thể bị chậm hơn HTTP ngay cả khi server response time là như nhau?**

Trả lời: TLS handshake thêm 1-2 RTT vào connection setup. Với TLS 1.2: 2 RTT thêm sau TCP. Với TLS 1.3: 1 RTT. Nếu user ở xa server (latency cao), 1 RTT có thể là 200ms+. Giải pháp: TLS 1.3, session resumption (0-RTT), `preconnect` để làm handshake sớm, HTTP/2 để reuse connection.

**Câu 2 (Cơ chế): Điều gì xảy ra khi browser gặp `<script src="app.js">` (không có async/defer) ở giữa HTML?**

Trả lời: (1) HTML parser dừng lại ngay tại vị trí đó. (2) Preload scanner tiếp tục scan phần còn lại của HTML để discover resources. (3) Browser download `app.js`. (4) Nếu có `<link rel="stylesheet">` nào trước đó chưa parse xong → script execution bị block thêm cho đến khi CSSOM sẵn sàng. (5) Execute script. (6) Parser tiếp tục từ vị trí bị dừng.

**Câu 3 (Điều gì xảy ra): Page của bạn có `window.addEventListener('unload', cleanup)`. User navigate đi rồi nhấn Back. Điều gì xảy ra khác với page không có listener này?**

Trả lời: Page có `unload` listener **không thể vào bfcache**. Khi user nhấn Back, browser phải thực hiện full navigation — DNS (có thể cached), TCP, TLS, HTTP request, HTML parse, script execute, render. Toàn bộ pipeline lại từ đầu. Với page không có unload listener, browser restore page từ memory trong vài milliseconds, JS state (variables, event listeners, form values) hoàn toàn nguyên vẹn. Fix: dùng `pagehide` event với `event.persisted` check.

**Câu 4 (Tại sao): Tại sao đặt tất cả CSS vào 1 file lớn lại có thể tệ hơn nhiều file nhỏ, dù nhiều file nhỏ tạo nhiều HTTP requests hơn?**

Trả lời: Câu hỏi này dựa trên giả định HTTP/1.1 (nhiều request = tệ). Với HTTP/2 (multiplexing), nhiều request nhỏ không đắt hơn 1 request lớn đáng kể. Nhưng quan trọng hơn: **CSS là render-blocking**. 1 file CSS lớn = browser phải download toàn bộ trước khi render. Nhiều file nhỏ với `media` queries đúng = browser chỉ render-block file CSS apply cho current viewport. Ví dụ: `print.css` với `media="print"` không block rendering trên màn hình. Split CSS theo route + lazy load = chỉ load CSS cần thiết.

**Câu 5 (Code có vấn đề gì): Đoạn code sau có vấn đề gì?**

```javascript
const images = document.querySelectorAll('.product-image');
images.forEach((img) => {
  const rect = img.getBoundingClientRect();
  img.style.width = rect.width * 1.2 + 'px';
  img.style.height = rect.height * 1.2 + 'px';
});
```

Trả lời: Đây là **layout thrashing**. Trong mỗi iteration: `getBoundingClientRect()` đọc layout geometry → browser phải force synchronous reflow để trả về giá trị chính xác (vì style đã thay đổi ở iteration trước). Sau đó write `style.width/height` → invalidate layout. Với N images → N forced reflows thay vì 1. Fix: đọc tất cả rects trước (1 reflow), rồi write tất cả styles (sẽ reflow 1 lần ở frame tiếp theo).

## CÂU HỎI ÔN TẬP — (Dành cho NotebookLM)

### Câu 1

**Hỏi:** Tại sao browser phải parse và build CSSOM hoàn toàn xong mới có thể render, trong khi DOM được build incremental (streaming)?

**Đáp án chi tiết:** CSS có tính chất cascade — quy tắc viết sau override quy tắc viết trước. Browser không thể áp dụng style cho một element chỉ với một phần CSS, vì phần còn lại có thể chứa selector cao hơn specificity hoặc rule override. Nếu browser render ngay với CSSOM chưa đầy đủ, user sẽ thấy "Flash of Unstyled Content" — page hiển thị sai rồi sau đó nhảy sang style đúng, gây trải nghiệm tệ. Ví dụ: CSS file có `.hero { color: black }` ở dòng 1 và `.hero { color: red }` ở dòng 1000. Nếu browser render sau dòng 1, hero sẽ màu đen, sau đó nhảy sang đỏ. Browser phải đọc toàn bộ stylesheet để biết final computed value. Đây là lý do tại sao CSS được coi là "render-blocking" — browser block rendering (không phải HTML parsing) cho đến khi CSSOM hoàn chỉnh.

### Câu 2

**Hỏi:** Giải thích sự khác biệt cơ chế giữa `async` và `defer` script. Trường hợp nào dùng `async` sẽ gây bug?

**Đáp án chi tiết:** `async`: Download song song với HTML parsing. **Execute ngay khi download xong, bất kể DOM đã parse xong hay chưa, bất kể script khác đã execute hay chưa.** `defer`: Download song song với HTML parsing. **Execute sau khi HTML hoàn toàn parse xong, trước DOMContentLoaded, theo đúng thứ tự xuất hiện trong HTML.**

Trường hợp `async` gây bug: (1) Script cần DOM — nếu script execute trước DOM parse xong, `document.querySelector('.hero')` trả về null. (2) Script phụ thuộc script khác — `async` không đảm bảo thứ tự, nên nếu `app.js` cần `jquery.js` nhưng cả hai đều async, app.js có thể execute trước jquery.js. (3) Script cần CSSOM — nếu script đọc computed styles nhưng CSS chưa parse xong. `defer` an toàn hơn cho hầu hết app scripts vì đảm bảo DOM ready và thứ tự execution. `async` chỉ phù hợp cho scripts hoàn toàn độc lập như analytics pixel.

### Câu 3

**Hỏi:** `preconnect`, `preload`, và `prefetch` khác nhau thế nào? Viết HTML cho scenario: font từ Google Fonts cần được load nhanh nhất có thể, và page tiếp theo trong checkout flow nên được load sẵn.

**Đáp án chi tiết:** Ba resource hints hoạt động theo cơ chế khác nhau:

- **`preconnect`**: Thực hiện DNS lookup + TCP handshake + TLS handshake cho một domain. **Không download resource.** Dùng khi biết sẽ cần resource từ domain đó nhưng chưa biết chính xác file nào.
- **`preload`**: Download **ngay lập tức với high priority** một resource cụ thể. Resource được cache, sẵn sàng khi cần. Phải có `as` attribute. Dùng cho resources cần trong current page nhưng browser discover muộn (fonts trong CSS, LCP images).
- **`prefetch`**: Download với **lowest priority trong idle time** để chuẩn bị cho future navigation. Dùng cho resources của page tiếp theo mà user có thể navigate tới.

HTML cho scenario:

```html
<!-- preconnect với fonts.gstatic.com (nơi chứa font files) -->
<!-- crossorigin BẮT BUỘC vì font request là CORS request -->
<link
  rel="preconnect"
  href="https://fonts.googleapis.com"
/>
<link
  rel="preconnect"
  href="https://fonts.gstatic.com"
  crossorigin
/>

<!-- Load font CSS (connection đã sẵn sàng nhờ preconnect) -->
<link
  href="https://fonts.googleapis.com/css2?family=Inter:wght@400;700"
  rel="stylesheet"
/>

<!-- Preload font file cụ thể nếu biết trước URL -->
<!-- as="font" + crossorigin bắt buộc cho font preload -->
<link
  rel="preload"
  href="https://fonts.gstatic.com/s/inter/v13/font.woff2"
  as="font"
  type="font/woff2"
  crossorigin
/>

<!-- Prefetch page tiếp theo trong checkout -->
<!-- Low priority, download trong idle time -->
<link
  rel="prefetch"
  href="/checkout/payment"
/>
```

Lý do `crossorigin` với fonts: Font requests luôn là CORS requests (cross-origin). Nếu `preconnect` hoặc `preload` không có `crossorigin`, browser tạo anonymous connection. Sau đó khi font request thực sự xảy ra (là CORS credentialed), browser phải tạo connection mới — preconnect/preload bị lãng phí. `crossorigin` đảm bảo connection type match.
