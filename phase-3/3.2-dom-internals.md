### A — DOM không phải JavaScript

Đây là điều hầu hết developer không để ý: **DOM không phải là một phần của JavaScript**. DOM là một **C++ object graph** được expose sang JavaScript thông qua Web IDL bindings.

```
JavaScript Engine (V8)          Browser Engine (Blink/Gecko)
┌─────────────────────┐         ┌──────────────────────────┐
│                     │  Binding│                          │
│  document.body      │◄───────►│  HTMLBodyElement (C++)   │
│  el.style           │         │  CSSStyleDeclaration(C++)│
│  el.appendChild()   │         │  Node::AppendChild() C++ │
│                     │         │                          │
└─────────────────────┘         └──────────────────────────┘
         ▲                                   ▲
    JS heap                          Renderer heap
    (V8 GC)                         (Blink GC — Oilpan)
```

**Mỗi lần bạn access một DOM property từ JavaScript, bạn đang vượt qua boundary giữa hai hệ thống.** Đây là lý do DOM access có cost — không phải vì DOM "chậm", mà vì mỗi lần access là một cross-boundary call với overhead.

**Hai GC riêng biệt:** V8 quản lý JS objects. Blink có GC riêng (Oilpan) quản lý DOM objects. Khi một element bị remove khỏi DOM nhưng JS vẫn giữ reference → element vẫn sống trong Blink heap, không được collect. Đây là nguồn gốc của "detached DOM node" memory leak.

### B — Node Types: Không phải mọi thứ trong DOM đều là Element

DOM tree gồm nhiều loại Node khác nhau, mỗi loại có `nodeType` constant:

```
Node (abstract base)
├── Document              nodeType = 9   → document
├── DocumentFragment      nodeType = 11  → document.createDocumentFragment()
├── Element               nodeType = 1   → <div>, <span>, v.v.
│   ├── HTMLElement
│   │   ├── HTMLDivElement
│   │   ├── HTMLInputElement
│   │   └── ... (mỗi tag có class riêng)
│   └── SVGElement
├── Attr                  nodeType = 2   → attributes (ít dùng trực tiếp)
├── Text                  nodeType = 3   → text nodes
├── Comment               nodeType = 8   → <!-- comment -->
└── ProcessingInstruction nodeType = 7   → <?xml ...?>
```

**Text nodes là invisible nhưng thật:**

```html
<div>
  Hello
  <span>world</span>
</div>
```

```
DOM tree thực sự:
div
├── Text: "\n  Hello\n  "    ← whitespace text node — thật sự tồn tại!
├── span
│   └── Text: "world"
└── Text: "\n"               ← trailing whitespace
```

**Tại sao quan trọng:**

```javascript
const div = document.querySelector('div');

div.childNodes; // NodeList(3): [Text, span, Text]  ← bao gồm text nodes
div.children; // HTMLCollection(1): [span]         ← chỉ Element nodes

div.firstChild; // Text node (whitespace!)
div.firstElementChild; // span element

// Bug phổ biến:
div.childNodes[0].nodeType === 3; // true — là Text, không phải Element
div.childNodes[0].tagName; // undefined — Text nodes không có tagName
```

### C — DOM Tree: Cơ chế thật của C++ data structure

Bên trong Blink, mỗi Node là một C++ object với các pointers:

```
Node (C++ object)
┌─────────────────────────────────┐
│ m_parentOrShadowHostNode*       │ → pointer lên parent
│ m_previous*                     │ → pointer sang sibling trước
│ m_next*                         │ → pointer sang sibling sau
│ m_firstChild*                   │ → pointer xuống first child
│ m_nodeFlags                     │ → bitfield: isConnected, isInShadowTree...
│ m_treeScope*                    │ → Document hoặc ShadowRoot
└─────────────────────────────────┘
```

**Đây là doubly-linked tree, không phải array.** Traversal theo chiều bất kỳ đều O(1) nhưng accessing by index là O(n).

```javascript
// O(1) — direct pointer access
el.parentNode;
el.firstChild;
el.nextSibling;
el.previousSibling;

// O(n) — phải traverse từ đầu
el.children[5]; // traverse 5 siblings
el.childNodes[10]; // traverse 10 nodes

// querySelector — không phải O(1)
// Phức tạp hơn nhiều, xem phần D
```

**isConnected flag:** Mỗi Node có flag `isConnected` — true khi node là một phần của document tree (có ancestor là Document). Khi node bị remove → flag set false. Đây là cách browser biết nhanh node có trong DOM hay không mà không cần traverse toàn bộ tree.

```javascript
const el = document.createElement('div'); // isConnected = false
document.body.appendChild(el); // isConnected = true
document.body.removeChild(el); // isConnected = false

// Kiểm tra:
el.isConnected; // false — node đã bị detach
```

### D — querySelector: Cơ chế thật

`querySelector` không phải "magic" — nó là một **selector matching engine** chạy trên DOM tree.

#### CSS Selector Matching: Right-to-Left

Một điều counter-intuitive: CSS selector được match **từ phải sang trái**:

```css
/* Selector: div.container > ul li.active */

/* Browser đọc từ phải sang trái:
   1. Tìm tất cả elements có class "active" (key selector)
   2. Với mỗi .active element, check: có phải là <li> không?
   3. Nếu có: tìm ancestor <ul>?
   4. Nếu có: tìm ancestor trực tiếp là div.container?
*/
```

**Tại sao right-to-left?** Vì bắt đầu từ "key selector" (rightmost) thường loại bỏ được nhiều candidates nhất. Bắt đầu từ trái (document root) sẽ phải traverse toàn bộ tree.

#### querySelector vs querySelectorAll: Live vs Static

```javascript
// querySelectorAll: trả về StaticNodeList — snapshot tại thời điểm gọi
const staticList = document.querySelectorAll('.item');
// Thêm element mới vào DOM
document.body.innerHTML += '<div class="item">new</div>';
staticList.length; // vẫn là số cũ — không update

// getElementsByClassName: trả về HTMLCollection — LIVE collection
const liveList = document.getElementsByClassName('item');
// Thêm element mới vào DOM
document.body.innerHTML += '<div class="item">new</div>';
liveList.length; // đã update — phản ánh DOM hiện tại
```

**Live collections có cost:** Mỗi lần access `.length` hoặc index trên live collection, browser có thể phải re-traverse DOM để đảm bảo collection up-to-date. Trong loop, đây là performance trap:

```javascript
// ❌ BUG + PERFORMANCE ISSUE: live collection trong loop
const items = document.getElementsByClassName('item'); // live!
for (let i = 0; i < items.length; i++) {
  // items.length re-evaluated mỗi iteration
  items[i].remove(); // modify DOM → live collection thay đổi → i++ bỏ qua element
}
// Kết quả: chỉ xóa một nửa elements!

// ✅ FIX: convert sang static array trước
const items = Array.from(document.getElementsByClassName('item'));
// hoặc dùng querySelectorAll (trả về StaticNodeList)
const items = document.querySelectorAll('.item');
items.forEach((el) => el.remove());
```

### E — innerHTML vs textContent vs innerText

Ba property này nhìn có vẻ tương tự nhưng cơ chế hoàn toàn khác nhau.

#### innerHTML

```javascript
el.innerHTML = '<strong>Hello</strong> world';
```

**Cơ chế khi SET:**

1. Parse string như HTML document fragment
2. Build DOM nodes từ parsed HTML
3. Remove tất cả existing children của `el`
4. Append tất cả nodes mới vào `el`

**Cơ chế khi GET:**

1. Serialize DOM subtree thành HTML string
2. Traverse tất cả child nodes, tạo string representation

**Cost:** Parse HTML + rebuild DOM. Với large subtrees, tốn kém.

**XSS Vulnerability:**

```javascript
// ❌ NGUY HIỂM: XSS
const userInput = '<img src=x onerror="steal(document.cookie)">';
el.innerHTML = userInput;
// Browser execute onerror handler → XSS

// ❌ Script tags trực tiếp sẽ không execute (spec quy định)
el.innerHTML = '<script>alert("xss")</script>';
// <script> inserted vào DOM nhưng KHÔNG execute

// ❌ Nhưng event handlers thì execute!
el.innerHTML = '<img src=x onerror=alert(1)>'; // execute
el.innerHTML = '<svg onload=alert(1)>'; // execute
el.innerHTML = '<div onclick=alert(1)>click me'; // execute khi click

// ✅ SAFE: textContent
el.textContent = userInput; // render như plain text, không parse HTML

// ✅ SAFE: DOMPurify cho trường hợp cần render HTML từ user input
el.innerHTML = DOMPurify.sanitize(userInput);
```

#### textContent

```javascript
el.textContent = 'Hello <strong>world</strong>';
// Render as: Hello <strong>world</strong>  ← literal text, không parse

el.textContent; // GET: concatenate text content của tất cả text node descendants
```

**Cơ chế khi SET:** Tạo một Text node duy nhất với string value. Remove tất cả children cũ. Nhanh hơn `innerHTML` vì không parse HTML.

**Cơ chế khi GET:** Traverse toàn bộ subtree, concatenate tất cả Text node content. Bao gồm cả hidden elements (`display: none`).

```javascript
// <div id="el">Hello <span style="display:none">hidden</span> world</div>
el.textContent; // "Hello hidden world" — bao gồm cả hidden text
```

#### innerText

```javascript
el.innerText; // GET: text như user "thấy" — layout-aware
```

**Cơ chế khi GET:**

1. Trigger layout (reflow!) để biết element nào visible
2. Return text của các visible elements theo visual order
3. Tính đến CSS: `display:none` bị loại trừ, `text-transform` được apply, `white-space: pre` được tôn trọng

```javascript
// <div id="el">Hello <span style="display:none">hidden</span> world</div>
el.innerText; // "Hello  world" — bỏ qua hidden
el.textContent; // "Hello hidden world" — bao gồm hidden

// CSS text-transform:
// <p style="text-transform: uppercase">hello</p>
p.innerText; // "HELLO"   — apply CSS
p.textContent; // "hello"   — raw text, không CSS
```

**Performance:**

```
textContent GET: Fast — chỉ traverse text nodes
innerHTML GET:  Medium — serialize DOM to HTML string
innerText GET:  SLOW — trigger layout (reflow) trước khi return
```

**Quy tắc chọn:**

```
Muốn set text an toàn (không parse HTML)?  → textContent
Muốn set HTML?                             → innerHTML (+ sanitize nếu user input)
Muốn get text của visible content?         → innerText (chấp nhận reflow cost)
Muốn get tất cả text kể cả hidden?         → textContent
Muốn get/set HTML?                         → innerHTML
```

### F — DOM Manipulation: Cơ chế thật

#### appendChild, insertBefore, removeChild

Mỗi DOM mutation operation làm những việc sau trong Blink:

```
appendChild(newChild):
1. Detach newChild từ parent cũ (nếu có)
2. Attach newChild vào parent mới
3. Update parent/child/sibling pointers
4. Dispatch "DOMNodeInserted" mutation event (nếu có listener)
5. Set "needs style recalculation" flag trên affected nodes
6. Invalidate layout (nếu node sẽ visible)
7. Schedule render update
```

**Điều quan trọng:** Bước 5-7 không xảy ra ngay. Browser **batch** style recalculation và layout. Nếu bạn append 100 elements trong 1 JS execution context, browser không reflow 100 lần — nó reflow 1 lần sau khi JS execution xong.

```javascript
// Browser không reflow sau mỗi append:
for (let i = 0; i < 100; i++) {
  const div = document.createElement('div');
  div.textContent = `Item ${i}`;
  container.appendChild(div); // Mark dirty, không reflow ngay
}
// Sau khi loop xong → JS callstack empty → browser reflow 1 lần
```

**Nhưng nếu bạn READ layout sau khi WRITE:**

```javascript
for (let i = 0; i < 100; i++) {
  const div = document.createElement('div');
  container.appendChild(div); // WRITE → mark dirty
  console.log(div.offsetHeight); // READ → force synchronous reflow ngay!
  // → 100 reflows
}
```

#### Reparenting — node không bị clone

```javascript
const el = document.getElementById('source');
const target = document.getElementById('target');

// Node bị MOVE, không copy
target.appendChild(el);
// el đã không còn trong source, không cần remove trước
// appendChild tự handle detach

// Nếu muốn copy:
target.appendChild(el.cloneNode(true)); // deep clone (bao gồm children)
target.appendChild(el.cloneNode(false)); // shallow clone (chỉ element, không children)
// Note: cloneNode KHÔNG clone event listeners
```

### G — DocumentFragment: Tại sao nó nhanh hơn

#### Vấn đề với direct DOM manipulation trong loop

```javascript
// ❌ Append trực tiếp: mỗi append trigger potential reflow
// Và quan trọng hơn: mỗi append khi DOM connected
// → dispatchEvent, update live collections, v.v.
const list = document.querySelector('ul');
items.forEach((item) => {
  const li = document.createElement('li');
  li.textContent = item.name;
  list.appendChild(li); // append vào connected DOM
});
```

#### DocumentFragment: Một "ghost" DOM container

```javascript
const fragment = document.createDocumentFragment();
```

DocumentFragment là một Node đặc biệt:

- **Không có parent** — không được attach vào document
- **Không render** — invisible, không trigger layout/paint khi mutate
- **Khi append vào DOM:** nó "disappears" — chỉ children của nó được inserted, không phải fragment itself
- **isConnected = false** cho đến khi children được moved vào real DOM

```javascript
// ✅ DocumentFragment approach
const fragment = document.createDocumentFragment();

items.forEach((item) => {
  const li = document.createElement('li');
  li.textContent = item.name;
  fragment.appendChild(li);
  // append vào disconnected fragment:
  // - Không trigger live collection updates
  // - Không trigger layout
  // - Không dispatch DOM mutation events lên document
});

// 1 lần append duy nhất vào real DOM:
list.appendChild(fragment);
// fragment.childNodes được moved vào list
// fragment itself không được inserted (trở thành empty)
// → 1 layout invalidation thay vì N
```

**Tại sao fragment nhanh hơn?**

Khi node có `isConnected = false`:

- Style recalculation không xảy ra sau mỗi mutation
- Live HTMLCollection không được notified
- `MutationObserver` vẫn observe (nếu observe fragment), nhưng không trigger document-level observers

**Thực tế với modern browsers:** Browser hiện đại đã optimize batch DOM operations khá tốt. Difference giữa fragment và direct append không lớn như xưa — nhưng pattern vẫn quan trọng khi:

1. Cần đảm bảo không trigger intermediate reflows (khi có JS đọc layout giữa chừng)
2. Cần atomic insert (tất cả hoặc không có gì)
3. Cần avoid triggering intermediate MutationObserver callbacks

### H — DOM Diffing: Nền tảng của Virtual DOM

#### Tại sao DOM diffing ra đời

Trước React/Virtual DOM, pattern phổ biến là:

```javascript
// Khi data thay đổi → wipe và re-render toàn bộ
container.innerHTML = renderTemplate(newData);
// Vấn đề:
// 1. Mất focus state (input đang focus bị destroy)
// 2. Mất scroll position của elements bên trong
// 3. Mất CSS transition state
// 4. Toàn bộ subtree bị destroy và recreate → GC pressure
// 5. Performance: nhiều DOM mutations không cần thiết
```

#### Concept của DOM Diffing

Thay vì wipe và re-render, so sánh "DOM cũ" với "DOM mới muốn có" và **chỉ apply sự khác biệt**:

```
Old Tree:         New Tree:         Diff (operations):
<ul>              <ul>
  <li>A</li>        <li>A</li>      → No change
  <li>B</li>        <li>B modified  → Update text "B" → "B modified"
  <li>C</li>        </li>
</ul>               <li>D</li>      → Insert new <li>D</li>
                  </ul>
                                    → Remove old <li>C</li>
```

**Virtual DOM là gì thật sự:**

Virtual DOM là một JavaScript object representation của DOM tree — "plain JS objects" thay vì actual DOM nodes:

```javascript
// DOM Node (C++ object, đắt để tạo):
// document.createElement('div') → cross-boundary call, allocate C++ object

// Virtual DOM Node (plain JS object, rẻ):
const vnode = {
  type: 'div',
  props: { className: 'container', id: 'main' },
  children: [
    { type: 'span', props: {}, children: ['Hello'] },
    { type: 'p', props: { className: 'text' }, children: ['World'] },
  ],
};
```

**Pipeline của React (simplified):**

```
State change
     ↓
Re-render: tạo new Virtual DOM tree (JS objects — nhanh)
     ↓
Reconciler: diff old vDOM vs new vDOM (JS comparison — nhanh)
     ↓
Commit phase: apply minimal DOM mutations (đắt — chỉ làm những gì thật sự thay đổi)
```

#### Tại sao diffing không trivial — Tree Edit Distance

Diff hai trees tổng quát (Tree Edit Distance problem) có complexity O(n³). React và Vue sử dụng **heuristics** để đạt O(n):

**Heuristic 1: Same level comparison**
Chỉ compare nodes cùng level, không compare cross-level:

```
Old:          New:
<div>          <div>
  <p>            <span>    ← Khác type → destroy p, create span
  </p>           </span>      (không cố tìm p ở level khác)
</div>         </div>
```

**Heuristic 2: Same type → update, khác type → destroy + recreate**

```javascript
// Old: <div className="old">
// New: <div className="new">
// → UPDATE className (giữ nguyên DOM node, chỉ update attribute)

// Old: <div>
// New: <span>
// → DESTROY div và tất cả children, CREATE span mới
```

**Heuristic 3: Keys cho list items**

```jsx
// ❌ Không có key — React không biết item nào match item nào
// Old: [A, B, C]
// New: [A, B, C, D]
// React compare: A↔A (same), B↔B (same), C↔C (same), null↔D (create)
// OK trong trường hợp này.

// Old: [A, B, C]
// New: [X, A, B, C]  ← prepend X
// React compare: A↔X (different → update!), B↔A (update!), C↔B (update!), null↔C (create!)
// → 3 updates + 1 create dù chỉ cần 1 create

// ✅ Với key — React biết chính xác node nào là node nào
// Old: [key=A: A, key=B: B, key=C: C]
// New: [key=X: X, key=A: A, key=B: B, key=C: C]
// React diff: X là mới (create), A/B/C không thay đổi (no-op)
// → 1 create duy nhất
```

**Tại sao `key={index}` là anti-pattern:**

```jsx
// ❌ key=index — key thay đổi khi thứ tự thay đổi
items.map((item, index) => <li key={index}>{item.name}</li>);

// Old: [key=0: Apple, key=1: Banana, key=2: Cherry]
// New (xóa Apple): [key=0: Banana, key=1: Cherry]
// React thấy: key=0 thay đổi từ Apple→Banana (update!)
//              key=1 thay đổi từ Banana→Cherry (update!)
//              key=2 bị xóa (remove)
// → 2 updates + 1 remove — nhưng lẽ ra chỉ cần 1 remove

// ✅ key=unique ID — key stable dù thứ tự thay đổi
items.map((item) => <li key={item.id}>{item.name}</li>);
// New (xóa Apple): React thấy item.id của Apple biến mất → remove đúng node
// → 1 remove duy nhất
```

### I — Observer APIs: Cơ chế thật

#### MutationObserver 🟡

**Tại sao MutationObserver ra đời:** Trước đây có `MutationEvents` (DOMNodeInserted, DOMAttrModified, v.v.). Events này fire **synchronously** trong DOM mutation — nghĩa là mỗi mutation gây re-entrant code execution, có thể gây thêm mutations, cascade không kiểm soát được. Browser vendors deprecated và replace bằng MutationObserver.

**Cơ chế của MutationObserver:**

```
DOM mutation xảy ra
     ↓
Browser ghi lại MutationRecord (không dispatch event)
     ↓
Tiếp tục thực thi JS hiện tại (không interrupt)
     ↓
Cuối microtask checkpoint:
Flush tất cả pending MutationRecords
Gọi observer callback với batch records
```

MutationObserver callback chạy như **microtask** — sau JS execution hiện tại, trước next macrotask.

```javascript
const observer = new MutationObserver((mutations) => {
  // mutations: MutationRecord[] — batch của tất cả thay đổi từ lần callback trước
  mutations.forEach((mutation) => {
    console.log(mutation.type); // 'childList' | 'attributes' | 'characterData'
    console.log(mutation.target); // Node bị mutate
    console.log(mutation.addedNodes); // NodeList nodes được thêm
    console.log(mutation.removedNodes); // NodeList nodes bị xóa
    console.log(mutation.oldValue); // Giá trị cũ (nếu attributeOldValue/characterDataOldValue)
  });
});

// Configuration options:
observer.observe(targetNode, {
  childList: true, // Observe direct children add/remove
  subtree: true, // Observe toàn bộ subtree (không chỉ direct children)
  attributes: true, // Observe attribute changes
  attributeOldValue: true, // Lưu giá trị attribute cũ trong record
  characterData: true, // Observe text content changes
  characterDataOldValue: true, // Lưu text content cũ
  attributeFilter: ['class', 'style'], // Chỉ observe specific attributes
});

// QUAN TRỌNG: Luôn disconnect khi không cần nữa
observer.disconnect();
// Nếu không disconnect: observer giữ reference đến targetNode
// → targetNode không thể được GC dù bị remove khỏi DOM
```

**Use case thực tế — Detect third-party DOM changes:**

```javascript
// Framework muốn biết khi nào third-party script modify DOM
// (ví dụ: chat widget inject elements, A/B testing tools thay đổi content)
const observer = new MutationObserver((mutations) => {
  for (const mutation of mutations) {
    for (const node of mutation.addedNodes) {
      if (node.nodeType === Node.ELEMENT_NODE) {
        // Xử lý element mới được inject
        processInjectedElement(node);
      }
    }
  }
});

observer.observe(document.body, { childList: true, subtree: true });
```

#### ResizeObserver 🟡

**Vấn đề trước ResizeObserver:** Không có API native để detect element size change. Developer dùng:

- `window.resize` event: Chỉ fire khi viewport resize, không detect element resize (ví dụ: sidebar collapse, panel resize)
- Polling với `setInterval`: Tốn performance, không chính xác timing
- Iframe tricks: Hack kinh điển — tạo iframe trong element, listen iframe's window resize

**Cơ chế của ResizeObserver:**

```
Browser hoàn thành Layout step
     ↓
ResizeObserver: check tất cả observed elements
     ↓
Với mỗi element có size thay đổi so với lần trước:
  Tạo ResizeObserverEntry
     ↓
```

#### IntersectionObserver 🟡

**Trước IntersectionObserver:** Để detect element có trong viewport không, phải:

```javascript
// Polling mỗi 100ms — tốn kém
setInterval(() => {
  const rect = el.getBoundingClientRect(); // FORCE REFLOW mỗi 100ms!
  const isVisible = rect.top < window.innerHeight && rect.bottom > 0;
  if (isVisible) loadImage(el);
}, 100);
```

**Cơ chế của IntersectionObserver:**

IntersectionObserver không chạy trên main thread theo polling. Nó được implement **off main thread** trong browser:

```
Browser compositing layer (GPU thread hoặc compositor thread)
     ↓
Biết vị trí elements và scroll position
     ↓
Tính intersection ratio
     ↓
Khi ratio vượt ngưỡng (threshold):
Enqueue callback vào main thread task queue
     ↓
Main thread: execute callback
```

**Zero layout thrashing:** IntersectionObserver không gọi `getBoundingClientRect()` — nó dùng compositor information đã có sẵn. Không trigger reflow.

```javascript
const observer = new IntersectionObserver(
  (entries, observer) => {
    entries.forEach((entry) => {
      // entry.isIntersecting: boolean — có đang intersect không
      // entry.intersectionRatio: 0.0 → 1.0 — bao nhiêu % element visible
      // entry.boundingClientRect: element's bounding rect
      // entry.intersectionRect: phần visible của element
      // entry.rootBounds: viewport (hoặc root element) bounds
      // entry.time: timestamp khi intersection thay đổi

      if (entry.isIntersecting) {
        // Lazy load image
        const img = entry.target;
        img.src = img.dataset.src; // load actual image
        observer.unobserve(img); // stop observing sau khi load
      }
    });
  },
  {
    root: null, // null = viewport
    rootMargin: '200px', // Expand viewport bounds 200px (load trước khi visible)
    threshold: [0, 0.25, 0.5, 0.75, 1.0], // Callback fire ở các mốc này
  },
);

document.querySelectorAll('img[data-src]').forEach((img) => {
  observer.observe(img);
});
```

### Lazy Loading với IntersectionObserver — Production Pattern

```javascript
class LazyLoader {
  constructor(options = {}) {
    this.options = {
      rootMargin: options.rootMargin || '200px 0px',
      threshold: options.threshold || 0,
      // Fallback cho browsers không support IntersectionObserver
      fallback: options.fallback || 'eager',
    };

    // Check support
    if (!('IntersectionObserver' in window)) {
      this._loadAllImmediately();
      return;
    }

    this.observer = new IntersectionObserver(
      this._handleIntersection.bind(this),
      {
        rootMargin: this.options.rootMargin,
        threshold: this.options.threshold,
      },
    );
  }

  observe(element) {
    if (!this.observer) {
      // Fallback: load immediately
      this._loadElement(element);
      return;
    }
    this.observer.observe(element);
  }

  _handleIntersection(entries) {
    entries.forEach((entry) => {
      if (!entry.isIntersecting) return;

      this._loadElement(entry.target);
      this.observer.unobserve(entry.target);
    });
  }

  _loadElement(element) {
    const tagName = element.tagName.toLowerCase();

    if (tagName === 'img') {
      // Image lazy loading
      if (element.dataset.srcset) {
        element.srcset = element.dataset.srcset;
      }
      element.src = element.dataset.src;
      element.removeAttribute('data-src');
      element.removeAttribute('data-srcset');
    } else if (tagName === 'video') {
      // Video lazy loading
      element.querySelectorAll('source').forEach((source) => {
        source.src = source.dataset.src;
      });
      element.load();
    } else if (element.dataset.component) {
      // Lazy load React/Vue component
      import(`/components/${element.dataset.component}.js`).then((module) =>
        module.default.mount(element),
      );
    }

    element.classList.add('loaded');
  }

  disconnect() {
    if (this.observer) {
      this.observer.disconnect();
    }
  }
}

// Sử dụng:
const lazyLoader = new LazyLoader({ rootMargin: '300px 0px' });

document.querySelectorAll('[data-src]').forEach((el) => {
  lazyLoader.observe(el);
});
```

```javascript
function safeSetHTML(element, html) {
  // Option 1: Browser native setHTML (Chrome 93+, còn experimental)
  if (element.setHTML) {
    element.setHTML(html, {
      sanitizer: new Sanitizer({
        allowElements: ['b', 'i', 'em', 'strong', 'a', 'p', 'br'],
        allowAttributes: { a: ['href'] },
      }),
    });
    return;
  }

  // Option 2: Parse với DOMParser trong safe context
  // DOMParser không execute scripts trong parsed document
  const parser = new DOMParser();
  const parsed = parser.parseFromString(html, 'text/html');

  // Remove dangerous elements/attributes
  const dangerous = parsed.querySelectorAll(
    'script, iframe, object, embed, form, [onerror], [onload], [onclick]',
  );
  dangerous.forEach((el) => el.remove());

  // Remove all event handler attributes
  parsed.querySelectorAll('*').forEach((el) => {
    Array.from(el.attributes).forEach((attr) => {
      if (attr.name.startsWith('on')) {
        el.removeAttribute(attr.name);
      }
      // javascript: URLs
      if (
        (attr.name === 'href' || attr.name === 'src') &&
        attr.value.trim().toLowerCase().startsWith('javascript:')
      ) {
        el.removeAttribute(attr.name);
      }
    });
  });

  // Clear và rebuild với DocumentFragment
  const fragment = document.createDocumentFragment();
  Array.from(parsed.body.childNodes).forEach((node) => {
    fragment.appendChild(node.cloneNode(true));
  });

  element.innerHTML = ''; // Clear
  element.appendChild(fragment); // Insert sanitized content
}
```

### MutationObserver — Auto-initialize Components

```javascript
// Pattern: Auto-initialize custom components khi được inserted vào DOM
// Dùng trong design systems, micro-frontend architectures

class ComponentRegistry {
  constructor() {
    this.components = new Map();
    this.initialized = new WeakSet(); // WeakSet không block GC

    this.observer = new MutationObserver((mutations) => {
      const addedNodes = [];

      mutations.forEach((mutation) => {
        mutation.addedNodes.forEach((node) => {
          if (node.nodeType === Node.ELEMENT_NODE) {
            addedNodes.push(node);
          }
        });
      });

      // Batch process added nodes
      if (addedNodes.length > 0) {
        requestAnimationFrame(() => this._initializeComponents(addedNodes));
        // requestAnimationFrame để batch nhiều mutations trong cùng frame
      }
    });

    this.observer.observe(document.body, {
      childList: true,
      subtree: true,
      // Không observe attributes hay characterData — không cần, tiết kiệm overhead
    });
  }

  register(selector, initFn) {
    this.components.set(selector, initFn);
    // Initialize existing elements ngay
    document.querySelectorAll(selector).forEach((el) => {
      this._initElement(el, initFn);
    });
  }

  _initializeComponents(addedNodes) {
    addedNodes.forEach((node) => {
      this.components.forEach((initFn, selector) => {
        // Check node itself
        if (node.matches(selector)) {
          this._initElement(node, initFn);
        }
        // Check descendants
        node.querySelectorAll(selector).forEach((el) => {
          this._initElement(el, initFn);
        });
      });
    });
  }

  _initElement(el, initFn) {
    if (!this.initialized.has(el)) {
      this.initialized.add(el);
      initFn(el);
    }
  }

  destroy() {
    this.observer.disconnect();
  }
}

// Sử dụng:
const registry = new ComponentRegistry();

registry.register('[data-component="datepicker"]', (el) => {
  new DatePicker(el, { format: el.dataset.format });
});

registry.register('[data-component="tooltip"]', (el) => {
  new Tooltip(el, { content: el.dataset.tooltip });
});

// Bất kỳ khi nào HTML mới được inject vào DOM
// (từ AJAX, React portals, innerHTML, v.v.)
// Components sẽ tự động initialize
```

### DocumentFragment — Virtual List (Infinite Scroll)

```javascript
class VirtualList {
  constructor(container, items, renderItem) {
    this.container = container;
    this.items = items;
    this.renderItem = renderItem;
    this.batchSize = 50;
  }

  // Render một batch items hiệu quả
  renderBatch(startIndex, endIndex) {
    // DocumentFragment để batch DOM operations
    const fragment = document.createDocumentFragment();

    for (let i = startIndex; i < Math.min(endIndex, this.items.length); i++) {
      const element = this.renderItem(this.items[i], i);
      fragment.appendChild(element);
      // Append vào disconnected fragment:
      // - Không trigger style recalculation
      // - Không trigger layout
      // - Không dispatch events lên document
    }

    // Một DOM operation duy nhất
    this.container.appendChild(fragment);
    // fragment bây giờ rỗng (children đã được moved)
  }

  // Infinite scroll với IntersectionObserver
  setupInfiniteScroll() {
    let currentIndex = 0;

    // Render batch đầu tiên
    this.renderBatch(currentIndex, this.batchSize);
    currentIndex += this.batchSize;

    // Sentinel element ở cuối list
    const sentinel = document.createElement('div');
    sentinel.className = 'sentinel';
    this.container.appendChild(sentinel);

    const observer = new IntersectionObserver(
      (entries) => {
        if (entries[0].isIntersecting && currentIndex < this.items.length) {
          // Load next batch
          // Remove sentinel tạm thời để reattach sau batch
          sentinel.remove();
          this.renderBatch(currentIndex, currentIndex + this.batchSize);
          currentIndex += this.batchSize;
          this.container.appendChild(sentinel);

          // Stop observe nếu đã hết items
          if (currentIndex >= this.items.length) {
            observer.disconnect();
            sentinel.remove();
          }
        }
      },
      {
        rootMargin: '200px', // Start loading 200px trước khi sentinel visible
      },
    );

    observer.observe(sentinel);
  }
}
```

## CÂU HỎI ÔN TẬP — (Dành cho NotebookLM)

### Câu 1

**Hỏi:** Tại sao DOM không phải là một phần của JavaScript, và điều này ảnh hưởng thế nào đến performance khi viết code thao tác DOM trong loop?

**Đáp án chi tiết:** DOM là một C++ object graph được implement trong browser engine (Blink trong Chrome, Gecko trong Firefox). JavaScript chạy trong V8 engine — một hệ thống hoàn toàn tách biệt. Hai hệ thống giao tiếp qua Web IDL bindings — mỗi lần JS access một DOM property hay method (ví dụ `el.offsetHeight`, `el.appendChild()`), đó là một cross-boundary call với overhead: type checking, context switching, và trong một số trường hợp là IPC (Inter-Process Communication). Hơn nữa, DOM objects được quản lý bởi Blink's GC (Oilpan), còn JS objects được quản lý bởi V8's GC — hai hệ thống GC phải coordinate với nhau khi có references cross boundary.

Hệ quả performance: Trong loop thực hiện nhiều DOM reads và writes xen kẽ, cost xảy ra từ hai phía: (1) Chi phí cross-boundary của mỗi DOM call, và (2) Quan trọng hơn — "layout thrashing" khi đọc layout properties (offsetWidth, getBoundingClientRect) sau khi đã write DOM. Browser phải invalidate cached layout và recompute synchronously để trả về giá trị đúng. Fix: cache DOM references ra ngoài loop, tách read phase và write phase, dùng DocumentFragment cho batch inserts.

### Câu 2

**Hỏi:** Giải thích sự khác biệt cơ chế giữa `textContent`, `innerHTML`, và `innerText`. Tại sao dùng `innerHTML` với user input lại nguy hiểm dù `<script>` tags không execute?

**Đáp án chi tiết:** `textContent` SET tạo một Text node duy nhất với giá trị là string — không parse HTML, không execute gì. GET traverse toàn bộ subtree và concatenate tất cả Text node content, bao gồm cả hidden elements. `innerHTML` SET parse string như HTML, build DOM nodes từ parsed result, remove existing children, append nodes mới. GET serialize DOM subtree thành HTML string. `innerText` GET trigger layout (reflow) trước khi return — browser phải biết element nào visible, CSS transform nào apply. Đây là lý do `innerText` rất chậm so với hai cái kia.

`innerHTML` nguy hiểm với user input dù `<script>` không execute vì: Spec quy định `<script>` inserted via innerHTML không execute (để tránh XSS quá dễ). Nhưng browser vẫn execute event handler attributes: `<img src=x onerror="steal(document.cookie)">` — khi browser cố load image từ "x" (fail), onerror handler execute. `<svg onload=maliciousCode()>` execute ngay khi SVG được inserted. `<a href="javascript:code()">` execute khi click. Attacker không cần `<script>` — bất kỳ event handler attribute nào trên bất kỳ HTML element nào đều có thể execute JavaScript. Fix: dùng `textContent` cho plain text, hoặc DOMPurify để sanitize HTML trước khi set vào `innerHTML`.

### Câu 3

**Hỏi:** MutationObserver, ResizeObserver, và IntersectionObserver có timing callback khác nhau như thế nào trong event loop? Tại sao điều này quan trọng khi dùng chúng để update UI?

**Đáp án chi tiết:** Ba Observer chạy ở những điểm khác nhau trong rendering pipeline, và điều này quyết định chúng có thể làm gì an toàn mà không trigger extra layout hay paint.

`MutationObserver` callback chạy như microtask — sau khi JS execution hiện tại kết thúc, trước next macrotask, và trước Layout. Điều này có nghĩa: nếu trong callback bạn đọc layout properties (offsetHeight, getBoundingClientRect), browser phải perform synchronous layout để trả về giá trị — có thể tốn kém. Nhưng nếu chỉ mutate DOM thêm trong callback, những mutations đó sẽ được batch với layout tiếp theo.

`ResizeObserver` callback chạy sau Layout, trước Paint — đây là timing lý tưởng. Browser đã tính toán xong kích thước (nên có thể đọc size từ entry mà không cần trigger reflow thêm), và chưa paint (nên có thể update DOM để ảnh hưởng visual). Tuy nhiên, nếu trong callback bạn thay đổi size của chính element đang observe, bạn sẽ trigger thêm Layout pass — có thể dẫn đến infinite loop (browser throw "ResizeObserver loop limit exceeded").

`IntersectionObserver` callback chạy muộn nhất — sau Composite (hoặc cuối frame, tùy implementation). Tại thời điểm này, frame đã được render xong. Bất kỳ DOM mutation nào trong callback sẽ ảnh hưởng frame tiếp theo, không phải frame hiện tại. Đây là lý do IntersectionObserver phù hợp cho lazy loading và analytics, nhưng không phù hợp cho animations cần response ngay lập tức với visibility change.
