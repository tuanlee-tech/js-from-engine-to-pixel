## A — HTTP Caching

### Tại sao HTTP Caching phức tạp hơn bạn nghĩ

Hầu hết người học chỉ biết "có cache thì nhanh hơn". Nhưng câu hỏi thật là: **browser quyết định dùng cache hay không bằng cơ chế gì? Và khi nào cache gây bug?**

### A.1 — Hai loại cache hoàn toàn khác nhau

Browser có nhiều tầng cache, nhưng quan trọng nhất là hai loại này:

```
Request đến → Memory Cache → Disk Cache → Network
                  ↑                ↑
             (RAM, nhanh,    (SSD/HDD, chậm hơn,
              mất khi         tồn tại qua sessions,
              tab đóng)       có thể hàng GB)
```

**Memory cache:** Lưu resources đã dùng trong session hiện tại. Tốc độ truy cập ~microseconds. Khi tab đóng → mất. Ví dụ: cùng một image xuất hiện nhiều lần trong page → chỉ download 1 lần, các lần sau lấy từ memory cache.

**Disk cache:** Tồn tại qua sessions. Khi mở tab mới vào lại site → vẫn còn cache. Kích thước giới hạn (Chrome tự quản lý, thường vài GB). Eviction dựa vào LRU + file size + access frequency.

**Service Worker cache:** Tầng thứ 3, hoàn toàn do developer kiểm soát. Intercept mọi request trước khi đến HTTP cache. Đây là nền tảng của PWA offline support.

```
Request
  ↓
Service Worker (nếu có) → có thể trả về cached response, fake response, v.v.
  ↓ (nếu không intercept)
Memory Cache
  ↓ (miss)
Disk Cache (HTTP Cache)
  ↓ (miss hoặc stale)
Network Request
```

### A.2 — Cache-Control: Cơ chế thật

`Cache-Control` là header quan trọng nhất. Đây là **chuỗi directives**, mỗi directive có ý nghĩa cụ thể:

#### `max-age=N`

```
Cache-Control: max-age=31536000
```

Resource được coi là **fresh** trong N giây kể từ lúc nhận response. Trong thời gian này, browser **không gửi network request** — dùng cache trực tiếp, kể cả khi refresh (trừ hard refresh).

```
T=0:    Browser nhận response với max-age=3600
T=1800: User refresh → Browser kiểm tra: 1800 < 3600 → FRESH → Dùng cache, không gửi request
T=4000: User refresh → Browser kiểm tra: 4000 > 3600 → STALE → Gửi request mới
```

**Đây là "strong cache"** — không có network round-trip khi fresh. Response status trong DevTools hiển thị **200 (from disk cache)** hoặc **200 (from memory cache)**.

#### `no-cache` — Tên gây hiểu nhầm nhiều nhất

```
Cache-Control: no-cache
```

`no-cache` **KHÔNG có nghĩa là không cache**. Nó có nghĩa là: **cache resource, nhưng phải validate với server trước mỗi lần dùng**.

```
Browser có cached response với no-cache
  ↓
User request resource
  ↓
Browser gửi conditional request lên server (với ETag hoặc Last-Modified)
  ↓
Server kiểm tra: resource thay đổi chưa?
  ├── Chưa thay đổi → 304 Not Modified (không gửi body) → Browser dùng cache
  └── Đã thay đổi   → 200 OK + new content → Browser update cache
```

`no-cache` vẫn tiết kiệm bandwidth (304 không có body) nhưng vẫn có network round-trip.

#### `no-store`

```
Cache-Control: no-store
```

**Thật sự không cache** — không lưu vào disk, không lưu vào memory. Mỗi request đều phải download lại hoàn toàn. Dùng cho sensitive data (banking transactions, medical records).

#### `private` vs `public`

```
Cache-Control: private, max-age=3600
Cache-Control: public, max-age=86400
```

`private`: Chỉ browser của user đó được cache. CDN và intermediate proxies không được cache. Dùng cho personalized content (user profile, dashboard).

`public`: CDN và proxies có thể cache. Dùng cho shared resources (JS bundles, images, fonts).

#### `immutable` — ít biết nhưng quan trọng

```
Cache-Control: public, max-age=31536000, immutable
```

`immutable` nói với browser: "Resource này sẽ **không bao giờ thay đổi** trong khoảng max-age. Đừng gửi conditional request kể cả khi user refresh."

Không có `immutable`: User refresh → browser gửi conditional request (dù resource fresh) → server trả 304 → vẫn có network round-trip.

Có `immutable`: User refresh → browser biết resource không thay đổi → dùng cache trực tiếp, zero network.

**Đây là cách content-hashed assets hoạt động:**

```
app.js → hash → app.abc123.js (content hash trong tên file)
Cache-Control: public, max-age=31536000, immutable
```

Nếu content thay đổi → hash thay đổi → URL mới → cache miss → download lại. Nếu content không đổi → URL cũ → cache hit, 1 năm không cần download lại.

#### `stale-while-revalidate` — Pattern quan trọng cho UX

```
Cache-Control: max-age=60, stale-while-revalidate=3600
```

**Cơ chế:**

- Trong 60 giây: Fresh → dùng cache trực tiếp
- 60s → 3660s: Stale, nhưng **vẫn trả về cache ngay lập tức** (để UX không bị delay), **đồng thời gửi revalidation request background**
- Sau 3660s: Stale quá lâu → phải chờ network

```
T=0:    Nhận response max-age=60, stale-while-revalidate=3600
T=30:   Request → FRESH → Cache (0ms delay)
T=90:   Request → STALE nhưng trong SWR window → Cache ngay (0ms delay)
                          + Background request để update cache
T=5000: Request → STALE, ngoài SWR window → Phải chờ network
```

**Tại sao quan trọng:** Với API responses không cần real-time, SWR cho phép instant response mọi lúc trong khi vẫn keep cache up-to-date. Next.js sử dụng pattern này cho `revalidate` trong ISR.

### A.3 — ETag & Conditional Requests: Cơ chế validation

Khi cache stale (hoặc `no-cache`), browser gửi **conditional request** thay vì download lại toàn bộ.

```
=== FIRST REQUEST ===
Browser → GET /api/data HTTP/1.1
Server  → 200 OK
          ETag: "abc123"              ← fingerprint của response
          Last-Modified: Mon, 01 Jan 2024 00:00:00 GMT
          [full response body]

=== SUBSEQUENT REQUEST (cache stale) ===
Browser → GET /api/data HTTP/1.1
          If-None-Match: "abc123"     ← gửi ETag nhận được
          If-Modified-Since: Mon, 01 Jan 2024 00:00:00 GMT

Server kiểm tra: data thay đổi từ "abc123" chưa?
├── Không thay đổi → 304 Not Modified (no body) ← tiết kiệm bandwidth
└── Đã thay đổi   → 200 OK + ETag: "xyz789" + [new body]
```

**ETag vs Last-Modified:**

- `ETag`: Hash/fingerprint của content. Chính xác hơn. Server tự tạo (thường là hash MD5/SHA của content hoặc version number).
- `Last-Modified`: Timestamp của lần cuối file thay đổi. Vấn đề: timestamp có thể thay đổi mà content không thay đổi (deploy lại cùng file), hoặc timestamp không đủ granular (1-second precision).
- Khi cả hai cùng có: `ETag` được ưu tiên.

### A.4 — Cache Invalidation: Bài toán khó nhất

> "There are only two hard things in Computer Science: cache invalidation and naming things." — Phil Karlton

**Vấn đề:** Bạn deploy version mới của `app.js`. Users đang có cached version cũ. Làm sao họ nhận được version mới?

**Với `max-age` còn hiệu lực:** Browser **không gửi request**, không biết version mới tồn tại. Phải chờ cache expire.

**Giải pháp: Cache Busting với Content Hash:**

```
# Build cũ
app.js → bundle → app.js
         Cache-Control: max-age=31536000
         # Problem: URL không đổi, cache không biết cần update

# Build mới (Webpack/Vite)
app.js → bundle → app.a1b2c3d4.js  ← hash của content
         Cache-Control: public, max-age=31536000, immutable
         # URL thay đổi khi content thay đổi → cache miss → download mới
```

**HTML thì sao?** HTML không thể content-hash vì HTML chứa URLs của JS/CSS (chicken-and-egg problem). HTML thường dùng:

```
Cache-Control: no-cache
# Hoặc
Cache-Control: max-age=0, must-revalidate
```

HTML luôn được validate, nhưng nếu không thay đổi → 304, không tốn bandwidth.

```
Tier 1: HTML               → no-cache (luôn validate, 304 nếu không đổi)
Tier 2: JS/CSS bundles     → max-age=1year + immutable + content hash in URL
Tier 3: Images, fonts      → max-age=1year + immutable + content hash hoặc version
Tier 4: API responses      → depends (no-cache, or short max-age + SWR)
```

## B — TCP Slow Start & The 14KB Rule

### B.1 — Tại sao TCP không gửi full speed ngay từ đầu?

TCP được thiết kế để **tự động tìm băng thông khả dụng** mà không làm nghẽn mạng. Cơ chế là **Congestion Control** với **Slow Start algorithm**.

```
Khi TCP connection mới được thiết lập:
  Congestion Window (cwnd) = 10 segments (≈ 14KB)

  Round 1: Gửi 10 segments (14KB)
    → Nhận ACK → cwnd = 20 segments (28KB)
  Round 2: Gửi 20 segments (28KB)
    → Nhận ACK → cwnd = 40 segments (56KB)
  Round 3: Gửi 40 segments (56KB)
    ...
```

**cwnd tăng theo exponential** cho đến khi:

1. Đạt **ssthresh** (slow start threshold) → chuyển sang Congestion Avoidance (tăng linear)
2. Packet loss xảy ra → reset cwnd, bắt đầu lại

```
cwnd (segments)
^
|                                          *
|                                      * *
|                                  * *
|          Congestion          * *
|           Avoidance      * *
|  Slow   /           * *
|  Start /        * *          ← packet loss → reset
|       /     * *
|      /  * *
|     / *
|    *
|   *
|  *
| *
+────────────────────────────────────────► time (RTTs)
   1  2  3  4  5  6  7  8  9
```

### B.2 — The 14KB Rule — Tại sao con số này

**10 segments × 1460 bytes/segment ≈ 14,600 bytes ≈ 14KB**

(1460 bytes = Maximum Segment Size của TCP trên Ethernet với IPv4, sau khi trừ TCP/IP headers)

Đây là lượng data browser nhận được **trong Round Trip đầu tiên**. Round trip thứ 2 nhận 28KB. Thứ 3 nhận 56KB.

**Với server ở Mỹ, user ở Việt Nam (RTT ~200ms):**

```
T=0ms:   Browser gửi HTTP request (sau khi TCP+TLS handshake xong)
T=200ms: Server nhận request, bắt đầu gửi response
T=400ms: Browser nhận 14KB đầu tiên
T=600ms: Browser nhận 28KB tiếp theo
T=800ms: Browser nhận 56KB tiếp theo
...
```

Nếu HTML critical rendering path nằm gọn trong 14KB đầu:

- Browser nhận HTML lúc T=400ms
- Bắt đầu parse và render ngay
- Trong khi chờ 14KB tiếp theo (T=600ms), browser đã có thể hiển thị first content

Nếu HTML > 14KB:

- First paint bị delay đến T=600ms (hoặc hơn)
- Mỗi extra RTT = +200ms delay cho user

### B.3 — Thực tế: 14KB có nghĩa là gì với HTML

```
Ngân sách 14KB (khoảng 14,336 bytes sau compression):

Ví dụ HTML cơ bản:
<html><head>
  Critical CSS (inline):     ~3KB
  Preload hints:             ~0.5KB
  Critical fonts metadata:   ~0.2KB
  Above-the-fold HTML:       ~5KB
  Meta tags, Open Graph:     ~0.5KB
</head><body>
  Hero section HTML:         ~2KB
  Navigation HTML:           ~1KB
</body></html>

Tổng: ~12.2KB — còn trong budget ✅

Vấn đề phổ biến:
  Server-side rendered component tree: 20KB+ ❌
  Không inline critical CSS → thêm 1 RTT để load CSS file ❌
  Uncompressed HTML → 14KB budget tiêu thụ 3-4x nhanh hơn ❌
```

**Gzip/Brotli compression quan trọng vì lý do này:** Brotli nén HTML trung bình 3-4x tốt hơn plain text. 14KB compressed ≈ 40-56KB plain text. Với `Content-Encoding: br`, ngân sách 14KB cho đầu tiên thực sự có thể chứa nhiều content hơn nhiều.

### B.4 — HTTP/3 (QUIC) thay đổi game như thế nào

TCP Slow Start là vấn đề của **TCP**. HTTP/3 dùng **QUIC** (chạy trên UDP) và giải quyết một số vấn đề:

```
TCP Head-of-Line Blocking:
Stream 1: [data][data][LOST][data][data]
Stream 2:                    WAIT → WAIT → WAIT
Stream 3:                    WAIT → WAIT → WAIT
(Tất cả streams phải chờ packet lost của Stream 1)

QUIC Per-Stream Loss Recovery:
Stream 1: [data][data][LOST][data][data] ← retransmit chỉ stream 1
Stream 2: [data][data][data][data] → chạy bình thường ✅
Stream 3: [data][data][data][data] → chạy bình thường ✅
```

QUIC vẫn có Slow Start nhưng có thể "remember" cwnd từ connection trước (connection ID migration), giúp subsequent connections không phải bắt đầu từ cwnd=10.

## C — Preload Scanner: Cơ chế thật

### C.1 — Vấn đề: Parser bị block, resources bị delay

Không có Preload Scanner:

```
HTML: <head>
        <link rel="stylesheet" href="styles.css">   ← CSS download
        <script src="jquery.js"></script>            ← Parser BLOCK ở đây
        ...                                          ← Parser chưa đến đây
      </head>
      <body>
        <img src="hero.jpg">                         ← Chưa discover!
        <script src="app.js"></script>               ← Chưa discover!
      </body>

Kết quả: hero.jpg và app.js chỉ được discover SAU KHI jquery.js execute xong
         Delay = thời gian download + execute jquery.js
```

### C.2 — Preload Scanner giải quyết như thế nào

Khi main HTML parser bị block (bởi script, hoặc chờ CSS), browser **kích hoạt một parser phụ** — Preload Scanner (còn gọi là Speculative Parser hoặc Look-ahead Parser). Parser phụ này:

- Scan raw HTML bytes phía trước (không build DOM)
- Tìm kiếm pattern: `src=`, `href=`, `action=` trong các tags cụ thể
- Enqueue download requests cho resources tìm thấy

```
Main Parser:         Parse HEAD
                     Gặp <script src="jquery.js"> → BLOCK ⬛
                     (Đang chờ jquery.js download + execute)

Preload Scanner:     Scan tiếp phần HTML còn lại (raw bytes)
                     Tìm thấy: hero.jpg → Enqueue download (thấp priority)
                     Tìm thấy: app.js  → Enqueue download (thấp priority)
                     Tìm thấy: logo.png → Enqueue download

Kết quả: hero.jpg, app.js, logo.png bắt đầu download
         TRONG KHI main parser đang block chờ jquery.js
```

**Visual timeline:**

```
Không có Preload Scanner:
──────────────────────────────────────────────────────
[Parse HEAD][jquery download][jquery exec][Parse BODY][hero.jpg download]
                                                       ↑ Phát hiện hero.jpg muộn

Có Preload Scanner:
──────────────────────────────────────────────────────
[Parse HEAD][jquery download]     [jquery exec][Parse BODY]
             [hero.jpg download──────────────]
             [app.js download────────────────]
              ↑ Preload Scanner discover và download song song
```

### C.3 — Giới hạn của Preload Scanner — quan trọng để hiểu đúng

Preload Scanner **chỉ scan HTML**. Nó không thể discover:

**1. Resources trong CSS:**

```css
/* styles.css */
.hero {
  background-image: url('/images/hero.jpg'); /* ← Preload Scanner không thấy */
}

@font-face {
  src: url('/fonts/custom.woff2'); /* ← Preload Scanner không thấy */
}
```

→ Đây là lý do `preload` hint trong HTML cần thiết cho critical CSS backgrounds và custom fonts.

**2. Resources được inject bởi JavaScript:**

```javascript
// Script này inject image sau khi execute
const img = document.createElement('img');
img.src = '/images/dynamic.jpg'; // ← Preload Scanner không thấy
document.body.appendChild(img);
```

→ Dynamic resources không được discovered sớm.

**3. Resources trong `<template>` tags:**

```html
<template>
  <img src="/images/template-img.jpg" />
  <!-- ← Preload Scanner bỏ qua -->
</template>
```

→ Template content không được parsed đến khi sử dụng.

**4. Resources sau lazy-loaded JS modules:**

```javascript
// app.js
import('/features/chart.js').then(({ Chart }) => {
  // Chart.js có thể cần thêm resources → không được preload
});
```

### C.4 — Implications: Tại sao biết điều này quan trọng

```javascript
// ❌ Tệ: Hero image trong CSS background
// Preload Scanner không thấy → chỉ discover sau khi:
// 1. CSS file download xong
// 2. CSSOM build xong
// 3. Render tree build xong
// 4. Browser biết .hero element cần background
// → Rất muộn

// .hero { background-image: url('/images/hero.jpg'); }

// ✅ Tốt: Preload trong HTML + img tag
// <link rel="preload" as="image" href="/images/hero.jpg">
// <img src="/images/hero.jpg" alt="Hero" class="hero">
// → Preload Scanner discover ngay từ đầu

// ❌ Tệ: Font chỉ trong @font-face
// @font-face { src: url('/fonts/Inter.woff2'); }
// → Chỉ discover sau khi parse CSS + build CSSOM + check font-display

// ✅ Tốt: Preload font trong HTML
// <link rel="preload" as="font" href="/fonts/Inter.woff2" type="font/woff2" crossorigin>
```

**Quy tắc thực hành:**

> Bất kỳ resource nào browser không thể discover trực tiếp từ HTML → cần `preload` hint trong `<head>`.

## D — Speculation Rules API

### D.1 — Lịch sử: Prerender 1.0 và tại sao nó thất bại

Chrome từng có `<link rel="prerender" href="/next-page">`. Nó render toàn bộ page trong background tab ẩn. Khi user navigate → swap tab → instant.

**Tại sao bị deprecated:**

- Render toàn bộ page = CPU + memory + bandwidth tốn kém
- Không thể cancel khi user không navigate đến
- Side effects: analytics fired, form submitted, videos started playing
- Không support nhiều candidates (chỉ 1 prerender tại một thời điểm)
- Không có control nào cho developer

### D.2 — Speculation Rules API (Chrome 109+)

Speculation Rules là successor, được thiết kế để giải quyết các vấn đề trên.

```html
<script type="speculationrules">
  {
    "prerender": [
      {
        "where": { "href_matches": "/product/*" },
        "eagerness": "moderate"
      }
    ],
    "prefetch": [
      {
        "where": { "selector_matches": ".nav-link" },
        "eagerness": "conservative"
      }
    ]
  }
</script>
```

#### Hai loại speculation:

**`prefetch`**: Download HTML của page đó vào cache. Khi navigate → vẫn phải parse HTML, execute JS, render. Nhưng nhanh hơn vì không cần network.

**`prerender`**: Render toàn bộ page trong background (hidden renderer). Khi navigate → swap rendered page → gần như instant (~instant FCP, LCP).

#### `eagerness` — kiểm soát khi nào speculation kích hoạt:

```
"immediate"    → Speculate ngay khi page load
                 (dùng cho resources chắc chắn cần)

"eager"        → Speculate khi có dấu hiệu sớm
                 (thường là khi link visible trong viewport)

"moderate"     → Speculate khi pointer hover gần link (~200ms before click)
                 (cân bằng giữa usefulness và waste)

"conservative" → Chỉ speculate khi user bắt đầu click/tap
                 (ít waste nhất, vẫn đủ thời gian để prefetch)
```

**Cơ chế "moderate" ví dụ:**

```
User di chuyển mouse về phía link
  ↓
~200ms trước khi click thường xảy ra
  ↓
Browser bắt đầu prerender /product/123
  ↓
User click
  ↓
Prerender hoàn tất (hoặc gần hoàn tất)
  ↓
Swap → Instant navigation
```

### D.3 — Speculation Rules giải quyết vấn đề cũ như thế nào

```
Vấn đề cũ                    Speculation Rules giải pháp
─────────────────────────────────────────────────────────
Không cancel được          → Auto-cancel khi user không navigate
                             Browser quản lý lifecycle

Chỉ 1 prerender            → Multiple candidates, browser prioritize

Side effects (analytics)   → Prerendered page biết nó đang prerendered
                             via document.prerendering === true
                             Có thể defer analytics đến khi activate

Không granular control     → eagerness levels, selector_matches, href_matches

Lãng phí tất cả           → Prefetch cho uncertain, prerender cho confident
```

### D.4 — Opt-out cho pages không muốn được prerendered

Một số pages có side effects khi load (charge user, start video call, v.v.) và không muốn bị prerender:

```
Supports-Loading-Mode: credentialed-prerender
# hoặc
Supports-Loading-Mode: no-prerender (opt out hoàn toàn)
```

### D.5 — Phát hiện trang đang được prerendered

```javascript
// Trong prerendered page:
if (document.prerendering) {
  // Page đang được prerender trong background
  // ĐỪNG:
  // - Fire analytics
  // - Start video autoplay
  // - Request permissions
  // - Bắt đầu timers quan trọng

  document.addEventListener('prerenderingchange', () => {
    // Page được activate (user navigate đến)
    // Bây giờ có thể fire analytics
    firePageViewAnalytics();
  });
} else {
  // Regular navigation
  firePageViewAnalytics();
}
```

## E — Core Web Vitals Map vào CRP

Đây là phần kết nối tất cả lại:

### E.1 — Từng metric map vào đâu trong pipeline

```
Navigation Timeline:
T=0                                                                T=end
│                                                                      │
├─DNS─┤TCP┤TLS┤────── Request/Response ──────┤── Rendering Pipeline ──┤

      TTFB                FCP           LCP                         CLS
      ─┤├─                ─┤├─          ─┤├─                        scattered

TTFB (Time to First Byte):
  = DNS + TCP + TLS + Server processing time
  Ảnh hưởng bởi: Server location, server performance, CDN
  Target: < 800ms

FCP (First Contentful Paint):
  = TTFB + HTML download + Parse HTML (trên fold) + Render first content
  Ảnh hưởng bởi: Render-blocking resources, HTML size, Critical CSS
  Target: < 1.8s

LCP (Largest Contentful Paint):
  = FCP + Load time của largest element (thường hero image/heading)
  Ảnh hưởng bởi: Image optimization, preload, CDN, render-blocking JS
  Target: < 2.5s

CLS (Cumulative Layout Shift):
  Không phải time metric — là stability metric
  Mỗi lần element nhảy vị trí sau render → cộng vào CLS score
  Nguyên nhân: Images không có width/height, dynamic content inject, fonts gây FOUT
  Target: < 0.1
```

### E.2 — Tại sao mỗi giây trong pipeline quan trọng

```
Mỗi 100ms TTFB tăng:
  → FCP tăng 100ms
  → LCP tăng 100ms
  → Bounce rate tăng ~1% (Google research)

Render-blocking CSS 500ms:
  → FCP delay 500ms minimum
  → LCP delay 500ms minimum

Hero image không được preload (phát hiện muộn 800ms):
  → LCP delay 800ms
  → Thường là difference giữa "Good" và "Needs Improvement" LCP
```

### Service Worker: Cache + Network Strategy

```javascript
// service-worker.js

const CACHE_NAME = 'app-v1';
const STATIC_ASSETS = [
  '/css/critical.css',
  '/fonts/Inter.woff2',
  '/images/logo.svg',
];

// Install: cache static assets
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then((cache) => cache.addAll(STATIC_ASSETS)),
  );
});

self.addEventListener('fetch', (event) => {
  const { request } = event;
  const url = new URL(request.url);

  // Strategy 1: Cache First (static assets với content hash)
  // → Trả về cache ngay, không cần network
  // → Dùng cho: JS bundles, CSS với hash, fonts
  if (url.pathname.startsWith('/_next/static/')) {
    event.respondWith(
      caches.match(request).then((cached) => {
        // Cache hit → trả về ngay (cực nhanh)
        if (cached) return cached;

        // Cache miss → fetch và cache cho lần sau
        return fetch(request).then((response) => {
          const clone = response.clone();
          caches.open(CACHE_NAME).then((cache) => cache.put(request, clone));
          return response;
        });
      }),
    );
    return;
  }

  // Strategy 2: Stale While Revalidate (API responses)
  // → Trả về cache ngay (nếu có) + background fetch để update
  // → Dùng cho: API responses, product listings
  if (url.pathname.startsWith('/api/products')) {
    event.respondWith(
      caches.open(CACHE_NAME).then(async (cache) => {
        const cached = await cache.match(request);

        // Background fetch để update cache
        const networkFetch = fetch(request).then((response) => {
          cache.put(request, response.clone());
          return response;
        });

        // Trả về cached ngay (nếu có), hoặc chờ network
        return cached || networkFetch;
      }),
    );
    return;
  }

  // Strategy 3: Network First (HTML, user data)
  // → Thử network trước, fallback cache nếu offline
  event.respondWith(
    fetch(request)
      .then((response) => {
        // Network thành công → cache và trả về
        const clone = response.clone();
        caches.open(CACHE_NAME).then((cache) => cache.put(request, clone));
        return response;
      })
      .catch(() => {
        // Offline → trả về cache
        return caches.match(request);
      }),
  );
});
```

```javascript
// Dynamic Speculation Rules (thêm/xóa dựa vào user behavior)
class SpeculationRulesManager {
  constructor() {
    this.scriptEl = null;
    this.rules = { prerender: [], prefetch: [] };
  }

  addPrerenderCandidate(url, eagerness = 'moderate') {
    // Khi user đã nhìn một product nhiều lần
    // → Tăng eagerness
    this.rules.prerender.push({
      urls: [url],
      eagerness,
    });
    this._apply();
  }

  addPrefetchUrls(urls) {
    this.rules.prefetch.push({
      urls,
      eagerness: 'conservative',
    });
    this._apply();
  }

  _apply() {
    // Xóa rule cũ
    if (this.scriptEl) {
      this.scriptEl.remove();
    }

    // Tạo rule mới
    this.scriptEl = document.createElement('script');
    this.scriptEl.type = 'speculationrules';
    this.scriptEl.textContent = JSON.stringify(this.rules);
    document.head.appendChild(this.scriptEl);
    // Browser tự handle: cancel prerender cũ, bắt đầu prerender mới
  }
}

// Trong page:
const speculationManager = new SpeculationRulesManager();

// Track user behavior
const viewCounts = new Map();

document.querySelectorAll('.product-card').forEach((card) => {
  const observer = new IntersectionObserver((entries) => {
    entries.forEach((entry) => {
      if (entry.isIntersecting) {
        const url = card.querySelector('a').href;
        const count = (viewCounts.get(url) || 0) + 1;
        viewCounts.set(url, count);

        // Sau lần thứ 2 nhìn vào product → prerender
        if (count >= 2) {
          speculationManager.addPrerenderCandidate(url, 'eager');
          observer.unobserve(entry.target);
        }
      }
    });
  });
  observer.observe(card);
});
```

## CÂU HỎI ÔN TẬP — (Dành cho NotebookLM)

### Câu 1

**Hỏi:** `Cache-Control: no-cache` và `Cache-Control: no-store` khác nhau như thế nào? Khi nào dùng cái nào?

**Đáp án chi tiết:** `no-cache` không có nghĩa là không cache. Nó có nghĩa là: lưu vào cache, nhưng trước mỗi lần dùng phải gửi conditional request lên server để validate (via `If-None-Match` với ETag, hoặc `If-Modified-Since`). Nếu server trả về 304 Not Modified, browser dùng cached version — không tốn bandwidth nhưng vẫn có network round-trip. `no-store` thật sự không lưu gì cả — mỗi lần phải download toàn bộ từ server. Dùng `no-cache` cho: HTML pages (cần validate để nhận deploy mới, nhưng không cần download lại nếu không đổi), API responses thường xuyên thay đổi nhưng muốn tận dụng 304 để tiết kiệm bandwidth. Dùng `no-store` cho: sensitive data thật sự không được cache ở bất kỳ đâu — banking transactions, medical records, temporary auth tokens, bất kỳ thứ gì mà việc cache tạo ra security risk.

### Câu 2

**Hỏi:** Giải thích tại sao "đặt hero image trong CSS background" lại ảnh hưởng xấu đến LCP. Cơ chế discover resource diễn ra như thế nào trong hai trường hợp: CSS background và `<img>` tag với preload?

**Đáp án chi tiết:** Với CSS background: browser phải hoàn thành chuỗi sau trước khi biết image cần load:

(1) Download HTML

→ (2) Parse HTML đến thẻ `<link rel="stylesheet">`

→ (3) Download CSS file

→ (4) Parse CSS và build CSSOM

→ (5) Kết hợp DOM + CSSOM để build Render Tree

→ (6) Traverse Render Tree, tìm `.hero` element

→ (7) Đọc `background-image` property

→ (8) Mới bắt đầu download image.

Toàn bộ chuỗi này có thể mất 500-1500ms trước khi download bắt đầu, tùy vào kích thước CSS và network speed.

Với `<img>` tag và preload hint: browser có hai con đường discover nhanh hơn: (1) Preload Scanner đọc raw HTML bytes, gặp `<link rel="preload" as="image">` trong `<head>` → bắt đầu download ngay lập tức, song song với mọi thứ khác.

(2) Khi Main Parser đến thẻ `<img src="...">` → cũng bắt đầu download (hoặc nhận ra preload đang chạy). Image download bắt đầu ngay từ những RTTs đầu tiên, thay vì sau khi toàn bộ CSS pipeline hoàn tất. Kết quả thực tế: difference thường 500-1500ms trong LCP — đủ để chuyển từ "Good" (<2.5s) sang "Needs Improvement" (>2.5s) trong Core Web Vitals.

### Câu 3

**Hỏi:** TCP Slow Start ảnh hưởng thế nào đến Critical Rendering Path? Tại sao "14KB đầu tiên" quan trọng, và một developer có thể làm gì để tận dụng con số này?

**Đáp án chi tiết:** TCP bắt đầu mỗi connection với Congestion Window (cwnd) nhỏ — khoảng 10 TCP segments tương đương ~14KB. Con số này tăng theo exponential (double mỗi RTT) cho đến khi đạt ngưỡng hoặc xảy ra packet loss. Điều này có nghĩa là: bytes đầu tiên browser nhận được trong Round Trip đầu tiên chỉ là 14KB. Với server ở xa (RTT ~200ms), Round Trip 1 xảy ra ở T≈400ms (sau TCP+TLS handshake), Round Trip 2 ở T≈600ms (+200ms), Round Trip 3 ở T≈800ms.

Nếu toàn bộ HTML critical path — HTML markup, inline critical CSS, preload hints — nằm gọn trong 14KB compressed đầu tiên, browser có thể bắt đầu parse và render ngay lúc T=400ms. Nếu HTML lớn hơn 14KB, first paint bị delay đến T=600ms tối thiểu, thêm 200ms.

Developer có thể tận dụng:

(1) Inline critical CSS thay vì load file riêng — loại bỏ extra round-trip cho CSS.

(2) Brotli/Gzip compression — 14KB compressed ≈ 40-56KB HTML raw text.

(3) Giảm HTML size: server render chỉ above-the-fold content, lazy render phần còn lại.

(4) Sử dụng `preload` hints trong `<head>` để Preload Scanner discover resources trong khi vẫn trong giai đoạn đầu.

(5) Tối ưu server TTFB — mọi ms server mất đều delay thời điểm 14KB đầu tiên đến browser.
