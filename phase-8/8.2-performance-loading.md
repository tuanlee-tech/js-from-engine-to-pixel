## 1. Cơ chế thật — HTML Parser & Resource Loading

Trước khi đi vào từng attribute, cần hiểu **HTML parser hoạt động như thế nào với resources**.

### Parser Threading Model

Browser HTML parser chạy trên **main thread**. Khi parser gặp resource cần load, có hai quyết định:

1. **Parser có phải dừng lại không?** (parser blocking)
2. **Resource được fetch ở thread nào?**

```
Main Thread:
HTML Parser ─────────────────────────────────────────────────────────▶
             Parse HTML   [BLOCKED]   Parse HTML   [BLOCKED]   Parse HTML
                         ↑                        ↑
                    Script fetch              Script exec
                    (network thread)          (main thread)

Network Thread:
                    ════════════════▶  ════════════════▶
                    fetch script 1     fetch script 2
```

Browser có một **preload scanner** — một secondary parser chạy **song song** với main parser để scan ahead và dispatch resource requests sớm. Đây là lý do `<link rel="preload">` trong `<head>` hiệu quả hơn dynamic insertion bằng JS.

```
Main Thread Parser:     <html> → <head> → <meta> → [blocked by script]...
                                                            ↑
Preload Scanner:        <html> → <head> → <meta> → <script> → <link> → <img>
(parallel)              Phát hiện resources trước, dispatch fetch sớm
```

### Resource Priority

Browser không fetch mọi thứ như nhau. Mỗi resource type có **default priority**:

```
HIGHEST:  HTML document, CSS (render-blocking), Fonts (in preload)
HIGH:     Script (parser-blocking), preload với as="script"/"style"
MEDIUM:   Script (async/defer), preload với as="font", Images (in viewport)
LOW:      Images (out of viewport), prefetch
LOWEST:   Everything else
```

Priority có thể bị override bằng `fetchpriority` attribute (🟡).

## 2. Script Loading (🔴) — Cơ chế chi tiết

### 4 cách load script — khác nhau thật sự

```
Case 1: <script src="app.js">
─────────────────────────────
HTML Parse ───▶ [STOP]
                ├── Fetch app.js (network)
                ├── Execute app.js (main thread)
                └── [RESUME] HTML Parse

Case 2: <script src="app.js" defer>
────────────────────────────────────
HTML Parse ─────────────────────────────────────────▶ [DOMContentLoaded]
               │                                               ↑
               └── Fetch app.js (parallel, ngay lập tức)      │
                   ═══════════════════════════▶ Execute ───────┘
                   (sau khi parse xong, trước DOMContentLoaded)

Case 3: <script src="app.js" async>
────────────────────────────────────
HTML Parse ──────────────────────[STOP]──────────────▶
               │                 ↑
               └── Fetch app.js  Execute (ngay khi fetch xong, không chờ parse)
                   ═════════════▶

Case 4: <script type="module" src="app.js">
────────────────────────────────────────────
HTML Parse ─────────────────────────────────────────▶
               │                                     ↑
               └── Fetch app.js + dependencies ──────┘ Execute
                   (module graph resolution)         (trước DOMContentLoaded, như defer)
```

### So sánh đầy đủ

|                       | Parser block? | Fetch               | Execute timing            | Order guarantee |
| --------------------- | ------------- | ------------------- | ------------------------- | --------------- |
| `<script>`            | ✅ Có         | Ngay khi gặp        | Ngay sau fetch            | Có              |
| `defer`               | ❌ Không      | Ngay khi gặp        | Sau parse xong, trước DCL | Có (theo order) |
| `async`               | ❌ Không      | Ngay khi gặp        | Ngay khi fetch xong       | Không           |
| `type="module"`       | ❌ Không      | Ngay khi gặp + deps | Sau parse xong, trước DCL | Có              |
| `type="module" async` | ❌ Không      | Ngay khi gặp + deps | Ngay khi fetch xong       | Không           |

### Ví dụ production — Quyết định dùng gì

```html
<!DOCTYPE html>
<html>
  <head>
    <meta charset="UTF-8" />

    <!--
    CSS: render-blocking theo mặc định.
    Browser không paint cho đến khi CSS load + parse xong.
    → Đặt CSS trong <head> để bắt đầu fetch sớm nhất có thể.
  -->
    <link
      rel="stylesheet"
      href="/static/main.css"
    />

    <!--
    Analytics: không cần DOM, không depend on other scripts
    → async: fetch parallel, execute ngay khi xong
    → Không quan tâm nếu nó execute sau một số DOM content
  -->
    <script
      async
      src="https://analytics.example.com/tracker.js"
    ></script>

    <!--
    Third-party widget không critical:
    → async vì không depend on DOM hay scripts khác
  -->
    <script
      async
      src="https://cdn.example.com/chat-widget.js"
    ></script>
  </head>
  <body>
    <!-- Content -->
    <div id="app"></div>

    <!--
    App bundle: cần DOM đã ready, cần execute theo thứ tự
    → defer: fetch parallel với HTML parse, execute sau khi parse xong
    → Order guarantee: vendor.js trước app.js
  -->
    <script
      defer
      src="/static/vendor.js"
    ></script>
    <script
      defer
      src="/static/app.js"
    ></script>

    <!--
    Module: cũng có defer behavior mặc định
    → Tốt cho code mới dùng ES Modules
    → Tự động strict mode
    → Không cần defer attribute (mặc định đã như defer)
  -->
    <script
      type="module"
      src="/static/main.js"
    ></script>

    <!--
    Polyfill: cần execute TRƯỚC app code, cần block
    → Nhưng thay vì block parser, inline nó nếu nhỏ
    → Hoặc dùng feature detection để conditional load
  -->
    <script>
      // Inline script không có src → execute ngay, không fetch
      // Nhỏ → cost của parser block nhỏ hơn network roundtrip
      if (!window.IntersectionObserver) {
        document.write(
          '<script src="/polyfills/intersection-observer.js"><\/script>',
        );
      }
    </script>
  </body>
</html>
```

### Module vs Script — Khác nhau kỹ thuật

```javascript
// type="module" tự động strict mode
// Không cần 'use strict'
const x = 10; // OK
delete x; // SyntaxError trong strict mode

// Module scope — không leak ra global
var moduleVar = 'hello';
console.log(window.moduleVar); // undefined — không phải global

// Script scope — var leak ra global
var scriptVar = 'hello';
console.log(window.scriptVar); // 'hello' — là global

// Module: import/export hoạt động
import { helper } from './utils.js';

// Module: import.meta
console.log(import.meta.url); // URL của module này
console.log(import.meta.env); // Với bundler như Vite

// Module: chạy một lần dù import nhiều nơi
// Script bình thường: mỗi <script src="..."> là một execution riêng
```

## 3. Resource Hints (🔴) — Cơ chế và Sử dụng

### 4 resource hints — khác nhau về timing và scope

```
preconnect:   Kết nối TCP+TLS tới domain (không fetch resource nào cụ thể)
dns-prefetch: Chỉ resolve DNS (rẻ hơn preconnect, ít lợi hơn)
preload:      Fetch resource cụ thể với priority cao, dùng trong trang hiện tại
prefetch:     Fetch resource cụ thể với priority thấp, dùng cho navigation sau
```

### DNS Prefetch vs Preconnect — Pipeline chi tiết

```
Mỗi request tới domain mới cần:

1. DNS Lookup:     domain → IP address          (~20-120ms)
2. TCP Handshake:  3-way handshake với server   (~RTT = ~50-200ms)
3. TLS Handshake:  Certificate exchange         (~RTT = ~50-200ms)
4. HTTP Request:   Actual resource fetch

dns-prefetch: Chỉ step 1
preconnect:   Step 1 + 2 + 3
```

```html
<!-- preconnect: tốt nhất cho third-party domains mà bạn chắc sẽ dùng -->
<!-- Browser sẽ TCP+TLS connect sẵn → khi cần fetch, tiết kiệm 150-400ms -->
<link
  rel="preconnect"
  href="https://fonts.googleapis.com"
/>
<link
  rel="preconnect"
  href="https://fonts.gstatic.com"
  crossorigin
/>
<!-- crossorigin: cần cho CORS requests (fonts, API calls cross-origin) -->

<!-- dns-prefetch: fallback cho browsers không support preconnect -->
<!-- Hoặc dùng cho third-party domains ít chắc hơn sẽ dùng -->
<link
  rel="dns-prefetch"
  href="https://analytics.example.com"
/>
<link
  rel="dns-prefetch"
  href="https://cdn.example.com"
/>
```

### Preload — Tại sao cần nó?

Preload giải quyết vấn đề **discovery latency**. Browser chỉ biết cần fetch một resource khi parser/renderer gặp nó. Một số resources bị **late discovered**:

```
Problem: Font được reference trong CSS
Timeline without preload:
  HTML parse → CSS fetch → CSS parse → Font URL discovered → Font fetch
  (Font fetch bắt đầu rất muộn → FOIT/FOUT)

Timeline with preload:
  HTML parse → [preload font] ─────────────────────────▶
               → CSS fetch → CSS parse → Font URL discovered (already fetched!)
```

```html
<head>
  <!-- Preload LCP image: image quan trọng nhất trên viewport -->
  <!-- Không có preload: image bị discover muộn sau khi parse HTML xong -->
  <link
    rel="preload"
    href="/images/hero.webp"
    as="image"
    type="image/webp"
  >

  <!-- Preload font: tránh FOIT (Flash of Invisible Text) -->
  <link
    rel="preload"
    href="/fonts/inter-var.woff2"
    as="font"
    type="font/woff2"
    crossorigin  <!-- BẮT BUỘC cho fonts dù cùng origin -->
  >

  <!-- Preload critical CSS cho above-the-fold (nếu dùng code splitting CSS) -->
  <link rel="preload" href="/css/critical.css" as="style">

  <!-- Preload script sẽ cần ngay (nhưng chưa execute, chỉ fetch) -->
  <link rel="preload" href="/js/checkout.js" as="script">

  <!--
    'as' attribute BẮT BUỘC — nếu thiếu:
    1. Priority sai (fetch as 'fetch' priority thay vì 'image' priority)
    2. Resource không được cache match đúng
    3. Không set Accept header đúng
  -->
</head>
```

## 4. Images — srcset, sizes, loading, decoding (🔴)

### Cơ chế: Responsive Images Selection Algorithm

Browser chọn source image nào để load theo thuật toán sau, **không phải CSS**:

```
Input vào thuật toán:
  1. srcset candidates (URLs + width descriptors hoặc density descriptors)
  2. sizes (media conditions → layout width)
  3. Device pixel ratio (window.devicePixelRatio)
  4. Viewport width

Thuật toán:
  1. Evaluate sizes → lấy effective size (layout width của image)
  2. Tính pixel width cần: effectiveSize × devicePixelRatio
  3. Chọn source nhỏ nhất trong srcset ≥ pixel width cần
```

```
Ví dụ:
  sizes="(max-width: 768px) 100vw, 50vw"
  Viewport: 1200px, DPR: 2

  → Viewport > 768px → effective size = 50vw = 600px
  → Pixel width cần = 600 × 2 = 1200px
  → Chọn source ≥ 1200px từ srcset
```

### srcset với width descriptors (w) — Phổ biến nhất

```html
<img
  src="/images/product-800.jpg"    <!-- fallback cho browsers không support srcset -->
  srcset="
    /images/product-400.jpg  400w,
    /images/product-800.jpg  800w,
    /images/product-1200.jpg 1200w,
    /images/product-1600.jpg 1600w
  "
  sizes="
    (max-width: 640px)  100vw,    <!-- Mobile: full width -->
    (max-width: 1024px) 50vw,     <!-- Tablet: half width -->
    33vw                           <!-- Desktop: 1/3 width (default) -->
  "
  alt="Laptop Pro M3"
  width="800"     <!-- BẮT BUỘC để tránh layout shift (CLS) -->
  height="600"    <!-- BẮT BUỘC: browser tính aspect ratio để reserve space -->
>
```

**Tại sao `width` và `height` quan trọng với CLS:**

```
Không có width/height:
  Browser load HTML → tạo img element với size 0×0
  → Layout: content sau image được đặt cao
  → Image load → image có size → content bị đẩy xuống → CLS spike

Có width/height:
  Browser thấy width=800 height=600 → tính aspect ratio = 4:3
  → CSS: aspect-ratio: 800/600 tự động (từ Chrome 88+)
  → Browser reserve đúng space trước khi image load
  → Content không shift khi image load → CLS = 0
```

### srcset với density descriptors (x) — Ít dùng hơn

```html
<!--
  x descriptors: không cần sizes, đơn giản hơn
  Dùng khi image luôn có fixed size trong layout
  (logo, icon, avatar với fixed CSS width)
-->
<img
  src="/images/logo.png"
  srcset="/images/logo.png 1x, /images/logo@2x.png 2x, /images/logo@3x.png 3x"
  alt="Company Logo"
  width="120"
  height="40"
/>
```

```html
  width="1200"
  height="600"

  src="/hero.jpg"
  loading="lazy"
  alt="Hero"
/>
<!-- LCP image nằm trong viewport → lazy load trì hoãn fetch → LCP tệ -->

<!-- ✅ FIX: Above-the-fold images không lazy load -->
<!-- Dùng lazy chỉ cho images chắc chắn không trong viewport ban đầu -->
```

### `decoding="async"` — Decode ngoài main thread

```html
<!--
  Image decoding: Sau khi fetch xong, browser phải DECODE bytes → pixel data
  Decode là CPU-intensive, nếu chạy trên main thread → janky scrolling

  decoding="sync" (default): decode trên main thread, block rendering cho đến xong
  decoding="async": decode ngoài main thread, không block rendering
  decoding="auto": browser tự quyết

  Dùng async cho:
  - Images không critical cho initial render
  - Images trong scroll-heavy lists
  - Large images

  Không dùng async cho LCP image:
  - async có thể delay khi image appear → LCP tệ
-->
<img
  src="/products/product-1.jpg"
  loading="lazy"
  decoding="async"    <!-- Decode off main thread → smooth scroll -->
  alt="Product"
  width="300"
  height="300"
>

<!-- LCP image: eager + không async decode -->
<img
  src="/hero.jpg"
  loading="eager"
  decoding="sync"    <!-- Hoặc bỏ attribute → sync là default -->
  fetchpriority="high"  <!-- 🟡 Signal browser đây là priority cao -->
  alt="Hero"
  width="1200"
  height="600"
>
```

## 5. Picture Element — Art Direction (🟡)

### Cơ chế: Source Selection Algorithm

`<picture>` khác `<img srcset>` về mục đích:

```
srcset + sizes: Cùng một image, khác resolution/size
                Browser chọn size phù hợp nhất
                (optimization)

<picture>:      Khác nhau hoàn toàn về image content/crop/format
                Developer dictate điều kiện (browser phải follow)
                (art direction + format negotiation)
```

```html
<!-- Art direction: image khác nhau theo viewport -->
<picture>
  <!-- Mobile: crop portrait, chỉ hiện subject -->
  <source
    media="(max-width: 640px)"
    srcset="/hero-mobile-400.webp 400w, /hero-mobile-800.webp 800w"
    sizes="100vw"
    type="image/webp"
  />
  <!-- Tablet: crop square -->
  <source
    media="(max-width: 1024px)"
    srcset="/hero-tablet-800.webp 800w, /hero-tablet-1200.webp 1200w"
    sizes="100vw"
    type="image/webp"
  />
  <!-- Desktop: landscape full image -->
  <source
    srcset="/hero-1200.webp 1200w, /hero-1600.webp 1600w, /hero-2400.webp 2400w"
    sizes="100vw"
    type="image/webp"
  />
  <!-- Fallback: phải có <img> cuối cùng -->
  <img
    src="/hero-1200.jpg"
    alt="Hero: Team họp mặt tại văn phòng"
    width="1600"
    height="900"
    loading="eager"
    fetchpriority="high"
  />
</picture>

<!-- Format negotiation: WebP với JPEG fallback -->
<picture>
  <!-- Browser support WebP? Dùng WebP -->
  <source
    srcset="/product.webp 1x, /product@2x.webp 2x"
    type="image/webp"
  />
  <!-- Browser support AVIF? Dùng AVIF (nếu thêm source này trước WebP) -->
  <source
    srcset="/product.avif 1x, /product@2x.avif 2x"
    type="image/avif"
  />
  <!-- Fallback JPEG -->
  <img
    src="/product.jpg"
    alt="Product"
    width="400"
    height="400"
  />
</picture>
```

**Thứ tự source quan trọng:** Browser chọn `<source>` đầu tiên match điều kiện. Đặt AVIF trước WebP trước JPEG.

## 6. Meta Viewport (🔴) — Responsive Foundation

### Cơ chế: Layout Viewport vs Visual Viewport

Browser có hai viewport khái niệm:

```
Layout Viewport: Kích thước mà CSS layout tính toán
Visual Viewport:  Phần thực sự user nhìn thấy trên màn hình
                  (có thể nhỏ hơn layout viewport khi zoom)

Mobile không có meta viewport:
  Layout viewport = 980px (browser assumption cho desktop sites)
  Device width = 390px (iPhone 14)
  → Browser scale down toàn bộ page → tiny text, unusable

Với meta viewport:
  Layout viewport = device width (390px)
  → CSS media queries dùng đúng giá trị thật
  → 1 CSS pixel = 1 logical pixel (trước khi tính DPR)
```

```html
<!--
  content="width=device-width": set layout viewport width = device width
  content="initial-scale=1": zoom 1:1 khi load, không zoom out
-->
<meta
  name="viewport"
  content="width=device-width, initial-scale=1"
/>

<!-- ❌ BUG: Tắt user zoom -->
<meta
  name="viewport"
  content="width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no"
/>
<!-- Vi phạm WCAG 1.4.4 Resize Text: user phải có thể zoom tới 200% -->
<!-- Nhiều apps làm vì "muốn control UX" → accessibility violation -->

<!-- ✅ FIX: Không dùng maximum-scale hoặc user-scalable=no -->
<meta
  name="viewport"
  content="width=device-width, initial-scale=1"
/>
```

### DPR (Device Pixel Ratio) và CSS pixels

```
Device pixel ratio: số physical pixels tương ứng với 1 CSS pixel

iPhone 14 Pro: DPR = 3
  → CSS: width: 100px = 300 physical pixels
  → img cần 300px để "sharp" trên màn hình này
  → Đó là lý do cần @2x, @3x images

window.devicePixelRatio → đọc DPR của device
```

## 7. fetchpriority (🟡)

### Cơ chế: Override Browser Priority Heuristics

Browser gán priority tự động dựa trên resource type và vị trí. `fetchpriority` cho phép developer hint override:

```html
<!-- LCP image: hint browser fetch với HIGH priority -->
<img
  src="/hero.jpg"
  fetchpriority="high"
  alt="Hero"
/>

<!-- Below-fold images: fetch với LOW priority để nhường bandwidth cho critical resources -->
<img
  src="/slide-2.jpg"
  fetchpriority="low"
  loading="lazy"
  alt="Slide 2"
/>

<!-- fetch() API: critical API call với high priority -->
<script>
  // Fetch user data với high priority
  fetch('/api/user', { priority: 'high' });

  // Prefetch non-critical data với low priority
  fetch('/api/recommendations', { priority: 'low' });
</script>

<!-- Preload hints với fetchpriority -->
<link
  rel="preload"
  href="/fonts/inter.woff2"
  as="font"
  fetchpriority="high"
  crossorigin
/>

<!-- Script không critical -->
<script
  src="/analytics.js"
  fetchpriority="low"
  async
></script>
```

## 8. 103 Early Hints (🟢)

### Cơ chế: Preload trước khi server xử lý xong

```
Không có Early Hints:
  Client → Request → Server processing (50-200ms) → 200 OK + HTML
  Client chỉ biết resources cần fetch sau khi nhận HTML

Với 103 Early Hints:
  Client → Request → Server gửi 103 ngay lập tức (trong khi vẫn processing)
                   → 103: Link: </styles.css>; rel=preload; as=style
                   → 103: Link: </fonts/inter.woff2>; rel=preload; as=font
                   Client bắt đầu fetch resources ngay
  → Server xong processing → 200 OK + HTML
  Resources có thể đã fetch xong hoặc gần xong khi HTML arrive
```

```http
HTTP/1.1 103 Early Hints
Link: </styles/main.css>; rel=preload; as=style
Link: </fonts/inter.woff2>; rel=preload; as=font; crossorigin
Link: </scripts/app.js>; rel=preload; as=script

HTTP/1.1 200 OK
Content-Type: text/html
...
```

Implement ở server level (Node.js/Express):

```javascript
// Express: gửi 103 trước khi process xong
app.get('/', async (req, res) => {
  // Gửi 103 ngay lập tức
  res.writeEarlyHints({
    link: [
      '</styles/main.css>; rel=preload; as=style',
      '</fonts/inter.woff2>; rel=preload; as=font; crossorigin',
    ],
  });

  // Process page (database queries, etc.)
  const data = await fetchPageData();

  // Gửi 200 với content
  res.send(renderPage(data));
});
```

## 5. Câu hỏi kiểm tra

**Q1:** Tại sao `defer` scripts execute theo order nhưng `async` thì không?

**A:** `defer` scripts được browser add vào một **deferred script list** theo thứ tự gặp trong HTML. Sau khi parser finish, browser execute từ list này tuần tự — giống như không có defer nhưng fetch được parallel. `async` không có list này — mỗi script execute ngay khi fetch xong, hoàn toàn độc lập. Script nhỏ hơn có thể fetch xong trước và execute trước dù nằm sau trong HTML.

---

**Q2:** Điều gì xảy ra nếu thêm `crossorigin` vào preload font nhưng CSS `@font-face` không có `crossorigin`?

**A:** Sẽ có **2 network requests** cho cùng font. Preload dispatch request với CORS mode (`crossorigin` attribute → credentials: 'same-origin' mode). Khi CSS parser sau đó gặp `@font-face url()` mà không có crossorigin attribute → dispatch request với **no-cors mode**. Hai requests này có khác nhau về CORS headers → cache key khác nhau → cache miss → browser fetch lần 2. Fix: `crossorigin` attribute phải nhất quán giữa preload hint và `@font-face` declaration. Fonts từ Google Fonts luôn cần `crossorigin` vì đó là cross-origin request.

---

**Q3:** Đoạn code này có vấn đề gì?

```html
<head>
  <link
    rel="preload"
    href="/js/app.js"
    as="script"
  />
  <link
    rel="preload"
    href="/images/hero.jpg"
    as="image"
  />
  <link
    rel="preload"
    href="/images/product-1.jpg"
    as="image"
  />
  <link
    rel="preload"
    href="/images/product-2.jpg"
    as="image"
  />
  <link
    rel="preload"
    href="/images/product-3.jpg"
    as="image"
  />
</head>
<body>
  <img
    src="/images/hero.jpg"
    fetchpriority="high"
    alt="Hero"
  />
  <!-- product images ở dưới fold -->
  <img
    src="/images/product-1.jpg"
    loading="lazy"
    alt="P1"
  />
  <img
    src="/images/product-2.jpg"
    loading="lazy"
    alt="P2"
  />
  <img
    src="/images/product-3.jpg"
    loading="lazy"
    alt="P3"
  />
</body>
```

**A:** Ba vấn đề: (1) Preload `app.js` nhưng không có `<script>` nào dùng nó trong 3 giây → unused preload warning, lãng phí bandwidth. (2) Preload 3 product images dù chúng có `loading="lazy"` — preload và lazy load mâu thuẫn nhau: preload fetch ngay với high priority, lazy load fetch muộn → preload thắng, nhưng vậy thì lazy load vô nghĩa, và fetch 3 large images ngay khi page load tranh bandwidth với hero image. (3) Hero image thiếu `width` và `height` → CLS. Fix: chỉ preload LCP image (hero), bỏ preload product images, thêm `width`/`height` cho hero, bỏ preload script nếu không có `<script>` tương ứng.

---

**Q4:** Tại sao `<picture>` override browser choice còn `<img srcset>` thì không?

**A:** Đây là **intentional design**. `<img srcset>` dùng `w` descriptors — browser biết pixel width của mỗi candidate và **tự chọn tối ưu** dựa trên viewport, DPR, và thậm chí network conditions (browser có thể chọn source nhỏ hơn nếu connection chậm). Developer không control final choice — đây là feature, không phải bug, vì browser có thông tin tốt hơn về runtime conditions. `<picture>` với `<source media="...">` là developer nói "ở viewport này, dùng image này" — browser **phải follow** điều kiện media query. Art direction cần developer control vì đây là design decision (crop khác nhau, composition khác nhau), không phải optimization decision.

---

**Q5:** Tại sao thêm `width` và `height` attribute vào `<img>` lại giúp CLS — cơ chế browser là gì?

**A:** Trước Chrome 88, `width`/`height` chỉ set intrinsic size — không reserve space trước khi image load. Từ Chrome 88+, browser áp dụng **default aspect-ratio style** từ `width` và `height` attributes: browser tự inject `img { aspect-ratio: attr(width) / attr(height) }` vào user-agent stylesheet. Vì aspect ratio được biết trước, CSS layout engine có thể tính height của image container ngay cả khi `width` là `100%` hay bất kỳ flexible value nào — không cần đợi image load. Content phía dưới được placed đúng vị trí ngay từ đầu, không bị pushed down khi image load, CLS = 0. Đây là lý do `width`/`height` giờ không còn là "fixed size" attribute mà là "aspect ratio hint".

## Câu hỏi ôn tập (NotebookLM)

---

**Câu 1:** Giải thích sự khác biệt giữa `defer` và `async` về execution timing và order guarantee — và tại sao `type="module"` về bản chất là defer?

**Đáp án:**

**`defer` — Guaranteed order, guaranteed timing:**

Browser fetch script ngay khi parser gặp `defer` attribute (parallel với HTML parsing, không block parser). Tuy nhiên, execution bị defer cho đến khi **HTML parsing hoàn tất**. Browser maintain một **deferred script queue** — scripts được add vào queue theo thứ tự gặp trong HTML. Sau khi parsing xong, browser execute scripts từ queue theo thứ tự, **trước khi** fire `DOMContentLoaded` event.

Order guarantee vì: execution theo queue (FIFO), không phải theo fetch completion time.

**`async` — No order, no timing guarantee:**

Browser fetch ngay khi gặp (parallel), nhưng execute **ngay khi fetch xong**, không quan tâm HTML parsing status. Nếu script nhỏ fetch xong trước khi parser finish, browser **dừng parser** và execute script, sau đó resume. Không có queue, không có order. Phù hợp cho scripts hoàn toàn độc lập (analytics, ads).

**`type="module"` là defer vì:**

ES Modules có **static import graph** — browser phải resolve tất cả `import` statements để build dependency graph trước khi execute bất kỳ module nào. Graph resolution cần biết toàn bộ dependency tree. Để resolve graph đúng, browser phải parse xong HTML (vì modules có thể được referenced từ nhiều nơi, và live bindings cần toàn bộ tree). Do đó, module execution được deferred cho đến sau HTML parsing — behavior giống hệt `defer`.

Ngoài ra, modules có **module map** — mỗi module URL được execute đúng một lần dù import nhiều lần. Behavior này cần deferred execution để đảm bảo module map được build đúng.

---

**Câu 2:** Mô tả chính xác thuật toán browser dùng để chọn source trong `srcset` với `w` descriptors và `sizes` attribute — và tại sao developer không control được final choice như `<picture>`?

**Đáp án:**

**Thuật toán selection (theo HTML spec):**

Bước 1 — Parse `sizes` attribute:

```
sizes="(max-width: 640px) 100vw, (max-width: 1024px) 50vw, 33vw"
```

Browser evaluate từng media condition từ trái sang phải. Lấy giá trị của condition đầu tiên match hiện tại viewport. Đây là **source size** — layout width mà image sẽ chiếm theo CSS.

Bước 2 — Tính **pixel density width**:

```
pixel density width = source size (CSS px) × window.devicePixelRatio
```

Ví dụ: source size = 600px, DPR = 2 → cần 1200 physical pixels.

Bước 3 — Chọn candidate từ `srcset`:

Browser parse tất cả candidates trong `srcset` (URL + w descriptor). Chọn **candidate nhỏ nhất có `w` ≥ pixel density width**. Nếu không có candidate nào đủ lớn → chọn candidate lớn nhất.

**Tại sao browser có thể override:**

Spec cho phép browser xem xét thêm **network conditions** khi chọn. Chrome có thể chọn source nhỏ hơn nếu connection chậm (Save-Data header, slow network), dù source đó không "đủ lớn" theo tính toán. Developer không biết network speed của user → browser có thông tin tốt hơn → nên để browser optimize.

**`<picture>` với `<source media>` khác:**

`media` attribute evaluate như CSS media query — **boolean match**. Nếu match → browser **phải** dùng source đó, không có optimization flexibility. Developer dictate hoàn toàn. Phù hợp cho art direction (khác crop, khác composition) vì đây là design decision không thể để browser override.

**Format negotiation với `type`:**

```html
<source
  type="image/avif"
  srcset="..."
/>
<source
  type="image/webp"
  srcset="..."
/>
<img src="fallback.jpg" />
```

Browser check `type` attribute — nếu browser **không support** format đó → skip source. Browser chọn source đầu tiên có `type` mà nó support. Đây là **capability detection**, không phải optimization → browser phải follow.

---

**Câu 3:** Giải thích tại sao `loading="lazy"` và `preload` trên cùng một image là mâu thuẫn, và mô tả chiến lược đúng để phân loại images thành các nhóm loading strategy khác nhau?

**Đáp án:**

**Tại sao mâu thuẫn:**

`preload` (`<link rel="preload" as="image">`) nói browser: "Fetch image này ngay bây giờ với priority cao, tôi sẽ cần nó sớm." Browser dispatch fetch request ngay lập tức.

`loading="lazy"` nói browser: "Đừng fetch image này cho đến khi nó gần viewport." Browser không dispatch fetch.

Khi cả hai cùng apply cho một image: preload thắng — image được fetch ngay. Nhưng vậy lazy load hoàn toàn vô nghĩa, và preload đã lãng phí bandwidth fetch image mà có thể không bao giờ được nhìn thấy (nếu user không scroll tới). Hai directives contradictory nhau.

**Chiến lược phân loại images:**

**Group 1 — LCP Image (above fold, highest priority):**

```html
<link
  rel="preload"
  href="/hero.jpg"
  as="image"
  fetchpriority="high"
/>
<img
  src="/hero.jpg"
  loading="eager"
  decoding="sync"
  fetchpriority="high"
/>
```

Rationale: LCP trực tiếp ảnh hưởng Core Web Vitals score. Cần preload để fetch sớm nhất có thể, trước khi HTML parser gặp `<img>`. `eager` và `sync` decode để browser render ngay.

**Group 2 — Above fold, non-LCP (trong viewport nhưng không phải largest element):**

```html
<img
  src="/logo.png"
  loading="eager"
  decoding="async"
  width="120"
  height="40"
/>
```

Rationale: Trong viewport → không lazy. Nhưng không phải LCP → không cần preload. `async` decode vì không phải element critical nhất.

**Group 3 — Below fold, likely to be seen (vài screens down):**

```html
<img
  src="/product.jpg"
  loading="lazy"
  decoding="async"
  width="400"
  height="400"
/>
```

Rationale: lazy với threshold mặc định (~1250px trước viewport ở Chrome) đủ để fetch trước khi user scroll tới. Không cần preload.

**Group 4 — Far below fold, uncertain (có thể không bao giờ thấy):**

```html
<img
  src="/slide-5.jpg"
  loading="lazy"
  decoding="async"
  fetchpriority="low"
  width="800"
  height="600"
/>
```

Rationale: lazy + low priority → browser càng trì hoãn càng tốt, nhường bandwidth cho critical resources.

**Rule of thumb:**

- `preload` chỉ cho LCP image (1-2 images tối đa mỗi page)
- `loading="eager"` cho above-fold images
- `loading="lazy"` cho below-fold images
- Không bao giờ `preload` + `loading="lazy"` cùng nhau
- Luôn có `width` và `height` cho tất cả images để tránh CLS
