## 1. Cơ chế thật — HTML là gì với browser parser?

Trước khi đi vào từng element, cần hiểu browser làm gì với HTML.

### Browser HTML Parser

Không phải browser đọc HTML từ trên xuống và ra DOM ngay. Pipeline thật:

```
HTML bytes (network)
      ↓
  Tokenizer (từng ký tự → token: StartTag, EndTag, Character, Comment)
      ↓
  Tree Construction Algorithm
  (token → gắn vào DOM tree theo "insertion mode")
      ↓
  DOM Tree (C++ objects trong heap của renderer process)
```

**Điểm quan trọng:** Tree construction **không phải là parsing**. Đây là 2 bước khác nhau. Tokenizer chạy trên luồng bytes. Tree construction nhận token và quyết định gắn vào đâu dựa trên **stack of open elements** và **insertion mode** hiện tại.

Đây là lý do HTML "tự sửa lỗi":

```html
<!-- Bạn viết: -->
<p>Text <div>Block</div></p>

<!-- Browser parse ra: -->
<p>Text</p>
<div>Block</div>
<p></p>
```

Khi parser gặp `<div>` bên trong `<p>`, nó biết `<p>` không thể chứa block-level element → tự đóng `<p>` trước, mở `<div>` → đây là behavior được **spec hóa**, không phải bug.

### Semantic HTML là gì với browser?

"Semantic" không phải khái niệm mơ hồ. Nó có hệ quả kỹ thuật cụ thể:

1. **Accessibility Tree** — Song song với DOM tree, browser build một **AX Tree** (Accessibility Tree). Screen reader không đọc DOM, nó đọc AX Tree. Element nào dùng thì AX Tree có node đó với role, name, state tương ứng.

2. **Implicit ARIA roles** — Mỗi HTML element có một implicit ARIA role. `<nav>` → `role="navigation"`. `<button>` → `role="button"`. Dùng đúng element = AX Tree đúng miễn phí.

3. **Default keyboard behavior** — `<button>` nhận focus, respond Enter/Space. `<a>` nhận focus, respond Enter. `<div>` thì không — bạn phải tự thêm bằng JS.

4. **Browser UI hints** — `<input type="email">` → mobile keyboard mở keyboard có @. `<input type="tel">` → numeric pad. Không phải magic — là browser spec.

```
DOM Tree                    AX Tree (Accessibility Tree)
──────────────             ────────────────────────────
<main>                      role=main
  <nav>                       role=navigation
    <a href="/">                role=link, name="Home"
      Home
    </a>
  </nav>
  <h1>Title</h1>              role=heading, level=1, name="Title"
  <button>Click</button>      role=button, name="Click"
  <div>Text</div>             role=generic (hoặc không có)
</main>
```

## 2. Document Outline — Heading Hierarchy (🔴)

### Cơ chế: Outline Algorithm

Browser có một **document outline algorithm** (từ HTML spec). Headings `h1`-`h6` tạo ra cấu trúc outline của document. Screen reader dùng outline này để navigate — giống như Table of Contents.

**Quan trọng:** HTML5 có "outline algorithm" dựa trên sectioning elements (`article`, `section`, v.v.) — nhưng **không một browser hay screen reader nào implement nó** (bị deprecated trong HTML spec 2022). Chỉ có heading-based outline thật sự hoạt động.

```
Document:
  h1: "Bài viết về Node.js"          ← Level 1
    h2: "Giới thiệu"                  ← Level 2
    h2: "Kiến trúc"                   ← Level 2
      h3: "Event Loop"                ← Level 3
      h3: "Worker Threads"            ← Level 3
    h2: "Kết luận"                    ← Level 2
```

### Bug phổ biến — Skip heading levels

```html
<!-- ❌ BUG: Skip từ h1 → h3, không có h2 -->
<h1>Trang chủ</h1>
<h3>Sản phẩm nổi bật</h3>
<!-- Screen reader bị confused -->

<!-- ✅ FIX: Hierarchy liên tục -->
<h1>Trang chủ</h1>
<h2>Sản phẩm nổi bật</h2>
<h3>Laptop Gaming</h3>
```

### Bug phổ biến 2 — Dùng heading cho styling, không phải structure

```html
<!-- ❌ BUG: Dùng h1 vì muốn chữ to, không phải vì nó là heading quan trọng nhất -->
<h1 class="section-title">Màu sắc yêu thích</h1>
<!-- Kết quả: screen reader announce "heading level 1: Màu sắc yêu thích" -->
<!-- Nhưng nội dung này không phải heading level 1 của document -->

<!-- ✅ FIX: Dùng đúng heading level, style bằng CSS -->
<h2 class="section-title">Màu sắc yêu thích</h2>

<!-- Hoặc nếu không phải heading thật sự -->
<p class="large-text">Màu sắc yêu thích</p>
```

**Rule thực tế:** Một page nên có đúng 1 `h1` (main title). Các section là `h2`. Sub-section là `h3`. Không bao giờ dùng heading level thấp hơn (h4, h5, h6) trừ khi thật sự cần nested structure đó.

## 3. Sectioning Elements (🔴)

### Các element và implicit ARIA role

| Element     | Implicit Role                                              | Dùng khi nào                                                          |
| ----------- | ---------------------------------------------------------- | --------------------------------------------------------------------- |
| `<main>`    | `main`                                                     | Content chính của page, chỉ 1 cái trên page                           |
| `<nav>`     | `navigation`                                               | Navigation links — header nav, sidebar nav                            |
| `<header>`  | `banner` (khi top-level), `group` (khi trong section)      | Page header hoặc section header                                       |
| `<footer>`  | `contentinfo` (khi top-level), `group` (khi trong section) | Page footer hoặc section footer                                       |
| `<aside>`   | `complementary`                                            | Content phụ, related nhưng không essential                            |
| `<article>` | `article`                                                  | Nội dung độc lập, có thể syndicate (blog post, comment, product card) |
| `<section>` | `region` (chỉ khi có accessible name)                      | Nhóm thematic content, cần label                                      |

### Cơ chế: "top-level" nghĩa là gì?

`<header>` và `<footer>` thay đổi role tùy vào context:

```html
<!-- <header> ở top-level → role="banner" (announced by screen reader) -->
<body>
  <header>
    <!-- role=banner -->
    <h1>My Site</h1>
  </header>

  <main>
    <article>
      <!-- <header> bên trong sectioning element → role=group (không announced) -->
      <header>
        <!-- role=group, không phải banner -->
        <h2>Article Title</h2>
        <time>2024-01-01</time>
      </header>
      <p>Content...</p>
    </article>
  </main>

  <footer>
    <!-- role=contentinfo -->
    <p>© 2024</p>
  </footer>
</body>
```

### article vs section — Confusion phổ biến nhất

```
article: Có nghĩa độc lập khi đứng một mình.
         Hỏi: "Nếu tách nội dung này ra khỏi page, nó vẫn có nghĩa không?"
         → Có: dùng article
         → Không: xem xét section hoặc div

section: Nhóm thematic content, cần một heading để label.
         Không có label → dùng div thay thế.
```

```html
<!-- ✅ article: Blog post list -->
<main>
  <h1>Blog</h1>

  <!-- Mỗi post là independent → article -->
  <article>
    <h2><a href="/post/1">Understanding V8 Engine</a></h2>
    <time datetime="2024-01-15">Jan 15, 2024</time>
    <p>V8 là JavaScript engine của Chrome...</p>
  </article>

  <article>
    <h2><a href="/post/2">Node.js Event Loop</a></h2>
    <time datetime="2024-01-20">Jan 20, 2024</time>
    <p>Event loop là trái tim của Node.js...</p>
  </article>
</main>

<!-- ✅ section: Các phần của một trang có cấu trúc -->
<main>
  <h1>Sản phẩm của chúng tôi</h1>

  <section aria-labelledby="laptop-heading">
    <h2 id="laptop-heading">Laptop</h2>
    <!-- products -->
  </section>

  <section aria-labelledby="phone-heading">
    <h2 id="phone-heading">Điện thoại</h2>
    <!-- products -->
  </section>
</main>

<!-- ❌ BUG: section không có label → dùng div thay -->
<section>
  <!-- role=region nhưng không có accessible name → vô nghĩa -->
  <p>Some content</p>
</section>

<!-- ✅ FIX -->
<div>
  <p>Some content</p>
</div>
```

## 4. Content Categories (🔴)

### Tại sao cần biết điều này?

Browser parser dùng content model để quyết định HTML nào hợp lệ. Khi bạn nest element sai, parser tự sửa — và sửa theo cách không ngờ tới, gây bug khó debug.

### Ba category chính

```
Metadata Content
  → Không render, affect document hoặc link với other docs
  → <head>: <title>, <meta>, <link>, <script>, <style>, <base>

Flow Content (gần như mọi thứ trong <body>)
  → Block-level: <div>, <p>, <section>, <article>, <h1>-<h6>, <ul>, <table>
  → Cả phrasing content

Phrasing Content (inline)
  → <span>, <a>, <strong>, <em>, <img>, <input>, <button>, <label>
  → Nằm trong flow content
  → Không được chứa flow content (với vài ngoại lệ)
```

### Bug parser tự sửa — cần biết để debug

```html
<!-- ❌ p không thể chứa div (div là flow, p chỉ accept phrasing) -->
<p>
  Text here
  <div>Block content</div>
  More text
</p>

<!-- Browser parse ra (kiểm tra DevTools): -->
<p>Text here</p>
<div>Block content</div>
<p>More text</p>
<!-- → Layout bị vỡ! -->

<!-- ❌ a có thể chứa block content (interactive exception) -->
<!-- nhưng không thể chứa interactive content khác -->
<a href="/page">
  <button>Click me</button>  <!-- ❌ interactive trong interactive -->
</a>

<!-- ✅ FIX: button trigger navigation bằng JS -->
<button onclick="window.location='/page'">Click me</button>
<!-- Hoặc dùng a bọc nội dung không interactive -->
<a href="/page">
  <div class="card">
    <img src="..." alt="...">
    <h3>Card Title</h3>
    <p>Description</p>
  </div>
</a>
```

### Ngoại lệ: `<a>` transparent content model

`<a>` có "transparent content model" — nó kế thừa content model của parent:

```html
<!-- a bên trong p (phrasing context) → a chỉ accept phrasing -->
<p>
  <a href="/page">Link text <span>inline</span></a> ✅
  <a href="/page"><div>Block</div></a> ❌
</p>

<!-- a bên trong div (flow context) → a accept flow content -->
<div>
  <a href="/page">
    <h2>Heading inside link</h2>
    ✅ (vì div context là flow)
    <p>Paragraph inside link</p>
    ✅
  </a>
</div>
```

## 5. Forms — Validation API & Accessibility (🔴)

### Cơ chế: Constraint Validation API

Browser có built-in form validation engine. Khi submit form, browser kiểm tra mỗi field theo **validity state object**:

```
HTMLInputElement.validity = {
  valueMissing: false,    // required nhưng empty
  typeMismatch: false,    // value không đúng type (email, url)
  patternMismatch: false, // không match pattern attribute
  tooLong: false,         // vượt maxlength
  tooShort: false,        // chưa đủ minlength
  rangeUnderflow: false,  // < min
  rangeOverflow: false,   // > max
  stepMismatch: false,    // không chia hết cho step
  badInput: false,        // browser không thể parse value
  customError: false,     // setCustomValidity() được gọi
  valid: true             // tất cả false → valid
}
```

### Cấu trúc form đúng chuẩn

```html
<form id="checkout-form" novalidate>  <!-- novalidate: tắt browser UI, dùng custom UI -->

  <!-- fieldset nhóm related fields -->
  <fieldset>
    <legend>Thông tin giao hàng</legend>

    <div class="field-group">
      <!-- label phải link với input qua htmlFor hoặc wrap -->
      <label for="full-name">
        Họ và tên
        <span aria-hidden="true">*</span>  <!-- dấu * decorative, không cần screen reader đọc -->
      </label>
      <input
        id="full-name"
        name="fullName"
        type="text"
        autocomplete="name"
        required
        aria-required="true"
        aria-describedby="full-name-error"
      />
      <!-- Error message liên kết với input qua aria-describedby -->
      <span
        id="full-name-error"
        role="alert"           <!-- screen reader announce khi content thay đổi -->
        aria-live="assertive"
        class="field-error"
        hidden                 <!-- ẩn khi không có error -->
      ></span>
    </div>

    <div class="field-group">
      <label for="email">Email</label>
      <input
        id="email"
        name="email"
        type="email"           <!-- mobile keyboard có @, browser validate format -->
        autocomplete="email"
        required
        aria-required="true"
        aria-describedby="email-error email-hint"  <!-- multiple descriptions -->
      />
      <span id="email-hint" class="field-hint">
        Chúng tôi sẽ gửi xác nhận đơn hàng tới email này
      </span>
      <span id="email-error" role="alert" hidden></span>
    </div>

  </fieldset>

  <fieldset>
    <legend>Phương thức thanh toán</legend>

    <!-- Radio group: fieldset+legend là pattern đúng -->
    <label>
      <input type="radio" name="payment" value="card" checked>
      Thẻ tín dụng/ghi nợ
    </label>
    <label>
      <input type="radio" name="payment" value="transfer">
      Chuyển khoản ngân hàng
    </label>
    <label>
      <input type="radio" name="payment" value="cod">
      Thanh toán khi nhận hàng
    </label>
  </fieldset>

  <button type="submit">Đặt hàng</button>
</form>
```

### Custom Validation với Constraint Validation API

```javascript
const form = document.getElementById('checkout-form');
const fullNameInput = document.getElementById('full-name');
const fullNameError = document.getElementById('full-name-error');
const emailInput = document.getElementById('email');
const emailError = document.getElementById('email-error');

// Validate một field, update UI
function validateField(input, errorEl) {
  // Browser đã check validity.valid dựa trên attributes
  if (!input.validity.valid) {
    let message;

    if (input.validity.valueMissing) {
      // Lấy label text để message có context
      const label = document.querySelector(`label[for="${input.id}"]`);
      message = `${label.textContent.trim()} là bắt buộc`;
    } else if (input.validity.typeMismatch) {
      message = 'Định dạng không hợp lệ';
    } else if (input.validity.tooShort) {
      message = `Tối thiểu ${input.minLength} ký tự`;
    } else if (input.validity.customError) {
      // setCustomValidity() đã được gọi trước đó
      message = input.validationMessage;
    }

    errorEl.textContent = message;
    errorEl.removeAttribute('hidden'); // show error
    input.setAttribute('aria-invalid', 'true');
    return false;
  }

  // Clear error
  errorEl.textContent = '';
  errorEl.setAttribute('hidden', '');
  input.removeAttribute('aria-invalid');
  return true;
}

// Async validation (check duplicate email)
async function validateEmailAsync(input, errorEl) {
  // Trước tiên check browser constraints
  if (!input.validity.valid) {
    return validateField(input, errorEl);
  }

  try {
    const res = await fetch(
      `/api/check-email?email=${encodeURIComponent(input.value)}`,
    );
    const { exists } = await res.json();

    if (exists) {
      // setCustomValidity đặt customError = true
      input.setCustomValidity('Email này đã được đăng ký');
      validateField(input, errorEl);
      return false;
    }

    // Clear custom error
    input.setCustomValidity('');
    return true;
  } catch {
    return true; // Network error → cho qua, server sẽ validate
  }
}

// Validate on blur (không validate realtime → UX annoying)
fullNameInput.addEventListener('blur', () =>
  validateField(fullNameInput, fullNameError),
);
emailInput.addEventListener('blur', () =>
  validateEmailAsync(emailInput, emailError),
);

// Validate on submit
form.addEventListener('submit', async (e) => {
  e.preventDefault();

  const nameValid = validateField(fullNameInput, fullNameError);
  const emailValid = await validateEmailAsync(emailInput, emailError);

  if (!nameValid || !emailValid) {
    // Focus vào field đầu tiên có lỗi
    form.querySelector('[aria-invalid="true"]')?.focus();
    return;
  }

  // Submit
  submitForm(new FormData(form));
});
```

## 6. Tables — Accessible Tables (🔴)

### Cơ chế: Screen reader đọc table như thế nào

Screen reader không đọc table theo thứ tự DOM. Nó build **internal table model** và cho phép user navigate: "cell A3, column: Price, row: MacBook Pro". Để điều này hoạt động, table phải có đúng semantic structure.

```html
<!-- ✅ Table đầy đủ semantic -->
<table>
  <!-- caption mô tả table, announced đầu tiên khi focus vào table -->
  <caption>
    Bảng giá sản phẩm tháng 1/2024
  </caption>

  <thead>
    <tr>
      <!-- scope="col": header này mô tả cả column dưới nó -->
      <th scope="col">Sản phẩm</th>
      <th scope="col">Giá</th>
      <th scope="col">Tình trạng</th>
    </tr>
  </thead>

  <tbody>
    <tr>
      <!-- scope="row": header này mô tả cả row bên phải nó -->
      <th scope="row">MacBook Pro M3</th>
      <td>52.000.000đ</td>
      <td>Còn hàng</td>
    </tr>
    <tr>
      <th scope="row">iPad Air M2</th>
      <td>18.000.000đ</td>
      <td>Hết hàng</td>
    </tr>
  </tbody>

  <tfoot>
    <tr>
      <td colspan="2">Tổng sản phẩm: 2</td>
      <td></td>
    </tr>
  </tfoot>
</table>
```

### Complex tables — headers attribute

Khi table có spanning cells, `scope` không đủ. Dùng `id` + `headers`:

```html
<table>
  <caption>
    Doanh thu theo khu vực và quý
  </caption>
  <thead>
    <tr>
      <td></td>
      <!-- ô trống góc trên trái -->
      <!-- colspan header cần id -->
      <th
        id="north"
        colspan="2"
        scope="colgroup"
      >
        Miền Bắc
      </th>
      <th
        id="south"
        colspan="2"
        scope="colgroup"
      >
        Miền Nam
      </th>
    </tr>
    <tr>
      <td></td>
      <th
        id="q1-north"
        scope="col"
      >
        Q1
      </th>
      <th
        id="q2-north"
        scope="col"
      >
        Q2
      </th>
      <th
        id="q1-south"
        scope="col"
      >
        Q1
      </th>
      <th
        id="q2-south"
        scope="col"
      >
        Q2
      </th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th
        id="laptop"
        scope="row"
      >
        Laptop
      </th>
      <!-- headers liệt kê TẤT CẢ th áp dụng cho cell này -->
      <td headers="north q1-north laptop">120</td>
      <td headers="north q2-north laptop">145</td>
      <td headers="south q1-south laptop">89</td>
      <td headers="south q2-south laptop">102</td>
    </tr>
  </tbody>
</table>
```

## 7. `<details>` / `<summary>` — Native Disclosure Widget (🟡)

### Cơ chế: Browser-native state machine

`<details>` là một trong số ít HTML elements có **internal state** được browser quản lý. Không cần JS để toggle.

```
details element có attribute "open":
  - không có "open" → collapsed (content hidden)
  - có "open" → expanded (content visible)

Browser handle:
1. Click <summary> → toggle "open" attribute
2. Fire "toggle" event trên <details>
3. Re-render content visibility
```

```html
<!-- Basic usage -->
<details>
  <summary>Chi tiết đơn hàng</summary>
  <!-- Content ẩn cho đến khi expand -->
  <div class="details-content">
    <p>Mã đơn: #12345</p>
    <p>Ngày đặt: 15/01/2024</p>
    <table>
      ...
    </table>
  </div>
</details>

<!-- Mặc định mở -->
<details open>
  <summary>Thông tin quan trọng (hiển thị sẵn)</summary>
  <p>Nội dung...</p>
</details>
```

### JS Integration — Khi cần custom behavior

```javascript
const details = document.querySelector('#order-details');

// Event "toggle" fire sau khi state thay đổi
details.addEventListener('toggle', () => {
  // details.open là boolean phản ánh attribute
  if (details.open) {
    // Lazy load content khi mở lần đầu
    loadOrderDetails(details.dataset.orderId);
    // Track analytics
    analytics.track('details_expanded', { id: details.dataset.orderId });
  }
});

// Programmatic control
function expandAll() {
  document.querySelectorAll('details').forEach((d) => (d.open = true));
}

function collapseAll() {
  document.querySelectorAll('details').forEach((d) => (d.open = false));
}
```

## 8. `<dialog>` — Native Modal (🟡)

### Cơ chế: Top Layer

Đây là cơ chế **cực kỳ quan trọng** mà nhiều người không biết.

Browser có một **top layer** — một layer nằm trên tất cả stacking contexts. Element trong top layer luôn render trên mọi thứ khác, kể cả `z-index: 9999999`. Không cần z-index hack.

Elements vào top layer: `<dialog>` (khi `.showModal()`), Fullscreen elements, Popover elements.

```
Render stack (từ dưới lên):
├── Normal stacking contexts (z-index 1, 2, 100, 9999...)
├── Position: fixed elements
└── TOP LAYER ← dialog, popover, fullscreen nằm ở đây
               (Luôn on top, bất kể z-index)
```

```html
<dialog
  id="confirm-dialog"
  aria-labelledby="dialog-title"
>
  <h2 id="dialog-title">Xác nhận xóa</h2>
  <p>
    Bạn có chắc muốn xóa sản phẩm này không? Hành động này không thể hoàn tác.
  </p>
  <div class="dialog-actions">
    <button id="cancel-btn">Hủy</button>
    <button
      id="confirm-btn"
      class="btn-danger"
    >
      Xóa
    </button>
  </div>
</dialog>
```

```javascript
const dialog = document.getElementById('confirm-dialog');
const cancelBtn = document.getElementById('cancel-btn');
const confirmBtn = document.getElementById('confirm-btn');

// showModal() → vào top layer + tạo backdrop + focus trap + Escape key
// show() → hiện như non-modal (không có backdrop, không trap focus)
function openConfirmDialog(onConfirm) {
  dialog.showModal();
  // Browser tự:
  // 1. Add vào top layer
  // 2. Render ::backdrop pseudo-element
  // 3. Trap focus bên trong dialog
  // 4. Listen Escape key để close
  // 5. Set aria-modal="true" internally

  // Focus element đầu tiên có ý nghĩa (không phải destructive action)
  cancelBtn.focus();

  // One-time listeners để tránh memory leak
  const handleConfirm = () => {
    onConfirm();
    dialog.close();
    cleanup();
  };

  const handleCancel = () => {
    dialog.close('cancelled'); // close() nhận return value
    cleanup();
  };

  const cleanup = () => {
    confirmBtn.removeEventListener('click', handleConfirm);
    cancelBtn.removeEventListener('click', handleCancel);
  };

  confirmBtn.addEventListener('click', handleConfirm, { once: true });
  cancelBtn.addEventListener('click', handleCancel, { once: true });
}

// "close" event fire khi dialog đóng (bằng Escape, close(), hay form[method=dialog])
dialog.addEventListener('close', () => {
  console.log('Dialog closed with:', dialog.returnValue);
  // Restore focus về trigger element!
  document.getElementById('delete-trigger')?.focus();
});

// Form bên trong dialog với method="dialog" → submit đóng dialog
// <form method="dialog">
//   <button value="confirmed">Xác nhận</button>
// </form>
// → dialog.returnValue = "confirmed"
```

## 10. Template Element (🟢)

### Cơ chế: Inert DOM

`<template>` chứa **inert DOM** — DOM được parse nhưng **không render, không execute, không load resources**. Nó nằm trong một `DocumentFragment` riêng gọi là **template content**.

```
<template> parse:
  ├── Template element tồn tại trong main DOM
  └── template.content → DocumentFragment (separate, inert)
      ├── Không attach vào main document
      ├── Scripts bên trong không run
      ├── Images không load
      └── CSS không apply
```

```html
<!-- Template định nghĩa một lần, dùng nhiều lần -->
<template id="product-card-template">
  <article class="product-card">
    <img
      class="product-image"
      alt=""
    />
    <div class="product-info">
      <h3 class="product-name"></h3>
      <p class="product-price"></p>
      <button
        class="add-to-cart"
        type="button"
      >
        Thêm vào giỏ
      </button>
    </div>
  </article>
</template>
```

```javascript
function createProductCard(product) {
  const template = document.getElementById('product-card-template');

  // cloneNode(true) clone toàn bộ template content
  // Phải clone mỗi lần — không thể reuse cùng fragment
  const clone = template.content.cloneNode(true);

  // clone là DocumentFragment, querySelector hoạt động trên nó
  clone.querySelector('.product-image').src = product.imageUrl;
  clone.querySelector('.product-image').alt = product.name;
  clone.querySelector('.product-name').textContent = product.name;
  clone.querySelector('.product-price').textContent = new Intl.NumberFormat(
    'vi-VN',
    { style: 'currency', currency: 'VND' },
  ).format(product.price);

  const addBtn = clone.querySelector('.add-to-cart');
  addBtn.addEventListener('click', () => addToCart(product.id));

  return clone; // DocumentFragment
}

// Render danh sách sản phẩm
function renderProducts(products, container) {
  // Dùng DocumentFragment để batch DOM insertion
  const fragment = document.createDocumentFragment();
  products.forEach((product) => {
    fragment.appendChild(createProductCard(product));
  });

  // Một lần DOM mutation duy nhất
  container.appendChild(fragment);
}
```

### Template vs innerHTML

```javascript
// innerHTML: parse HTML string → tạo DOM
// Vấn đề: XSS nếu content từ user
// Vấn đề: Mỗi lần parse là tốn kém
container.innerHTML = `<div class="card">${userContent}</div>`; // ❌ XSS risk

// Template: clone pre-parsed DOM
// An toàn hơn (không eval HTML string), nhanh hơn (đã parse sẵn)
```

## 11. Slot Element — Web Components (🟢)

### Cơ chế: Shadow DOM Composition

Slot là cơ chế để Web Components **compose** content từ light DOM (bên ngoài) vào shadow DOM (bên trong component). Đây là **distributed nodes** — content không move, chỉ được "projected" vào slot.

```
Light DOM (outside component):
<my-card>
  <span slot="title">Product Name</span>     ← slotted content
  <span slot="price">500,000đ</span>         ← slotted content
  <img src="product.jpg" slot="image">       ← slotted content
</my-card>

Shadow DOM (inside component):
<div class="card">
  <div class="card-header">
    <slot name="image"></slot>    ← receives <img slot="image">
  </div>
  <div class="card-body">
    <slot name="title"></slot>    ← receives <span slot="title">
    <slot name="price"></slot>    ← receives <span slot="price">
  </div>
  <slot></slot>                   ← default slot: receives unslotted children
</div>
```

```javascript
class ProductCard extends HTMLElement {
  constructor() {
    super();
    // attachShadow tạo shadow root — encapsulated DOM
    const shadow = this.attachShadow({ mode: 'open' });
    // mode: 'open' → shadow root accessible từ ngoài qua element.shadowRoot
    // mode: 'closed' → không accessible (dùng cho third-party, không phải internal)

    // Template + slot
    const template = document.getElementById('product-card-template');
    shadow.appendChild(template.content.cloneNode(true));

    // Styles bên trong shadow DOM không leak ra ngoài
    const style = document.createElement('style');
    style.textContent = `
      .card { border: 1px solid #ddd; border-radius: 8px; padding: 16px; }
      /* CSS này chỉ apply bên trong component, không affect ngoài */
    `;
    shadow.prepend(style);
  }

  connectedCallback() {
    // Element added to DOM
    this.shadowRoot
      .querySelector('slot')
      .addEventListener('slotchange', (e) => {
        // Khi slotted content thay đổi
        const slottedNodes = e.target.assignedNodes({ flatten: true });
        console.log('Slotted nodes:', slottedNodes);
      });
  }
}

customElements.define('product-card', ProductCard);
```

```html
<!-- Usage -->
<product-card>
  <img
    slot="image"
    src="/products/laptop.jpg"
    alt="Laptop Pro"
  />
  <span slot="title">Laptop Pro M3</span>
  <span slot="price">52.000.000đ</span>
  <!-- Unslotted content → vào default slot -->
  <p>Bảo hành 12 tháng</p>
</product-card>
```

```jsx
// React controlled form với Constraint Validation API
function CheckoutForm() {
  const [errors, setErrors] = useState({});

  const handleSubmit = (e) => {
    e.preventDefault();
    const form = e.target;

    // checkValidity() trả về false nếu bất kỳ field nào invalid

      const newErrors = {};
      Array.from(form.elements).forEach((el) => {
        if (!el.validity.valid) {
          newErrors[el.name] = el.validationMessage;
        }
      });
      setErrors(newErrors);
      return;
    }

    // Submit
  };

  return (

      onSubmit={handleSubmit}
      noValidate
    >
      <div>
        <label htmlFor='email'>Email</label>
        <input
          id='email'
          name='email'
          type='email'
          required
          aria-invalid={!!errors.email}
          aria-describedby={errors.email ? 'email-error' : undefined}
        />
        {errors.email && (

            id='email-error'

          </span>
```

## 5. Câu hỏi kiểm tra

**Q1:** Tại sao `<section>` không có accessible name thì tốt hơn nên dùng `<div>`?

**A:** `<section>` có implicit ARIA role là `region` — nhưng `region` role chỉ được exposed trong AX Tree khi element có accessible name (qua `aria-label` hoặc `aria-labelledby`). Nếu không có name, nhiều browsers và screen readers vẫn expose nó như `region` không có label, khiến screen reader announce "region" không có context — gây confusion hơn là có ích. `<div>` không có semantic role, không được announced — tốt hơn khi không có gì để nói.

---

**Q2:** Điều gì xảy ra bên trong browser khi `dialog.showModal()` được gọi?

**A:**

1. Browser add dialog element vào **top layer** — một layer đặc biệt render trên mọi stacking context, kể cả `z-index: 2147483647`.
2. Browser render `::backdrop` pseudo-element phía sau dialog trong top layer.
3. Browser kích hoạt **focus trap**: chỉ elements bên trong dialog có thể nhận focus. Tab key wrap around bên trong dialog. Elements bên ngoài nhận `inert` behavior (không thể focus, không thể interact).
4. Browser listen Escape keydown → nếu detect → gọi `.close()` automatically.
5. Browser set `open` attribute trên `<dialog>` element.
6. Sự khác biệt với `dialog.show()` (non-modal): không vào top layer một cách exclusive, không có backdrop, không trap focus, không Escape behavior.

---

**Q3:** Đoạn code này có vấn đề gì?

```html
<form>
  <p>
    Họ tên:
    <input
      type="text"
      name="fullName"
    />
  </p>
  <p>
    Email:
    <input
      type="email"
      name="email"
    />
  </p>
  <button>Gửi</button>
</form>
```

**A:** Ba vấn đề:

1. **Không có `<label>`** — screen reader không biết input nào là "Họ tên", input nào là "Email". Placeholder không thay thế label được (biến mất khi gõ, low contrast, screen reader không luôn announce).
2. **Không có `id` để link label** — ngay cả khi thêm label cũng cần `for`/`id` pair.
3. **Dùng `<p>` để bọc form fields** — không có vấn đề kỹ thuật lớn nhưng sai semantic. Nên dùng `<div>` hoặc dedicated form group pattern.

Fix:

```html
<form>
  <div class="field">
    <label for="full-name">Họ tên</label>
    <input
      id="full-name"
      type="text"
      name="fullName"
      required
    />
  </div>
  <div class="field">
    <label for="email">Email</label>
    <input
      id="email"
      type="email"
      name="email"
      required
    />
  </div>
  <button type="submit">Gửi</button>
</form>
```

---

**Q4:** Tại sao `popover` cần top layer nhưng tooltip truyền thống (dùng `position: absolute + z-index`) hay bị vấn đề overflow?

**A:** `position: absolute` bị **clipped bởi `overflow: hidden/clip` của ancestor**. Nếu tooltip nằm trong một container có `overflow: hidden`, tooltip bị cắt — dù `z-index` bao nhiêu cũng không giúp được. Top layer không phải là một stacking context trong document flow — nó là separate rendering layer của browser, không bị ảnh hưởng bởi bất kỳ CSS property nào của ancestors. Đây là lý do tại sao `popover` và `dialog` với top layer giải quyết triệt để vấn đề tooltip bị cắt mà `z-index` không giải quyết được.

---

**Q5:** Tại sao `<template>` được gọi là "inert" — và điều đó có nghĩa gì về performance?

**A:** "Inert" nghĩa là nội dung bên trong `<template>` được parse thành DOM (AST đã build) nhưng không được **connected** vào main document. Hệ quả kỹ thuật:

- `<script>` bên trong không execute.
- `<img>` không fire HTTP request để load.
- CSS không apply (không có style recalculation).
- Không tham gia vào layout/paint.

Về performance: Template là pre-parsed — structure đã là DOM object trong memory. Khi `.cloneNode(true)`, browser copy C++ objects (rẻ hơn nhiều so với parse HTML string). So với `innerHTML`: innerHTML phải tokenize → parse → build DOM mỗi lần gọi. Template chỉ phải clone đã-parsed structure. Trong trường hợp render nhiều items (virtual list, product cards), template significantly faster.

## Câu hỏi ôn tập (NotebookLM)

---

**Câu 1:** Tại sao `<header>` có thể có hai ARIA role khác nhau tùy context, và browser xác định role nào bằng cơ chế gì?

**Đáp án:**

`<header>` có **implicit ARIA role thay đổi theo vị trí** trong DOM:

- **`role="banner"`** khi `<header>` là **top-level sectioning content** — nghĩa là không nằm bên trong `<article>`, `<aside>`, `<main>`, `<nav>`, `<section>` nào. Đây là page-level header.
- **`role="group"`** khi `<header>` nằm **bên trong** một sectioning element (article, aside, main, nav, section). Đây là section-level header.

Cơ chế browser xác định: Khi build **AX Tree** (Accessibility Tree), browser với mỗi `<header>` element sẽ walk up DOM tree kiểm tra ancestors. Nếu gặp sectioning element trước khi gặp `<body>` → role là `group`. Nếu đến `<body>` mà không gặp sectioning element → role là `banner`.

`banner` role được screen reader announce và liệt kê trong landmark navigation. `group` role không được liệt kê — chỉ là một container vô danh.

Hệ quả thực tế: Nếu cả page chỉ có một `<header>` → nó tự động là `banner`. Nhưng nếu mỗi `<article>` trong blog post list đều có `<header>` bên trong → đó là `group`, không phải `banner` — đây là behavior đúng và mong muốn.

**Câu 2:** Khi `dialog.showModal()` được gọi, điều gì xảy ra với focus và tại sao behavior này không thể thực hiện bằng CSS/JS thuần?

**Đáp án:**

Khi `dialog.showModal()` được gọi, browser thực hiện **focus management tự động**:

1. **Focus trap được kích hoạt**: Browser thêm dialog vào **top layer** và đồng thời mark tất cả elements **bên ngoài dialog** là không thể focus (tương đương `inert`). Tab key chỉ cycle qua elements bên trong dialog. Shift+Tab cycle ngược chiều. Khi reach cuối list → wrap về đầu.

2. **Focus move vào dialog**: Browser focus vào element đầu tiên có thể focus bên trong dialog (hoặc chính dialog element nếu không có focusable children).

3. **Focus restoration**: Khi dialog đóng bằng `dialog.close()` hay Escape, browser **không** tự restore focus về trigger element — đây là trách nhiệm của developer. Phải tự save reference và restore.

**Tại sao không thể replicate bằng JS thuần:**

Focus trap thuần JS dùng `addEventListener('keydown')` để intercept Tab — nhưng vẫn có edge cases: screen reader virtual cursor (không dùng Tab), `aria-activedescendant` pattern, browser native focusable elements (`<video>`, `<embed>`). Native dialog focus trap là browser-level, không phải JS event — nên handle tất cả edge cases, bao gồm cả accessibility tree navigation mà JS events không capture được.

Top layer là **render architecture**, không phải CSS concept. Không thể tạo top layer behavior bằng `z-index` vì top layer bypass tất cả stacking contexts — kể cả `position: fixed` trong một `transform` ancestor (transform tạo stacking context, fix `position: fixed` vào đó thay vì viewport). Top layer không có vấn đề này.

**Câu 3:** Giải thích Constraint Validation API — browser quyết định field valid hay không bằng cơ chế gì, và tại sao nên dùng `novalidate` + custom validation UI thay vì để browser tự handle?

**Đáp án:**

**Cơ chế Constraint Validation:**

Browser build một **validity state object** cho mỗi form control element (`<input>`, `<textarea>`, `<select>`). Object này là `ValidityState` với các boolean properties: `valueMissing`, `typeMismatch`, `patternMismatch`, `tooLong`, `tooShort`, `rangeUnderflow`, `rangeOverflow`, `stepMismatch`, `badInput`, `customError`, và `valid`.

Các properties này được update **realtime** khi value thay đổi, nhưng browser chỉ **show UI** khi form submit (hoặc `reportValidity()` được gọi). Browser check theo thứ tự: nếu `required` và empty → `valueMissing = true` → không check tiếp các constraint khác.

`checkValidity()` trả về boolean và fire `invalid` event nếu invalid. `reportValidity()` làm như `checkValidity()` nhưng thêm hiện browser validation UI.

**Tại sao dùng `novalidate` + custom UI:**

1. **Inconsistent UI**: Browser validation UI hoàn toàn khác nhau giữa Chrome, Firefox, Safari — không style được bằng CSS. Không match design system.

2. **UX không kiểm soát**: Browser hiện error ngay lập tức khi submit — không có "validate on blur" pattern (validate khi rời field, không phải ngay khi gõ). Gây frustration khi user chưa kịp điền xong.

3. **Không support async validation**: Không thể check duplicate email với server bằng browser native validation. Constraint Validation API có `setCustomValidity()` nhưng phải gọi trước khi browser check.

4. **Vị trí error message**: Browser hiện error tooltip tại vị trí field — không thể hiện error trong một summary box ở đầu form, không thể scroll đến error, không thể persist error sau khi user dismiss tooltip.

Pattern đúng: dùng `novalidate` trên `<form>` để tắt browser UI, nhưng **vẫn dùng** HTML attributes (`required`, `type`, `pattern`, `min`, `max`) và `ValidityState` API để check — browser vẫn tính toán validity state, mình chỉ tắt phần hiển thị UI và tự build UI tốt hơn.
