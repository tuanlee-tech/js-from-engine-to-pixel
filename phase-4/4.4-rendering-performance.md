## 1. Cơ chế thật

### 1.1 — Virtual Scrolling — Tại sao DOM size giết performance

Trước khi học virtual scrolling, cần hiểu **tại sao** render 10,000 DOM nodes lại chậm.

```
Render pipeline với DOM lớn:

10,000 <div> elements trong DOM:

1. Style calculation:
   Browser phải tính computed style cho MỖI element
   CSS selector matching: kiểm tra mọi rule trên mọi element
   10,000 elements × N CSS rules = O(n×m) work
   → Style recalc: ~50-200ms

2. Layout:
   Mỗi element có position, size phụ thuộc lẫn nhau
   Thay đổi 1 element → reflow cascades
   10,000 elements: layout tree traversal rất tốn
   → Layout: ~30-100ms

3. Paint:
   Rasterize pixels cho 10,000 elements
   Dù chỉ thấy 20 elements trong viewport!
   → Paint: ~20-80ms

4. Memory:
   Mỗi DOM node = ~1-2KB overhead
   10,000 nodes = 10-20MB DOM overhead
   + Event listeners, data attributes...

5. Interaction cost:
   querySelector, getBoundingClientRect → scan toàn bộ tree
   React reconciliation → diff 10,000 nodes mỗi render
   → Any interaction feels sluggish
```

**Virtual scrolling — render chỉ cái user nhìn thấy:**

```
Physical DOM (không virtual):
┌─────────────────────────────┐
│  Item 1  ← visible          │
│  Item 2  ← visible          │
│  Item 3  ← visible          │  Viewport
│  Item 4  ← visible          │
│  Item 5  ← visible          │
└─────────────────────────────┘
│  Item 6  ← in DOM, not seen │
│  Item 7  ← in DOM, not seen │
│  ...                        │
│  Item 9999 ← in DOM!        │
│  Item 10000 ← in DOM!       │
→ 10,000 DOM nodes


Virtual Scrolling:
┌─────────────────────────────┐
│  [spacer: height = 0*50px]  │ ← Empty div, giữ scroll position
│  Item 1  ← visible          │
│  Item 2  ← visible          │
│  Item 3  ← visible          │  Viewport (50px/item, 5 visible)
│  Item 4  ← visible          │
│  Item 5  ← visible          │
│  [buffer Item 6]            │ ← Render thêm vài items ngoài viewport
│  [buffer Item 7]            │   để tránh flicker khi scroll
│  [spacer: height = 9993*50px] ← Empty div, giữ đúng scroll height
└─────────────────────────────┘
→ ~10-15 DOM nodes bất kể list dài bao nhiêu!
```

**Cơ chế tính toán:**

```
Scroll container: height = 400px
Item height: 50px (fixed — dễ tính)
Total items: 10,000
Total scrollable height: 10,000 × 50 = 500,000px

User scrolls to position scrollTop = 5000px:

First visible index = Math.floor(scrollTop / itemHeight)
  = Math.floor(5000 / 50) = 100

Last visible index = Math.ceil((scrollTop + containerHeight) / itemHeight)
  = Math.ceil((5000 + 400) / 50) = 109

Render items 100 → 109 (+ buffer: 95 → 114 for smooth scroll)
Spacer top: 100 × 50 = 5000px (đẩy items xuống đúng vị trí)
Spacer bottom: (10000 - 114) × 50 = 494,300px (giữ scroll height)

DOM: spacer-top + 20 items + spacer-bottom
→ 22 DOM nodes thay vì 10,000
```

**Variable height items — phức tạp hơn nhiều:**

```
Fixed height: O(1) để tính index từ scrollTop
Variable height: Không có công thức đơn giản

Approaches:

1. Estimated height + correction:
   Assume average height (ví dụ: 80px)
   Sau khi render, đo actual height
   Cập nhật offset map: [0, 75, 160, 240, ...]
   → Scroll position có thể "jump" khi estimates sai

2. Binary search với offset map:
   Duy trì cumulative heights array: [0, 75, 160, 240, ...]
   Binary search để tìm first visible index từ scrollTop
   → O(log n) thay vì O(1), nhưng vẫn rất nhanh

3. ResizeObserver per item:
   Watch mỗi rendered item để detect size changes
   Update offset map dynamically
   → Accurate nhưng overhead cao hơn
```

### 1.2 — IntersectionObserver — Cơ chế native lazy loading

Trước IntersectionObserver, lazy loading phải dùng scroll event + `getBoundingClientRect()` — pattern cực kỳ tốn kém.

```
Old pattern (KHÔNG dùng nữa):

window.addEventListener('scroll', () => {
  images.forEach(img => {
    const rect = img.getBoundingClientRect(); // FORCE REFLOW!
    if (rect.top < window.innerHeight) {
      img.src = img.dataset.src;
    }
  });
});

Vấn đề:
  - scroll event: fire ~60 lần/giây khi cuộn
  - getBoundingClientRect(): mỗi lần gọi force synchronous layout
  - 60 calls/s × N images × forced reflow = main thread paralysis
```

**IntersectionObserver — tại sao nhanh hơn:**

```
IntersectionObserver chạy NGOÀI main thread (một phần):

Browser có Intersection Detection module chạy trong:
  - Compositor thread (không phải main thread!)
  - Hoặc background timer

Không force reflow, không fire liên tục.
Chỉ notify khi intersection THAY ĐỔI (element vào/ra viewport).

Cơ chế:
  1. JS gọi observer.observe(element) trên main thread
  2. Browser register element trong intersection tracking system
  3. Browser check intersections sau mỗi scroll/resize event,
     nhưng ASYNCHRONOUSLY — không block main thread
  4. Khi element cross threshold: queue IntersectionObserver callback
     vào microtask/task queue
  5. Callback chạy trên main thread khi idle
```

**Threshold và rootMargin — fine-tuning:**

```javascript
const observer = new IntersectionObserver(callback, {
  // root: null = viewport (default)
  // root: element = scroll container
  root: null,

  // rootMargin: expand/shrink root bounds
  // "200px 0px" = trigger 200px TRƯỚC khi element vào viewport
  // Giúp pre-load trước khi user thực sự thấy
  rootMargin: '200px 0px 200px 0px', // top right bottom left

  // threshold: % visible để trigger callback
  // 0 = ngay khi 1 pixel visible
  // 0.5 = khi 50% visible
  // [0, 0.25, 0.5, 0.75, 1] = trigger ở nhiều mốc
  threshold: 0,
});

// rootMargin: '200px' cho lazy loading
// → Image bắt đầu load khi còn 200px trước khi vào viewport
// → User scroll đến, ảnh đã load xong → không thấy placeholder
```

### 1.3 — Skeleton Screens — Performance Perception

Đây không chỉ là UX decision — là **performance perception engineering**.

```
Human perception của loading:

< 100ms: Instant — không nhận ra có delay
100-300ms: Hệ thống đang phản hồi — OK
300ms-1s: Đang xử lý — user bắt đầu thấy chờ
1-3s: Task nặng — user thất vọng, cần progress indicator
> 3s: Context switch — user làm việc khác

Spinner:
  "System đang làm gì đó, không biết bao lâu"
  → Tăng perceived waiting time
  → Brain không có gì để "nhìn vào" → thời gian trôi chậm hơn

Skeleton screen:
  "Content sắp xuất hiện ở đây"
  → Não nhận ra layout sắp có → anticipation thay vì chờ đợi
  → Trao cho brain một "bản đồ" để chuẩn bị đọc
  → Perceived wait time giảm 20-40% (theo các nghiên cứu UX)

Progressive content loading:
  Hiển thị data ngay khi có (partial data first)
  → Tốt nhất cho perceived performance
  → Cần streaming data từ server (Streaming SSR)
```

**Khi nào dùng gì:**

```
Spinner:
  ✅ Action ngắn (< 300ms): button loading state
  ✅ Kết quả không biết layout trước (search, error state)
  ✅ Overlay blocking action (form submission)
  ❌ Full page loading — tệ cho perceived performance

Skeleton:
  ✅ Content có layout có thể predict (feed, card list, profile)
  ✅ Loading time 300ms - 3s
  ✅ Multiple independent content blocks loading
  ❌ Dynamic/unknown layouts
  ❌ < 200ms loading (flash of skeleton là anti-pattern)

Optimistic UI:
  ✅ Actions có high success rate (like, follow, send message)
  Show result ngay lập tức → rollback nếu fail
  → Fastest perceived performance
  ❌ Destructive actions (delete, payment)
```

### 1.4 — CSR vs SSR vs SSG — Trade-offs Thật

Không có silver bullet. Mỗi approach phù hợp với use case khác nhau.

```
CSR (Client-Side Rendering):

Server gửi:           Browser nhận:         Sau JS execute:
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ <html>       │      │ Blank page   │      │ Full page    │
│ <body>       │  →   │              │  →   │ with content │
│   <div id=   │      │ (spinner)    │      │              │
│    "root"/>  │      │              │      │              │
│ <script>     │      └──────────────┘      └──────────────┘
└──────────────┘
t=0: HTML (empty)     t=0: Blank             t=2000ms: Content
t=500: JS bundle      t=500: JS loading
t=1000: JS parse      t=1000: JS parsing
t=1500: API fetch     t=1500: Waiting API
t=2000: Render        t=2000: Content

Problems:
  - LCP cao (phụ thuộc JS bundle + API)
  - SEO kém (crawlers thấy empty HTML)
  - Core Web Vitals tệ trên slow devices

Good for:
  - Dashboards (authenticated, SEO không quan trọng)
  - Complex interactive apps (admin panels)
  - Apps chạy sau login
```

```
SSR (Server-Side Rendering):

Server process:        Browser nhận:         After hydration:
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ API fetch    │      │ Full HTML    │      │ Interactive  │
│ React render │  →   │ with content │  →   │ React app    │
│ HTML string  │      │ (not yet     │      │              │
│              │      │  interactive)│      │              │
└──────────────┘      └──────────────┘      └──────────────┘
t=0: Request          t=200ms: HTML          t=1500ms: Interactive
t=100: DB query       (LCP fast! ✅)         (TTI still slow ❌)
t=200: HTML ready     t=200: Content visible
                      t=1500: JS loaded
                      t=1500: Hydration done

Problems:
  - TTFB cao nếu server chậm (DB queries)
  - "Hydration gap": HTML visible nhưng JS chưa attached
    Click trong khoảng này = lost events
  - Server cost cao hơn (compute per request)
  - Caching phức tạp hơn (dynamic per user)

Good for:
  - Marketing pages (SEO critical)
  - E-commerce product pages
  - News/blog with personalization
```

```
SSG (Static Site Generation):

Build time:            CDN:                  Browser:
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ Pre-render   │      │ Static HTML  │      │ Full content │
│ ALL pages    │  →   │ files on CDN │  →   │ instantly    │
│ at build     │      │              │      │              │
└──────────────┘      └──────────────┘      └──────────────┘
t=build: All pages    t=0: CDN edge          t=50ms: Content
pre-rendered          serves instantly       (TTFB ~10ms ✅✅)

Problems:
  - Stale content (rebuild required for updates)
  - Build time tăng với nhiều pages (10,000 pages = slow build)
  - Không personalized (same HTML cho mọi user)
  - Không phù hợp với real-time data

Good for:
  - Marketing sites
  - Documentation
  - Blog posts
  - Product catalogs (không thay đổi thường)

ISR (Incremental Static Regeneration — Next.js):
  Hybrid: pre-generate pages, rebuild on-demand hoặc theo schedule
  revalidate: 60 → rebuild page sau 60 giây nếu có request
  Giải quyết "stale content" problem của SSG
```

### 1.5 — Streaming SSR — React 18 + Server Components

**Vấn đề với SSR truyền thống:**

```
Traditional SSR — "All or nothing":

Server phải:
  1. Fetch tất cả data (user, products, recommendations, cart...)
  2. Render toàn bộ component tree thành HTML string
  3. GỬI toàn bộ HTML

User nhìn thấy: BLANK → đột ngột toàn bộ page xuất hiện

Bottleneck:
  Slow component (ví dụ: recommendations cần ML inference 800ms)
  → TOÀN BỘ page bị hold cho đến khi nó xong
  → TTFB = thời gian của component chậm nhất
```

**Streaming SSR — gửi HTML theo từng phần:**

```
Streaming SSR với React Suspense:

Server gửi từng chunk HTML khi sẵn sàng:

t=0ms:   <html><head>...</head><body>
t=50ms:  <Header /> (fast — static)
t=100ms: <ProductInfo /> (fast — simple DB query)
t=100ms: <!--$?--><template id="B:0"/><!--/$--> ← Suspense placeholder
t=800ms: <Recommendations /> resolved →
         <script>replaceContent('B:0', '<RecommendationList>...')</script>

Browser:
  t=50ms:  Header visible
  t=100ms: Product info visible
           Skeleton shown for Recommendations
  t=800ms: Recommendations swap in

TTFB = ~50ms (first byte of HTML)
LCP = ~100ms (product info, the likely LCP element)
Recommendations: show late but don't block everything else
```

**React Server Components (RSC) — khác với SSR:**

```
SSR: Components render trên server → HTML → ship to browser
     NHƯNG: JS bundle vẫn include component code để hydrate

RSC: Components chạy ONLY trên server
     → Zero JS sent to client for server components
     → Cannot have state, event handlers, browser APIs

Component categories:

Server Component (default trong Next.js App Router):
  - Fetch data trực tiếp (no API round-trip)
  - Access DB, filesystem, secrets
  - Zero JS bundle cost
  - Cannot: useState, useEffect, onClick, browser APIs

Client Component ('use client' directive):
  - Interactivity, state, effects
  - Chạy trên cả server (SSR) lẫn client (hydration)
  - JS được ship to browser

Boundary:
  Server Component → can render Client Component ✅
  Client Component → CANNOT render Server Component ❌
    (Client component đang ở browser, không thể "call" server)
    Exception: pass Server Component as children prop ✅
```

**RSC Wire Format — không phải HTML:**

```
RSC không gửi HTML — gửi một serialized component tree:

// RSC payload (simplified):
J0:["$","div",null,{"children":["$","h1",null,{"children":"Product Title"}]}]
J1:["$","ul",null,{"children":[
  ["$","li",null,{"children":"Review 1"}],
  ["$","li",null,{"children":"Review 2"}]
]}]

Client-side React runtime:
  Nhận RSC payload → merge vào React tree
  Không re-render client components
  Chỉ update phần server component thay đổi

→ Navigation giữa các pages:
  Không reload HTML (như CSR)
  Không re-render mọi thứ (như SSR)
  Chỉ fetch và merge RSC payload của page mới
```

### 1.6 — Partial Hydration & Islands Architecture

**Vấn đề với Full Hydration:**

```
Full Hydration (React, Next.js pages router):

Server gửi full HTML (good for FCP/LCP)
Browser download entire JS bundle:
  - React runtime: ~40KB
  - All component code: ?KB
  - All dependencies: ?KB

Hydration: React "walks" through the entire DOM,
  recreates the virtual DOM, attaches event listeners
  to EVERY interactive element
  → Even static content (footer, header text) must be hydrated
  → Hydration cost = O(number of components)

Hydration "gap":
  HTML visible → JS loaded → Hydration → Interactive
  [content visible but not interactive for N seconds]
  Clicks during gap: ❌ lost (no event handlers attached yet)
  → INP metric bị ảnh hưởng nặng
```

**Islands Architecture — chỉ hydrate phần interactive:**

```
Page như một "đại dương" HTML tĩnh,
với các "islands" là interactive components.

┌────────────────────────────────────────────────┐
│  <header>Logo + Nav links</header>              │ ← Static HTML, no JS
│                                                  │
│  <article>Blog post content...</article>        │ ← Static HTML, no JS
│                                                  │
│  ┌──────────────────────┐                       │
│  │  🏝️ CommentBox        │ ← Island: hydrated    │
│  │  (useState, handlers) │   with React          │
│  └──────────────────────┘                       │
│                                                  │
│  <footer>Links</footer>                         │ ← Static HTML, no JS
│                                                  │
│  ┌──────────────────────┐                       │
│  │  🏝️ ShareButtons      │ ← Island: hydrated    │
│  │  (onClick handlers)  │   independently       │
│  └──────────────────────┘                       │
└────────────────────────────────────────────────┘

Benefits:
  - Ship ZERO JS for static regions
  - Each island loads its JS independently (parallel)
  - Page interactive as soon as CRITICAL islands hydrate
  - Not waiting for entire page

Frameworks: Astro (primary), Marko, Fresh (Deno)
```

### 1.7 — Resumability (Qwik) — Khái niệm mới nhất

```
Hydration (React/Vue/Angular model):

Server: Render HTML + serialize state
Client: Download framework + all component code
        Re-execute component tree (reconstruct what server already did!)
        Attach event listeners

Problem: "Work done twice" — server rendered, client re-renders

Resumability (Qwik model):

Server: Render HTML + serialize COMPLETE execution state
        (component closures, event handlers, reactivity graph)
        Serialize into HTML as <script type="qwikloader">

Client: Download minimal ~1KB Qwik loader
        Page instantly interactive!

        When user clicks button:
          → Qwik loader sees: "this click needs handler X"
          → Download ONLY the code for handler X (lazy!)
          → Execute handler
          → Download only what's needed for the result

Key difference:
  Hydration: "I need to rebuild the entire app to handle any interaction"
  Resumability: "I have all the state serialized, just pick up where server left off"

Tradeoff:
  Qwik: Faster startup, slower first interaction (lazy load on demand)
  React: Slower startup (hydration), faster subsequent interactions (all code loaded)

Best for: Content-heavy sites where most users don't interact deeply
```

### Streaming SSR Timeline

```
Traditional SSR:
Server: [─── fetch ALL data ───────────────][─── render HTML ───][send]
Client: [wait...................................][FCP/LCP]

Streaming SSR:
Server: [fetch critical][send shell]
        [─────────── fetch slow data ───────][send chunk 2]
Client: [FCP — shell visible][skeleton for slow part][LCP — slow part]

Time saved = Time of slowest component avoided at FCP/LCP
```

## 3. Ví dụ code

### 3.1 — Virtual Scrolling từ scratch (fixed height)

```javascript
class VirtualScroller {
  #container;
  #items;
  #itemHeight;
  #visibleCount;
  #scrollTop = 0;
  #startIndex = 0;
  #buffer = 3; // Render thêm N items ngoài viewport để tránh flash

  constructor({ container, items, itemHeight, renderItem }) {
    this.#container = container;
    this.#items = items;
    this.#itemHeight = itemHeight;
    this.renderItem = renderItem;

    // Total scrollable height = tất cả items
    this.#container.style.position = 'relative';
    this.#container.style.overflowY = 'scroll';

    const containerHeight = container.clientHeight;
    // Số items visible đồng thời (ceiling để không có gap)
    this.#visibleCount = Math.ceil(containerHeight / itemHeight) + 1;

    // Spacers giữ scroll height đúng
    this.#spacerTop = this.#createSpacer();
    this.#spacerBottom = this.#createSpacer();

    // Content wrapper
    this.#content = document.createElement('div');
    container.appendChild(this.#spacerTop);
    container.appendChild(this.#content);
    container.appendChild(this.#spacerBottom);

    // Set tổng scroll height
    this.#spacerBottom.style.height = `${items.length * itemHeight}px`;

    // Throttle scroll handler để không overwhelm
    this.#container.addEventListener(
      'scroll',
      this.#throttle(this.#onScroll.bind(this), 16), // ~60fps
    );

    this.#render(); // Initial render
  }

  #onScroll = () => {
    const newScrollTop = this.#container.scrollTop;
    const newStartIndex = Math.max(
      0,
      Math.floor(newScrollTop / this.#itemHeight) - this.#buffer,
    );

    // Chỉ re-render nếu window thật sự thay đổi
    // Tránh render khi user scroll nhẹ trong window hiện tại
    if (newStartIndex !== this.#startIndex) {
      this.#startIndex = newStartIndex;
      this.#scrollTop = newScrollTop;
      this.#render();
    }
  };

  #render() {
    const endIndex = Math.min(
      this.#items.length - 1,
      this.#startIndex + this.#visibleCount + this.#buffer * 2,
    );

    // Offset spacer-top để push items xuống đúng vị trí
    const offsetY = this.#startIndex * this.#itemHeight;
    this.#spacerTop.style.height = `${offsetY}px`;

    // Update spacer-bottom
    const renderedHeight = (endIndex - this.#startIndex + 1) * this.#itemHeight;
    const totalHeight = this.#items.length * this.#itemHeight;
    this.#spacerBottom.style.height = `${totalHeight - offsetY - renderedHeight}px`;

    // DocumentFragment: batch DOM updates
    // Tránh layout thrashing từ multiple appendChild
    const fragment = document.createDocumentFragment();

    for (let i = this.#startIndex; i <= endIndex; i++) {
      const el = this.renderItem(this.#items[i], i);
      el.style.height = `${this.#itemHeight}px`;
      // Key cho DOM recycling (nếu implement)
      el.dataset.index = i;
      fragment.appendChild(el);
    }

    // Single DOM update
    this.#content.replaceChildren(fragment);
  }

  // Update items (ví dụ: khi sort/filter)
  updateItems(newItems) {
    this.#items = newItems;
    this.#startIndex = 0;
    this.#container.scrollTop = 0;
    this.#spacerBottom.style.height = `${newItems.length * this.#itemHeight}px`;
    this.#render();
  }

  #createSpacer() {
    const spacer = document.createElement('div');
    spacer.style.height = '0px';
    spacer.setAttribute('aria-hidden', 'true');
    return spacer;
  }

  #throttle(fn, delay) {
    let lastCall = 0;
    return function (...args) {
      const now = performance.now();
      if (now - lastCall >= delay) {
        lastCall = now;
        fn.apply(this, args);
      }
    };
  }
}
```

```javascript
// ─── 1. Lazy image loading ─────────────────────────────────────────
class LazyImageLoader {
  #observer;

  constructor() {
    // Dùng native loading="lazy" nếu browser support
    // IO chỉ làm fallback và cho advanced use cases
    if (!('loading' in HTMLImageElement.prototype)) {
      this.#initObserver();
    }
  }

  #initObserver() {
    this.#observer = new IntersectionObserver(
      (entries) => {
        entries.forEach((entry) => {
          if (!entry.isIntersecting) return;

          const img = entry.target;
          this.#loadImage(img);

          // Unobserve sau khi đã load — không cần track nữa
          this.#observer.unobserve(img);
        });
      },
      {
        // 200px trước khi vào viewport → start loading sớm
        rootMargin: '200px 0px',
        threshold: 0,
      },
    );
  }

  observe(img) {
    if ('loading' in HTMLImageElement.prototype) {
      // Native lazy loading supported — browser handles it
      img.loading = 'lazy';
      img.src = img.dataset.src;
      if (img.dataset.srcset) img.srcset = img.dataset.srcset;
    } else {
      // Fallback: IntersectionObserver
      this.#observer?.observe(img);
    }
  }

  #loadImage(img) {
    const src = img.dataset.src;
    if (!src) return;

    // Preload để tránh layout shift khi src được set
    const tempImg = new Image();
    tempImg.onload = () => {
      img.src = src;
      if (img.dataset.srcset) {
        img.srcset = img.dataset.srcset;
      }
      img.classList.add('loaded');
      img.removeAttribute('data-src');
    };
    tempImg.onerror = () => {
      img.classList.add('error');
    };
    tempImg.src = src;
  }
}

// ─── 2. Infinite scroll với IntersectionObserver ──────────────────
class InfiniteScroller {
  #sentinel; // Element ở cuối list để watch
  #observer;
  #loading = false;
  #page = 1;
  #hasMore = true;

  constructor({ container, loadMore, sentinel }) {
    this.container = container;
    this.loadMore = loadMore;
    this.#sentinel = sentinel;

    this.#observer = new IntersectionObserver(
      this.#handleIntersection.bind(this),
      {
        // Trigger khi sentinel còn 300px trước khi visible
        // User chưa thấy bottom → data đã sẵn sàng
        rootMargin: '300px',
        threshold: 0,
      },
    );

    this.#observer.observe(this.#sentinel);
  }

  #handleIntersection = async (entries) => {
    const [entry] = entries;

    // Chỉ load khi sentinel visible VÀ không đang loading
    if (!entry.isIntersecting || this.#loading || !this.#hasMore) return;

    this.#loading = true;
    this.#showLoadingIndicator();

    try {
      const { items, hasMore } = await this.loadMore(this.#page);

      this.#appendItems(items);
      this.#hasMore = hasMore;
      this.#page++;

      if (!hasMore) {
        // Không còn data → stop observing
        this.#observer.unobserve(this.#sentinel);
        this.#showEndMessage();
      }
    } catch (error) {
      this.#showError(error);
    } finally {
      this.#loading = false;
      this.#hideLoadingIndicator();
    }
  };

  #appendItems(items) {
    // DocumentFragment để batch DOM inserts
    const fragment = document.createDocumentFragment();
    items.forEach((item) => {
      fragment.appendChild(this.#renderItem(item));
    });
    // Insert TRƯỚC sentinel để sentinel vẫn ở cuối
    this.container.insertBefore(fragment, this.#sentinel);
  }

  // ... render helpers
}

// Sử dụng:
const infiniteScroller = new InfiniteScroller({
  container: document.getElementById('feed'),
  sentinel: document.getElementById('load-more-sentinel'),
  loadMore: async (page) => {
    const res = await fetch(`/api/feed?page=${page}&limit=20`);
    return res.json(); // { items: [...], hasMore: boolean }
  },
});
```

## 5. Câu hỏi kiểm tra

**Q1:** Tại sao virtual scrolling giải quyết performance vấn đề, và "spacer element" đóng vai trò gì trong cơ chế này?

> **A:** Virtual scrolling giải quyết 2 vấn đề cốt lõi: **DOM size** và **rendering cost**. Với 10,000 items trong DOM, browser phải style recalculate, layout, và paint tất cả — dù chỉ 10 items visible. Style recalculation là O(n×rules), layout traversal O(n), memory ~1-2KB/node. Virtual scrolling chỉ render ~10-20 items trong DOM bất kể list dài bao nhiêu. **Spacer elements** là hai `<div>` empty ở đầu và cuối content: spacer-top có `height = startIndex × itemHeight` (đẩy rendered items xuống đúng vị trí trong scroll container), spacer-bottom có `height = (totalItems - endIndex) × itemHeight` (giữ scroll container đúng tổng height để scrollbar hiển thị đúng). Không có spacers, scroll container sẽ chỉ cao bằng ~15 items và user mất scrollbar cùng khả năng navigate.

**Q2:** Streaming SSR với React Suspense khác traditional SSR như thế nào, và điều gì xảy ra ở network level khi server "stream" HTML?

> **A:** Traditional SSR là "all-or-nothing": server phải fetch tất cả data, render toàn bộ component tree thành HTML string, rồi mới gửi — TTFB bằng thời gian của component chậm nhất. Streaming SSR dùng HTTP chunked transfer encoding (header `Transfer-Encoding: chunked`) — server gửi HTML theo từng chunk khi sẵn sàng mà không cần biết trước response size. React Suspense boundaries tạo các "holes" trong HTML ban đầu (placeholder comments), browser render shell ngay lập tức, sau đó khi slow data sẵn sàng, server gửi thêm HTML chunk chứa resolved content và một script nhỏ để swap placeholder bằng content thật. Ở network level: browser nhận và parse các chunks dần dần (streaming parsing), FCP xảy ra khi chunk đầu tiên có meaningful content đến, LCP có thể xảy ra sớm hơn nhiều so với traditional SSR.

**Q3:** Điều gì xảy ra khi React hydration gặp mismatch giữa server HTML và client render, và tại sao điều này ảnh hưởng đến performance?

> **A:** Khi React hydrate, nó "walk" qua DOM và so sánh với virtual DOM được generate từ component code. Nếu mismatch (ví dụ: server render `"10:30:45"` nhưng client generate `"10:30:47"`), React phải **throw away server-rendered HTML và re-render từ đầu** — toàn bộ component tree, không phải chỉ component bị mismatch. Performance impact: (1) Phí công server render HTML (vì bị bỏ đi); (2) Client phải render lại → CPU spike → long task → INP impact; (3) CLS: content visually thay đổi khi re-render → layout shift. Hydration mismatch cũng có thể gây flash của sai content. Nguyên nhân phổ biến: `Date.now()`, `Math.random()`, `window`/`document`, cookies/localStorage được dùng trong initial render.

**Q4:** Tại sao Islands Architecture ship ít JavaScript hơn so với full hydration, và đâu là limitation của approach này?

> **A:** Full hydration (React) ship **toàn bộ component code** dù phần lớn page là static — vì React cần reconstruct entire component tree trong memory để attach event listeners. Islands Architecture chỉ ship JS cho **"islands"** — những components thật sự có interactivity (onClick, useState). Static regions (header text, article content, footer) không có JS — chúng chỉ là raw HTML, browser không cần JS để render chúng. Ví dụ: một blog post với comment box — chỉ comment box là island, toàn bộ article content là static HTML. JS saving có thể 70-90% với content-heavy sites. **Limitations:** (1) Islands độc lập — chúng không share state trực tiếp. Nếu Island A cần biết state của Island B, cần shared store (signals, nanostores) hoặc event bus. (2) Không phù hợp với highly-interactive SPAs (dashboards, editors) — khi hầu hết UI là interactive thì Islands không lợi. (3) Mental model phức tạp hơn — devs phải suy nghĩ rõ về ranh giới server/client.

**Q5:** Tại sao `requestAnimationFrame` là cách đúng để implement virtual scrolling animation, và dùng scroll event trực tiếp có vấn đề gì?

> **A:** Scroll event fire ~60 lần/giây ở 60fps, đôi khi nhiều hơn trên high-refresh displays. Nếu scroll handler gọi `getBoundingClientRect()` hay đọc layout properties → forced synchronous reflow mỗi lần. Nếu handler cũng update DOM (render items mới) → layout thrashing: đọc/ghi xen kẽ trong loop. Kết quả: mỗi scroll event = forced reflow + re-render = jank. `requestAnimationFrame` gom updates vào đúng 1 lần per frame (16.67ms ở 60fps): scroll event được throttle bởi rAF, tất cả DOM reads xảy ra trước rAF callback, tất cả DOM writes xảy ra trong rAF callback — không interleave, không thrashing. Browser timing đảm bảo update xảy ra đúng lúc trước frame paint. Với `passive: true` trên scroll listener, browser không cần chờ JS handler xong mới scroll — scroll animation trên compositor thread không bị block.

## Câu hỏi ôn tập (cho NotebookLM)

### Câu 1: Virtual scrolling hoạt động như thế nào ở cấp độ DOM, và tại sao "spacer elements" là thành phần không thể thiếu của cơ chế này?

**Đáp án:**

Virtual scrolling là kỹ thuật chỉ render **subset nhỏ của list** vào DOM tại bất kỳ thời điểm nào, bất kể list có bao nhiêu items.

**Cơ chế DOM:**

Scroll container chứa 3 phần: spacer-top (empty div), rendered items (~15-20 nodes), spacer-bottom (empty div). Khi user scroll:

1. `scroll` event fire → đọc `container.scrollTop`
2. Tính `startIndex = Math.floor(scrollTop / itemHeight)`
3. Tính `endIndex = startIndex + visibleCount + buffer`
4. Update `spacer-top.style.height = startIndex × itemHeight` (px)
5. Render items từ startIndex đến endIndex vào DOM (replace cũ)
6. Update `spacer-bottom.style.height = (total - endIndex) × itemHeight` (px)

**Vai trò của spacers — tại sao không thể thiếu:**

Spacer-top đẩy rendered items xuống **đúng vị trí** trong scroll container. Nếu user đang ở scroll position 5000px và items bắt đầu từ index 100, spacer-top = 100 × 50 = 5000px đảm bảo item 100 xuất hiện ở đúng y=5000px trong container. Không có spacer-top, items sẽ luôn ở top của container dù scroll position là bao nhiêu.

Spacer-bottom duy trì **tổng scrollable height**. Nếu không có nó, scroll container chỉ cao bằng ~15 items đang render → scrollbar sẽ chỉ có chiều cao nhỏ → user không thể scroll đến cuối list → tổng trải nghiệm broken. Spacer-bottom = `(totalItems - endIndex) × itemHeight` giữ đúng scroll height như thể tất cả 10,000 items đều trong DOM.

### Câu 2: SSR, SSG, và Streaming SSR khác nhau như thế nào trong cách chúng ảnh hưởng đến TTFB, FCP, LCP, và TTI?

**Đáp án:**

**CSR (Client-Side Rendering):**

- TTFB: ~50ms (empty HTML, nhanh)
- FCP: ~1500-3000ms (phải chờ JS download + execute + API)
- LCP: ~2000-4000ms (content từ JS/API)
- TTI: tốt (không có hydration gap, interactive ngay sau render)
- SEO: kém (crawlers thấy empty HTML)

**SSR (Server-Side Rendering):**

- TTFB: ~200-800ms (server phải fetch data, render HTML)
- FCP: TTFB + ~100ms (HTML có content ngay)
- LCP: tốt (content trong initial HTML)
- TTI: chậm (phải chờ JS download + full hydration)
- Hydration gap: HTML visible nhưng không interactive

**SSG (Static Site Generation):**

- TTFB: ~10-50ms (CDN edge, pre-built HTML)
- FCP: ~50-150ms (excellent)
- LCP: ~50-200ms (excellent)
- TTI: tốt (ít JS cần hydrate)
- Stale content: phải rebuild để update

**Streaming SSR:**

- TTFB: ~20-50ms (gửi shell ngay, không chờ slow data)
- FCP: ~50-100ms (shell + fast components stream first)
- LCP: ~100-300ms (fast components thường là LCP element)
- TTI: trung bình (slow components không block hydration của fast parts)
- Benefit chính: slow data không block fast data — each Suspense boundary independent

Key insight: Streaming SSR decouples "time to show critical content" khỏi "time to fetch all data". Traditional SSR: LCP = max(all_component_times). Streaming SSR: LCP ≈ time_of_critical_components only.

### Câu 3: React Server Components (RSC) khác với SSR như thế nào, và "zero bundle cost" của Server Components có nghĩa gì trong thực tế?

**Đáp án:**

**SSR (bao gồm cả Server Components trong Next.js):**
Render HTML trên server → gửi HTML cho browser → browser download JS → hydrate (React re-creates component tree, attaches handlers). Trong traditional SSR, **component code được ship đến browser** để hydration có thể xảy ra.

**React Server Components — sự khác biệt cốt lõi:**
Server Components là components chạy **exclusively on the server**, không bao giờ được ship đến browser. Chúng không có lifecycle trên client. Thay vì gửi component code + HTML, server gửi **RSC payload** — một serialized representation của rendered output (không phải HTML, không phải JS, mà là React's own wire format). Client React runtime interpret RSC payload để update component tree mà không cần component code.

**"Zero bundle cost" trong thực tế:**

```javascript
// Server Component — access DB trực tiếp
async function ProductDescription({ id }) {
  const product = await db.products.findById(id);
  const markdown = await markdownLib.parse(product.description);
  // markdownLib là heavy library (~50KB)

  return <div dangerouslySetInnerHTML={{ __html: markdown }} />;
}
```

`markdownLib` ở đây là ~50KB library. Với SSR truyền thống, library này **được include trong client bundle** để hydration. Với RSC, component code và dependencies không bao giờ rời server → user không download markdownLib → 50KB savings.

**Limitations:** Server Components không thể có `useState`, `useEffect`, event handlers, hay bất kỳ browser API nào. Nếu cần interactivity, component đó phải là Client Component với `'use client'` directive — và Client Component code **được** ship to browser. Architecture pattern: Server Components wrap Client Components, pass data as props. Client Components giữ nhỏ, chỉ chứa interactive logic, không chứa data-fetching hay heavy dependencies.
