## 1. Cơ chế thật

### 1.1 — V8 JIT và tại sao "object shape" quyết định tốc độ

Để hiểu tại sao code JS "nhanh" hay "chậm", cần hiểu V8 làm gì với code đó.

**Pipeline của V8 (đã học Phase 1, nhắc lại ngắn gọn):**

```
Source Code
    │
    ▼
  Parser → AST
    │
    ▼
  Ignition (Interpreter)
  → Tạo Bytecode
  → Chạy code ngay
  → Thu thập "type feedback" (object này có shape gì? hàm này nhận kiểu gì?)
    │
    │  (Nếu hàm được gọi nhiều lần = "hot")
    ▼
  TurboFan (Optimizing Compiler)
  → Dùng type feedback để generate machine code
  → ASSUME: "hàm này luôn nhận object có shape {x: number, y: number}"
  → Generate code cực kỳ nhanh dựa trên assumption đó
    │
    │  (Nếu assumption sai → DEOPTIMIZE)
    ▼
  Back to Ignition (chậm lại)
```

**Hidden Classes (V8 gọi là "Maps") — cơ chế thật:**

Khi bạn tạo object, V8 không lưu nó như một hash map thông thường. V8 tạo một **Hidden Class** — một cấu trúc metadata mô tả "shape" của object đó (thứ tự properties, kiểu của mỗi property, offset trong memory).

```
// Khi bạn viết:
const p1 = {};
p1.x = 10;
p1.y = 20;

// V8 thật sự làm:
// Step 1: p1 = {} → V8 tạo Hidden Class C0 (empty object)
//
// Step 2: p1.x = 10 → V8 tạo Hidden Class C1
//   C1: { x: number at offset 0 }
//   C0 có "transition": "add property x → go to C1"
//   p1 bây giờ point to C1
//
// Step 3: p1.y = 20 → V8 tạo Hidden Class C2
//   C2: { x: number at offset 0, y: number at offset 8 }
//   C1 có "transition": "add property y → go to C2"
//   p1 bây giờ point to C2

Hidden Class Transition Tree:
C0 (empty)
  └─[add x]──→ C1 {x}
                  └─[add y]──→ C2 {x, y}
```

**Tại sao thứ tự thêm property quan trọng:**

```javascript
const p1 = {};
p1.x = 10; // C0 → C1{x} → C2{x,y}
p1.y = 20;

const p2 = {};
p2.y = 30; // C0 → C3{y} → C4{y,x}  ← DIFFERENT hidden class!
p2.x = 40;

// p1 và p2 có cùng properties nhưng KHÁC hidden class
// Vì thứ tự add property khác nhau

// Khi V8 optimize hàm này:
function getX(point) {
  return point.x; // V8 nhớ: "x ở offset 0 trong C2"
}

getX(p1); // Fast: p1 dùng C2, x ở offset 0 ✅
getX(p2); // Slow: p2 dùng C4, x ở offset 8 → inline cache miss
```

**Inline Cache (IC) — tại sao hàm monomorphic nhanh hơn:**

```
Monomorphic (1 hidden class):
  getX(p1), getX(p2), getX(p3) → tất cả cùng hidden class C2
  → V8 nhớ: "luôn là C2, x ở offset 0"
  → Lần 2 trở đi: truy cập x không cần lookup, TRỰC TIẾP từ offset
  → Extremely fast (như native struct access)

Polymorphic (2-4 hidden classes):
  → V8 nhớ danh sách các classes và offsets tương ứng
  → Mỗi lần phải check "class nào?" rồi tìm offset
  → Slower, nhưng V8 còn optimize được

Megamorphic (>4 hidden classes):
  → V8 từ bỏ inline caching
  → Mỗi property access = full hash map lookup
  → Chậm như HashMap thông thường
```

**Deoptimization — khi V8 "bail out":**

```javascript
// V8 đã optimize hàm này dựa trên assumption:
// "arg luôn là number"
function add(a, b) {
  return a + b;
}

add(1, 2); // V8: "cả 2 đều number" → optimize
add(3, 4); // Fast path ✅
add(5, 6); // Fast path ✅
add('hello', 'world'); // DEOPTIMIZE! Assumption sai → back to Ignition

// Sau deoptimize: hàm chạy lại từ bytecode
// V8 có thể RE-optimize sau khi thu thập feedback mới
// nhưng với type feedback "polymorphic", code sẽ không nhanh bằng trước
```

### 1.2 — Time Complexity trong JS Context

Lý thuyết Big-O bạn đã biết. Trong JS thực tế, vấn đề là **hidden constants** và **engine-specific behavior**.

```
Operation costs (approximate, V8):

Array operations:
  arr[i]                     → O(1), rất nhanh (pointer arithmetic)
  arr.push(x)                → O(1) amortized
  arr.unshift(x)             → O(n) — shift tất cả elements!
  arr.splice(0, 1)           → O(n) — shift tất cả elements!
  arr.indexOf(x)             → O(n) linear scan
  arr.includes(x)            → O(n) linear scan
  new Set(arr).has(x)        → O(1) hash lookup — nhưng có O(n) setup cost

Object operations:
  obj[key] (string key)      → O(1) với inline cache hit
  obj[key] (dynamic key)     → O(1) nhưng megamorphic nếu nhiều shapes
  Object.keys(obj)           → O(n) — tạo array mới
  JSON.stringify(obj)        → O(n) — traverse toàn bộ object

DOM operations (đặc biệt đắt):
  element.offsetHeight       → O(?) — force synchronous layout!
  querySelector('#id')       → O(1) amortized với index
  querySelectorAll('.class') → O(n) với n = số matching elements
  appendChild(el)            → O(1) nhưng triggers style recalculation
```

**Hot path là gì và tại sao quan trọng:**

```javascript
// "Hot path" = code chạy nhiều lần nhất (trong loop, event handler, render function)

// ❌ Ví dụ: O(n²) trong hot path
function filterAndSort(items, searchTerm) {
  // Hàm này được gọi mỗi khi user gõ phím (onInput event)
  // items.length = 10,000

  return (
    items
      .filter((item) =>
        item.name.toLowerCase().includes(searchTerm.toLowerCase()),
      )
      // toLowerCase() gọi 2 lần cho MỖI item = 20,000 lần gọi
      // Với searchTerm dài: includes() = O(m*n) với m = searchTerm.length
      .sort((a, b) => a.name.localeCompare(b.name))
  );
  // localeCompare rất chậm (Unicode-aware) = O(n log n) với constant cao
}

// ✅ Fix: move work ra ngoài hot path
function createSearchIndex(items) {
  // Chạy 1 lần khi data load, không phải mỗi keystroke
  return items.map((item) => ({
    ...item,
    _searchKey: item.name.toLowerCase(), // Pre-compute
  }));
}

const indexedItems = createSearchIndex(rawItems);

function filterAndSort(items, searchTerm) {
  const term = searchTerm.toLowerCase(); // 1 lần thay vì n lần

  return items
    .filter((item) => item._searchKey.includes(term))
    .sort((a, b) => {
      // So sánh string đơn giản nhanh hơn localeCompare nhiều lần
      if (a._searchKey < b._searchKey) return -1;
      if (a._searchKey > b._searchKey) return 1;
      return 0;
    });
}
```

### 1.3 — Debounce vs Throttle — Cơ chế thật

Hai pattern này không phải chỉ là "gọi function ít hơn". Chúng có **khác biệt về guarantee** quan trọng khi hiểu timing.

```
Scenario: User scroll liên tục trong 500ms

Event fires: |----|----|----|----|----|----|----| (mỗi | là 1 event, ~60fps = 16ms)
             0   16   32   48   64   80   96  112ms

DEBOUNCE (wait: 100ms):
  → Mỗi lần event fire, reset timer 100ms
  → Function chỉ chạy khi không có event MỚI trong 100ms
  → Đây là "trailing edge" debounce (default)

  Event: |----|----|----|----|----| ... silence 100ms ... → EXECUTE
                                                                ↑
                                                       Chỉ chạy 1 lần
                                                       sau khi user DỪNG scroll

THROTTLE (wait: 100ms):
  → Function chạy tối đa 1 lần per 100ms, bất kể event fires bao nhiêu lần
  → GUARANTEE: luôn có ít nhất 1 lần execute per interval

  Event: |----|----|----|----|----|----|----|----|
  Execute:↑                  ↑                  ↑
          0ms               100ms              200ms
          Chạy ngay     Chạy sau 100ms    Chạy sau 200ms
```

**Khi nào dùng cái nào:**

```
Debounce:
  ✅ Search input — chỉ gọi API sau khi user dừng gõ
  ✅ Window resize handler — chỉ recalculate layout sau khi resize xong
  ✅ Form validation — validate sau khi user dừng nhập
  ❌ Không dùng cho: progress tracking, real-time visualization (user thấy lag)

Throttle:
  ✅ Scroll handler cần update UI liên tục (sticky header, parallax)
  ✅ Mouse move tracking (drag, tooltip follow cursor)
  ✅ Rate limiting API calls nhưng vẫn cần responsiveness
  ❌ Không dùng cho: form submit (có thể bị execute khi không muốn)
```

**Cơ chế bên trong — tự implement để hiểu:**

```javascript
// DEBOUNCE — cơ chế timer reset
function debounce(fn, wait) {
  let timerId = null;

  return function debounced(...args) {
    // Mỗi lần gọi: clear timer cũ (nếu có), đặt timer mới
    // Timer cũ bị cancel → function không chạy
    clearTimeout(timerId);

    // Context (this) và args được capture trong closure
    // Chỉ timer MỚI NHẤT mới thật sự chạy fn
    timerId = setTimeout(() => {
      fn.apply(this, args);
      timerId = null; // Reset để GC có thể dọn dẹp
    }, wait);
  };
}

// Leading edge debounce — chạy ngay lần đầu, rồi ignore tiếp theo
function debounceLeading(fn, wait) {
  let timerId = null;

  return function debounced(...args) {
    const shouldCallNow = timerId === null; // Chưa có timer → lần đầu

    clearTimeout(timerId);
    timerId = setTimeout(() => {
      timerId = null; // Reset sau khi cooldown xong
    }, wait);

    if (shouldCallNow) {
      fn.apply(this, args); // Chạy ngay lần đầu
    }
    // Các lần tiếp theo trong cooldown bị ignore
  };
}

// THROTTLE — cơ chế timestamp
function throttle(fn, wait) {
  let lastCallTime = 0;
  let timerId = null;

  return function throttled(...args) {
    const now = performance.now();
    const remaining = wait - (now - lastCallTime);

    if (remaining <= 0) {
      // Đủ thời gian kể từ lần cuối → execute ngay
      if (timerId) {
        clearTimeout(timerId);
        timerId = null;
      }
      lastCallTime = now;
      fn.apply(this, args);
    } else if (!timerId) {
      // Chưa đủ thời gian, nhưng schedule trailing call
      // Đảm bảo lần gọi cuối cùng LUÔN được execute
      timerId = setTimeout(() => {
        lastCallTime = performance.now();
        timerId = null;
        fn.apply(this, args);
      }, remaining);
    }
    // Nếu đã có timerId: ignore (trailing call đã được schedule)
  };
}
```

### 1.4 — Code Splitting — Cơ chế bên trong bundler

Bundler (Webpack, Rollup, Vite) không tự biết phải split ở đâu. Nó nhìn vào **dynamic `import()`** làm signal.

```
Static import (không split được):
  import { Button } from './Button';
  → Bundler thấy: "Button cần thiết ngay khi module này load"
  → Đưa vào cùng bundle chunk

Dynamic import (split point):
  const { Button } = await import('./Button');
  → Bundler thấy: "Button chỉ cần khi code này chạy"
  → Tạo chunk riêng: button.[hash].js
  → Main bundle KHÔNG chứa Button code
  → Browser download button.[hash].js chỉ khi import() được execute

Bundle output:
  main.[hash].js         (eager — download ngay)
  button.[hash].js       (lazy — chỉ download khi cần)
  checkout.[hash].js     (lazy — chỉ download khi cần)
  admin.[hash].js        (lazy — chỉ download khi cần)
```

**Waterfall problem và cách tránh:**

```
❌ Waterfall — load lần lượt:

t=0ms   : Browser download main.js
t=500ms : main.js parse xong
t=500ms : import('./Chart') — browser bắt đầu download Chart.js
t=800ms : Chart.js parse xong
t=800ms : Chart.js import('./D3') — browser bắt đầu download D3.js
t=1200ms: D3.js xong → component render được

Total: 1200ms, user thấy blank


✅ Fix: preload lazy chunks khi user có khả năng sẽ cần

// Webpack magic comment syntax:
import(/* webpackPrefetch: true */ './Chart');
// → Browser download Chart.js trong idle time, TRƯỚC khi cần
// → Khi user navigate đến route có Chart, nó đã trong cache

import(/* webpackPreload: true */ './CriticalModal');
// → Download song song với parent chunk (ngay lập tức)
// → Dùng khi biết chắc sẽ cần ngay sau khi parent load
```

**Route-based splitting với React:**

```javascript
import { lazy, Suspense } from 'react';

// lazy() wrap dynamic import → React biết đây là lazy component
// Bundler tạo chunk riêng cho mỗi lazy()
const HomePage = lazy(() => import('./pages/HomePage'));
const CheckoutPage = lazy(() => import('./pages/CheckoutPage'));
const AdminDashboard = lazy(() => import('./pages/AdminDashboard'));

// Suspense: hiển thị fallback trong khi chunk download
// Khi chunk xong, React swap fallback → component thật
function App() {
  return (
    <Suspense fallback={<PageSkeleton />}>
      <Routes>
        <Route
          path='/'
          element={<HomePage />}
        />
        <Route
          path='/checkout'
          element={<CheckoutPage />}
        />
        <Route
          path='/admin'
          element={<AdminDashboard />}
        />
      </Routes>
    </Suspense>
  );
}

// User vào /admin → browser download admin.[hash].js → AdminDashboard render
// User KHÔNG vào /admin trong session → admin chunk không bao giờ download
```

### 1.5 — Tree Shaking — Cơ chế tại sao ES Modules là điều kiện tiên quyết

Tree shaking là quá trình bundler **loại bỏ dead code** — exports không được import bởi ai.

**Tại sao CommonJS không tree-shakeable:**

```javascript
// CommonJS (Node.js legacy) — DYNAMIC exports
const utils = require('./utils');
// require() là function call xảy ra lúc RUNTIME
// Bundler không biết trước bạn sẽ dùng gì từ utils
// → Phải include TẤT CẢ utils vào bundle

// Thậm chí:
const method = someCondition ? 'formatDate' : 'formatCurrency';
utils[method](); // Bundler không thể predict → include tất cả
```

```javascript
// ES Modules — STATIC imports/exports
import { formatDate } from './utils';
// import là KHAI BÁO, không phải function call
// Bundler phân tích statically: "chỉ formatDate được import"
// → Bundler biết chắc: formatCurrency, parseDate, v.v. không cần
// → Có thể safely loại bỏ (tree shake) chúng khỏi bundle
```

**Side effects — lý do tree shaking fail:**

```javascript
// utils.js
export function formatDate(date) {
  /* ... */
}
export function formatCurrency(amount) {
  /* ... */
}

// BUG (implicit side effect):
console.log('utils module loaded'); // Dòng này chạy khi module được import
// Bundler không dám remove module này dù bạn không dùng formatCurrency
// vì removing module sẽ remove side effect này luôn
// → Không tree-shakeable!

// ✅ Fix: Khai báo trong package.json
// "sideEffects": false
// → Nói với bundler: "module này an toàn để tree shake"
// → Hoặc liệt kê: "sideEffects": ["./src/polyfills.js", "*.css"]
```

**Bundler phân tích dependency graph:**

```
Entry point: main.js
  import { formatDate } from './utils'
  import { Button } from './components'

utils.js exports:
  - formatDate  ← được import → KEEP
  - formatCurrency  ← không ai import → REMOVE
  - parseDate   ← không ai import → REMOVE

components/index.js exports:
  - Button  ← được import → KEEP
  - Modal   ← không ai import → trace dependencies...
    Modal imports: animation.js → REMOVE (vì Modal bị remove)
  - Tooltip ← không ai import → REMOVE

Final bundle: main.js + formatDate + Button code only
```

### 1.6 — Web Workers — Cơ chế thật

JS là single-threaded. Web Workers là cách chạy code trên **thread riêng biệt** của OS.

```
Browser Process Architecture:

Main Thread (Renderer Process)
├── JavaScript execution (V8)
├── DOM manipulation
├── CSS styling + Layout
├── Painting
└── User event handling
    ← Tất cả điều này share cùng 1 thread!
    ← Long tasks block TẤT CẢ

Worker Thread (riêng biệt)
├── JavaScript execution (V8 instance riêng)
├── KHÔNG có DOM access
├── KHÔNG có window object
├── Chạy song song với Main Thread ✅
└── Communicate qua postMessage()
```

**Structured Clone Algorithm — cơ chế truyền data:**

Khi `postMessage(data)`, browser phải truyền data giữa 2 V8 instances (2 threads khác nhau, memory riêng biệt). Browser dùng **Structured Clone Algorithm**:

```
Main Thread Memory          Worker Thread Memory
┌────────────────────┐      ┌────────────────────┐
│                    │      │                    │
│  originalData      │      │  clonedData        │
│  {items: [...]}    │─────→│  {items: [...]}    │
│                    │ COPY │                    │
│  (vẫn còn ở đây)  │      │  (bản sao độc lập) │
└────────────────────┘      └────────────────────┘

Chi phí: O(n) serialize + O(n) deserialize
Với data lớn (1MB array): có thể mất 10-50ms chỉ để truyền data!
```

**Transferable Objects — zero-copy transfer:**

```javascript
// ❌ Clone: tốn bộ nhớ và thời gian
const bigArray = new Float64Array(1_000_000); // 8MB
worker.postMessage(bigArray); // COPY 8MB → mất time

// ✅ Transfer: chuyển ownership, không copy
worker.postMessage(bigArray, [bigArray.buffer]);
// bigArray.buffer được CHUYỂN sang worker
// Main thread: bigArray.buffer bị detach (neutered) → không dùng được nữa
// Worker: nhận được buffer nguyên vẹn, O(1) thay vì O(n)
```

### 1.7 — Task Scheduling — Chunking Long Tasks

Vấn đề: Main thread bị block bởi long task → Input delay cao → INP cao.

**Giải pháp: Yield về event loop giữa chừng:**

```
Không yield:
Main Thread: [─────────────────── Long Task (200ms) ───────────────────]
                                                                         ↑
                                              Click event phải đợi 200ms!

Yield định kỳ:
Main Thread: [Chunk 1][yield][Chunk 2][yield][Chunk 3][yield][Chunk 4]
                       ↑             ↑             ↑
                   Click có thể   Click có thể   Click có thể
                   chen vào đây   chen vào đây   chen vào đây
```

**Cơ chế yield — `scheduler.yield()` (mới nhất) và fallback:**

```javascript
// ❌ BUG: Block main thread với large dataset processing
async function processLargeDataset(items) {
  // 10,000 items × 1ms each = 10 giây blocking!
  return items.map((item) => heavyTransform(item));
}

// ✅ FIX 1: setTimeout(0) yield (classic, nhưng có overhead)
function processInChunks(items, chunkSize = 100) {
  return new Promise((resolve) => {
    const results = [];
    let index = 0;

    function processChunk() {
      const end = Math.min(index + chunkSize, items.length);

      // Process chunk này (sync, nhanh)
      while (index < end) {
        results.push(heavyTransform(items[index]));
        index++;
      }

      if (index < items.length) {
        // setTimeout(0): yield về macrotask queue
        // Event loop có thể process input events, rAF callbacks
        // trước khi chunk tiếp theo chạy
        setTimeout(processChunk, 0);
        // Downside: setTimeout có minimum delay ~1ms, nhiều chunks = overhead
      } else {
        resolve(results);
      }
    }

    processChunk();
  });
}

// ✅ FIX 2: scheduler.yield() — modern, ưu tiên user input
async function processInChunksModern(items) {
  const results = [];

  for (let i = 0; i < items.length; i++) {
    results.push(heavyTransform(items[i]));

    // Yield sau mỗi item (hoặc mỗi N items)
    // scheduler.yield() có prioritization:
    //   - User input được ưu tiên hơn tiếp tục task này
    //   - Nếu không có input pending, resume ngay (ít overhead hơn setTimeout)
    if (i % 50 === 0) {
      await scheduler.yield();
      // Sau yield: tiếp tục vòng lặp từ chỗ dừng
      // Closure giữ state (i, results) an toàn
    }
  }

  return results;
}

// ✅ FIX 3: isInputPending() — chỉ yield khi thực sự cần
async function processAdaptive(items) {
  const results = [];

  for (let i = 0; i < items.length; i++) {
    results.push(heavyTransform(items[i]));

    // Chỉ yield nếu có input đang pending
    // Tránh yield không cần thiết → throughput cao hơn
    if (navigator.scheduling?.isInputPending()) {
```

### 3.2 — Memoization patterns thực tế

```javascript
// ─── 1. Simple memo cho pure functions với primitive args ──────────
function memoize(fn) {
  const cache = new Map();

  return function memoized(...args) {
    // Key từ args — chỉ hoạt động tốt với primitive args
    const key = JSON.stringify(args);

    if (cache.has(key)) {
      return cache.get(key); // Cache hit: O(1)
    }

    const result = fn.apply(this, args);
    cache.set(key, result);
    return result;
  };
}

// Ví dụ: fibonacci với memoization
const fib = memoize(function (n) {
  if (n <= 1) return n;
  return fib(n - 1) + fib(n - 2); // Recursive calls cũng hit cache
});

fib(40); // First call: compute và cache all sub-problems
fib(40); // Second call: instant, from cache
fib(41); // Only compute fib(40) + fib(39), rest cached

// ─── 2. WeakMap memo cho object args (không memory leak) ──────────
function memoizeByReference(fn) {
  // WeakMap key phải là object
  // Khi object bị GC, cache entry cũng bị GC → không memory leak!
  const cache = new WeakMap();

  return function memoized(obj) {
    if (cache.has(obj)) {
      return cache.get(obj);
    }

    const result = fn(obj);
    cache.set(obj, result);
    return result;
  };
}

// Ví dụ: expensive calculation on product object
const calculateDiscount = memoizeByReference(function (product) {
  // Giả sử đây là expensive calculation
  return product.basePrice * getDiscountRate(product.category, product.tags);
});

// product object reference giữ nguyên trong session
// → Gọi 100 lần vẫn chỉ compute 1 lần per unique object
calculateDiscount(product); // compute
calculateDiscount(product); // hit cache
calculateDiscount(product); // hit cache

// ─── 3. LRU Cache — khi memo không giới hạn gây memory pressure ──
class LRUCache {
  #capacity;
  #cache; // Map maintains insertion order

  constructor(capacity) {
    this.#capacity = capacity;
    this.#cache = new Map();
  }

  get(key) {
    if (!this.#cache.has(key)) return undefined;

    // Move to end (most recently used)
    const value = this.#cache.get(key);
    this.#cache.delete(key);
    this.#cache.set(key, value);
    return value;
  }

  set(key, value) {
    if (this.#cache.has(key)) {
      this.#cache.delete(key);
    } else if (this.#cache.size >= this.#capacity) {
      // Delete least recently used (first item in Map)
      const lruKey = this.#cache.keys().next().value;
      this.#cache.delete(lruKey);
    }
    this.#cache.set(key, value);
  }
}

// Ví dụ: cache API responses
const apiCache = new LRUCache(100); // max 100 entries

async function fetchProduct(id) {
  const cached = apiCache.get(id);
  if (cached) return cached;

  const product = await fetch(`/api/products/${id}`).then((r) => r.json());
  apiCache.set(id, product);
  return product;
}
```

### 3.3 — Web Worker cho heavy computation

```javascript
// ─── main.js ──────────────────────────────────────────────────────
class WorkerPool {
  #workers = [];
  #queue = [];
  #maxWorkers;

  constructor(workerUrl, maxWorkers = navigator.hardwareConcurrency || 4) {
    this.#maxWorkers = maxWorkers;
    this.workerUrl = workerUrl;
  }

  // Run task trên available worker
  run(data, transferables = []) {
    return new Promise((resolve, reject) => {
      const task = { data, transferables, resolve, reject };

      const idleWorker = this.#workers.find((w) => !w.busy);

      if (idleWorker) {
        this.#runOnWorker(idleWorker, task);
      } else if (this.#workers.length < this.#maxWorkers) {
        // Tạo worker mới (lazy initialization)
        const worker = this.#createWorker();
        this.#runOnWorker(worker, task);
      } else {
        // Tất cả workers đang bận → queue task
        this.#queue.push(task);
      }
    });
  }

  #createWorker() {
    const worker = {
      instance: new Worker(this.workerUrl, { type: 'module' }),
      busy: false,
    };

    worker.instance.onmessage = ({ data }) => {
      worker.currentTask.resolve(data);
      worker.busy = false;
      worker.currentTask = null;

      // Process queued task nếu có
      if (this.#queue.length > 0) {
        this.#runOnWorker(worker, this.#queue.shift());
      }
    };

    worker.instance.onerror = (error) => {
      worker.currentTask?.reject(error);
      worker.busy = false;
    };

    this.#workers.push(worker);
    return worker;
  }

  #runOnWorker(worker, task) {
    worker.busy = true;
    worker.currentTask = task;
    worker.instance.postMessage(task.data, task.transferables);
  }

  terminate() {
    this.#workers.forEach((w) => w.instance.terminate());
    this.#workers = [];
  }
}

// ─── image-processor.worker.js ────────────────────────────────────
// Worker không có DOM access, nhưng có OffscreenCanvas, fetch, etc.
self.onmessage = async ({ data }) => {
  const { imageBuffer, width, height, filter } = data;

  // Chạy trên Worker thread → không block main thread
  const processed = applyImageFilter(imageBuffer, width, height, filter);

  // Transfer buffer về main thread (zero-copy)
  self.postMessage({ processedBuffer: processed }, [processed]);
};

  // Heavy pixel manipulation loop

      const avg = (pixels[i] + pixels[i + 1] + pixels[i + 2]) / 3;
      pixels[i] = pixels[i + 1] = pixels[i + 2] = avg;
    }
    // ...other filters
  }

  return pixels.buffer;
}

// ─── Sử dụng trong component ──────────────────────────────────────
```

## 5. Câu hỏi kiểm tra

**Q1:** Tại sao thứ tự add property vào object ảnh hưởng đến performance, và điều gì xảy ra trong V8 khi 2 objects có cùng properties nhưng được tạo theo thứ tự khác nhau?

> **A:** V8 dùng **Hidden Classes** (internal Maps) để represent object shape. Khi bạn tạo object và add properties, V8 duyệt qua **transition tree** — mỗi hidden class có "transitions" sang hidden class tiếp theo khi property mới được thêm. Thứ tự add property quyết định path trong transition tree. `{x, y}` và `{y, x}` tạo ra 2 hidden classes khác nhau dù có cùng properties, vì chúng đi qua paths khác nhau trong transition tree. Khi Inline Cache (IC) trong V8 thấy nhiều objects với shapes khác nhau truyền vào cùng một hàm, cache trở thành **polymorphic** (2-4 shapes) hoặc **megamorphic** (>4 shapes). Megamorphic IC bị V8 từ bỏ hoàn toàn → mỗi property access phải làm full hash lookup thay vì direct offset read → có thể chậm hơn 10x+.

**Q2:** Điều gì xảy ra khi `postMessage()` một object lớn sang Worker, và tại sao Transferable Objects giải quyết vấn đề đó?

> **A:** `postMessage()` dùng **Structured Clone Algorithm** — deep clone toàn bộ object sang memory space của Worker thread. Chi phí là O(n) để serialize + O(n) để deserialize, cộng với bộ nhớ bị double (main thread vẫn giữ bản gốc, Worker có bản copy). Với 8MB Float64Array, đây có thể tốn 20-50ms chỉ để transfer — tự tạo ra một Long Task. **Transferable Objects** (ArrayBuffer, MessagePort, OffscreenCanvas) giải quyết bằng cơ chế **ownership transfer**: browser thay đổi pointer trong V8 instance của Worker trỏ vào cùng vùng memory, rồi **neutered** (vô hiệu hoá) reference trong main thread. Đây là O(1) operation — không copy gì cả. Downside: sau khi transfer, main thread không dùng được buffer nữa (ném RangeError nếu cố access).

**Q3:** Đoạn code này có vấn đề gì về performance? Có thể gây ra bug gì?

```javascript
const cache = {};

function getCachedData(url) {
  if (cache[url]) {
    return Promise.resolve(cache[url]);
  }
  return fetch(url)
    .then((r) => r.json())
    .then((data) => {
      cache[url] = data;
      return data;
    });
}
```

> **A:** Có 3 vấn đề: **(1) Cache key không an toàn:** URL có thể là `__proto__`, `constructor`, `hasOwnProperty` — những keys này tồn tại trên Object prototype, có thể gây prototype pollution hoặc false cache hit. Fix: dùng `Map` hoặc `Object.create(null)`. **(2) Race condition:** Nếu `getCachedData(url)` được gọi 2 lần nhanh liên tiếp trước khi fetch xong, cả 2 thấy `cache[url]` là undefined → cả 2 đều gọi fetch → 2 requests thay vì 1. Fix: cache **Promise** thay vì result: `cache[url] = fetchPromise` ngay khi tạo. **(3) Cache không bao giờ expire và không giới hạn kích thước:** `cache` object grow mãi → memory leak với ứng dụng chạy lâu. Fix: dùng LRU cache hoặc set TTL.

**Q4:** Tại sao `Array.prototype.unshift()` chậm hơn `Array.prototype.push()` về mặt time complexity, và điều gì xảy ra bên trong V8 khi bạn gọi `unshift()`?

> **A:** `push()` thêm element vào cuối array → O(1) amortized: V8 chỉ cần set `arr[length]` và tăng length. Đôi khi cần reallocate backing store khi capacity đầy, nhưng amortized vẫn là O(1). `unshift()` thêm vào đầu array → O(n): V8 phải **shift toàn bộ existing elements sang phải 1 position** để nhường index 0. Với array 100,000 elements, `unshift()` cần 100,000 memory write operations. Trong V8, small integer arrays được lưu dưới dạng contiguous typed arrays trong heap (optimized representation), và shift operation là physical memory move — tương tự `memmove()` trong C. Nếu cần thường xuyên thêm vào đầu, dùng `unshift()` là anti-pattern → cân nhắc dùng `push()` rồi reverse, hoặc dùng linked list/deque structure.

**Q5:** Debounce và throttle đều giảm số lần hàm được gọi. Khi nào throttle tốt hơn debounce cho scroll event handler?

> **A:** **Throttle** tốt hơn cho scroll handler khi UI cần **continuous feedback** — ví dụ: sticky header hiển thị/ẩn dựa trên scroll position, parallax effect, hay progress indicator. Với throttle, handler đảm bảo chạy đều đặn (mỗi 100ms) trong khi user đang scroll, cho phép UI update liên tục. **Debounce** sẽ fail ở đây vì nó chỉ chạy **sau khi user DỪNG scroll** — user sẽ thấy sticky header không cập nhật trong khi scroll, rồi jump ngay khi dừng. Ngược lại, debounce tốt hơn cho **fire-once-after-done** scenarios: tính toán layout sau khi resize xong, gọi search API sau khi user dừng gõ. Rule of thumb: nếu user cần thấy feedback **trong khi** action đang diễn ra → throttle; nếu chỉ cần kết quả **sau khi** action xong → debounce.

## Câu hỏi ôn tập (cho NotebookLM)

### Câu 1: Monomorphic, Polymorphic, và Megamorphic Inline Cache là gì, và điều gì xảy ra trong V8 khi một hàm chuyển từ monomorphic sang megamorphic?

**Đáp án:**

**Inline Cache (IC)** là cơ chế V8 dùng để cache kết quả property lookups. Khi hàm `getX(point)` chạy lần đầu với object có Hidden Class C2, V8 ghi nhớ: "với C2, property `x` ở offset 0". Lần sau, thay vì lookup lại, V8 kiểm tra "object này có class C2 không?" rồi đọc offset 0 trực tiếp — không cần hash lookup.

**Monomorphic IC** (1 hidden class): V8 tối ưu hoàn toàn, property access gần như bằng native struct access. Fastest case.

**Polymorphic IC** (2-4 hidden classes): V8 duy trì danh sách nhỏ (stub) các classes và offsets tương ứng. Mỗi call phải check "class nào trong danh sách này?" rồi dùng offset tương ứng. Chậm hơn monomorphic nhưng vẫn được optimize.

**Megamorphic IC** (>4 hidden classes): V8 từ bỏ hoàn toàn việc cache. Mọi property access phải tra cứu trong global IC table (essentially một hash map). Đây là worst case — performance tương đương không có optimization.

Khi hàm chuyển từ mono → mega: TurboFan **deoptimize** hàm đó, discard machine code đã compile, quay về chạy Ignition bytecode. V8 có thể re-optimize sau khi thu thập type feedback mới, nhưng với type profile "megamorphic", code được generate sẽ chứa các guard checks và fallbacks thay vì fast-path assumptions.

Hậu quả thực tế: hàm `processItem(item)` chạy 10,000 lần/giây, nếu `item` có 5+ hidden classes → hàm megamorphic → có thể chậm hơn 5-10x so với monomorphic equivalent. Đây là lý do tại sao stable object shapes là performance best practice quan trọng nhất trong V8 optimization.

### Câu 2: Tree shaking hoạt động như thế nào, và tại sao CommonJS modules không thể được tree-shaken trong khi ES Modules có thể?

**Đáp án:**

Tree shaking là quá trình bundler **statically analyze** dependency graph và loại bỏ code không được sử dụng. "Static" là từ khoá then chốt.

**ES Modules tree-shakeable vì:**

- `import` và `export` là **khai báo static** — bundler biết chính xác tại compile time module nào import gì từ module nào
- Bundler xây dựng **dependency graph** dạng cây/DAG: entry point → imports → sub-imports
- Bắt đầu từ entry point, bundler mark tất cả code **reachable**. Code không reachable = dead code = removed

```
Entry: import { formatDate } from './utils'
                                       ↓
utils.js: export formatDate ✅ (reachable)
          export formatCurrency ❌ (không ai import → removed)
          export parseDate ❌ (removed)
```

**CommonJS không tree-shakeable vì:**

- `require()` là **function call** — xảy ra tại **runtime**, không phải compile time
- Bundler không biết trước giá trị của `require()` argument
- `module.exports` có thể được modify tại runtime: `module.exports.newMethod = function() {}`
- Pattern `const { x } = require('./module')` trông như destructuring nhưng thật ra là: gọi require → nhận object → destructure → bundler phải include toàn bộ object

**Side effects — điều kiện để tree shaking hoạt động:**
Ngay cả với ES Modules, nếu module có side effects (code chạy khi module được import, như `document.addEventListener(...)` hay `window.xxx = ...`), bundler không dám remove module dù không có export nào được dùng — removing module đồng nghĩa removing side effect. Khai báo `"sideEffects": false` trong `package.json` là contract với bundler: "file này không có side effects, an toàn để remove nếu exports không được dùng."

### Câu 3: Tại sao `scheduler.yield()` tốt hơn `setTimeout(fn, 0)` cho task scheduling, và cơ chế nào khiến chúng xử lý user input khác nhau?

**Đáp án:**

Cả hai đều là cách **yield về event loop** — cho phép browser xử lý pending tasks khác trước khi tiếp tục. Nhưng cơ chế và priority khác nhau:

**`setTimeout(fn, 0)` — Macrotask Queue:**

- Callback được đặt vào **Macrotask Queue** (còn gọi Task Queue)
- Minimum delay thực tế ~1ms (browsers enforce minimum 1ms, nested setTimeouts ~4ms)
- Không có priority — callback của bạn xếp hàng cùng với mọi macrotask khác
- Browser xử lý input events trong macrotask queue — nhưng không có guarantee về thứ tự

**`scheduler.yield()` — Priority-aware Task Scheduler:**

- Là Web API mới (Chrome 115+, với polyfill)
- Khi resume sau yield, task được schedule với **"user-visible" priority**
- Nếu có input event pending (click, key), browser ưu tiên xử lý input trước khi resume task của bạn
- Không có minimum delay overhead như setTimeout
- `navigator.scheduling.isInputPending()` cho phép **adaptive yielding** — chỉ yield khi thực sự có input pending, tránh overhead không cần thiết

**Thực tế về throughput:**

```
setTimeout(0) approach:
  1000 chunks × 1ms minimum delay = thêm ít nhất 1 giây overhead!
  Dù actual computation là 0.1ms/chunk

scheduler.yield() approach:
  Nếu không có input pending → resume gần như ngay lập tức
  Nếu có input pending → xử lý input → resume
  Overhead chỉ khi cần thiết
```

**Pattern kết hợp tối ưu:**

```javascript
for (let i = 0; i < items.length; i++) {
  process(items[i]);
  // Chỉ yield khi input đang chờ VÀ đã xử lý đủ work
  if (i % 50 === 0 && navigator.scheduling?.isInputPending()) {
    await scheduler.yield();
  }
}
```

Pattern này giữ throughput cao khi không có input, nhưng responsive khi user interact — đây là approach được Google khuyến nghị cho INP optimization.
