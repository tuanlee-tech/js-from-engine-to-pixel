## 1. Cơ chế thật

### 1.1 — HTTP Protocol Evolution

Để hiểu HTTP/2 và HTTP/3 giải quyết vấn đề gì, phải hiểu HTTP/1.1 bị giới hạn ở đâu.

#### HTTP/1.1 — Text protocol, connection-per-request

```
HTTP/1.1 request/response cycle (text-based):

Client                              Server
  │                                   │
  │──── TCP handshake (1.5 RTT) ──────│
  │──── TLS handshake (1-2 RTT) ──────│  ← HTTPS
  │                                   │
  │─── GET /index.html ───────────────│
  │                                   │  (server processes)
  │←── HTTP/1.1 200 OK \r\n ──────────│
  │    Content-Type: text/html \r\n   │
  │    \r\n                           │
  │    <html>...</html>               │
  │                                   │
  │─── GET /styles.css ───────────────│  ← Phải đợi response trước
  │←── HTTP/1.1 200 OK ... ───────────│
  │                                   │
  │─── GET /main.js ──────────────────│  ← Phải đợi lần nữa
  │←── HTTP/1.1 200 OK ... ───────────│

Head-of-Line (HOL) Blocking:
  Connection 1: [req1]────[res1][req2]────[res2][req3]────[res3]
                     ↑ idle      ↑ idle        ↑ idle
                  Waiting for    Waiting for   Waiting for
                  response       response      response
```

**HTTP/1.1 workarounds (và tại sao chúng là hacks):**

```
1. Connection pooling: browsers mở 6 connections/domain
   → 6 parallel requests thay vì 1
   → Nhưng 6× TCP/TLS overhead, server phải handle nhiều connections hơn

2. Domain sharding: chia resources sang nhiều domains
   api.example.com, cdn1.example.com, cdn2.example.com
   → Bypass 6 connections/domain limit
   → Nhưng mỗi domain = thêm DNS lookup + TCP/TLS handshake
   → HTTP/2 render domain sharding là ANTI-PATTERN

3. Bundling/concatenation: ghép nhiều JS/CSS file thành 1
   → Giảm số requests
   → Nhưng một thay đổi nhỏ = invalidate toàn bộ bundle
   → HTTP/2 render cách này cũng thành anti-pattern

4. Inlining: nhúng CSS/JS thẳng vào HTML
   → Không cần request riêng
   → Nhưng không cacheable
```

#### HTTP/2 — Binary framing, multiplexing

HTTP/2 thay đổi fundamental cách data được truyền. Không còn là text — là **binary frames**.

```
HTTP/2 Connection Model:

Single TCP Connection
├── Stream 1 (GET /index.html)
│   ├── HEADERS frame
│   └── DATA frame(s)
├── Stream 3 (GET /styles.css)
│   ├── HEADERS frame
│   └── DATA frame(s)
├── Stream 5 (GET /main.js)
│   ├── HEADERS frame
│   └── DATA frame(s)
└── Stream 7 (GET /api/data)
    ├── HEADERS frame
    └── DATA frame(s)

Tất cả streams chạy ĐỒNG THỜI trên 1 TCP connection!
Frames được interleave:

Wire: [S1 HEADERS][S3 HEADERS][S5 HEADERS][S1 DATA][S3 DATA][S5 DATA]
       ↑ Stream IDs giúp client ghép lại đúng response
```

**HTTP/2 features:**

```
1. Multiplexing:
   Nhiều request/response đồng thời trên 1 connection
   Không có HOL blocking ở APPLICATION layer
   (Vẫn có HOL blocking ở TCP layer — xem HTTP/3)

2. Header Compression (HPACK):
   HTTP/1.1: headers lặp lại mỗi request (Cookie, User-Agent, Accept...)
   → 500-800 bytes overhead mỗi request

   HTTP/2 HPACK: duy trì header table cả 2 phía
   → Gửi index thay vì full header string
   → Repeated headers: 1-2 bytes thay vì 50-100 bytes

3. Server Push (deprecated in practice):
   Server gửi resources TRƯỚC khi client request
   Lý thuyết: gửi CSS/JS khi thấy HTML request
   Thực tế: browsers loại bỏ vì cache invalidation khó, bị thay bằng 103 Early Hints

4. Stream Prioritization:
   Client hint server nên ưu tiên stream nào
   (Ít được implement đúng trong practice)
```

**Tại sao HTTP/2 vẫn có HOL blocking:**

```
TCP HOL Blocking:
  TCP đảm bảo delivery in-order
  Nếu packet bị mất: tất cả data SAU packet đó phải đợi retransmit

  HTTP/2 trên TCP:
  [S1 frame][S3 frame][S5 frame][LOST PACKET][S1 frame][S3 frame]
                                      ↑
                     Tất cả streams bị BLOCK cho đến khi packet này
                     được retransmit và nhận → mọi streams stall!

  HTTP/1.1 với 6 connections:
  Packet loss chỉ ảnh hưởng 1 trong 6 connections
  → Paradoxically, HTTP/1.1 có thể resilient hơn trên lossy networks
```

#### HTTP/3 — QUIC protocol, UDP-based

HTTP/3 giải quyết TCP HOL blocking bằng cách **thay TCP bằng QUIC** (chạy trên UDP).

```
HTTP/3 / QUIC Architecture:

Application Layer:  HTTP/3 (binary frames, same semantics as HTTP/2)
Transport Layer:    QUIC (thay TCP+TLS)
Network Layer:      UDP

QUIC innovations:

1. Independent streams:
   Mỗi stream có sequence space riêng
   Packet loss chỉ ảnh hưởng stream đó, không block streams khác!

   HTTP/2 trên TCP:  [stream1][stream2][LOSS]──wait──[stream1][stream2]
   HTTP/3 trên QUIC: [stream1][stream2][LOSS][stream2 continues!]
                                         ↑ stream1 wait, stream2 không bị ảnh hưởng

2. 0-RTT connection establishment:
   TLS 1.3 + QUIC: có thể resume connection mà không cần handshake
   First visit: 1 RTT (thay vì 3 RTT với TCP+TLS 1.2)
   Subsequent visits: 0 RTT (gửi data ngay trong first packet!)

3. Connection migration:
   QUIC connections identified by Connection ID, không phải IP:port
   → Khi WiFi → 4G, connection TIẾP TỤC (không bị reset)
   → TCP: IP thay đổi = connection drop = reconnect từ đầu

Latency comparison (rough):
  HTTP/1.1 + TLS 1.2: 3 RTT trước khi byte đầu tiên
  HTTP/2 + TLS 1.3:   2 RTT
  HTTP/3 (new):        1 RTT
  HTTP/3 (resume):     0 RTT
```

**Khi nào HTTP/3 không nhanh hơn:**

```
UDP bị firewall block:
  Nhiều corporate networks block UDP trên port lạ
  Browsers fallback sang HTTP/2 tự động (ALPN negotiation)
  → HTTP/3 không phải always-on, là progressive enhancement

Low packet loss networks (fiber, LAN):
  HTTP/2 và HTTP/3 performance gần như nhau
  HTTP/3 shine khi packet loss > 1% (mobile, lossy WiFi)
```

### 1.2 — HTTP Caching — Cơ chế đầy đủ

Caching là một trong những topics bị hiểu sai nhiều nhất. Có nhiều tầng cache với **các headers khác nhau điều khiển từng tầng**.

```
Request journey với caching:

Browser                  Service Worker    HTTP Cache      CDN Edge    Origin Server
   │                          │               │               │              │
   │─── fetch('/api/data') ──→│               │               │              │
   │                          │               │               │              │
   │         (SW intercept?)  │               │               │              │
   │                    [NO SW]               │               │              │
   │──────────────────────────────────────────│               │              │
   │                                          │               │              │
   │                                    (Cache hit?)          │              │
   │                                    [MISS - expired]      │              │
   │──────────────────────────────────────────────────────────│              │
   │                                                          │              │
   │                                                    (CDN hit?)           │
   │                                                    [MISS]               │
   │──────────────────────────────────────────────────────────────────────── │
   │                                                                         │
   │←───────────────────────────────── 200 OK + headers ──────────────────── │
```

#### Cache-Control — Directive đầy đủ

```
Response Header: Cache-Control: <directive>, <directive>

max-age=<seconds>
  → Cache valid trong N giây từ lúc nhận response
  → Browser cache và CDN đều respect

s-maxage=<seconds>
  → Chỉ áp dụng cho shared caches (CDN, proxy)
  → Override max-age cho CDN
  → Ví dụ: max-age=60, s-maxage=86400
    Browser cache: 60 giây
    CDN cache: 24 giờ

no-cache
  ← TÊN GÂY HIỂU LẦM: KHÔNG phải "không cache"
  → Cache ĐƯỢC LƯU, nhưng phải validate với server trước khi dùng
  → Server trả 304 Not Modified nếu còn fresh → dùng cached version
  → Server trả 200 với new content nếu stale → update cache

no-store
  → THẬT SỰ không cache gì cả
  → Dùng cho data nhạy cảm (banking, medical)
  → Mỗi request phải đi đến server

public
  → CDN và shared proxies được phép cache
  → Default cho static assets

private
  → Chỉ browser cache, không CDN
  → Dùng cho user-specific data (profile, cart)

immutable
  → File KHÔNG BAO GIỜ thay đổi trong max-age window
  → Browser không send revalidation request dù user hard refresh
  → Best practice cho content-hashed assets

stale-while-revalidate=<seconds>
  → Trả về stale cache NGAY LẬP TỨC trong khi fetch fresh data background
  → User thấy content ngay (fast!), fresh data dùng cho request kế tiếp
  → Chrome và modern browsers support

stale-if-error=<seconds>
  → Dùng stale cache nếu server lỗi (5xx, network error)
  → Resilience cho origin failures
```

**Caching strategies cho từng loại asset:**

```javascript
// ─── Immutable static assets (content hash trong filename) ────────
// Filename: main.abc123.js — hash thay đổi khi content thay đổi
res.setHeader('Cache-Control', 'public, max-age=31536000, immutable');
// 1 năm cache, không bao giờ revalidate
// Hash trong filename đảm bảo: nội dung mới = URL mới = cache miss → fresh

// ─── HTML files — ngắn hoặc no-cache ─────────────────────────────
// HTML chứa links đến hashed assets → phải fresh để nhận links mới
res.setHeader('Cache-Control', 'no-cache');
// Hoặc: 'public, max-age=0, must-revalidate'
// Browser cache HTML nhưng luôn validate → 304 nếu không thay đổi

// ─── API responses — tùy theo data freshness requirement ──────────
// Real-time data:
res.setHeader('Cache-Control', 'no-store');

// Semi-static (product catalog):
res.setHeader(
  'Cache-Control',
  'public, max-age=300, stale-while-revalidate=60',
);
// CDN cache 5 phút, serve stale trong 1 phút khi revalidating

// User-specific data:
res.setHeader('Cache-Control', 'private, max-age=60');
// Browser cache 1 phút, CDN không cache
```

#### ETag và Conditional Requests — Revalidation mechanism

```
First request:
  Client → GET /data.json
  Server → 200 OK
           ETag: "abc123"              ← hash của response body
           Last-Modified: Thu, 20 Mar 2025 10:00:00 GMT
           Cache-Control: no-cache
           Body: {...}

Cache expires → Revalidation request:
  Client → GET /data.json
           If-None-Match: "abc123"     ← ETag từ lần trước
           If-Modified-Since: Thu, 20 Mar 2025 10:00:00 GMT

  Server (data CHƯA thay đổi):
         → 304 Not Modified            ← No body! Save bandwidth
           ETag: "abc123"

  Server (data ĐÃ thay đổi):
         → 200 OK
           ETag: "xyz789"              ← ETag mới
           Body: {...new data...}

304 Not Modified:
  → Không có response body = tiết kiệm bandwidth đáng kể
  → Browser dùng cached body nhưng update headers (expiry reset)
  → Latency vẫn tốn 1 RTT (request + response)
  → Tốt hơn full request, nhưng tệ hơn cache hit không cần revalidate
```

**ETag strong vs weak:**

```
Strong ETag: "abc123"
  → Byte-for-byte identical response
  → Dùng cho exact matching

Weak ETag: W/"abc123"
  → Semantically equivalent (có thể khác minor formatting)
  → Dùng khi server generate dynamic content nhưng content tương đương
  → Không dùng được với Range requests (byte serving)
```

### 1.3 — Service Workers — Programmable Network Proxy

Service Worker không phải chỉ là "offline support". Nó là một **JavaScript proxy ngồi giữa browser và network**, với khả năng intercept, modify, cache mọi request.

```
Architecture:

Browser Tab                    Service Worker                 Network
(Main Thread)                  (SW Thread)                   (CDN/Origin)
     │                              │                              │
     │── fetch('/api/users') ──────→│                              │
     │                              │                              │
     │                    [SW intercept: onfetch]                  │
     │                              │                              │
     │                    [Cache strategy logic]                   │
     │                              │── GET /api/users ───────────→│
     │                              │←── 200 OK ──────────────────│
     │                              │                              │
     │←── Response ─────────────────│                              │
     │                              │                              │

SW Registration:
  navigator.serviceWorker.register('/sw.js')
  → Browser download và parse sw.js
  → SW chạy trong background thread riêng
  → SW "installed" → "activated" → controls pages

SW Lifecycle:
  Installing → Waiting → Activating → Activated
       ↑                                   ↓
  (New version                      Controls all fetch
   downloaded)                      requests in scope
```

**SW chỉ hoạt động trên HTTPS** (trừ localhost) — vì SW là man-in-the-middle, phải secure.

**Cache strategies — cơ chế từng strategy:**

```
1. Cache First (Offline First):
   Request → Check cache → HIT: return immediately
                         → MISS: fetch network → cache → return

   Tốt cho: static assets (images, fonts, immutable JS)
   Tệ cho: data thay đổi thường xuyên

   Timeline: [SW cache lookup (sync-ish)] vs [Network (1+ RTT)]
   Với cache hit: ~1ms
   Với cache miss: ~300ms (network) + lưu vào cache

2. Network First (Freshness First):
   Request → Fetch network → SUCCESS: cache + return
                           → FAIL (offline): Check cache → return

   Tốt cho: API data cần fresh nhưng có thể offline fallback
   Tệ cho: slow networks (user luôn đợi network trước)

3. Stale-While-Revalidate:
   Request → Check cache → HIT: return stale immediately
                               + fetch network in background
                               + update cache for NEXT request
                         → MISS: fetch network → cache → return

   Tốt cho: content thay đổi không thường xuyên (news, catalog)
   User thấy instant (stale), fresh data ở request kế tiếp

4. Cache Only:
   Request → Cache → HIT: return / MISS: error
   Dùng cho assets bạn pre-cache khi install (app shell)

5. Network Only:
   Request → Network only (bypass SW cache)
   Dùng cho analytics, real-time data không nên cache
```

### 1.4 — Compression — Tại sao Brotli tốt hơn Gzip

```
HTTP Compression negotiation:

Client → Request: Accept-Encoding: gzip, deflate, br, zstd
Server → Response: Content-Encoding: br
                   (compressed body)

Client decompress body trước khi parse.
```

**Gzip vs Brotli — cơ chế khác nhau:**

```
Gzip (1992):
  Algorithm: DEFLATE (LZ77 + Huffman coding)
  Dictionary: 32KB sliding window
  Speed: Fast compression/decompression
  Ratio: baseline

Brotli (Google, 2015):
  Algorithm: LZ77 + Huffman + 2nd-order context modeling
  Dictionary: Built-in 120KB static dictionary của common web content
              (HTML tags, CSS properties, JS keywords đã có sẵn!)
  + Dynamic dictionary từ content
  Speed: Slower to compress (server CPU), same/faster decompress
  Ratio: 20-26% smaller than gzip cho text content

Brotli built-in dictionary trick:
  Khi compress "Content-Type: text/html", Brotli có thể reference
  "Content-Type: text/html" TỪ STATIC DICTIONARY bằng vài bits
  thay vì encode từng byte.
  Gzip không có static dictionary → phải encode from scratch.
```

**Compression levels và trade-offs:**

```
Brotli levels: 0-11
  Level 1: ~gzip speed, slightly better ratio (dùng cho dynamic content)
  Level 4-6: Balanced (CDN default thường ở đây)
  Level 11: Best compression, ~100× slower (pre-compress static assets)

Rule of thumb:
  Static assets (JS, CSS, fonts): Pre-compress tại build time với Brotli-11
  Gzip-9 as fallback → store both, serve based on Accept-Encoding

  Dynamic content (API responses): Brotli-4 hoặc gzip-6
  Không pre-compress được → trade off speed vs ratio

  Binary files (images, video): KHÔNG compress
  Đã compressed rồi, thêm gzip = chậm hơn và to hơn
```

**Minification — không phải chỉ remove whitespace:**

```
JavaScript minification (Terser/esbuild):
  1. Remove whitespace, comments
  2. Rename variables: descriptiveVariableName → a
  3. Simplify expressions: if(x===true) → if(x)
  4. Dead code elimination (basic)
  5. Constant folding: const x = 2 + 3 → const x = 5
  6. Mangle property names (với configuration)

Before:
  function calculateTotalPrice(cartItems, discountRate) {
    return cartItems.reduce((total, item) => {
      return total + (item.price * item.quantity);
    }, 0) * (1 - discountRate);
  }

After Terser:
  function c(a,d){return a.reduce((b,e)=>b+e.price*e.quantity,0)*(1-d)}

CSS minification:
  Remove whitespace, comments
  Merge identical selectors
  Shorten hex colors: #ffffff → #fff
  Remove unnecessary units: 0px → 0

Size reduction (typical):
  Raw JS: 100KB → Minified: 40KB → Gzip: 15KB → Brotli: 12KB
  80-90% reduction từ raw đến compressed!
```

### 1.5 — CDN — Edge Caching Mechanics

```
Without CDN:
  User in Singapore → Origin server in US East
  Latency: ~200ms RTT (physical distance limit)
  Every request travels 15,000km+

With CDN:
  User in Singapore → CDN Edge in Singapore
  Latency: ~5ms RTT (same city)
  Edge serves cached content

  Cache MISS: Edge → Origin (1 time per cache period)
  Cache HIT:  Edge → User (tất cả requests sau đó)

CDN Network topology:
  Origin ──→ Regional POP (Point of Presence) ──→ Edge Node ──→ User
   US East        Singapore datacenter              SG edge
               (may cache here too - tiered cache)
```

**Cache invalidation tại CDN:**

```
CDN caches theo URL. Khi bạn deploy code mới:

Strategy 1: Content-based hashing (recommended)
  main.js → main.abc123.js (hash trong filename)
  Deploy mới → main.xyz789.js (different URL → automatic cache miss)
  Cũ: main.abc123.js (vẫn còn trong CDN, valid)
  CDN không cần invalidate gì!

Strategy 2: Cache-busting query param (fragile)
  main.js?v=1 → main.js?v=2
  Một số CDN treat ?v=1 và ?v=2 khác nhau ✅
  Một số CDN ignore query string → không work ❌

Strategy 3: CDN Purge API (expensive)
  Explicit API call để invalidate URLs
  Instant nhưng tốn tiền, cần coordination với deployment
  Dùng cho: HTML files, hoặc khi không có hashed filenames

CDN cache headers (s-maxage):
  Cache-Control: public, max-age=60, s-maxage=86400
  Browser cache: 60 giây
  CDN cache: 24 giờ
  → User thấy fresh content mỗi 60s
  → CDN tiết kiệm origin traffic với 24h cache
```

### 1.6 — Image Optimization — Formats và Loading

```
Image format comparison:

JPEG:  Lossy, tốt cho photos, không support transparency
PNG:   Lossless, tốt cho graphics với transparency, lớn
GIF:   Animation, chỉ 256 colors, lỗi thời
WebP:  Lossy + lossless, ~30% nhỏ hơn JPEG/PNG, transparency ✅
AVIF:  AV1-based, ~50% nhỏ hơn JPEG, better quality, slower decode
SVG:   Vector, vô hạn scale, text-based (gzip rất tốt), chỉ cho graphics

Browser support (2026):
  WebP:  ~97% (safe to use as primary)
  AVIF:  ~93% (use with fallback)

Responsive images — srcset mechanism:
```

**Tại sao `srcset` tốt hơn CSS media queries cho images:**

```html
<!-- Browser biết:
  - Viewport width
  - Device pixel ratio (DPR): Retina = 2x, 3x
  - Network speed (đôi khi)
  - Trước khi CSS load!
  
  → Browser chọn src tốt nhất TRƯỚC khi parser đọc CSS
  → Media queries CSS load SAU → too late for optimal image loading
-->

<img
  src="hero-800.webp"
  srcset="
    hero-400.webp   400w,
    hero-800.webp   800w,
    hero-1200.webp 1200w,
    hero-1600.webp 1600w
  "
  sizes="
    (max-width: 600px) 100vw,
    (max-width: 1200px) 80vw,
    1200px
  "
  alt="Hero image"
  loading="lazy"
  decoding="async"
  fetchpriority="high"
/>

<!-- sizes attribute:
  "Tôi đang render image này với width bao nhiêu?"
  (max-width: 600px) 100vw → ở viewport 600px: image width = 600px
  (max-width: 1200px) 80vw → ở viewport 1200px: image width = 960px
  1200px → ở viewport lớn hơn: image width luôn là 1200px
  
  Browser tính: viewport 375px DPR 2x → cần 375*2=750px wide image
  → Chọn hero-800.webp (gần nhất ≥ 750px)
-->
```

**`loading="lazy"` — cơ chế IntersectionObserver native:**

```
loading="lazy":
  Browser KHÔNG tải image nếu nằm ngoài viewport
  Khi image approach viewport (threshold ~1200px trước khi visible):
    → Browser bắt đầu tải

  Không cần JavaScript! Browser handle natively

  Cẩn thận: KHÔNG dùng lazy cho LCP image!
  LCP image = image lớn nhất visible in viewport on load
  Nếu lazy → LCP bị delay → LCP score tệ
  Dùng fetchpriority="high" cho LCP image thay vào đó
```

### 1.7 — Font Optimization

Fonts là silent performance killer — thường bị bỏ qua cho đến khi FOIT/FOUT xảy ra.

```
Font Loading Timeline (default behavior):

0ms    : Browser parse HTML, thấy <link rel="stylesheet" href="style.css">
50ms   : CSS downloaded and parsed
50ms   : Browser thấy font-face declaration trong CSS
50ms   : Browser CHƯA download font (lazy — chờ đến khi text cần render)
100ms  : Layout xong, browser thấy text node cần font X
100ms  : Font request bắt đầu
400ms  : Font downloaded
400ms  : FOIT: Flash of Invisible Text (text ẩn trong khi chờ)
```

### 3.2 — Service Worker đầy đủ (production pattern)

```javascript
// sw.js

const CACHE_VERSION = 'v3'; // Tăng để force update
const STATIC_CACHE = `static-${CACHE_VERSION}`;
const DYNAMIC_CACHE = `dynamic-${CACHE_VERSION}`;

// Assets cần thiết cho App Shell — pre-cache khi install
const APP_SHELL_URLS = [
  '/',
  '/offline.html',
  '/styles/critical.css',
  '/scripts/app.js',
  '/fonts/inter-var.woff2',
];

// ─── Install: pre-cache App Shell ─────────────────────────────────
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(STATIC_CACHE).then((cache) => {
      // addAll là atomic: nếu 1 URL fail → tất cả fail
      // → SW không install → user vẫn dùng phiên cũ
      return cache.addAll(APP_SHELL_URLS);
    }),
  );

  // Skip waiting: activate ngay thay vì chờ tab cũ đóng
  // Cẩn thận: pages đang mở có thể dùng SW mới + cache cũ
  self.skipWaiting();
});

// ─── Activate: dọn dẹp cache cũ ──────────────────────────────────
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches
      .keys()
      .then((cacheNames) => {
        return Promise.all(
          cacheNames
            .filter((name) => name !== STATIC_CACHE && name !== DYNAMIC_CACHE)
            .map((name) => {
              console.log('[SW] Deleting old cache:', name);
              return caches.delete(name);
            }),
        );
      })
      .then(() => {
        // Claim: SW control các tabs đang mở ngay (không cần reload)
        return self.clients.claim();
      }),
  );
});

// ─── Fetch: intercept và apply strategy ───────────────────────────
self.addEventListener('fetch', (event) => {
  const { request } = event;
  const url = new URL(request.url);

  // Chỉ intercept GET requests từ same origin
  if (request.method !== 'GET') return;
  if (url.origin !== self.location.origin && !isTrustedCDN(url)) return;

  // Routing: áp dụng strategy phù hợp
  if (isStaticAsset(url)) {
    event.respondWith(cacheFirst(request));
  } else if (isAPIRequest(url)) {
    event.respondWith(networkFirstWithFallback(request));
  } else if (isPageRequest(request)) {
    event.respondWith(staleWhileRevalidate(request));
  }
});

// ─── Strategy implementations ─────────────────────────────────────

// Cache First: static assets với content hash
async function cacheFirst(request) {
  const cached = await caches.match(request);
  if (cached) return cached;

  try {
    const response = await fetch(request);
    if (response.ok) {
      // Clone vì response chỉ đọc được 1 lần
      const cache = await caches.open(STATIC_CACHE);
      cache.put(request, response.clone());
    }
    return response;
  } catch {
    // Nếu offline và không có cache → 404
    return new Response('Asset not available offline', { status: 404 });
  }
}

// Network First với offline fallback
async function networkFirstWithFallback(request) {
  try {
    const response = await fetch(request);

    if (response.ok) {
      // Cache successful API responses
      const cache = await caches.open(DYNAMIC_CACHE);
      cache.put(request, response.clone());
    }

    return response;
  } catch {
    // Offline: serve từ cache
    const cached = await caches.match(request);
    if (cached) return cached;

    // Không có cache: trả về JSON error
    return new Response(
      JSON.stringify({ error: 'You are offline', offline: true }),
      {
        status: 503,
        headers: { 'Content-Type': 'application/json' },
      },
    );
  }
}

// Stale-While-Revalidate cho HTML pages
async function staleWhileRevalidate(request) {
  const cache = await caches.open(DYNAMIC_CACHE);
  const cached = await cache.match(request);

  // Fetch in background (không await)
  const fetchPromise = fetch(request)
    .then((response) => {
      if (response.ok) {
        cache.put(request, response.clone());
      }
      return response;
    })
    .catch(() => null);

  // Nếu có cache: trả ngay, fetch in background
  if (cached) return cached;

  // Không có cache: đợi fetch
  const response = await fetchPromise;
  return response || caches.match('/offline.html');
}

// ─── Helpers ──────────────────────────────────────────────────────
function isStaticAsset(url) {
  return (
    /\.(js|css|woff2|png|webp|avif|svg)$/.test(url.pathname) ||
    url.pathname.includes('/static/')
  );
}

function isAPIRequest(url) {
  return url.pathname.startsWith('/api/');
}

function isPageRequest(request) {
  return (
    request.mode === 'navigate' ||
    request.headers.get('accept')?.includes('text/html')
  );
}

function isTrustedCDN(url) {
  const trustedDomains = ['cdn.example.com', 'fonts.gstatic.com'];
  return trustedDomains.some((domain) => url.hostname === domain);
}
```

### 3.3 — Cache-Control headers (Express middleware)

```javascript
// middleware/caching.js

// Fingerprint-based (content hash trong filename)
// Dùng cho: /static/main.abc123.js, /static/styles.xyz.css
function immutableAssets(req, res, next) {
  // Chỉ áp dụng cho files có hash pattern trong tên
  if (/\.[a-f0-9]{8,}\.(js|css|woff2)$/.test(req.path)) {
    res.set({
      'Cache-Control': 'public, max-age=31536000, immutable',
      // Vary: Accept-Encoding → CDN cache riêng cho gzip vs brotli
      Vary: 'Accept-Encoding',
    });
  }
  next();
}

// HTML: không cache hoặc no-cache
function htmlPages(req, res, next) {
  if (req.path.endsWith('.html') || req.accepts('html')) {
    res.set('Cache-Control', 'no-cache');
    // no-cache = validate mỗi lần, nhưng 304 nếu không thay đổi
  }
  next();
}

// API: short cache với stale-while-revalidate
function apiResponses(req, res, next) {
  // Chỉ GET requests
  if (req.method === 'GET') {
    res.set({
      'Cache-Control': 'public, max-age=60, stale-while-revalidate=30',
      Vary: 'Accept-Encoding, Accept',
      // ETag tự động nếu dùng express-etag middleware
    });
  } else {
    // POST/PUT/DELETE: không cache
    res.set('Cache-Control', 'no-store');
  }
  next();
}

// Security headers cho sensitive routes
function sensitiveData(req, res, next) {
  res.set({
    'Cache-Control': 'no-store, private',
    Pragma: 'no-cache', // HTTP/1.0 backward compat
  });
  next();
}

// Sử dụng
app.use('/static', immutableAssets, express.static('public'));
app.use('/api/public', apiResponses);
app.use('/api/user', sensitiveData);
app.use('/', htmlPages);
```

## 5. Câu hỏi kiểm tra

**Q1:** Tại sao HTTP/2 không giải quyết hoàn toàn Head-of-Line Blocking, và HTTP/3 giải quyết vấn đề đó như thế nào?

> **A:** HTTP/2 giải quyết **application-layer HOL blocking** (nhiều requests trên cùng connection, không phải chờ nhau), nhưng vẫn bị **TCP-layer HOL blocking**. TCP là reliable ordered delivery protocol — nếu một packet bị mất, TCP stall toàn bộ byte stream cho đến khi packet đó được retransmit và nhận, dù các HTTP/2 streams khác không liên quan gì đến packet đó. HTTP/3 thay TCP bằng **QUIC** (chạy trên UDP), trong đó mỗi stream có sequence numbering riêng biệt trong QUIC layer. Packet loss chỉ stall stream liên quan đến packet đó — các streams khác tiếp tục bình thường. Thêm vào đó, QUIC tích hợp TLS 1.3, cho phép 0-RTT connection resumption và Connection Migration (khi đổi network IP, connection không bị reset).

**Q2:** `Cache-Control: no-cache` và `no-store` khác nhau như thế nào? Điều gì xảy ra khi browser gặp mỗi directive?

> **A:** `no-cache` tên gây hiểu lầm — nó **không phải "không cache"**. Nó có nghĩa là "cache được, nhưng phải validate với server trước khi dùng". Browser lưu response, nhưng mỗi lần cần dùng phải gửi conditional request (If-None-Match hoặc If-Modified-Since). Nếu server trả 304, browser dùng cached body — tiết kiệm bandwidth. Nếu server trả 200, update cache. `no-store` là "thật sự không cache" — browser không lưu response xuống disk hay memory. Mỗi request phải fetch đầy đủ từ server. Dùng cho sensitive data (banking, medical records) mà không nên tồn tại trong cache sau khi session kết thúc.

**Q3:** Đoạn code Service Worker này có vấn đề gì?

```javascript
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then((response) => {
      return (
        response ||
        fetch(event.request).then((fetchResponse) => {
          caches.open('my-cache').then((cache) => {
            cache.put(event.request, fetchResponse);
          });
          return fetchResponse;
        })
      );
    }),
  );
});
```

> **A:** Có 3 vấn đề. **(1) Response chỉ đọc được một lần:** `fetchResponse` được return về browser VÀ `cache.put(request, fetchResponse)` cùng lúc. Sau khi response body được browser đọc, cache.put cố đọc body đã consumed → lỗi hoặc empty cache entry. Fix: `cache.put(request, fetchResponse.clone())`. **(2) Cache không selective:** Mọi GET request đều bị cache, kể cả API calls với sensitive data, third-party requests, analytics. Cần filter theo URL/type trước khi cache. **(3) Không handle errors:** Nếu `fetch()` throw (offline, network error) và không có cache → SW crash với unhandled rejection, browser nhận network error. Cần wrap trong try/catch và trả về meaningful offline response.

**Q4:** Tại sao `<link rel="preload" as="font" crossorigin>` cần `crossorigin` attribute ngay cả khi font nằm cùng origin với trang?

> **A:** Fonts luôn được fetch bằng **CORS mode** (`credentials: 'omit'` — anonymous CORS) bất kể same-origin hay cross-origin, vì đây là spec requirement cho font loading. Nếu `preload` không có `crossorigin`, browser fetch font với **no-cors mode**. Khi actual font-face rule cần font đó, browser fetch lại bằng CORS mode — hai requests có khác nhau về mode → browser coi là hai requests khác nhau → **font bị download 2 lần**. `crossorigin` trong preload tag đảm bảo preload request và font-face request dùng cùng mode (CORS) → browser nhận ra là cùng resource → chỉ download 1 lần, preloaded version được reuse.

**Q5:** Brotli compression tốt hơn Gzip trên text content nhờ cơ chế nào, và khi nào KHÔNG nên dùng Brotli?

> **A:** Brotli vượt trội nhờ **built-in static dictionary** chứa ~120KB các patterns phổ biến trong web content — HTML tags, CSS property names, JavaScript keywords, common HTTP strings. Khi compress, Brotli có thể reference "Content-Type: application/json" bằng vài bits (dictionary index) thay vì encode từng byte như Gzip. Thêm vào đó, Brotli dùng thuật toán context modeling phức tạp hơn để predict bytes tiếp theo tốt hơn. Kết quả: ~20-26% nhỏ hơn Gzip cho text content. **Không nên dùng Brotli khi:** (1) Content đã được compressed: images (JPEG/WebP/AVIF), video, audio, ZIP files — thêm compression chỉ lãng phí CPU, output còn có thể lớn hơn; (2) Dynamic content trên server slow: Brotli level cao (9-11) compress chậm hơn Gzip nhiều lần — dùng Brotli 4-6 hoặc chỉ pre-compress static assets; (3) Clients không support: Brotli yêu cầu HTTPS, và một số older HTTP clients chỉ support gzip — cần fallback.

## Câu hỏi ôn tập (cho NotebookLM)

### Câu 1: HTTP/2 multiplexing là gì, tại sao nó là cải tiến lớn so với HTTP/1.1, và điểm yếu nào HTTP/2 vẫn chưa giải quyết được?

**Đáp án:**

**HTTP/1.1 vấn đề cốt lõi:** Mỗi request phải chờ response trước đó xong mới gửi được trên cùng TCP connection (Head-of-Line Blocking ở application layer). Workaround là mở 6 connections/domain — nhưng mỗi connection tốn TCP handshake + TLS handshake, server phải maintain nhiều connections.

**HTTP/2 Multiplexing:** Toàn bộ communication xảy ra trên **1 TCP connection** thông qua **binary frames**. Mỗi request/response là một **Stream** có ID riêng. Frames từ nhiều streams được **interleave** trên connection — browser gửi GET /css, GET /js, GET /img cùng một lúc, server gửi frames từ cả 3 streams đan xen nhau. Client dùng Stream ID để ghép lại đúng response. Kết quả: không còn HOL blocking ở application layer, không cần 6 connections, không cần domain sharding.

**Điểm yếu còn lại — TCP HOL Blocking:** TCP là protocol đảm bảo delivery theo thứ tự. Khi một packet bị mất trên network, TCP stall toàn bộ byte stream — dù HTTP/2 có 20 streams đang chạy, tất cả đều bị freeze cho đến khi packet mất được retransmit và nhận. Trên mạng có packet loss cao (mobile, lossy WiFi), HTTP/1.1 với 6 connections paradoxically có thể nhanh hơn HTTP/2 vì packet loss chỉ ảnh hưởng 1/6 connections. HTTP/3 + QUIC giải quyết điều này bằng cách cho mỗi stream có sequence space riêng ở transport layer.

### Câu 2: Service Worker stale-while-revalidate strategy hoạt động như thế nào ở cấp độ cơ chế, và tại sao nó thường là lựa chọn tốt hơn network-first cho HTML pages?

**Đáp án:**

**Cơ chế stale-while-revalidate:**

Khi request đến SW:

1. SW check Cache API ngay lập tức (synchronous-ish, chạy trong memory)
2. Nếu có cached response: **return ngay lập tức** cho browser — user thấy content trong ~1ms
3. Đồng thời (parallel, không blocking): SW gửi fetch request lên network
4. Khi network response về: SW cập nhật Cache API entry
5. **Request hiện tại đã được serve bằng stale** — network response được dùng cho request KẾ TIẾP

Nếu không có cache: đợi network response, cache, return.

**Tại sao tốt hơn network-first cho HTML pages:**

Network-first luôn chờ network response trước — trên mobile 3G có thể 500-2000ms. Với stale-while-revalidate, user thấy page gần như ngay lập tức (cached HTML từ lần trước). Trade-off: user có thể thấy content cũ 1 request. Đây thường là acceptable vì HTML thay đổi không quá thường xuyên, và user sẽ thấy fresh content ở page reload tiếp theo.

**Quan trọng về implementation:** Response body chỉ readable một lần. Khi trả stale response về browser VÀ muốn fetch mới, cần tránh conflict — fetch network là independent request, không clone stale response. Pattern đúng là: `cache.match()` → trả cached response → `fetch()` độc lập → `cache.put()` với response mới, không liên quan đến stale response đã gửi đi.

### Câu 3: Tại sao content-hashing trong filename (`main.abc123.js`) là caching strategy tốt hơn query string versioning (`main.js?v=2`), và cơ chế nào đảm bảo user luôn nhận code mới nhất khi dùng content hash?

**Đáp án:**

**Query string versioning — vấn đề:**

- Nhiều CDN và proxy được cấu hình **ignore query strings** khi tạo cache key (vì query strings thường là tracking params, không ảnh hưởng content). `main.js?v=1` và `main.js?v=2` bị treat như cùng resource → CDN serve version cũ dù version đã tăng.
- Phải coordination: deploy code mới + update version number trong HTML + purge CDN — nhiều bước, dễ lỗi
- Không phản ánh nội dung thật: `?v=2` không nói gì về content đã thay đổi như thế nào

**Content hash trong filename — cơ chế:**

Bundler (Webpack, Vite) tính hash từ nội dung file: `hash(file_content)` → `abc123`. Filename: `main.abc123.js`.

Khi code thay đổi → nội dung thay đổi → hash thay đổi → **URL thay đổi hoàn toàn**: `main.xyz789.js`. URL mới = cache miss trên CDN = browser phải fetch fresh. URL cũ (`main.abc123.js`) vẫn valid trong CDN — rollback cũng instant vì content vẫn đang được cache.

**Cách user nhận code mới:** HTML file (`index.html`) được serve với `Cache-Control: no-cache`. Browser validate HTML với server mỗi lần. HTML mới chứa `<script src="/main.xyz789.js">` — URL mới → browser fetch fresh JS. Toàn bộ caching strategy dựa trên một điểm: **HTML phải luôn fresh để chứa URLs mới nhất, còn JS/CSS/fonts có thể cache vĩnh viễn vì URL thay đổi khi content thay đổi**.
