## 1. Cơ chế thật

### Performance Timeline — Bộ não đằng sau mọi thứ

Trước khi học từng API, cần hiểu **infrastructure chung** mà tất cả chúng dùng.

Browser duy trì một **Performance Timeline Buffer** — một danh sách có thứ tự của các `PerformanceEntry` object. Mỗi sự kiện đáng chú ý trong lifecycle của trang đều được ghi vào đây.

```
Performance Timeline Buffer (trong Renderer Process)
┌─────────────────────────────────────────────────────────────────┐
│  PerformanceEntry[]                                              │
│  ┌──────────────────┐  ┌──────────────────┐  ┌───────────────┐  │
│  │ type: navigation │  │ type: resource   │  │ type: mark    │  │
│  │ name: document   │  │ name: main.js    │  │ name: app-    │  │
│  │ startTime: 0     │  │ startTime: 120   │  │  start        │  │
│  │ duration: 3200   │  │ duration: 45     │  │ startTime:230 │  │
│  └──────────────────┘  └──────────────────┘  └───────────────┘  │
│                                                                  │
│  ┌─────────────────────────┐  ┌──────────────────────────────┐  │
│  │ type: largest-          │  │ type: layout-shift           │  │
│  │  contentful-paint       │  │ startTime: 1800              │  │
│  │ startTime: 1200         │  │ value: 0.08                  │  │
│  │ element: <img>          │  │ sources: [...]               │  │
│  └─────────────────────────┘  └──────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
          ↑
          │  getEntries(), getEntriesByType(), getEntriesByName()
          │  PerformanceObserver
          │
    JavaScript có thể đọc
```

**Tất cả timestamp trong Performance API đều là `DOMHighResTimeStamp`** — số milliseconds thập phân (vd: `1234.5678`) tính từ `performance.timeOrigin`. Khác với `Date.now()` trả về unix timestamp millisecond nguyên.

**Tại sao không dùng `Date.now()` để đo performance?**

```
Date.now()           → millisecond precision, có thể bị skew bởi system clock adjustment
performance.now()    → microsecond precision (~5µs), monotonic (chỉ tăng, không bao giờ giảm)
                       bị cố ý làm tròn để chống Spectre timing attack (cross-origin)
                       same-origin: độ chính xác cao hơn
```

Browser làm tròn `performance.now()` thành bội số của 0.1ms (cross-origin) hoặc giữ µs precision (same-origin với COOP/COEP headers). Đây là quyết định bảo mật sau Spectre 2018.

### Core Web Vitals — Cơ chế bên trong

#### LCP (Largest Contentful Paint) — Cơ chế tracking

LCP không phải một điểm duy nhất — browser **liên tục cập nhật** LCP candidate khi trang render.

```
Timeline của LCP tracking:

t=0ms    : Trang bắt đầu load
t=200ms  : <h1> render → LCP candidate: h1, size=500px²
t=400ms  : <img> bắt đầu load (chưa decode xong)
t=800ms  : <img> decode xong, paint → LCP candidate: img, size=200000px²
           (img lớn hơn, thay thế candidate cũ)
t=1000ms : User scroll/click/keydown → LCP FINALIZED = 800ms
           (browser ngừng update LCP khi user interact)
t=1200ms : <video> thumbnail render → BỎ QUA (đã finalized)
```

**Element types được xét cho LCP:**

- `<img>` (kể cả `background-image` trong một số trường hợp)
- `<video>` poster image
- Element có CSS background image (từ URL, không phải gradient)
- Block-level element chứa text

**Những gì KHÔNG ảnh hưởng LCP:**

- Element có `opacity: 0`
- Element ngoài viewport
- Element bị clip hoàn toàn

**Điểm quan trọng:** Browser dùng **element's paint time** (khi pixel thực sự được composite lên màn hình), không phải decode time hay load time.

```
Resource Load Timeline vs LCP:

fetchStart ──→ responseEnd ──→ decode ──→ PAINT ←── LCP timestamp ở đây
                                              ↑
                               Pixel composited lên screen
```

#### INP (Interaction to Next Paint) — Thay thế FID từ 2024

**Tại sao FID bị retire?** FID chỉ đo **first interaction** và chỉ đo **input delay** (thời gian từ user action đến khi event handler bắt đầu chạy). Nó bỏ qua:

- Processing time (event handler chạy bao lâu)
- Presentation delay (từ khi handler xong đến khi frame paint)
- Mọi interaction sau interaction đầu tiên

**INP đo toàn bộ interaction latency:**

```
Anatomy của một Interaction:

User clicks button
      │
      ▼
┌─────────────────────────────────────────────────────────────────┐
│ INPUT DELAY        │ PROCESSING TIME     │ PRESENTATION DELAY   │
│                    │                     │                       │
│ Pointerdown event  │ Event handlers run  │ Browser style/layout  │
│ queued, waiting    │ (your JS code)      │ /paint/composite      │
│ for main thread    │                     │                       │
│ to be free         │                     │                       │
└────────────────────┴─────────────────────┴───────────────────────┘
         │                    │                        │
         └────────────────────┴────────────────────────┘
                              │
                    INTERACTION LATENCY
                    (INP tracks this for ALL interactions,
                     reports the 98th percentile)
```

**INP = 98th percentile** của tất cả interaction latencies trong session. Không phải worst case, không phải average — 98th percentile. Nếu user click 50 lần, INP ≈ latency của interaction chậm thứ 1 (bỏ qua 1 cái tệ nhất).

**Thresholds:**

```
LCP: ≤ 2.5s  (Good) | ≤ 4.0s (Needs Improvement) | > 4.0s (Poor)
INP: ≤ 200ms (Good) | ≤ 500ms (Needs Improvement) | > 500ms (Poor)
CLS: ≤ 0.1   (Good) | ≤ 0.25 (Needs Improvement)  | > 0.25 (Poor)
```

#### CLS (Cumulative Layout Shift) — Score được tính như thế nào

CLS không phải "số lần shift". Đây là một **score tích luỹ** dựa trên formula:

```
Layout Shift Score = Impact Fraction × Distance Fraction

Impact Fraction: % of viewport area bị ảnh hưởng
  = (union của vùng trước + sau khi shift) / viewport area

Distance Fraction: % of viewport mà element đã di chuyển
  = max(horizontal_distance, vertical_distance) / viewport dimension

Ví dụ:
  Button shift từ top 10% xuống top 60% của viewport
  Button chiếm 30% viewport height

  Impact Fraction = (10% + 60% + 30%) / 100% ≈ không tính thế này
  Thật ra: union area của vị trí trước (10%→40%) và sau (60%→90%) = 10%→90% = 80%
  Impact Fraction = 80%

  Distance Fraction = |60% - 10%| / 100% = 50%

  Shift Score = 0.8 × 0.5 = 0.4
```

**Session windows:** CLS không cộng dồn tất cả shifts. Nó dùng **session window** — nhóm các shifts xảy ra trong vòng 1 giây của nhau, tối đa 5 giây per window. CLS = score của window lớn nhất.

**Shift nào KHÔNG tính vào CLS:**

- Shift do **user interaction** trong vòng 500ms trước (click, scroll, key press)
- Shift do `transform` animation (không làm reflow)

### Performance API — Cơ chế `performance.now()`

```
Browser process architecture:

┌─────────────────────────────────────────┐
│           Renderer Process              │
│                                         │
│  ┌──────────────────────────────────┐   │
│  │  Blink (rendering engine)        │   │
│  │                                  │   │
│  │  performance.timeOrigin          │   │
│  │  = absolute time trang bắt đầu  │   │
│  │    load (Unix timestamp ms)      │   │
│  │                                  │   │
│  │  performance.now()               │   │
│  │  = monotonic clock từ timeOrigin │   │
│  │  ≈ Date.now() - timeOrigin       │   │
│  │  (nhưng chính xác và không skew) │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**Monotonic** nghĩa là: ngay cả khi user chỉnh đồng hồ hệ thống hoặc NTP sync, `performance.now()` không nhảy ngược. Nó chỉ tăng dần. `Date.now()` có thể giảm nếu system clock được điều chỉnh.

### Navigation Timing — Từng phase có nghĩa gì

Navigation Timing V2 (`PerformanceNavigationTiming`) extend `PerformanceResourceTiming`. Nó track toàn bộ lifecycle của document navigation.

```
Navigation Timing Timeline (tất cả là DOMHighResTimeStamp):

0                                                            loadEventEnd
│                                                                 │
▼                                                                 ▼
────────────────────────────────────────────────────────────────────────
│ redirect │ cache/ │ DNS  │ TCP  │TLS│ Request │Response│ DOM Process │load│
│          │ App    │lookup│ conn │   │         │        │             │    │

startTime (= 0 nếu same-origin)
         │
         unloadEventEnd (unload page cũ)
                │
                fetchStart ←── App Cache check
                        │
                        domainLookupStart
                                │
                                domainLookupEnd
                                        │
                                        connectStart
                                                │
                                                secureConnectionStart (HTTPS only)
                                                        │
                                                        connectEnd
                                                                │
                                                                requestStart
                                                                       │
                                                                       responseStart ← TTFB
                                                                               │
                                                                               responseEnd
                                                                                      │
                                                                              domInteractive
                                                                                      │
                                                                     domContentLoadedEventStart
                                                                                      │
                                                                     domContentLoadedEventEnd
                                                                                      │
                                                                              domComplete
                                                                                      │
                                                                              loadEventStart
                                                                                      │
                                                                              loadEventEnd
```

**Công thức tính các metrics phổ biến:**

```javascript
const nav = performance.getEntriesByType('navigation')[0];

// TTFB (Time To First Byte) — server response speed
const ttfb = nav.responseStart - nav.requestStart;

// DNS lookup time
const dnsTime = nav.domainLookupEnd - nav.domainLookupStart;

// TCP connection time
const tcpTime = nav.connectEnd - nav.connectStart;

// TLS negotiation (HTTPS only)
const tlsTime = nav.connectEnd - nav.secureConnectionStart;

// Total download time (HTML document)
const downloadTime = nav.responseEnd - nav.responseStart;

// DOM processing time (HTML parse + resource load + JS execute)
const domProcessing = nav.domComplete - nav.responseEnd;

// Time until page interactive (DOMContentLoaded)
const domInteractiveTime = nav.domInteractive - nav.fetchStart;

// Total page load
const totalLoadTime = nav.loadEventEnd - nav.fetchStart;
```

**Sự khác biệt quan trọng:**

- `domInteractive`: Parser xong, deferred scripts chưa chạy xong
- `domContentLoadedEventEnd`: `DOMContentLoaded` event đã fire và handler đã xong
- `domComplete`: Tất cả resources (images, subframes) đã load xong
- `loadEventEnd`: `load` event đã fire và handler đã xong

### Resource Timing — Mỗi resource được track riêng

Mỗi resource (JS, CSS, image, font, XHR, fetch) tạo ra một `PerformanceResourceTiming` entry với **cùng timestamp fields** như Navigation Timing.

```
Resource Timing Buffer:
Mặc định giới hạn 250 entries! (configurable với performance.setResourceTimingBufferSize())

Khi buffer đầy:
  - Entries mới BỊ DROP
  - resourcetimingbufferfull event fire
  - Cần clear buffer hoặc tăng size
```

**Cross-origin resources bị redact một số fields:**

```javascript
// Same-origin resource — đầy đủ thông tin
{
  name: "https://example.com/api/data.json",
  fetchStart: 100,
  domainLookupStart: 100,  // ✅ có
  domainLookupEnd: 105,    // ✅ có
  connectStart: 105,       // ✅ có
  requestStart: 110,       // ✅ có
  responseStart: 150,      // ✅ có (TTFB per resource)
  responseEnd: 160,        // ✅ có
  transferSize: 2048,      // ✅ có
  encodedBodySize: 1800,   // ✅ có
  decodedBodySize: 5200    // ✅ có
}

// Cross-origin resource KHÔNG có Timing-Allow-Origin header
{
  name: "https://cdn.other.com/lib.js",
  fetchStart: 200,
  domainLookupStart: 0,    // ❌ zeroed out
  domainLookupEnd: 0,      // ❌ zeroed out
  connectStart: 0,         // ❌ zeroed out
  requestStart: 0,         // ❌ zeroed out
  responseStart: 0,        // ❌ zeroed out (không có TTFB)
  responseEnd: 250,        // ✅ có (startTime và duration vẫn có)
  transferSize: 0,         // ❌ zeroed out
}
// Fix: server phải trả Timing-Allow-Origin: * header
```

### PerformanceObserver — Observer Pattern cho Performance Timeline

```
PerformanceObserver mechanism:

┌─────────────────────────────────────────────────────────────────┐
│  Renderer Process                                               │
│                                                                 │
│  ┌─────────────────┐        ┌──────────────────────────────┐   │
│  │ Performance     │        │ PerformanceObserver           │   │
│  │ Timeline Buffer │──push──│                              │   │
│  │                 │        │ observe({ type: 'lcp',       │   │
│  │ [entry, entry,  │        │           buffered: true })  │   │
│  │  entry, ...]    │        │                              │   │
│  └─────────────────┘        │ callback(list) {             │   │
│                             │   list.getEntries()          │   │
│  Khi entry mới được push    │     .forEach(entry => ...)   │   │
│  vào buffer, observer được  │ }                            │   │
│  notify qua microtask queue │                              │   │
│  (không block rendering)    └──────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘

buffered: true → nhận entries đã có trong buffer trước khi observe() được gọi
              → QUAN TRỌNG cho LCP/CLS vì chúng có thể xảy ra trước code của bạn chạy
```

### Long Tasks — Tại sao 50ms?

```
Human perception threshold:

<100ms  → Instant (user không cảm nhận delay)
100-300ms → Minor delay (vẫn OK)
300-1000ms → Machine is working (user bắt đầu thất vọng)
>1s     → Context switch (user mất focus)

16.67ms → 1 frame ở 60fps

Long Task threshold = 50ms:
  = ~3 frames bị block
  = Đủ để user cảm nhận UI "freeze"

Main Thread:
──────────────────────────────────────────────────────────────
│  Task  │ │  Task  │     LONG TASK (>50ms)       │  Task  │
│  15ms  │ │  20ms  │         80ms                │  10ms  │
──────────────────────────────────────────────────────────────
                     ↑
              Input events, animations
              bị BLOCK trong khoảng này
              → User thấy page unresponsive
```

## 3. Ví dụ code

### 3.1 — Core Web Vitals measurement (production pattern)

```javascript
// web-vitals library source được đơn giản hoá để học cơ chế
// Production: dùng https://github.com/GoogleChrome/web-vitals

// ─── LCP ─────────────────────────────────────────────────────────
function observeLCP(callback) {
  // Browser sẽ report nhiều LCP candidates, candidate cuối cùng trước
  // khi user interact là giá trị thật
  let lcpValue = 0;

  const observer = new PerformanceObserver((entryList) => {
    // Mỗi lần có LCP candidate mới (element lớn hơn xuất hiện),
    // browser push entry vào đây
    const entries = entryList.getEntries();
    const lastEntry = entries[entries.length - 1]; // Luôn lấy entry mới nhất

    // lastEntry.startTime = khi element được paint lên screen
    // lastEntry.element = DOM element (null nếu đã bị remove)
    // lastEntry.size = area in px²
    // lastEntry.url = resource URL (nếu là image/video)
    // lastEntry.loadTime = khi resource load xong (0 cho text)
    // lastEntry.renderTime = khi element paint xong (0 nếu cross-origin image không có TAO header)

    lcpValue = lastEntry.startTime;
  });

  // buffered: true → nhận entries xảy ra TRƯỚC khi observer này được tạo
  // Cực kỳ quan trọng nếu script load muộn
  observer.observe({ type: 'largest-contentful-paint', buffered: true });

  // LCP finalized khi user interact hoặc tab hidden
  // Phải listen cả 2 để capture đúng
  ['keydown', 'click', 'pointerdown'].forEach((eventType) => {
    addEventListener(
      eventType,
      () => {
        observer.disconnect();
        callback({ name: 'LCP', value: lcpValue });
      },
      { once: true, capture: true },
    ); // capture: true để bắt trước propagation
  });

  // Tab bị hide (switch tab, minimize)
  addEventListener(
    'visibilitychange',
    () => {
      if (document.visibilityState === 'hidden') {
        observer.disconnect();
        callback({ name: 'LCP', value: lcpValue });
      }
    },
    { once: true },
  );
}

// ─── INP ─────────────────────────────────────────────────────────
function observeINP(callback) {
  const interactions = new Map(); // interactionId → worst latency cho interaction đó

  const observer = new PerformanceObserver((entryList) => {
    for (const entry of entryList.getEntries()) {
      // entry.interactionId: unique id cho mỗi interaction (click, key, tap)
      // Một interaction có thể sinh nhiều events (pointerdown, pointerup, click)
      // Tất cả share cùng interactionId

      if (!entry.interactionId) continue; // Skip non-interaction events

      // duration = input delay + processing time + presentation delay
      // = thời gian từ event đến next paint
      const existing = interactions.get(entry.interactionId);
      if (!existing || entry.duration > existing) {
        interactions.set(entry.interactionId, entry.duration);
      }
    }
  });

  observer.observe({ type: 'event', buffered: true, durationThreshold: 16 });
  // durationThreshold: 16ms → bỏ qua interactions nhanh hơn 1 frame

  addEventListener(
    'visibilitychange',
    () => {
      if (document.visibilityState === 'hidden') {
        observer.disconnect();

        // INP = 98th percentile của tất cả interactions
        const sortedLatencies = [...interactions.values()].sort(
          (a, b) => b - a,
        );
        const p98Index = Math.floor(sortedLatencies.length * 0.02);
        const inp = sortedLatencies[p98Index] ?? 0;

        callback({ name: 'INP', value: inp });
      }
    },
    { once: true },
  );
}

// ─── CLS ─────────────────────────────────────────────────────────
function observeCLS(callback) {
  let sessionValue = 0; // Score của current session window
  let sessionEntries = []; // Entries trong current window
  let clsValue = 0; // Score của worst session window (đây là CLS thật)

  const observer = new PerformanceObserver((entryList) => {
    for (const entry of entryList.getEntries()) {
      // entry.hadRecentInput = true nếu shift xảy ra trong 500ms sau user interaction
      // Những shift này KHÔNG tính vào CLS (ví dụ: expand accordion khi click)
      if (entry.hadRecentInput) continue;

      const firstEntry = sessionEntries[0];
      const lastEntry = sessionEntries[sessionEntries.length - 1];

      // Session window rules:
      // 1. Gap giữa các shift ≤ 1 giây
      // 2. Tổng duration của window ≤ 5 giây
      if (
        sessionEntries.length === 0 ||
        (entry.startTime - lastEntry.startTime < 1000 &&
          entry.startTime - firstEntry.startTime < 5000)
      ) {
        // Cùng session window
        sessionValue += entry.value;
        sessionEntries.push(entry);
      } else {
        // Bắt đầu session window mới
        sessionValue = entry.value;
        sessionEntries = [entry];
      }

      // CLS = worst session window
      if (sessionValue > clsValue) {
        clsValue = sessionValue;
      }
    }
  });

  observer.observe({ type: 'layout-shift', buffered: true });

  addEventListener(
    'visibilitychange',
    () => {
      if (document.visibilityState === 'hidden') {
        observer.disconnect();
        callback({ name: 'CLS', value: clsValue });
      }
    },
    { once: true },
  );
}
```

### 3.2 — Navigation Timing & Resource Timing (production)

```javascript
// Đợi page load xong hoàn toàn mới đọc
// Nếu đọc sớm hơn, loadEventEnd = 0
window.addEventListener('load', () => {
  // Dùng setTimeout để đảm bảo loadEventEnd đã được ghi
  // (load event handler chưa xong khi event fire)
  setTimeout(() => {
    const [navEntry] = performance.getEntriesByType('navigation');

    const metrics = {
      // Time To First Byte — server latency + network
      ttfb: navEntry.responseStart - navEntry.requestStart,

      // DNS — thường 0 nếu cached hoặc reused connection
      dns: navEntry.domainLookupEnd - navEntry.domainLookupStart,

      // TCP connection — 0 nếu reused connection (HTTP/2 connection reuse)
      tcp: navEntry.connectEnd - navEntry.connectStart,

      // TLS handshake — chỉ có với HTTPS, 0 nếu reused connection
      tls:
        navEntry.secureConnectionStart > 0
          ? navEntry.connectEnd - navEntry.secureConnectionStart
          : 0,

      // Download HTML document
      htmlDownload: navEntry.responseEnd - navEntry.responseStart,

      // DOM processing — parse HTML, load resources, execute JS
      domProcessing: navEntry.domComplete - navEntry.responseEnd,

      // Time to Interactive (approximate — dùng domInteractive)
      // Không phải TTI chuẩn, chỉ là proxy
      domInteractive: navEntry.domInteractive - navEntry.fetchStart,

      // DOMContentLoaded
      dcl: navEntry.domContentLoadedEventEnd - navEntry.fetchStart,

      // Total load
      totalLoad: navEntry.loadEventEnd - navEntry.fetchStart,

      // Loại navigation: 'navigate', 'reload', 'back_forward', 'prerender'
      type: navEntry.type,

      // Kích thước document
      transferSize: navEntry.transferSize, // Bytes truyền qua network (compressed)
      decodedBodySize: navEntry.decodedBodySize, // Bytes sau decompress
    };

    console.table(metrics);
    // Thường send lên analytics endpoint
    sendToAnalytics(metrics);
  }, 0);
});

// ─── Resource Timing — tìm bottleneck ─────────────────────────────
function analyzeResourceBottlenecks() {
  const resources = performance.getEntriesByType('resource');

  return resources
    .map((r) => ({
      name: r.name.split('/').pop(), // Tên file
      type: r.initiatorType, // 'script', 'link', 'img', 'fetch', 'xmlhttprequest'

      // Thời gian từ fetch đến response end (tổng network time)
      totalNetworkTime: r.responseEnd - r.fetchStart,

      // Nếu = 0, resource được serve từ cache!
      // transferSize = 0 nghĩa là disk cache
      // transferSize > 0 nhưng bằng encodedBodySize nghĩa là không compress
      fromCache: r.transferSize === 0,

      // Khi nào trả về first byte — nếu cao = server slow hoặc network slow
      ttfb: r.responseStart > 0 ? r.responseStart - r.requestStart : null,

      // Nếu connectEnd > connectStart → new TCP connection (không reuse)
      // HTTP/2 nên reuse connection cho tất cả same-origin resources
      newConnection: r.connectEnd > r.connectStart,

      transferSize: r.transferSize,
      decodedBodySize: r.decodedBodySize,

      // Compression ratio
      compressionRatio:
        r.decodedBodySize > 0
          ? ((1 - r.encodedBodySize / r.decodedBodySize) * 100).toFixed(1) + '%'
          : 'N/A',
    }))
    .sort((a, b) => b.totalNetworkTime - a.totalNetworkTime); // Sort by slowest
}
```

### 3.3 — User Timing API + Long Tasks (custom instrumentation)

```javascript
// ─── User Timing — đo custom operations ───────────────────────────
class PerformanceTracer {
  // performance.mark() đặt timestamp vào Performance Timeline
  // performance.measure() tạo entry với duration giữa 2 marks

  startTrace(name) {
    // Mark được lưu vào Performance Timeline Buffer
    // Có thể thấy trong DevTools > Performance tab!
    performance.mark(`${name}:start`);
  }

  endTrace(name, metadata = {}) {
    performance.mark(`${name}:end`);

    // Tạo measure entry — visible trong DevTools flame chart
    const measure = performance.measure(name, `${name}:start`, `${name}:end`);

    // measure.duration = thời gian thật (ms)
    console.log(`[Trace] ${name}: ${measure.duration.toFixed(2)}ms`, metadata);

    // Clean up marks (buffer có giới hạn)
    performance.clearMarks(`${name}:start`);
    performance.clearMarks(`${name}:end`);

    return measure.duration;
  }
}

const tracer = new PerformanceTracer();

// Trong React component

// ─── Long Tasks — phát hiện JS blocking main thread ───────────────
function observeLongTasks() {
  if (!('PerformanceObserver' in window)) return;

  const longTasks = [];

  const observer = new PerformanceObserver((list) => {
    for (const entry of list.getEntries()) {
      // entry.duration > 50ms (luôn luôn, đó là định nghĩa của long task)
      // entry.startTime = khi task bắt đầu
      // entry.attribution = [{containerType, containerSrc, containerId, containerName}]
      //   → nói task thuộc iframe/window nào

      const task = {
        start: entry.startTime,
        duration: entry.duration,
        // Attribution giúp biết task đến từ code của mình hay third-party iframe
        attribution: entry.attribution.map((a) => a.containerType),
      };

      longTasks.push(task);

      // Long task > 100ms thường cần fix ngay
      if (entry.duration > 100) {
        console.warn(
          `[LongTask] ${entry.duration.toFixed(0)}ms task detected at ${entry.startTime.toFixed(0)}ms`,
        );
        // Trong production: send to error tracking (Sentry, Datadog)
      }
    }
  });

  observer.observe({ type: 'longtask', buffered: true });

  return () => observer.disconnect();
}
```

### 3.4 — RUM (Real User Monitoring) pattern

```javascript
// Pattern đầy đủ để collect và send performance data
// Đây là bản đơn giản hoá của những gì New Relic, Datadog, SpeedCurve làm

class RUMCollector {
  #metrics = {};
  #observers = [];

  constructor({ endpoint, sampleRate = 1.0 }) {
    this.endpoint = endpoint;
    // Sample rate: 0.1 = chỉ collect từ 10% users (giảm cost)
    this.shouldSample = Math.random() < sampleRate;
  }

  collect() {
    if (!this.shouldSample) return;

    this.#collectCoreWebVitals();
    this.#collectNavigationTiming();
    this.#collectDeviceInfo();
    this.#scheduleSend();
  }

  #collectCoreWebVitals() {
    // Dùng PerformanceObserver như đã implement ở trên
    observeLCP((metric) => (this.#metrics.lcp = metric.value));
    observeINP((metric) => (this.#metrics.inp = metric.value));
    observeCLS((metric) => (this.#metrics.cls = metric.value));
  }

  #collectNavigationTiming() {
    // Không cần observer, dùng getEntriesByType
    // Nhưng phải đợi sau 'load' event
    addEventListener('load', () => {
      setTimeout(() => {
        const [nav] = performance.getEntriesByType('navigation');
        this.#metrics.ttfb = nav.responseStart - nav.requestStart;
        this.#metrics.fcp = this.#getFCP(); // First Contentful Paint
        this.#metrics.pageLoadTime = nav.loadEventEnd - nav.fetchStart;
      }, 0);
    });
  }

  #getFCP() {
    // FCP là paint entry, không phải navigation entry
    const [fcp] = performance.getEntriesByName('first-contentful-paint');
    return fcp?.startTime ?? null;
  }

  #collectDeviceInfo() {
    this.#metrics.deviceInfo = {
      // Connection quality — ảnh hưởng nhiều đến network metrics
      connection: navigator.connection
        ? {
            effectiveType: navigator.connection.effectiveType, // '4g', '3g', '2g', 'slow-2g'
            rtt: navigator.connection.rtt, // Round-trip time (ms)
            downlink: navigator.connection.downlink, // Bandwidth (Mbps)
            saveData: navigator.connection.saveData, // User bật data saver mode
          }
        : null,

      // Memory (chỉ có trong Chrome)
      memory: performance.memory
        ? {
            used: Math.round(performance.memory.usedJSHeapSize / 1048576), // MB
            total: Math.round(performance.memory.totalJSHeapSize / 1048576),
            limit: Math.round(performance.memory.jsHeapSizeLimit / 1048576),
          }
        : null,

      // Viewport
      viewport: `${window.innerWidth}x${window.innerHeight}`,
    };
  }

  #scheduleSend() {
    // Gửi data khi user rời trang
    // KHÔNG dùng 'unload' — phá bfcache và unreliable
    // DÙNG 'visibilitychange' + sendBeacon (fire-and-forget, không block page unload)
    addEventListener('visibilitychange', () => {
      if (document.visibilityState === 'hidden') {
        this.#send();
      }
    });

    // Fallback: pagehide event
    addEventListener('pagehide', () => this.#send());
  }

  #send() {
    const payload = JSON.stringify({
      url: location.href,
      timestamp: Date.now(),
      metrics: this.#metrics,
      // Session ID để correlate multiple requests từ cùng user session
      sessionId: this.#getSessionId(),
    });

    // sendBeacon: fire-and-forget, browser sẽ gửi ngay cả khi tab đóng
    // Không dùng fetch() ở đây vì có thể bị cancel khi tab đóng
    navigator.sendBeacon(this.endpoint, payload);
  }

  #getSessionId() {
    let id = sessionStorage.getItem('rum_session_id');
    if (!id) {
      id = crypto.randomUUID();
      sessionStorage.setItem('rum_session_id', id);
    }
    return id;
  }
}

// Khởi tạo
const rum = new RUMCollector({
  endpoint: '/api/rum',
  sampleRate: 0.1, // 10% users
});
rum.collect();
```

## 4. Ứng dụng thực tế

### 4.1 — Ảnh hưởng đến performance ra sao

**LCP thường bị ảnh hưởng bởi:**

```
LCP > 2.5s thường do một trong các nguyên nhân:

1. Slow server (TTFB cao)
   → responseStart - requestStart > 600ms
   → Fix: server-side caching, CDN, database query optimization

2. Render-blocking resources
   → CSS/JS blocking HTML parser
   → Fix: async/defer scripts, critical CSS inline, preload LCP image

3. Slow image
   → <img> là LCP candidate nhưng không được preload
   → Fix: <link rel="preload" as="image" href="hero.jpg">
   → Fix: fetchpriority="high" attribute trên LCP image
   → Fix: srcset, WebP/AVIF format

4. Client-side rendering
   → HTML trống, React render sau khi JS load
   → LCP element chưa tồn tại khi browser parse HTML lần đầu
   → Fix: SSR hoặc SSG
```

**INP thường bị ảnh hưởng bởi:**

```
INP > 200ms thường do:

1. Long Tasks trên main thread
   → Heavy computation trong event handler
   → Fix: Web Workers, async computation, task chunking

2. Excessive DOM size
   → querySelector, layout calculations slow với DOM lớn
   → Fix: Virtual scrolling, reduce DOM nodes

3. Third-party scripts
   → Analytics, ads chiếm main thread
   → Fix: Load in Web Worker (Partytown), defer, facade

4. React: unnecessary re-renders
   → Interaction trigger re-render toàn bộ subtree
   → Fix: React.memo, useMemo, useCallback, state colocation
```

## 5. Câu hỏi kiểm tra

**Q1:** Tại sao `performance.now()` trả về giá trị được làm tròn trong cross-origin iframe, và cơ chế làm tròn hoạt động như thế nào?

> **A:** Sau lỗ hổng Spectre (2018), attacker có thể dùng high-resolution timer để suy luận nội dung bộ nhớ của process khác thông qua side-channel attacks. Browser response bằng cách cố ý làm tròn `performance.now()` thành bội số của 0.1ms trong cross-origin contexts, giảm precision đủ để Spectre exploit không hiệu quả. Trong same-origin contexts với COOP (`Cross-Origin-Opener-Policy: same-origin`) và COEP (`Cross-Origin-Embedder-Policy: require-corp`) headers, browser cho phép precision cao hơn vì cross-origin isolation đảm bảo các processes đã được cô lập.

**Q2:** Điều gì xảy ra với LCP measurement nếu script của bạn load sau khi LCP element đã paint? Và tại sao `buffered: true` là mandatory cho LCP observer?

> **A:** Nếu LCP element paint ở t=800ms nhưng script load ở t=1200ms, không có `buffered: true` thì PerformanceObserver sẽ không nhận được entry đó — nó chỉ nhận entries xảy ra SAU khi observer được tạo. `buffered: true` khiến browser flush lại tất cả entries đã có trong Performance Timeline Buffer ngay khi `observe()` được gọi, gọi callback ngay lập tức với những entries đó. Đây là lý do tất cả web-vitals library đều set `buffered: true`.

**Q3:** Đoạn code này có vấn đề gì về đo performance?

```javascript
function measureApiCall() {
  const start = Date.now();

  fetch('/api/data')
    .then((res) => res.json())
    .then((data) => {
      const duration = Date.now() - start;
      analytics.track('api_duration', { duration, data });
    });
}
```

> **A:** Có 3 vấn đề: (1) Dùng `Date.now()` thay vì `performance.now()` — kém chính xác hơn (ms vs µs) và có thể bị skew bởi system clock adjustment; (2) Đo từ trước khi fetch bắt đầu thay vì dùng Resource Timing API — bỏ qua DNS, TCP, TLS time; để đo đúng network time của fetch nên dùng `PerformanceObserver` observe `resource` entries với URL filter. (3) Gửi `data` cùng với analytics payload — nếu `data` lớn, toàn bộ response payload được send lên analytics server, đây là security/privacy issue và waste bandwidth. Fix đúng: `performance.mark('api:start')`, khi xong `performance.measure('api', 'api:start')`.

**Q4:** Tại sao INP dùng 98th percentile thay vì worst case (100th percentile)?

> **A:** 100th percentile sẽ capture outliers — ví dụ một lần user click vào moment trình duyệt đang GC heavy, hoặc laptop bị thermal throttling. Những outlier này không đại diện cho trải nghiệm thực sự và không phải lỗi của code. 98th percentile bỏ qua 2% worst interactions, đủ để loại bỏ noise nhưng vẫn capture các pattern thực sự slow. Đây là standard p98/p99 approach trong SLO/SLA của distributed systems — cùng tư duy: đo "typical worst case" không phải "absolute worst case".

**Q5:** Tại sao `navigator.sendBeacon()` được khuyến nghị để gửi analytics data khi page unload, thay vì `fetch()`?

> **A:** Khi user đóng tab/navigate away, browser có thể cancel pending `fetch()` requests trong flight — không có guarantee data được gửi. `sendBeacon()` được browser xử lý như một fire-and-forget queued request: browser giữ request alive và đảm bảo nó được gửi ngay cả khi tab đóng, không cần connection còn sống. Thêm vào đó, `sendBeacon()` không block page unload (không delay navigation), không thể bị cancel bởi page unload, và không trả về response (đúng với use case analytics). Ngoài ra, dùng `unload`/`beforeunload` event để gửi fetch thường fail và phá vỡ bfcache (Back/Forward Cache) — browser không thể cache page nếu nó có `unload` listener.

## Câu hỏi ôn tập (cho NotebookLM)

### Câu 1: Performance Timeline Buffer hoạt động như thế nào, và tại sao thứ tự gọi observe() ảnh hưởng đến data bạn nhận được?

**Đáp án:**

Browser duy trì một **Performance Timeline Buffer** — một mảng có thứ tự của `PerformanceEntry` objects trong Renderer Process. Mỗi khi có sự kiện đáng chú ý (resource load, paint, layout shift, long task...), browser tạo một entry và push vào buffer này.

`PerformanceObserver` là cơ chế subscribe vào buffer này. Khi `observe()` được gọi, observer đăng ký nhận entries thuộc type đã chỉ định **từ thời điểm đó trở đi**. Vì vậy, nếu LCP element đã paint trước khi script của bạn load và gọi `observe()`, bạn sẽ không nhận được entry đó — nó đã nằm trong buffer nhưng observer chưa tồn tại để nhận.

`buffered: true` là giải pháp: khi option này được set, browser sẽ gọi callback ngay lập tức với tất cả entries đã có trong buffer tương ứng với type được observe. Đây là cơ chế "replay" — sau đó observer tiếp tục nhận entries mới theo real-time.

Điều này có nghĩa là cho các metrics như LCP và CLS — những thứ có thể xảy ra rất sớm — **luôn luôn** phải dùng `buffered: true`, ngay cả khi script được inlined vào `<head>`. Không có `buffered: true`, bạn risk bỏ qua toàn bộ metric.

### Câu 2: Tại sao CLS (Cumulative Layout Shift) dùng session windows để tính score thay vì cộng dồn tất cả shifts trong suốt lifetime của page?

**Đáp án:**

Cộng dồn tất cả shifts tạo ra bất công cho long-lived pages (như dashboard, email client, social media feed). Một trang có 10 shifts nhỏ trong 10 phút sẽ có CLS cao hơn một trang có 5 shifts lớn trong 5 giây, mặc dù trang thứ hai gây khó chịu hơn nhiều cho user. User không ở lại trang lâu một cách liên tục — họ interact, đọc, rồi quay lại sau.

Session windows giải quyết điều này bằng cách nhóm shifts theo **thời gian gần nhau**:

- Shifts cách nhau ≤ 1 giây thuộc cùng một session
- Tổng duration của mỗi window tối đa 5 giây
- CLS = score của **window tệ nhất**, không phải tổng tất cả windows

Điều này capture "burst" của shifts — ví dụ: khi ad network inject 3 ads liên tiếp gây layout shift — như một vấn đề, trong khi những layout adjustments thỉnh thoảng xảy ra trong session dài không bị penalize không công bằng.

Thêm vào đó, shifts trong vòng **500ms sau user interaction** bị loại khỏi CLS hoàn toàn. Nếu user click "Show more" và content expand gây shift, đó là expected behavior. Browser dùng `hadRecentInput` flag trong entry để mark những shifts này.

### Câu 3: Điều gì xảy ra bên trong browser khi một INP interaction xảy ra, và tại sao một event handler "nhẹ" vẫn có thể có INP cao?

**Đáp án:**

Một INP interaction bao gồm **3 phases**, không phải chỉ event handler:

**1. Input Delay:** Khoảng thời gian từ khi user interact (pointerdown, keydown...) đến khi event handler bắt đầu chạy. Input delay cao xảy ra khi **main thread đang bận** với task khác — ví dụ: JS đang chạy analytics script, GC đang diễn ra, hay một timer callback đang xử lý. Event được queue vào Web APIs/Callback Queue nhưng không thể execute vì Call Stack chưa trống.

**2. Processing Time:** Event handler của bạn chạy trong bao lâu. Đây là phần bạn kiểm soát trực tiếp.

**3. Presentation Delay:** Sau khi handler return, browser cần chạy **Style → Layout → Paint → Composite** trước khi pixel thật sự thay đổi trên màn hình. Nếu handler setState() trong React và trigger re-render của component tree lớn, hoặc gây layout thrashing, presentation delay có thể cao ngay cả khi handler chạy nhanh.

Ví dụ: event handler `onClick` chạy trong 5ms (processing time thấp), nhưng nó setState() → React re-render 500 components → style recalculation + layout mất 200ms → INP = 205ms (Poor).

Đây là lý do tại sao INP đo **toàn bộ** từ interaction đến next paint, không chỉ handler time — và tại sao React performance optimization (memo, state splitting, concurrent rendering) trực tiếp cải thiện INP.
