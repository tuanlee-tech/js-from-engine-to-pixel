## 2. Senior vs Staff vs Principal: Sự khác biệt thật sự

Đây không phải về title. Đây về **scope of impact** và **mode of operation**.

```
┌─────────────────────────────────────────────────────────────────────┐
│              Scope of Impact Spectrum                               │
│                                                                     │
│  JUNIOR          SENIOR          STAFF           PRINCIPAL          │
│                                                                     │
│  "Tôi làm       "Tôi làm       "Tôi làm        "Tôi xác định       │
│  task được      task tốt       đúng task       task nào cần        │
│  giao"          và làm         cho team"       làm ở cấp           │
│                 nhanh"                         công ty"            │
│                                                                     │
│  Scope: self    Scope: team    Scope: 2-4      Scope: org/         │
│                               teams           company              │
│                                                                     │
│  Output:        Output:        Output:         Output:              │
│  Code           Features       Systems         Direction            │
│                               & Processes                          │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.1 — Cái gì giữ Senior mãi là Senior

Phổ biến nhất: **over-indexing on execution, under-indexing on leverage**.

```
Senior Engineer trap:
├── Luôn là người ship fastest
├── Luôn là người debug best
├── Luôn là người code review nhiều nhất
└── → Impact bị bounded bởi bao nhiêu giờ bạn có

Staff Engineer:
├── Code ít hơn Senior
├── Nhưng code của họ là infrastructure mà 10 engineers dùng
├── Review design documents thay vì chỉ code
├── Ask "why are we building this?" trước "how do we build this?"
└── → Impact không còn bounded bởi giờ làm việc cá nhân
```

### 2.2 — Leverage: Khái niệm cốt lõi của Staff+ level

Leverage = Impact / Time Invested

```
Low leverage activities (cần nhưng không scale):
├── Writing code that only you maintain
├── Being the only person who can fix a class of bugs
├── Solving the same problem repeatedly
└── Knowledge siloed in your head

High leverage activities (multiply your impact):
├── Writing code/docs that upskills your team
├── Creating systems that prevent bugs, not just fix them
├── RFC that aligns 30 engineers before anyone writes code
├── ADR that saves 10 engineers from re-investigating a decision
├── Performance budget that prevents regressions proactively
└── Shared library that solves a problem once for 5 teams
```

Cách nhận ra bạn đang doing high leverage work: **khi bạn nghỉ phép 2 tuần, team vẫn productive vì bạn đã create systems và processes thay vì chỉ execute tasks**.

## 3. The Staff Engineer's Operating System

### 3.1 — Weekly operating rhythm

Staff Engineer không có "sprint tasks" theo nghĩa truyền thống. Họ operate theo một rhythm khác:

```
Weekly Rhythm của một Staff Engineer:

MONDAY:
├── Review: Có RFC/ADR nào cần feedback không?
├── Scan: Có tech debt items nào escalating không?
└── 1:1 với tech lead của team B (adjacent)

TUESDAY - THURSDAY:
├── Deep work: RFC draft, architecture design, POC code
├── Code review: Focus vào design level, không chỉ implementation
└── Async: Answer questions, unblock engineers

FRIDAY:
├── Reflect: Team đang bị block bởi gì? Root cause là gì?
├── Write: Blog post nội bộ, documentation, decision records
└── Look ahead: Cái gì sẽ là problem trong 6 tháng tới mà chưa ai nhìn ra?
```

**"Look ahead" là phần quan trọng nhất và thường bị bỏ qua nhất.**

Senior engineer giải quyết problems hiện tại.
Staff engineer **anticipates** problems tương lai và create conditions để chúng không xảy ra.

### 3.2 — The "Glue Work" awareness

Có một loại work quan trọng nhưng thường invisible — "glue work":

```
Glue work examples:
├── Onboarding documentation cho engineers mới
├── Organizing knowledge sharing sessions
├── Noticing team A và team B có misalignment, facilitating conversation
├── Writing postmortem đầy đủ với action items
├── Triaging tech debt backlog
├── Reviewing interview loops
└── Mentoring engineers không trong direct report chain
```

Glue work tạo ra leverage lớn nhưng thường không được counted trong perf reviews vì nó không produce visible artifacts.

**Staff engineer cần làm hai việc:**

1. **Do** một lượng glue work nhất định — đây là impact thật
2. **Make it visible** — document nó, share nó, ensure nó được recognized

Không phải để PR, mà vì nếu glue work invisible, org không biết nó có giá trị và sẽ không invest vào nó.

### 3.3 — Technical Taste: Thứ không thể teach từ docs

Một điểm phân biệt Staff engineer là "technical taste" — khả năng assess nhanh xem một solution có "đúng" không, kể cả khi không thể articulate ngay tại sao.

Technical taste được build từ:

- **Pattern recognition từ nhiều codebases** — đọc source code của React, Next.js, Prisma, undici
- **Post-mortem learning** — đọc incident reports, failure analyses
- **Production experience** — debug real incidents với real consequences
- **Historical context** — biết tại sao các design decisions được made (Lịch sử ECMAScript trong roadmap này chính là việc build context đó)

Ví dụ về technical taste trong action:

```javascript
// Junior engineer nhìn vào đây và thấy: "works fine"
const results = await Promise.all(
  userIds.map((id) => db.query('SELECT * FROM users WHERE id = $1', [id])),
);

// Staff engineer nhìn vào và ngay lập tức thấy:
// 1. N parallel queries thay vì 1 batch query — N có thể là 10,000
// 2. SELECT * — over-fetching, không biết sẽ dùng field nào
// 3. Không có error handling — 1 failure = toàn bộ Promise.all reject
// 4. Không có connection pool awareness — N connections cùng lúc

// Fix:
const results = await db.query(
  'SELECT id, name, email FROM users WHERE id = ANY($1)',
  [userIds],
);
```

Technical taste không phải magic. Đó là **pattern recognition được build qua deliberate exposure**.

## 4. Putting It All Together: Một ngày trong vai Staff Engineer

Scenario thực tế để connect tất cả:

```
Tình huống: E-commerce platform, bạn là Staff Engineer

09:00 — Morning review
├── Pull Slack: Team A đang plan thêm caching layer bằng Memcached
│   Team B vừa mới setup Redis cho session storage tuần trước
│   → CROSS-TEAM MISALIGNMENT: Hai teams chuẩn bị build duplicate infra
│   → Action: Facilitate conversation, propose shared Redis cluster
│   → Tool: Informal RFC để document decision

10:00 — Code Review
├── PR từ engineer mới: pagination cho product list
│   Code works nhưng:
│   - SELECT * thay vì chỉ fetch needed fields
│   - Offset-based pagination (sẽ chậm ở large offsets)
│   - Không có index trên sort column
│
│   → Comment không chỉ "đây là bug", mà explain WHY:
│   "Offset pagination với 1M rows: OFFSET 990000 LIMIT 20
│    buộc DB scan 990,020 rows, chỉ return 20. Cursor-based
│    pagination fix này. Đây là pattern chúng ta document
│    trong ADR-0023."
│   → Link ADR, giúp engineer learn, không chỉ fix

11:30 — Tech Debt triage meeting
├── Review debt backlog với engineering manager
│   Item nổi bật: Auth service không có rate limiting
│   Present theo business framing:
│   "Credential stuffing attacks tăng 300% industry-wide Q3.
│    Nếu bị tấn công, ước tính 4-6 hours downtime để contain.
│    Implementation estimate: 3 days. Đề xuất Q4 P0."

13:00 — Architecture design session
├── Team đề xuất rewrite search service từ custom solution
│   sang Elasticsearch
│
│   → Apply Buy/Build framework:
│   "Custom solution hiện tại chiếm ~15% backend team bandwidth.
│    Elasticsearch managed (Elastic Cloud) cost $800/month.
│    Break-even: 800 × 12 / (engineer_cost × 0.15) = ~8 months.
│    Đề xuất: Elastic Cloud, kèm ADR document decision."

15:00 — Performance review
├── Weekly metrics review
│   LCP trên product pages: trending từ 2.1s → 2.7s (4 tuần)
│
│   → Dig into: deployment correlation? Nào PR mới nào merge?
│   → Bundle analyzer: +47KB từ PR #1823 (new image carousel)
│
│   → Action: Retroactive budget enforcement không work.
│   Implement Lighthouse CI trước khi PR có thể merge.
│   → Viết RFC ngắn cho process change này.

17:00 — Write
├── 30 phút viết ADR cho Redis decision (từ 09:00)
│   Khi còn context, trước khi quên.
│
│   ADR-0041: Shared Redis cho caching và session storage
│   Context → Decision → Consequences
│   Done trong 20 phút. Sẽ save hours of confusion sau này.
```

Đây là Staff engineer operating: **không code nhiều nhất, nhưng mỗi action multiply impact của người khác**.

## 6. Câu hỏi ôn tập cuối

---

### Câu 1: Tại sao "glue work" visibility quan trọng không kém bản thân việc làm glue work?

**Đáp án:**

Glue work — documentation, cross-team facilitation, onboarding, postmortems — tạo ra real impact nhưng không produce easily-counted artifacts như merged PRs hay shipped features.

Visibility quan trọng vì 3 lý do:

**Lý do 1 — Personal impact:**
Nếu glue work invisible trong perf review, người làm nó không được credit xứng đáng. Overtime, rational response là stop doing glue work và focus vào visible work. Kết quả: org mất valuable infrastructure work.

**Lý do 2 — Org learning:**
Nếu org không see glue work, org không biết nó có giá trị. Không invest vào nó. Không hire cho nó. Không create conditions để nó thể xảy ra (đủ time, recognition). Making it visible = teaching org what matters.

**Lý do 3 — Replication:**
Khi glue work được documented và shared, nó trở thành template mà người khác có thể replicate. Một good onboarding doc được share internally → 5 người khác improve theirs. Visible = multiplied impact.

Practical: Khi làm glue work, **write it down in a format that others can find**. Không cần announce, nhưng "ADR-0041 trong docs/decisions" là visible. "Tôi đã figure out Redis architecture trong đầu" là invisible.

---

### Câu 2: Điều gì xảy ra khi một Staff Engineer sử dụng technical authority để mandate adoption thay vì build consensus?

**Đáp án:**

Short term: Solution được adopted nhanh hơn.

Long term: Series of problems xuất hiện:

**Trust erosion:** Engineers cảm thấy không có voice trong decisions ảnh hưởng tới work của họ. Họ comply nhưng không commit. Khi implementation hits problems, họ không proactively raise concerns — "đây không phải quyết định của tôi."

**Brittle adoption:** Mandated adoption thường là surface-level. Teams implement the letter of the mandate nhưng không the spirit. Ví dụ: mandate "phải viết tests" → teams viết tests nhưng tests không meaningful, chỉ để pass CI.

**Staff engineer becomes bottleneck:** Nếu adoption là do authority chứ không phải conviction, mọi enforcement phải đến từ Staff engineer. Scale không được: một người không thể police mọi decision trong org.

**Talent loss:** Strong engineers không thích environments với top-down technical mandates. Họ leave. Remaining pool là people who are okay with không có agency.

**Đúng approach:** Build shared understanding trước khi propose solution. RFC process là formalization của điều này. Khi people hiểu problem và tham gia vào solution, họ adopt vì convinced, không vì told. Staff engineer's influence phải come from **being right about important things** và **helping others succeed**, không phải từ title.

### Câu 3: Codebase sau đây cho thấy organization dysfunction gì? Đề xuất intervention nào phù hợp nhất?

```
// auth-service/src/utils/email.ts
export async function sendEmail(to: string, subject: string, body: string) {
  // Using Nodemailer
  const transporter = nodemailer.createTransport({ ... });
  return transporter.sendMail({ from: 'noreply@company.com', to, subject, html: body });
}

// notification-service/src/helpers/mailer.ts
export async function sendNotificationEmail(recipient: string, message: string) {
  // Using SendGrid
  const sgMail = require('@sendgrid/mail');
  sgMail.setApiKey(process.env.SENDGRID_API_KEY);
  return sgMail.send({ to: recipient, from: 'notifications@company.com', subject: 'Notification', text: message });
}

// order-service/src/lib/emailClient.ts
export async function dispatchOrderEmail(customerEmail: string, orderDetails: object) {
  // Using AWS SES
  const ses = new AWS.SES();
  return ses.sendEmail({
    Destination: { ToAddresses: [customerEmail] },
    Message: { Body: { Html: { Data: buildEmailHtml(orderDetails) } }, Subject: { Data: 'Order Update' } },
    Source: 'orders@company.com',
  }).promise();
}
```

**Đáp án:**

**Dysfunction được reveal:**

Ba services, ba email implementations khác nhau (Nodemailer, SendGrid, AWS SES), ba từ khác nhau cho cùng operation (sendEmail, sendNotificationEmail, dispatchOrderEmail), ba from addresses khác nhau. Đây là classic symptom của **lack of cross-team technical alignment** và **no shared standards**.

Implications của dysfunction:

- Nếu cần thay đổi email provider (cost, deliverability), phải update 3 places
- Email logging/tracking không centralized — không có single view về emails sent
- Unsubscribe/compliance logic (GDPR, CAN-SPAM) có thể được implement inconsistently hoặc missing ở một số places
- Rate limiting, bounce handling, không consistent
- Engineers joining bất kỳ team nào phải learn different email abstraction

**Root cause:**
Teams built in isolation. Không có mechanism để discover "team B đã build này rồi" trước khi team A build. Không có shared standards cho "how we do email at this company."

**Intervention:**

1. **Immediate (không cần RFC):** Create `@company/email-client` shared package với unified interface. Implement adapter pattern — under the hood có thể dùng SendGrid (tốt nhất cho deliverability), nhưng interface thống nhất.

2. **Short-term:** RFC hoặc Tech Radar entry: "SendGrid là standard email provider, @company/email-client là required wrapper. Alternatives require RFC justification."

3. **Process fix (root cause):** Cross-team weekly sync hoặc Tech Radar review mỗi quarter — nơi teams share "chúng ta đang plan build X" trước khi build. Catch duplications before they happen.

4. **Migration:** Branch by Abstraction — từng service migrate sang shared client, không big bang. Auth service first (simplest), sau đó notification, sau đó order service.

**Đây không phải code quality problem. Đây là organizational problem với code as symptom.**
