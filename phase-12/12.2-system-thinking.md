# Phase 12.2 — System Thinking

> **Độ ưu tiên:** 🟢 — Tất cả 4 nội dung con đều Staff/Principal territory.
> **Định hướng bài này:** System Thinking không phải về distributed systems hay system design interview. Đây là về **cách một Staff/Principal engineer nhìn nhận toàn bộ sociotechnical system** — code, con người, process, business constraints — và đưa ra decisions ở scope rộng hơn một feature hay một team.

## 1. Buy vs Build vs Open Source

### 1.1 — Tại sao đây là một trong những quyết định quan trọng nhất

Mọi engineer giỏi đều có reflex "build it ourselves" — hiểu problem sâu hơn ai, control hoàn toàn, không phụ thuộc vendor. Reflex này đúng trong nhiều trường hợp. Nhưng ở Staff level, bạn phải biết khi nào reflex đó là trap.

Ví dụ thật:

```
2019: Team quyết định tự build search infrastructure
      thay vì dùng Elasticsearch.
      Lý do: "Chúng ta có use case đặc biệt."

2021: Search infrastructure chiếm 30% engineering bandwidth
      để maintain. 2 senior engineers gần như full-time
      chỉ để keep nó running.

      Trong khi đó: Algolia, Typesense, Elasticsearch đều
      đã có feature set đủ cho use case của họ từ 2019.

Cost: 2 năm × 2 engineers × $150k/year = $600k+
vs. Algolia plan: $500/month = $12k/2 years
```

Đây không phải trường hợp ngoại lệ. Nó cực kỳ phổ biến.

### 1.2 — Decision Framework: Buy vs Build vs Open Source

Không có formula tuyệt đối, nhưng có một set of questions để think through:

```
┌─────────────────────────────────────────────────────────────────┐
│              Buy vs Build vs Open Source Decision Tree          │
│                                                                 │
│  1. Đây có phải core competency của công ty không?             │
│     ├── YES → Lean toward BUILD                                 │
│     └── NO  → Continue to question 2                           │
│                                                                 │
│  2. Có solution nào ngoài thị trường đáp ứng 80%+ requirements?│
│     ├── NO  → Probably BUILD                                    │
│     └── YES → Continue to question 3                           │
│                                                                 │
│  3. Yêu cầu về customization, security, data residency?        │
│     ├── STRICT → Open source (self-hosted) hoặc BUILD          │
│     └── FLEXIBLE → BUY (SaaS)                                  │
│                                                                 │
│  4. Total cost comparison (3-5 year horizon)?                   │
│     ├── Build cheaper → BUILD (if team has capacity)           │
│     └── Buy cheaper   → BUY                                    │
│                                                                 │
│  5. Vendor risk acceptable?                                     │
│     ├── YES → BUY                                               │
│     └── NO  → Open source self-hosted                          │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 — Hidden costs khi tính toán

Khi compare "build vs buy", engineers thường chỉ tính **initial build cost**. Đây là sai lầm nghiêm trọng.

**True cost of building:**

```
True Cost = Build Cost + Maintenance + Opportunity Cost

Build Cost:
├── Initial implementation (thường estimate thấp hơn thực tế 2-3x)
├── Testing và QA
└── Documentation

Maintenance (recurring, often underestimated):
├── Bug fixes
├── Security patches
├── Performance tuning
├── Keeping up with dependencies
├── Oncall/incident response
└── Onboarding new engineers

Opportunity Cost:
└── Những feature product bị delayed vì bandwidth dành cho infrastructure
```

**True cost of buying (SaaS):**

```
True Cost = License + Integration + Vendor Risk

License: Monthly/annual fee × years

Integration:
├── Initial integration work
├── Ongoing API compatibility maintenance
└── Data migration cost (nếu switch vendor)

Vendor Risk:
├── Price increase risk
├── Feature deprecation risk
├── Vendor going out of business
└── Data portability / lock-in
```

### 1.4 — Open Source: Middle path với hidden complexity

Open source thường được xem là "free" — đây là misconception nguy hiểm.

**Open source là "free như puppies, không phải free như beer."**

Khi bạn self-host Elasticsearch, Kafka, Temporal, bạn đang "mua" source code miễn phí nhưng **paying với engineering time**:

```
Self-hosted Kafka cluster:
├── Initial setup: 2 weeks (nếu chưa có kinh nghiệm: 4-6 weeks)
├── Ongoing operations:
│   ├── Monitoring và alerting setup
│   ├── Capacity planning
│   ├── Upgrade management
│   ├── Disaster recovery testing
│   └── Security patching
└── Incident response khi cluster có vấn đề
```

**Heuristic cho open source decision:**

- Nếu công ty không có **dedicated platform/infra team**, self-hosting complex distributed systems (Kafka, Elasticsearch, Cassandra) thường là bad idea
- Managed cloud versions (MSK, OpenSearch, Astra) thường worth the cost
- Simple open source libraries (utils, parsers, formatters): ngược lại — usually fine, low maintenance burden

### 1.5 — "Not Invented Here" syndrome vs legitimate reasons to build

**NIH syndrome:** Build because "we can do it better" hoặc vì engineers muốn interesting problems, không phải vì có business reason.

**Legitimate reasons to build:**

1. **Core differentiation:** Nếu bạn là một search company, tự build search engine là đúng. Nếu bạn là một e-commerce company, đừng build search engine.

2. **Unique constraints không ai solve:** Regulatory requirements, scale requirements, latency requirements ở extreme end.

3. **Existing solutions are genuinely bad fit:** Sau khi thực sự evaluate alternatives, không phải sau khi assume.

4. **Onboarding cost của third-party cao hơn build:** Đôi khi API của một service quá complex, learning curve quá cao cho use case đơn giản.

**Rule of thumb:**

> Nếu bạn đang build something mà có thể describe bằng một category name (search, auth, payments, email, logging, monitoring, feature flags), rất likely có solution tốt ngoài thị trường rồi. Prove yourself wrong trước khi build.

## 2. Incremental Migration Strategies

### 2.1 — Tại sao incremental migration là skill quan trọng

Bất kỳ codebase nào đủ lớn và đủ lâu đời đều có accumulated decisions cần thay đổi. Problem: bạn không thể stop the world để rewrite.

"Big Bang rewrite" — stop everything, rewrite từ đầu, ship khi xong — là một trong những cách phổ biến nhất để fail:

```
Classic Big Bang Rewrite failure pattern:
├── Month 1-3: Excitement, good progress
├── Month 4-6: Discover edge cases cũ không được port
├── Month 6-9: "Old system" vẫn getting features, new system falls behind
├── Month 9-12: Scope creep vì "while we're at it..."
├── Month 12+: "We'll ship when it's ready"
└── Month 18+: Either abandoned or shipped with new set of problems
```

Incremental migration là alternative: **migrate piece by piece, keep system working at all times**.

### 2.2 — The Strangler Fig Pattern

Named after strangler fig trees — cây leo bắt đầu từ ngọn cây cũ, dần dần phát triển xuống và bao quanh, cuối cùng cây cũ chết và cây mới đứng độc lập.

```
Phase 1: Legacy handles everything
┌──────────────────────────────────┐
│           LEGACY SYSTEM          │
│   Feature A | Feature B | ...    │
└──────────────────────────────────┘
          ↑ all traffic

Phase 2: Route new features to new system
┌──────────┐     ┌────────────────┐
│  ROUTER  │────►│   NEW SYSTEM   │
│          │     │  Feature A     │
│          │────►│────────────────│
│          │     │ LEGACY SYSTEM  │
└──────────┘     │ Feature B | ...│
                 └────────────────┘

Phase 3: Migrate features one by one
┌──────────┐     ┌────────────────┐
│  ROUTER  │────►│   NEW SYSTEM   │
│          │     │ Feature A      │
│          │     │ Feature B      │
│          │────►│────────────────│
│          │     │ LEGACY SYSTEM  │
└──────────┘     │ Feature C | ..│
                 └────────────────┘

Phase 4: Legacy retired
┌──────────┐     ┌────────────────┐
│  ROUTER  │────►│   NEW SYSTEM   │
│          │     │ Feature A      │
│          │     │ Feature B      │
│          │     │ Feature C      │
└──────────┘     └────────────────┘
```

**Ví dụ thực tế: Migrate từ Pages Router sang App Router (Next.js)**

```
Không làm:
└── Tạo branch "app-router-migration"
    Rewrite toàn bộ trong 2 tháng
    Merge vào main
    [Disaster]

Làm đúng với Strangler Fig:
├── Week 1: Setup App Router bên cạnh Pages Router
│   (/app directory coexists với /pages)
│
├── Week 2-3: Migrate /about, /contact (static pages, low risk)
│   Validate behavior identical.
│
├── Week 4-6: Migrate /blog (read-only, medium complexity)
│   A/B test traffic nếu cần.
│
├── Week 7-10: Migrate /products (data fetching complexity)
│   Feature flag to control rollout.
│
├── Week 11-14: Migrate /checkout (highest risk)
│   Shadow mode: run both, compare responses.
│   Gradual rollout: 1% → 10% → 50% → 100%
│
└── Week 15: Delete /pages, remove old dependencies
```

### 2.3 — Feature Flags cho safe migration

Feature flags là critical infrastructure cho incremental migration:

```javascript
// Flag-driven migration: không phải feature toggle
// mà là migration control plane

import { getFlag } from '@company/feature-flags';

async function getProductData(productId: string) {
  const useNewDataLayer = await getFlag('use-new-product-data-layer', {
    userId: currentUser.id,
    // Rollout strategy được control ở flag service
    // không phải hardcode ở đây
  });

  if (useNewDataLayer) {
    // New implementation
    return productService.getProduct(productId);
  }

  // Legacy implementation — vẫn chạy được
  return legacyProductApi.fetch(productId);
}
```

**Rollout strategy thường dùng:**

```
1% internal users
    ↓ (24h, no incidents)
5% of users (canary)
    ↓ (48h, monitor metrics)
25% of users
    ↓ (48h)
50% → 100%
    ↓ (cleanup: remove flag + old code)
```

### 2.4 — The Branch by Abstraction pattern

Dùng khi cần thay thế một dependency deep inside codebase, không phải ở edge (routing level).

```
Step 1: Tạo abstraction layer (interface/wrapper)

// Trước: direct dependency
import { sendEmail } from 'old-email-library';

// Sau step 1: abstraction
interface EmailService {
  send(to: string, subject: string, body: string): Promise<void>;
}

class LegacyEmailAdapter implements EmailService {
  async send(to, subject, body) {
    // wrap old library
    return sendEmail({ to, subject, body });
  }
}

// Toàn bộ code giờ dùng EmailService interface
// không phải old-email-library trực tiếp

────────────────────────────────────────────

Step 2: Build new implementation behind same interface

class NewEmailAdapter implements EmailService {
  async send(to, subject, body) {
    // new implementation (SendGrid, Resend, etc.)
    return newEmailClient.send({ to, subject, body });
  }
}

────────────────────────────────────────────

Step 3: Switch implementations via dependency injection

// DI container hoặc feature flag controls which adapter
const emailService: EmailService = featureFlag('new-email-provider')
  ? new NewEmailAdapter()
  : new LegacyEmailAdapter();

────────────────────────────────────────────

Step 4: Validate, cleanup

// Khi new implementation stable:
// 1. Remove LegacyEmailAdapter
// 2. Remove feature flag
// 3. Remove old-email-library dependency
```

**Benefit:** Bất kỳ lúc nào trong quá trình migration, bạn có thể rollback bằng cách flip flag. Không cần revert code.

### 2.5 — Database migration strategies

Database migration là phần nguy hiểm nhất của bất kỳ migration nào vì:

- Data không thể rollback dễ dàng
- Schema changes thường require downtime nếu làm naive

**Expand-Contract pattern (Blue-Green schema migration):**

```
Scenario: Rename column `user_name` → `username`

NAIVE (requires downtime):
ALTER TABLE users RENAME COLUMN user_name TO username;
-- All code reading user_name now broken until deploy

EXPAND-CONTRACT (zero downtime):

Phase 1 — EXPAND:
ALTER TABLE users ADD COLUMN username VARCHAR(255);
-- Deploy code that writes to BOTH columns
-- Backfill: UPDATE users SET username = user_name;

Phase 2 — MIGRATE READS:
-- Deploy code that reads from NEW column (username)
-- Still writes to both

Phase 3 — CONTRACT:
-- Deploy code that only writes to NEW column
ALTER TABLE users DROP COLUMN user_name;
-- Cleanup complete
```

```
Timeline:
─────────────────────────────────────────────────────►
│         │              │              │
Deploy    Deploy         Deploy         Drop
expand    dual-write     read-new       old column
schema    code           code
          +backfill

Gap giữa mỗi phase: đủ để confident data correct
(thường ít nhất 24-48h production traffic)
```

## 3. Backward Compatibility Planning

### 3.1 — Tại sao backward compatibility matter

Nếu bạn chỉ làm việc trong một monolith và deploy toàn bộ cùng lúc, backward compatibility ít quan trọng hơn.

Nhưng trong hầu hết production systems:

```
Scenario: API breaking change không có backward compat planning

Before deploy:
├── Server: v2.0 (new API shape)
└── Client (mobile): v1.8 (old API expectations)
    // Mobile users không update app ngay
    // Có users dùng app version từ 6 tháng trước

After deploy:
└── Mobile app v1.8 calls /api/products
    Server responds với new shape
    App crashes hoặc shows wrong data

    → 1-star reviews, support tickets, revenue impact
```

Backward compatibility là **contract** bạn giữ với consumers của API/library của mình.

### 3.2 — API Versioning Strategies

**Strategy 1: URL versioning**

```
/v1/products     ← deprecated nhưng vẫn hoạt động
/v2/products     ← current
/v3/products     ← new (in rollout)
```

Pros: Explicit, easy to route, consumers biết rõ version họ dùng
Cons: URL proliferation, code duplication nếu không careful

**Strategy 2: Header versioning**

```http
GET /products
API-Version: 2024-01-15
```

Pros: Clean URLs
Cons: Less visible, easier to forget

**Strategy 3: Additive changes only (no versioning needed)**

Approach này powerful nhất nếu discipline cao:

```
Rules:
✅ Add new fields to response
✅ Add new optional parameters
✅ Add new endpoints
✅ Add new enum values (với care)

❌ Remove fields from response
❌ Change field types
❌ Change field semantics (meaning, not just type)
❌ Make optional params required
❌ Change URL structure
❌ Change HTTP method
```

Với rules này, v1 client vẫn works với v2 server vì tất cả changes là additive.

### 3.3 — Deprecation process

Khi phải make breaking change, đây là process professional:

```
Timeline: Breaking change deprecation

T-0: Announce deprecation
├── Add deprecation warning vào API response headers:
│   Deprecation: true
│   Sunset: Sat, 01 Jun 2025 00:00:00 GMT
│   Link: <https://docs.example.com/migration/v2>; rel="deprecation"
│
├── Update documentation
├── Email/notify all known API consumers
└── Add to changelog

T+3 months: Reminder notifications
├── API response warning tăng visibility
└── Direct outreach tới high-traffic consumers

T+6 months: Sunset date
└── Old API returns 410 Gone (không phải 404)
    với body: {
      "error": "API version deprecated",
      "migration_guide": "https://docs.example.com/migration/v2",
      "sunset_date": "2025-06-01"
    }
```

**Sunset period thực tế:**

- Internal API (same company): 4-8 weeks đủ
- External API, small developer base: 3-6 months
- External API, large developer base, mobile apps: 6-12 months (mobile release cycles chậm)
- Public API với millions of users (Twitter, GitHub): 12-24 months

### 3.4 — Semantic Versioning và khi nào dùng

SemVer: `MAJOR.MINOR.PATCH`

```
PATCH (1.0.0 → 1.0.1): Bug fixes, no API changes
MINOR (1.0.0 → 1.1.0): New features, backward compatible
MAJOR (1.0.0 → 2.0.0): Breaking changes
```

**Ứng dụng cho public libraries:**

```javascript
// package.json của consumers:
{
  "dependencies": {
    "your-library": "^1.2.0"  // ^ = accept MINOR and PATCH bumps
    // Consumers biết MAJOR bump = phải đọc migration guide
  }
}
```

**Thực tế phũ phàng:** SemVer chỉ hoạt động nếu maintainers discipline. "Accidental breaking changes" trong MINOR releases là nguồn gốc của vô số incidents.

**Hyrum's Law** — quan trọng để hiểu:

> "Với đủ users của một API, tất cả observable behaviors của implementation sẽ được depend on bởi ai đó, dù spec có nói gì đi nữa."

Nghĩa là: dù bạn viết doc rõ ràng "behavior này là undefined, không rely on", users sẽ rely on nó. Khi bạn change "internal" behavior, ai đó sẽ break.

Implication cho planning: **assume everything is public API**, kể cả những thứ bạn không intend as public.

## 4. Performance Budgets và Monitoring

### 4.1 — Performance budget là gì và tại sao cần

Performance budget là **explicit constraints** được set trước khi build, không phải sau khi ship và nhận complaints.

Không có performance budget: Performance là "vague concern" — mọi người biết là quan trọng nhưng không ai chịu trách nhiệm cụ thể. PR vẫn được merge dù bundle size tăng 20KB vì "không có rule nói không được".

Có performance budget: Performance là **measurable, enforceable constraint** giống như test coverage hay type safety.

```
Example: E-commerce Product Page Budget

Core Web Vitals targets:
├── LCP (Largest Contentful Paint): < 2.5s (P75)
├── INP (Interaction to Next Paint): < 200ms
└── CLS (Cumulative Layout Shift): < 0.1

JavaScript bundle limits:
├── Initial JS (critical path): < 150KB gzipped
├── Total JS (including lazy): < 400KB gzipped
└── Per-route code split chunk: < 50KB gzipped

Network:
├── Total page weight (First load): < 1MB
├── Images: < 200KB per image (WebP/AVIF)
└── Fonts: < 50KB total (subsetted)

Runtime:
├── Time to Interactive: < 3.5s (3G, mid-range mobile)
└── Main thread blocking time: < 50ms per task
```

### 4.2 — Enforcing budgets trong CI/CD

Budget không có enforcement là aspirational document, bukan constraint.

**Bundlesize checks:**

```javascript
// bundlesize.config.js
module.exports = {
  files: [
    {
      path: './dist/static/js/main.*.js',
      maxSize: '150 kB',
      // CI fail nếu vượt budget
    },
    {
      path: './dist/static/js/*.chunk.js',
      maxSize: '50 kB',
    },
    {
      path: './dist/static/css/main.*.css',
      maxSize: '30 kB',
    },
  ],
};
```

**Lighthouse CI:**

```yaml
# .github/workflows/ci.yml
- name: Run Lighthouse CI
  run: |
    npm install -g @lhci/cli
    lhci autorun
  env:
    LHCI_GITHUB_APP_TOKEN: ${{ secrets.LHCI_GITHUB_APP_TOKEN }}

# lighthouserc.js
module.exports = {
  ci: {
    assert: {
      assertions: {
        'categories:performance': ['error', { minScore: 0.9 }],
        'first-contentful-paint': ['error', { maxNumericValue: 2000 }],
        'largest-contentful-paint': ['error', { maxNumericValue: 2500 }],
        'cumulative-layout-shift': ['error', { maxNumericValue: 0.1 }],
        'total-byte-weight': ['error', { maxNumericValue: 1000000 }],
      },
    },
  },
};
```

Khi Lighthouse CI chạy trên mỗi PR:

- PR vượt budget → CI fail → không thể merge
- Engineer phải optimize trước khi merge
- Budget degradation được caught ở PR level, không phải sau khi ship

### 4.3 — Real User Monitoring (RUM) vs Synthetic Monitoring

Hai loại monitoring bổ sung cho nhau, không phải thay thế:

```
┌─────────────────────────────────────────────────────────────────┐
│              Synthetic vs RUM                                   │
├────────────────────────┬────────────────────────────────────────┤
│  SYNTHETIC             │  REAL USER MONITORING (RUM)            │
├────────────────────────┼────────────────────────────────────────┤
│  Controlled environment│  Real devices, real networks           │
│  Consistent baseline   │  Variable conditions                   │
│  Catches regressions   │  Shows actual user experience          │
│  Pre-production        │  Post-production                       │
│                        │                                        │
│  Tools:                │  Tools:                                │
│  • Lighthouse CI       │  • web-vitals library                  │
│  • WebPageTest         │  • Datadog RUM                         │
│  • k6 (load testing)   │  • New Relic                           │
│                        │  • SpeedCurve                          │
│  Answers:              │  • Google CrUX                         │
│  "Did we regress?"     │                                        │
│                        │  Answers:                              │
│                        │  "Are real users suffering?"           │
└────────────────────────┴────────────────────────────────────────┘
```

**RUM implementation với web-vitals library:**

```javascript
// vitals.ts — report Core Web Vitals tới analytics

import { onCLS, onINP, onLCP, onFCP, onTTFB } from 'web-vitals';

type VitalMetric = {
  name: string;
  value: number;
  rating: 'good' | 'needs-improvement' | 'poor';
  navigationType: string;
};

function reportVital(metric: VitalMetric) {
  // Send to your analytics endpoint
  fetch('/api/vitals', {
    method: 'POST',
    body: JSON.stringify({
      metric: metric.name,
      value: metric.value,          // Raw value (ms, score)
      rating: metric.rating,        // 'good' | 'needs-improvement' | 'poor'
      page: window.location.pathname,
      connection: navigator.connection?.effectiveType, // '4g', '3g', etc.
      deviceMemory: navigator.deviceMemory,
      timestamp: Date.now(),
    }),
    headers: { 'Content-Type': 'application/json' },
  });
}

// Register all vitals
onCLS(reportVital);
onINP(reportVital);
onLCP(reportVital);
onFCP(reportVital);
onTTFB(reportVital);
```

### 4.4 — Performance budgets theo business context

Không phải mọi performance budget đều giống nhau. Budget phải tied to business outcomes:

```
E-commerce (direct revenue impact):
└── 100ms improvement in LCP → ~1% conversion rate improvement
    (Amazon, Walmart internal studies)
    Budget: aggressive, strictly enforced

Internal admin tool:
└── Performance annoying nhưng không gây revenue loss
    Budget: relaxed, aspirational

B2B SaaS với enterprise clients:
└── SLA commitments, enterprise clients expect professional experience
    Budget: moderate, tied to SLA commitments
```

**The Performance Budget Conversation với Product/Management:**

```
Engineer nói:
"Chúng ta cần maintain LCP < 2.5s."

Manager nghe:
"Engineer muốn thêm constraints mà không rõ lý do."

Engineer nên nói:
"Mỗi 100ms tăng trong page load time là ~0.5% conversion drop
theo A/B test của chúng ta tháng trước. Chúng ta đang ở 2.8s.
Nếu chúng ta không have LCP budget, tôi dự đoán chúng ta sẽ
hit 3.5s vào Q3 khi các features mới ship — tương đương ~$200k
annual revenue impact dựa trên current conversion rates."
```

### 4.5 — Performance monitoring dashboard (what to track)

### Câu 1: Tại sao "Big Bang Rewrite" thường thất bại, và điều gì xảy ra về mặt organizational dynamics khi nó fail?

**Đáp án:**

Big Bang Rewrite thất bại do tổ hợp của technical và organizational factors:

**Technical:**

Rewrite bắt đầu từ một clean slate nhưng legacy system đã tích lũy years of edge case handling, bug fixes, và implicit business logic không được documented. Người viết rewrite không biết những implicit logic này tồn tại cho đến khi users complain. Classic quote: "It took us 2 years to rewrite, and the new system had all the same bugs plus new ones."

Thêm vào đó, trong thời gian rewrite đang diễn ra, legacy system tiếp tục nhận features và bug fixes (business không stop). New system phải "catch up" liên tục — một đích di chuyển.

**Organizational:**

1. **Scope creep:** Vì đang rewrite anyway, mọi người add requirements ("Trong khi chúng ta đang ở đó, hãy thêm feature X"). Scope expand, timeline slip.

2. **Morale degradation:** Rewrite kéo dài → engineers loses motivation → key people leave → knowledge walks out the door.

3. **Business pressure:** Sau 6-12 tháng không ship, management bắt đầu đặt câu hỏi về value. Engineer team phải justify ongoing investment trong khi không có user-visible output.

4. **Sunk cost trap:** Đã invested nhiều, rất khó admit failure và stop. Project tiếp tục dù signals rõ ràng là không ổn.

**Incremental migration tránh tất cả những điều trên:** System luôn shippable, changes được validated incrementally, legacy knowledge được preserved và migrated piece by piece, không phải relearned.

### Câu 2: Hyrum's Law ảnh hưởng thế nào tới backward compatibility planning? Đưa ra ví dụ cụ thể.

**Đáp án:**

Hyrum's Law: "Với đủ users, tất cả observable behaviors của implementation sẽ được depend on bởi ai đó."

Điều này có nghĩa là **implementation details bạn không intend as part of API sẽ trở thành API** theo thời gian.

**Ví dụ cụ thể:**

Bạn có một endpoint trả về danh sách users. Implementation hiện tại vô tình sort users theo `created_at` DESC vì query không có explicit ORDER BY (database internal behavior). Documentation không mention ordering.

Một số consumers bắt đầu depend on việc first user trong response là newest user — vì "nó luôn như vậy". Không ai document điều này, không ai explicitly test nó.

Sau đó bạn add index, query plan thay đổi, default ordering không còn là `created_at`. Consumers bị break mặc dù bạn không change API.

**Implication cho planning:**

1. Khi thay đổi "internal" implementation, assume có ai đó depend on current behavior. Test widely before removing.

2. Nếu muốn có flexibility để change behavior, phải **explicitly document it as undefined/unspecified** từ rất sớm. "Response order is undefined and may change" phải được document trước khi users start depend on nó.

3. Large API surfaces với many users = almost nothing can be changed safely. Google's internal APIs được governed bởi nguyên tắc này — họ phải maintain backward compat cho internal code từ 15 năm trước vì teams khắp nơi trong Google depend on undocumented behaviors.

4. Sunset period cần dài hơn khi user base lớn — chính xác vì nhiều hidden dependencies hơn.

### Câu 3: Đoạn code migration sau đây có vấn đề gì?

```javascript
// Week 1: Add feature flag
async function getProductList() {
  const useNewApi = await getFlag('new-product-api');

  if (useNewApi) {
    return newProductService.list();
  }
  return legacyProductApi.getProducts();
}

// Week 3: Flag đã 100% rollout, xóa legacy code
async function getProductList() {
  return newProductService.list();
}

// package.json: xóa legacy-product-api dependency
```

**Đáp án:**

Có 3 vấn đề theo thứ tự severity:

**Vấn đề 1: Cleanup quá nhanh, không có validation period**

Chỉ 3 weeks từ lúc flag on đến lúc xóa legacy code là quá ngắn với production system. Best practice: sau khi flag hit 100%, **observe ít nhất 1-2 full business cycles** (thường 1-2 tuần minimum, đôi khi 1 tháng nếu có seasonal traffic) trước khi xóa fallback.

Nếu có incident sau cleanup — deploy bị rollback, nhưng legacy code đã bị xóa — không có easy fallback nữa.

**Vấn đề 2: Không có rollback plan explicit**

Trước khi flag 100%, có rollback plan rõ ràng: turn flag off. Sau khi xóa legacy code, rollback plan là: revert commit + redeploy. Cái sau chậm hơn đáng kể trong incident.

Fix: Keep legacy code trong một full sprint sau 100% rollout. Chỉ xóa khi confident không cần rollback.

**Vấn đề 3: Flag value được fetch trong hot path mà không có caching**

`await getFlag('new-product-api')` trong mỗi request có thể là network call tới flag service. Trong một API endpoint được gọi nhiều lần, đây là added latency + potential failure point (nếu flag service down, mọi request fail).

Flag SDKs tốt cache flag values locally và sync in background. Phải verify `getFlag` implementation có caching hay không. Nếu không, wrap với local cache:

```javascript
// Với TTL cache
const flagCache = new Map<string, { value: boolean; expiry: number }>();

async function getCachedFlag(name: string): Promise<boolean> {
  const cached = flagCache.get(name);
  if (cached && cached.expiry > Date.now()) {
    return cached.value;
  }
  const value = await getFlag(name);
  flagCache.set(name, { value, expiry: Date.now() + 30_000 }); // 30s TTL
  return value;
}
```
