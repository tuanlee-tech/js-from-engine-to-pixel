# Phase 12.1 — Technical Leadership

> **Độ ưu tiên:** 🟢 — Staff/Principal territory. Tất cả 4 nội dung con đều là 🟢.
> **Lưu ý về approach:** Phase này khác hoàn toàn với Phase 1–11. Không có V8 internals, không có memory layout. Cơ chế ở đây là **organizational mechanics** — cách một kỹ sư giỏi về technical nhưng fail ở Staff level vì không biết cách operate trong tổ chức. Bài này sẽ đi thẳng vào đó.

---

## Tại sao Phase 12 tồn tại trong roadmap này?

Có một cái bẫy rất phổ biến: bạn học hết Phase 1–11, code rất giỏi, debug rất nhanh, nhưng không lên được Staff. Không phải vì thiếu kỹ năng kỹ thuật. Mà vì **scope of impact** còn quá nhỏ.

Senior Engineer: impact trong team, giải quyết task được giao, làm tốt những gì có sẵn.

Staff Engineer: **xác định** task nào cần làm, **tạo ra** context để người khác làm được, impact across teams.

4 thứ trong 12.1 là tools để mở rộng scope đó.

## 1. RFC (Request for Comments) Process

### 1.1 — Cơ chế thật

RFC không phải là "viết document rồi xin approval". Đó là cách nhiều người hiểu sai và dẫn tới RFC trở thành bureaucracy vô nghĩa.

RFC thật sự là một **alignment protocol** — một cơ chế để move fast mà không break things ở scale.

Tại sao cần RFC?

Khi team còn 3 người: bạn Slack nhau, align trong 5 phút, ship. Không cần RFC.

Khi team là 30 người, 5 teams: nếu không có RFC, một trong các tình huống sau sẽ xảy ra:

- Hai team xây hai thứ solve cùng problem, discover sau khi cả hai đã ship
- Một người thay đổi một shared library, không ai biết, 3 team bị break
- Decision được làm trong "corridor conversation" — người không ở đó không biết context, sau 6 tháng không ai nhớ tại sao design đó được chọn
- Senior engineer unilaterally make a big call, people feel excluded, morale suffers

RFC giải quyết tất cả những thứ trên bằng cách **externalize decision-making** — đưa nó ra khỏi đầu một người, vào một artifact mà nhiều người có thể react.

### 1.2 — RFC Process hoạt động như thế nào trong thực tế

```
┌─────────────────────────────────────────────────────────────────┐
│                     RFC Lifecycle                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. DRAFT           2. COMMENT          3. DECISION            │
│                                                                 │
│  Tác giả viết  →   Share rộng     →    Owner quyết định       │
│  problem + options  Collect feedback    Accepted / Rejected /  │
│                     Async hoặc          Withdrawn              │
│                     sync meeting                               │
│                          │                                      │
│                          ↓                                      │
│                   4. IMPLEMENTATION     5. ARCHIVE             │
│                                                                 │
│                   RFC số → ADR          RFC thành lịch sử      │
│                   (nếu accepted)        Reference mãi mãi      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**RFC không phải approval gate.** Đây là điểm mà nhiều công ty implement sai. Nếu RFC biến thành "mọi thứ phải được approve mới được làm" → bottleneck, engineers sẽ ghét RFC.

RFC đúng nghĩa: tác giả RFC vẫn là **decision owner**. RFC chỉ là cơ chế để họ thu thập input trước khi quyết định. Họ có thể reject feedback và vẫn proceed — nhưng họ phải **acknowledge** feedback đó và giải thích tại sao họ disagree.

### 1.3 — RFC Template (Production-Grade)

Đây là template được dùng ở nhiều công ty lớn (Facebook, Rust lang, IETF đều có variant):

```markdown

## Summary (1-2 câu)

Đề xuất migrate Product API từ REST sang GraphQL để giải quyết
vấn đề over-fetching và N+1 requests đang gây ra 40% unnecessary
database load.

---

## Motivation / Problem

<!-- QUAN TRỌNG: Mô tả vấn đề, không phải solution. -->
<!-- Người đọc phải đồng ý với problem trước khi evaluate solution. -->

Hiện tại, mobile client cần load product detail page phải make
5 separate REST calls:

- GET /products/:id
- GET /products/:id/images
- GET /products/:id/reviews?limit=5
- GET /users/:sellerId
- GET /categories/:categoryId

Điều này dẫn tới:

1. 5 round trips → ~500ms added latency trên 3G
2. Over-fetching: /products/:id trả về 40 fields, client chỉ dùng 8
3. Mobile team phải tự orchestrate data fetching, dễ race condition

**Metric hiện tại:**

- Median page load (product detail): 1.2s
- 95th percentile: 3.4s
- Database queries per product page load: ~45 (N+1 từ reviews)

---

## Proposed Solution

Implement GraphQL layer (Apollo Server) trước existing REST services.
GraphQL sẽ là BFF (Backend for Frontend) — không replace backend services,
chỉ aggregate chúng.
```

Mobile/Web → GraphQL → REST Services → Database
↓
(batching, DataLoader,
field selection)

```

**Implementation plan:**
- Phase 1: Setup Apollo Server, schema cho Product domain (2 weeks)
- Phase 2: Migrate mobile app (4 weeks)
- Phase 3: Migrate web app (3 weeks)
- Phase 4: Deprecate old REST endpoints (2 weeks after migration done)

### Option B: GraphQL Federation
Pros: More scalable, teams own their schemas
Cons: Complex setup, overkill cho team size hiện tại (15 engineers)
**Rejected:** Complexity không justify benefit ở scale hiện tại.

### Option C: BFF với REST (tự viết aggregation layer)
Pros: Familiar technology, no new paradigm
Cons: Sẽ reinvent nhiều thứ GraphQL đã có (batching, introspection, codegen)
**Rejected:** Higher maintenance cost dài hạn.

### Option D: gRPC
Pros: Performant, strongly typed
Cons: Browser support kém, không có rich ecosystem cho web clients
**Rejected:** Không phù hợp cho browser clients.
```

### 1.4 — RFC anti-patterns phổ biến

**Anti-pattern 1: RFC quá dài trước khi share**

Bạn bỏ 2 tuần viết 10-page RFC cực kỳ detailed → share → người đọc thấy investment quá lớn, ngại reject dù có vấn đề.

Fix: Share Draft sớm, khi còn rough. "Đây là draft, tôi đang explore — input welcome." Khi đã spent ít, feedback dễ tiếp nhận hơn.

**Anti-pattern 2: RFC là rubber stamp**

Decision đã được made privately, RFC chỉ để "formalize". Người comment thật sự sẽ nhận ra và dừng engage.

Fix: RFC chỉ nên bắt đầu khi tác giả thật sự còn undecided, hoặc khi muốn surface blind spots.

**Anti-pattern 3: Không có deadline**

RFC tồn tại 6 tháng không ai đụng tới.

Fix: Mỗi RFC phải có explicit comment deadline. Sau deadline, tác giả make decision và move on.

**Anti-pattern 4: Quá nhiều RFC**

Viết RFC cho mọi thứ, kể cả quyết định nhỏ.

Heuristic: RFC phù hợp khi:

- Decision affects > 1 team
- Khó reverse sau khi implement
- Requires significant engineering investment (> 1 sprint)
- Introduces new technology/pattern vào stack

## 2. Architecture Decision Records (ADR)

### 2.1 — ADR khác RFC như thế nào

RFC là **forward-looking** — "chúng ta sẽ làm gì".
ADR là **backward-looking** — "chúng ta đã làm gì và tại sao".

ADR là **log của decisions đã được made**. RFC có thể (nên) kết thúc bằng một ADR.

Vấn đề ADR giải quyết: **Archeological debugging**.

Bạn join một codebase 3 năm tuổi và thấy một pattern kỳ lạ:

```javascript
// Tại sao lại có wrapper function này?
// Tại sao không dùng fetch trực tiếp?
const apiClient = createApiClient({
  baseURL: process.env.API_URL,
  withCredentials: true,
  // Tại sao timeout là 8000, không phải 5000 hay 10000?
  timeout: 8000,
});
```

Không có ADR: bạn hỏi Slack, người viết code đó đã nghỉ từ lâu. Bạn không dám thay đổi vì không biết context.

Có ADR: bạn tìm `ADR-0015-api-client-wrapper.md`, đọc được:

> "Timeout 8000ms là kết quả sau khi profile thấy P95 latency của auth service là 6.2s dưới peak load. 5000ms timeout gây 15% của requests fail unnecessarily. 10000ms quá dài, người dùng đã abandon. 8000ms là sweet spot."

### 2.2 — ADR Format (Michael Nygard's original + modern additions)

```markdown
# ADR-0015: Sử dụng custom API client wrapper thay vì fetch trực tiếp

**Date:** 2023-03-10
**Status:** Accepted

<!-- Possible statuses: Proposed | Accepted | Deprecated | Superseded by ADR-XXXX -->

**Context:**

Frontend app cần communicate với 12 backend services khác nhau.
Mỗi service có:

- Authentication requirements khác nhau
- Timeout requirements khác nhau
- Error handling conventions khác nhau

Khi mỗi team tự gọi fetch, chúng tôi gặp:

- Inconsistent error handling (một số places throw, một số return null)
- Auth token refresh logic bị duplicate ở 6 places
- Network errors không được log centrally → khó debug production issues

**Decision:**

Tạo một ApiClient class đóng gói:

1. Automatic auth token injection
2. Token refresh với request queuing khi token expired
3. Centralized error normalization
4. Request/response interceptors cho logging
5. Configurable timeout per-service (default 8000ms)

Tất cả network calls phải đi qua client này. Direct fetch chỉ được dùng
cho external URLs không qua auth layer (ví dụ: CDN images).

**Consequences:**

Positive:

- Auth logic tập trung một chỗ — khi auth flow thay đổi, chỉ fix ở 1 nơi
- Consistent error format → UI error handling đơn giản hơn
- Request logs centralized → production debugging dễ hơn

Negative:

- Thêm một layer abstraction — engineers mới phải learn client API
- Potential single point of failure nếu client có bug
- Bundle size tăng ~3KB (minified)

Neutral:

- Teams không còn flexibility để customize individual requests ngoài
  những gì client expose → acceptable trade-off

**Notes:**

Timeout 8000ms: Xem [incident report #234] — auth service P95 latency
dưới peak load đo được 6.2s. 5000ms timeout gây 15% unnecessary failures.

Token refresh logic: Implement request queue để prevent race condition
khi nhiều requests cùng fail với 401 — xem implementation notes ở
src/api/client.ts line 145.
```

### 2.3 — Nơi lưu ADR và cách tổ chức

Convention phổ biến nhất: **docs/decisions/** trong repo chính.

```
project/
├── src/
├── docs/
│   ├── decisions/
│   │   ├── 0001-use-nextjs-for-frontend.md
│   │   ├── 0002-postgresql-as-primary-database.md
│   │   ├── 0003-redis-for-session-storage.md
│   │   ├── 0014-migrate-from-redux-to-zustand.md
│   │   │   (status: Superseded by ADR-0031)
│   │   ├── 0031-migrate-from-zustand-to-react-query-plus-context.md
│   │   └── README.md  ← index của tất cả ADRs
│   └── architecture/
│       └── overview.md
```

Tại sao để trong repo thay vì Confluence/Notion:

- ADR có thể được link vào PR — review code hiểu ngay context
- Versioning tự động — git blame, git log show history của decision
- ADR không bị separated khỏi code theo thời gian
- Search bằng `grep` hay GitHub search

### 2.4 — ADR anti-patterns

**Anti-pattern 1: ADR quá late**

Viết ADR sau khi code đã được deployed, implementation đã fixed. ADR lúc này chỉ là documentation, không còn là decision record.

Fix: ADR nên được viết **trong quá trình** decision-making, ít nhất là cùng lúc với implementation, không phải sau.

**Anti-pattern 2: ADR không có "Alternatives Considered"**

```markdown
# ADR-0007: Dùng Jest

**Decision:** Dùng Jest cho testing.
**Consequences:** Có thể viết tests.
```

Đây không phải ADR. Đây là documentation nói "chúng ta dùng Jest". Không có context, không có trade-off. Người sau không hiểu tại sao không dùng Vitest, Mocha, hoặc không test gì cả.

**Anti-pattern 3: ADR không được update**

ADR-0005 nói "Dùng Redux cho state management" — nhưng codebase đã migrate sang Zustand từ 2022.

Fix: Khi supersede một ADR, update status của ADR cũ: `Status: Superseded by ADR-0031`. Đừng xóa ADR cũ — lịch sử vẫn có giá trị.

## 3. Tech Debt Triage

### 3.1 — Tech debt là gì thật sự (không phải "code xấu")

Ward Cunningham — người đặt ra khái niệm — có ý như sau: Tech debt là metaphor cho trade-off. Khi bạn viết code chạy được nhưng chưa phản ánh đủ understanding của problem, bạn đang "borrow" từ tương lai. Bạn sẽ phải trả "interest" dưới dạng việc code đó khó thay đổi khi understanding tốt hơn.

Vấn đề: khái niệm này bị misuse để justify mọi thứ là "tech debt":

- "Code này xấu" → tech debt
- "Chúng ta không có tests" → tech debt
- "Tôi muốn rewrite cái này" → tech debt
- "Library này đã outdated" → tech debt

Khi mọi thứ đều là tech debt, không thứ gì là ưu tiên.

Senior/Staff engineer cần có mental model **phân loại** tech debt để triage đúng.

### 3.3 — Triage Model cho Staff Engineer

Khi bạn có 20 items trong tech debt backlog và cần prioritize, dùng 3 axes:

```
Priority Score = (Business Impact × Probability of Trigger) / Cost to Fix
```

**Business Impact:** Nếu debt này "triggers" (gây ra vấn đề), impact ra sao?

- 3: Production incident, data loss, security breach
- 2: Slowdown significant feature development
- 1: Inconvenience, minor slowdown

**Probability of Trigger:** Debt này có khả năng gây ra vấn đề không?

- 3: Đang chạm vào area này hàng tuần, growing traffic
- 2: Đôi khi, không consistent
- 1: Legacy code không ai touch

**Cost to Fix:** Effort để address?

- 1: Ít (< 1 sprint)
- 2: Vừa (1-3 sprints)
- 3: Lớn (> 3 sprints)

Ví dụ triage thực tế:

```
Item                                    Impact  Trigger  Cost  Score
─────────────────────────────────────────────────────────────────────
Auth service không có rate limiting       3       2       1    6.0 ← ưu tiên cao
Product search N+1 query (growing)        2       3       2    3.0
Outdated React version (16→18)           2       2       3    1.3
No unit tests on payment module          3       1       3    1.0
Old CSS methodology (no CSS modules)     1       3       1    3.0
```

### 3.4 — Cách present tech debt lên management

Engineer thường fail ở bước này vì present theo kiểu "code xấu cần fix". Management không care về code quality, họ care về business outcomes.

Template framing đúng:

```
SAI:
"Chúng ta cần refactor authentication module vì code đang rất messy
và hard to maintain."

ĐÚNG:
"Authentication module hiện tại không support multi-factor auth —
đây là requirement từ enterprise clients Q2. Estimate 6 weeks để add
MFA nếu giữ code hiện tại, vs 3 weeks nếu refactor trước (1 week).
Đề xuất: 1 week refactor, unblocks MFA + 3 upcoming enterprise features."
```

Ba cách frame debt cho business:

1. **Risk mitigation:** "Nếu không fix, X% khả năng có production incident trong Y months dựa trên current growth rate."

2. **Velocity tax:** "Mỗi feature trong payment area mất 40% longer hơn cần thiết vì architecture hiện tại. Đây là 3 sprints/quarter bị waste."

3. **Opportunity cost:** "Chúng ta không thể implement feature X vì dependency trên legacy system Y. Fix Y unblocks 3 roadmap items."

### 3.5 — "Debt-as-first-class-citizen" model

Model phổ biến ở mature engineering orgs: treat tech debt như feature work, không cần justify riêng.

```
Sprint capacity allocation:
├── 70% — New features
├── 20% — Tech debt (planned)
└── 10% — Unplanned/incidents/hotfixes
```

20% debt allocation không phải idle time. Nó được managed như feature backlog — có owner, có acceptance criteria, có "definition of done".

Staff engineer role: maintain debt backlog này, triage regularly, ensure 20% được spent on highest-leverage items.

## 4. Cross-team Technical Alignment

### 4.1 — Tại sao cross-team alignment là hard problem

Trong một team: mọi người share context, communicate daily, friction thấp.

Across teams: mỗi team có:

- Priorities riêng (có thể conflict)
- Mental models khác nhau về system
- "Local optima" — decision tốt cho team A có thể bad cho system tổng thể
- Communication channels ít hơn, slower

Ví dụ về cross-team misalignment:

```
Team A (Web)                    Team B (Mobile)
────────────────                ────────────────
Quyết định dùng                 Đang build
pagination cho                  infinite scroll —
Product List API                cần cursor-based
                                pagination

        ↓
  API design conflict
  discover 2 weeks
  trước launch
```

### 4.2 — Mechanisms for Cross-team Alignment

**Mechanism 1: Shared Design Documents**

Trước khi build API, viết API contract document và **share với tất cả consumers** trước khi implement. Đây là horizontal RFC — không phải propose technical change mà là propose interface.

```markdown
# API Contract: Product List v2

**Proposed by:** Team API Platform
**Consumers:** Web (Team A), Mobile (Team B), Partner integrations (Team C)
**Review deadline:** 2024-02-01
**Status:** DRAFT

## Proposed Interface

GET /v2/products

Query params:

- limit: number (1-100, default: 20)
- cursor: string (opaque cursor, base64 encoded)
- sort: "relevance" | "price_asc" | "price_desc" | "newest"

Response:
{
"data": Product[],
"pagination": {
"nextCursor": string | null,
"hasMore": boolean,
"total": number // Approximate for performance
}
}

## Open Questions for Consumers

1. Mobile cần total count chính xác hay approximate là đủ?
   (Exact count requires expensive COUNT query)
2. Web có cần offset-based pagination không, hay cursor là đủ?
```

**Mechanism 2: Architecture Review Board / Design Review**

Ở một số orgs, có regular meeting (weekly hoặc biweekly) nơi cross-cutting architectural decisions được discussed.

Không phải approval gate. Là a forum for:

- Surface potential conflicts trước khi commit
- Share knowledge về decisions đang được made
- Identify opportunities to reuse vs rebuild

Staff Engineer thường là member hoặc organizer của forum này.

Inspired by ThoughtWorks Tech Radar. Mỗi technology được classified:

- **Adopt:** Standard — default choice mới
- **Trial:** Promising, một số teams đang dùng, hãy try nhưng share learnings
- **Assess:** Worth exploring nhưng chưa ready cho production
- **Hold:** Đừng start mới, plan migration ra cho existing usage

Staff Engineer role: maintain radar này, update quarterly, ensure cross-team consistency.

**Mechanism 4: Inner Source / Shared Libraries**

Khi nhiều teams build similar things, Staff Engineer xác định opportunity để extract thành shared library:

```
Before:
Team A: có auth utilities
Team B: có auth utilities (slightly different)
Team C: có auth utilities (slightly different again)

After:
@company/auth-utils (owned by Platform team)
→ All teams depend on this
→ Bug fixed once, benefits all
→ Security patch only needs one PR
```

Trade-off thật: shared library phải được governed. Ai own nó? Ai review PRs? Ai decide API changes? Không có clear owner → bị abandoned.

### 4.3 — Staff Engineer Operating Model cho Cross-team Work

Có một mental model quan trọng: Staff Engineer không làm việc **cho** một team. Họ làm việc **across** teams. Điều này có nghĩa:

**Communication surface mở rộng:**

```
Senior Engineer:
└── Works with: immediate team (5-8 people)

Staff Engineer:
├── Immediate team: 5-8 people
├── Adjacent teams: 2-3 teams (20-25 people)
├── Stakeholders: Product, Design, Data, Management
└── External: Partners, vendors (đôi khi)
```

**Operating model cụ thể:**

1. **Weekly 1:1s với tech leads của adjacent teams** — không phải để manage, mà để stay informed về mỗi team đang build gì, catch potential conflicts sớm.

2. **"Embedded sprint"** — Đôi khi spend 1 sprint với một team khác để understand their problems deeply. Sau đó trở về với better context để build shared solutions.

3. **Written communication is default** — Decisions, proposals, context phải được viết down (RFC, ADR, Slack threads được summarized) vì bạn không thể meeting với 30 people.

4. **Build trust trước khi có influence** — Bạn không thể mandate adoption across teams nếu họ không trust technical judgment của bạn. Trust được build qua: shipping things that work, being right about important calls, helping other teams succeed.

### Câu 1: Tại sao RFC process thường fail khi nó được implement như một "approval gate"?

**Đáp án:**

Khi RFC trở thành approval gate, một số dynamics xấu xuất hiện:

**Flow bị block:** Nếu mọi decision phải được "approved" trước khi bắt đầu, và approvers là bottleneck (bận, không responsive), engineering velocity sụt giảm. Engineers bắt đầu bypass RFC để ship nhanh, dẫn tới RFC không còn được dùng.

**Ownership bị diluted:** Khi cần "approval", tác giả không còn là decision owner — họ là người xin phép. Điều này giảm accountability. Khi project fail, không ai chịu trách nhiệm vì "đã được approved".

**Investment bias:** Nếu RFC được submitted chỉ sau khi đã suy nghĩ kỹ và committed, người đọc cảm thấy investment cao → ngại reject → sycophantic approval → RFC mất giá trị như feedback mechanism.

RFC đúng design: tác giả **vẫn là decision owner**, RFC chỉ là structured way để collect input. Tác giả có thể receive negative feedback và vẫn proceed — nhưng phải explicitly acknowledge và explain disagreement. Điều này maintain speed while ensuring visibility và alignment.

### Câu 2: Tại sao ADR nên được store trong Git repo thay vì Confluence/Notion?

**Đáp án:**

Có 3 lý do kỹ thuật và 1 lý do cultural:

**Kỹ thuật:**

1. **Co-location với code:** ADR-0015 về api-client timeout có thể được referenced trực tiếp trong PR comment hay code comment. `// See docs/decisions/0015` ngay trong code dẫn người đọc đến context. Với Confluence, link rất dễ rot (page bị move, rename, delete).

2. **Versioning tự động:** `git log --all docs/decisions/0015-api-client.md` show toàn bộ history của decision đó — khi nó thay đổi, ai thay đổi, thay đổi gì. Confluence có version history nhưng không integrated với code changes.

3. **Code review integration:** ADR có thể được submitted cùng PR — reviewer review cả code change lẫn decision rationale trong cùng context. ADR trong Confluence phải được reviewed separately, dễ bị skip.

**Cultural:**

Khi ADR sống trong repo, engineers encounter nó trong daily workflow (searching codebase, reading PRs). ADR trong Confluence bị "out of sight, out of mind" — chỉ được đọc khi ai đó actively tìm kiếm.

### Câu 3: Đoạn code này có vấn đề gì từ góc độ tech debt management?

```javascript
// TODO: Fix this later
async function getUser(id) {
  // This is slow but works for now
  const user = await db.query(`SELECT * FROM users WHERE id = ${id}`);
  const orders = await db.query(`SELECT * FROM orders WHERE user_id = ${id}`);
  const addresses = await db.query(
    `SELECT * FROM addresses WHERE user_id = ${id}`,
  );
  return { user, orders, addresses };
}
```

**Đáp án:**

Có 3 vấn đề, mỗi cái thuộc một loại debt khác nhau:

**Vấn đề 1: SQL Injection (Security debt — không phải tech debt thông thường)**
`SELECT * FROM users WHERE id = ${id}` là string interpolation trực tiếp vào SQL query. Đây là SQL injection vulnerability. Phải dùng parameterized query: `WHERE id = $1` với `[id]` là parameter.

Đây không phải tech debt thông thường — đây là security issue cần fix ngay, không thể defer.

**Vấn đề 2: N+1 / Sequential queries (Performance debt)**
3 queries đang chạy sequential (await từng cái) thay vì parallel. Fix với `Promise.all`:

```javascript
const [user, orders, addresses] = await Promise.all([
  db.query('SELECT * FROM users WHERE id = $1', [id]),
  db.query('SELECT * FROM orders WHERE user_id = $1', [id]),
  db.query('SELECT * FROM addresses WHERE user_id = $1', [id]),
]);
```

Đây là deliberate debt nếu team biết vấn đề, inadvertent nếu không.

**Vấn đề 3: "TODO: Fix this later" không có context**
Đây là anti-pattern trong tech debt management. "Fix later" không có:

- Tại sao đây là shortcut
- Điều kiện để fix (khi nào "later"?)
- Ai sẽ fix
- Severity

TODO comment đúng:

```javascript
// DEBT(perf): Sequential queries — should use Promise.all
// Deferred from sprint 23 due to deadline. Impact: ~300ms added latency.
// Owner: Backend team. Target: Sprint 26.
// Ticket: PROJ-1234
```

Hoặc tốt hơn: tạo ticket thật trong Jira/Linear thay vì để TODO trong code — tickets được tracked, prioritized, không bị buried.
