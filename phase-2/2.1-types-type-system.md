# Phase 2 — Bài 2.1: Types & Type System

> Tất cả nội dung 🔴 đều quan trọng ngang nhau ở topic này — đây là nền tảng của mọi bug khó hiểu trong JS. Bài này sẽ dài nhưng sau khi xong, bạn sẽ hiểu tại sao 90% JS "gotcha" trên internet tồn tại.

## 1. Cơ chế thật

### V8 nhìn nhận "type" như thế nào?

JavaScript là ngôn ngữ **dynamically typed** — nhưng V8 không hề "mù" về type. V8 liên tục _đoán_ type của từng variable để tối ưu. Hiểu điều này là hiểu tại sao type system của JS được thiết kế như vậy.

Ở cấp độ spec, JS chia thành **2 loại giá trị**:

```
Primitive              Reference (Object)
─────────────────      ──────────────────
undefined              Object
null                   Array (là Object)
boolean                Function (là Object)
number                 Map, Set, Date, ...
bigint
string
symbol
```

Sự khác biệt cốt lõi không phải syntax — mà là **nơi lưu trữ và cách truy cập**:

```
STACK                          HEAP
──────────────────────         ────────────────────────────
let age = 25          ──►  25 (giá trị thật nằm thẳng ở stack)
let name = "Alice"    ──►  "Alice" (string ngắn: V8 intern vào pool)
                                      │
let user = { id: 1 }  ──►  [ ref ] ──►  { id: 1 } (object nằm ở heap)
                         (stack chứa     (heap chứa object thật)
                          địa chỉ bộ nhớ)
```

Đây là lý do `primitive` copy by value, `object` copy by reference — không phải quy ước ngẫu nhiên, mà là hệ quả tất yếu của memory layout.

### Primitive types — cơ chế lưu trữ trong V8

**`undefined`**

Giá trị mặc định khi V8 allocate một variable chưa được assign. Trong internal representation của V8, `undefined` là một singleton object đặc biệt — không phải "không có gì", mà là một giá trị cụ thể có địa chỉ cố định.

```
// Khi V8 parse: let x;
// V8 tạo một slot trong lexical environment và gán tag "undefined"
// Đây là lý do typeof undeclaredVar === 'undefined' — không throw ReferenceError
// (với typeof — đây là ngoại lệ được thiết kế cố tình để safe-check global)
```

**`null`**

Đây là **giá trị có chủ đích** — developer tự assign, nghĩa là "ô này có tồn tại nhưng rỗng". Khác với `undefined` là "ô này chưa có giá trị bao giờ".

```javascript
typeof null === 'object'; // true — tại sao???
```

Đây là **bug từ năm 1995** còn tồn tại đến hôm nay. Lý do lịch sử:

Trong implementation đầu tiên của JavaScript (Netscape Navigator), mỗi value được biểu diễn bằng một 32-bit word:

- 3 bits đầu là **type tag**
- `000` = object
- `null` được biểu diễn là `NULL pointer` = `0x00000000` — toàn bộ 32 bit là 0
- Type tag của `null` → `000` → bị đọc nhầm là "object"

TC39 đã propose fix trong ES6 nhưng bị reject vì sẽ break quá nhiều code cũ đang check `typeof x === 'object'`.

**`number`**

V8 dùng **IEEE 754 double-precision 64-bit floating point** cho tất cả số. Không có "integer" thật sự trong JS — `42` và `42.0` là cùng một representation.

```
64-bit floating point layout:
┌──────┬──────────────┬─────────────────────────────────────────────────────┐
│ sign │   exponent   │                     mantissa                        │
│  1   │     11 bit   │                      52 bit                         │
└──────┴──────────────┴─────────────────────────────────────────────────────┘
```

Điều này dẫn đến:

- `Number.MAX_SAFE_INTEGER = 2^53 - 1 = 9007199254740991`
- Số lớn hơn mất độ chính xác
- `0.1 + 0.2 !== 0.3` (floating point precision)

V8 optimization: Khi V8 phát hiện một number luôn là integer nhỏ, nó dùng **Smi (Small Integer)** — integer 31-bit nằm thẳng trong pointer slot, không cần heap allocation. Đây là lý do integer arithmetic trong V8 nhanh hơn float arithmetic.

**`string`**

String trong V8 là **immutable** và được **intern** (deduplicated). V8 có nhiều internal string representations:

- `SeqString`: String liên tục trong bộ nhớ
- `ConsString`: Kết quả của `a + b` — V8 không copy ngay, mà lưu dạng tree `(a, b)` để lazy concatenation
- `SlicedString`: Substring — chỉ lưu pointer + offset, không copy
- **String internment**: Literal strings như `"hello"` được cache — hai nơi dùng `"hello"` có thể trỏ đến cùng địa chỉ

```javascript
// V8 intern string literals
const a = 'hello';
const b = 'hello';
// a và b có thể trỏ đến cùng địa chỉ trong heap — V8 dedup
// Đây là lý do string comparison trong JS nhanh với short strings
```

**`symbol`**

Symbol là **unique identifier** được đảm bảo không bao giờ trùng. V8 cấp phát một unique 64-bit ID cho mỗi Symbol được tạo ra.

```javascript
const s1 = Symbol('debug-label');
const s2 = Symbol('debug-label');
s1 === s2; // false — cùng description nhưng khác identity
// V8 giữ một global symbol registry nếu dùng Symbol.for()
```

### `typeof` operator — cơ chế và quirks

`typeof` không gọi runtime function — nó là **compile-time operator** được V8 handle đặc biệt. Kết quả là string, không throw exception ngay cả với undeclared variable.

```
typeof result map:
────────────────────────────────────────
undefined    → "undefined"
null         → "object"    ← BUG LỊCH SỬ
boolean      → "boolean"
number       → "number"
bigint       → "bigint"
string       → "string"
symbol       → "symbol"
function     → "function"  ← object nhưng callable
object       → "object"    (array, plain object, null đều ra đây)
────────────────────────────────────────
```

Quirk quan trọng: `typeof` với undeclared variable không throw `ReferenceError`:

```javascript
// Đây là intentional design — để check feature detection
if (typeof window !== 'undefined') {
  // Safe — không throw dù window không tồn tại
}

// Ngược lại:
if (window !== undefined) {
  // Throw ReferenceError nếu window không tồn tại
}
```

### Type Coercion — Abstract Equality (`==`) Algorithm

Đây là phần phức tạp nhất và gây nhiều bug nhất. V8 implement đúng theo spec, không có magic — chỉ là spec rất dài và không ai đọc.

**Explicit coercion** (bạn viết rõ):

```javascript
Number('42'); // 42
String(42); // "42"
Boolean(0); // false
```

**Implicit coercion** (JS tự làm):

```javascript
'5' + 3; // "53"  — number bị coerce sang string
'5' - 3; // 2     — string bị coerce sang number
```

Tại sao `+` và `-` hành xử khác nhau? Vì `+` có hai nghĩa: **string concatenation** và **numeric addition**. Khi một operand là string, `+` chọn concatenation. `-` chỉ có một nghĩa: numeric subtraction — nên luôn coerce sang number.

**Abstract Equality (`==`) algorithm — spec thật:**

```
x == y:

1. Nếu cùng type → dùng Strict Equality (===)
2. null == undefined → true (và chỉ 2 cái này == nhau)
3. null/undefined == anything else → false
4. Nếu một bên là number, một bên là string → coerce string sang number, so lại
5. Nếu một bên là boolean → coerce boolean sang number (true→1, false→0), so lại
6. Nếu một bên là object, một bên là primitive → gọi ToPrimitive(object), so lại
7. Còn lại → false
```

Visualize một vài case kinh điển:

```
[] == false

Step 1: [] là object, false là boolean → khác type
Step 5: boolean → number: false → 0
        [] == 0

Step 6: [] là object, 0 là number → ToPrimitive([])
        ToPrimitive([]) gọi [].toString() → ""
        "" == 0

Step 4: "" là string, 0 là number → Number("") → 0
        0 == 0

Step 1: cùng type, cùng value → TRUE
```

```
"" == false

Step 5: false → 0
        "" == 0

Step 4: "" → Number("") → 0
        0 == 0  →  TRUE

null == false
Step 3: null chỉ == undefined, không == false → FALSE
```

### Falsy/Truthy — danh sách đầy đủ

V8 implement **ToBoolean()** như sau — đây là danh sách **exhaustive** của falsy values:

```
Falsy values (tất cả, không thêm không bớt):
──────────────────────────────────────────
false
0
-0
0n          (BigInt zero)
""          (empty string)
null
undefined
NaN
──────────────────────────────────────────
TẤT CẢ còn lại đều truthy
```

Edge cases cần nhớ:

```javascript
// Những cái này TRUTHY — không phải falsy
Boolean('false'); // true  — string không rỗng
Boolean('0'); // true  — string không rỗng
Boolean([]); // true  — object (dù rỗng)
Boolean({}); // true  — object (dù rỗng)
Boolean(new Boolean(false)); // true  — object!

// Bẫy phổ biến:
if ([]) {
} // chạy — [] là truthy
[] == false; // true  — coercion khác với ToBoolean!
```

### NaN — tại sao `NaN !== NaN`

`NaN` là viết tắt của **Not-a-Number** — nhưng `typeof NaN === 'number'`. Nghịch lý này là do NaN là một giá trị số hợp lệ trong IEEE 754 dùng để biểu diễn "kết quả phép tính không hợp lệ".

**Tại sao `NaN !== NaN`?**

IEEE 754 spec định nghĩa: _"NaN là unordered — bất kỳ so sánh nào có NaN đều trả về false"_, kể cả `NaN === NaN`. Lý do: có nhiều loại NaN khác nhau trong IEEE 754 (quiet NaN, signaling NaN), và so sánh hai NaN không có nghĩa xác định.

JS implement đúng theo IEEE 754, không override.

```javascript
NaN === NaN; // false — IEEE 754
NaN == NaN; // false — same

// Cách check NaN:
isNaN('hello'); // true  — BAD: coerce sang number trước
isNaN(undefined); // true  — BAD: undefined → NaN
Number.isNaN('hello'); // false — GOOD: check type trước
Number.isNaN(NaN); // true  — GOOD: chỉ true với NaN thật

// Cách isNaN hoạt động:
// isNaN(x) = Number.isNaN(Number(x)) — coerce trước, check sau
// Number.isNaN(x) = typeof x === 'number' && x !== x
```

## 2. Visualize

### Abstract Equality decision tree

```
x == y
   │
   ▼
Same type? ──YES──► Strict equality (===)
   │
   NO
   │
   ▼
null/undefined? ──YES──► null==undefined: TRUE
   │                      others: FALSE
   NO
   │
   ▼
One is number, other is string? ──YES──► ToNumber(string), retry
   │
   NO
   │
   ▼
One is boolean? ──YES──► ToNumber(boolean), retry
   │                      true→1, false→0
   NO
   │
   ▼
One is object, other is primitive? ──YES──► ToPrimitive(object), retry
   │                                         calls valueOf() then toString()
   NO
   │
   ▼
FALSE
```

### ToPrimitive flow

```
ToPrimitive(obj, hint)
   │
   ├── hint = "number" (default cho math ops)
   │      → try valueOf() first
   │      → if not primitive, try toString()
   │
   └── hint = "string" (template literals, String())
          → try toString() first
          → if not primitive, try valueOf()

Array:
  valueOf() → [] (not primitive)
  toString() → "" (primitive!) ──► kết quả: ""

Plain object:
  valueOf() → {} (not primitive)
  toString() → "[object Object]" ──► kết quả: "[object Object]"

Date (đặc biệt):
  hint = "string" → toString()
  hint = "number" → valueOf() → timestamp
```

### Type coercion trong production code

```javascript
// BUG THƯỜNG GẶP: dùng == trong điều kiện
function processUserId(id) {
  // V8 thực hiện: id == null
  // Abstract Equality: null == undefined → TRUE
  // Đây là pattern HỢP LỆ để check cả null và undefined
  if (id == null) {
    throw new Error('id required');
  }
  return fetch(`/users/${id}`);
}

processUserId(null); // throw ✓
processUserId(undefined); // throw ✓
processUserId(0); // KHÔNG throw — 0 == null là FALSE ✓
```

```javascript
// BUG: dùng || để set default value
function createUser(options) {
  // V8 evaluate: options.timeout || 5000
  // ToBoolean(0) → false → fallback sang 5000
  // BUG: timeout = 0 là valid nhưng bị overwrite!
  const timeout = options.timeout || 5000;

  // FIX: dùng ?? (nullish coalescing — ES2020)
  // ?? chỉ fallback khi null/undefined, không fallback khi 0 hay ""
  const timeoutFixed = options.timeout ?? 5000;
}

createUser({ timeout: 0 });
// timeout = 5000  ← BUG với ||
// timeoutFixed = 0 ← CORRECT với ??
```

```javascript
// BUG: sort array số mà không truyền comparator
const ids = [10, 9, 2, 100, 21];

// V8 thực hiện: coerce mỗi phần tử sang string rồi sort lexicographically
ids.sort();
// → [10, 100, 2, 21, 9]  ← SAI

// FIX: truyền comparator để V8 dùng numeric comparison
ids.sort((a, b) => a - b);
// → [2, 9, 10, 21, 100]  ← ĐÚNG
```

```javascript
// NaN propagation — âm thầm phá data pipeline
function calculateDiscount(price, discountStr) {
  // Nếu discountStr = "abc" → Number("abc") → NaN
  // NaN * anything = NaN
  // NaN propagate qua toàn bộ chain
  const discount = Number(discountStr);
  const finalPrice = price * (1 - discount / 100);

  // finalPrice = NaN — không có error thrown
  // Bug âm thầm đến database hoặc UI
  return finalPrice;
}

// FIX: validate trước
function calculateDiscountSafe(price, discountStr) {
  const discount = Number(discountStr);

  // Number.isNaN — không coerce, check type trước
  if (Number.isNaN(discount)) {
    throw new TypeError(`Invalid discount: ${discountStr}`);
  }

  return price * (1 - discount / 100);
}
```

### Symbol — use case thực tế

```javascript
// Well-known Symbol: Symbol.toPrimitive
// Override cách V8 coerce object sang primitive

class Temperature {
  constructor(celsius) {
    this.celsius = celsius;
  }

  // V8 gọi cái này khi cần coerce Temperature sang primitive
  [Symbol.toPrimitive](hint) {
    // hint = "number" khi dùng trong math expression
    if (hint === 'number') return this.celsius;
    // hint = "string" khi dùng trong template literal
    if (hint === 'string') return `${this.celsius}°C`;
    // hint = "default" khi dùng với == hay +
    return this.celsius;
  }
}

const temp = new Temperature(25);
console.log(+temp); // 25      — hint: "number"
console.log(`${temp}`); // "25°C"  — hint: "string"
console.log(temp + 5); // 30      — hint: "default"
```

```javascript
// Symbol làm non-enumerable metadata key
// Không bị lộ ra khi iterate object

const INTERNAL_ID = Symbol('internal-id');

class ApiResponse {
  constructor(data, requestId) {
    this.data = data;
    // Symbol key không xuất hiện trong JSON.stringify, for...in, Object.keys
    this[INTERNAL_ID] = requestId;
  }

  getRequestId() {
    return this[INTERNAL_ID];
  }
}

const res = new ApiResponse({ user: 'Alice' }, 'req-123');
JSON.stringify(res); // {"data":{"user":"Alice"}}  — Symbol key bị skip
Object.keys(res); // ["data"]  — Symbol key không xuất hiện
res.getRequestId(); // "req-123"  — vẫn truy cập được
```

## 4. Ứng dụng thực tế

### Performance

**V8 Monomorphism** — liên quan trực tiếp đến type:

```javascript
// V8 tối ưu tốt nhất khi một function luôn nhận cùng type
// Đây gọi là "monomorphic call site"

function add(a, b) {
  return a + b;
}

// Monomorphic — V8 compile thành optimized machine code
add(1, 2);
add(3, 4);
add(5, 6);

// Polymorphic — V8 phải deoptimize, dùng generic handler
add(1, 2); // number + number
add('a', 'b'); // string + string — V8 "bail out"
add(1, '2'); // mixed — V8 dùng slow path
```

**Implication trong React**: Đây là lý do TypeScript giúp performance không chỉ vì DX — type consistency giúp V8 giữ code trong optimized state.

### Debugging

**Tìm type bug với DevTools:**

```javascript
// Console trick để debug coercion
console.log(typeof value, value); // Xem type và value
console.log(Object.prototype.toString.call(value));
// "[object Array]", "[object Null]", "[object Undefined]"
// Đây là cách reliable nhất để check type chính xác
```

**Trong React codebase:**

```javascript
// Bug phổ biến: state là 0 nhưng bị treat là falsy
function Counter({ count }) {
  // BUG: count = 0 → falsy → render nothing
  return <div>{count && <span>{count}</span>}</div>;

  // FIX 1: explicit boolean
  return <div>{count > 0 && <span>{count}</span>}</div>;

  // FIX 2: ternary
  return <div>{count !== null ? <span>{count}</span> : null}</div>;
}
```

### BigInt — khi nào cần

```javascript
// Database IDs lớn hơn Number.MAX_SAFE_INTEGER
// Twitter/X dùng int64 cho tweet IDs — vượt quá giới hạn JS number
const tweetId = 9007199254740993n; // BigInt literal
const tweetIdNum = 9007199254740993; // MẤT CHÍNH XÁC
console.log(tweetIdNum === 9007199254740992); // true — off by 1!

// BigInt không tự động coerce với number
tweetId + 1; // TypeError
tweetId + 1n; // OK

// JSON không support BigInt — cần serialize thủ công
JSON.stringify({ id: tweetId }); // TypeError
JSON.stringify({ id: tweetId.toString() }); // OK — lưu dạng string
```

---

**Câu 1: Tại sao `typeof null === 'object'` và tại sao TC39 không fix?**

**Đáp án:**

Đây là bug từ implementation đầu tiên của JavaScript năm 1995. JavaScript lưu mỗi value dưới dạng 32-bit word với 3 bit đầu là type tag. Object có type tag là `000`. `null` được implement là null pointer — toàn bộ 32 bit bằng 0 — nên type tag của nó cũng là `000`, bị đọc nhầm thành "object".

TC39 đã đề xuất fix trong quá trình chuẩn bị ES6 (`typeof null === 'null'`), nhưng bị reject vì sẽ break code cũ đang sử dụng pattern `typeof x === 'object'` để check cả object lẫn null. Với hàng tỷ trang web đang chạy, đây là breaking change không chấp nhận được.

Cách check null đúng: dùng strict equality `x === null`.

---

**Câu 2: Đoạn code sau có vấn đề gì? Giải thích từng bước V8 thực hiện.**

```javascript
const input = '';
const port = input || 3000;
console.log(port); // 3000

const timeout = 0;
const finalTimeout = timeout || 5000;
console.log(finalTimeout); // ???
```

**Đáp án:**

`finalTimeout` sẽ là `5000` — đây là bug.

Cơ chế: Operator `||` không so sánh equality, nó gọi **ToBoolean()** trên operand trái. `ToBoolean(0)` trả về `false` vì `0` nằm trong danh sách falsy values của spec. Do đó `0 || 5000` evaluate thành `5000`.

Điều này đúng với `""` (empty string cũng là falsy), nhưng sai với `0` vì `timeout = 0` là giá trị hợp lệ (có thể có nghĩa là "không timeout").

Fix: Dùng nullish coalescing operator `??` (ES2020). `??` chỉ fallback khi operand trái là `null` hoặc `undefined`, không fallback với falsy values khác:

```javascript
const finalTimeout = timeout ?? 5000;
// 0 ?? 5000 → 0  (correct)
// null ?? 5000 → 5000  (correct)
// undefined ?? 5000 → 5000  (correct)
```

---

**Câu 3: Tại sao `Number.isNaN` an toàn hơn `isNaN` trong production code?**

**Đáp án:**

`isNaN(x)` là global function — nó gọi `Number(x)` trước để coerce argument sang number, sau đó mới check. Điều này dẫn đến false positives:

```javascript
isNaN('hello'); // true  — Number("hello") = NaN, nhưng "hello" không phải NaN
isNaN(undefined); // true  — Number(undefined) = NaN
isNaN({}); // true  — Number({}) = NaN
```

`Number.isNaN(x)` là static method được thêm vào ES6 — nó **không coerce**, check type trước: `typeof x === 'number' && x !== x`. Lợi dụng đặc tính duy nhất của NaN là giá trị duy nhất không bằng chính nó (theo IEEE 754).

```javascript
Number.isNaN('hello'); // false — "hello" là string, không phải NaN
Number.isNaN(undefined); // false — undefined là undefined
Number.isNaN(NaN); // true  — đúng
```

Trong production code xử lý input từ user hoặc API, luôn dùng `Number.isNaN()` để tránh false positives gây silent bug trong data pipeline.
