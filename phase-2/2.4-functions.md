## 1. Cơ chế thật

### Function declaration vs expression vs arrow — sự khác biệt thật sự

Ba cú pháp tạo function — nhưng V8 xử lý chúng khác nhau ở 3 điểm: **hoisting**, **`this` binding**, và **`arguments` object**.

```
                    Function Declaration   Function Expression   Arrow Function
─────────────────────────────────────────────────────────────────────────────
Hoisting            Toàn bộ (declaration   Chỉ var binding      Không (TDZ nếu let/const)
                    + body)                (nếu var) = undefined
this binding        Dynamic                Dynamic               Lexical (từ outer scope)
arguments object    Có                     Có                    KHÔNG có
new.target          Có                     Có                    KHÔNG có
Dùng làm           OK                      OK                    KHÔNG được
constructor
prototype           Có                     Có                    KHÔNG có
property
```

**Ý nghĩa thực tế của bảng này:**

Arrow function không phải "function viết ngắn hơn" — nó là một loại function **fundamentally khác**: không có `this` riêng, không có `arguments`, không thể dùng làm constructor. Đây là lý do tại sao không thể thay thế function thường bằng arrow function mọi nơi.

### `this` binding — 4 rules và thứ tự ưu tiên

`this` trong JavaScript không phải variable — nó là một **implicit parameter** được V8 truyền vào mỗi function call. Giá trị của `this` được xác định **lúc gọi**, không phải lúc định nghĩa (ngoại trừ arrow function).

V8 xác định `this` theo 4 rules, theo thứ tự ưu tiên từ cao đến thấp:

---

#### Rule 1 — `new` binding (ưu tiên cao nhất)

Khi gọi function với `new`, V8 làm 4 việc:

1. Tạo object mới trong heap
2. Set `[[Prototype]]` của object mới = `Constructor.prototype`
3. Gọi constructor với `this` = object mới vừa tạo
4. Return object mới (trừ khi constructor explicitly return object khác)

```javascript
function User(name, role) {
  // `this` = object mới được tạo bởi `new`
  // V8 đã tạo object {} và truyền vào đây làm `this`
  this.name = name;
  this.role = role;
  // implicit: return this
}

const alice = new User('Alice', 'admin');
// alice = { name: "Alice", role: "admin" }
// alice.__proto__ === User.prototype
```

---

#### Rule 2 — Explicit binding: `call`, `apply`, `bind`

`call` và `apply` gọi function ngay với `this` được chỉ định. `bind` tạo function mới với `this` được "đóng băng".

```javascript
// call và apply: gọi ngay
function greet(greeting, punctuation) {
  return `${greeting}, ${this.name}${punctuation}`;
}

const user = { name: 'Alice' };

// call: truyền arguments riêng lẻ
greet.call(user, 'Hello', '!'); // "Hello, Alice!"
// V8: tạo execution context với this = user

// apply: truyền arguments dạng array
greet.apply(user, ['Hello', '!']); // "Hello, Alice!"
// V8: spread array thành individual arguments

// bind: tạo function mới, this bị lock
const greetAlice = greet.bind(user);
greetAlice('Hi', '.'); // "Hi, Alice."
// greetAlice.[[BoundThis]] = user — không thể override dù dùng call/apply sau
```

**`bind` tạo ra Bound Function Exotic Object** — một loại function object đặc biệt trong V8 với internal slot `[[BoundThis]]` và `[[BoundArguments]]`. Khi được gọi, V8 luôn dùng `[[BoundThis]]` thay vì tính toán `this` bình thường. Bound function thậm chí **không thể bị override bởi `new`** — nhưng `[[BoundThis]]` bị ignore khi dùng với `new`.

---

#### Rule 3 — Implicit binding: method call

Khi function được gọi như method của object (`obj.method()`), `this` = object đứng trước dấu chấm.

```javascript
const api = {
  baseUrl: 'https://api.example.com',

  fetch(endpoint) {
    // this = api — vì được gọi là api.fetch()
    return `${this.baseUrl}/${endpoint}`;
  },
};

api.fetch('users'); // this = api ✓

// BUG KINH ĐIỂN: method bị tách ra khỏi object
const fetchFn = api.fetch;
fetchFn('users'); // this = undefined (strict mode) hoặc window
// Vì fetchFn() không có object trước dấu chấm

// Cũng xảy ra với callback:
setTimeout(api.fetch, 100); // this sẽ bị mất
// setTimeout nhận function reference — không phải method call

// FIX 1: bind
setTimeout(api.fetch.bind(api), 100);

// FIX 2: arrow wrapper
setTimeout(() => api.fetch('users'), 100);
// Arrow function không có this riêng, nhưng arrow này gọi api.fetch()
// — đây là method call → this = api ✓
```

**Tại sao `this` bị mất khi tách method?**

Khi bạn viết `const fetchFn = api.fetch`, bạn copy **reference đến function object** — không copy "context" gì cả. Function object không nhớ nó thuộc về `api`. `this` được xác định mỗi lần gọi dựa trên cú pháp gọi, không dựa trên nơi function được lấy ra.

---

#### Rule 4 — Default binding (ưu tiên thấp nhất)

Khi không có rule nào ở trên apply, `this` = `global` (non-strict) hoặc `undefined` (strict mode).

```javascript
function showThis() {
  console.log(this);
}

showThis(); // window (browser, non-strict) hoặc undefined (strict)

// Strict mode bắt "this pollution" vào global
('use strict');
function showThisStrict() {
  console.log(this); // undefined — không silently access global
}
```

---

#### Arrow function — Lexical `this`

Arrow function không có `this` riêng. Khi V8 gặp `this` bên trong arrow function, nó **không tính toán `this` mới** — nó đi lên scope chain và lấy `this` từ outer lexical scope. Đây giống hệt cách closure capture variable thông thường, nhưng áp dụng cho `this`.

```javascript
class Timer {
  constructor() {
    this.seconds = 0;
  }

  start() {
    // `this` ở đây = Timer instance (method call)

    // BUG: regular function — this bị mất trong callback
    setInterval(function () {
      this.seconds++; // this = undefined (strict) hoặc window
    }, 1000);

    // FIX: arrow function — capture `this` từ start() scope
    setInterval(() => {
      // V8 tìm `this` trong arrow fn scope → không có
      // → đi lên outer scope (start method scope) → tìm thấy this = Timer instance
      this.seconds++;
    }, 1000);
  }
}
```

**Cơ chế bên trong V8:** Arrow function có internal slot `[[ThisMode]] = lexical`. Khi execute arrow function, V8 không tạo `this` binding mới trong execution context. Thay vào đó, mọi reference đến `this` trong arrow function được resolve thông qua scope chain, giống như một variable thông thường.

### `arguments` object vs rest parameters

`arguments` là object giống array (không phải array thật) được V8 tự động tạo trong mỗi non-arrow function call. Nó chứa tất cả arguments được truyền vào.

```javascript
function oldStyle() {
  // V8 tạo `arguments` object — linked đến actual parameters
  console.log(arguments); // Arguments [1, 2, 3]
  console.log(arguments[0]); // 1
  console.log(Array.isArray(arguments)); // false — không phải array thật

  // Không có array methods:
  arguments.map((x) => x * 2); // TypeError

  // Phải convert:
  Array.from(arguments).map((x) => x * 2); // [2, 4, 6]
  [...arguments].map((x) => x * 2); // [2, 4, 6]
}

oldStyle(1, 2, 3);
```

**Rest parameters** — cách đúng từ ES6:

```javascript
function newStyle(...args) {
  // args là Array thật — không phải Arguments object
  // V8 tạo array từ remaining arguments
  console.log(Array.isArray(args)); // true
  return args.map((x) => x * 2); // [2, 4, 6] — array method hoạt động
}

newStyle(1, 2, 3);

// Rest chỉ capture "phần còn lại" — phải đứng cuối
function request(method, url, ...headers) {
  // method = "GET", url = "/users", headers = ["Auth: Bearer...", "Accept: json"]
}

request('GET', '/users', 'Auth: Bearer token', 'Accept: application/json');
```

**Tại sao `arguments` tồn tại?** Vì pre-ES6 không có rest parameters. Và tại sao không dùng nữa? Vì `arguments`:

- Không hoạt động trong arrow functions
- Không phải real array — phải convert
- Không thể dùng với destructuring ngay
- Tên cố định, không descriptive
- V8 khó optimize hơn rest parameters

### Default parameters — evaluation timing

Default parameters không chỉ là "giá trị mặc định" — chúng được **evaluate mỗi lần function được gọi** nếu argument tương ứng là `undefined`.

```javascript
// Default value là expression — evaluated lazily
function createRequest(
  url,
  timestamp = Date.now(), // evaluated mỗi lần gọi — không phải lúc định nghĩa
  id = crypto.randomUUID(), // tạo UUID mới mỗi lần
) {
  return { url, timestamp, id };
}

createRequest('/users'); // timestamp và id khác nhau mỗi lần gọi

// Default có thể reference parameters trước nó:
function createUser(
  name,
  slug = name.toLowerCase().replace(/\s+/g, '-'), // reference `name`
) {
  return { name, slug };
}

createUser('Alice Smith'); // { name: "Alice Smith", slug: "alice-smith" }

// Default parameters tạo SCOPE RIÊNG (parameter scope)
// Nằm giữa outer scope và function body scope
let multiplier = 2;

function scale(value, factor = multiplier * 2) {
  // `multiplier` được tìm trong parameter scope → không thấy
  // → leo lên outer scope → tìm thấy multiplier = 2
  // factor = 2 * 2 = 4
  return value * factor;
}
```

**TDZ trong default parameters:**

```javascript
// Parameters có TDZ với nhau
function broken(a = b, b = 1) {
  // Khi evaluate `a = b`, `b` chưa được initialized → ReferenceError
}

broken(); // ReferenceError: Cannot access 'b' before initialization

function ok(a = 1, b = a * 2) {
  // `a` đã initialized trước khi evaluate `b = a * 2` → OK
  return [a, b];
}

ok(); // [1, 2]
```

### Spread operator — shallow copy implications

```javascript
// Spread với array — shallow copy
const original = [1, 2, { value: 3 }];
const copy = [...original];

copy[0] = 99; // original không bị ảnh hưởng — primitive copied by value
copy[2].value = 99; // original[2].value cũng = 99!
// Vì spread chỉ copy references — object trong heap không bị duplicate

// Cùng vấn đề với object spread
const user = { name: 'Alice', address: { city: 'Hanoi' } };
const userCopy = { ...user };

userCopy.name = 'Bob'; // user.name vẫn "Alice" — string là primitive
userCopy.address.city = 'HCMC'; // user.address.city cũng = "HCMC"!
// address là reference — cả hai point đến cùng object trong heap

// Deep copy — các options:
// 1. structuredClone (ES2022, modern browsers + Node 17+)
const deepCopy = structuredClone(user); // true deep copy

// 2. JSON round-trip — chỉ cho data đơn giản, mất functions/undefined/Symbol
const jsonCopy = JSON.parse(JSON.stringify(user));
```

### Pure functions — định nghĩa thực sự

Một function là **pure** khi thỏa 2 điều kiện — **cả hai**, không phải một:

1. **Deterministic**: Với cùng input, luôn trả về cùng output
2. **No side effects**: Không thay đổi bất cứ thứ gì bên ngoài function scope

```javascript
// PURE — deterministic + no side effects
function calculateTax(amount, rate) {
  return amount * rate;
}

// IMPURE — không deterministic (phụ thuộc external state)
let taxRate = 0.1;
function calculateTaxImpure(amount) {
  return amount * taxRate; // kết quả thay đổi khi taxRate thay đổi
}

// IMPURE — side effect (mutate input)
function addItem(cart, item) {
  cart.items.push(item); // mutate argument — side effect
  return cart;
}

// PURE version:
function addItem(cart, item) {
  return {
    ...cart,
    items: [...cart.items, item], // tạo object mới — không mutate
  };
}

// IMPURE — side effect (I/O, network, DOM)
function fetchUser(id) {
  return fetch(`/users/${id}`); // network call = side effect
}

// IMPURE — side effect (console.log cũng là side effect)
function debugAdd(a, b) {
  console.log('adding', a, b); // side effect
  return a + b;
}
```

**Tại sao pure functions quan trọng cho performance?**

V8 và runtime có thể optimize pure functions mạnh hơn:

- **Memoization** chỉ safe với pure functions — vì cùng input = cùng output
- **Parallel execution** safe với pure functions — không có shared state mutation
- **Tree shaking** hiệu quả hơn — bundler biết function không có side effects

Trong React: Pure component/function component không có side effects → React có thể render ở Strict Mode hai lần để detect bugs. Đây là lý do `useEffect` tồn tại — để isolate side effects khỏi render logic.

### 🟡 Generator functions — lazy evaluation

Generator là function có thể **pause** và **resume**. Mỗi lần gặp `yield`, V8 lưu toàn bộ execution state (local variables, instruction pointer) và trả quyền điều khiển về caller.

```javascript
// function* tạo generator function
// Gọi nó trả về generator object (iterator)
// Code bên trong KHÔNG chạy ngay
function* idGenerator() {
  let id = 1;
  while (true) {
    // infinite loop — ok vì lazy
    yield id++; // pause ở đây, trả về id, tăng id cho lần sau
  }
}

const gen = idGenerator(); // tạo generator object — chưa chạy gì

gen.next(); // { value: 1, done: false } — chạy đến yield đầu, pause
gen.next(); // { value: 2, done: false } — resume từ sau yield, chạy tiếp
gen.next(); // { value: 3, done: false }

// Thực tế: generate ID cho records
function* generateRequestIds(prefix) {
  let seq = 0;
  while (true) {
    yield `${prefix}-${Date.now()}-${seq++}`;
  }
}

const requestIdGen = generateRequestIds('REQ');
requestIdGen.next().value; // "REQ-1708123456789-0"
requestIdGen.next().value; // "REQ-1708123456790-1"
```

**Lazy evaluation — xử lý large dataset mà không load hết vào memory:**

```javascript
// BẮT BUỘC LOAD HẾT — tốn memory nếu dataset lớn
function getExpensiveItems(items) {
  return items
    .filter((item) => item.price > 1000)
    .map((item) => ({ ...item, tax: item.price * 0.1 }))
    .slice(0, 10);
  // Filter và map chạy trên TOÀN BỘ array trước khi slice
}

// LAZY với generator — chỉ process items cần thiết
function* getExpensiveItemsLazy(items) {
  let count = 0;
  for (const item of items) {
    if (count >= 10) return; // stop sớm
    if (item.price > 1000) {
      yield { ...item, tax: item.price * 0.1 }; // process từng item khi cần
      count++;
    }
  }
}

// Chỉ process đủ items để lấy 10 cái đầu tiên
const result = [...getExpensiveItemsLazy(millionItems)];
```

### 🟡 Async generators — streaming data

Kết hợp `async` và generator để handle streaming data — ví dụ paginated API:

```javascript
// Async generator: yield + await trong cùng một function
async function* fetchAllUsers(pageSize = 100) {
  let page = 1;
  let hasMore = true;

  while (hasMore) {
    // await trong generator — pause CŨNG chờ Promise
    const response = await fetch(`/users?page=${page}&size=${pageSize}`);
    const data = await response.json();

    for (const user of data.users) {
      yield user; // trả từng user về caller
    }

    hasMore = data.hasNextPage;
    page++;
  }
}

// Dùng với for-await-of
async function processAllUsers() {
  for await (const user of fetchAllUsers()) {
    // Xử lý từng user khi nó arrive — không cần load hết
    await updateUserIndex(user);
  }
}
```

## 2. Visualize

### 4 rules của `this` — thứ tự ưu tiên

```
Khi gọi function, V8 check theo thứ tự này:

1. new binding?
   └── new Foo()  →  this = newly created object
         ↓ không
2. Explicit binding?
   └── foo.call(obj)    →  this = obj
   └── foo.apply(obj)   →  this = obj
   └── foo.bind(obj)()  →  this = obj (locked, không override được)
         ↓ không
3. Implicit binding?
   └── obj.foo()  →  this = obj (object before the dot)
         ↓ không
4. Default binding
   └── foo()  →  strict mode: this = undefined
              →  non-strict:  this = global (window/global)

NGOẠI LỆ: Arrow function — bỏ qua tất cả rules trên
   └── this = lexical this từ outer scope (không đổi khi gọi)
```

### `this` binding bị mất — khi nào xảy ra

```
obj.method()        ← Implicit binding, this = obj  ✓

const fn = obj.method
fn()                ← Chỉ là function call, no object → Default binding  ✗

[1,2,3].forEach(obj.method)
                    ← forEach nhận function reference → Default binding  ✗

setTimeout(obj.method, 100)
                    ← setTimeout nhận function reference → Default binding  ✗

class A {
  method = () => {} ← Arrow function trong class field
}                      → this = instance, luôn luôn  ✓
```

### Generator execution model

```
function* gen() {
  console.log("A");
  yield 1;          ← PAUSE POINT 1
  console.log("B");
  yield 2;          ← PAUSE POINT 2
  console.log("C");
}

const g = gen();     → Tạo generator object, CHƯA chạy gì

g.next()             → Chạy từ đầu đến yield 1
                       Print "A"

                       Print "B"

                       Return { value: undefined, done: true }

                       Return { value: undefined, done: true }
```

### `this` trong class methods — bug và fix

```javascript
class EventEmitter {
  constructor() {
    this.listeners = new Map();
    this.eventCount = 0;
  }

  // BUG: method bị tách khỏi instance khi pass như callback
  on(event, callback) {
    if (!this.listeners.has(event)) {
      // `this` ở đây đúng vì on() được gọi là method
      this.listeners.set(event, []);
    }
    this.listeners.get(event).push(callback);
  }

  emit(event, data) {
    this.eventCount++; // nếu emit bị tách ra khỏi instance, this.eventCount bị undefined
    const callbacks = this.listeners.get(event) ?? [];
    callbacks.forEach((cb) => cb(data));
  }
}

const emitter = new EventEmitter();

// Truyền emit như callback — this bị mất
const trigger = emitter.emit;
trigger('click', {}); // TypeError: Cannot read properties of undefined

// FIX 1: bind trong constructor — phù hợp khi method thường xuyên pass như callback
class EventEmitter {
  constructor() {
    this.listeners = new Map();
    this.eventCount = 0;
    // Tạo bound version ngay trong constructor
    this.emit = this.emit.bind(this);
    this.on = this.on.bind(this);
  }
  // ...
}

// FIX 2: class field arrow function — this luôn là instance
class EventEmitter {
  listeners = new Map();
  eventCount = 0;

  // Arrow function trong class field: [[Environment]] capture `this` của constructor
  emit = (event, data) => {
    this.eventCount++;
    // `this` luôn là instance — không bao giờ bị mất
    const callbacks = this.listeners.get(event) ?? [];
    callbacks.forEach((cb) => cb(data));
  };
}

// Trade-off của FIX 2:
// Mỗi instance có function object riêng cho `emit`
// Không share qua prototype — tốn memory hơn với nhiều instances
```

### Function composition — pure functions + closures

```javascript
// compose: áp dụng functions từ phải sang trái
// pipe: áp dụng functions từ trái sang phải

const compose =
  (...fns) =>
  (x) =>
    fns.reduceRight((acc, fn) => fn(acc), x);

const pipe =
  (...fns) =>
  (x) =>
    fns.reduce((acc, fn) => fn(acc), x);

// Pure functions — không mutate, deterministic

const toLowerCase = (str) => str.toLowerCase();
const removeSpaces = (str) => str.replace(/\s+/g, '-');

const createSlug = pipe(trim, toLowerCase, removeSpaces, addPrefix('post'));

createSlug('  React Hooks   '); // "post-react-hooks"
```

### Generator cho pagination — không load hết

```javascript
// Thực tế: iterate qua toàn bộ records mà không load hết vào memory
async function* paginatedFetch(url, pageSize = 50) {
  let cursor = null;
  let pageCount = 0;

  do {
    // Build URL với cursor-based pagination
    const pageUrl = cursor
      ? `${url}?cursor=${cursor}&limit=${pageSize}`
      : `${url}?limit=${pageSize}`;

    const response = await fetch(pageUrl);

    if (!response.ok) {
      throw new Error(`HTTP ${response.status}: ${pageUrl}`);
    }

    const { data, nextCursor } = await response.json();
    cursor = nextCursor;
    pageCount++;

    // Yield từng item — caller nhận và xử lý trước khi fetch page tiếp
    for (const item of data) {
      yield item;
    }

    console.log(`Fetched page ${pageCount}`);
  } while (cursor);
}

// Consumer: xử lý từng record mà không giữ hết trong memory
async function exportToCSV(apiUrl) {
  const rows = [];

  for await (const record of paginatedFetch(apiUrl)) {
    // Process từng record ngay khi nhận được
    rows.push(formatRecord(record));

    // Có thể break sớm nếu cần
    if (rows.length >= 1000) break;
  }

  return rows.join('\n');
}
```

### React — tại sao class component cần bind

```javascript
// Giải thích bằng cơ chế this binding
class Button extends React.Component {
  constructor(props) {
    super(props);
    this.state = { count: 0 };

    // Cần bind vì React gọi handleClick như regular function call
    // không phải method call trên instance
    // this.handleClick.bind(this) tạo Bound Function với [[BoundThis]] = instance
    this.handleClick = this.handleClick.bind(this);
  }

  handleClick() {
    // Khi React gọi onClick handler:
    // React làm: handler() — không phải this.handler()
    // → Default binding → this = undefined (strict mode)
    // → Crash nếu không bind
    this.setState({ count: this.state.count + 1 });
  }

  render() {
    return <button onClick={this.handleClick}>Click</button>;
  }
}

// Function component + hooks: vấn đề this không còn
// Vì function component không dùng this
function Button() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount((c) => c + 1)}>{count}</button>;
}
```

**Câu 1: V8 xác định `this` như thế nào? Mô tả 4 rules theo thứ tự ưu tiên và tại sao arrow function là ngoại lệ.**

**Đáp án:**

V8 xác định `this` mỗi khi một function được gọi — không phải lúc định nghĩa. Quy trình check theo thứ tự ưu tiên giảm dần:

**Rule 1 — new binding:** Khi gọi `new Foo()`, V8 tạo object mới trong heap, set prototype của nó = `Foo.prototype`, gọi `Foo` với `this` = object mới đó, rồi return object đó. `this` = newly created object.

**Rule 2 — Explicit binding:** Khi dùng `.call(obj)`, `.apply(obj)`, hoặc `.bind(obj)`. V8 trực tiếp dùng `obj` làm `this`. Với `bind`, V8 tạo Bound Function Exotic Object với `[[BoundThis]] = obj` — không thể override bằng `call`/`apply` sau đó.

**Rule 3 — Implicit binding:** Khi gọi dạng `obj.method()`. V8 dùng object trước dấu chấm làm `this`. Nếu method bị tách ra (`const fn = obj.method; fn()`) thì không còn object trước dấu chấm — rule này không apply.

**Rule 4 — Default binding:** Không có rule nào ở trên apply. Strict mode: `this = undefined`. Non-strict: `this = global object`.

**Arrow function ngoại lệ:** Arrow function có `[[ThisMode]] = lexical`. V8 không tạo `this` binding mới khi execute arrow function. Mọi reference đến `this` bên trong arrow function được resolve qua scope chain — giống variable thông thường — lấy `this` từ lexical outer scope tại thời điểm arrow function được định nghĩa. Không có rule nào ở trên apply cho arrow function.

**Câu 2: Tại sao `arguments` object bị coi là problematic và ES6 thay thế bằng rest parameters? Liệt kê ít nhất 3 vấn đề cụ thể.**

**Đáp án:**

`arguments` có 5 vấn đề cụ thể:

**1. Không phải real Array:** `arguments` là Array-like object — có `length` và numeric indices, nhưng không có Array methods (`map`, `filter`, `reduce`...). Phải convert thủ công qua `Array.from(arguments)` hoặc `[...arguments]` trước khi dùng.

**2. Không hoạt động trong arrow functions:** Arrow function không có `arguments` binding riêng. Nếu dùng `arguments` trong arrow function, nó sẽ reference `arguments` của outer non-arrow function — behavior gây nhầm lẫn và khó debug.

**3. Không có tên mô tả:** `arguments[0]`, `arguments[1]` không nói lên ý nghĩa. Rest parameter `...userIds` hay `...items` rõ ràng hơn nhiều về intent.

**4. Khó optimize cho V8:** V8 phải tạo `arguments` object mỗi lần function được gọi, kể cả khi function không dùng. Với rest parameters, V8 chỉ tạo array khi thực sự cần. Trong hot paths, sự khác biệt là đo được.

**5. Không thể dùng với destructuring ngay:** `const [first, ...rest] = arguments` phải convert trước. Rest parameters thì tự nhiên: `function fn([first, ...rest])` hoặc `function fn(...args)` rồi destructure `args`.

**Câu 3: Pure function là gì theo định nghĩa cơ chế — không phải định nghĩa sách giáo khoa? Tại sao React yêu cầu component render logic là pure?**

**Đáp án:**

Pure function là function thỏa **đồng thời** 2 điều kiện:

**1. Referential transparency (deterministic):** Với cùng input arguments, luôn trả về cùng output — không phụ thuộc vào external state (global variables, closure over mutable state, thời gian, random...). Có thể thay thế bất kỳ lần gọi nào bằng giá trị return của nó mà không đổi behavior của chương trình.

**2. No side effects:** Không thay đổi bất cứ thứ gì nằm ngoài function scope — không mutate arguments, không ghi vào global/closure state, không làm I/O (network, file, console, DOM), không gọi impure functions.

**Tại sao React yêu cầu render logic là pure:**

React Strict Mode (development) cố tình gọi render function **hai lần** cho mỗi render — chính xác để detect side effects trong render logic. Nếu render là pure, gọi hai lần cho cùng props/state phải trả về cùng JSX — không có gì thay đổi. Nếu có side effect (API call, DOM mutation, counter++...), gọi hai lần sẽ lộ bug.

Ngoài ra, React 18's Concurrent Mode có thể pause, abandon, và restart renders. Nếu render logic có side effects, restarting render sẽ trigger side effects nhiều lần — gây data inconsistency. Đây là lý do `useEffect` được tách biệt: để isolate side effects khỏi render, chỉ chạy sau khi DOM được commit, không bị restart giữa chừng.
