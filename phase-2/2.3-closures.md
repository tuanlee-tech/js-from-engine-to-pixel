# Phase 2 — Bài 2.3: Closures

> **Độ ưu tiên:** Tất cả 🔴, một phần 🟡 — Closures là cơ chế nền tảng của module pattern, React hooks, event handlers, memoization, và hầu hết mọi thứ "thú vị" trong JS. Hiểu sai closure = debug sai hướng mãi mãi.

---

## 1. Cơ chế thật

### Closure không phải "function nhớ biến" — đó là định nghĩa sai

Định nghĩa phổ biến: _"Closure là function có thể truy cập biến từ scope bên ngoài"_. Định nghĩa này sai về cơ chế — nó mô tả **kết quả** chứ không phải **cơ chế**.

Định nghĩa đúng về cơ chế: **Closure là sự kết hợp giữa một function object và Lexical Environment tại thời điểm function đó được tạo ra.** Cụ thể hơn: mỗi function object trong V8 có một internal slot tên `[[Environment]]` — một **reference** đến Lexical Environment nơi function được định nghĩa. Đây không phải copy, đây là **reference trực tiếp**.

---

### Lexical Environment là gì trong V8

Mỗi khi V8 bắt đầu execute một block code (function call, block `{}`, module), nó tạo ra một **Lexical Environment** — một cấu trúc data gồm 2 phần:

```
Lexical Environment
┌─────────────────────────────────────┐
│  Environment Record                  │  ← nơi lưu actual bindings
│  ┌─────────────────────────────────┐│
│  │  name: "Alice"                  ││
│  │  count: 0                       ││
│  │  increment: [Function]          ││
│  └─────────────────────────────────┘│
│                                      │
│  outer: [ref to outer environment]   │  ← pointer đến scope cha
└─────────────────────────────────────┘
```

Khi một function được **tạo ra** (không phải gọi — tạo ra), V8 làm 2 việc:

1. Tạo function object trong heap
2. Gán `function.[[Environment]] = currentLexicalEnvironment` — ghi reference đến environment hiện tại vào trong function object

Khi function đó được **gọi**, V8 tạo một Lexical Environment mới cho lần gọi đó, và set `outer` của environment mới = `function.[[Environment]]`. Đây là cách scope chain được nối lại.

---

### Lifecycle của Lexical Environment

Đây là phần quan trọng nhất — và là nguồn gốc của memory leak liên quan đến closure.

**Bình thường — không có closure:**

```javascript
function createUser(name) {
  // Khi createUser được gọi:
  // V8 tạo Lexical Environment cho lần gọi này
  // Environment chứa binding: name = "Alice"

  const greeting = `Hello ${name}`;
  return greeting;

  // Khi createUser return:
  // Không có gì còn reference đến Lexical Environment này
  // GC sẽ collect nó → memory được giải phóng
}

createUser('Alice'); // Env được tạo rồi destroyed
```

**Với closure — environment sống sót:**

```javascript
function makeCounter(initial) {
  // V8 tạo Lexical Environment E1 cho lần gọi makeCounter này
  // E1.count = initial = 0

  let count = initial;

  // Khi function object `increment` được tạo:
  // increment.[[Environment]] = E1  ← reference đến E1
  function increment() {
    count++; // access E1.count thông qua [[Environment]]
    return count;
  }

  return increment;
  // makeCounter return → nhưng E1 KHÔNG bị GC
  // Vì `increment` object vẫn đang reference E1 qua [[Environment]]
  // E1 sẽ sống chừng nào `increment` còn sống
}

const counter = makeCounter(0);
// counter = increment function object
// counter.[[Environment]] → E1 (count = 0)

counter(); // 1 — đọc và ghi E1.count
counter(); // 2 — cùng E1.count
counter(); // 3

// Khi counter = null:
// increment function object bị GC
// Không còn reference đến E1
// E1 bị GC → count bị giải phóng
```

### Sharing model — khi nào share, khi nào không

Đây là nguồn gốc của nhiều bug closure. Quy tắc: **các functions được tạo trong cùng một Lexical Environment thì share cùng Environment Record — chúng nhìn thấy cùng binding, không phải copy.**

```javascript
function makeCounter(initial) {
  let count = initial; // binding trong E1

  // Cả increment và decrement được tạo trong cùng E1
  // Cả hai có [[Environment]] = E1
  // Cả hai share cùng binding `count`
  return {
    increment: () => ++count, // [[Environment]] → E1
    decrement: () => --count, // [[Environment]] → E1 (cùng chỗ)
    getCount: () => count, // [[Environment]] → E1 (cùng chỗ)
  };
}

const c = makeCounter(0);
c.increment(); // count = 1
c.increment(); // count = 2
c.decrement(); // count = 1
c.getCount(); // 1 — tất cả đang đọc cùng 1 binding
```

**Ngược lại — mỗi lần gọi tạo environment riêng:**

```javascript
const c1 = makeCounter(0); // tạo E1, count riêng
const c2 = makeCounter(0); // tạo E2, count riêng — KHÁC E1

c1.increment(); // E1.count = 1
c2.increment(); // E2.count = 1
c2.increment(); // E2.count = 2

c1.getCount(); // 1 — E1 không bị ảnh hưởng bởi E2
c2.getCount(); // 2
```

### 3 loại bug closure kinh điển

#### Bug 1: Shared binding trong loop (async)

```javascript
// Đã cover ở bài 2.2 nhưng cần hiểu sâu hơn về cơ chế
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// Output: 3, 3, 3

// Cơ chế:
// Loop tạo 3 function objects (callbacks)
// Mỗi callback.[[Environment]] → CÙNG một Environment (function/global scope)
// CÙNG một binding `i`
// Khi callbacks chạy, `i` đã là 3
```

#### Bug 2: Async timing — closure capture binding, không phải value

```javascript
// Đây là bug tinh tế hơn — không phải loop, nhưng cùng cơ chế

async function processOrders(orderIds) {
  let currentOrder = null;

  for (const id of orderIds) {
    currentOrder = id; // reassign binding

    // BUG: closure capture BINDING của currentOrder
    // không phải VALUE tại thời điểm này
    await fetch(`/orders/${id}`).then((result) => {
      // Nếu các fetch overlap, `currentOrder` có thể đã thay đổi
      console.log(`Processing ${currentOrder}`); // có thể sai
    });
  }
}

// Tại sao? Vì arrow function.[[Environment]] → scope của processOrders
// scope đó chứa binding `currentOrder`
// Khi callback chạy, binding đó có thể đã bị reassign
```

#### Bug 3: Stale closure — function "đóng băng" environment cũ

```javascript
// Pattern phổ biến trong React hooks
function useDebounce(callback, delay) {
  // callback ở đây là closure — nó capture state/props tại thời điểm render
  const timeoutRef = useRef(null);

  // BUG: debouncedFn.[[Environment]] capture `callback` lần render đầu
  // Các render sau tạo `callback` mới, nhưng debouncedFn cũ vẫn giữ callback cũ
  const debouncedFn = useCallback(() => {
    clearTimeout(timeoutRef.current);
    timeoutRef.current = setTimeout(callback, delay); // callback CŨ
  }, []); // empty deps → debouncedFn không bao giờ được recreate

  return debouncedFn;
}

// Fix pattern: dùng ref để "escape" stale closure
function useDebounce(callback, delay) {
  const callbackRef = useRef(callback);

  // Luôn update ref với callback mới nhất
  // useLayoutEffect để update trước khi bất kỳ effect nào chạy
  useLayoutEffect(() => {
    callbackRef.current = callback;
  });

  const debouncedFn = useCallback(() => {
    clearTimeout(timeoutRef.current);
    // Dùng ref thay vì closure — ref.current luôn là giá trị mới nhất
    timeoutRef.current = setTimeout(() => callbackRef.current(), delay);
  }, [delay]); // chỉ recreate khi delay thay đổi

  return debouncedFn;
}
```

### 🟡 Closure-based private state vs class private fields (`#`)

Hai cách tạo "private" data — trade-offs khác nhau:

```javascript
// Closure-based private:
function makeUser(name, age) {
  // name, age nằm trong Lexical Environment
  // Không thể access từ ngoài — thật sự private
  let _age = age;

  return {
    getName: () => name,
    getAge: () => _age,
    birthday: () => {
      _age++;
    },
  };
}

const user = makeUser('Alice', 30);
user._age; // undefined — không phải private, property không tồn tại
user.getAge(); // 30 — chỉ access được qua closure

// Class private fields (#):
class User {
  #age; // thật sự private — V8 enforce ở engine level

  constructor(name, age) {
    this.name = name;
    this.#age = age;
  }

  getAge() {
    return this.#age;
  }
  birthday() {
    this.#age++;
  }
}

const user2 = new User('Alice', 30);
user2.#age; // SyntaxError — ngay cả devtools cũng không đọc được
```

**Trade-offs:**

|             | Closure                                 | Class `#`                                 |
| ----------- | --------------------------------------- | ----------------------------------------- |
| Memory      | Mỗi instance tạo function objects riêng | Methods shared qua prototype              |
| Performance | Chậm hơn với nhiều instances            | Nhanh hơn — V8 optimize prototype methods |
| Inheritance | Khó                                     | Native với `extends`                      |
| Debugging   | Không thấy trong DevTools               | Hiện trong DevTools với `[[Scopes]]`      |
| Use case    | Factory function, module pattern        | Class-based OOP                           |

**Về memory:** Với closure pattern, mỗi instance của `makeUser` tạo 3 function objects mới (`getName`, `getAge`, `birthday`) trong heap. Với class, tất cả instances share cùng method objects qua prototype chain. Với 10,000 instances, sự khác biệt là rõ ràng.

### 🟡 Memory implications trong hot paths

```javascript
// VẤND ĐỀ: closure giữ reference đến large data
function processLargeDataset(data) {
  // `data` là array 100MB
  const summary = data.reduce((acc, item) => acc + item.value, 0);

  // BUG: callback này capture TOÀN BỘ `data` qua [[Environment]]
  // dù callback chỉ cần `summary`
  return {
    getSummary: () => summary,
    // ^ [[Environment]] → scope chứa cả `data` lẫn `summary`
    // `data` 100MB không thể bị GC chừng nào object này còn sống
  };
}

const result = processLargeDataset(hugeArray);
// hugeArray vẫn chiếm 100MB trong heap!

// FIX: tạo scope mới, chỉ capture thứ cần thiết
function processLargeDataset(data) {
  const summary = data.reduce((acc, item) => acc + item.value, 0);
  // data không còn cần thiết sau đây

  // IIFE tạo scope mới — chỉ capture `summary`, không capture `data`
  return (() => {
    const capturedSummary = summary; // copy value vào scope mới
    return {
      getSummary: () => capturedSummary,
      // [[Environment]] → scope của IIFE, chỉ chứa `capturedSummary`
      // `data` không có trong scope này → có thể bị GC
    };
  })();
}
```

### Stale closure — React render cycle

```
Render 1:
  state = { query: "" }
  effect được tạo → effect.[[Environment]] → E_render1
  E_render1.query = ""

  [ effect1: () => fetch(query) ]   query = "" ← stale nếu không cleanup

Render 2 (query changed to "Alice"):
  state = { query: "Alice" }
  effect mới được tạo → effect.[[Environment]] → E_render2
  E_render2.query = "Alice"

  Nếu deps array sai → effect1 vẫn đang chạy với query = ""
  Nếu deps array đúng → effect1 bị cleanup, effect2 chạy với query = "Alice"

KEY INSIGHT: Mỗi render tạo closure mới với snapshot của state/props lúc đó.
"Stale closure" = closure cũ vẫn đang chạy sau khi state đã thay đổi.
```

## 3. Ví dụ code

### Module pattern — closure để tạo private state thật sự

```javascript
// Closure-based module: private state không expose ra ngoài
function createAuthService(config) {
  // Các biến này private — nằm trong Lexical Environment của createAuthService
  // Không thể access từ outside trừ qua public API bên dưới
  let accessToken = null;
  let refreshToken = null;
  let tokenExpiry = null;

  // Private helper — cũng là closure, capture cùng environment
  async function refreshAccessToken() {
    const response = await fetch('/auth/refresh', {
      method: 'POST',
      body: JSON.stringify({ refreshToken }), // access private binding
    });
    const data = await response.json();
    accessToken = data.accessToken; // mutate private binding
    tokenExpiry = Date.now() + data.expiresIn * 1000;
  }

  // Public API — chỉ expose những gì cần thiết
  return {
    async login(username, password) {
      const res = await fetch('/auth/login', {
        method: 'POST',
        body: JSON.stringify({ username, password }),
      });
      const data = await res.json();
      accessToken = data.accessToken; // mutate private binding
      refreshToken = data.refreshToken; // mutate private binding
      tokenExpiry = Date.now() + data.expiresIn * 1000;
    },

    async getToken() {
      // access private binding để check expiry
      if (Date.now() > tokenExpiry - 60000) {
        await refreshAccessToken(); // gọi private function
      }
      return accessToken; // return private binding value
    },

    logout() {
      // reset tất cả private bindings
      accessToken = null;
      refreshToken = null;
      tokenExpiry = null;
    },
  };
}

const auth = createAuthService({ baseUrl: '/api' });
auth.accessToken; // undefined — không tồn tại trên object
await auth.getToken(); // OK — thông qua public API
```

### Memoization — closure cache kết quả

```javascript
// Closure giữ `cache` Map sống trong suốt lifetime của memoized function
function memoize(fn) {
  // `cache` nằm trong Lexical Environment của memoize call này
  // Tất cả lần gọi memoizedFn đều share cùng `cache` binding
  const cache = new Map();

  return function memoizedFn(...args) {
    // closure capture `cache` và `fn` từ outer scope
    const key = JSON.stringify(args);

    if (cache.has(key)) {
      return cache.get(key); // cache hit — không gọi fn
    }

    const result = fn(...args);
    cache.set(key, result); // mutate shared `cache` binding
    return result;
  };
}

const expensiveCalc = memoize((n) => {
  // simulate expensive work
  return fibonacci(n);
});

expensiveCalc(40); // tính, lưu vào cache
expensiveCalc(40); // cache hit — trả về ngay
expensiveCalc(40); // cache hit

// CẢNH BÁO: cache không bao giờ bị clear → memory leak tiềm năng
// Fix cho production: dùng LRU cache với giới hạn size
function memoizeWithLimit(fn, maxSize = 100) {
  const cache = new Map(); // Map giữ insertion order

  return function (...args) {
    const key = JSON.stringify(args);

    if (cache.has(key)) {
      // LRU: move to end (delete rồi re-insert)
      const value = cache.get(key);
      cache.delete(key);
      cache.set(key, value);
      return value;
    }

    const result = fn(...args);

    if (cache.size >= maxSize) {
      // Xóa entry cũ nhất (first item trong Map)
      cache.delete(cache.keys().next().value);
    }

    cache.set(key, result);
    return result;
  };
}
```

### Partial application và currying — closure giữ arguments

```javascript
// Partial application: fix một số arguments, trả về function nhận arguments còn lại
function createApiClient(baseUrl, authToken) {
  // baseUrl và authToken được capture trong closure
  // Không cần truyền lại mỗi lần gọi

  async function request(method, endpoint, body) {
    // closure access baseUrl, authToken từ outer scope
    return fetch(`${baseUrl}${endpoint}`, {

        Authorization: `Bearer ${authToken}`,

    get: (endpoint) => request('GET', endpoint),
    post: (endpoint, body) => request('POST', endpoint, body),
    put: (endpoint, body) => request('PUT', endpoint, body),
    delete: (endpoint) => request('DELETE', endpoint),
  };
}

const api = createApiClient('https://api.example.com', 'token-xyz');
// Mỗi method là closure capture `request`, mà `request` capture `baseUrl` và `authToken`
```

### Memory leak pattern trong production — cách nhận ra

```javascript
// LEAK: closure trong event listener không được remove
class VideoPlayer {
  constructor(element) {
    this.element = element;
    this.playCount = 0;

    // Arrow function capture `this` (là closure capture outer `this` binding)
    // Event listener giữ reference đến arrow function
    // Arrow function giữ reference đến `this` (VideoPlayer instance)
    // Nếu không remove listener → VideoPlayer không bao giờ bị GC
    this.element.addEventListener('play', () => {
      this.playCount++;
      this.updateAnalytics(); // `this` từ closure
    });
  }

  // Nếu không có destroy() → memory leak
  destroy() {
    // Phải lưu reference đến function để có thể remove
    // Arrow function ẩn danh không remove được
  }
}

// FIX: lưu reference, remove khi destroy
class VideoPlayer {
  constructor(element) {
    this.element = element;
    this.playCount = 0;

    // Bind để tạo named function có thể remove
    this._onPlay = this.handlePlay.bind(this);
    this.element.addEventListener('play', this._onPlay);
  }

  handlePlay() {
    this.playCount++;
  }

  destroy() {
    this.element.removeEventListener('play', this._onPlay);
    // Sau khi remove, _onPlay không còn referenced bởi DOM
    // VideoPlayer instance có thể bị GC
  }
}
```

### React — hiểu hooks qua lens closure

```javascript
// useState, useEffect, useCallback đều dựa vào closure
// Mỗi render call của component tạo một Lexical Environment mới

function SearchBox() {
  // Mỗi render: Environment E_n được tạo
  // query, setQuery là bindings trong E_n
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);

  // useEffect callback là closure
  // callback.[[Environment]] → E_n (environment của render này)
  // Nếu deps array đúng, React tạo closure mới mỗi khi query thay đổi
  useEffect(() => {
    // Đây là closure: capture `query` từ E_n
    // `query` ở đây là VALUE tại thời điểm render này
    // Không phải "live reference" — là snapshot của render đó
    if (!query) return;

    fetchResults(query).then((data) => {
      setResults(data); // setResults là stable reference từ useState
    });
  }, [query]); // deps: re-create closure khi query thay đổi

  return (
    <input
      value={query}
      onChange={(e) => setQuery(e.target.value)}
    />
  );
}

// KEY INSIGHT:
// React hooks không phải magic — chúng là closures + một array of states
// bên ngoài component function để persist data giữa các renders
```

---

**Câu 1: `[[Environment]]` internal slot là gì và nó được gán khi nào trong lifecycle của một function?**

**Đáp án:**

`[[Environment]]` là một internal slot — một field ẩn bên trong mỗi function object trong V8, không accessible từ JS code. Nó lưu một **reference** (pointer) đến Lexical Environment tại thời điểm function được **tạo ra** (không phải gọi).

Khi V8 execute một expression tạo function — `function foo() {}`, `() => {}`, `function* gen() {}` — nó thực hiện: tạo function object trong heap, sau đó gán `functionObject.[[Environment]] = currentLexicalEnvironment`. `currentLexicalEnvironment` là environment đang active tại dòng code đó.

Khi function được gọi sau này, V8 tạo Execution Context mới với Lexical Environment mới, và set `newEnv.outer = function.[[Environment]]`. Đây là cách scope chain được nối — closure có thể nhìn thấy biến từ outer scope vì outer được link qua `[[Environment]]`.

Đây không phải copy của environment — là reference trực tiếp. Nên nếu binding trong outer environment thay đổi, closure sẽ thấy giá trị mới (không phải snapshot tại thời điểm tạo closure).

---

**Câu 2: Tại sao closure có thể gây memory leak, và làm sao nhận ra nó bằng DevTools?**

**Đáp án:**

Garbage Collector chỉ thu hồi memory khi không còn reference nào đến object đó. Lexical Environment là object trong heap — nó bị GC khi không còn gì reference đến nó.

Memory leak xảy ra khi: một closure (function object) được giữ sống lâu hơn cần thiết (qua event listener, timer, global variable, module-level cache...), và closure đó hold reference đến một Lexical Environment lớn qua `[[Environment]]`. V8 không thể GC environment đó dù data trong đó không còn được dùng — vì về mặt lý thuyết, closure có thể access bất kỳ binding nào trong environment.

Ví dụ: một callback trong `addEventListener` capture một Lexical Environment chứa 100MB data. Dù callback không dùng data đó, V8 vẫn giữ toàn bộ environment sống.

Nhận ra bằng DevTools: `Memory tab → Take Heap Snapshot`. Tìm các object có retained size lớn bất thường. Click vào object → xem `retaining path` — nếu thấy `(closure) → Lexical Environment → large data`, đó là dấu hiệu. Trong `Sources`, kiểm tra `Scope panel` khi breakpoint — nếu thấy binding lớn trong Closure scope mà function không cần dùng, đó là candidate để fix.

**Câu 3: Stale closure là gì? Tại sao nó xảy ra trong React hooks và cách fix theo cơ chế?**

**Đáp án:**

Stale closure xảy ra khi một closure được tạo tại thời điểm A, capture các binding có giá trị X, nhưng vẫn đang chạy sau khi binding đó đã được reassign sang giá trị Y — và closure vẫn thấy giá trị cũ X vì nó reference environment của thời điểm A.

Trong React, mỗi render tạo một Lexical Environment mới với snapshot của `state` và `props` lúc đó. Các closures (event handlers, effect callbacks, memo callbacks) được tạo trong render đó có `[[Environment]]` trỏ đến environment của render đó.

Nếu một closure tồn tại qua nhiều renders mà không được recreate (do deps array sai hoặc `useCallback([])`) — nó vẫn giữ `[[Environment]]` của render cũ. Khi state thay đổi và component re-render, binding trong global state đã đổi, nhưng closure cũ không thấy vì nó trỏ vào environment của render cũ.

**Fix theo 2 cách:**

1. **Recreate closure khi dependency thay đổi**: đặt đúng deps vào `useEffect`/`useCallback`. React sẽ tạo closure mới với `[[Environment]]` mới sau mỗi render relevant.

2. **Dùng ref để escape closure**: Lưu giá trị mới nhất vào `useRef`. Ref object là stable reference — cùng object qua mọi render. Closure capture ref object (không đổi), nhưng đọc `ref.current` để lấy giá trị mới nhất. Pattern này phù hợp khi không muốn trigger re-render hay recreate closure.
