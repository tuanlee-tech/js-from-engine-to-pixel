# Phase 2 — Bài 2.2: Variables & Scoping

> **Độ ưu tiên:** Tất cả 🔴 — đây là nguồn gốc của một lớp bug cực kỳ phổ biến, từ loop closures đến React stale state. Sau bài này bạn sẽ đọc được code và biết ngay variable nào "sống" ở đâu, "chết" khi nào.

---

## 1. Cơ chế thật

### V8 xử lý variable như thế nào trước khi chạy một dòng code?

Trước khi execute bất kỳ dòng code nào, V8 trải qua một bước gọi là **compilation phase** (còn gọi là parse phase). Trong bước này, V8 đọc toàn bộ source code, xây dựng AST (Abstract Syntax Tree), và **quyết định scope của từng variable**. Đây là nền tảng của mọi thứ trong phần này.

Điều quan trọng cần nhớ: **scope được xác định lúc viết code, không phải lúc chạy**. Đây gọi là **lexical scoping** (hay static scoping). Khi bạn nhìn vào code, bạn có thể xác định scope của mọi variable mà không cần chạy code.

---

### `var` — hoisting và function scope

`var` là cơ chế cũ nhất, được thiết kế từ 1995. Nó có 2 đặc điểm kỳ lạ:

**1. Hoisting — declaration bị "kéo lên" đầu function scope**

```javascript
// Code bạn viết:
function process() {
  console.log(name); // undefined — không phải ReferenceError
  var name = 'Alice';
  console.log(name); // "Alice"
}
```

```javascript
// Cách V8 thật sự thấy code này sau compilation phase:
function process() {
  var name; // Declaration bị hoist lên đầu function
  // V8 tạo binding trong function scope ngay từ đầu
  // và gán giá trị undefined
  console.log(name); // undefined — binding tồn tại, nhưng chưa có value
  name = 'Alice'; // Assignment giữ nguyên vị trí — không bị hoist
  console.log(name); // "Alice"
}
```

**Cơ chế bên trong V8:** Trong compilation phase, V8 scan toàn bộ function body, tìm tất cả `var` declarations, và tạo binding trong **Variable Environment** của function ngay từ đầu — gán `undefined`. Chỉ declaration bị hoist, không phải assignment.

**2. Function scope — không phải block scope**

```javascript
// var xuyên qua block {} — đây là behavior cố tình
function loadConfig() {
  if (true) {
    var timeout = 5000; // V8 đặt binding này vào FUNCTION scope
    // không phải if-block scope
  }
  console.log(timeout); // 5000 — vẫn accessible
}

// Hệ quả nguy hiểm trong loop:
for (var i = 0; i < 3; i++) {
  // var i thuộc về function scope (hoặc global scope)
  // không phải loop-block scope
}
console.log(i); // 3 — i vẫn tồn tại sau loop
```

### `let` và `const` — block scope và TDZ

ES6 giới thiệu `let` và `const` để fix các vấn đề của `var`. Chúng có 2 đặc điểm khác biệt:

**1. Block scope**

```javascript
function loadConfig() {
  if (true) {
    let timeout = 5000; // V8 tạo binding trong IF-BLOCK scope
    const limit = 100; // cũng trong IF-BLOCK scope
  }
  console.log(timeout); // ReferenceError — binding không tồn tại ở đây
}
```

V8 tạo một **Lexical Environment** mới cho mỗi block `{}`. `let`/`const` bindings chỉ tồn tại trong Lexical Environment của block chứa chúng.

**2. Temporal Dead Zone (TDZ)**

Đây là cơ chế quan trọng nhất và hay bị hiểu sai nhất.

`let` và `const` **cũng bị hoist** — nhưng không được khởi tạo với `undefined`. Thay vào đó, V8 đặt chúng vào trạng thái **"uninitialized"**. Khoảng thời gian từ khi scope bắt đầu đến khi declaration được execute gọi là **Temporal Dead Zone**.

```javascript
// V8 thật sự làm gì:
{
  // ← TDZ của `name` bắt đầu ở đây
  // V8 đã biết `name` tồn tại (từ compilation phase)
  // nhưng đánh dấu là "uninitialized" — không phải undefined

  console.log(name); // ReferenceError: Cannot access 'name' before initialization
  // ^ V8 check: binding tồn tại không? CÓ. Initialized chưa? CHƯA → throw

  let name = 'Alice'; // ← TDZ kết thúc ở đây — binding được khởi tạo
  // ^ Lúc này V8 mới gán "Alice" vào binding

  console.log(name); // "Alice"
}
```

**Tại sao TDZ được thiết kế như vậy?**

Để bắt lỗi sớm hơn. Với `var`, accessing trước declaration trả về `undefined` — silent bug. Với `let`/`const`, bạn nhận được error ngay — fail fast, dễ debug hơn.

**TDZ với `const`:**

`const` có thêm một ràng buộc: binding phải được initialize ngay tại declaration và không thể reassign:

```javascript
const x; // SyntaxError: Missing initializer — V8 phát hiện lúc parse
const y = 5;
y = 10; // TypeError: Assignment to constant variable
// ^ V8 check "writable" flag của binding — false với const
```

**Quan trọng:** `const` không làm object immutable — chỉ ngăn reassign binding:

```javascript
const user = { name: 'Alice' };
user = { name: 'Bob' }; // TypeError — binding bị reassign
user.name = 'Bob'; // OK — object property thay đổi
// const chỉ lock binding, không lock object trong heap
```

### Lexical Scoping — scope chain lookup

Khi V8 gặp một identifier (tên variable), nó tìm kiếm theo **scope chain** — từ scope hiện tại lên outer scope, cho đến global scope. Quá trình này xảy ra lúc runtime nhưng **cấu trúc chain được xác định lúc compile time**.

```javascript
const apiBase = 'https://api.example.com'; // global scope

function createRequest(endpoint) {
  // function scope
  const method = 'GET'; // function scope

  function buildUrl() {
    // inner function scope
    // V8 tìm `endpoint`:
    //   1. buildUrl scope? KHÔNG
    //   2. createRequest scope? CÓ → dùng giá trị này
    //
    // V8 tìm `apiBase`:
    //   1. buildUrl scope? KHÔNG
    //   2. createRequest scope? KHÔNG
    //   3. global scope? CÓ → dùng giá trị này
    return `${apiBase}/${endpoint}`;
  }

  return { url: buildUrl(), method };
}
```

**Đây chính là closure** — `buildUrl` "nhìn thấy" `endpoint` và `apiBase` không phải vì được truyền vào, mà vì scope chain của nó include scope của function cha, được xác định lúc `buildUrl` được định nghĩa.

### Hoisting đầy đủ — tất cả các loại

V8 hoist nhiều thứ hơn chỉ `var`. Thứ tự ưu tiên từ cao đến thấp:

```
1. Function declarations      ← hoist TOÀN BỘ (declaration + body)
2. var declarations           ← hoist declaration, init = undefined
3. let/const declarations     ← hoist declaration, nhưng TDZ (uninitialized)
```

```javascript
// Function declaration — hoisted hoàn toàn
console.log(greet('Alice')); // "Hello Alice" — gọi được trước declaration

function greet(name) {
  return `Hello ${name}`;
}

// Function expression — KHÔNG hoisted (chỉ var binding bị hoist)
console.log(greet2('Bob')); // TypeError: greet2 is not a function
// var greet2 bị hoist → undefined
// undefined("Bob") → TypeError

var greet2 = function (name) {
  return `Hello ${name}`;
};

// Arrow function expression — tương tự function expression
console.log(greet3('Carol')); // ReferenceError (nếu let) hoặc TypeError (nếu var)
const greet3 = (name) => `Hello ${name}`;
```

**Tại sao function declaration được hoist toàn bộ?**

Lý do thực dụng: cho phép viết code theo thứ tự tự nhiên — gọi function ở đầu, định nghĩa ở cuối. JavaScript được thiết kế để dễ dùng cho người mới, không phải cho compiler.

### Module scope

ES Modules tạo ra một loại scope mới — **module scope**. Mỗi file `.js` với `type="module"` (browser) hoặc `.mjs` (Node.js) có scope riêng, không leak vào global.

```javascript
// utils.js — module scope
const API_KEY = "secret"; // Không phải global — chỉ tồn tại trong module này
export function fetchUser(id) { ... }

// main.js
import { fetchUser } from './utils.js';
console.log(API_KEY); // ReferenceError — không access được từ ngoài
```

### `var` vs `let` trong loop — memory layout

```javascript
// VẤN ĐỀ với var trong loop + closure:
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// Output: 3, 3, 3 — không phải 0, 1, 2
```

```
Scope layout với var:

  Loop Scope (FUNCTION scope)
  ┌──────────────────────────────┐
  │  i                           │  ← CHỈ MỘT binding cho cả 3 iterations
  │  (shared across all iters)   │
  └──────────┬───────────────────┘
             │ tất cả 3 closures đều
             │ reference CÙNG một binding
             ▼
  [closure 1] ──►  i  (đọc lúc chạy: 3)
  [closure 2] ──►  i  (đọc lúc chạy: 3)
  [closure 3] ──►  i  (đọc lúc chạy: 3)

Khi setTimeout callbacks chạy, loop đã xong, i = 3 rồi.
```

```
Scope layout với let:

  Iteration 1 scope    Iteration 2 scope    Iteration 3 scope
  ┌─────────────┐      ┌─────────────┐      ┌─────────────┐
  │  i = 0      │      │  i = 1      │      │  i = 2      │
  └──────┬──────┘      └──────┬──────┘      └──────┬──────┘
         │                    │                    │
         ▼                    ▼                    ▼
  [closure 1]          [closure 2]          [closure 3]
  reads i=0            reads i=1            reads i=2

let tạo MỘT binding MỚI cho mỗi iteration của loop.
```

## 3. Ví dụ code

### Classic loop closure bug

```javascript
// BUG: var trong loop + async callback
function setupButtonHandlers(buttons) {
  for (var i = 0; i < buttons.length; i++) {
    // var i thuộc về function scope của setupButtonHandlers
    // tất cả event handlers share cùng binding `i`
    buttons[i].addEventListener('click', () => {
      console.log(`Button ${i} clicked`);
      // Khi user click, loop đã xong → i = buttons.length
    });
  }
}
// Kết quả: mọi button đều log index cuối cùng

// FIX 1: dùng let — tạo new binding mỗi iteration
function setupButtonHandlers(buttons) {
  for (let i = 0; i < buttons.length; i++) {
    // let i tạo binding mới cho mỗi iteration
    buttons[i].addEventListener('click', () => {
      console.log(`Button ${i} clicked`); // đúng index
    });
  }
}

// FIX 2 (legacy — trước ES6): IIFE để tạo scope mới
function setupButtonHandlers(buttons) {
  for (var i = 0; i < buttons.length; i++) {
    (function (index) {
      // IIFE tạo function scope mới, copy giá trị i vào param `index`
      buttons[index].addEventListener('click', () => {
        console.log(`Button ${index} clicked`);
      });
    })(i);
  }
}
```

### Scope leak trong legacy code

```javascript
// BUG: var trong if block thoát ra ngoài
function processPayment(amount) {
  if (amount > 1000) {
    var fee = amount * 0.02; // V8 đặt fee vào function scope
    var discounted = amount - fee;
  }

  // Không có guard → fee và discounted là undefined nếu amount <= 1000
  // Không phải ReferenceError — var tồn tại nhưng = undefined
  return discounted; // undefined — silent bug, không throw
}

// FIX: let/const block scope
function processPayment(amount) {
  if (amount > 1000) {
    const fee = amount * 0.02; // scope: if block
    const discounted = amount - fee; // scope: if block
    return discounted;
  }
  return amount;
  // Sau if block, fee và discounted không tồn tại
  // Nếu vô tình access → ReferenceError ngay → dễ debug
}
```

### Shadowing — scope cha bị "che" bởi scope con

```javascript
// Shadowing là tính năng, không phải bug — nhưng hay gây nhầm lẫn
const config = { timeout: 5000 };

function fetchWithRetry(url) {
  const config = { timeout: 1000, retries: 3 };
  // Binding `config` này shadow binding `config` ở outer scope
  // V8 tìm `config` → thấy ở local scope → dừng tìm

  console.log(config.timeout); // 1000 — local config, không phải global
}

fetchWithRetry('/api/users');
console.log(config.timeout); // 5000 — global config không bị affect

// THỰC TẾ: Đây là nguồn gốc bug khi đặt tên trùng nhau
// DevTools → Sources → hover lên variable → hiện scope nào nó belong to
```

## 4. Ứng dụng thực tế

### Performance — `var` vs `let`/`const` trong V8

`let`/`const` không chỉ safer mà còn giúp V8 optimize tốt hơn:

- Block scope hạn chế **scope chain length** mà V8 phải traverse khi lookup variable
- `const` signals cho V8 rằng binding không thay đổi → V8 có thể aggressively inline giá trị
- Tránh được một số deoptimization path mà `var` trong closures hay trigger

Trong hot paths (functions gọi hàng triệu lần), đây là sự khác biệt đo được.

### React — stale closure và scope

```javascript
// Stale closure trong React hooks — nguồn gốc từ lexical scoping
function SearchComponent() {
  const [query, setQuery] = useState('');

  useEffect(() => {
    // Closure này capture `query` lúc effect chạy
    // Nếu query thay đổi, effect cũ vẫn giữ giá trị query cũ
    const timer = setTimeout(() => {
      fetchResults(query); // `query` ở đây là giá trị CŨ
    }, 500);

    return () => clearTimeout(timer);
  }, []); // deps array rỗng → effect chỉ chạy 1 lần → query luôn là ""

  // FIX: thêm query vào deps array
  // hoặc dùng useRef để không trigger stale closure
}
```

**Tại sao đây là vấn đề scoping?** Vì closure capture binding tại thời điểm function được tạo. Mỗi render tạo một closure mới với `query` mới — nhưng nếu deps array sai, effect cũ không bị cleanup, và closure cũ vẫn hold `query` cũ.

### Debugging với DevTools

```
Chrome DevTools → Sources tab → đặt breakpoint:

1. Nhìn vào panel "Scope" bên phải:
   - Local: variables trong current function scope
   - Block: variables trong current block (let/const)
   - Closure: variables từ outer scope mà current closure capture
   - Global: global scope

2. Hover lên bất kỳ variable nào trong code → tooltip show giá trị hiện tại
   + tên scope nó thuộc về

3. Khi gặp "Cannot access X before initialization":
   → X đang trong TDZ
   → Tìm declaration của X, xem có code nào access trước đó không
```

**Câu 1: Tại sao `let` và `const` cũng bị hoist, nhưng lại throw `ReferenceError` khi access trước declaration? Điều gì xảy ra bên trong V8 trong khoảng thời gian đó?**

**Đáp án:**

`let` và `const` bị hoist theo nghĩa: trong compilation phase, V8 đã biết sự tồn tại của binding đó trong scope, và tạo entry cho nó trong Lexical Environment. Tuy nhiên, khác với `var` được khởi tạo ngay với `undefined`, V8 đánh dấu `let`/`const` binding là **uninitialized** — một trạng thái đặc biệt.

Khoảng thời gian từ khi scope bắt đầu (binding được hoist nhưng chưa initialized) đến khi dòng declaration được execute gọi là **Temporal Dead Zone (TDZ)**. Khi V8 gặp một identifier lookup trong runtime, nó kiểm tra: "binding này tồn tại không?" — CÓ. "Nó đã initialized chưa?" — CHƯA → throw `ReferenceError: Cannot access before initialization`.

Đây là behavior cố tình: thay vì trả về `undefined` như `var` (silent bug), TDZ buộc developer phải fix thứ tự code ngay lập tức. Fail fast = dễ debug hơn.

---

**Câu 2: Đoạn code sau output gì và tại sao? Giải thích từng bước V8 thực hiện.**

```javascript
var x = 1;

function outer() {
  console.log(x); // (A)
  var x = 2;

  function inner() {
    console.log(x); // (B)
  }
  inner();
}

outer();
console.log(x); // (C)
```

**Đáp án:**

- **(A)** → `undefined`
- **(B)** → `2`
- **(C)** → `1`

**Giải thích:**

**Compilation phase** — V8 scan và hoist:

- Global scope: `var x` → `x = undefined`, `function outer` → hoisted hoàn toàn
- `outer` scope: `var x` → `x = undefined` (đây là binding **khác** với global `x` — shadow), `function inner` → hoisted
- `inner` scope: không có local variable

**Execution phase:**

- Global `x = 1` → global `x` được assign `1`
- `outer()` được gọi → tạo execution context cho `outer`
- **(A)**: V8 tìm `x` trong `outer` scope → tìm thấy local `x` (đã hoist, giá trị hiện tại là `undefined`) → log `undefined`. **Đây là hoisting** — declaration của `var x = 2` bên dưới đã được hoist lên đầu `outer`, shadow global `x`, nhưng assignment `= 2` chưa chạy.
- `outer`'s local `x = 2` → local `x` được assign `2`
- `inner()` được gọi → tạo execution context
- **(B)**: V8 tìm `x` trong `inner` scope → không có → leo lên `outer` scope → tìm thấy `x = 2` → log `2`
- **(C)**: V8 tìm `x` trong global scope → `x = 1` (chưa bị thay đổi — local `x` trong `outer` là binding riêng) → log `1`

**Câu 3: Tại sao `for (var i...)` với async callback là bug, trong khi `for (let i...)` lại đúng? Sự khác biệt nằm ở đâu trong cách V8 tạo scope?**

**Đáp án:**

Sự khác biệt nằm ở số lượng **binding** được tạo ra.

Với `var`: `i` thuộc function scope (hoặc global scope nếu không có function bao ngoài). Toàn bộ vòng lặp **chỉ có một binding duy nhất** cho `i`. Ba closure được tạo ra trong 3 iterations đều reference cùng một binding đó. Khi các async callbacks (setTimeout, event handlers) chạy sau khi loop kết thúc, chúng đọc giá trị hiện tại của binding đó — lúc này là giá trị cuối cùng sau loop.

Với `let`: Theo spec ES6, `let` trong `for` loop khiến V8 tạo một **Lexical Environment mới** cho mỗi iteration. Mỗi iteration có **binding riêng** cho `i`, được khởi tạo bằng giá trị của iteration đó. Closure được tạo trong mỗi iteration capture binding của iteration đó — không phải binding chung. Do đó mỗi closure có snapshot riêng của `i`.

Nói cách khác: `var` → 1 binding, N closures trỏ vào cùng chỗ. `let` → N bindings, mỗi closure trỏ vào binding của iteration mình. Đây là lý do tại sao `let` giải quyết loop closure bug mà không cần IIFE workaround.
