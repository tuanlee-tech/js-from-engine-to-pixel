## 1. libuv & Event Loop Phases

### Cơ chế thật

Node.js **không phải** là JavaScript engine. Node.js là một **runtime** bao gồm:

```
┌─────────────────────────────────────────────────────┐
│                    Node.js Process                   │
│                                                     │
│  ┌────────────┐    ┌──────────────────────────────┐ │
│  │    V8      │    │           libuv              │ │
│  │  (JS exec) │◄──►│  (async I/O, event loop,     │ │
│  │            │    │   thread pool, OS bindings)   │ │
│  └────────────┘    └──────────────────────────────┘ │
│                                                     │
│  ┌──────────────────────────────────────────────┐   │
│  │         Node.js Core Modules                 │   │
│  │  (fs, net, http, crypto, path, ...)          │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

**libuv** là C library viết cross-platform, handle:

- Event loop
- Thread pool (default 4 threads) cho operations không thể async ở OS level (file I/O, DNS lookup, crypto)
- Async TCP/UDP networking
- Child processes
- Signal handling

**Điểm quan trọng nhất:** Browser event loop và Node.js event loop khác nhau. Browser không có "phases" — Node có.

---

### Node.js Event Loop: 6 Phases

```
   ┌───────────────────────────┐
┌─►│           timers          │  ← setTimeout, setInterval callbacks
│  └─────────────┬─────────────┘
│                │
│  ┌─────────────▼─────────────┐
│  │     pending callbacks     │  ← I/O errors từ vòng trước
│  └─────────────┬─────────────┘
│                │
│  ┌─────────────▼─────────────┐
│  │       idle, prepare       │  ← internal only, bạn không cần
│  └─────────────┬─────────────┘
│                │
│  ┌─────────────▼─────────────┐
│  │           poll            │  ← fetch I/O events, execute I/O callbacks
│  └─────────────┬─────────────┘
│                │
│  ┌─────────────▼─────────────┐
│  │           check           │  ← setImmediate callbacks
│  └─────────────┬─────────────┘
│                │
│  ┌─────────────▼─────────────┐
└──│      close callbacks      │  ← socket.on('close'), cleanup
   └───────────────────────────┘

Giữa MỖI phase transition:
   → Drain toàn bộ microtask queue (Promise.then + queueMicrotask)
   → Drain toàn bộ process.nextTick queue (TRƯỚC microtasks)
```

**Cơ chế chi tiết từng phase:**

#### Phase 1: Timers

```
libuv kiểm tra: "có setTimeout/setInterval nào đã expired chưa?"
→ Nếu có: execute callbacks
→ Không phải "đúng giờ" — là "đã qua ngưỡng threshold"
```

`setTimeout(fn, 100)` không nghĩa là "chạy sau đúng 100ms". Nghĩa là "chạy khi event loop đến timers phase VÀ đã qua ít nhất 100ms từ khi register."

#### Phase 4: Poll (quan trọng nhất)

Đây là phase Node.js dành phần lớn thời gian.

```
poll phase:
  1. Tính toán blocking time (bao lâu nên chờ I/O)
  2. Nếu poll queue KHÔNG rỗng:
     → Execute callbacks trong queue cho đến hết hoặc đến system limit
  3. Nếu poll queue RỖng:
     a. Nếu có setImmediate pending → đi tới check phase
     b. Không có setImmediate → chờ (block) cho đến khi:
        - I/O callback mới xuất hiện
        - Timer threshold đạt
        - Process receive signal
```

#### Phase 5: Check

`setImmediate` callbacks chạy ở đây — SAU poll, TRƯỚC timers của vòng tiếp theo.

---

### Visualize: Execution Order

```javascript
const fs = require('fs');

// Đây sẽ chạy theo thứ tự nào?
setTimeout(() => console.log('1: timeout'), 0);
setImmediate(() => console.log('2: immediate'));

fs.readFile('/etc/hostname', () => {
  // Bây giờ đang ở trong poll callback
  setTimeout(() => console.log('3: timeout inside I/O'), 0);
  setImmediate(() => console.log('4: immediate inside I/O'));
  Promise.resolve().then(() => console.log('5: promise inside I/O'));
  process.nextTick(() => console.log('6: nextTick inside I/O'));
});

console.log('7: synchronous');
```

```
Output:
7: synchronous          ← sync luôn chạy trước
1: timeout              ← không đảm bảo thứ tự với setImmediate ngoài I/O
2: immediate            ← hoặc có thể 2 trước 1 (race condition)

Sau khi readFile callback chạy:
6: nextTick inside I/O  ← nextTick drain trước microtask
5: promise inside I/O   ← microtask sau nextTick
4: immediate inside I/O ← setImmediate (check phase, cùng vòng loop)
3: timeout inside I/O   ← setTimeout (timers phase, vòng loop TIẾP THEO)
```

**Tại sao `setImmediate` thắng `setTimeout` khi ở trong I/O callback?**

Khi đang trong poll callback, sau khi callback xong, event loop đi tới **check phase** (setImmediate) TRƯỚC KHI quay lại timers phase. Đây là thiết kế có chủ ý — đảm bảo I/O callbacks có thể schedule work ngay sau khi I/O xong.

### Thread Pool: Điều mà hầu hết người không biết

Không phải tất cả Node.js operations đều thật sự async ở OS level:

```
Thật sự async (OS-level):
  ✅ TCP/UDP networking (epoll/kqueue/IOCP)
  ✅ Unix domain sockets
  ✅ Signals, child processes

Fake async (chạy trong thread pool):
  ❌ File I/O (fs.readFile, fs.writeFile, ...)
  ❌ DNS lookup (dns.lookup, KHÔNG phải dns.resolve)
  ❌ CPU-heavy crypto (bcrypt, pbkdf2, scrypt, randomBytes >threshold)
  ❌ zlib (compression)
```

```
Main Thread (libuv event loop)
       │
       │ fs.readFile('big.txt', callback)
       │
       ▼
  ┌─────────────┐
  │ Thread Pool │  ← 4 threads by default (UV_THREADPOOL_SIZE)
  │  Thread 1   │──► sys_read() ← blocking OS call
  │  Thread 2   │──► (available)
  │  Thread 3   │──► (available)
  │  Thread 4   │──► (available)
  └─────────────┘
       │
       │ Thread 1 done → push callback to poll queue
       │
       ▼
  Main Thread picks up callback in poll phase
```

**Production impact:** Nếu bạn có nhiều concurrent `fs.readFile` hoặc `bcrypt.hash`, default 4 threads sẽ là bottleneck.

```javascript
// Tăng thread pool size trước khi app start
process.env.UV_THREADPOOL_SIZE = '16'; // Max 128

// PHẢI set trước khi require bất kỳ module nào dùng thread pool
const bcrypt = require('bcrypt');
```

## 2. process.nextTick vs setImmediate vs Promise.resolve

### Cơ chế thật

Ba thứ này đều "defer" execution nhưng ở các điểm KHÁC NHAU trong event loop:

```
Synchronous code
       ↓
process.nextTick queue (drain hoàn toàn)
       ↓
Microtask queue / Promise.then (drain hoàn toàn)
       ↓
Event Loop Phase Transition
       ↓
process.nextTick queue (drain lại)  ← giữa mỗi phase
       ↓
Microtask queue (drain lại)
       ↓
Next phase...
```

**`process.nextTick`** không phải là một phase trong event loop. Nó là một **"nextTick queue"** được drain sau TỪNG operation, trước cả microtasks.

```javascript
// Demo thứ tự thực thi
console.log('start');

process.nextTick(() => {
  console.log('nextTick 1');
  process.nextTick(() => console.log('nextTick 2 (nested)'));
  // ← nested nextTick cũng chạy TRONG lần drain này, trước promise
});

Promise.resolve()
  .then(() => console.log('promise 1'))
  .then(() => console.log('promise 2'));

setImmediate(() => console.log('setImmediate'));
setTimeout(() => console.log('setTimeout'), 0);

console.log('end');
```

```
Output:
start
end
nextTick 1          ← nextTick queue drain (bao gồm nested)
nextTick 2 (nested) ← vẫn trong nextTick queue drain
promise 1           ← microtask queue
promise 2           ← microtask queue tiếp
setTimeout          ← timers phase (hoặc sau setImmediate tùy race)
setImmediate        ← check phase
```

### Tại sao `nextTick` tồn tại?

Lý do lịch sử: `process.nextTick` được thêm trước khi Promises tồn tại. Dùng cho pattern:

```javascript
// Pattern: Emit event sau khi constructor hoàn tất
class MyEmitter extends EventEmitter {
  constructor() {
    super();

    // BUG: emit ngay trong constructor
    // this.emit('ready'); // Caller chưa có cơ hội attach listener!

    // FIX: defer đến sau khi synchronous code hoàn tất
    process.nextTick(() => {
      this.emit('ready'); // Bây giờ caller đã attach listener rồi
    });
  }
}

const emitter = new MyEmitter();
emitter.on('ready', () => console.log('I caught this!')); // Sẽ catch được
```

### Nguy hiểm: nextTick starves event loop

```javascript
// BUG: Recursive nextTick chặn event loop mãi mãi
function recursiveNextTick() {
  process.nextTick(recursiveNextTick); // NEVER STOPS
  // I/O callbacks sẽ không bao giờ chạy
  // Timers không bao giờ fire
}

// FIX: Dùng setImmediate nếu muốn yield cho I/O
function recursiveSetImmediate() {
  setImmediate(recursiveSetImmediate); // Cho phép I/O callbacks chạy giữa
}
```

## 3. Streams

### Cơ chế thật

Streams giải quyết một vấn đề cụ thể: **Memory**

```javascript
// WITHOUT STREAMS — memory problem
fs.readFile('10gb-video.mp4', (err, data) => {
  // 'data' là Buffer chứa 10GB trong RAM
  res.end(data);
  // RAM usage: 10GB+
  // Nếu 100 requests đồng thời: 1TB RAM
});

// WITH STREAMS — memory efficient
fs.createReadStream('10gb-video.mp4').pipe(res);
// RAM usage: ~64KB (default highWaterMark) bất kể file size
// 100 requests đồng thời: ~6.4MB RAM
```

### 4 loại Stream

```
┌─────────────────────────────────────────────────────────────┐
│ Readable     ← Nguồn dữ liệu                               │
│   fs.createReadStream, http.IncomingMessage, process.stdin  │
├─────────────────────────────────────────────────────────────┤
│ Writable     ← Đích dữ liệu                                │
│   fs.createWriteStream, http.ServerResponse, process.stdout │
├─────────────────────────────────────────────────────────────┤
│ Duplex       ← Cả hai (không connected)                    │
│   net.Socket (TCP socket)                                   │
├─────────────────────────────────────────────────────────────┤
│ Transform    ← Duplex nhưng output transform từ input       │
│   zlib.createGzip, crypto.createCipheriv                    │
└─────────────────────────────────────────────────────────────┘
```

### Backpressure — Cơ chế quan trọng nhất

**Backpressure** là mechanism giúp Readable không produce data nhanh hơn Writable có thể consume:

```
Producer                    Consumer
(fs.createReadStream)       (network socket)
      │                           │
      │    chunk 1 (64KB)         │
      │──────────────────────────►│
      │                           │ writableBuffer: 64KB
      │    chunk 2 (64KB)         │
      │──────────────────────────►│
      │                           │ writableBuffer: 128KB
      │    chunk 3 (64KB)         │
      │──────────────────────────►│
      │                           │ writableBuffer: 192KB
      │                           │
      │                           │ hwm exceeded!
      │    write() returns false  │
      │◄──────────────────────────│
      │                           │
      │    PAUSE (stop reading)   │
      │                           │
      │    (consumer drains buffer)
      │                           │
      │    'drain' event          │
      │◄──────────────────────────│
      │                           │
      │    RESUME (start reading) │
      │──────────────────────────►│
```

```javascript
// Manual backpressure implementation (để hiểu mechanism)
const readable = fs.createReadStream('large-file.dat');
const writable = fs.createWriteStream('destination.dat');

readable.on('data', (chunk) => {
  // write() trả về false khi internal buffer đầy
  const canContinue = writable.write(chunk);

  if (!canContinue) {
    // Dừng đọc — tránh memory overflow
    readable.pause();

    // Khi writable drain xong, tiếp tục đọc
    writable.once('drain', () => {
      readable.resume();
    });
  }
});

readable.on('end', () => writable.end());
```

```javascript
// pipe() tự handle backpressure
// Đây là lý do nên dùng pipe thay vì manual
readable.pipe(writable);

// pipeline() (Node 10+) còn tốt hơn: proper error handling
const { pipeline } = require('stream/promises');

await pipeline(
  fs.createReadStream('input.txt'),
  zlib.createGzip(),
  fs.createWriteStream('output.txt.gz'),
);
// pipeline tự cleanup nếu có error — pipe không làm vậy
```

### Object Mode

Streams mặc định làm việc với `Buffer` hoặc `string`. **Object mode** cho phép stream bất kỳ JS value:

```javascript
const { Transform } = require('stream');

// Transform: CSV text → JS objects
const csvParser = new Transform({
  objectMode: true, // output là objects thay vì Buffers

  transform(chunk, encoding, callback) {
    const lines = chunk.toString().split('\n');

    lines.forEach((line) => {
      if (line.trim()) {
        const [name, age, city] = line.split(',');
        this.push({ name, age: parseInt(age), city }); // push object
      }
    });

    callback();
  },
});

// Readable stream → Transform (text→object) → Writable
fs.createReadStream('users.csv')
  .pipe(csvParser)
  .on('data', (user) => {
    // user là { name: 'Alice', age: 30, city: 'Hanoi' }
    db.insert(user);
  });
```

### Custom Readable Stream

```javascript
const { Readable } = require('stream');

// Pagination stream — fetch pages cho đến hết
class PaginationStream extends Readable {
  constructor(fetchPage) {
    super({ objectMode: true });
    this.fetchPage = fetchPage;
    this.page = 1;
    this.done = false;
  }

  async _read() {
    // _read() được gọi khi consumer muốn thêm data
    // Đây là "pull-based" model

    if (this.done) {
      this.push(null); // null = stream ended
      return;
    }

    try {
      const results = await this.fetchPage(this.page++);

      if (results.length === 0) {
        this.done = true;
        this.push(null);
        return;
      }

      // push từng item để consumer nhận từng cái
      results.forEach((item) => this.push(item));
    } catch (err) {
      this.destroy(err); // propagate error qua stream
    }
  }
}

// Sử dụng:
const userStream = new PaginationStream((page) =>
  db.users.findMany({ skip: (page - 1) * 100, take: 100 }),
);

for await (const user of userStream) {
  await sendEmail(user); // Xử lý từng user mà không load tất cả vào RAM
}
```

## 4. Buffer

### Cơ chế thật

`Buffer` là **fixed-size raw memory allocation** nằm NGOÀI V8 heap.

```
V8 Heap (managed by GC)         Outside V8 (raw memory)
┌─────────────────────┐         ┌──────────────────────┐
│  JavaScript Objects  │         │  Buffer data         │
│  { name: 'Alice' }  │         │  FF D8 FF E0 00 10   │
│  [1, 2, 3]          │         │  4A 46 49 46 00 01   │
│                     │         │  01 00 00 01 00 01   │
│  Buffer object ─────┼────────►│  00 00 FF FE 00 00   │
│  (just metadata)    │         │  (actual bytes here) │
└─────────────────────┘         └──────────────────────┘
```

Tại sao không trong V8 heap?

- I/O operations cần stable memory address — GC có thể move objects
- Tránh double-copy: kernel write vào raw memory, Buffer trỏ thẳng vào đó
- Không có GC overhead

### Tạo Buffer

```javascript
// 3 cách tạo Buffer

// 1. Từ data đã có (most common)
const buf1 = Buffer.from('Hello, World!', 'utf8');
// → <Buffer 48 65 6c 6c 6f 2c 20 57 6f 72 6c 64 21>

const buf2 = Buffer.from([0xff, 0xfe, 0xaa]);
// → raw bytes

const buf3 = Buffer.from(arrayBuffer, byteOffset, length);
// → share memory với ArrayBuffer (zero-copy)

// 2. Allocate mới (zeroed)
const buf4 = Buffer.alloc(16);
// → 16 bytes, tất cả là 0x00 (safe, chậm hơn)

// 3. Allocate chưa initialized (KHÔNG SAFE — chứa old memory)
const buf5 = Buffer.allocUnsafe(16);
// → 16 bytes, nội dung ngẫu nhiên (nhanh hơn, dùng khi biết mình sẽ ghi đè hết)
```

### Encoding

```javascript
const data = 'Xin chào Việt Nam';

// utf8: variable-length, 1-4 bytes per char
const utf8Buf = Buffer.from(data, 'utf8');
console.log(utf8Buf.length); // > 18 vì tiếng Việt có dấu

// utf16le: 2 bytes mỗi char (Windows native)
const utf16Buf = Buffer.from(data, 'utf16le');

// latin1/binary: 1 byte per char (giới hạn ASCII+)
// base64: decode base64 string sang raw bytes
const imgBuf = Buffer.from(base64ImageString, 'base64');

// hex: decode hex string
const keyBuf = Buffer.from('deadbeef', 'hex');
// → <Buffer de ad be ef>
```

### Buffer trong Network Programming

```javascript
// Đọc binary protocol (ví dụ: custom TCP protocol)
// Header format: [msgType:1byte][payloadLen:4bytes][payload:N bytes]

function parseMessage(buf) {
  // Đọc 1 byte đầu làm message type
  const msgType = buf.readUInt8(0);

  // Đọc 4 bytes tiếp theo (big-endian) làm payload length
  const payloadLen = buf.readUInt32BE(1);

  // Đọc payload
  const payload = buf.slice(5, 5 + payloadLen);

  return { msgType, payload: payload.toString('utf8') };
}

function createMessage(type, content) {
  const contentBuf = Buffer.from(content, 'utf8');

  // Allocate header (5 bytes) + content
  const header = Buffer.alloc(5);
  header.writeUInt8(type, 0); // 1 byte: type
  header.writeUInt32BE(contentBuf.length, 1); // 4 bytes: length

  return Buffer.concat([header, contentBuf]);
}
```

### Pitfall: Buffer không tự resize

```javascript
// BUG: Streaming TCP data — chunks có thể không align với messages
let buffer = Buffer.alloc(0);

socket.on('data', (chunk) => {
  // chunk là một phần của stream, không phải một message hoàn chỉnh
  buffer = Buffer.concat([buffer, chunk]);

  // Cần loop: một chunk có thể chứa nhiều messages
  while (buffer.length >= 5) {
    // ít nhất đủ header
    const payloadLen = buffer.readUInt32BE(1);
    const totalLen = 5 + payloadLen;

    if (buffer.length < totalLen) break; // chưa đủ data

    const message = parseMessage(buffer.slice(0, totalLen));
    processMessage(message);

    buffer = buffer.slice(totalLen); // remove processed bytes
  }
});
```

## 5. Cluster Module

### Cơ chế thật

Node.js là single-threaded. Nếu server của bạn chạy trên máy 16 core, bạn đang lãng phí 15 core.

Cluster module cho phép spawn nhiều child processes, tất cả share cùng port:

```
                    ┌──────────────────────┐
                    │    Master Process     │
                    │                      │
                    │  - Không handle HTTP │
                    │  - Manage workers    │
                    │  - Restart khi crash │
                    └──────┬───────────────┘
                           │ fork()
              ┌────────────┼────────────┐
              │            │            │
    ┌─────────▼──┐  ┌──────▼─────┐  ┌──▼──────────┐
    │  Worker 1  │  │  Worker 2  │  │  Worker 3   │
    │  (V8 + JS) │  │  (V8 + JS) │  │  (V8 + JS)  │
    │  Port 3000 │  │  Port 3000 │  │  Port 3000  │
    └────────────┘  └────────────┘  └─────────────┘
          ▲               ▲               ▲
          └───────────────┴───────────────┘
                  OS load balancing
                  (round-robin default)
```

**Cơ chế share port:** Master mở socket, pass file descriptor xuống workers qua IPC. Workers không thật sự "share" port — master distribute connections.

```javascript
const cluster = require('cluster');
const http = require('http');
const numCPUs = require('os').cpus().length;

if (cluster.isPrimary) {
  // Master process: chỉ quản lý workers
  console.log(`Master ${process.pid} started`);

  // Fork workers
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }

  // Auto-restart worker khi crash
  cluster.on('exit', (worker, code, signal) => {
    console.log(`Worker ${worker.process.pid} died (${signal || code})`);

    // Restart trừ khi exit gracefully
    if (!worker.exitedAfterDisconnect) {
      console.log('Restarting worker...');
      cluster.fork();
    }
  });

  // Zero-downtime restart pattern
  process.on('SIGUSR2', () => {
    const workers = Object.values(cluster.workers);

    function restartNext() {
      const worker = workers.shift();
      if (!worker) return;

      worker.once('exit', () => {
        const newWorker = cluster.fork();
        newWorker.once('listening', restartNext); // restart next sau khi new worker ready
      });

      worker.send('shutdown'); // graceful shutdown signal
      worker.disconnect();
      setTimeout(() => worker.kill(), 5000); // force kill sau 5s nếu không shutdown
    }

    restartNext();
  });
} else {
  // Worker process: actual HTTP server
  const server = http.createServer((req, res) => {
    res.writeHead(200);
    res.end(`Handled by worker ${process.pid}\n`);
  });

  server.listen(3000);

  // Handle graceful shutdown
  process.on('message', (msg) => {
    if (msg === 'shutdown') {
      server.close(() => process.exit(0));
    }
  });

  console.log(`Worker ${process.pid} started`);
}
```

### Cluster vs worker_threads

```
Cluster (multi-process):
  ✅ True isolation — crash trong worker không ảnh hưởng master
  ✅ Memory isolation — separate heap cho mỗi process
  ✅ Tốt cho HTTP servers (stateless request handling)
  ❌ Mỗi process duplicate toàn bộ memory (không share heap)
  ❌ Communication qua IPC (JSON serialization = overhead)
  ❌ Không share state trực tiếp

worker_threads (multi-thread trong 1 process):
  ✅ Share memory via SharedArrayBuffer
  ✅ Nhẹ hơn — không duplicate toàn bộ process
  ✅ Tốt cho CPU-intensive tasks (image processing, ML inference)
  ❌ Crash trong thread có thể crash cả process
  ❌ Cần careful với shared state
```

## 6. worker_threads

### Cơ chế thật

Trước `worker_threads`, Node.js không có true parallelism cho JS code. CPU-bound tasks block toàn bộ event loop.

`worker_threads` spawn threads thật trong cùng process, mỗi thread có V8 isolate riêng:

```
Process Memory Space
┌────────────────────────────────────────────────────────────┐
│                                                            │
│  Main Thread                    Worker Thread 1            │
│  ┌──────────────────┐           ┌──────────────────┐       │
│  │   V8 Isolate 1   │           │   V8 Isolate 2   │       │
│  │   (own heap)     │           │   (own heap)     │       │
│  │   Event Loop     │           │   Event Loop     │       │
│  └──────────────────┘           └──────────────────┘       │
│          │                               │                 │
│          └───────── SharedArrayBuffer ───┘                 │
│                   (shared raw memory)                      │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

```javascript
// main.js — CPU-intensive task với worker_threads
const {
  Worker,
  isMainThread,
  parentPort,
  workerData,
} = require('worker_threads');

if (isMainThread) {
  // Main thread: tạo worker pool

  function runWorker(data) {
    return new Promise((resolve, reject) => {
      const worker = new Worker(__filename, { workerData: data });

      worker.on('message', resolve);
      worker.on('error', reject);
      worker.on('exit', (code) => {
        if (code !== 0)
          reject(new Error(`Worker stopped with exit code ${code}`));
      });
    });
  }

  // Chạy 4 CPU tasks song song
  Promise.all([
    runWorker({ task: 'fibonacci', n: 45 }),
    runWorker({ task: 'fibonacci', n: 44 }),
    runWorker({ task: 'fibonacci', n: 43 }),
    runWorker({ task: 'fibonacci', n: 42 }),
  ]).then((results) => console.log('Results:', results));

  // Main thread event loop vẫn chạy bình thường
  setInterval(() => console.log('Main thread still responsive!'), 1000);
} else {
  // Worker thread: chạy CPU task
  function fibonacci(n) {
    if (n <= 1) return n;
    return fibonacci(n - 1) + fibonacci(n - 2);
  }

  const result = fibonacci(workerData.n);
  parentPort.postMessage(result);
}
```

### SharedArrayBuffer — Shared Memory

```javascript
// Giao tiếp zero-copy giữa main và worker
const {
  Worker,
  isMainThread,
  parentPort,
  workerData,
} = require('worker_threads');

if (isMainThread) {
  // Allocate shared memory
  const sharedBuffer = new SharedArrayBuffer(4); // 4 bytes
  const sharedArray = new Int32Array(sharedBuffer);

  sharedArray[0] = 0; // initial value: counter = 0

  const worker = new Worker(__filename, {
    workerData: { sharedBuffer },
  });

  // Main thread increment
  setTimeout(() => {
    Atomics.add(sharedArray, 0, 10); // thread-safe add
    console.log('After main add:', sharedArray[0]);
  }, 100);
} else {
  const sharedArray = new Int32Array(workerData.sharedBuffer);

  // Worker increment — không cần serialize/deserialize
  Atomics.add(sharedArray, 0, 1); // thread-safe operation

  // Notify main thread
  Atomics.notify(sharedArray, 0, 1); // wake up threads waiting on index 0

  parentPort.postMessage('done');
}
```

**Atomics là bắt buộc** khi làm việc với SharedArrayBuffer vì race conditions.

## Ứng dụng thực tế & Production Patterns

### Pattern 1: Worker Pool cho image processing

```javascript
// worker-pool.js
const { Worker } = require('worker_threads');
const os = require('os');

class WorkerPool {
  constructor(workerScript, size = os.cpus().length) {
    this.workers = [];
    this.queue = [];

    for (let i = 0; i < size; i++) {
      this.addWorker(workerScript);
    }
  }

  addWorker(script) {
    const worker = new Worker(script);
    worker.on('message', ({ id, result, error }) => {
      // Find pending task
      const taskIndex = this.queue.findIndex((t) => t.id === id);
      if (taskIndex === -1) return;

      const { resolve, reject } = this.queue.splice(taskIndex, 1)[0];
      error ? reject(error) : resolve(result);

      worker.busy = false;
    });

    worker.busy = false;
    this.workers.push(worker);
  }

  run(data) {
    return new Promise((resolve, reject) => {
      const id = Math.random().toString(36);
      const availableWorker = this.workers.find((w) => !w.busy);

      if (availableWorker) {
        availableWorker.busy = true;
        availableWorker.postMessage({ id, data });
        this.queue.push({ id, resolve, reject });
      } else {
        // Queue for later
        this.queue.push({ id, data, resolve, reject, pending: true });
      }
    });
  }
}

// Usage
const imagePool = new WorkerPool('./image-worker.js', 4);
await imagePool.run({ image: imageBuffer, resize: { w: 800, h: 600 } });
```

### Câu 1: Tại sao `setImmediate` luôn thắng `setTimeout(fn, 0)` khi nằm trong I/O callback, nhưng kết quả không đảm bảo khi nằm ngoài?

**Đáp án:**

Khi code chạy trong **I/O callback** (ví dụ `fs.readFile` callback), event loop đang ở **poll phase**. Sau khi callback xong, event loop đi:

```
poll phase (đang ở đây)
    → check phase (setImmediate)  ← gần hơn
    → close callbacks
    → timers phase (setTimeout)   ← xa hơn, vòng tiếp theo
```

`setImmediate` ở **check phase** nằm ngay SAU poll phase trong cùng vòng loop. `setTimeout` ở **timers phase** phải chờ sang vòng loop tiếp theo.

**Ngoài I/O callback** (code synchronous ban đầu), không có guarantee về timing vì:

- Event loop chưa bước vào poll phase
- `setTimeout(fn, 0)` thật ra là `setTimeout(fn, 1)` (minimum delay là 1ms)
- Nếu timer expired trước khi event loop bắt đầu → timers phase chạy trước
- Nếu chưa expired → poll phase trước → check phase (setImmediate) trước

Đây là race condition thật và **không nên code dựa vào thứ tự này**.

### Câu 2: Đoạn code sau có vấn đề gì? Tại sao và sửa như thế nào?

```javascript
const http = require('http');
const fs = require('fs');

const server = http.createServer((req, res) => {
  fs.readFile('./large-video.mp4', (err, data) => {
    if (err) {
      res.writeHead(500);
      res.end();
      return;
    }
    res.writeHead(200, { 'Content-Type': 'video/mp4' });
    res.end(data);
  });
});

server.listen(3000);
```

**Đáp án:**

**Vấn đề 1: Memory exhaustion**
`fs.readFile` load toàn bộ file vào RAM trước khi gửi. Với file 500MB và 100 concurrent requests → 50GB RAM. Server crash.

**Vấn đề 2: No backpressure**
Ngay cả khi dùng `res.write(data)` từng chunk, không có backpressure handling. Nếu client download chậm, server tiếp tục gửi data vào writable buffer → memory leak.

**Vấn đề 3: No error handling**
`res.end(data)` không có timeout. Nếu client disconnect, server vẫn giữ resource.

**Fix:**

```javascript
const server = http.createServer((req, res) => {
  const stream = fs.createReadStream('./large-video.mp4');

  // Handle stream errors (file not found, permission denied, etc.)
  stream.on('error', (err) => {
    res.writeHead(404);
    res.end();
  });

  // Handle client disconnect — cleanup stream
  req.on('close', () => stream.destroy());

  res.writeHead(200, { 'Content-Type': 'video/mp4' });

  // pipe tự handle backpressure
  stream.pipe(res);
});
```

Hoặc tốt hơn với `pipeline`:

```javascript
const { pipeline } = require('stream/promises');

const server = http.createServer(async (req, res) => {
  try {
    res.writeHead(200, { 'Content-Type': 'video/mp4' });
    await pipeline(fs.createReadStream('./large-video.mp4'), res);
  } catch (err) {
    if (!res.headersSent) {
      res.writeHead(500);
      res.end();
    }
  }
});
```

### Câu 3: Điều gì xảy ra với memory khi bạn tạo `Buffer.allocUnsafe(1024)`? Tại sao nó nhanh hơn `Buffer.alloc(1024)`, và khi nào KHÔNG được dùng?

**Đáp án:**

**`Buffer.alloc(1024)`:**

1. OS cấp phát 1024 bytes raw memory
2. libuv/Node.js **zero-fill** toàn bộ memory (ghi 0x00 vào từng byte)
3. Trả về Buffer với 1024 bytes sạch

**`Buffer.allocUnsafe(1024)`:**

1. OS cấp phát 1024 bytes raw memory
2. **KHÔNG zero-fill** — memory chứa bất cứ thứ gì đã ở đó trước (old data từ process memory, data từ request trước, etc.)
3. Trả về Buffer ngay lập tức

**Tại sao nhanh hơn:** Bỏ qua bước ghi 1024 bytes. Với buffer nhỏ không đáng kể, nhưng với buffer lớn (MB range) trong hot path → measurable difference.

**Khi KHÔNG được dùng:**

- **Bất kỳ khi nào bạn không ghi đè TOÀN BỘ bytes trước khi send cho client**
- Đây là **security vulnerability**: bạn có thể vô tình leak memory từ previous request

```javascript
// VULNERABILITY: leak old data
const buf = Buffer.allocUnsafe(1024);
res.write(buf); // Gửi 1024 bytes chứa dữ liệu từ request trước!

// SAFE: chỉ dùng khi bạn sẽ fill hết
const buf = Buffer.allocUnsafe(1024);
buf.fill(0); // explicit zero-fill
// hoặc
data.copy(buf, 0); // copy data vào toàn bộ buf

// THỰC TẾ: Node.js có threshold
// Buffer.allocUnsafe() với size < 4096 bytes thật ra lấy từ pre-allocated pool
// đã được zero-filled → vẫn safe, nhưng API vẫn không guarantee
```

**Rule of thumb:** Dùng `allocUnsafe` chỉ khi: (1) bạn sẽ ghi đè 100% bytes ngay sau đó, và (2) bạn đã measure rằng đây thật sự là bottleneck.
