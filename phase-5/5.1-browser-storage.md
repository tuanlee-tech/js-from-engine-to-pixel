## 1. Cơ chế thật

### localStorage — Không phải "database nhỏ", là synchronous I/O trên main thread

Trước khi nói cơ chế, cần hiểu `localStorage` nằm ở đâu trong kiến trúc browser.

```
┌─────────────────────────────────────────────────┐
│              Browser Process                     │
│  ┌──────────────────────────────────────────┐   │
│  │           Renderer Process               │   │
│  │  ┌─────────────────────────────────┐    │   │
│  │  │     Main Thread                 │    │   │
│  │  │  ┌──────────┐  ┌─────────────┐ │    │   │
│  │  │  │ JS Engine│  │  DOM/Layout │ │    │   │
│  │  │  │  (V8)    │  │   Engine    │ │    │   │
│  │  │  └────┬─────┘  └─────────────┘ │    │   │
│  │  │       │ localStorage.getItem() │    │   │
│  │  │       │ ← SYNCHRONOUS CALL     │    │   │
│  │  └───────┼─────────────────────────    │   │
│  │          │                             │   │
│  │  ┌───────▼─────────────────────────┐  │   │
│  │  │   SQLite (per-origin .sqlite)   │  │   │
│  │  │   /path/to/profile/             │  │   │
│  │  │   localStorage.sqlite           │  │   │
│  │  └─────────────────────────────────┘  │   │
│  └──────────────────────────────────────────┘  │
│                                                  │
│  ┌──────────────────────────────────────────┐   │
│  │  Storage Service (Browser Process)       │   │
│  │  Manages disk I/O, quota enforcement     │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

**Điều thật sự xảy ra khi bạn gọi `localStorage.setItem('key', 'value')`:**

1. JS Engine (V8) gọi qua **Blink binding layer** — một lớp C++ nối V8 với Web APIs
2. Blink gọi **Storage Service** — chạy trên Browser Process hoặc đôi khi Renderer Process
3. Storage Service ghi xuống **SQLite file** trên disk của user (`localStorage.sqlite` trong Chrome profile directory)
4. **SQLite flush** xuống disk xong mới trả về
5. JS tiếp tục chạy

Bước 3–4 là synchronous từ góc độ JS thread. **Main thread bị block** trong suốt thời gian SQLite write xảy ra.

> **Tại sao Chromium chọn SQLite?** localStorage ra đời năm 2009. SQLite là cơ sở dữ liệu nhúng đáng tin cậy nhất lúc đó, có ACID guarantees. Trade-off là synchronous I/O.

**Tại sao `sessionStorage` khác `localStorage` về vòng đời nhưng KHÔNG khác về cơ chế I/O?**

`sessionStorage` cũng synchronous. Điểm khác biệt duy nhất là **scope**:

```
localStorage:    Tồn tại vĩnh viễn trên disk, shared giữa tất cả tabs cùng origin
sessionStorage:  Tồn tại trong memory (+ optional disk), scoped per tab + per browsing context
```

`sessionStorage` thực ra lưu trong memory của Renderer Process, không nhất thiết flush xuống disk sau mỗi write. Nhưng nó vẫn synchronous về mặt API.

### In-memory cache — Lý do localStorage "nhanh" hơn bạn nghĩ

Một điều mà nhiều người không biết: **localStorage không đọc từ disk mỗi lần bạn gọi `getItem()`**.

Khi trang load, browser đọc toàn bộ localStorage của origin đó vào **in-memory cache** (một `HashMap` trong C++). Sau đó:

```
localStorage.getItem('userToken')
→ Lookup trong in-memory HashMap
→ Trả về ngay, không có disk I/O
```

```
localStorage.setItem('userToken', 'abc123')
→ Ghi vào in-memory HashMap
→ Schedule disk write (asynchronous về phía implementation)
→ Trả về ngay cho JS (nhưng JS vẫn bị block đợi bước này)
```

> **Vậy tại sao localStorage vẫn bị coi là "blocking"?**

Vì khi trang **lần đầu load**, browser phải đọc SQLite file vào memory. Nếu origin có 5MB localStorage, browser mất vài ms đọc file đó. Trong thời gian đó, nếu script cần localStorage, browser phải đợi. Đây là lý do tại sao localStorage lớn làm **tăng Time to First Byte (TTFB) cảm nhận được**.

```
Trang load → Parser gặp <script> → Script chạy
                                          ↓
                             localStorage.getItem() được gọi
                                          ↓
                        [Nếu cache chưa warm] Đọc SQLite từ disk
                                          ↓
                                   Block! Đợi I/O
```

### Vì sao localStorage chỉ lưu được string

Đây không phải giới hạn tùy tiện. `localStorage` được thiết kế theo **Web Storage spec (HTML5 2009)** với mục tiêu đơn giản: key-value store cho web apps. Spec định nghĩa storage là `DOMString → DOMString`.

Khi bạn set một non-string:

```javascript
localStorage.setItem('count', 42);
// V8 gọi toString() trên 42 → "42"
// Lưu string "42" vào SQLite

localStorage.setItem('user', { name: 'Alice' });
// V8 gọi toString() trên object → "[object Object]"
// Lưu string "[object Object]" — classic bug
```

Không có auto-serialization. Không có type metadata. Chỉ có string.

### IndexedDB — Một database thật trong browser

IndexedDB là một **NoSQL document database** chạy trong browser, được implement khác hoàn toàn so với localStorage.

```
┌─────────────────────────────────────────────────────────┐
│                    Renderer Process                      │
│                                                          │
│  ┌─────────────┐    postMessage (IPC)    ┌────────────┐ │
│  │ Main Thread │ ──────────────────────► │  IDB I/O   │ │
│  │             │                         │  Thread    │ │
│  │  JS Engine  │ ◄────────────────────── │            │ │
│  │  (V8)       │    callback/event        │            │ │
│  └─────────────┘                         └─────┬──────┘ │
│                                                │        │
└────────────────────────────────────────────────┼────────┘
                                                 │ IPC
                                                 ▼
┌────────────────────────────────────────────────────────┐
│                  Browser Process                        │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │            Storage Service                       │   │
│  │  ┌─────────────────────────────────────────┐    │   │
│  │  │  LevelDB (Chromium)                      │    │   │
│  │  │  /profile/IndexedDB/                     │    │   │
│  │  │  https_example.com_0.indexeddb.leveldb/  │    │   │
│  │  └─────────────────────────────────────────┘    │   │
│  └─────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────┘
```

**Điểm khác biệt cốt lõi với localStorage:**

1. **Storage engine khác nhau:** Chrome dùng **LevelDB** (một LSM-tree based key-value store của Google) cho IndexedDB, không phải SQLite. LevelDB được thiết kế cho write-heavy workloads với sequential I/O.

2. **I/O model khác nhau:** IndexedDB operations chạy trên **một thread riêng** (I/O thread). Main thread không block.

3. **Transaction-based:** Mọi operation phải nằm trong một transaction. Transactions có 3 modes: `readonly`, `readwrite`, `versionchange`.

4. **Structured Clone Algorithm:** IndexedDB dùng Structured Clone để serialize data — hỗ trợ được `Blob`, `ArrayBuffer`, `Map`, `Set`, `Date`, v.v. Không chỉ string.

---

### Structured Clone Algorithm — Tại sao IndexedDB lưu được object phức tạp

Đây là thứ phân biệt IndexedDB với localStorage.

**Structured Clone** là một algorithm trong HTML spec cho phép deep-copy các JS values bao gồm:

```
✅ Primitives (string, number, boolean, null, undefined, BigInt)
✅ Date
✅ RegExp
✅ Blob, File
✅ ArrayBuffer, TypedArrays (Int8Array, Float32Array...)
✅ Map, Set
✅ Object, Array (deep clone)
✅ ImageData, ImageBitmap
✅ CryptoKey

❌ Function (không clone được — throws DataCloneError)
❌ DOM nodes (không clone được)
❌ Symbol (không clone được)
❌ Object với circular reference có thể handle được tùy implementation
```

Khi bạn `store.put(data)`, browser chạy Structured Clone trên `data`, serialize nó thành binary format, rồi ghi xuống LevelDB.

---

### IndexedDB Transaction model và cursor

IndexedDB là **event-driven API** (được thiết kế trước Promise). Mọi operation là async và communicate qua `onsuccess`/`onerror` events.

```
IDBDatabase.transaction()
    └─► IDBTransaction
            └─► IDBObjectStore
                    ├─► IDBRequest (put/get/delete/clear)
                    └─► IDBIndex
                              └─► IDBCursor (for iteration)
```

**Transaction lifecycle:**

```
transaction.oncomplete  → Tất cả requests trong transaction thành công, đã commit xuống disk
transaction.onerror     → Có request fail, transaction bị rollback
transaction.onabort     → transaction.abort() được gọi, hoặc unhandled error
```

> **Tại sao transaction tự động commit?** IndexedDB transaction tự commit ngay khi không còn pending requests và không có thêm requests được add vào. Đây là nguồn gốc của một bug phổ biến.

### IndexedDB Transaction auto-commit trap

```javascript
// Bug scenario:
const tx = db.transaction('users', 'readwrite');
const store = tx.objectStore('users');

const getReq = store.get(userId);

getReq.onsuccess = function () {
  const user = getReq.result;

  // ❌ Nếu fetch() ở đây, transaction đã commit rồi!
  // Vì sau getReq.onsuccess, không có pending request nào
  // → Transaction tự commit
  // → Khi fetch xong và bạn cố gọi store.put(), transaction đã closed

  fetch('/api/extra-data').then((data) => {
    store.put({ ...user, extra: data }); // TransactionInactiveError!
  });
};
```

## 3. Ví dụ code

### localStorage — Đúng vs Sai

```javascript
// ❌ SAI: Không handle lỗi, không handle storage full
function saveUserPreferences(prefs) {
  localStorage.setItem('prefs', JSON.stringify(prefs));
}

// ✅ ĐÚNG: Wrap trong try/catch
// Tại sao: localStorage.setItem() throw QuotaExceededError khi đầy (5-10MB tùy browser)
// V8 propagate exception từ Blink binding layer lên JS — nếu không catch, unhandled exception
function saveUserPreferences(prefs) {
  try {
    const serialized = JSON.stringify(prefs);
    localStorage.setItem('prefs', serialized);
  } catch (error) {
    if (error.name === 'QuotaExceededError') {
      // Storage đầy. Xử lý: xóa data cũ, thông báo user, hoặc dùng fallback
      console.warn('localStorage full, clearing stale data');
      clearStaleLocalStorageEntries();

      // Thử lại một lần
      try {
        localStorage.setItem('prefs', JSON.stringify(prefs));
      } catch {
        // Vẫn đầy — không làm gì được, log cho monitoring
        reportStorageError(error);
      }
    }
  }
}

// ❌ SAI: Đọc localStorage trong render loop
// Mỗi iteration đọc từ in-memory cache, nhưng vẫn là sync call có overhead
function renderItems(items) {
  return items.map((item) => {
    const theme = localStorage.getItem('theme'); // ← Gọi N lần!
    return `<div class="${theme}">${item.name}</div>`;
  });
}

// ✅ ĐÚNG: Cache localStorage value trong JS variable
// Tại sao: Mặc dù localStorage đọc từ in-memory cache, mỗi getItem() vẫn phải
// đi qua Blink binding layer (C++ → JS boundary crossing) — có overhead nhỏ nhưng tích lũy
function renderItems(items) {
  const theme = localStorage.getItem('theme'); // Đọc 1 lần
  return items.map((item) => {
    return `<div class="${theme}">${item.name}</div>`;
  });
}
```

### IndexedDB — Wrapper đúng cách

```javascript
// IndexedDB API gốc rất verbose. Đây là wrapper minimal nhưng đúng cơ chế.
// Hiểu wrapper này = hiểu cách IndexedDB hoạt động

class IDBWrapper {
  constructor(dbName, version, upgradeCallback) {
    this.dbName = dbName;
    this.version = version;
    this.upgradeCallback = upgradeCallback;
    this.db = null;
  }

  async open() {
    return new Promise((resolve, reject) => {
      // IDBFactory.open() trả về IDBOpenDBRequest
      // Đây là request mở connection đến LevelDB file
      const request = indexedDB.open(this.dbName, this.version);

      // onupgradeneeded fires khi:
      // 1. Database chưa tồn tại (version: undefined → 1)
      // 2. version mới > version hiện tại
      // Đây là nơi DUY NHẤT được phép tạo/xóa object stores
      request.onupgradeneeded = (event) => {
        const db = event.target.result;
        // event.oldVersion: version trước đó (0 nếu DB mới)
        // event.newVersion: version mới

        // Chạy trong versionchange transaction — tự động committed sau khi handler xong
        this.upgradeCallback(db, event.oldVersion, event.newVersion);
      };

      request.onsuccess = (event) => {
        this.db = event.target.result;

        // Lắng nghe versionchange — xảy ra khi tab khác upgrade DB
        this.db.onversionchange = () => {
          // Nếu không đóng connection, tab khác bị block ở onblocked
          this.db.close();
          // Thông báo user reload
          window.location.reload();
        };

        resolve(this.db);
      };

      request.onerror = () => reject(request.error);

      // onblocked: fires khi tab khác đang upgrade nhưng tab này vẫn có connection
      request.onblocked = () => {
        console.warn('IDB upgrade blocked by another tab');
      };
    });
  }

  async get(storeName, key) {
    return new Promise((resolve, reject) => {
      // Mỗi operation cần một transaction
      // 'readonly' transaction: nhiều readonly transactions có thể chạy song song
      // 'readwrite' transaction: exclusive lock trên object store
      const tx = this.db.transaction(storeName, 'readonly');
      const store = tx.objectStore(storeName);

      const request = store.get(key);
      // Browser gửi IPC message đến I/O thread
      // I/O thread query LevelDB
      // Kết quả trả về qua IPC, fire onsuccess

      request.onsuccess = () => resolve(request.result);
      request.onerror = () => reject(request.error);
      // tx.oncomplete không cần thiết ở đây vì readonly
    });
  }

  async put(storeName, value) {
    return new Promise((resolve, reject) => {
      // 'readwrite' mode — acquire write lock
      // Nếu có transaction readwrite khác đang chạy trên cùng store,
      // transaction này sẽ queue lại, không fail ngay
      const tx = this.db.transaction(storeName, 'readwrite');
      const store = tx.objectStore(storeName);

      const request = store.put(value);
      // Structured Clone được chạy ở đây (trong JS thread)
      // Result được gửi qua IPC đến I/O thread để write vào LevelDB

      // ✅ Đúng: resolve khi transaction complete, không phải khi request success
      // Tại sao: request.onsuccess fires khi request được queued,
      // nhưng data chưa chắc đã được flush xuống disk
      // tx.oncomplete đảm bảo đã commit (durability guarantee)
      tx.oncomplete = () => resolve(request.result);
      tx.onerror = () => reject(tx.error);
      tx.onabort = () => reject(tx.error);
    });
  }

  // ✅ Batch write trong cùng một transaction — QUAN TRỌNG về performance
  // Mỗi transaction readwrite có overhead (IPC round trip, lock acquisition)
  // 1000 items với 1 transaction vs 1000 transactions: chênh lệch hàng trăm ms
  async putMany(storeName, items) {
    return new Promise((resolve, reject) => {
      const tx = this.db.transaction(storeName, 'readwrite');
      const store = tx.objectStore(storeName);

      // Add tất cả requests vào cùng một transaction
      // IDB engine batch chúng trong cùng một LevelDB write batch
      items.forEach((item) => store.put(item));

      tx.oncomplete = () => resolve();
      tx.onerror = () => reject(tx.error);
    });
  }
}

// Sử dụng
const db = new IDBWrapper('app-db', 1, (db, oldVersion, newVersion) => {
  // Migration pattern — switch không có break để chạy tất cả migrations cần thiết
  switch (oldVersion) {
    case 0:
      // Version 0 → 1: tạo stores ban đầu
      const userStore = db.createObjectStore('users', { keyPath: 'id' });
      // Index cho phép query theo field khác ngoài keyPath
      userStore.createIndex('by-email', 'email', { unique: true });

    case 1:
      // Version 1 → 2: thêm store mới
      // (sẽ chạy nếu upgrade từ 0→2 hoặc 1→2)
      // db.createObjectStore('posts', { keyPath: 'id' });
      break;
  }
});

await db.open();
```

### IndexedDB — Transaction auto-commit bug và fix

```javascript
// ❌ BUG: Async operation trong transaction handler
async function updateUserWithExtraData(userId) {
  const tx = db.transaction('users', 'readwrite');
  const store = tx.objectStore('users');

  const getReq = store.get(userId);

  getReq.onsuccess = async function () {
    const user = getReq.result;

    // ❌ CHẾT Ở ĐÂY: await làm main thread nhường quyền kiểm soát
    // Trong thời gian fetch() đang chạy, không có pending IDB requests
    // → IDB engine thấy transaction idle → auto-commit transaction
    // → Khi fetch xong và cố gọi store.put(), transaction đã closed
    const extra = await fetch('/api/user-extra/' + userId).then((r) =>
      r.json(),
    );

    store.put({ ...user, ...extra }); // TransactionInactiveError!
  };
}

// ✅ FIX: Fetch data TRƯỚC khi mở transaction
async function updateUserWithExtraData(userId) {
  // Fetch data trước — ngoài transaction
  const extra = await fetch('/api/user-extra/' + userId).then((r) => r.json());

  // Bây giờ mở transaction khi đã có đủ data
  return new Promise((resolve, reject) => {
    const tx = db.transaction('users', 'readwrite');
    const store = tx.objectStore('users');

    const getReq = store.get(userId);

    getReq.onsuccess = function () {
      const user = getReq.result;
      store.put({ ...user, ...extra }); // ✅ Vẫn trong cùng event loop tick
    };

    tx.oncomplete = () => resolve();
    tx.onerror = () => reject(tx.error);
  });
}

// ✅ FIX ALTERNATIVE: Dùng idb library (wrapper của Jake Archibald)
// idb wrap IndexedDB trong Promise-based API và handle transaction lifecycle đúng cách
import { openDB } from 'idb';
```

### IndexedDB — Cursor để iterate large datasets

```javascript
// Nếu có 10,000 records, getAll() load hết vào memory
// Cursor cho phép iterate từng record, tiết kiệm memory

async function processLargeDataset(storeName) {
  return new Promise((resolve, reject) => {
    const tx = db.transaction(storeName, 'readonly');
    const store = tx.objectStore(storeName);

    // openCursor() trả về cursor ở record đầu tiên
    const cursorReq = store.openCursor();

    const results = [];

    cursorReq.onsuccess = function (event) {
      const cursor = event.target.result;

      if (cursor) {
        // cursor.value là record hiện tại
        // Browser chỉ load record này vào memory, không phải toàn bộ dataset

        if (shouldInclude(cursor.value)) {
          results.push(processRecord(cursor.value));
        }

        // cursor.continue() gửi IPC request lấy record tiếp theo
        // Fires onsuccess lại với cursor ở vị trí mới
        cursor.continue();
      } else {
        // cursor === null → đã hết records
        resolve(results);
      }
    };

    cursorReq.onerror = () => reject(cursorReq.error);
  });
}

// Cursor với range — chỉ iterate records trong khoảng
async function getUsersByAgeRange(minAge, maxAge) {
  return new Promise((resolve, reject) => {
    const tx = db.transaction('users', 'readonly');
    const store = tx.objectStore('users');
    const index = store.index('by-age'); // Cần có index trên field 'age'

    // IDBKeyRange định nghĩa range để query
    const range = IDBKeyRange.bound(minAge, maxAge, false, false);
    // false, false = inclusive bounds (không exclusive)

    const results = [];
    const cursorReq = index.openCursor(range);

    cursorReq.onsuccess = function (event) {
      const cursor = event.target.result;
      if (cursor) {
        results.push(cursor.value);
        cursor.continue();
      } else {
        resolve(results);
      }
    };

    cursorReq.onerror = () => reject(cursorReq.error);
  });
}
```

## 4. Ứng dụng thực tế

### Khi nào dùng cái nào

```
localStorage/sessionStorage:
  ✅ User preferences (theme, language)
  ✅ Feature flags, A/B test assignments
  ✅ Small state cần persist (< 1KB per item)
  ✅ Cross-tab communication qua storage events
  ✅ Auth tokens (nhưng xem xét HttpOnly cookie cho security)

  ❌ Large data (ảnh, documents)
  ❌ Structured queries
  ❌ Offline data sync
  ❌ Binary data

IndexedDB:
  ✅ Offline-first apps (PWA)
  ✅ Large datasets (email client cache, document editor)
  ✅ Binary data (images, files)
  ✅ Structured queries với indexes
  ✅ Client-side database cho web apps

  ❌ Small, simple key-value (overkill)
  ❌ Khi cần sync ngay lập tức (vẫn async)
```

## 5. Câu hỏi kiểm tra

**Q1: Tại sao gọi `localStorage.setItem()` 1000 lần trong vòng lặp KHÔNG nhất thiết gây ra 1000 disk writes?**

A: Vì localStorage có **in-memory cache layer**. Khi `setItem()` được gọi, data được ghi vào in-memory HashMap ngay lập tức và trả về cho JS. Disk write được **schedule** bởi browser's storage service — implementation có thể **batch** nhiều writes thành một disk flush. Điều JS thread thấy là 1000 synchronous calls, nhưng thực tế số disk I/O operations ít hơn nhiều. Tuy nhiên, mỗi `setItem()` vẫn là **synchronous boundary crossing** qua Blink C++ binding layer, có overhead không nhỏ khi lặp 1000 lần.

---

**Q2: Điều gì xảy ra khi hai tabs cùng origin mở cùng lúc và cả hai đều gọi `localStorage.setItem('counter', value + 1)` để implement distributed counter?**

A: **Race condition** — classic lost update problem.

```
Tab A: đọc counter = 5
Tab B: đọc counter = 5
Tab A: ghi counter = 6
Tab B: ghi counter = 6 (ghi đè — mất 1 increment!)
```

localStorage không có transactions hay atomic compare-and-swap. Không có cách fix race condition này với localStorage thuần túy. Giải pháp đúng: dùng **BroadcastChannel API** để coordinate giữa tabs, hoặc dùng **SharedWorker** làm single source of truth, hoặc dùng server-side counter.

---

**Q3: Đoạn code này có vấn đề gì?**

```javascript
async function saveAndNavigate(data) {
  const tx = db.transaction('drafts', 'readwrite');
  const store = tx.objectStore('drafts');
  store.put(data);

  // Navigate đến trang khác
  window.location.href = '/success';
}
```

A: **Có thể mất data.** `store.put(data)` là async — nó chỉ tạo một IDBRequest, không đợi write hoàn thành. Khi `window.location.href` thay đổi ngay sau đó, browser bắt đầu navigate → **destroy Renderer Process hiện tại** → transaction bị abort trước khi có cơ hội commit. Data không được ghi xuống LevelDB.

Fix:

```javascript
async function saveAndNavigate(data) {
  await new Promise((resolve, reject) => {
    const tx = db.transaction('drafts', 'readwrite');
    const store = tx.objectStore('drafts');
    store.put(data);
    tx.oncomplete = () => resolve(); // Đợi transaction commit xong
    tx.onerror = () => reject(tx.error);
  });

  // Chỉ navigate sau khi đã commit
  window.location.href = '/success';
}
```

**Q4: Tại sao IndexedDB dùng LevelDB thay vì SQLite (như localStorage)?**

A: **Use case khác nhau về I/O pattern:**

- **SQLite** được tối ưu cho **random read/write** với complex queries (SQL JOINs, WHERE clauses). Write pattern là B-tree updates — nhiều random I/O trên disk.

- **LevelDB** là **LSM-tree (Log-Structured Merge-tree)** được tối ưu cho **write-heavy workloads**. Writes đi vào MemTable (in-memory) trước, sau đó batch flush xuống SSTable files theo sequential I/O — nhanh hơn SQLite cho workloads nhiều writes nhỏ.

IndexedDB designed cho web apps cần store nhiều small records thường xuyên (offline sync, caching), nên LevelDB phù hợp hơn. localStorage ít writes hơn (chỉ user preferences) nên SQLite với ACID guarantees mạnh hơn là lựa chọn tốt.

Ngoài ra, Firefox dùng SQLite cho IndexedDB (implementation khác nhau!), còn Safari dùng SQLite-based backend riêng. Đây là lý do đôi khi performance IndexedDB khác nhau giữa browsers.

---

**Q5: Tại sao `sessionStorage` không bị share giữa các tabs dù cùng origin, trong khi `localStorage` thì share?**

A: Đây là thiết kế cố ý theo **HTML spec**. `sessionStorage` được scoped theo **browsing context** — mỗi tab là một browsing context riêng. Lý do:

1. **Security:** Form data tạm thời, wizard state, sensitive temporary data không nên leak sang tab khác dù cùng origin.

2. **Isolation:** Mỗi tab đang làm một "session" riêng biệt với app. Ví dụ: hai tabs mở hai orders khác nhau — sessionStorage cho phép chúng hoàn toàn độc lập.

Có một edge case: khi user **Ctrl+Click** một link (mở tab mới từ tab hiện tại), browser **copy sessionStorage** từ tab cha sang tab con lúc mở. Nhưng sau đó chúng độc lập — thay đổi ở tab con không ảnh hưởng tab cha. `storage` event cũng không fire giữa chúng.

## Câu hỏi ôn tập (NotebookLM)

### Câu 1: Tại sao `localStorage.setItem()` có thể làm chậm trang web ngay cả khi data được cache trong memory?

**Đáp án:**

`localStorage` sử dụng in-memory cache (một HashMap trong C++), vì vậy phần lớn thời gian `setItem()` không cần đọc disk trực tiếp. Tuy nhiên, nó vẫn gây chậm trang web qua ba cơ chế:

**1. Cold start — đọc SQLite khi trang load lần đầu:** Khi browser load trang, nó phải đọc toàn bộ localStorage của origin từ SQLite file vào memory. Nếu origin có 5MB localStorage, quá trình này chiếm vài ms và block main thread. Scripts cần localStorage trong giai đoạn này phải chờ.

**2. Synchronous Blink binding layer crossing:** Mỗi lần gọi `setItem()`, execution phải cross từ V8 (JavaScript engine) sang Blink C++ layer. Đây là boundary crossing có chi phí, dù nhỏ. Nếu gọi trong vòng lặp 1000 lần, chi phí tích lũy đủ để đo được.

**3. Main thread blocking:** Toàn bộ API là synchronous — không thể "không block" main thread. Trong khi localStorage operation đang chạy, không có JS nào khác có thể execute, không có rendering nào xảy ra. Trên main thread mà có nhiều localStorage operations, frame budget 16.67ms có thể bị ăn vào.

**Hệ quả thực tế:** Không nên lưu data lớn vào localStorage (> vài KB), không nên đọc trong render loops, và không nên dùng thay cho IndexedDB khi cần lưu data phức tạp.

### Câu 2: Giải thích cơ chế auto-commit của IndexedDB transaction và tại sao nó gây bug khi dùng với async/await.

**Đáp án:**

**Cơ chế auto-commit:**

IndexedDB transaction tự động commit khi hai điều kiện đồng thời xảy ra:

1. Tất cả requests trong transaction đã hoàn thành (không còn pending requests)
2. Không có request mới nào được thêm vào transaction trong cùng event loop tick

Browser theo dõi một "pending request counter" cho mỗi transaction. Khi counter về 0 và JavaScript trả quyền kiểm soát về event loop, browser coi transaction là "idle" và commit nó.

**Tại sao async/await gây bug:**

```javascript
const tx = db.transaction('store', 'readwrite');
const store = tx.objectStore('store');
const req = store.get(key);

req.onsuccess = async function () {
  // Tại thời điểm này, req đã complete
  // Pending request counter = 0

  const data = req.result;

  // 'await' tạo ra một Promise và nhường quyền kiểm soát về event loop
  // → Browser thấy: transaction idle, không có pending requests
  // → Transaction auto-commits ngay đây!
  const extra = await fetch('/api/data');

  // Khi fetch resolve và code tiếp tục, transaction đã closed từ lâu
  store.put({ ...data, extra }); // TransactionInactiveError
};
```

`await` không đơn giản là "chờ" — nó tạo một microtask/macrotask và **trao lại quyền kiểm soát cho event loop**. Trong khoảng thời gian event loop chạy, browser có cơ hội kiểm tra và commit transaction.

**Fix đúng:** Tất cả data cần thiết phải được fetch trước khi mở transaction, hoặc phải đảm bảo không bao giờ có `await` trong transaction handler mà lại cần tiếp tục sử dụng transaction sau đó.

### Câu 3: Khi nào nên chọn IndexedDB thay vì localStorage, và sự khác biệt về storage engine (LevelDB vs SQLite) ảnh hưởng như thế nào đến quyết định này?

**Đáp án:**

**Chọn IndexedDB khi:**

- Data lớn hơn vài KB (localStorage giới hạn 5-10MB toàn origin)
- Cần lưu binary data (ArrayBuffer, Blob, File)
- Cần structured queries (tìm kiếm theo index, range queries)
- Offline-first PWA cần sync data lớn
- Cần không block main thread (IndexedDB hoàn toàn async)

**Chọn localStorage khi:**

- Data nhỏ, đơn giản (< 1KB total)
- Cần synchronous access (init code cần value ngay)
- Cross-tab sync qua storage events
- User preferences, theme, settings

**Ảnh hưởng của storage engine:**

Chrome dùng **LevelDB** (LSM-tree) cho IndexedDB:

- Write operations đi vào MemTable in-memory → sau đó batch flush xuống SSTable files theo sequential write
- Tối ưu cho write-heavy workloads: lưu nhiều small records nhanh (offline sync, caching)
- Read có thể chậm hơn SQLite với complex queries vì LSM-tree cần merge nhiều levels

Chrome dùng **SQLite** cho localStorage:

- B-tree structure tối ưu cho read-heavy workloads với random access
- ACID guarantees mạnh hơn, WAL (Write-Ahead Logging)
- Phù hợp với few writes, many reads — pattern của user preferences

Firefox và Safari implement IndexedDB bằng SQLite-based backend — đây là lý do IndexedDB performance đôi khi khác nhau đáng kể giữa browsers khi workload nặng về write.

**Kết luận thực tế:** Khi đánh giá storage solution, không chỉ nhìn vào API capability — cần hiểu storage engine phía sau ảnh hưởng đến performance profile của ứng dụng cụ thể. Một web app với nhiều small writes (chat messages, real-time sync) sẽ thấy IndexedDB trên Chrome (LevelDB) nhanh hơn đáng kể so với pattern đọc nhiều/ghi ít.
