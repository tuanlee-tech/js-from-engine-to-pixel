## 1. Cơ chế thật

### Cookies — Không phải "storage", là HTTP mechanism

Cookies thường bị nhầm thành một loại storage giống localStorage. Sự thật: **cookie là HTTP header mechanism**, browser chỉ là nơi lưu trữ. Hiểu điều này thay đổi toàn bộ cách bạn nhìn về security và behavior của cookie.

**Luồng hoàn chỉnh:**

```
1. Browser gửi request đến server
2. Server trả về response với header:
   Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=86400
3. Browser parse header, lưu cookie vào Cookie Jar (SQLite file trên disk)
4. Mọi request tiếp theo đến cùng origin, browser tự động đính kèm:
   Cookie: sessionId=abc123
5. JS không bao giờ thấy sessionId nếu HttpOnly được set
```

**Cookie Jar nằm ở đâu trong browser architecture:**

```
┌──────────────────────────────────────────────────────────────┐
│                    Browser Process                            │
│                                                               │
│  ┌───────────────────────────────────────────────────────┐   │
│  │              Network Service                           │   │
│  │                                                        │   │
│  │  ┌──────────────────────────────────────────────┐    │   │
│  │  │  Cookie Manager                               │    │   │
│  │  │  (Reads/writes SQLite cookie store)           │    │   │
│  │  │                                               │    │   │
│  │  │  Chrome path: Profile/Cookies (SQLite file)  │    │   │
│  │  └──────────────┬───────────────────────────────┘    │   │
│  │                 │ Intercept mọi HTTP request           │   │
│  │                 │ Attach Cookie header                 │   │
│  │                 │ Parse Set-Cookie header              │   │
│  └─────────────────┼──────────────────────────────────── ┘   │
│                    │                                          │
└────────────────────┼──────────────────────────────────────── ┘
                     │
         ┌───────────┼──────────────┐
         │           │              │
    Renderer    Renderer      Renderer
    Process     Process       Process
    (Tab 1)     (Tab 2)       (Tab 3)
```

**Điểm quan trọng:** Cookie Manager nằm trong **Network Service của Browser Process**, không phải Renderer Process. Điều này có nghĩa là:

1. `HttpOnly` cookies được đọc/ghi bởi Network Service — Renderer Process (nơi JS chạy) không bao giờ thấy chúng
2. Ngay cả XSS attack cũng không thể đọc `HttpOnly` cookie vì JS bị cô lập trong Renderer Process
3. Cookie được tự động attached vào mọi request, kể cả request từ `<img>`, `<script>`, `<iframe>` — đây là nền tảng của CSRF attack

### Cookie Attributes — Cơ chế từng attribute

#### `HttpOnly` — Cô lập khỏi JavaScript

```
Không có HttpOnly:
┌─────────────────────┐     ┌──────────────┐
│  Renderer Process   │     │  Cookie Jar   │
│  (JS chạy ở đây)    │────►│  sessionId=x  │
│  document.cookie    │◄────│              │
└─────────────────────┘     └──────────────┘
→ JS đọc được sessionId, XSS đọc được sessionId

Có HttpOnly:
┌─────────────────────┐     ┌──────────────┐
│  Renderer Process   │  ✗  │  Cookie Jar   │
│  document.cookie    │─────│  sessionId=x  │
│  (blocked)          │     │  [HttpOnly]   │
└─────────────────────┘     └──────┬───────┘
                                   │ Chỉ Network Service đọc được
                                   ▼
                          HTTP Request Headers
                          Cookie: sessionId=x
```

`document.cookie` là một getter/setter được implement trong Blink. Khi JS gọi `document.cookie`, Blink check xem cookie có flag `HttpOnly` không — nếu có, **filter ra khỏi kết quả**. Cookie vẫn tồn tại trong Cookie Jar và vẫn được gửi qua HTTP, nhưng JS không thấy.

#### `Secure` — Chỉ gửi qua HTTPS

```
Request đến http://example.com/api
→ Network Service check: cookie có Secure flag?
→ Có → Không đính kèm vào request
→ Không → Đính kèm bình thường

Request đến https://example.com/api
→ Network Service check: cookie có Secure flag?
→ Có → Đính kèm
```

Đây là enforcement ở tầng Network Service, không phải JS. JS không thể override.

#### `SameSite` — Chống CSRF, kiểm soát cross-site sending

Đây là attribute phức tạp nhất và thường hiểu sai nhất.

**Định nghĩa "same-site" vs "same-origin":**

```
Same-Origin: scheme + host + port phải khớp CHÍNH XÁC
  https://app.example.com:443  vs  https://app.example.com:443  → same-origin ✅
  https://app.example.com      vs  https://api.example.com      → cross-origin ❌

Same-Site: registrable domain (eTLD+1) phải khớp, scheme có thể khác nhau*
  https://app.example.com  vs  https://api.example.com  → same-site ✅
  https://example.com      vs  https://evil.com         → cross-site ❌
  https://example.com      vs  http://example.com       → cross-site ❌ (scheme khác)
  * "Schemeful same-site" là default từ Chrome 86
```

**Ba mode của SameSite:**

```
SameSite=Strict:
  Cookie CHỈ được gửi khi request xuất phát từ cùng site

  User đang ở evil.com, click link → bank.com:
  ← Cookie của bank.com KHÔNG được gửi
  User phải navigate trực tiếp vào bank.com để có cookie

  Use case: Extremely sensitive actions (banking, admin)

  Nhược điểm: User click email link → bank.com
  → Không có cookie → Phải login lại
  → UX rất tệ

SameSite=Lax (DEFAULT từ Chrome 80, 2020):
  Cookie được gửi cho top-level navigation với safe methods (GET)
  Cookie KHÔNG được gửi cho subrequests (fetch, XHR, img src, iframe)

  Cross-site requests và Lax:
  ┌────────────────────────────────────────────────────────────┐
  │ Request Type          │ Cookie sent? │ Lý do               │
  │───────────────────────│──────────────│─────────────────────│
  │ <a href="..."> click  │ YES          │ Top-level nav, GET  │
  │ <form method="GET">   │ YES          │ Top-level nav, GET  │
  │ <form method="POST">  │ NO           │ POST không safe     │
  │ fetch() cross-site    │ NO           │ Subrequest          │
  │ <img src="...">       │ NO           │ Subrequest          │
  │ <iframe src="...">    │ NO           │ Subrequest          │
  │ XMLHttpRequest        │ NO           │ Subrequest          │
  └────────────────────────────────────────────────────────────┘

SameSite=None (+ Secure required):
  Cookie được gửi trong mọi context, kể cả cross-site
  Dùng cho: third-party widgets, embedded payment forms, analytics
  PHẢI kết hợp với Secure, nếu không browser reject
```

**Tại sao Lax ngăn CSRF nhưng không ngăn hoàn toàn:**

```
CSRF attack cổ điển (bị block bởi Lax):
evil.com có form ẩn POST đến bank.com/transfer
→ Lax không gửi cookie cho cross-site POST
→ CSRF thất bại ✅

CSRF còn sót với Lax:
evil.com redirect user đến bank.com/action?amount=1000 (GET)
→ Lax GỬI cookie cho top-level GET navigation
→ Nếu bank.com có state-changing GET endpoint, vẫn có thể bị CSRF ❌
→ Đây là lý do state-changing operations PHẢI dùng POST/PUT/DELETE
```

#### `Expires` / `Max-Age` — Session vs Persistent cookie

```
Không có Expires/Max-Age → Session cookie:
  Chỉ tồn tại trong memory của browser session
  Bị xóa khi browser đóng (không phải tab, là toàn bộ browser)
  Không bao giờ được ghi xuống SQLite? SAI! Modern browsers có session restore
  → Browser ghi xuống disk để restore session, nhưng intent là "session only"

Expires=<date> → Persistent cookie:
  Được ghi xuống SQLite với expiry timestamp
  Network Service check expiry trước khi attach vào request
  Tự động bị xóa sau khi hết hạn

Max-Age=<seconds> → Relative expiry (ưu tiên hơn Expires nếu cả hai có mặt):
  Max-Age=0 hoặc Max-Age=-1 → Xóa cookie ngay lập tức (cách delete cookie)
  Max-Age=86400 → Hết hạn sau 24 giờ
```

### Cache API — Request/Response store cho Service Workers

Cache API khác hoàn toàn với localStorage và IndexedDB. Nó không lưu arbitrary data — nó lưu **HTTP Request/Response pairs**, được thiết kế cho offline-first web apps.

**Storage engine:**

```
Chrome implementation của Cache API:
  Không dùng LevelDB (như IndexedDB)
  Không dùng SQLite (như localStorage/cookies)
  Dùng: disk_cache format (Simple Backend)

  Mỗi cache entry:
  ├── Metadata file: Request headers, Response headers, timestamps
  └── Data file: Response body (binary)

  Path: Profile/Service Worker/CacheStorage/<origin hash>/
```

**Quan hệ với Service Worker:**

Cache API có thể được dùng từ window context, nhưng **power thật sự** chỉ thể hiện trong Service Worker, vì Service Worker có thể intercept network requests.

```
Không có Service Worker:
Browser → Network → Server → Response

Có Service Worker + Cache API:
Browser → Service Worker (intercept) → Cache API (hit?) → YES → Trả về ngay
                                                         → NO  → Network → Server
                                                                          → Cache API (store)
                                                                          → Trả về response
```

**Tại sao Cache API khác HTTP cache của browser:**

```
Browser HTTP Cache:
  - Browser tự quản lý dựa trên HTTP headers (Cache-Control, ETag, v.v.)
  - Developer không control được khi nào invalidate
  - Không thể dùng khi offline hoàn toàn (vì logic cache nằm trong browser)

Cache API:
  - Developer control hoàn toàn: khi nào cache, khi nào invalidate, version nào
  - Hoạt động offline (Service Worker intercept request trước khi đến network)
  - Có thể cache response không có Cache-Control header
  - Có thể cache cross-origin responses (với mode: 'cors')
```

### Storage Quota — Shared budget và eviction

Đây là thứ nhiều người không biết cho đến khi gặp bug production.

**Quota architecture trong Chrome:**

```
Disk Space Available: 100GB

Origin quota (per origin):
  Default: min(60% of available disk, some cap)
  → Nếu disk có 100GB free → origin có thể dùng tối đa ~60GB
  → Nhưng thực tế browser có minimum floor (~6MB) và các giới hạn khác

Storage shared giữa các APIs trong cùng origin:
┌─────────────────────────────────────────────┐
│  Origin: https://example.com                │
│  Total Quota: 500MB                         │
│  ┌──────────┬────────────┬────────────────┐ │
│  │IndexedDB │ Cache API  │ OPFS           │ │
│  │  50MB    │   400MB    │  50MB          │ │
│  └──────────┴────────────┴────────────────┘ │
│  localStorage: ~5-10MB (separate limit)     │
│  sessionStorage: memory, không count quota  │
│  Cookies: separate limit (~4KB/cookie)      │
└─────────────────────────────────────────────┘
```

**Eviction policy:**

```
Best-Effort storage (default):
  Browser CÓ THỂ xóa data khi:
  1. Disk space thấp (OS-level pressure)
  2. Browser storage pressure
  3. User clear site data
  4. Inactive origins (Chrome có thể evict origins không dùng lâu)

  Eviction order: LRU (Least Recently Used) theo origin
  Toàn bộ origin bị evict cùng lúc — không phải từng item

Persistent storage:
  navigator.storage.persist() → Request permission
  Nếu granted: data KHÔNG bị evict tự động (chỉ user xóa thủ công)

  Browser tự động grant persistent storage nếu:
  - User đã bookmark site
  - User đã install PWA
  - Site có high engagement score
  - User đã grant notification permission (Chrome)
```

### OPFS — File System trong browser, không cần permission

OPFS (Origin Private File System) là API mới nhất và ít được biết đến nhất, nhưng là quan trọng nhất cho **high-performance client-side data processing**.

**Tại sao OPFS tồn tại:**

```
Problem: SQLite compiled to WASM cần đọc/ghi file

  Attempt 1: Dùng IndexedDB làm backend
  → Slow: mỗi read/write là async IPC round-trip
  → SQLite cần synchronous I/O (blocking reads)
  → Emulate sync trên async API = overhead lớn

  Attempt 2: Dùng localStorage
  → String-only, không phải binary file I/O
  → Size limit quá nhỏ

  Solution: OPFS với synchronous access handle
  → Worker thread có thể đọc/ghi file SYNCHRONOUSLY
  → Không cần IPC round-trip cho mỗi operation
  → Hoàn hảo cho SQLite file I/O
```

**Hai modes của OPFS:**

```
Mode 1: Async API (từ main thread hoặc worker)
  → Giống File System Access API về API shape
  → Mỗi operation là Promise
  → Không có permission prompt (private to origin)

Mode 2: Sync API với FileSystemSyncAccessHandle (CHỈ trong Worker)
  → Synchronous reads/writes
  → Extreme performance — gần với native file I/O
  → Không thể dùng từ main thread (sẽ block)
```

**Tại sao Sync mode chỉ có trong Worker:**

```
Main Thread:
  Chạy JS + DOM rendering + event handling
  Không được phép block (sẽ freeze UI)
  → Sync file I/O ở đây = unacceptable

Dedicated Worker:
  Thread riêng, không có DOM access
  ĐƯỢC PHÉP block thread của mình
  Main thread vẫn responsive
  → Sync file I/O hoàn toàn OK
```

## 3. Ví dụ code

### Cookies — đọc/ghi đúng cách từ JS

```javascript
// document.cookie là một "magic string" API — thiết kế rất kỳ lạ
// Đọc: trả về tất cả cookies dưới dạng "key=value; key2=value2"
// Ghi: SET một cookie (không phải replace tất cả!)

// ❌ Sai: Nhiều người nghĩ đây là ghi lại toàn bộ cookie string
document.cookie = 'theme=dark; username=alice';
// Thực tế: Chỉ set MỘT cookie với key="theme=dark; username=alice"
// Đây là một bug cổ điển

// ✅ Đúng: Mỗi lần set một cookie
document.cookie = 'theme=dark; path=/; max-age=2592000'; // 30 days
document.cookie = 'username=alice; path=/; max-age=2592000';

// Đọc cookie theo key — phải parse thủ công hoặc dùng helper
function getCookie(name) {
  // document.cookie: "theme=dark; username=alice; sessionId=[filtered if HttpOnly]"
  // HttpOnly cookies không xuất hiện ở đây dù chúng vẫn được gửi qua HTTP

  const cookies = document.cookie.split('; ');
  const cookie = cookies.find((c) => c.startsWith(name + '='));
  return cookie ? decodeURIComponent(cookie.split('=')[1]) : null;
}

// Delete cookie: set Max-Age=0 hoặc Expires trong quá khứ
// Phải specify CÙNG path và domain khi set lúc đầu, nếu không sẽ không delete được
function deleteCookie(name, path = '/') {
  // Đặt Max-Age=0 → Network Service xóa cookie khỏi Cookie Jar
  document.cookie = `${name}=; path=${path}; max-age=0`;
}

// ✅ Modern: CookieStore API (Chrome 87+) — async, nicer API
// Vẫn chỉ access được non-HttpOnly cookies
async function modernCookieUsage() {
  // Ghi
  await cookieStore.set({
    name: 'theme',
    value: 'dark',
    maxAge: 2592000,
    sameSite: 'lax',
  });

  // Đọc (trả về object, không phải string khó parse)
  const cookie = await cookieStore.get('theme');
  console.log(cookie?.value); // 'dark'

  // Xóa
  await cookieStore.delete('theme');

  // Watch for changes (useful trong Service Worker)
  cookieStore.addEventListener('change', (event) => {
    console.log('Cookies changed:', event.changed, event.deleted);
  });
}
```

### Cookies — Security implementation đúng

```javascript
// Server-side (Node.js Express) — set cookie đúng cho auth
app.post('/login', async (req, res) => {
  const { username, password } = req.body;
  const user = await verifyCredentials(username, password);

  if (!user) {
    return res.status(401).json({ error: 'Invalid credentials' });
  }

  const sessionId = generateSecureToken(); // crypto.randomBytes(32).toString('hex')
  await saveSession(sessionId, user.id);

  res.cookie('sessionId', sessionId, {
    httpOnly: true, // JS không đọc được → XSS không steal được
    secure: true, // Chỉ HTTPS
    sameSite: 'lax', // Ngăn CSRF attack qua POST
    maxAge: 7 * 24 * 60 * 60 * 1000, // 7 days in ms
    path: '/',
    // domain: '.example.com' // Nếu muốn share giữa subdomains
  });

  // Không trả sessionId trong response body — chỉ trong cookie!
  res.json({ success: true, user: { id: user.id, name: user.name } });
});

// CSRF protection pattern:
// Cookie sessionId (HttpOnly) = authenticate identity
// CSRF token (trong form/header) = verify request intentionality
app.post('/transfer', csrfProtection, async (req, res) => {
  // csrfProtection middleware verify X-CSRF-Token header
  // Kết hợp HttpOnly sessionId + CSRF token = double protection
});
```

### Cache API — Service Worker với multiple strategies

```javascript
// service-worker.js
const CACHE_VERSION = 'v2';
const STATIC_CACHE = `static-${CACHE_VERSION}`;
const DYNAMIC_CACHE = `dynamic-${CACHE_VERSION}`;

// Install event: pre-cache static assets
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches
      .open(STATIC_CACHE)
      .then((cache) => {
        // cache.addAll(): fetch + cache nhiều URLs
        // Nếu BẤT KỲ URL nào fail → toàn bộ installation fail
        return cache.addAll(['/', '/app.js', '/app.css', '/offline.html']);
      })
      .then(() => {
        // skipWaiting: activate SW ngay, không đợi old SW deactivate
        // Cẩn thận: có thể gây issue nếu old client + new SW có incompatible caches
        return self.skipWaiting();
      }),
  );
});

// Activate event: clean up old caches
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches
      .keys()
      .then((cacheNames) => {
        return Promise.all(
          cacheNames
            .filter((name) => name !== STATIC_CACHE && name !== DYNAMIC_CACHE)
            .map((name) => {
              console.log('Deleting old cache:', name);
              return caches.delete(name);
            }),
        );
      })
      .then(() => {
        // clients.claim(): SW kiểm soát ngay các tabs đang open
        // Không claim: chỉ control tabs mở SAU khi SW activate
        return self.clients.claim();
      }),
  );
});

// Fetch event: intercept network requests
self.addEventListener('fetch', (event) => {
  const { request } = event;
  const url = new URL(request.url);

  // Không cache cross-origin requests không phải API của mình
  if (
    url.origin !== location.origin &&
    !url.hostname.includes('cdn.example.com')
  ) {
    return; // Let browser handle normally
  }

  // Static assets → Cache First
  if (isStaticAsset(url.pathname)) {
    event.respondWith(cacheFirst(request));
    return;
  }

  // API calls → Network First
  if (url.pathname.startsWith('/api/')) {
    event.respondWith(networkFirst(request));
    return;
  }

  // HTML pages → Stale While Revalidate
  if (request.headers.get('Accept')?.includes('text/html')) {
    event.respondWith(staleWhileRevalidate(request));
    return;
  }
});

// Strategy implementations
async function cacheFirst(request) {
  const cached = await caches.match(request);
  if (cached) return cached;

  try {
    const response = await fetch(request);
    // Clone vì response stream chỉ đọc được một lần
    // Một bản để cache, một bản để trả về browser
    const cache = await caches.open(STATIC_CACHE);
    cache.put(request, response.clone());
    return response;
  } catch {
    // Offline và không có cache → trả offline page
    return caches.match('/offline.html');
  }
}

async function networkFirst(request) {
  try {
    const response = await fetch(request);

    // Chỉ cache successful responses
    if (response.ok) {
      const cache = await caches.open(DYNAMIC_CACHE);
      cache.put(request, response.clone());
    }

    return response;
  } catch {
    // Network fail → fallback to cache
    const cached = await caches.match(request);
    if (cached) return cached;

    // API không có cache → trả error response
    return new Response(JSON.stringify({ error: 'Offline', cached: false }), {
      status: 503,
      headers: { 'Content-Type': 'application/json' },
    });
  }
}

async function staleWhileRevalidate(request) {
  const cache = await caches.open(DYNAMIC_CACHE);
  const cached = await cache.match(request);

  // Fetch in background dù có cache hay không
  const fetchPromise = fetch(request)
    .then((response) => {
      if (response.ok) {
        cache.put(request, response.clone());
      }
      return response;
    })
    .catch(() => null); // Ignore network errors khi đã có cache

  // Trả cached ngay nếu có, không đợi fetch
  return cached || fetchPromise;
}

function isStaticAsset(pathname) {
  return /\.(js|css|png|jpg|webp|woff2|svg)$/.test(pathname);
}
```

```javascript
// Prune cache khi sắp đầy
async function pruneOldCacheEntries() {
  const cacheNames = await caches.keys();

  for (const cacheName of cacheNames) {
    const cache = await caches.open(cacheName);
    const requests = await cache.keys();

    // Xóa entries cũ hơn 7 ngày
    const oneWeekAgo = Date.now() - 7 * 24 * 60 * 60 * 1000;

    for (const request of requests) {
      const response = await cache.match(request);
      const dateHeader = response?.headers.get('date');

      if (dateHeader) {
        const cacheDate = new Date(dateHeader).getTime();
        if (cacheDate < oneWeekAgo) {
          await cache.delete(request);
        }
      }
    }
  }
}

  const k = 1024;
  const sizes = ['Bytes', 'KB', 'MB', 'GB'];

  return parseFloat((bytes / Math.pow(k, i)).toFixed(2)) + ' ' + sizes[i];
```

## 5. Câu hỏi kiểm tra

**Q1: Tại sao `HttpOnly` cookie không thể bị đánh cắp bởi XSS, nhưng CSRF attack vẫn có thể lợi dụng `HttpOnly` cookie?**

A: Đây là câu hỏi về hai attack vectors khác nhau hoàn toàn.

**XSS (Cross-Site Scripting):** Attacker inject script vào page, script chạy trong Renderer Process và cố đọc cookie qua `document.cookie`. `HttpOnly` block điều này vì Blink filter cookies có flag `HttpOnly` ra khỏi kết quả của `document.cookie`. JS không thể đọc → không thể gửi về server của attacker.

**CSRF (Cross-Site Request Forgery):** Attacker KHÔNG cần đọc cookie. Attacker làm browser của victim gửi request đến bank.com. Network Service của browser tự động đính kèm cookie (kể cả `HttpOnly` cookie) vào mọi request đến bank.com — đây là cách cookies được thiết kế để hoạt động. `HttpOnly` chỉ ngăn JS đọc, không ngăn browser tự động gửi trong HTTP requests.

→ `HttpOnly` ngăn theft của cookie value, nhưng không ngăn việc browser tự động dùng cookie khi bị trick gửi request. Đó là lý do cần `SameSite` để chống CSRF.

**Q2: Đoạn code Service Worker này có vấn đề gì?**

```javascript
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then((response) => {
      if (response) return response;
      return fetch(event.request).then((networkResponse) => {
        caches.open('dynamic').then((cache) => {
          cache.put(event.request, networkResponse);
        });
        return networkResponse;
      });
    }),
  );
});
```

A: Có hai vấn đề:

**Vấn đề 1 — Response stream bị consumed:** `cache.put(event.request, networkResponse)` và `return networkResponse` đều cố dùng cùng một `Response` object. Response body là một **ReadableStream** — chỉ đọc được một lần. Khi browser đọc response để render, stream bị consumed. Cache nhận một stream đã rỗng → cache entry bị hỏng (0 bytes body).

Fix: `networkResponse.clone()` trước khi cache.

**Vấn đề 2 — Uncaught promise race:** `caches.open().then(cache.put())` là một Promise không được await và không được handle. Nếu cache operation fail (quota exceeded, disk error), error bị swallow silently. Không ảnh hưởng đến response trả về cho browser nhưng làm bug khó track.

```javascript
// Fix
return fetch(event.request).then((networkResponse) => {
  const responseToCache = networkResponse.clone(); // Clone trước khi dùng
  caches.open('dynamic').then((cache) => {
    cache.put(event.request, responseToCache); // Cache bản clone
  });
  return networkResponse; // Trả original cho browser
});
```

**Q3: Tại sao `SameSite=Lax` (default) không đủ cho một số apps và cần `SameSite=Strict`? Cho ví dụ cụ thể.**

A: `Lax` vẫn gửi cookie trong **top-level GET navigation**. Điều này đủ an toàn với hầu hết apps, nhưng có một số scenarios nguy hiểm:

**Scenario: Banking app với "magic link" URL:**

Nếu bank.com có endpoint `GET /confirm-transfer?id=abc123&amount=1000` để confirm transfer (thậm chí "chỉ là GET để confirm"), một attacker có thể gửi link đó qua email. Victim click link → top-level GET navigation → `Lax` cookie được gửi → transfer được confirm. Đây là CSRF via GET.

**Scenario: Admin panel:**

Admin đang login vào admin.example.com. Attacker gửi email chứa link hợp lệ `https://admin.example.com/users/delete?id=123`. Admin click → top-level GET → session cookie được gửi (Lax) → user bị xóa.

Với `SameSite=Strict`: Click link từ email → cross-site navigation → cookie KHÔNG được gửi → admin thấy trang login thay vì action được thực thi.

Nhược điểm của Strict: Admin phải login lại khi navigate từ email, Slack message, hay bất kỳ external source nào — UX xấu nhưng security tốt hơn.

**Q4: Tại sao FileSystemSyncAccessHandle của OPFS chỉ available trong Worker thread, không phải main thread?**

A: Vì đây là blocking I/O API. `syncHandle.read(buffer, { at: 0 })` block thread cho đến khi read hoàn thành.

Nếu cho phép trên main thread, sẽ gây **jank** nghiêm trọng:

```
Main thread bị block vì sync I/O:
  │
  ├── UI events queue up (click, scroll không response)
  ├── rAF callbacks không chạy
  ├── Style recalculation không xảy ra
  ├── Browser UI (loading spinner, tabs) có thể freeze
  │
  └── "Aw, Snap!" hoặc "Page Unresponsive" dialog
```

Worker thread không có DOM access và không tham gia vào rendering pipeline. Blocking worker thread chỉ delay computation trong worker đó — main thread hoàn toàn unaffected, UI vẫn mượt.

Đây là lý do design pattern đúng: SQLite WASM chạy trong Worker với sync OPFS → main thread communicate qua async `postMessage()` → không có blocking nào trên main thread.

**Q5: Tại sao quota của localStorage (5-10MB) và Cookie (4KB/cookie) lại tách biệt khỏi quota chung của IndexedDB/Cache API/OPFS?**

A: Lý do lịch sử và design:

**localStorage (2009):** Ra đời trước khi có unified storage quota model. Spec đặt hard limit là 5MB per origin (browsers implement từ 5-10MB). Không tích hợp với `StorageManager.estimate()` vì thiếu API để track usage.

**Cookies (1994):** Cực kỳ cũ. Giới hạn 4KB/cookie xuất phát từ HTTP header size limits và thời đại network băng thông thấp. Cookies được gửi trong **mọi request** → càng nhỏ càng tốt. Nếu cookie 1MB, mọi request sẽ mang theo 1MB overhead. Đây là design constraint của protocol, không chỉ của browser.

**IndexedDB/Cache API/OPFS:** Modern APIs được thiết kế từ đầu với unified quota model. `StorageManager` biết về tất cả chúng và có thể báo cáo usage. Browser eviction policy cũng quản lý chúng theo cùng một cơ chế.

Hệ quả thực tế: Khi kiểm tra storage với `navigator.storage.estimate()`, kết quả **không bao gồm** localStorage hay cookies usage — đây là điểm blind spot cần biết khi monitor storage health.

## Câu hỏi ôn tập (NotebookLM)

### Câu 1: Giải thích tại sao `HttpOnly` cookie ngăn được XSS attack nhưng không ngăn được CSRF attack, và cần kết hợp thêm cơ chế gì để protect đầy đủ.

**Đáp án:**

**Cơ chế bảo vệ của HttpOnly:**

`HttpOnly` là một attribute của cookie được Browser Process's Network Service enforce. Khi cookie có flag này, nó được lưu trong Cookie Jar (SQLite file) bình thường, nhưng khi Renderer Process (nơi JS chạy) gọi `document.cookie` qua Blink binding layer, Blink **filter out** tất cả `HttpOnly` cookies trước khi trả về string cho JS. Cookie vẫn được Network Service đính kèm vào HTTP requests, nhưng JS không bao giờ thấy value của nó.

XSS attack inject malicious script vào page. Script đó chạy trong Renderer Process và cố gọi `document.cookie` để steal session token. `HttpOnly` block điều này tại tầng Blink — JS nhận empty string hoặc string không có HttpOnly cookies.

**Tại sao CSRF vẫn hoạt động:**

CSRF không cần JavaScript và không cần đọc cookie. CSRF exploit cách browser tự động đính kèm cookies vào mọi request đến matching origin/domain. Attacker tạo một page trên evil.com với form POST hoặc image request đến bank.com. Khi victim load evil.com, browser gửi request đến bank.com và Network Service **tự động attach tất cả matching cookies** — kể cả HttpOnly cookies — vào request đó. Server nhận request với valid session cookie và execute action.

**Defense in depth:**

1. `SameSite=Lax` (minimum): Ngăn cookie được gửi trong cross-site POST requests và subrequests. Còn để lộ với top-level GET navigation.
2. `SameSite=Strict` (maximum): Không bao giờ gửi trong cross-site context. UX trade-off: user phải re-login khi navigate từ external links.
3. **CSRF tokens:** Server tạo random token, embed vào form hidden field hoặc custom header. Token không thể bị attacker biết (không có `document.cookie` access và không có CORS). Server verify token trước khi process request.
4. Double Submit Cookie: Gửi CSRF token cả trong cookie (JS-readable, không HttpOnly) lẫn trong request body/header. Server verify chúng match. Attacker không thể forge vì cần đọc cookie của victim.

Kết hợp `HttpOnly` + `SameSite=Lax` + CSRF tokens là triple-layered defense tiêu chuẩn cho production auth.

### Câu 2: Giải thích cơ chế eviction của Browser Storage và tại sao một PWA có thể bị mất toàn bộ offline data mà không có cảnh báo.

**Đáp án:**

**Storage model theo mức độ persistence:**

Browser chia storage thành hai tier:

- **Best-Effort (default):** Tất cả storage (IndexedDB, Cache API, OPFS) của một origin mặc định là "best-effort" — browser có thể xóa bất cứ lúc nào dưới storage pressure mà không cần hỏi user.
- **Persistent:** Origin request qua `navigator.storage.persist()`. Nếu granted, browser không tự evict — chỉ user action (clear site data) mới xóa được.

**Eviction process:**

Khi OS báo disk space thấp (hoặc browser phát hiện storage pressure):

1. Browser Storage Service kiểm tra tất cả origins có best-effort storage
2. Sắp xếp theo LRU — origin nào không được access lâu nhất sẽ bị evict trước
3. **Toàn bộ origin bị xóa cùng lúc:** IndexedDB + Cache API + OPFS + Service Worker registrations của origin đó bị xóa — không phải từng entry
4. Không có callback hay warning cho origin đang bị evict

**Tại sao PWA bị ảnh hưởng đặc biệt:**

PWA dùng Service Worker + Cache API để serve offline content. Nếu:

1. User cài PWA
2. Không dùng trong 2-3 tuần
3. Disk của user gần đầy
4. Browser evict origin theo LRU

→ Lần sau user mở PWA khi offline: Service Worker tồn tại nhưng Cache API empty → tất cả requests fail → app crash hoặc hiện error page.

**Fix đúng:**

```javascript
// Khi install PWA hoặc user đăng nhập lần đầu
async function requestPersistentStorageForPWA() {
  const isPersisted = await navigator.storage.persisted();
  if (!isPersisted) {
    const granted = await navigator.storage.persist();
    if (!granted) {
      // Fallback: warn user, sync data to server more frequently
      scheduleMoreFrequentSync();
    }
  }
}

// Monitor và report storage health
async function storageHealthCheck() {
  const { quota, usage } = await navigator.storage.estimate();
  const percent = (usage / quota) * 100;

  if (percent > 85) {
    // Prune oldest cache entries
    await pruneOldCaches();
    // Notify analytics
    reportEvent('storage_pressure', { percent });
  }
}
```

Chrome tự động grant persistent storage nếu: site được bookmark, installed as PWA, notification permission granted, hoặc high engagement score. Firefox và Safari yêu cầu explicit user permission prompt.

### Câu 3: Giải thích tại sao OPFS với `FileSystemSyncAccessHandle` có thể chạy SQLite với performance gần native, trong khi IndexedDB không đáp ứng được requirement này.

**Đáp án:**

**Vấn đề của SQLite với IndexedDB:**

SQLite là một database engine được thiết kế với **synchronous, random-access file I/O** làm foundation. Internal architecture của SQLite:

- Đọc/ghi theo **pages** (thường 4KB mỗi page)
- Một query có thể trigger hàng trăm random page reads/writes
- Mỗi operation cần immediate result để decide next operation
- B-tree traversal: đọc node → decide direction → đọc child node → repeat

IndexedDB là **asynchronous**: mỗi operation là IPC round-trip từ Renderer Process → Browser Process → LevelDB backend → response IPC. Để emulate SQLite synchronous page I/O trên IndexedDB:

```
SQLite đọc page 47:
→ Tạo IndexedDB get request
→ Chờ IPC round-trip (có thể 1-5ms)
→ Nhận result
→ Process result
→ Quyết định đọc page 83
→ Tạo IndexedDB get request
→ Chờ IPC round-trip lần nữa
→ ...repeat hàng trăm lần cho một query

Total latency: hàng trăm ms cho một query đơn giản
```

**Tại sao OPFS Sync Access Handle giải quyết được:**

`FileSystemSyncAccessHandle` cung cấp **direct synchronous file I/O** trong Worker thread:

```
Worker thread (không có DOM, có thể block):

SQLite đọc page 47:
→ syncHandle.read(buffer, { at: 47 * 4096 })
→ OS file read (có thể có OS-level cache)
→ Trả về ngay (microseconds, không phải milliseconds)
→ SQLite process result ngay lập tức
→ SQLite đọc page 83:
→ syncHandle.read(buffer, { at: 83 * 4096 })
→ Trả về ngay
→ Total: microseconds, gần native SQLite
```

**Tại sao Worker là prerequisite:**

Sync I/O block thread. Main thread không thể block (UI freeze). Worker thread không có DOM → blocking là acceptable → sync I/O có thể.

Pattern hoàn chỉnh của production implementation (như Notion, Linear):

```
Main Thread                    SQLite Worker
     │                               │
     │ postMessage({sql, params})    │
     ├──────────────────────────────►│ Nhận request
     │                               │ SQLite engine chạy
     │                               │ OPFS sync reads/writes
     │                               │ (blocking, nhưng chỉ block worker)
     │                               │ Query hoàn thành
     │ postMessage({rows: [...]}) ◄──┤
     │                               │
     │ Nhận kết quả async            │
     │ Main thread KHÔNG bao giờ block
```

Kết quả: SQLite WASM với OPFS backend đạt **90-95% performance của native SQLite** trong benchmark, vượt xa bất kỳ IndexedDB-based storage nào. Đây là lý do các apps cần offline-first database (document editors, email clients, kanban boards) đang migrate sang pattern này.
