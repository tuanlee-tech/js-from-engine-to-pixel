## Phần 0 — Mental Model: Tại Sao Cần CI/CD & Code Quality Tools

Đặt vấn đề đúng trước khi học tool:

```
Vấn đề khi team lớn hơn:

1 developer:     Nhớ rules trong đầu, tự enforce
10 developers:   "Mỗi người code theo style khác nhau"
                 "Bug pass qua review vì reviewer không để ý"
                 "Deploy lúc 11pm → production down → không ai biết tại sao"

Code quality tools giải quyết:
  ESLint:    Enforce rules AUTOMATICALLY — không cần human review trivial issues
  Prettier:  Eliminate style debates — format là không cần discuss
  Git hooks: Enforce BEFORE code reaches repo — catch sớm nhất có thể
  CI/CD:     Enforce BEFORE code reaches production — automated safety net

Principle: Shift left — bắt lỗi càng sớm càng rẻ

  Cost of fixing bug:
  Development: $1
  Code review: $10
  QA/Staging:  $100
  Production:  $1000+
```

## Phần 1 — ESLint: Cơ Chế Thật

### 1.1 — ESLint Hoạt Động Như Thế Nào

ESLint không phải "search và replace" hay regex matching. Nó là **static code analyzer** dựa trên AST.

```
Source code (string)
         │
         ▼
    PARSER (Espree mặc định, hoặc @typescript-eslint/parser)
    → Parse code thành AST (Abstract Syntax Tree)
         │
         ▼
    AST (JavaScript Object)
    {
      type: "Program",
      body: [
        {
          type: "VariableDeclaration",
          kind: "var",           ← ESLint "no-var" rule check ở đây
          declarations: [...]
        }
      ]
    }
         │
         ▼
    RULES RUN (mỗi rule là visitor pattern trên AST)
    Rule "no-var":
      → Khi gặp VariableDeclaration node với kind === "var"
      → Report lỗi
         │
         ▼
    REPORT (array of errors/warnings với location)
    [
      {
        ruleId: "no-var",
        message: "Unexpected var, use let or const instead",
        line: 5, column: 3
      }
    ]
```

**Tại sao AST thay vì regex?**

```javascript
// Regex approach fail với:
const message = "var x = 1"; // String chứa "var" — false positive
// Hoặc:
var                           // var trên dòng riêng — regex không catch

// AST approach:
// Parser hiểu ngữ nghĩa của code
// VariableDeclaration node với kind="var" = chắc chắn là khai báo var
// Không có false positive
```

### 1.2 — Cấu Hình ESLint: Flat Config (Modern)

ESLint v9 chuyển sang **Flat Config** (`eslint.config.js`). Legacy `.eslintrc` vẫn hoạt động nhưng deprecated.

```javascript
// eslint.config.js — modern flat config
import js from '@eslint/js';
import typescript from '@typescript-eslint/eslint-plugin';
import tsParser from '@typescript-eslint/parser';
import react from 'eslint-plugin-react';
import reactHooks from 'eslint-plugin-react-hooks';
import importPlugin from 'eslint-plugin-import';

export default [
  // 1. BASE CONFIG — áp dụng cho tất cả files
  js.configs.recommended,

  // 2. GLOBAL IGNORES (thay thế .eslintignore)
  {
    ignores: ['dist/**', 'node_modules/**', '*.min.js', 'coverage/**'],
  },

  // 3. TYPESCRIPT FILES CONFIG
  {
    files: ['**/*.{ts,tsx}'],

    // Parser: biến TS code thành AST mà ESLint hiểu
    languageOptions: {
      parser: tsParser,
      parserOptions: {
        // Project: cho phép rules dùng type information
        // (một số rules cần TypeScript type checker)
        project: './tsconfig.json',
        tsconfigRootDir: import.meta.dirname,
      },
    },

    plugins: {
      '@typescript-eslint': typescript,
      react: react,
      'react-hooks': reactHooks,
      import: importPlugin,
    },

    rules: {
      // ── TYPESCRIPT RULES ──────────────────────────────

      // Không dùng any — weakens type safety
      '@typescript-eslint/no-explicit-any': 'error',

      // Phải handle promise — tránh unhandled rejection
      '@typescript-eslint/no-floating-promises': 'error',
      // Lý do: await someFunction() mà không handle → lỗi bị nuốt im lặng

      // Không dùng non-null assertion (!) bừa bãi
      '@typescript-eslint/no-non-null-assertion': 'warn',

      // Consistent type imports — tránh circular import issues
      '@typescript-eslint/consistent-type-imports': [
        'error',
        {
          prefer: 'type-imports',
          // import type { User } thay vì import { User }
          // Giúp bundler tree shake type-only imports
        },
      ],

      // ── REACT RULES ────────────────────────────────────

      // Hook rules — QUAN TRỌNG
      'react-hooks/rules-of-hooks': 'error', // Hooks chỉ ở top level
      'react-hooks/exhaustive-deps': 'warn', // deps array phải đủ

      // ── IMPORT RULES ───────────────────────────────────

      // Không import default từ file chỉ có named exports
      'import/no-default-export': 'off', // Tuỳ team preference

      // Đảm bảo import order nhất quán
      'import/order': [
        'error',
        {
          groups: [
            'builtin', // node built-ins (path, fs)
            'external', // node_modules
            'internal', // aliased paths (@/components)
            ['parent', 'sibling', 'index'], // relative imports
          ],
          'newlines-between': 'always',
          alphabetize: { order: 'asc' },
        },
      ],

      // ── GENERAL RULES ──────────────────────────────────

      'no-console': [
        'warn',
        {
          allow: ['warn', 'error'], // console.warn và console.error OK
          // console.log không OK trong production code
        },
      ],

      'no-debugger': 'error', // Không commit debugger statement

      'prefer-const': 'error', // Dùng const khi không reassign

      'no-var': 'error', // Không dùng var

      // Không dùng == (dùng ===)
      eqeqeq: ['error', 'always', { null: 'ignore' }],
      // null: ignore → x == null còn OK (check cả null và undefined)
    },
  },

  // 4. TEST FILES — rules khác cho test
  {
    files: ['**/*.{test,spec}.{ts,tsx}', '**/__tests__/**'],
    rules: {
      // Test có thể dùng any nhiều hơn
      '@typescript-eslint/no-explicit-any': 'off',
      // console.log trong test OK
      'no-console': 'off',
    },
  },
];
```

### 1.3 — Hiểu Severity Levels

```javascript
// 3 levels:
'rule-name': 'off'    // 0 — disabled
'rule-name': 'warn'   // 1 — warning (không fail build)
'rule-name': 'error'  // 2 — error (fail build, non-zero exit code)

// Khi nào dùng warn vs error:
// ERROR:  Rules mà violation là CHẮC CHẮN bug hoặc security issue
//         no-debugger, no-var, react-hooks/rules-of-hooks
//
// WARN:   Rules mà violation là code smell, đôi khi có lý do hợp lệ
//         no-console (đôi khi cần log trong dev),
//         @typescript-eslint/no-explicit-any (migration phase)
//
// OFF:    Rules không phù hợp với project của bạn

// Khi chạy trong CI:
// ESLint exit code 0 = success
// ESLint exit code 1 = có errors
// → CI check exit code, fail build nếu có errors
// Warnings KHÔNG fail CI (exit code vẫn 0)
```

ESLint với TypeScript type-aware rules có thể chậm vì phải chạy TypeScript compiler.

```bash
# Output:
```

## Phần 2 — Prettier: Cơ Chế và Integration

### 2.1 — Prettier Khác ESLint Như Thế Nào

```
ESLint:    LINTER — tìm bugs, enforce coding patterns
           Rules có opinions về LOGIC và STYLE
           Nhiều rules auto-fixable, nhiều rules không

Prettier:  FORMATTER — chỉ quan tâm tới FORMATTING
           Không quan tâm bugs hay patterns
           Opinionated — ít config options
           ALWAYS auto-fix (đó là điểm mạnh)

Vấn đề khi dùng cả hai: overlap
  ESLint có rules về indent, quotes, semicolons
  Prettier cũng format những thứ đó
  → Conflict → ESLint error về thứ Prettier vừa format

Solution: eslint-config-prettier
  → Tắt tất cả ESLint rules conflict với Prettier
  → ESLint làm code quality, Prettier làm formatting
```

### 2.2 — Cấu Hình Prettier

```javascript
// prettier.config.js (hoặc .prettierrc)
export default {
  // Độ dài line tối đa — Prettier tự xuống dòng
  printWidth: 100,

  // 2 spaces indent (không phải tabs)
  tabWidth: 2,
  useTabs: false,

  // Semicolons: true = luôn có, false = chỉ khi cần (ASI)
  semi: true,

  // Single quotes cho JS, double quotes cho JSX attributes
  singleQuote: true,
  jsxSingleQuote: false,

  // Trailing commas — ES5: arrays và objects
  // "all" = trailing comma ở function params (ES2017+)
  trailingComma: 'es5',

  // Spaces trong object literals: { foo: bar }
  bracketSpacing: true,

  // Arrow function params: avoid = không cần parens khi 1 param
  // x => x  vs  (x) => x
  arrowParens: 'always', // luôn có parens — consistent, dễ add types

  // Prettier không re-order imports (đó là job của ESLint)
};
```

### 2.3 — Prettier vs ESLint: Phân Chia Trách Nhiệm Rõ Ràng

```javascript
// eslint.config.js — thêm prettier config CUỐI CÙNG
import prettierConfig from 'eslint-config-prettier';

export default [
  js.configs.recommended,
  // ... các configs khác ...

  // PHẢI ở cuối — override/disable ESLint rules conflict với Prettier
  prettierConfig,
];

// Sau khi setup:
// ESLint checks: no-var, no-console, react-hooks, import/order, v.v.
// Prettier handles: indent, quotes, semicolons, line length, v.v.
// Không còn conflict
```

## Phần 3 — Git Hooks: Enforce Trước Khi Code Vào Repo

### 3.1 — Cơ Chế: Git Hooks Là Gì

```
Git hooks = scripts chạy tự động tại các lifecycle events của Git

Vị trí: .git/hooks/  (local, không commit được)
        → Đây là vấn đề: mỗi developer phải setup thủ công

Solution: Husky — quản lý git hooks qua package.json
          → Hooks được lưu trong repo (.husky/)
          → npm install tự động setup hooks
```

**Lifecycle quan trọng nhất:**

```
git commit -m "feat: add login"
     │
     ├── pre-commit hook CHẠY TRƯỚC
     │   → Nếu exit code != 0: commit bị CANCEL
     │   → Dùng để: lint, format, type check
     │
     ├── prepare-commit-msg hook
     │
     ├── commit-msg hook CHẠY VỚI COMMIT MESSAGE
     │   → Validate commit message format
     │   → Nếu fail: commit bị CANCEL
     │
     └── post-commit hook (sau khi commit xong)
         → Không thể cancel commit
         → Dùng để: notifications, logging

git push
     │
     └── pre-push hook
         → Chạy trước khi push lên remote
         → Dùng để: run tests (nặng hơn pre-commit)
```

### 3.2 — Setup Husky + lint-staged

```bash
# Install
npm install --save-dev husky lint-staged

# Khởi tạo husky (thêm "prepare" script vào package.json)
npx husky init
# Tạo file .husky/pre-commit với content mẫu
```

```json
// package.json
{
  "scripts": {
    // "prepare" chạy tự động sau npm install
    // → Đảm bảo mọi developer có hooks sau khi clone repo
    "prepare": "husky",
    "lint": "eslint src/",
    "lint:fix": "eslint src/ --fix",
    "format": "prettier --write src/",
    "type-check": "tsc --noEmit"
  },

  // lint-staged config: chỉ lint FILES ĐANG ĐƯỢC STAGED
  // Không lint toàn bộ codebase mỗi commit → nhanh hơn
  "lint-staged": {
    // TypeScript và TSX files:
    "*.{ts,tsx}": [
      "prettier --write", // Format trước
      "eslint --fix", // Lint và auto-fix
      "eslint" // Nếu còn errors sau fix → fail
    ],
    // JSON, CSS, Markdown:
    "*.{json,css,md}": ["prettier --write"]
  }
}
```

```bash
# .husky/pre-commit
#!/bin/sh
npx lint-staged
# Exit code từ lint-staged → nếu != 0, commit bị cancel
```

### 3.3 — Commitlint: Enforce Commit Message Format

**Conventional Commits** là standard cho commit messages:

```
<type>(<scope>): <description>

[optional body]

[optional footer]

Types:
  feat:     New feature (minor version bump nếu dùng semantic-release)
  fix:      Bug fix (patch version bump)
  docs:     Documentation only
  style:    Formatting, không đổi logic
  refactor: Refactoring, không phải fix hay feature
  perf:     Performance improvement
  test:     Adding/updating tests
  build:    Build system, dependencies
  ci:       CI configuration
  chore:    Maintenance tasks

Ví dụ:
  feat(auth): add OAuth Google login
  fix(api): handle 429 rate limit response
  perf(table): virtualize rows for large datasets
  feat!: remove deprecated API endpoints  ← "!" = breaking change
```

```bash
# Install
npm install --save-dev @commitlint/cli @commitlint/config-conventional
```

```javascript
// commitlint.config.js
export default {
  extends: ['@commitlint/config-conventional'],

  // Customize rules nếu cần:
  rules: {
    // Type phải thuộc danh sách này
    'type-enum': [
      2,
      'always',
      [
        'feat',
        'fix',
        'docs',
        'style',
        'refactor',
        'perf',
        'test',
        'build',
        'ci',
        'chore',
        'revert',
      ],
    ],

    // Subject (description) không quá 100 chars
    'subject-max-length': [2, 'always', 100],

    // Subject không viết hoa chữ đầu
    'subject-case': [2, 'never', ['start-case', 'pascal-case', 'upper-case']],

    // Không có dấu chấm cuối subject
    'subject-full-stop': [2, 'never', '.'],
  },
};
```

```bash
# .husky/commit-msg
#!/bin/sh
npx --no -- commitlint --edit $1
# $1 = path tới file chứa commit message (do Git cung cấp)
```

**Tại sao Conventional Commits quan trọng:**

```
1. AUTOMATED CHANGELOG:
   conventional-changelog tự generate CHANGELOG.md từ commits

2. SEMANTIC VERSIONING:
   semantic-release tự bump version:
   feat: → minor (1.0.0 → 1.1.0)
   fix: → patch (1.0.0 → 1.0.1)
   feat!: → major (1.0.0 → 2.0.0)

3. READABLE GIT HISTORY:
   git log --oneline:

   BAD:                        GOOD:
   fix stuff                   fix(auth): resolve token refresh race condition
   WIP                         feat(dashboard): add export to CSV feature
   asdf                        perf(table): virtualize 10k row rendering
   changes                     docs(api): add rate limiting documentation
```

### 3.4 — Vấn Đề: Developer Bypass Hooks

```bash
# Developer có thể bypass hooks:
git commit -m "fix" --no-verify  # -n hoặc --no-verify bypass tất cả hooks

# Đây là lý do hooks KHÔNG phải là defense cuối cùng
# Hooks = first line of defense (fast feedback, developer experience)
# CI = second line of defense (không thể bypass, blocks merge)

# Nguyên tắc: Mọi check trong hook PHẢI cũng chạy trong CI
```

## Phần 4 — GitHub Actions: CI/CD Pipeline

### 4.1 — Cơ Chế: GitHub Actions Hoạt Động Như Thế Nào

```
Khi developer push code hoặc open PR:

GitHub:
  1. Detect event (push, pull_request, v.v.)
  2. Tìm workflow files trong .github/workflows/*.yml
  3. Tạo "workflow run" — log, UI trong Actions tab
  4. Allocate runner (virtual machine: Ubuntu, Windows, macOS)
  5. Runner clone repo
  6. Chạy từng step trong jobs

Runner là máy ảo có sẵn:
  ubuntu-latest:  Ubuntu 22.04 (~2 vCPU, 7GB RAM, 14GB SSD)
  windows-latest: Windows Server 2022
  macos-latest:   macOS Ventura

Billing:
  Public repos:   FREE (unlimited minutes)
  Private repos:  Free tier 2000 min/month, sau đó tính phí
                  ubuntu: $0.008/min
                  windows: $0.016/min (2x)
                  macos: $0.08/min (10x)
```

### 4.2 — Anatomy của Workflow File

```yaml
# .github/workflows/ci.yml

# Tên hiện trong GitHub UI
name: CI

# TRIGGERS — khi nào workflow chạy
on:
  # Chạy khi có push lên các branches này
  push:
    branches: [main, develop]

  # Chạy khi có PR targeting branches này
  pull_request:
    branches: [main, develop]
    # Chỉ trigger khi PR được opened, reopened, hoặc có new commits
    types: [opened, synchronize, reopened]

# JOBS — các jobs chạy song song (mặc định)
jobs:
  # JOB 1: Code Quality
  quality:
    name: Code Quality
    # Runner OS
    runs-on: ubuntu-latest

    steps:
      # Step 1: Checkout code vào runner
      - name: Checkout
        uses: actions/checkout@v4
        # "uses" = dùng action có sẵn từ GitHub Marketplace
        # actions/checkout@v4 = official action để clone repo

      # Step 2: Setup Node.js
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          # Cache npm để không download lại mỗi run
          cache: 'npm'

      # Step 3: Install dependencies
      - name: Install dependencies
        run: npm ci
        # npm ci: clean install, dùng package-lock.json
        # Nhanh hơn npm install, đảm bảo reproducible

      # Step 4: Type check
      - name: TypeScript check
        run: npm run type-check
        # Fail nếu có TypeScript errors

      # Step 5: Lint
      - name: ESLint
        run: npm run lint
        # Fail nếu có ESLint errors (exit code != 0)

      # Step 6: Format check
      - name: Prettier check
        run: npx prettier --check src/
        # --check: không format, chỉ kiểm tra
        # Fail nếu có files chưa được format

  # JOB 2: Tests (chạy SONG SONG với quality job)
  test:
    name: Tests
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci

      - name: Run tests
        run: npm run test -- --coverage
        # -- để pass args xuống vitest/jest

      # Upload coverage để track trends
      - name: Upload coverage
        uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
          # secrets: encrypted variables stored in GitHub
          # Settings → Secrets and variables → Actions

  # JOB 3: Build (chỉ chạy khi quality VÀ test pass)
  build:
    name: Build
    runs-on: ubuntu-latest
    needs: [quality, test] # ← Dependency: chờ jobs khác xong

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci

      - name: Build
        run: npm run build

      # Upload build artifacts để jobs khác hoặc deployments dùng
      - name: Upload build artifacts
        uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
          retention-days: 7
```

**Job dependency graph:**

```
          ┌─────────────┐  ┌─────────────┐
          │   quality   │  │    test     │
          │   ~2 min    │  │   ~3 min    │
          └──────┬──────┘  └──────┬──────┘
                 │                │
                 └───────┬────────┘
                         │ (cả hai phải pass)
                         ▼
                  ┌─────────────┐
                  │    build    │
                  │   ~2 min    │
                  └─────────────┘

Tổng thời gian: ~5 min (jobs chạy parallel)
Không parallel: ~7 min (sequential)
```

### 4.3 — Caching: Tăng Speed CI Lên 3-5x

```yaml
# Caching node_modules — giảm từ 60s → 10s install time

- name: Cache node_modules
  uses: actions/cache@v4
  with:
    # Thư mục cần cache
    path: ~/.npm
    # Cache key: nếu key này tồn tại → restore cache, skip install
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
    # Restore key: nếu exact key không có, thử key này
    restore-keys: |
      ${{ runner.os }}-node-

# Cơ chế:
# hashFiles('**/package-lock.json') → hash của lockfile
# Nếu lockfile không đổi → hash không đổi → cache hit → restore node_modules
# Nếu lockfile thay đổi (thêm/xóa package) → cache miss → npm ci → save cache mới

# actions/setup-node@v4 có built-in caching:
- uses: actions/setup-node@v4
  with:
    node-version: '20'
    cache: 'npm' # Tự động cache ~/.npm
```

### 4.5 — Matrix Strategy: Test Nhiều Versions


### 6.2 — Anatomy của ESLint Rule

```javascript
// Mỗi ESLint rule là object với:
// 1. meta: mô tả rule
// 2. create(context): trả về visitor object

// Rule: no-direct-infrastructure-import
// Không cho domain layer import từ infrastructure layer

// eslint-plugin-architecture/rules/no-direct-infrastructure-import.js
export default {
  meta: {
    type: 'problem',
    docs: {
      description: 'Domain layer must not import from infrastructure layer',
      recommended: true,
    },
    schema: [], // Không có options
    messages: {
      noDirectImport:
        'Domain layer cannot import from infrastructure: {{importPath}}',
    },
  },

  create(context) {
    // context: thông tin về file đang được checked
    //   context.getFilename() → đường dẫn file hiện tại
    //   context.report() → báo lỗi

    const filename = context.getFilename();
    const isDomainFile = filename.includes('/domain/');

    // Nếu không phải domain file, không cần check
    if (!isDomainFile) return {};

    // Visitor object: keys là AST node types
    // ESLint gọi các functions này khi traverse AST
    return {
      // ImportDeclaration = bất kỳ import statement nào
      ImportDeclaration(node) {
        const importPath = node.source.value;
        // node.source.value = string trong quotes của import
        // import { X } from './path' → './path'

        if (
          importPath.includes('/infrastructure/') ||
          importPath.includes('/infrastructure')
        ) {
          context.report({
            node, // Node gây ra lỗi (dùng để highlight)
            messageId: 'noDirectImport',
            data: { importPath }, // Dữ liệu cho message template
          });
        }
      },
    };
  },
};
```

## Câu Hỏi Kiểm Tra

**Q1: Tại sao ESLint dùng AST thay vì regex để analyze code?**

A: Regex chỉ match text patterns — không hiểu ngữ nghĩa. String `"var x = 1"` chứa chữ "var" sẽ bị regex flag nhầm. Code `var\n  x` trên 2 dòng có thể bị regex bỏ qua. AST (Abstract Syntax Tree) là structural representation của code — parser hiểu ngữ nghĩa, biết `VariableDeclaration` với `kind: "var"` là khai báo var dù được viết như thế nào. Rules operate trên AST nodes → không có false positive, không bỏ sót edge cases.

**Q2: Điều gì xảy ra nếu lint-staged fail trong pre-commit hook?**

A: `lint-staged` trả về exit code khác 0 khi có lỗi. Git kiểm tra exit code của hook script — nếu khác 0, commit bị **cancel hoàn toàn**. Files staged vẫn giữ nguyên trạng thái (chưa commit). Developer phải fix lỗi, `git add` lại các files đã được auto-fix bởi Prettier/ESLint, rồi commit lại. Quan trọng: lint-staged chỉ chạy trên **staged files** (files trong `git add`), không phải toàn bộ codebase, nên fast ngay cả với project lớn.

**Q3: Tại sao job `build` cần `needs: [quality, test]` thay vì chạy song song?**

A: Dependency chain thể hiện logic: không có ý nghĩa khi build code có TypeScript errors hoặc failing tests. Nếu `quality` hoặc `test` fail, `build` job không chạy → tiết kiệm CI minutes (build có thể tốn 2-5 phút). Nếu chạy song song và `quality` fail → build waste resources và tạo ra artifact từ broken code. `needs` đảm bảo chỉ build khi code đã pass quality gate. Đây là pattern "fail fast" — dừng pipeline sớm nhất có thể khi phát hiện vấn đề.

**Q4: Đoạn workflow này có vấn đề gì?**

```yaml
on:
  push:
    branches: ['**'] # tất cả branches

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      - run: npx vercel --token ${{ secrets.VERCEL_TOKEN }} --prod
```

A: Nhiều vấn đề nghiêm trọng: (1) **Deploy production từ mọi branch** — push lên feature branch cũng trigger production deploy. (2) **Không có quality gate** — deploy mà không lint, test, type check. (3) **Không có `needs`** — nếu thêm quality job, deploy vẫn chạy song song với nó. (4) **Không có environment protection** — production nên có required reviewers. Fix: trigger chỉ từ `main`, thêm `needs: [quality, test, build]`, dùng GitHub Environments với protection rules, dùng `--prod` flag chỉ khi deploy từ main.

**Q5: `npm install` vs `npm ci` — tại sao CI luôn dùng `npm ci`?**

A: `npm install`: đọc `package.json`, resolve versions (trong semver ranges), có thể install versions khác nhau tùy thời điểm chạy, update `package-lock.json` nếu có inconsistency. `npm ci`: đọc **chính xác** `package-lock.json`, fail nếu `package-lock.json` không có hoặc mismatch với `package.json`, **không bao giờ update** lockfile, delete `node_modules` trước khi install (clean state). Kết quả: reproducible builds — cùng lockfile = cùng dependencies = cùng behavior, mọi lúc, mọi machine. Đây là yêu cầu cơ bản của CI: build phải deterministic.

## Câu Hỏi Ôn Tập (NotebookLM)

### Câu 1: ESLint hoạt động theo cơ chế nào ở cấp độ engine? Giải thích pipeline từ source code đến error report, và tại sao "type-aware rules" trong @typescript-eslint chậm hơn regular rules đáng kể?

**Đáp án:**

ESLint pipeline gồm 4 bước:

**Bước 1 — Parse:** Source code (string) được parser chuyển thành AST. Default parser là Espree (JavaScript). Với TypeScript files, dùng `@typescript-eslint/parser` — nó wrap TypeScript compiler để parse TS syntax thành ESLint-compatible AST.

**Bước 2 — AST Traversal:** ESLint traverse AST theo visitor pattern. Mỗi rule đăng ký "visitors" — functions được gọi khi gặp specific node types. Ví dụ rule `no-var` đăng ký visitor cho `VariableDeclaration` nodes — khi gặp node có `kind: "var"`, report lỗi.

**Bước 3 — Rule Execution:** Tất cả rules chạy trong cùng một traversal pass (single-pass). ESLint không traverse AST N lần cho N rules — nó traverse một lần và gọi tất cả relevant visitors.

**Bước 4 — Report:** Tổng hợp tất cả errors/warnings với location information (line, column), sort, format và output.

**Tại sao type-aware rules chậm:** Regular rules chỉ cần AST — parsing nhanh (~milliseconds/file). Type-aware rules (ví dụ `@typescript-eslint/no-floating-promises`) cần **type information** — biết return type của function là Promise hay không. Để có type information, ESLint phải chạy TypeScript type checker (`tsc`) trên toàn bộ project, không chỉ file đang check. TypeScript type checker resolve imports, check types across files, maintain type graph — đây là expensive operation tốn vài giây đến vài chục giây tùy project size. Đây là lý do `parserOptions.project` trong config cần trỏ tới `tsconfig.json` — TypeScript compiler cần biết toàn bộ project context.

### Câu 2: Giải thích cơ chế của Git hooks và tại sao hooks không phải là defense cuối cùng. Trong một team setup hoàn chỉnh, hooks và CI đóng vai trò gì khác nhau?

**Đáp án:**

**Cơ chế Git hooks:** Git hooks là executable scripts nằm trong `.git/hooks/`. Git chạy chúng tại các lifecycle events. Nếu script trả về exit code khác 0, Git abort operation tương ứng. `pre-commit` → abort commit. `commit-msg` → abort commit. `pre-push` → abort push.

**Vấn đề với `.git/hooks/`:** Thư mục `.git/` không được track bởi Git — không thể commit vào repo. Mỗi developer phải setup hooks thủ công khi clone. Husky giải quyết bằng cách lưu hooks trong `.husky/` (committed vào repo) và dùng npm `prepare` script để symlink hoặc copy vào `.git/hooks/` sau `npm install`.

**Tại sao không phải defense cuối cùng:** Developer có thể bypass với `git commit --no-verify`. Hooks là **local** — chỉ chạy trên máy developer, không chạy trên CI server. Nếu developer tắt hooks, xóa `.git/hooks/`, hoặc dùng Git GUI client không support hooks → không có protection.

**Phân chia trách nhiệm:**

Hooks = **Developer Experience Layer**: Fast feedback (chạy trong giây), catch lỗi trước khi vào repo, giảm PR review cycles. Chi phí là developer time khi commit.

CI = **Safety Net Layer**: Chạy trên clean environment (không có cached state, không có developer machine quirks), không thể bypass, blocks merge/deploy nếu fail. Chi phí là build minutes và wait time.

Nguyên tắc: **mọi check trong hook phải cũng có trong CI**. Hooks là optimization (fail fast trên máy developer), CI là enforcement (fail safe trước production). Nếu chỉ có hooks, bad code có thể vào repo. Nếu chỉ có CI, developer phải đợi 5 phút để biết có lỗi format. Cả hai cùng nhau: fast feedback + guaranteed enforcement.

### Câu 3: GitHub Actions workflow có `concurrency` group và `cancel-in-progress: true`. Giải thích cơ chế này hoạt động như thế nào, khi nào nên dùng, và khi nào KHÔNG nên dùng?

**Đáp án:**

**Cơ chế:** `concurrency` group là một named mutex. Khi workflow run mới trigger với cùng group name, GitHub kiểm tra: có run nào đang chạy với group này không? Nếu có và `cancel-in-progress: true` → cancel run cũ, chạy run mới. Nếu `cancel-in-progress: false` (default) → run mới queue, chờ run cũ xong.

```yaml
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  # group name = "CI/CD Pipeline-refs/heads/feature/login"
  # Mỗi branch có group riêng
  cancel-in-progress: true
```

**Scenario:** Developer push commit A lên PR → workflow start. 30 giây sau, push thêm commit B (fix typo). Workflow cho commit B trigger → cancel workflow của commit A (đang chạy) → chạy workflow mới cho commit B. Kết quả: không waste CI minutes chạy build cho commit A (sẽ bị overwrite bởi B).

**Khi nên dùng:**

- PR workflows — chỉ cần kết quả của commit mới nhất
- Development branches — intermediate commits không quan trọng
- Deploy preview environments — chỉ cần preview của latest code

**Khi KHÔNG nên dùng:**

- Production deployments — cancel một deploy đang chạy giữa chừng có thể để server ở inconsistent state (deploy cũ chưa xong, deploy mới chưa bắt đầu → partial deploy)
- Release workflows — mỗi release tag quan trọng, không nên cancel
- Workflows có side effects không thể rollback (database migrations, external API calls)

**Fix cho production:**

```yaml
concurrency:
  group: production-deploy
  cancel-in-progress: false # Queue, không cancel
  # Deploys chạy tuần tự, không cancel nhau
```
