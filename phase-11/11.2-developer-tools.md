## Phần 0 — Mental Model: DevTools Là Gì Thật Sự

DevTools không phải là "xem code". DevTools là **cửa sổ nhìn vào runtime state của browser engine**.

```
Bạn viết code → V8 + Blink (rendering engine) + Network stack xử lý
                      │
                      ▼
              DevTools Protocol (CDP)
              Chrome DevTools Protocol
              = WebSocket-based protocol
              browser expose để external tools inspect runtime
                      │
                      ▼
              DevTools UI (chính nó là 1 web app viết bằng JS)
              visualize data từ CDP

Điều này có nghĩa:
- Mọi thứ bạn thấy trong DevTools = live data từ engine
- Performance panel không "simulate" — nó record thật
- Memory panel không estimate — nó dump heap thật từ V8
- Network panel = actual requests từ browser network stack
```

**Hệ quả quan trọng:** DevTools bản thân cũng tốn resources. Khi mở DevTools, JavaScript profiling overhead tăng ~10-15%. Memory panel khi mở giữ thêm references. Luôn đo performance với DevTools **closed** trước, sau đó mở để investigate.

### 1.2 — Computed Styles: CSSOM Resolution

```
Styles panel có 2 tab quan trọng:

1. "Styles" tab:
   Hiện tất cả CSS rules match element
   Theo thứ tự specificity (cao → thấp)

```

**Debug workflow: Tại sao CSS không apply?**

```
Symptom: margin-top: 20px không có effect

Check Styles tab:
  ├── Strikethrough? → Override bởi rule khác (check specificity)
  ├── Yellow warning icon? → Invalid value
  ├── Greyed out? → Property không áp dụng cho display type này
  │   (margin-top không work trên display:inline)
  └── Không thấy rule? → Selector không match

Check Computed tab:
  → Search "margin-top"
  → Thấy value được compute từ đâu (click arrow → jump to rule)
```

## Phần 2 — Network Panel: Request Lifecycle

### 2.1 — Cơ Chế: Waterfall Chart Là Gì

Mỗi row trong Network panel là 1 request. Mỗi row có waterfall bar. Hiểu màu sắc = hiểu tại sao request chậm.

```
Waterfall bar anatomy:
┌─────────────────────────────────────────────────────────────────┐
│ Queueing    │ Stalled  │ DNS │ Connect │ TLS  │ TTFB  │ Download│
│ (white)     │ (grey)   │     │         │      │(green)│ (blue)  │
└─────────────────────────────────────────────────────────────────┘

Queueing:   Request đang chờ trong queue
            HTTP/1.1: max 6 parallel connections/domain
            → Nhiều requests cùng domain → queue
            HTTP/2: multiplexing, không có giới hạn này

Stalled:    Request đã gửi nhưng browser chưa nhận byte đầu tiên
            Có thể do: network latency, server think time

DNS Lookup: Resolve hostname → IP address
            Lần đầu: 20-120ms
            Sau đó: cached (TTL theo DNS record)
            → preconnect/dns-prefetch hints giúp ở đây

Connect:    TCP handshake (SYN → SYN-ACK → ACK)
            ~1 RTT (Round Trip Time)

TLS:        TLS negotiation (nếu HTTPS)
            TLS 1.3: 1 RTT (cải thiện từ TLS 1.2 là 2 RTT)

TTFB:       Time To First Byte
            = Server processing time + network time
            TTFB cao = server chậm (database, computation)
            Target: < 200ms cho cached, < 600ms cho dynamic

Download:   Nhận response body
            Phụ thuộc bandwidth và response size
```

**Ví dụ đọc waterfall:**

```
Request A: [████████████████████████] 2s total
           [   Queueing 1.5s  ][TTFB][DL]
           → Vấn đề: HTTP/1.1 connection limit
           → Fix: HTTP/2, domain sharding (workaround cũ), hoặc reduce requests

Request B: [████████████████████████] 2s total
           [Connect][TLS][TTFB 1.8s ][DL]
           → Vấn đề: Server slow (TTFB cao)
           → Fix: Server-side caching, CDN, optimize DB queries

Request C: [████████████████████████] 2s total
           [Connect][TLS][TTFB][████ Download 1.5s ████]
           → Vấn đề: Response quá lớn
           → Fix: Compression (brotli/gzip), pagination, reduce payload
```

### 3.3 — Call Tree, Bottom-Up, Flame Chart — 3 Views

```javascript
// Ví dụ code để trace:
function handleSubmit(event) {
  event.preventDefault();
  const data = collectFormData();     // 5ms
  const validated = validateData(data); // 2ms
  sendToServer(validated);            // 0.1ms (async, không count)
}

function collectFormData() {
  // gọi 10 functions nhỏ
  return { name: getNameField(), email: getEmailField(), ... }
}
```

**Flame Chart view** (top-down, timeline):

```
handleSubmit ──────────────────────── 7ms
  collectFormData ──────────────── 5ms
    getNameField ─── 1ms
    getEmailField ─── 0.8ms
    ... (8 functions khác)
  validateData ──── 2ms
```

→ Hiện **call hierarchy theo time**. Dễ thấy "đường đi" của execution.

**Call Tree view** (top-down, aggregated):

```
handleSubmit        7ms  100%
  collectFormData   5ms   71%
    getNameField    1ms   14%
    getEmailField   0.8ms 11%
  validateData      2ms   29%
```

→ Hiện **% thời gian** mỗi function chiếm. Dễ thấy bottleneck.

**Bottom-Up view** (bottom-up, self time):

```
Function            Self Time   Total Time
getNameField        1ms         1ms
getEmailField       0.8ms       0.8ms
validateData        2ms         2ms
collectFormData     0.2ms       5ms     ← Self time nhỏ
handleSubmit        0.1ms       7ms     ← Self time rất nhỏ
```

**Self Time** = thời gian function tự chạy, không tính children.

→ **Bottom-Up là view quan trọng nhất để tìm bottleneck thật.** Function có Self Time cao = chính nó chậm (không phải gọi function khác chậm).

### 3.5 — Phân Tích Long Tasks

**Long Task** = bất kỳ task nào > 50ms trên main thread.

Tại sao 50ms? Để đạt 60fps, mỗi frame phải complete trong 16.67ms. Browser cần ~6ms cho rendering. Còn ~10ms cho JS. Nhưng thực tế task dài hơn 50ms mới gây noticeable jank.

```javascript
// PATTERN: Tìm và fix long tasks

// TRƯỚC (long task 200ms):
function processLargeDataset(items) {
  // Chạy 200ms liên tục, block main thread
  return items.map((item) => heavyTransform(item)); // 1000 items × 0.2ms
}

// SAU (chunked, yield to browser):
async function processLargeDataset(items) {
  const results = [];

  for (let i = 0; i < items.length; i++) {
    results.push(heavyTransform(items[i]));

    // Mỗi 50 items, yield cho browser
    // Browser có cơ hội: handle input, render frame, run microtasks
    if (i % 50 === 0) {
      await scheduler.yield();
      // Hoặc: await new Promise(resolve => setTimeout(resolve, 0));
    }
  }

  return results;
}

// Trong Flame Chart:
// TRƯỚC: [████████████████████████ 200ms long task ████████████████]
//                                                  ↑ RED outline
// SAU:   [████ 10ms][yield][████ 10ms][yield][████ 10ms]...
//         Browser có thể respond to input giữa các chunks
```

## Phần 4 — Memory Panel: Tìm Memory Leaks

### 4.1 — V8 Heap: Ôn Lại Cơ Chế

```
V8 Heap structure:
┌─────────────────────────────────────────────────────────┐
│                     V8 Heap                             │
│  ┌──────────────────┐  ┌──────────────────────────────┐ │
│  │   Young Gen      │  │        Old Gen               │ │
│  │  (Scavenger GC)  │  │     (Mark & Sweep GC)        │ │
│  │                  │  │                              │ │
│  │  New objects     │  │  Objects survived Young Gen  │ │
│  │  (~1-8MB)        │  │  (no size limit)             │ │
│  │                  │  │                              │ │
│  │  Minor GC:       │  │  Major GC:                   │ │
│  │  frequent,       │  │  less frequent, but          │ │
│  │  very fast       │  │  pauses longer               │ │
│  └──────────────────┘  └──────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘

Memory leak = object không được dùng nữa nhưng vẫn có reference
→ GC không thể collect
→ Object stuck trong Old Gen
→ Heap grow dần theo thời gian
```

### 4.2 — Heap Snapshot: Kỹ Thuật Đọc

```
Memory panel → "Take heap snapshot"

Snapshot chứa:
  - Tất cả objects trong heap tại thời điểm đó
  - Size của từng object (Shallow Size + Retained Size)
  - Reference graph (object nào giữ object nào)

Hai concepts quan trọng:

SHALLOW SIZE:
  Memory object đó tự chiếm
  Một array với 1000 references: shallow size nhỏ (chỉ array header + pointer slots)

RETAINED SIZE:
  Tổng memory sẽ được giải phóng nếu object này bị GC
  = Shallow size + tất cả objects chỉ được reachable qua object này

  Đây là con số quan trọng để tìm leak:
  Object có Retained Size lớn = đang giữ nhiều thứ khác không thể release
```

**Workflow tìm memory leak:**

```
STEP 1: Take snapshot BASELINE (trước khi thực hiện action)
STEP 2: Thực hiện action nghi ngờ gây leak (mở modal, navigate, v.v.)
STEP 3: Close/undo action
STEP 4: Force GC (bin icon trong Memory panel)
STEP 5: Take snapshot 2
STEP 6: So sánh: dropdown → "Comparison" → chọn snapshot 1

Kết quả comparison view:
  # New: Objects mới xuất hiện (chưa có trong snapshot 1)
  # Deleted: Objects đã bị GC
  # Delta: New - Deleted

Nếu sau khi close modal mà vẫn còn # New objects:
  → Những objects đó là candidates của leak
  → Click vào object → xem "Retainers" (ai đang giữ object này)
```

### 4.3 — Memory Leak Patterns Phổ Biến

**Pattern 1: Event listener không được remove**

```javascript
// ❌ LEAK
class DataVisualizer extends React.Component {
  componentDidMount() {
    // Add listener khi component mount
    window.addEventListener('resize', this.handleResize);
    document.addEventListener('keydown', this.handleKeydown);
  }

  // QUÊN componentWillUnmount
  // → Component unmount nhưng listeners vẫn tồn tại
  // → Listeners giữ reference tới component instance
  // → Component không bị GC dù đã unmount
  // → Mỗi lần mount/unmount = 1 leak thêm

  handleResize = () => {
    /* update chart */
  };
}

// ✅ FIX
class DataVisualizer extends React.Component {
  componentDidMount() {
    window.addEventListener('resize', this.handleResize);
  }

  componentWillUnmount() {
    // PHẢI remove với cùng reference function
    window.removeEventListener('resize', this.handleResize);
    // Lý do cần cùng reference:
    // window.removeEventListener so sánh function reference
    // → Arrow function trong class field: OK (same reference)
    // → Inline function: KHÔNG OK (new function mỗi lần)
  }
}

// React hooks equivalent:
useEffect(() => {
  window.addEventListener('resize', handleResize);
  return () => {
    window.removeEventListener('resize', handleResize); // Cleanup function
  };
}, []);
```

**Pattern 2: Closure giữ large data**

```javascript
// ❌ LEAK
function setupAnalytics() {
  // largeDataset: 50MB array được fetch về để build report
  const largeDataset = fetchHugeDataset(); // 50MB

  // processedSummary chỉ cần 1KB data
  const processedSummary = computeSummary(largeDataset);

  // Closure này capture toàn bộ scope
  // → largeDataset bị giữ trong memory miễn là handler còn sống
  document.getElementById('export-btn').addEventListener('click', () => {
    // Chỉ dùng processedSummary, nhưng largeDataset vẫn bị capture
    downloadFile(processedSummary);
  });
}

// ✅ FIX: Giải phóng reference sau khi không cần
function setupAnalytics() {
  const largeDataset = fetchHugeDataset();
  const processedSummary = computeSummary(largeDataset);

  // largeDataset không còn trong scope của closure
  setupExportButton(processedSummary);
}

function setupExportButton(summary) {
  // Closure này chỉ capture summary (1KB), không capture largeDataset
  document.getElementById('export-btn').addEventListener('click', () => {
    downloadFile(summary);
  });
}
```

**Pattern 3: Detached DOM nodes**

```javascript
// ❌ LEAK
let cachedNodes = [];

function buildList(items) {
  const container = document.getElementById('list');
  container.innerHTML = ''; // Clear old content

  // Nodes cũ bị remove khỏi DOM nhưng...
  // Nếu chúng ta giữ reference:
  cachedNodes.push(...container.childNodes); // ← các nodes này đã detach

  // Build new list
  items.forEach((item) => {
    const li = document.createElement('li');
    li.textContent = item.name;
    container.appendChild(li);
  });
}

// cachedNodes giữ detached DOM nodes → không GC được
// Detached nodes vẫn có full DOM subtree attached
// → Potential MB of leaked memory với large lists

// Trong DevTools Heap Snapshot:
// Search "Detached" → hiện tất cả detached DOM nodes còn trong memory
// → Đây là quick way để find DOM leaks
```

**Pattern 4: Forgotten timers/intervals**

```javascript
// ❌ LEAK
function startPolling(userId) {
  const interval = setInterval(async () => {
    const data = await fetchUserStatus(userId);
    updateUI(data); // updateUI có thể reference DOM, component state, v.v.
  }, 5000);

  // Không có cách nào stop interval này
  // → interval chạy mãi, giữ reference tới closure
  // → closure giữ reference tới userId, fetchUserStatus, updateUI
}

// ✅ FIX
function startPolling(userId) {
  const interval = setInterval(async () => {
    const data = await fetchUserStatus(userId);
    updateUI(data);
  }, 5000);

  // Return cleanup function
  return () => clearInterval(interval);
}

// Dùng:
const stopPolling = startPolling(userId);
// Khi component unmount hoặc user logout:
stopPolling();

// Trong React:
useEffect(() => {
  const stopPolling = startPolling(userId);
  return stopPolling; // Cleanup tự động khi unmount
}, [userId]);
```

### 4.4 — Allocation Timeline: Track Leak Real-time

```
Memory panel → "Allocation instrumentation on timeline"

→ Hiện KHI NÀO objects được allocate

  2. Perform action (navigate, open modal, process data)
  3. Wait 10-20 seconds
  4. Repeat action 5-10 lần
  5. Stop recording
  6. Observe: grey bars có tích tụ không?
```

## Câu Hỏi Kiểm Tra

**Q1: Tại sao khi mở DevTools, performance measurement có thể không accurate?**

A: DevTools Protocol (CDP) có overhead. CPU sampling profiler của V8 (kích hoạt khi record Performance) inject code vào mỗi function call để capture call stack mỗi 100μs. Điều này làm JS chậm thêm 10-15%. Memory panel giữ thêm references để enable inspection, ảnh hưởng GC behavior. Network panel tạo thêm internal events. Đây là lý do luôn measure performance **closed** DevTools trước để có baseline, sau đó open để investigate.

**Q2: Tại sao `window.removeEventListener('resize', () => handleResize())` không remove được listener?**

A: Vì `removeEventListener` so sánh **function reference** (không phải function code). Mỗi lần viết `() => handleResize()` tạo ra **new function object** với reference khác nhau. Browser so sánh reference, thấy khác → không remove. Fix: lưu function reference vào variable (`this.boundHandler = () => handleResize()`), dùng cùng reference cho add và remove. Class field arrow functions (`handleResize = () => {...}`) tự động có stable reference vì được tạo một lần trong constructor.

**Q3: Điều gì xảy ra trong V8 heap khi có "Detached DOM node" trong Memory snapshot?**

A: Khi `container.innerHTML = ''` hoặc `removeChild()`, browser tách node khỏi live DOM tree (detach). Thông thường, GC sẽ collect node đó vì không còn reachable. Tuy nhiên, nếu JS code giữ reference tới node đó (trong array, closure, variable), node vẫn reachable từ JS heap. V8 không thể GC object còn reachable. Node bị "stranded" — không có trong visible DOM, không được GC, tồn tại trong Old Generation heap. Nguy hiểm vì node DOM có thể có full subtree (children elements, event listeners, style objects) → MB of memory per "detached" node.

**Q4: Đoạn code này gây ra vấn đề gì nhìn thấy trong Performance panel?**

```javascript
const rows = document.querySelectorAll('.table-row');
rows.forEach((row) => {
  const height = row.offsetHeight; // đọc
  row.style.minHeight = height + 'px'; // ghi
});
```

A: **Layout Thrashing**. Mỗi iteration: đọc `offsetHeight` (force browser recalculate layout để trả về accurate value) → ghi style (invalidate layout). Vòng tiếp theo lại đọc → browser phải recalculate lại. N rows = N forced synchronous layouts.

Trong Performance panel: thấy xen kẽ purple (Layout) và yellow (JS) bars nhiều lần trong cùng 1 frame. Nếu nhìn vào "Forced reflow is a likely performance bottleneck" warning. Fix: tách read và write — đọc tất cả heights vào array trước (`const heights = [...rows].map(r => r.offsetHeight)`), sau đó ghi tất cả trong một batch (`rows.forEach((r, i) => r.style.minHeight = heights[i] + 'px')`). Hoặc bọc trong `requestAnimationFrame`.

**Q5: Lighthouse score 90 nhưng users report trang chậm — tại sao và debug như thế nào?**

A: Lighthouse là **lab data** — chạy trên máy development trong controlled conditions. Users có thiết bị yếu hơn, mạng thật (variable latency, packet loss), geographic distance tới server khác, browser extensions, nhiều tabs open, RAM đầy. Gap giữa lab và field là bình thường. Debug: (1) Check **CrUX data** trong Google Search Console → Core Web Vitals report → thấy P75 metric từ real users. (2) Implement **RUM** (Real User Monitoring) với `PerformanceObserver` để capture LCP, INP, CLS từ real sessions. (3) Run Lighthouse với **4x CPU + Slow 3G throttle** để simulate low-end mobile. (4) Dùng **WebPageTest** với real devices ở real locations.

## Câu Hỏi Ôn Tập (NotebookLM)

### Câu 1: Giải thích cơ chế Heap Snapshot trong Chrome DevTools. "Retained Size" khác "Shallow Size" như thế nào, và tại sao Retained Size là metric quan trọng hơn khi tìm memory leak?

**Đáp án:**

Khi bạn click "Take heap snapshot", DevTools thông qua Chrome DevTools Protocol (CDP) yêu cầu V8 dump toàn bộ trạng thái heap tại thời điểm đó. V8 tạo serialized representation của tất cả objects còn sống (reachable từ GC roots), bao gồm: object type, properties, values, và quan trọng nhất là **reference graph** (object nào giữ reference tới object nào).

**Shallow Size** là memory vật lý mà object đó tự chiếm. Một object `{name: "An"}` có shallow size nhỏ — chỉ vài bytes cho header và property slots. Một array `[ref1, ref2, ..., ref1000]` có shallow size nhỏ dù chứa 1000 references.

**Retained Size** là tổng memory sẽ được **giải phóng** nếu object đó bị GC — tức là shallow size của chính nó cộng với shallow size của tất cả objects chỉ reachable qua nó. Nếu array trên là cách duy nhất để reach 1000 objects lớn, retained size của array sẽ rất lớn.

Ví dụ: `EventEmitter` object có shallow size 200 bytes, nhưng nó giữ references tới 50 listener functions, mỗi function là closure giữ reference tới large dataset 1MB. Retained size của `EventEmitter` = 200 bytes + 50 functions + 50MB data = ~50MB. Shallow size gây misleading (chỉ 200 bytes), retained size mới cho thấy thật sự đang giữ bao nhiêu memory.

Khi tìm leak, sort theo Retained Size descending → objects đứng đầu là candidates leak cần investigate trước.

### Câu 2: Giải thích chi tiết "Layout Thrashing" — nó xảy ra khi nào, tại sao gây performance vấn đề, và cách phát hiện trong Performance panel?

**Đáp án:**

**Cơ chế:** Browser maintain một "dirty flag" cho layout. Khi JS thay đổi DOM hoặc CSS (write), browser đánh dấu layout là dirty nhưng **không recalculate ngay** — defer tới cuối frame để batch nhiều changes. Tuy nhiên, khi JS đọc layout properties như `offsetWidth`, `offsetHeight`, `getBoundingClientRect()`, `scrollTop`, browser **phải** trả về giá trị accurate. Nếu layout đang dirty, browser buộc phải **recalculate layout synchronously** ngay lúc đó — forced synchronous layout hay "forced reflow".

**Layout Thrashing xảy ra khi:** Trong một loop hoặc sequential code, xen kẽ read (trigger forced reflow) và write (invalidate layout) nhiều lần:

```javascript
elements.forEach((el) => {
  const h = el.offsetHeight; // READ → forced reflow (layout dirty do write trước)
  el.style.height = h * 2; // WRITE → invalidates layout
  // Vòng tiếp theo: READ lại → forced reflow lần nữa
});
// N elements = N forced reflows trong 1 frame
```

**Tại sao chậm:** Layout là expensive — browser phải traverse DOM tree, tính toán box model cho tất cả elements bị ảnh hưởng (không chỉ element bạn đang modify). Forced synchronous layout không thể defer hay batch → N thrash = N full layout calculations.

**Phát hiện trong Performance panel:**

- Trong Main thread, thấy xen kẽ nhiều purple bars (Layout) và yellow bars (JS) trong cùng một task
- Hover vào Layout bar → tooltip hiện "Forced reflow" warning với call stack trỏ về dòng code gây ra
- Bottom-Up view: `Layout` có high Self Time với nhiều entries
- DevTools cũng hiển thị warning "Forced reflow is a likely performance bottleneck" trong Summary

**Fix:** Tách read và write thành 2 phases riêng biệt. Đọc tất cả values cần thiết → store vào array → sau đó ghi tất cả. Hoặc dùng `requestAnimationFrame` để đảm bảo reads xảy ra ở đầu frame (sau khi browser đã layout) và writes xảy ra batched.

### Câu 3: Chrome DevTools Performance panel có "Flame Chart", "Call Tree", và "Bottom-Up" views. Mỗi view được dùng để làm gì, và trong tình huống nào Bottom-Up là view quan trọng nhất để tìm performance bottleneck?

**Đáp án:**

Cả ba views đều represent cùng một profiling data, nhưng theo các góc nhìn khác nhau:

**Flame Chart** là visualization theo **timeline** (trục x = thời gian). Chiều rộng của bar = thời gian. Stack (chiều cao) = call hierarchy. Bars nằm trên được gọi bởi bars bên dưới. Dùng để: hiểu **sequence of execution**, thấy khi nào functions được gọi, identify một frame cụ thể bị chậm.

**Call Tree** là top-down aggregation. Bắt đầu từ entry points, đi xuống call hierarchy, hiện total time (tự chạy + tất cả children). Dùng để: hiểu **cấu trúc** của execution, thấy paths nào tốn nhiều time nhất theo percentage.

**Bottom-Up** là bottom-up aggregation, hiện **Self Time** — thời gian function tự chạy không kể time của children nó gọi. Sort by Self Time descending → functions đứng đầu là những gì **chính chúng** chậm.

**Bottom-Up quan trọng nhất trong các tình huống này:**

Khi Flame Chart hiện `handleSearch → processQuery → filterResults → sortItems → compareItems` với `compareItems` là leaf function chạy 10.000 lần. Call Tree sẽ hiện `handleSearch` với total time 500ms (cao nhất) vì nó là root — nhưng `handleSearch` tự nó chỉ mất 1ms. Bottom-Up hiện `compareItems` với Self Time 450ms — đây mới là bottleneck thật.

Tương tự, khi nhiều functions khác nhau tất cả gọi cùng một helper function nặng, Bottom-Up aggregate tất cả calls của helper đó lại → thấy total self time cao → tìm được bottleneck. Call Tree không aggregate cross-paths nên khó thấy pattern này. Quy tắc: dùng Flame Chart để **orient và hiểu flow**, dùng Bottom-Up để **tìm bottleneck thật sự**.
