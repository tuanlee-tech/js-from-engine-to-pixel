Phase 1–9 là về cơ chế engine/runtime. Phase 10 là về cơ chế **kiến trúc phần mềm** — tức là điều gì xảy ra với **codebase** (khả năng thay đổi, khả năng test, coupling, cohesion) khi bạn áp dụng hoặc vi phạm các nguyên tắc này. ## 1. SOLID Principles

### S — Single Responsibility Principle (SRP)

#### Cơ chế thật

SRP không phải "mỗi class chỉ làm một việc" — định nghĩa đó quá mơ hồ và dẫn đến over-engineering. Robert Martin định nghĩa chính xác hơn:

> **"A module should have one, and only one, reason to change."**

"Reason to change" = một **actor** (người dùng, business rule, deployment concern) có thể yêu cầu thay đổi. Nếu hai người khác nhau có thể yêu cầu bạn sửa cùng một file vì lý do khác nhau — file đó vi phạm SRP.

Khi vi phạm SRP, cơ chế hỏng như sau:

```
Actor A yêu cầu thay đổi logic X
         ↓
Bạn sửa file chứa cả X và Y
         ↓
Logic Y vô tình bị ảnh hưởng (side effect)
         ↓
Actor B (dùng Y) bị break — không ai báo trước
```

#### Ví dụ production

```typescript
// ❌ VI PHẠM SRP
// File này có 3 reasons to change:
// 1. Product team đổi business rule tính giá
// 2. DBA đổi schema database
// 3. DevOps đổi format log / monitoring

class OrderService {
  async processOrder(order: Order) {
    // Reason 1: Business logic — Product team owns
    const discount = order.items.length > 5 ? 0.1 : 0;
    const total = order.items.reduce((sum, item) => sum + item.price, 0);
    const finalTotal = total * (1 - discount);

    // Reason 2: Persistence — DBA/Backend owns
    await db.query(`INSERT INTO orders (id, total, user_id) VALUES (?, ?, ?)`, [
      order.id,
      finalTotal,
      order.userId,
    ]);
    // Nếu DBA đổi schema → phải sửa file này
    // Nhưng file này cũng chứa business logic → nguy cơ break

    // Reason 3: Observability — DevOps/Platform owns
    console.log(`[ORDER] ${order.id} processed: $${finalTotal}`);
    // Nếu DevOps muốn đổi sang structured logging (JSON) → phải sửa file này
  }
}

// ✅ ÁP DỤNG SRP
// Mỗi class có đúng 1 reason to change

class OrderPricingService {
  // Chỉ Product team có quyền yêu cầu thay đổi file này
  calculateTotal(order: Order): number {
    const discount = order.items.length > 5 ? 0.1 : 0;
    const subtotal = order.items.reduce((sum, item) => sum + item.price, 0);
    return subtotal * (1 - discount);
  }
}

class OrderRepository {
  // Chỉ DBA/Backend có quyền yêu cầu thay đổi file này
  async save(order: Order, total: number): Promise<void> {
    await db.query(`INSERT INTO orders (id, total, user_id) VALUES (?, ?, ?)`, [
      order.id,
      total,
      order.userId,
    ]);
  }
}

class OrderLogger {
  // Chỉ DevOps/Platform có quyền yêu cầu thay đổi file này
  logProcessed(orderId: string, total: number): void {
    logger.info({ event: 'order_processed', orderId, total });
  }
}

class OrderService {
  // Orchestrator — chỉ thay đổi khi workflow tổng thể thay đổi
  constructor(
    private pricing: OrderPricingService,
    private repo: OrderRepository,
    private logger: OrderLogger,
  ) {}

  async processOrder(order: Order): Promise<void> {
    const total = this.pricing.calculateTotal(order);
    await this.repo.save(order, total);
    this.logger.logProcessed(order.id, total);
  }
}
```

**Tại sao điều này quan trọng ở production:**

- Khi `OrderRepository` thay đổi (migrate sang Prisma, đổi schema), bạn chỉ sửa 1 file
- Unit test `OrderPricingService` không cần mock database
- 3 team khác nhau có thể làm việc trên 3 file đó song song, không conflict

### O — Open/Closed Principle (OCP)

#### Cơ chế thật

> **"Software entities should be open for extension, but closed for modification."**

Nghĩa thực: Khi có requirement mới, bạn thêm code mới thay vì sửa code đang chạy tốt. Code đang chạy tốt và đã được test — sửa nó là rủi ro.

Cơ chế vi phạm điển hình: **switch/if-else phân nhánh theo type** — mỗi lần thêm type mới, phải sửa tất cả những nơi có switch đó.

```typescript
// ❌ VI PHẠM OCP
// Mỗi lần thêm payment method mới → phải sửa file này
// File này đang "closed for extension, open for modification" — ngược lại

type PaymentMethod = 'credit_card' | 'paypal' | 'crypto'; // Thêm 'bank_transfer' → sửa type
// và tất cả switch bên dưới

function processPayment(method: PaymentMethod, amount: number): Promise<void> {
  switch (
    method // Thêm method mới → phải sửa đây
  ) {
    case 'credit_card':
      return processCreditCard(amount);
    case 'paypal':
      return processPaypal(amount);
    case 'crypto':
      return processCrypto(amount);
    // Thêm 'bank_transfer' → thêm case ở đây, và ở mọi switch khác
  }
}

function getPaymentFee(method: PaymentMethod, amount: number): number {
  switch (
    method // Phải sửa đây nữa
  ) {
    case 'credit_card':
      return amount * 0.029;
    case 'paypal':
      return amount * 0.034;
    case 'crypto':
      return 0;
  }
}

// ✅ ÁP DỤNG OCP
// Thêm payment method mới = thêm 1 class mới, không sửa class nào cũ

interface PaymentProcessor {
  process(amount: number): Promise<PaymentResult>;
  calculateFee(amount: number): number;
}

class CreditCardProcessor implements PaymentProcessor {
  async process(amount: number): Promise<PaymentResult> {
    return stripe.charge(amount);
  }
  calculateFee(amount: number): number {
    return amount * 0.029;
  }
}

class PaypalProcessor implements PaymentProcessor {
  async process(amount: number): Promise<PaymentResult> {
    return paypal.createPayment(amount);
  }
  calculateFee(amount: number): number {
    return amount * 0.034;
  }
}

// Thêm bank transfer: tạo file mới, không sửa gì cũ ✅
class BankTransferProcessor implements PaymentProcessor {
  async process(amount: number): Promise<PaymentResult> {
    return bankApi.initTransfer(amount);
  }
  calculateFee(amount: number): number {
    return 0; // Free
  }
}

// PaymentService không đổi khi thêm method mới
class PaymentService {
  constructor(private processors: Map<string, PaymentProcessor>) {}

  async processPayment(method: string, amount: number): Promise<PaymentResult> {
    const processor = this.processors.get(method);
    if (!processor) throw new Error(`Unknown payment method: ${method}`);
    return processor.process(amount);
  }
}
```

**Trong React context:** OCP hiện ra rõ nhất ở component design.

```typescript
// ❌ Button component vi phạm OCP
function Button({ variant }: { variant: 'primary' | 'danger' | 'ghost' }) {
  // Thêm variant mới → sửa component này
  const styles = {
    primary: 'bg-blue-500 text-white',
    danger:  'bg-red-500 text-white',
    ghost:   'bg-transparent border',
  }[variant];

  return <button className={styles} />;
}

// ✅ OCP compliant — extend qua className/composition
function Button({ className, children, ...props }: ButtonProps) {
  return (
    <button
      className={cn('base-button-styles', className)} // caller mở rộng, không sửa Button
      {...props}
    >
      {children}
    </button>
  );
}

// Caller thêm variant mới mà không sửa Button
<Button className="bg-blue-500 text-white">Primary</Button>
<Button className="bg-red-500 text-white">Danger</Button>
<Button className="bg-purple-500 text-white">New variant</Button>
```

### L — Liskov Substitution Principle (LSP)

#### Cơ chế thật

> **Nếu S là subtype của T, thì objects của type T có thể được thay thế bằng objects của type S mà không làm thay đổi tính đúng đắn của chương trình.**

Nôm na: Subclass phải có thể được dùng thay thế cho superclass mà không gây surprise. Violation xảy ra khi:

- Subclass throws exception mà superclass không throw
- Subclass strengthens precondition (yêu cầu input chặt hơn)
- Subclass weakens postcondition (output kém đảm bảo hơn)

```typescript
// ❌ VI PHẠM LSP — Classic Rectangle/Square problem
class Rectangle {
  constructor(
    protected width: number,
    protected height: number,
  ) {}

  setWidth(w: number) {
    this.width = w;
  }
  setHeight(h: number) {
    this.height = h;
  }
  getArea() {
    return this.width * this.height;
  }
}

class Square extends Rectangle {
  // Square muốn width === height luôn
  // → phải override cả hai setter → thay đổi behavior
  setWidth(w: number) {
    this.width = w;
    this.height = w; // ← unexpected side effect!
  }
  setHeight(h: number) {
    this.height = h;
    this.width = h; // ← unexpected side effect!
  }
}

// Code dùng Rectangle — không biết gì về Square
function doubleWidth(rect: Rectangle) {
  const originalHeight = rect.getArea() / rect.getWidth?.() || rect['height'];
  rect.setWidth(10);
  rect.setHeight(5);
  // Với Rectangle: area = 10 * 5 = 50 ✓
  // Với Square:    area = 5 * 5 = 25  ✗ — không ai mong đợi điều này
  console.assert(rect.getArea() === 50, 'Area phải là 50');
}

const rect = new Rectangle(2, 3);
doubleWidth(rect); // Passes ✓

const square = new Square(5, 5);
doubleWidth(square); // FAILS ✗ — LSP violated
```

**LSP trong TypeScript thực tế — interface contracts:**

```typescript
interface Cache<T> {
  get(key: string): T | undefined;
  set(key: string, value: T, ttl?: number): void;
  delete(key: string): void;
}

class InMemoryCache<T> implements Cache<T> {
  private store = new Map<string, T>();

  get(key: string): T | undefined {
    return this.store.get(key); // Luôn sync, luôn trả value hoặc undefined
  }

  set(key: string, value: T): void {
    this.store.set(key, value); // Luôn thành công
  }

  delete(key: string): void {
    this.store.delete(key);
  }
}

// ❌ Vi phạm LSP — RedisCache "strengthens postcondition" bằng cách throw
class RedisCache<T> implements Cache<T> {
  get(key: string): T | undefined {
    // ← Nếu Redis down, throw Error
    // Consumer của Cache<T> không cần try/catch vì InMemoryCache không throw
    // Nhưng RedisCache throw → Consumer bị surprise
    if (!this.redis.isConnected) throw new Error('Redis disconnected');
    return this.redis.get(key);
  }
}

// ✅ LSP compliant — handle failure gracefully, không change contract
class RedisCache<T> implements Cache<T> {
  get(key: string): T | undefined {
    try {
      return this.redis.isConnected ? this.redis.get(key) : undefined;
      // Trả undefined khi lỗi — giống behavior của InMemoryCache khi key không tồn tại
    } catch {
      return undefined; // Fallback, không throw — contract intact
    }
  }
}
```

### I — Interface Segregation Principle (ISP)

#### Cơ chế thật

> **Clients should not be forced to depend on interfaces they don't use.**

Vi phạm ISP xảy ra khi interface quá lớn (fat interface). Client phải implement hoặc mock toàn bộ interface dù chỉ cần 1-2 method. Hậu quả:

- Test phức tạp hơn (phải mock cả interface to)
- Thay đổi method không dùng vẫn trigger recompile/re-test
- Coupling cao hơn mức cần thiết


### D — Dependency Inversion Principle (DIP)

#### Cơ chế thật

> **High-level modules should not depend on low-level modules. Both should depend on abstractions.**

"High-level" = business logic. "Low-level" = implementation details (database, HTTP, file system, email service).

Vi phạm DIP: Business logic import trực tiếp từ implementation. Khi implementation thay đổi, business logic phải sửa.

```
❌ WITHOUT DIP:
OrderService ──depends on──► PostgresOrderRepo (concrete)
                              ↑
                              Thay sang MongoDB → sửa OrderService

✅ WITH DIP:
OrderService ──depends on──► IOrderRepository (abstract)
                                     ↑
                PostgresOrderRepo ───┘ (cả hai depend on abstraction)
                MongoOrderRepo ───────┘
```

```typescript
// ❌ VI PHẠM DIP — business logic coupled với infrastructure
import { PrismaClient } from '@prisma/client';
import { Resend } from 'resend';

class UserRegistrationService {
  // High-level module import trực tiếp low-level implementation
  private db = new PrismaClient(); // ← concrete dependency
  private email = new Resend(API_KEY); // ← concrete dependency

  async register(data: RegisterDTO): Promise<User> {
    // Business logic bị coupled với Prisma schema
    const user = await this.db.user.create({
      data: { email: data.email, passwordHash: hash(data.password) },
    });

    // Business logic bị coupled với Resend API
    await this.email.emails.send({
      from: 'noreply@app.com',
      to: user.email,
      subject: 'Welcome!',
      html: `<p>Hi ${user.email}</p>`,
    });

    return user;
  }
}
// Test: phải mock Prisma VÀ Resend — phức tạp và brittle
// Swap sang Drizzle/Mongoose → phải sửa business logic
// Swap sang SendGrid → phải sửa business logic

// ✅ ÁP DỤNG DIP

// Abstractions (không phụ thuộc vào ai)
interface IUserRepository {
  create(data: CreateUserData): Promise<User>;
  findByEmail(email: string): Promise<User | null>;
}

interface IEmailService {
  sendWelcome(to: string, name: string): Promise<void>;
}

// High-level module depend vào abstraction
class UserRegistrationService {
  constructor(
    private userRepo: IUserRepository, // ← abstract
    private emailService: IEmailService, // ← abstract
  ) {}

  async register(data: RegisterDTO): Promise<User> {
    const existing = await this.userRepo.findByEmail(data.email);
    if (existing) throw new Error('Email already registered');

    const user = await this.userRepo.create({
      email: data.email,
      passwordHash: hash(data.password),
    });

    await this.emailService.sendWelcome(user.email, user.name);
    return user;
  }
}

// Low-level modules implement abstraction
class PrismaUserRepository implements IUserRepository {
  constructor(private prisma: PrismaClient) {}

  create(data: CreateUserData): Promise<User> {
    return this.prisma.user.create({ data });
  }

  findByEmail(email: string): Promise<User | null> {
    return this.prisma.user.findUnique({ where: { email } });
  }
}

class ResendEmailService implements IEmailService {
  constructor(private resend: Resend) {}

  async sendWelcome(to: string, name: string): Promise<void> {
    await this.resend.emails.send({
      from: 'noreply@app.com',
      to,
      subject: 'Welcome!',
      html: `<p>Hi ${name}, welcome!</p>`,
    });
  }
}

// Test: mock cực đơn giản
class MockUserRepository implements IUserRepository {
  private users: User[] = [];
  create(data: CreateUserData): Promise<User> {
    const user = { id: 'test-id', ...data } as User;
    this.users.push(user);
    return Promise.resolve(user);
  }
  findByEmail(email: string): Promise<User | null> {
    return Promise.resolve(this.users.find((u) => u.email === email) ?? null);
  }
}

// Wiring tại composition root (main.ts / DI container)
const service = new UserRegistrationService(
  new PrismaUserRepository(prisma),
  new ResendEmailService(resend),
);
```

**Tóm tắt SOLID relationships:**

```
S — Mỗi module có 1 reason to change (actor-based)
O — Extend qua thêm code mới, không sửa code cũ
L — Subtype thay thế supertype không gây surprise
I — Interface nhỏ, focused — không force implement thứ không dùng
D — Business logic depend vào interface, không depend vào implementation
```

## 2. DRY, YAGNI, KISS

### DRY — Don't Repeat Yourself

#### Cơ chế thật

DRY thường bị hiểu sai là "không copy-paste code". Định nghĩa đúng của Andy Hunt & Dave Thomas:

> **"Every piece of knowledge must have a single, unambiguous, authoritative representation within a system."**

"Knowledge" — không phải "code". Hai đoạn code trông giống nhau nhưng represent different knowledge thì KHÔNG vi phạm DRY khi tách ra.

**Khi nào không nên DRY — Sandro Mancuso's Rule of Three:**

- Lần 1: viết thẳng
- Lần 2: thấy giống, ghi chú, nhưng chưa abstract
- Lần 3: abstract

Vì lý do: "incidental duplication" — hai thứ trông giống nhau nhưng thực ra không cùng knowledge. Nếu abstract sớm, khi một thứ thay đổi, bạn bị forced thay đổi cả thứ kia.

```typescript
// ❌ OVER-DRY — abstract nhầm "incidental duplication"

// Hai hàm này trông giống nhau:
function validateUserAge(age: number): boolean {
  return age >= 18 && age <= 120;
}

function validateProductQuantity(quantity: number): boolean {
  return quantity >= 1 && quantity <= 1000;
}

// Engineer "thông minh" DRY nó thành:
function validateRange(value: number, min: number, max: number): boolean {
  return value >= min && value <= max;
}

// Bây giờ: Product team muốn đổi max quantity theo subscription tier
// Nhưng validateRange đang chứa cả user age logic
// → Hai business rules khác nhau bị entangled trong 1 function
// → "Coupling by coincidence"

// ✅ DRY đúng chỗ — chỉ abstract khi same knowledge

// Cùng một validation rule → DRY là đúng
function formatCurrency(amount: number, currency = 'USD'): string {
  return new Intl.NumberFormat('en-US', { style: 'currency', currency }).format(
    amount,
  );
}
// Dùng ở cart, invoice, receipt, refund — tất cả cùng knowledge "format tiền tệ"
// Nếu đổi locale → đổi 1 chỗ ✓
```

**DRY trong React — component extraction:**


### YAGNI — You Aren't Gonna Need It

#### Cơ chế thật

> **Đừng implement thứ gì bạn chưa cần ngay bây giờ, dựa trên dự đoán tương lai.**

YAGNI không phải "đừng nghĩ về tương lai". YAGNI là: **chi phí implement thứ không cần ngay > chi phí refactor sau nếu thật sự cần.**

Công thức của Ron Jeffries:

```
Cost of implementing now + Cost of maintaining unused code
    vs.
Cost of implementing later IF actually needed
```

Trường hợp thứ 2 thường thắng vì: (a) thường không cần, (b) khi cần bạn hiểu requirements rõ hơn.

```typescript
// ❌ VI PHẠM YAGNI — "phòng khi sau này cần"

interface UserRepository {
  findById(id: string): Promise<User>;
  findByEmail(email: string): Promise<User>;

  // "Sau này chắc sẽ cần filtering nâng cao"
  findWithFilters(filters: {
    role?: Role;
    createdAfter?: Date;
    createdBefore?: Date;
    isActive?: boolean;
    emailVerified?: boolean;
    subscriptionTier?: SubscriptionTier;
```

### KISS — Keep It Simple, Stupid

#### Cơ chế thật

> **Hầu hết các hệ thống hoạt động tốt nhất khi chúng đơn giản hơn là phức tạp. Do đó, sự đơn giản nên là mục tiêu thiết kế chính.**

"Stupid" trong KISS không phải chửi — ý là "đơn giản đến mức ngớ ngẩn" (simple to the point of stupidity).

Complexity cost: mỗi lớp abstraction thêm vào là cognitive load cho người đọc. Code được đọc nhiều hơn được viết (tỷ lệ 10:1). Code đơn giản = ít bugs, ít meeting, ít onboarding time.

```typescript
// ❌ OVER-ENGINEERED — KISS violation

// Cần: lấy user từ DB theo ID, return undefined nếu không tìm thấy
// Reality: 1 function, 3 dòng

// "Engineer sáng tạo" làm:
abstract class BaseEntityFetcher<T, ID> {
  abstract getRepository(): Repository<T>;
  abstract transformResult(raw: unknown): T;

  async fetchById(id: ID): Promise<Option<T>> {
    const result = await this.getRepository().findOne(id);
    return result ? Some(this.transformResult(result)) : None;
  }
}

class UserFetcher extends BaseEntityFetcher<User, string> {
  getRepository() {
    return userRepository;
  }
  transformResult(raw: unknown): User {
    return raw as User;
  }
}

const fetcher = new UserFetcher();
const result = await fetcher.fetchById(userId);
const user = result.fold(
  (u) => u, // Some case
  () => {
    throw new UserNotFoundError(userId);
  }, // None case
);

// Đọc code này mất 5 phút để hiểu. Có 4 abstraction layers.

// ✅ KISS — đúng level of complexity cho problem này
async function getUserById(id: string): Promise<User | undefined> {
  return userRepository.findOne(id) ?? undefined;
}

// Đọc code này mất 5 giây. Clear. Done.
```

**Khi nào complexity là justified:**

```typescript
// Complexity này là JUSTIFIED — không phải KISS violation
// Vì problem thật sự phức tạp: distributed cache với fallback, invalidation, telemetry

class UserCacheService {
  private readonly CACHE_PREFIX = 'user:';
  private readonly DEFAULT_TTL = 3600;

  constructor(
    private redis: Redis,
    private userRepo: IUserRepository,
    private telemetry: ITelemetry,
  ) {}

  async getUser(id: string): Promise<User | null> {
    const cacheKey = `${this.CACHE_PREFIX}${id}`;

    // Try cache first
    const cached = await this.redis.get(cacheKey);
    if (cached) {
      this.telemetry.increment('cache.hit', { entity: 'user' });
      return JSON.parse(cached);
    }

    // Cache miss — fetch from DB
    this.telemetry.increment('cache.miss', { entity: 'user' });
    const user = await this.userRepo.findById(id);

    if (user) {
      await this.redis.setex(cacheKey, this.DEFAULT_TTL, JSON.stringify(user));
    }

    return user;
  }
}
// Complexity này là để giải quyết problem thật: performance + observability
// Không thể đơn giản hơn mà vẫn đúng
```

**KISS vs DRY vs YAGNI — conflict resolution:**

```
KISS và DRY conflict: Code bị duplicate nhưng abstracting nó tạo ra complexity
→ Ưu tiên KISS — duplication ít tệ hơn wrong abstraction

KISS và YAGNI conflict: Cần implement thêm complexity ngay để tránh rewrite sau
→ Ưu tiên YAGNI — nhưng nếu refactor sau có cost cao (breaking API), làm ngay
```

## 3. Composition over Inheritance

#### Cơ chế thật

Inheritance tạo ra coupling chặt (tight coupling) giữa parent và child. Khi parent thay đổi, tất cả children bị ảnh hưởng — ngay cả khi chúng không liên quan đến thay đổi đó. Đây gọi là **fragile base class problem.**

Composition (HAS-A) vs Inheritance (IS-A):

- Inheritance: "Một Dog IS-A Animal"
- Composition: "Một Robot HAS-A Motor, HAS-A Camera, HAS-A AI Brain"

```typescript
// ❌ INHERITANCE CHAIN — fragile base class problem

class Animal {
  breathe() {
    console.log('breathing');
  }
  eat(food: string) {
    console.log(`eating ${food}`);
  }
}

class LivingCreature extends Animal {
  sleep() {
    console.log('sleeping');
  }
}

class Vertebrate extends LivingCreature {
  hasSpine = true;
}

class Mammal extends Vertebrate {
  regulateTemperature() {
    console.log('warm-blooded');
  }
}

class Dog extends Mammal {
  bark() {
    console.log('woof');
  }
}

// Vấn đề: Muốn tạo RobotDog
// - RobotDog không breathe, không eat, không sleep
// - Nhưng có thể "bark", move, sense
// → Không thể extend Animal hierarchy
// → Phải duplicate hoặc contort class hierarchy

// ❌ Giải pháp tệ: force fit vào hierarchy
class RobotDog extends Dog {
  breathe() {
    /* overide to do nothing — vi phạm LSP */
  }
  eat(food: string) {
    /* override to throw — vi phạm LSP */
  }
}

// ✅ COMPOSITION — behaviors như building blocks

// Behaviors — independent, composable
const canBreathe = () => ({
  breathe: () => console.log('breathing'),
});

const canEat = (diet: string) => ({
  eat: (food: string) => console.log(`eating ${food} (${diet} diet)`),
});

const canBark = () => ({
  bark: () => console.log('woof!'),
});

const canMove = (speed: number) => ({
  move: (direction: string) =>
    console.log(`moving ${direction} at ${speed}mph`),
});

const hasSensors = () => ({
  sense: (target: string) => console.log(`detecting ${target}`),
});

// Compose dogs from behaviors
function createDog(name: string) {
  return {
    name,
    ...canBreathe(),
    ...canEat('omnivore'),
    ...canBark(),
    ...canMove(15),
  };
}

function createRobotDog(name: string, speedMph: number) {
  return {
    name,
    // Không có breathe, eat — vì không cần
    ...canBark(), // Chung với real dog
    ...canMove(speedMph),
    ...hasSensors(), // Chỉ robot có
  };
}

const rex = createDog('Rex');
const k9 = createRobotDog('K9', 30);

rex.breathe(); // ✓
k9.bark(); // ✓
// k9.eat(...)  ← TypeScript error: method không tồn tại ✓
```

```typescript
// ❌ Inheritance trong React — anti-pattern (React docs khuyến cáo không dùng)
```

## 4. Separation of Concerns (SoC)

#### Cơ chế thật

> **Mỗi module, class, function nên chịu trách nhiệm về một concern (khía cạnh) duy nhất của chức năng.**

SoC và SRP có liên quan nhưng khác nhau:

- SRP nói về "reason to change" (organizational/actor-based)
- SoC nói về "domain of responsibility" (functional separation)

Concerns phổ biến cần tách:

- **Presentation** — Hiển thị data như thế nào
- **Business Logic** — Tính toán, validation, rules
- **Data Access** — Fetch, persist, cache
- **Infrastructure** — HTTP, logging, monitoring, config

```typescript
// ❌ VI PHẠM SOC — React component làm tất cả mọi thứ
function UserProfilePage({ userId }: Props) {
  const [user, setUser] = useState<User | null>(null);
  const [isLoading, setLoading] = useState(false);
  const [error, setError] = useState<Error | null>(null);

  // ← DATA ACCESS concern trộn vào presentation
  useEffect(() => {
    setLoading(true);
    fetch(`/api/users/${userId}`)
      .then(r => r.json())
      .then(data => {
        // ← BUSINESS LOGIC concern trộn vào presentation
        if (data.role === 'admin' && !data.permissions.includes('profile:view')) {
          throw new Error('Insufficient permissions');
        }
        setUser(data);
      })
      .catch(setError)
      .finally(() => setLoading(false));
  }, [userId]);

  // ← BUSINESS LOGIC: tính display name
  const displayName = user
    ? `${user.firstName} ${user.lastName}` + (user.isPremium ? ' ⭐' : '')
    : '';

  // ← PRESENTATION
  if (isLoading) return <Spinner />;
  if (error) return <ErrorMessage error={error} />;
  return <div>{displayName}</div>;
}

// ✅ ÁP DỤNG SOC — tách rõ từng concern

// CONCERN 1: Data Access — custom hook
function useUser(userId: string) {
  return useQuery({
    queryKey: ['user', userId],
    queryFn: () => userApi.getById(userId),
  });
}

// CONCERN 2: Business Logic — pure function, testable
function formatDisplayName(user: User): string {
  const base = `${user.firstName} ${user.lastName}`;
  return user.isPremium ? `${base} ⭐` : base;
}

function canViewProfile(user: User): boolean {
  return user.role !== 'admin' || user.permissions.includes('profile:view');
}

// CONCERN 3: Presentation — chỉ biết render, không biết fetch
function UserProfile({ user }: { user: User }) {
  const displayName = formatDisplayName(user); // Business logic từ nơi khác
  return <div className="user-profile">{displayName}</div>;
}

// CONCERN 4: Composition — orchestrate các concerns
function UserProfilePage({ userId }: Props) {
  const { data: user, isLoading, error } = useUser(userId);

  if (isLoading) return <Spinner />;
  if (error)     return <ErrorMessage error={error} />;
  if (!user || !canViewProfile(user)) return <AccessDenied />;

  return <UserProfile user={user} />;
}
```

```typescript
// Controller — HTTP concern chỉ
class UserController {
  constructor(private userService: UserService) {}

  async createUser(req: Request, res: Response) {

    const dto = CreateUserDTO.parse(req.body); // Validate shape

    // ← HTTP response

class UserService {
  constructor(
    private userRepo: IUserRepository,
    private emailService: IEmailService,
    private eventBus: IEventBus,
  ) {}

    // ← Business rules — không biết HTTP, không biết DB schema
    const existing = await this.userRepo.findByEmail(dto.email);
    if (existing) throw new DomainError('EMAIL_TAKEN');

    const passwordHash = await bcrypt.hash(dto.password, 12);

    // Business rule: gửi welcome email sau register

    // Domain event
    await this.eventBus.publish(new UserRegisteredEvent(user.id));

    return user;
  }
}
```

## 5. Single Source of Truth (SSOT)

#### Cơ chế thật

> **Mỗi piece of data chỉ được stored và managed ở một nơi duy nhất. Tất cả các references khác đều derive từ nơi đó.**

Vi phạm SSOT = data được store ở nhiều nơi và có thể diverge. Khi diverge, bạn có **inconsistency bug** — thứ khó debug nhất vì không có error, data chỉ là sai.

```typescript
// ❌ VI PHẠM SSOT — state được lưu ở nhiều nơi

function ShoppingCart() {
  const [cartItems, setCartItems] = useState<CartItem[]>([]);
  const [itemCount, setItemCount] = useState(0); // ← derived from cartItems, nhưng stored separately
  const [totalPrice, setTotalPrice] = useState(0); // ← derived from cartItems, nhưng stored separately

  const addItem = (item: Product) => {
    const newItems = [...cartItems, { product: item, qty: 1 }];
    setCartItems(newItems);

    // Phải manually sync derived state
    setItemCount(newItems.length); // ← Có thể quên dòng này
    setTotalPrice((prev) => prev + item.price); // ← Dễ tính sai

    // Nếu removeItem quên sync itemCount → inconsistency
  };

  const removeItem = (productId: string) => {
    const filtered = cartItems.filter((i) => i.product.id !== productId);
    setCartItems(filtered);
    // BUG: quên update itemCount và totalPrice ← silent inconsistency
  };
}

// ✅ SSOT — single source, derive everything else

function ShoppingCart() {
  // SINGLE SOURCE OF TRUTH: chỉ có cartItems là state thật
  const [cartItems, setCartItems] = useState<CartItem[]>([]);

  // Derived state — tính từ SSOT, không stored separately
  const itemCount = cartItems.length;
  const totalPrice = cartItems.reduce(
    (sum, item) => sum + item.product.price * item.qty,
    0,
  );

  const addItem = (item: Product) => {
    setCartItems((prev) => [...prev, { product: item, qty: 1 }]);
    // itemCount và totalPrice tự động correct — không cần sync
  };

  const removeItem = (productId: string) => {
    setCartItems((prev) => prev.filter((i) => i.product.id !== productId));
    // itemCount và totalPrice tự động correct
  };
}
```

| Nguyên tắc  | Khi vi phạm → Production problem                         |
| ----------- | -------------------------------------------------------- |
| SRP         | File lớn, merge conflicts thường xuyên, team blocker     |
| OCP         | Bug regression mỗi lần thêm feature, vì phải sửa code cũ |
| LSP         | Runtime errors không dự đoán được, khó debug             |
| ISP         | Test setup phức tạp, circular dependency khó thấy        |
| DIP         | Không thể test business logic in isolation (phải có DB)  |
| DRY         | Sửa bug phải sửa nhiều chỗ, miss chỗ → inconsistency     |
| YAGNI       | Dead code, tech debt, onboarding chậm                    |
| KISS        | Bugs ẩn trong abstractions phức tạp, khó debug           |
| Composition | Inheritance hierarchy cứng nhắc, tái sử dụng kém         |
| SoC         | Component test phải setup toàn bộ stack (DB, API, etc.)  |
| SSOT        | Stale data bugs — khó reproduce vì intermittent          |

### Hay gặp trong React codebase

```typescript
// Pattern: "Container/Presenter" — SoC trong React
// Container: data fetching, state management
// Presenter: pure render, no side effects — testable với snapshot

// Vi phạm SoC phổ biến trong React:
// - useEffect trong presentational component để fetch data
// - Business logic trong JSX (`{items.length > 0 && items[0].price * 0.9}`)
// - Direct API calls trong component thay vì qua hook/service

// Pattern: Custom hooks là Composition over Inheritance
// Mỗi hook là một behavior, component compose nhiều hooks
// → Không cần abstract base component

// Vi phạm SSOT phổ biến:
// - useState cho cả total và items trong cart (total là derived)
// - Cùng data trong Redux store và local component state
// - URL params và state không sync (URL là SSOT cho shareable state)
```

**Q1:** Đây là SOLID principle nào bị vi phạm, và tại sao?

```typescript
class NotificationService {
  async sendEmailNotification(userId: string, message: string) {
    const user = await this.db.findUser(userId);
    if (!user.emailVerified) return;

    const html = `<html><body>${message}</body></html>`;
    await sgMail.send({ to: user.email, subject: 'Notification', html });
    await this.db.createNotificationLog({ userId, type: 'email', message });
  }
}
```

**A1:** Vi phạm **SRP** và **DIP**.

- **SRP**: class có 3 reasons to change: (1) Business rule thay đổi (ai được nhận email), (2) Email provider thay đổi (từ SendGrid sang SES), (3) Logging schema thay đổi. Ba actors khác nhau: Product, DevOps/Platform, DBA.
- **DIP**: `sgMail` (concrete) và `this.db` (concrete) được depend trực tiếp từ business logic. Không thể test `sendEmailNotification` mà không mock cả SendGrid API và database thật.

Fix: Tách thành `UserRepository`, `EmailProvider` (interface), `NotificationLogger`, `NotificationService` (orchestrate qua interfaces).

**Q2:** Tại sao đoạn code sau vi phạm LSP, và điều gì sẽ xảy ra ở runtime?

```typescript
interface Logger {
  log(message: string): void;
  error(message: string): void;
}

class ConsoleLogger implements Logger {
  log(message: string): void {
    console.log(message);
  }
  error(message: string): void {
    console.error(message);
  }
}

class SilentLogger implements Logger {
  log(message: string): void {
    /* no-op */
  }
  error(message: string): void {
    throw new Error('Cannot call error on SilentLogger'); // ← Vấn đề ở đây
  }
}

function processOrder(order: Order, logger: Logger) {
  try {
    // ... business logic
    logger.log('Order processed');
  } catch (err) {
    logger.error(`Failed: ${err.message}`); // ← Caller không expect throw
  }
}
```

**A2:** `SilentLogger` vi phạm LSP vì nó **strengthens postcondition** — `Logger.error()` theo contract không throws, nhưng `SilentLogger.error()` throws. Consumer (`processOrder`) viết code dựa trên contract của `Logger`, không có try/catch quanh `logger.error()`. Khi thay `ConsoleLogger` bằng `SilentLogger`, một unhandled exception xuất hiện trong catch block — tức là code xử lý error lại bị lỗi, paradox.

Fix: `SilentLogger.error()` nên no-op hoặc store error silently, không throw — giống behavior của `ConsoleLogger` về mặt "không throw".

**Q3:** Đoạn code React sau vi phạm SSOT ở đâu? Điều gì xảy ra khi user xóa item?

```typescript
function CartPage() {
  const [items, setItems] = useState<CartItem[]>([]);
  const [total, setTotal] = useState(0);
  const [discountedTotal, setDiscountedTotal] = useState(0);

  const removeItem = (id: string) => {
    const removed = items.find(i => i.id === id)!;
    setItems(prev => prev.filter(i => i.id !== id));
    setTotal(prev => prev - removed.price);
    // BUG: quên update discountedTotal
  };

  return (
    <div>
      <div>Total: ${total}</div>
      <div>After discount: ${discountedTotal}</div> {/* ← stale */}
    </div>
  );
}
```

**A3:** Vi phạm SSOT: `total` và `discountedTotal` là **derived state** (tính được từ `items`) nhưng được stored như state riêng. Khi `removeItem` cập nhật `items` và `total` nhưng quên `discountedTotal`, UI hiển thị sai giá trị — silent inconsistency bug không có error message, rất khó debug.

Xảy ra gì: `discountedTotal` hiển thị giá tiền cũ (trước khi xóa item), trong khi `total` đã đúng. User thấy số không khớp nhau → trust issue với app.

Fix:

```typescript
const [items, setItems] = useState<CartItem[]>([]);
// Derive từ SSOT — không bao giờ stale
const total = items.reduce((sum, i) => sum + i.price, 0);
const discount = total > 100 ? 0.1 : 0;
const discountedTotal = total * (1 - discount);
```

### Câu 1: Tại sao DRY không đồng nghĩa với "không copy-paste code", và khi nào duplication là đúng?

**Đáp án:**

DRY — "Don't Repeat Yourself" — nguyên nghĩa gốc là: "Every piece of **knowledge** must have a single, unambiguous, authoritative representation." Từ khóa là **knowledge** (kiến thức, business rule), không phải **code syntax**.

Hai đoạn code có cú pháp giống hệt nhau nhưng có thể represent hai knowledge khác nhau. Ví dụ:

```typescript
function validateUserAge(age: number): boolean {
  return age >= 18 && age <= 120;
}
function validateProductQuantity(qty: number): boolean {
  return qty >= 1 && qty <= 100;
}
```

Hai hàm này trông giống nhau (cùng pattern range check) nhưng represent hai business rules hoàn toàn khác nhau — user age validation thuộc legal/compliance domain, product quantity validation thuộc inventory domain. Nếu "DRY" chúng thành `validateRange(min, max)`, khi business muốn đổi quantity cap theo subscription tier, bạn buộc phải sửa một function đang chứa cả age logic — đây gọi là **coupling by coincidence**.

**Khi nào duplication là đúng:**

1. **Rule of Three:** Lần thứ nhất viết thẳng. Lần thứ hai thấy giống, ghi chú. Lần thứ ba mới abstract — lúc này bạn hiểu pattern đủ để abstract đúng.
2. **Incidental duplication:** Code trông giống nhưng business domain khác → không abstract.
3. **Wrong abstraction worse than duplication:** Sandro Mancuso, Sandi Metz đều viết về điều này — duplication ít tệ hơn wrong abstraction. Wrong abstraction tạo ra coupling ẩn, khó refactor. Duplication thì chỉ cần sửa nhiều chỗ.

Cách phân biệt: nếu hai thứ trông giống nhau thay đổi vì **cùng reason** → DRY là đúng. Nếu chúng có thể thay đổi **độc lập vì lý do khác nhau** → để chúng tách biệt.

### Câu 2: Điều gì xảy ra khi Dependency Inversion Principle bị vi phạm trong context unit testing, và tại sao nó thực sự là vấn đề ở production?

**Đáp án:**

Khi DIP bị vi phạm, business logic import trực tiếp từ concrete implementations:

```typescript
import { PrismaClient } from '@prisma/client';

class UserService {
  private db = new PrismaClient(); // ← concrete dependency

  async register(email: string, password: string): Promise<User> {
    const existing = await this.db.user.findUnique({ where: { email } });
    if (existing) throw new Error('Email taken');
    return this.db.user.create({
      data: { email, passwordHash: hash(password) },
    });
  }
}
```

**Điều xảy ra khi test:**

`UserService` không thể được unit tested mà không có Postgres running. Test phải:

- Spin up real database (chậm, flaky, CI/CD phức tạp)
- Hoặc mock toàn bộ PrismaClient (phức tạp, brittle — mock phải match Prisma API)
- Hoặc không test (nguy hiểm)

Kết quả: Test suite chậm (minutes instead of seconds), developers skip unit tests, bugs reach production.

**Production problems khi không có DIP:**

1. **Swap implementation = rewrite business logic:** Muốn migrate từ Prisma sang Drizzle, hoặc từ Postgres sang MongoDB → phải sửa `UserService` dù business rules không đổi. Mỗi lần sửa là risk.

2. **Feature flag / A/B testing khó:** Muốn dùng Redis cache cho 50% users, Postgres cho 50% → không thể swap implementation ở runtime mà không sửa business logic.

3. **Microservice extraction:** Khi muốn extract `UserService` thành service riêng → nó kéo theo toàn bộ Prisma dependency, không thể move clean.

**Fix với DIP:**

```typescript
interface IUserRepository {
  findByEmail(email: string): Promise<User | null>;
  create(data: CreateUserData): Promise<User>;
}

class UserService {
  constructor(private repo: IUserRepository) {} // ← abstract dependency

  async register(email: string, password: string): Promise<User> {
    const existing = await this.repo.findByEmail(email);
    if (existing) throw new Error('Email taken');
    return this.repo.create({ email, passwordHash: hash(password) });
  }
}

// Test: 5 dòng mock, chạy in-memory, milliseconds
class InMemoryUserRepository implements IUserRepository {
  private users: User[] = [];
  findByEmail(email: string) {
    return Promise.resolve(this.users.find((u) => u.email === email) ?? null);
  }
  create(data: CreateUserData) {
    const user = { id: uuid(), ...data } as User;
    this.users.push(user);
    return Promise.resolve(user);
  }
}

const service = new UserService(new InMemoryUserRepository());
// Unit test chạy mà không cần DB
```

DIP không chỉ là "design pattern" — nó là điều kiện để business logic **independently testable**, **independently deployable**, và **independently changeable**.

### Câu 3: Composition over Inheritance giải quyết vấn đề gì của inheritance, và tại sao React đi theo hướng này (từ class components sang hooks)?

**Đáp án:**

**Fragile Base Class Problem — vấn đề cốt lõi của inheritance:**

Inheritance tạo ra **tight coupling** theo chiều dọc — subclass phụ thuộc vào implementation của superclass, không chỉ interface. Khi superclass thay đổi internal implementation (không phải public API), subclass có thể bị break mà không có compile error.

```
Animal (base) ──── breathe(), eat(), sleep()
     │
     ├── Mammal ──── regulateTemperature()
     │     │
     │     └── Dog ──── bark()
     │           │
     │           └── PoliceDog ──── sniff(), apprehend()
     │
     └── Robot ??  (Robot không breathe, không eat)
                   → Không thể fit vào hierarchy
                   → Phải duplicate hoặc hack
```

Vấn đề:

1. **Gorilla/Banana problem:** Muốn dùng `bark()` phải lấy cả `Animal`, `Mammal`, `Dog` — mang theo behaviors không cần.
2. **Diamond problem:** JS không hỗ trợ multiple inheritance — không thể inherit từ cả `FlyingCreature` lẫn `Mammal`.
3. **Hierarchy không flexible:** Reality thay đổi nhanh hơn hierarchy — robot dog, drone dog, virtual dog không fit vào tree.

**Composition giải quyết:**

```typescript
// Behaviors như LEGO blocks — compose tùy ý
const canBark = () => ({ bark: () => 'woof' });
const canFly = () => ({ fly: () => 'whoosh' });
const canMove = (speed: number) => ({
  move: (dir: string) => `moving ${dir} at ${speed}mph`,
});

const realDog = { ...canBark(), ...canMove(15) };
const robotDog = { ...canBark(), ...canMove(30), ...hasSensors() };
const flyingDog = { ...canBark(), ...canMove(10), ...canFly() };
// Mỗi dog chỉ có behaviors nó cần, không phụ thuộc vào hierarchy
```

**Tại sao React chuyển từ Class sang Hooks:**

React team gặp đúng "fragile base class problem" ở production scale:

1. **Logic reuse bất khả thi với inheritance:** Muốn share "fetch data" logic giữa `UserList` và `ProductList`, với class phải dùng HOC (Higher-Order Component) hoặc Render Props — cả hai đây là workaround của "can't share logic without inheritance". HOC tạo ra "wrapper hell":

```
<WithAuth>
  <WithTheme>
    <WithData>
      <MyComponent />
    </WithData>
  </WithTheme>
</WithAuth>
```

2. **Lifecycle coupling:** `componentDidMount` + `componentDidUpdate` + `componentWillUnmount` là 3 methods nhưng thường chứa 1 concern (setup/cleanup subscription). Logic bị split across lifecycle methods thay vì được nhóm theo concern.

3. **`this` confusion:** Class methods và `this` binding là nguồn bug phổ biến — `onClick={this.handleClick}` mà quên `.bind(this)` → runtime error.

**Hooks là Composition pattern thuần túy:**

```typescript
// Mỗi hook = 1 behavior/concern = 1 LEGO block
function useUserData(userId: string) {
  /* data concern */
}
function usePermissions(userId: string) {
  /* auth concern */
}
function useAnalytics() {
  /* tracking concern */
}

// Component compose behaviors mà không cần class hierarchy
function UserDashboard({ userId }: Props) {
  const { user } = useUserData(userId);
  const { can } = usePermissions(userId);
  const { track } = useAnalytics();
  // 3 behaviors, 0 inheritance, tất cả testable independently
}
```

Logic reuse giờ chỉ là extract custom hook — không cần HOC, không cần Render Props, không cần class hierarchy. Đây là lý do React docs nói: "We don't recommend rewriting your existing components overnight but you can start using Hooks in new components" — Hooks giải quyết structural problem mà class components không thể giải quyết cleanly.
