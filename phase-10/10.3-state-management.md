## Tại sao State Management là vấn đề khó

State là **nguồn gốc của hầu hết bugs** trong frontend. React founder Dan Abramov nói:

> "I think the real problem that we're trying to solve is: the state is being mutated somewhere, and by the time you render, you don't know what state you're in."

Cơ chế hỏng điển hình:

```
Component A mutates shared state
         ↓
Component B reads stale version of that state
         ↓
UI shows inconsistent data
         ↓
Bug report: "Sometimes the cart shows wrong total"
         ↓
Impossible to reproduce reliably
```

Đây là lý do state management là topic cực kỳ quan trọng — không phải vì có nhiều libraries, mà vì **bản chất của mutation là non-deterministic** khi nhiều parties đọc/ghi cùng lúc.

## 1. Local State vs Shared State vs Server State

### Cơ chế thật — phân loại đúng

Đây là **bước quan trọng nhất** trong state management. Engineer thường dùng global state store cho tất cả mọi thứ — đây là nguồn gốc của over-engineering và bugs.

Ba loại state có **ownership model khác nhau hoàn toàn**:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    STATE CLASSIFICATION                             │
│                                                                     │
│  LOCAL STATE                                                        │
│  ├── Owner: 1 component                                             │
│  ├── Lifetime: component lifetime                                   │
│  ├── Examples: form input, dropdown open/closed, tab selection      │
│  ├── Tool: useState, useReducer                                     │
│  └── Rule: nếu chỉ 1 component cần → giữ local                     │
│                                                                     │
│  SHARED STATE (Client)                                              │
│  ├── Owner: nhiều components, cùng client session                   │
│  ├── Lifetime: session hoặc app lifetime                            │
│  ├── Examples: auth user, theme, shopping cart, UI preferences      │
│  ├── Tool: Context, Zustand, Redux, Jotai                           │
│  └── Rule: nếu nhiều components ở nhiều nơi cần → lift/share       │
│                                                                     │
│  SERVER STATE                                                       │
│  ├── Owner: server — client chỉ có cached copy                     │
│  ├── Lifetime: phụ thuộc vào server, có thể stale bất kỳ lúc nào   │
│  ├── Examples: user profile, product list, orders, search results  │
│  ├── Tool: TanStack Query, SWR, Apollo Client                       │
│  └── Rule: nếu đến từ API → KHÔNG phải client state                │
└─────────────────────────────────────────────────────────────────────┘
```

### Server State là loại hay bị hiểu sai nhất

```typescript
// ❌ Sai lầm phổ biến: treat server data như client state
function UserProfile() {
  // Store server data trong local state — sai từ đầu
  const [user, setUser] = useState<User | null>(null);
  const [isLoading, setLoading] = useState(false);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    setLoading(true);
    fetch('/api/user/me')
      .then(r => r.json())
      .then(data => { setUser(data); setLoading(false); })
      .catch(err => { setError(err); setLoading(false); });
  }, []);

  // Vấn đề:
  // - Không có caching: mỗi mount = 1 API call
  // - Không có background refetch: data stale sau khi user update ở tab khác
  // - Loading/error state phải viết tay mỗi nơi
  // - Race condition: nếu component unmount trước khi fetch complete
  // - Không có retry
}

// ✅ Server state dùng TanStack Query
function UserProfile() {
  // Query tự handle: caching, background refetch, deduplication, retry, stale-while-revalidate
  const { data: user, isLoading, error } = useQuery({
    queryKey: ['user', 'me'],
    queryFn: () => api.getMe(),
    staleTime: 5 * 60 * 1000, // Consider fresh for 5 min
    gcTime: 10 * 60 * 1000,   // Keep in cache 10 min after unmount
  });

  if (isLoading) return <Skeleton />;
  if (error) return <ErrorState error={error} />;
  return <Profile user={user} />;
}
```

**Tại sao server state cần tool riêng:**

Server state có những đặc điểm mà client state tools (Redux, Zustand) không design để xử lý:

```
Server State Challenges:
├── Caching: Không re-fetch khi data còn fresh
├── Deduplication: 5 components mount cùng lúc, chỉ 1 request
├── Background updates: Refetch khi window focus, network reconnect
├── Stale-while-revalidate: Show cached data ngay, update background
├── Pagination: Infinite scroll, cursor-based pagination
├── Mutations + invalidation: Sau khi POST, invalidate cache tương ứng
├── Optimistic updates: Update UI ngay trước khi server confirm
└── Race condition handling: Cancel outdated requests
```

### Khi nào dùng loại state nào — Decision Tree

```
Câu hỏi: "State này thuộc loại gì?"
│
├── Nó đến từ server/API?
│   YES → Server State (TanStack Query, SWR)
│         Đừng bao giờ put server data vào Redux/Zustand
│
└── NO → Client state
    │
    ├── Chỉ 1 component dùng?
    │   YES → Local State (useState/useReducer)
    │         Đừng lift lên global store chỉ vì "có thể cần sau này" (YAGNI)
    │
    └── NO → Shared State
        │
        ├── Chỉ cần share trong 1 subtree của component tree?
        │   YES → Context API (theme, locale, modal context)
        │
        └── Cần share toàn app, hoặc có complex update logic?
            YES → Zustand / Jotai / Redux
                  Chọn theo complexity: Zustand < Jotai < Redux
```

## 2. Flux Architecture — Unidirectional Data Flow

### Cơ chế thật

Flux ra đời tại Facebook năm 2014 như một phản ứng với **MVC hai chiều** gây ra cascading updates không dự đoán được.

**MVC hai chiều — vấn đề gốc:**

```
MVC Two-Way Binding Problem:

View A ──updates──► Model X
Model X ──triggers──► View B update
View B ──updates──► Model Y
Model Y ──triggers──► View A update  ← Vòng lặp! Cascading!

Khi app lớn:
   Models: 20+
   Views: 100+
   Relationships: không track được
   → "Không biết tại sao View này thay đổi"
   → "Update cái này làm cái kia break"
```

**Flux giải quyết bằng Unidirectional Data Flow:**

```
┌─────────────────────────────────────────────────────────┐
│              FLUX UNIDIRECTIONAL DATA FLOW              │
│                                                         │
│   Action ──► Dispatcher ──► Store ──► View              │
│      ▲                                  │               │
│      └──────────────────────────────────┘               │
│                 User interaction                        │
│                                                         │
│  Không bao giờ: View → Store trực tiếp                  │
│  Không bao giờ: Store → Store trực tiếp                 │
│  Luôn luôn: View → Action → Dispatcher → Store → View   │
└─────────────────────────────────────────────────────────┘
```

**Tại sao unidirectional quan trọng:**

- Bất kỳ state change nào đều có **một path duy nhất** — dễ trace
- Store chỉ update khi nhận Action — không có side-channel
- **Time-travel debugging** trở nên khả thi: replay actions = replay state changes

### Redux — Flux implementation phổ biến nhất

Redux có 3 principles:

1. **Single source of truth** — toàn bộ app state trong 1 store
2. **State is read-only** — chỉ actions có thể trigger state change
3. **Changes made with pure functions** — reducers là pure functions

```
Redux Data Flow:

User clicks "Add to Cart"
    │
    ▼
dispatch({ type: 'cart/addItem', payload: product })
    │                                     ← Action: plain object mô tả WHAT happened
    ▼
rootReducer(currentState, action)
    │                                     ← Reducer: pure function, (state, action) → newState
    ▼
newState = { ...currentState, cart: updatedCart }
    │                                     ← Immutable: không mutate, tạo object mới
    ▼
Store notifies subscribers
    │
    ▼
React re-renders components đang subscribe tới cart data
```

```typescript
// Redux Toolkit — modern Redux, không cần boilerplate cũ

import { createSlice, PayloadAction, createSelector } from '@reduxjs/toolkit';

interface CartItem {
  product: Product;
  qty: number;
}

interface CartState {
  items: CartItem[];
  promoCode: string | null;
  promoDiscount: number;
}

const cartSlice = createSlice({
  name: 'cart',
  initialState: {
    items: [],
    promoCode: null,
    promoDiscount: 0,
  } as CartState,

  reducers: {
    addItem(state, action: PayloadAction<Product>) {
      // Redux Toolkit dùng Immer internally — bạn "mutate" nhưng thật ra là immutable
      const existing = state.items.find(
        (i) => i.product.id === action.payload.id,
      );
      if (existing) {
        existing.qty += 1; // ← Trông như mutation nhưng Immer tạo new object
      } else {
        state.items.push({ product: action.payload, qty: 1 });
      }
    },

    removeItem(state, action: PayloadAction<string>) {
      state.items = state.items.filter((i) => i.product.id !== action.payload);
    },

    updateQty(
      state,
      action: PayloadAction<{ productId: string; qty: number }>,
    ) {
      const item = state.items.find(
        (i) => i.product.id === action.payload.productId,
      );
      if (item) {
        item.qty = Math.max(1, action.payload.qty); // Minimum qty: 1
      }
    },

    applyPromo(
      state,
      action: PayloadAction<{ code: string; discount: number }>,
    ) {
      state.promoCode = action.payload.code;
      state.promoDiscount = action.payload.discount;
    },

    clearCart(state) {
      state.items = [];
      state.promoCode = null;
      state.promoDiscount = 0;
    },
  },
});

export const { addItem, removeItem, updateQty, applyPromo, clearCart } =
  cartSlice.actions;

// Selectors — derived state, memoized với createSelector
// Selector là pure function: (state: RootState) → derived value
export const selectCartItems = (state: RootState) => state.cart.items;

export const selectCartSummary = createSelector(
  // Input selectors — recompute chỉ khi inputs thay đổi
  selectCartItems,
  (state: RootState) => state.cart.promoDiscount,

  // Result function — chỉ chạy khi inputs thay đổi
  (items, promoDiscount) => {
    const subtotal = items.reduce((sum, i) => sum + i.product.price * i.qty, 0);
    const discount = subtotal * promoDiscount;
    const total = subtotal - discount;
    const itemCount = items.reduce((sum, i) => sum + i.qty, 0);

    return { subtotal, discount, total, itemCount };
  },
);
// createSelector dùng shallow equality để skip recompute
// Nếu items không đổi và promoDiscount không đổi → return cached result
// Component không re-render dù parent re-render

// Async action với createAsyncThunk
export const applyPromoCode = createAsyncThunk(
  'cart/applyPromoCode',
  async (code: string, { rejectWithValue }) => {
    try {
      const result = await promoApi.validate(code);
      return result; // { code: string, discount: number }
    } catch (error) {
      return rejectWithValue('Invalid promo code');
    }
  },
);

// Extra reducers để handle async action states
const cartSliceWithAsync = createSlice({
  name: 'cart',
  initialState: {
    ...initialState,
    promoStatus: 'idle' as 'idle' | 'loading' | 'success' | 'error',
    promoError: null as string | null,
  },
  reducers: {
    /* ... same as above ... */
  },
  extraReducers: (builder) => {
    builder
      .addCase(applyPromoCode.pending, (state) => {
        state.promoStatus = 'loading';
        state.promoError = null;
      })
      .addCase(applyPromoCode.fulfilled, (state, action) => {
        state.promoStatus = 'success';
        state.promoCode = action.payload.code;
        state.promoDiscount = action.payload.discount;
      })
      .addCase(applyPromoCode.rejected, (state, action) => {
        state.promoStatus = 'error';
        state.promoError = action.payload as string;
      });
  },
});
```

**Immer — cơ chế thật đằng sau Redux Toolkit:**

```typescript
// Immer dùng JavaScript Proxy để intercept mutations
// Khi bạn "mutate" trong Immer produce(), nó track changes và create new object

import { produce } from 'immer';

const state = { items: [{ id: '1', qty: 2 }] };

// Immer produce():
// 1. Tạo Proxy của state (draft)
// 2. Bạn mutate draft
// 3. Proxy ghi lại mọi mutation
// 4. Immer create new object dựa trên mutations (structural sharing)
// 5. Parts không đổi → giữ nguyên reference (không copy)

const newState = produce(state, (draft) => {
  const item = draft.items.find((i) => i.id === '1');
  if (item) item.qty = 5; // ← Trông như mutation
});

console.log(state.items[0].qty); // 2 — unchanged
console.log(newState.items[0].qty); // 5 — new object
console.log(state === newState); // false — new root
console.log(state.items === newState.items); // false — new array (because item changed)

// Structural sharing: nếu bạn có 1000 items và chỉ đổi 1 item
// → 999 items giữ nguyên reference (không copy)
// → Chỉ item thay đổi và ancestors của nó được copy
// → Memory efficient
```

## 3. Immutability — Tại sao quan trọng, Structural Sharing

### Cơ chế thật

Immutability không phải về performance — thực ra tạo object mới **chậm hơn** mutation. Immutability là về **predictability và change detection**.

**Tại sao mutable state khó debug:**

```typescript
// Mutable state: reference identity không thể dùng để detect change
const state = { count: 0 };
const snapshot = state; // snapshot trỏ vào cùng object

state.count = 5; // Mutate

console.log(snapshot.count); // 5 — snapshot cũng thay đổi!
// Không có way nào biết state đã thay đổi nếu bạn chỉ có reference cũ

// Immutable state: reference identity = change detection
const state = { count: 0 };
const snapshot = state; // snapshot trỏ vào object gốc

const newState = { ...state, count: 5 }; // Tạo object mới

console.log(snapshot.count); // 0 — snapshot không thay đổi
console.log(state === newState); // false — reference khác → changed!
console.log(state === snapshot); // true — original unchanged
```

**React re-rendering dựa vào immutability:**

```typescript
// React's reconciler dùng Object.is() để compare state/props
// Object.is() với objects: so sánh reference, không phải value

// ❌ Mutation phá vỡ React's change detection
function BadComponent() {
  const [items, setItems] = useState([{ id: 1, name: 'Item 1' }]);

  const updateItem = () => {
    items[0].name = 'Updated'; // ← MUTATION — phá vỡ immutability
    setItems(items); // ← Cùng reference! React thấy không đổi
    // Component KHÔNG re-render dù data đã thay đổi
  };
}

// ✅ Immutable update → React re-renders correctly
function GoodComponent() {
  const [items, setItems] = useState([{ id: 1, name: 'Item 1' }]);

  const updateItem = () => {
    setItems((prev) =>
      prev.map(
        (item) =>
          item.id === 1
            ? { ...item, name: 'Updated' } // ← New object reference
            : item, // ← Existing reference (structural sharing)
      ),
    );
    // New array reference → React detects change → re-render
  };
}
```

**Structural Sharing — tại sao immutability không tốn nhiều memory:**

```typescript
// Structural sharing: chỉ những phần thay đổi mới được copy
// Phần còn lại giữ nguyên reference

const originalState = {
  user: { name: 'John', age: 30 }, // Reference A
  cart: { items: [1, 2, 3] }, // Reference B
  settings: { theme: 'dark' }, // Reference C
};

// Chỉ update user.age
const newState = {
  ...originalState,
  user: { ...originalState.user, age: 31 }, // New reference D
  // cart và settings giữ nguyên reference B và C — không copy!
};

console.log(newState.cart === originalState.cart); // true — same reference
console.log(newState.settings === originalState.settings); // true — same reference
console.log(newState.user === originalState.user); // false — new reference

// Memory: chỉ user object mới được allocate
// cart và settings objects không bị duplicate trong memory
```

```
Visual: Structural Sharing

originalState ──► { user: ──► UserObj_v1
                    cart: ──► CartObj      ← shared
                    settings: ► SettingsObj ← shared
                  }

newState ──────► { user: ──► UserObj_v2  ← new
                   cart: ──► CartObj      ← same pointer
                   settings: ► SettingsObj ← same pointer
                 }
```

## 4. useReducer — Local Complex State

### Cơ chế thật

`useReducer` là Redux pattern ở local component level. Khi `useState` không đủ vì có **nhiều sub-states liên quan đến nhau** hoặc **next state phụ thuộc vào current state theo nhiều cách**, dùng `useReducer`.

```typescript
// Khi nào useReducer tốt hơn useState:
// - State object có nhiều fields được update cùng nhau
// - Next state phụ thuộc vào current state theo nhiều cách
// - Update logic phức tạp cần test riêng (reducer là pure function)
// - State transitions có business rules cụ thể

type FetchState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; error: Error };

type FetchAction<T> =
  | { type: 'FETCH_START' }
  | { type: 'FETCH_SUCCESS'; payload: T }
  | { type: 'FETCH_ERROR'; payload: Error }
  | { type: 'RESET' };

// Reducer là pure function — có thể test hoàn toàn tách biệt với React
function fetchReducer<T>(
  state: FetchState<T>,
  action: FetchAction<T>,
): FetchState<T> {
  switch (action.type) {
    case 'FETCH_START':
      return { status: 'loading' };

    case 'FETCH_SUCCESS':
      return { status: 'success', data: action.payload };

    case 'FETCH_ERROR':
      // Chỉ transition tới error nếu đang loading
      // Ignore race condition responses (state đã thay đổi)
      if (state.status !== 'loading') return state;
      return { status: 'error', error: action.payload };

    case 'RESET':
      return { status: 'idle' };

    default:
      return state;
  }
}

// Hook với useReducer
function useFetch<T>(url: string) {
  const [state, dispatch] = useReducer(fetchReducer<T>, { status: 'idle' });

  useEffect(() => {
    if (!url) return;

    let cancelled = false; // Handle race condition

    dispatch({ type: 'FETCH_START' });

    fetch(url)
      .then((r) => r.json())
      .then((data) => {
        if (!cancelled) dispatch({ type: 'FETCH_SUCCESS', payload: data });
      })
      .catch((error) => {
        if (!cancelled) dispatch({ type: 'FETCH_ERROR', payload: error });
      });

    return () => {
      cancelled = true;
    }; // Cleanup: prevent dispatch to unmounted
  }, [url]);

  return state;
}

// Test reducer độc lập với React — cực kỳ dễ
describe('fetchReducer', () => {
  it('should transition to loading on FETCH_START', () => {
    const result = fetchReducer({ status: 'idle' }, { type: 'FETCH_START' });
    expect(result).toEqual({ status: 'loading' });
  });

  it('should ignore FETCH_ERROR if not loading (race condition)', () => {
    const successState: FetchState<string> = {
      status: 'success',
      data: 'data',
    };
    const result = fetchReducer(successState, {
      type: 'FETCH_ERROR',
      payload: new Error('late error'),
    });
    // State không đổi vì không phải 'loading'
```

## 🟡 5. Finite State Machines (XState)

### Cơ chế thật

FSM (Finite State Machine) là mô hình toán học: một system có **hữu hạn states**, tại **một thời điểm chỉ ở một state**, và chuyển đổi giữa states thông qua **events**.

**Tại sao FSM quan trọng:**

Complex UI có vô số trạng thái ẩn. Nếu không model rõ ràng, sẽ có **impossible states** và **invalid transitions**:

```
Boolean states = 2^n combinations:
isLoading + isError + isSuccess + isEmpty = 2^4 = 16 combinations
Nhưng thực tế chỉ có ~5 valid states

isLoading=true && isSuccess=true  ← impossible state
isError=true && isEmpty=true      ← ambiguous state
```

```typescript

  // Nhưng code không prevent điều này
}
```

## 🟡 6. Optimistic Updates

### Cơ chế thật

Optimistic update: Update UI **ngay lập tức** như thể request đã thành công, sau đó revert nếu thất bại. Mục tiêu: UX nhanh hơn, không phải wait spinner.

```
PESSIMISTIC (thông thường):
User action → Show spinner → Wait for server → Update UI
Latency: 200-500ms visible wait

OPTIMISTIC:
User action → Update UI immediately → Server request (background)
           → Success: keep UI (nothing changes visually)
           → Failure: revert UI + show error
Latency: 0ms visible (với risk of revert)
```

**Khi nào dùng optimistic updates:**

- Actions có high success rate (like, follow, cart add/remove)
- Server response không thay đổi data đáng kể (không có server-side computation)
- Revert không confusing user (không phải financial transaction)

**Khi nào KHÔNG dùng:**

- Financial transactions — user cần confirmation
- Actions với business validation phức tạp phía server
- Actions có low success rate

```typescript
// ✅ Optimistic Update với TanStack Query

function useToggleLike(postId: string) {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (liked: boolean) => api.toggleLike(postId, liked),

    // onMutate: chạy TRƯỚC khi mutation request gửi đi
    onMutate: async (newLikedState: boolean) => {
      // 1. Cancel outgoing refetches (tránh overwrite optimistic update)
      await queryClient.cancelQueries({ queryKey: ['post', postId] });

      // 2. Snapshot current data để có thể rollback
      const previousPost = queryClient.getQueryData<Post>(['post', postId]);

      // 3. Optimistically update cache
      queryClient.setQueryData<Post>(['post', postId], old => {
        if (!old) return old;
        return {
          ...old,
          liked: newLikedState,
          likeCount: newLikedState ? old.likeCount + 1 : old.likeCount - 1
        };
      });

      // 4. Return snapshot — accessible trong onError
      return { previousPost };
    },

    // onError: chạy nếu mutation fail — rollback
    onError: (error, newLikedState, context) => {
      // Restore previous data
      if (context?.previousPost) {
        queryClient.setQueryData(['post', postId], context.previousPost);
      }
      toast.error('Failed to update like. Please try again.');
    },

    // onSettled: chạy sau success hoặc error
    // Sync với server để đảm bảo UI chính xác
    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ['post', postId] });
    }
  });
}

// Component
function PostCard({ postId }: Props) {
  const { data: post } = useQuery(['post', postId], () => api.getPost(postId));
  const { mutate: toggleLike, isPending } = useToggleLike(postId);

  return (
    <div>
      <button
        onClick={() => toggleLike(!post?.liked)}
        disabled={isPending} // Prevent double-click
        className={post?.liked ? 'liked' : ''}
      >
        ♥ {post?.likeCount}
        {/* UI cập nhật ngay — không cần spinner */}
      </button>
    </div>
```

```typescript
    likePostOptimistic(

    },

  };
```

## 🟢 7. Event Sourcing cho Client-Side State

### Cơ chế thật

Event Sourcing: thay vì lưu **current state**, lưu **sequence of events** dẫn đến state. State là **derived** bằng cách replay events.

```
NORMAL STATE:
  Store: { count: 5 }
  → Không biết làm sao đến count=5
  → Không có history

EVENT SOURCING:
  Events: [
    { type: 'INCREMENT', amount: 3, timestamp: T1 },
    { type: 'INCREMENT', amount: 4, timestamp: T2 },
    { type: 'DECREMENT', amount: 2, timestamp: T3 },
  ]
  Current state = replay(events) = 0 + 3 + 4 - 2 = 5
  → Full audit trail
  → Time travel: replay events up to T2 → count = 7
  → Undo: remove last event → replay remaining
```

```typescript
// Event Sourcing pattern cho collaborative editor

type DocumentEvent =
  | {
      type: 'INSERT';
      position: number;
      text: string;
      userId: string;
      timestamp: number;
    }
  | {
      type: 'DELETE';
      position: number;
      length: number;
      userId: string;
      timestamp: number;
    }
  | {
      type: 'FORMAT';
      start: number;
      end: number;
      format: string;
      userId: string;
      timestamp: number;
    };

class DocumentStore {
  private events: DocumentEvent[] = [];

  // Apply event
  applyEvent(event: DocumentEvent): void {
    this.events.push(event);
  }

  // Derive current state từ events
  getCurrentState(): DocumentState {
    return this.events.reduce(applyEventToDocument, initialDocumentState);
  }

  // Time travel
  getStateAtTime(timestamp: number): DocumentState {
    const relevantEvents = this.events.filter((e) => e.timestamp <= timestamp);
    return relevantEvents.reduce(applyEventToDocument, initialDocumentState);
  }

  // Undo last event của user
  undoLastUserEvent(userId: string): void {
    const lastUserEventIndex = [...this.events]
      .reverse()
      .findIndex((e) => e.userId === userId);

    if (lastUserEventIndex !== -1) {
      const actualIndex = this.events.length - 1 - lastUserEventIndex;
      this.events.splice(actualIndex, 1);
    }
  }

  // Audit trail
  getEventsByUser(userId: string): DocumentEvent[] {
    return this.events.filter((e) => e.userId === userId);
  }
}

// Redux là lightweight Event Sourcing:
// actions = events, store = current state (không lưu history)
// Redux DevTools time travel = replay actions
```

## Câu hỏi ôn tập (NotebookLM)

### Câu 1: Tại sao "server state" cần tools khác với "client state", và điều gì xảy ra khi team treat chúng như nhau (put server data vào Redux)?

**Đáp án:**

Server state và client state có **ownership model hoàn toàn khác nhau**. Client state do client own — nó không tồn tại nơi nào khác, client là source of truth. Server state do server own — client chỉ có **cached copy** có thể stale bất kỳ lúc nào.

Sự khác biệt này dẫn đến những yêu cầu kỹ thuật khác nhau:

**Client state cần:** Global accessibility, ability to update from anywhere, predictable mutations. Redux, Zustand giải quyết tốt.

**Server state cần:** Caching (không re-fetch khi data còn fresh), deduplication (nhiều components mount cùng lúc = 1 request), background refetch (window focus, reconnect), stale-while-revalidate, pagination, mutation + invalidation. TanStack Query / SWR design cho những điều này.

**Điều gì xảy ra khi treat server data như client state:**

```typescript
// Redux cho server data — vấn đề thực tế
const productsSlice = createSlice({
  name: 'products',
  initialState: { items: [], isLoading: false, error: null },
  reducers: {
    /* ... */
  },
  extraReducers: (builder) =>
    builder
      .addCase(fetchProducts.pending, (state) => {
        state.isLoading = true;
      })
      .addCase(fetchProducts.fulfilled, (state, action) => {
        state.items = action.payload;
        state.isLoading = false;
      }),
});
```

**Vấn đề 1: Không có caching strategy.** Mỗi lần `ProductListPage` mount (user navigate away rồi quay lại) = 1 API call mới dù data không đổi.

**Vấn đề 2: Không có background sync.** Nếu user mở 2 tabs, tab 1 update product, tab 2 vẫn hiển thị stale data. Redux không có mechanism refetch khi window focus.

**Vấn đề 3: Boilerplate explosion.** Mỗi entity (users, products, orders) cần loading/error state, fetch thunk, slice — hàng trăm dòng code, mostly identical.

**Vấn đề 4: Stale data sau mutation.** Sau khi POST new product, phải manually dispatch `fetchProducts()` lại. Nếu quên → UI hiển thị outdated data.

**Vấn đề 5: Race condition không được xử lý.** Nếu user navigate nhanh (Products → Users → Products), responses về theo thứ tự khác request → UI có thể hiển thị sai data. TanStack Query handle điều này tự động.

Với TanStack Query, tất cả 5 vấn đề trên được giải quyết out of the box với ~10 dòng code thay vì ~100 dòng Redux boilerplate.

### Câu 2: Structural sharing là gì, tại sao nó làm cho immutability practical (không tốn nhiều memory), và khi nào immutability thật sự có performance cost?

**Đáp án:**

**Structural sharing** là kỹ thuật: khi tạo một version mới của data structure, chỉ những phần **thay đổi** mới được copy. Những phần không thay đổi **giữ nguyên reference** (pointer) tới objects cũ.

```
State tree:
  root
  ├── user (object A)
  ├── cart (object B)
  │   ├── items (array C) ← chỉ phần này thay đổi
  │   └── promoCode
  └── settings (object D)

Sau khi add item vào cart:
  newRoot (new)
  ├── user (object A) ← SAME reference, không copy
  ├── cart (new object)
  │   ├── items (new array) ← Copy vì thay đổi
  │   └── promoCode ← giá trị primitive copy
  └── settings (object D) ← SAME reference, không copy
```

**Memory cost thực tế:** Với app state 1MB, nếu bạn update 1 field nhỏ:

- Structural sharing: chỉ path từ root tới field đó được copy (logarithmic)
- Full copy: 1MB mới
- Immutable.js, Immer đều implement structural sharing

**Khi immutability có performance cost thực sự:**

**Cost 1: GC pressure.** Nhiều short-lived objects được tạo → GC phải chạy thường xuyên hơn. Trong hot path (event handler gọi 60fps, tight loop), điều này đo được.

```typescript
// ❌ Immutable update trong animation loop — tạo object mới mỗi frame
function animate() {
  setParticles((prev) =>
    prev.map((p) => ({
      ...p, // ← New object mỗi particle mỗi frame
      x: p.x + p.vx, // 1000 particles × 60fps = 60,000 objects/sec bị GC
      y: p.y + p.vy,
    })),
  );
  requestAnimationFrame(animate);
}

// ✅ Mutable buffer cho animation — không GC pressure
const particlesRef = useRef<Float32Array>(new Float32Array(1000 * 4)); // x, y, vx, vy

function animate() {
  for (let i = 0; i < 1000; i++) {
    particlesRef.current[i * 4] += particlesRef.current[i * 4 + 2]; // x += vx
    particlesRef.current[i * 4 + 1] += particlesRef.current[i * 4 + 3]; // y += vy
  }
  // Mutate TypedArray directly — zero GC
  canvas.render(particlesRef.current);
  requestAnimationFrame(animate);
}
```

**Cost 2: Deep object graphs.** Nếu bạn immutably update một deeply nested field, mọi ancestor node trên path đó cần được copy. Với 10 levels deep, đó là 10 object creations per update.

**Rule of thumb:**

- Business logic, UI state, Redux → immutability luôn đúng, cost không đáng kể
- Animation, simulation, WebGL, game loop → use mutable buffers (TypedArray, object pools)
- Trong doubt → profile trước khi optimize, đừng premature optimize

**Đáp án:**

Code này có **5 vấn đề** liên quan đến state management:

**Vấn đề 1: Server data trong local state (sai classification)**

`user` và `orders` đến từ API — đây là server state, không nên dùng `useState`. Hậu quả: không có caching, không có background refetch, mỗi Dashboard mount = 2 API calls dù data không đổi.

Fix:

```typescript

  queryKey: ['user', 'me'],
  queryFn: () => fetch('/api/user/me').then((r) => r.json()),
});

const { data: orders = [], isLoading: isLoadingOrders } = useQuery({
  queryKey: ['orders'],
  queryFn: () => fetch('/api/orders').then((r) => r.json()),
});
```

**Vấn đề 2: Derived state được stored (vi phạm SSOT)**

`totalRevenue` là computed từ `orders` nhưng được stored như separate state. Trong `cancelOrder`, phải manually update cả `orders` lẫn `totalRevenue` — nếu quên một trong hai → inconsistency.

Fix:

```typescript
// Không store totalRevenue — compute từ SSOT

```

**Vấn đề 3: Không có error handling**

`fetch` không xử lý network error hay non-2xx responses. Nếu API fails → `setUser(undefined)` (vì `.json()` sẽ fail) → silent bug.

Fix: TanStack Query handle error states tự động. Nếu dùng raw fetch, cần `.catch()`.

**Vấn đề 4: Race condition trong `cancelOrder`**

`cancelOrder` update local state optimistically (tốt), nhưng không handle failure — nếu API call thất bại, UI đã xóa order khỏi list nhưng server vẫn có order đó. Không có rollback.

Fix:

```typescript
const cancelOrder = useMutation({
  mutationFn: (orderId: string) =>
    fetch(`/api/orders/${orderId}/cancel`, { method: 'POST' }),
  onMutate: async (orderId) => {
    await queryClient.cancelQueries({ queryKey: ['orders'] });
    const previousOrders = queryClient.getQueryData<Order[]>(['orders']);
    queryClient.setQueryData<Order[]>(
      ['orders'],
      (prev) => prev?.filter((o) => o.id !== orderId) ?? [],
    );
    return { previousOrders };
  },
  onError: (err, orderId, context) => {
    queryClient.setQueryData(['orders'], context?.previousOrders);
    toast.error('Failed to cancel order');
  },
  onSettled: () => queryClient.invalidateQueries({ queryKey: ['orders'] }),
});
```

**Vấn đề 5: Memory leak potential**

`useEffect` không có cleanup. Nếu component unmount trước khi fetch complete → `setUser` / `setOrders` gọi trên unmounted component → React warning (và state update lost).

Fix với TanStack Query: tự động handle cleanup. Fix với raw fetch:

```typescript
useEffect(() => {
  const controller = new AbortController();
  fetch('/api/user/me', { signal: controller.signal })
    .then((r) => r.json())
    .then(setUser)
    .catch((e) => {
      if (e.name !== 'AbortError') setError(e);
    });
  return () => controller.abort(); // Cleanup cancel fetch
}, []);
```
