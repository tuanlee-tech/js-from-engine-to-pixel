## Tại sao Testing là skill quan trọng của Senior+

Junior viết code. Senior viết code **có thể verify được**. Staff viết code mà **cả team tin tưởng thay đổi mà không sợ**.

> "Code không có tests không phải là code — đó là ý định." — Michael Feathers

Cơ chế hỏng khi không có tests:

```
Feature A được thêm
       ↓
Tháng sau: Feature B thêm vào
       ↓
Feature B vô tình break Feature A (regression)
       ↓
Không ai biết cho đến khi user báo cáo
       ↓
Hotfix rush → break Feature C
       ↓
Team sợ thay đổi code → velocity giảm → technical debt tích lũy
```

Testing không phải về "tìm bugs". Testing là về **confidence to change**.

## 1. Testing Pyramid

### Cơ chế thật

Testing Pyramid của Mike Cohn mô tả **tỷ lệ tối ưu** giữa các loại test dựa trên cost/benefit:

```
                    ╱╲
                   ╱  ╲
                  ╱ E2E ╲          ← Ít nhất: chậm, brittle, đắt
                 ╱  (5%) ╲
                ╱──────────╲
               ╱Integration ╲      ← Vừa: realistic, moderate cost
              ╱    (20-30%)   ╲
             ╱─────────────────╲
            ╱    Unit Tests      ╲  ← Nhiều nhất: nhanh, stable, rẻ
           ╱       (65-75%)       ╲
          ╱─────────────────────────╲
```

**Tại sao hình kim tự tháp, không phải hình khác:**

```
Unit Tests:
  ├── Tốc độ:     Milliseconds per test
  ├── Reliability: Deterministic — không phụ thuộc network, DB, browser
  ├── Feedback:   Ngay lập tức khi save file
  ├── Cost:       Rẻ nhất để viết và maintain
  └── Scope:      Test 1 function/component trong isolation

Integration Tests:
  ├── Tốc độ:     Tens of milliseconds — seconds
  ├── Reliability: Mostly deterministic (mock external, real internal)
  ├── Feedback:   Pre-commit / CI
  ├── Cost:       Moderate
  └── Scope:      Test nhiều units làm việc cùng nhau

E2E Tests:
  ├── Tốc độ:     Seconds — minutes per test
  ├── Reliability: Brittle (network, timing, browser quirks)
  ├── Feedback:   CI chạy xong (minutes — hours)
  ├── Cost:       Đắt nhất — setup, maintain, flakiness
  └── Scope:      Test toàn bộ system từ UI tới database
```

**Ice Cream Cone Anti-Pattern — đừng làm điều này:**

```
         ╱──────────────────────────╲
        ╱       E2E Tests             ╲   ← QA team viết manual/automated
       ╱────────────────────────────────╲
      ╱         Integration Tests        ╲
     ╱────────────────────────────────────╲
    ╱               Unit Tests              ╲  ← Ít, sau-thoughts
   ╱──────────────────────────────────────────╲

Hậu quả:
- CI chạy 30+ phút
- Flaky tests làm team mất tin tưởng
- Bugs chỉ phát hiện muộn
- Refactor rất khó vì tests coupled với UI
```

## 2. Unit Testing — Vitest/Jest

### Cơ chế thật

Unit test kiểm tra **một unit** (function, class, component) **trong isolation**. "Isolation" nghĩa là tất cả dependencies bên ngoài đều bị mock/stub.

```
Unit Test Execution:
  describe('calculateDiscount')
         │
         ▼
  Vitest runner finds test file
         │
         ▼
  Import module under test (no real dependencies)
         │
         ▼
  Mock external dependencies (DB, API, Date)
         │
         ▼
  Call function with specific inputs
         │
         ▼
  Assert output/behavior (expect)
         │
         ▼
  Pass/fail in milliseconds
```

### Anatomy của một test tốt — AAA Pattern

```typescript
// Arrange → Act → Assert

import { describe, it, expect, beforeEach } from 'vitest';
import { calculateOrderTotal } from './order-pricing';

describe('calculateOrderTotal', () => {
  // beforeEach: setup dùng chung, chạy trước mỗi test
  // Tránh shared mutable state giữa tests
  let baseOrder: Order;

  beforeEach(() => {
    baseOrder = {
      items: [
        { productId: 'p1', price: 100, qty: 2 },
        { productId: 'p2', price: 50, qty: 1 },
      ],
      promoDiscount: 0,
    };
  });

  it('should calculate subtotal correctly', () => {
    // ARRANGE: đã setup trong beforeEach

    // ACT: gọi function under test
    const result = calculateOrderTotal(baseOrder);

    // ASSERT: kiểm tra output
    expect(result.subtotal).toBe(250); // 100*2 + 50*1
    expect(result.discount).toBe(0);
    expect(result.total).toBe(250);
  });

  it('should apply percentage discount correctly', () => {
    // ARRANGE: modify base order
    const order = { ...baseOrder, promoDiscount: 0.1 };

    // ACT
    const result = calculateOrderTotal(order);

    // ASSERT
    expect(result.subtotal).toBe(250);
    expect(result.discount).toBe(25); // 10% of 250
    expect(result.total).toBe(225);
  });

  it('should return zero total for empty cart', () => {
    const emptyOrder = { items: [], promoDiscount: 0 };
    const result = calculateOrderTotal(emptyOrder);
    expect(result.total).toBe(0);
  });

  // Test edge cases và error cases
  it('should throw when item price is negative', () => {
    const badOrder = {
      items: [{ productId: 'p1', price: -10, qty: 1 }],
      promoDiscount: 0,
    };
    expect(() => calculateOrderTotal(badOrder)).toThrow(
      'Item price cannot be negative',
    );
  });
});
```

### Mocks, Stubs, Spies — phân biệt

```typescript
import { vi, expect, describe, it } from 'vitest';

// STUB: Replace implementation, trả về giá trị cố định
// Dùng khi: muốn control output của dependency
const mockUserRepo = {
  findById: vi
    .fn()
    .mockResolvedValue({ id: '1', name: 'John', email: 'john@example.com' }),
  save: vi.fn().mockResolvedValue(undefined),
};

// SPY: Wrap real implementation, track calls
// Dùng khi: muốn verify behavior của real function
const emailService = {
  send: vi.fn(), // Mock — không gọi real send
};

// MOCK MODULE: Replace entire module
vi.mock('../services/email', () => ({
  sendEmail: vi.fn().mockResolvedValue({ messageId: 'mock-id' }),
}));

// Test với mocks
describe('UserRegistrationService', () => {
  it('should save user and send welcome email', async () => {
    // Arrange
    const service = new UserRegistrationService(mockUserRepo, emailService);
    const dto = {
      name: 'Alice',
      email: 'alice@example.com',
      password: 'secret123',
    };

    // Act
    await service.register(dto);

    // Assert behavior — không chỉ output
    expect(mockUserRepo.save).toHaveBeenCalledOnce();
    expect(mockUserRepo.save).toHaveBeenCalledWith(
      expect.objectContaining({
        // objectContaining: partial match
        email: 'alice@example.com',
        name: 'Alice',
        // passwordHash không cần biết exact value
      }),
    );

    expect(emailService.send).toHaveBeenCalledOnce();
    expect(emailService.send).toHaveBeenCalledWith(
      expect.objectContaining({ to: 'alice@example.com' }),
    );
  });

  it('should not send email if user save fails', async () => {
    // Arrange: make save throw
    mockUserRepo.save.mockRejectedValueOnce(new Error('DB error'));
    const service = new UserRegistrationService(mockUserRepo, emailService);

    // Act & Assert
    await expect(
      service.register({
        name: 'Bob',
        email: 'bob@example.com',
        password: 'pass',
      }),
    ).rejects.toThrow('DB error');

    // Email không được gửi khi DB fail
    expect(emailService.send).not.toHaveBeenCalled();
  });
});
```

## 3. Testing Library — Test như User

### Cơ chế thật

Testing Library được build trên triết lý:

> "The more your tests resemble the way your software is used, the more confidence they can give you." — Kent C. Dodds

**Guiding principle:** Query DOM elements theo cách **user thấy và interact**, không theo cách developer implement.

```
Testing Library Query Priority (từ cao xuống thấp):
  1. getByRole       ← Accessible name + ARIA role — cách screen reader thấy
  2. getByLabelText  ← Form labels — cách user tìm input
  3. getByPlaceholder← Placeholder text
  4. getByText       ← Visible text content
  5. getByDisplayValue ← Current form value
  6. getByAltText    ← Image alt text
  7. getByTitle      ← Title attribute
  8. getByTestId     ← data-testid — last resort

High priority queries test behavior → tests không break khi refactor implementation
Low priority queries test implementation → brittle tests
```

```typescript
import { render, screen, fireEvent, waitFor, within } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { LoginForm } from './LoginForm';

describe('LoginForm', () => {
  // userEvent > fireEvent vì simulate real browser behavior
  // userEvent.type() = focus + keydown + keypress + keyup + input events
  // fireEvent.change() = chỉ input event — không realistic
  const user = userEvent.setup();

  it('should show validation error when email is invalid', async () => {
    render(<LoginForm onLogin={vi.fn()} />);

    // Query by role — cách user/screen reader finds element
    const emailInput = screen.getByRole('textbox', { name: /email/i });
    const submitButton = screen.getByRole('button', { name: /log in/i });

    // Simulate realistic user interaction
    await user.type(emailInput, 'not-an-email');
    await user.click(submitButton);

    // Assert: user thấy gì trên screen?
    expect(screen.getByRole('alert')).toHaveTextContent(/invalid email/i);
    // Không assert: component state, CSS class names, internal implementation
  });

  it('should call onLogin with credentials when form is valid', async () => {
    const onLogin = vi.fn().mockResolvedValue(undefined);
    render(<LoginForm onLogin={onLogin} />);

    await user.type(screen.getByLabelText(/email/i), 'john@example.com');
    await user.type(screen.getByLabelText(/password/i), 'password123');
    await user.click(screen.getByRole('button', { name: /log in/i }));

    expect(onLogin).toHaveBeenCalledWith({
      email: 'john@example.com',
      password: 'password123',
    });
  });

  it('should disable submit button while loading', async () => {
    // onLogin returns pending promise — simulate loading
    let resolveLogin: () => void;
    const onLogin = vi.fn().mockReturnValue(
      new Promise<void>(resolve => { resolveLogin = resolve; })
    );

    render(<LoginForm onLogin={onLogin} />);

    await user.type(screen.getByLabelText(/email/i), 'john@example.com');
    await user.type(screen.getByLabelText(/password/i), 'password123');
    await user.click(screen.getByRole('button', { name: /log in/i }));

    // Button disabled during loading
    expect(screen.getByRole('button', { name: /logging in/i })).toBeDisabled();

    // Resolve loading
    resolveLogin!();

    // Button re-enabled
    await waitFor(() => {
      expect(screen.getByRole('button', { name: /log in/i })).toBeEnabled();
    });
  });
});
```

## 4. Integration Testing — Test User Flows

### Cơ chế thật

Integration tests verify **nhiều units làm việc cùng nhau**. Không mock implementation details — chỉ mock external boundaries (real HTTP calls, actual DB).

```
What to mock in Integration Tests:
  ├── External APIs (via MSW)      ← Mock
  ├── Email services               ← Mock
  ├── Payment processors           ← Mock
  ├── Internal services            ← Real
  ├── React components             ← Real
  ├── Custom hooks                 ← Real
  ├── State management             ← Real
  └── Business logic               ← Real
```

```typescript
// Integration test: test entire cart workflow
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { QueryClientProvider } from '@tanstack/react-query';
import { CartProvider } from '../context/CartContext';
import { ProductListPage } from '../pages/ProductListPage';
import { CartPage } from '../pages/CartPage';

// Helper: render với tất cả real providers
function renderWithProviders(ui: React.ReactElement) {
  const queryClient = new QueryClient({
    defaultOptions: { queries: { retry: false } } // Không retry trong tests
  });

  return render(
    <QueryClientProvider client={queryClient}>
      <CartProvider>
        <MemoryRouter>
          {ui}
        </MemoryRouter>
      </CartProvider>
    </QueryClientProvider>
  );
}

describe('Shopping Cart Integration', () => {
  const user = userEvent.setup();

  it('should allow user to add products and checkout', async () => {
    // MSW đã setup mock handlers cho /api/products và /api/orders

    renderWithProviders(<ProductListPage />);

    // Wait for products to load
    const addButtons = await screen.findAllByRole('button', { name: /add to cart/i });
    expect(addButtons).toHaveLength(3); // 3 products from mock

    // Add first product
    await user.click(addButtons[0]);
    expect(screen.getByRole('status', { name: /cart count/i })).toHaveTextContent('1');

    // Add second product
    await user.click(addButtons[1]);
    expect(screen.getByRole('status', { name: /cart count/i })).toHaveTextContent('2');

    // Navigate to cart
    await user.click(screen.getByRole('link', { name: /view cart/i }));

    // Cart page shows correct items
    expect(screen.getByText(/2 items/i)).toBeInTheDocument();

    // Place order
    await user.click(screen.getByRole('button', { name: /place order/i }));

    // Success state
    await screen.findByRole('heading', { name: /order confirmed/i });
    expect(screen.getByText(/order-123/i)).toBeInTheDocument();
  });

  it('should persist cart across page navigations', async () => {
    renderWithProviders(<App />);

    // Navigate to products, add item
    await user.click(screen.getByRole('link', { name: /products/i }));
    await screen.findAllByRole('button', { name: /add to cart/i });
    await user.click(screen.getAllByRole('button', { name: /add to cart/i })[0]);

    // Navigate away
    await user.click(screen.getByRole('link', { name: /home/i }));

    // Navigate back to cart — item still there
    await user.click(screen.getByRole('link', { name: /cart/i }));
    expect(screen.getByText(/1 item/i)).toBeInTheDocument();
  });
});
```

## 5. Testing React-specific Patterns

### Testing Custom Hooks

```typescript
import { renderHook, act } from '@testing-library/react';
import { useCounter } from './useCounter';

describe('useCounter', () => {
  it('should initialize with given value', () => {
    const { result } = renderHook(() => useCounter(10));
    expect(result.current.count).toBe(10);
  });

  it('should increment counter', () => {
    const { result } = renderHook(() => useCounter(0));

    act(() => {
      result.current.increment();
    });
    // act() ensures all state updates are processed

    expect(result.current.count).toBe(1);
  });

  it('should not go below minimum', () => {
    const { result } = renderHook(() => useCounter(0, { min: 0 }));

    act(() => {
      result.current.decrement();
    });

    expect(result.current.count).toBe(0); // Stays at min
  });
});
```

## 🟡 7. Accessibility Testing

### Cơ chế thật

a11y tests đảm bảo component có thể được dùng bởi người dùng với disabilities. Kết hợp automated testing (phát hiện ~30% issues) với manual testing (phần còn lại).

```typescript
// axe-core với Vitest — automated a11y checks
import { render } from '@testing-library/react';
import { axe, toHaveNoViolations } from 'jest-axe';
import { expect } from 'vitest';

expect.extend(toHaveNoViolations);

describe('Modal accessibility', () => {
  it('should have no accessibility violations', async () => {
    const { container } = render(
      <Modal isOpen={true} onClose={vi.fn()} title="Confirm Delete">
        <p>Are you sure you want to delete this item?</p>
        <button>Cancel</button>
        <button>Delete</button>
      </Modal>
    );

    const results = await axe(container);
    expect(results).toHaveNoViolations();
    // axe check: role="dialog", aria-modal, aria-labelledby, focus trap, etc.
  });

  it('should trap focus within modal when open', async () => {
    const user = userEvent.setup();
    render(
      <Modal isOpen={true} onClose={vi.fn()} title="Dialog">
        <button>First</button>
        <button>Second</button>
      </Modal>
    );

    await user.tab();
    expect(screen.getByText('First')).toHaveFocus();

    await user.tab();
    expect(screen.getByText('Second')).toHaveFocus();

    // Tab from last focusable → wraps to first (focus trap)
    await user.tab();
    expect(screen.getByText('First')).toHaveFocus();

        <button>OK</button>
      </Modal>
    );

    expect(onClose).toHaveBeenCalled();
  });

  it('should return focus to trigger when closed', async () => {
    const { rerender } = render(
      <>
        <button id="trigger">Open Modal</button>
        <Modal isOpen={true} onClose={vi.fn()} title="Dialog">
          <button>Action</button>
        </Modal>
      </>
    );

    // Close modal
    rerender(

        <button id="trigger">Open Modal</button>
        <Modal isOpen={false} onClose={vi.fn()} title="Dialog">
          <button>Action</button>
```

## 🟢 8. Property-Based Testing (fast-check)

### Cơ chế thật

Thay vì viết specific examples, property-based testing **generates many random inputs** và kiểm tra **properties** (invariants) phải luôn đúng.

```typescript
import * as fc from 'fast-check';

// Unit test: specific examples
it('should sort array correctly', () => {
  expect(sortNumbers([3, 1, 2])).toEqual([1, 2, 3]); // 1 example
});

// Property-based: many random examples
it('should satisfy sorting properties', () => {
  fc.assert(
    fc.property(
      fc.array(fc.integer()), // Generate random arrays of integers
      (numbers) => {
        const sorted = sortNumbers(numbers);

        // Property 1: Output length same as input
        expect(sorted).toHaveLength(numbers.length);

        // Property 2: Output is ordered
        for (let i = 0; i < sorted.length - 1; i++) {
          expect(sorted[i]).toBeLessThanOrEqual(sorted[i + 1]);
        }

        // Property 3: Same elements (no duplicates added/removed)
        expect(sorted.slice().sort()).toEqual(numbers.slice().sort());
      },
    ),
  );
  // fast-check generates 100 random arrays and checks all properties
  // Khi tìm thấy failure, nó "shrinks" input xuống minimum failing case
});

// Useful cho: pure functions, business rules, data transformations
it('calculateDiscount should always return non-negative result', () => {
  fc.assert(
    fc.property(
      fc.float({ min: 0, max: 10000 }), // Random prices
      fc.float({ min: 0, max: 1 }), // Random discount rates 0-100%
      (price, discountRate) => {
        const result = applyDiscount(price, discountRate);
        expect(result).toBeGreaterThanOrEqual(0);
        expect(result).toBeLessThanOrEqual(price);
      },
    ),
  );
});
```

## Câu hỏi ôn tập (NotebookLM)

### Câu 1: Tại sao Testing Library ưu tiên `getByRole` hơn `getByTestId`, và điều này phản ánh triết lý gì của testing tốt?

**Đáp án:**

Testing Library's query priority phản ánh một triết lý cốt lõi: **tests nên verify behavior từ góc nhìn của user, không phải implementation details của developer**.

`getByRole` query theo **ARIA role và accessible name** — đây là cách **screen reader và assistive technology** navigate và interact với UI. Nó reflect cách user thật sự "thấy" interface.

```typescript
// getByRole: query theo accessible semantics
screen.getByRole('button', { name: /submit order/i });
// → Tìm element với role="button" và accessible name "Submit Order"
// → Đúng với: <button>Submit Order</button>
// → Đúng với: <div role="button" aria-label="Submit Order">...</div>
// → Sai với: <button aria-hidden="true">Submit Order</button> ← không accessible

// getByTestId: query theo implementation artifact
screen.getByTestId('submit-button');
// → Tìm element với data-testid="submit-button"
// → Hoàn toàn opaque — không nói lên gì về accessible behavior
// → data-testid là "fake" attribute chỉ tồn tại cho test
```

**Tại sao getByRole tốt hơn:**

1. **Tự động test accessibility:** Nếu `getByRole('button', { name: /submit/i })` không tìm thấy element, có thể vì button thiếu accessible label → bug đôi khi là a11y issue, không chỉ test issue.

2. **Resilient với refactor:** Nếu đổi từ `<button>` sang `<input type="submit">` → getByRole vẫn tìm thấy (cả hai có role button). getByTestId sẽ fail nếu data-testid bị remove.

3. **Documentation:** Test code đọc như user story: "Tìm button 'Submit Order' và click" thay vì "Tìm element có testid 'submit-btn-v2' và click".

**Khi nào getByTestId hợp lý:**

- Element không có semantic HTML role (decorative elements)
- Role không unique đủ để distinguish (nhiều buttons cùng tên)
- Testing infrastructure elements (loading spinners, error containers)

Triết lý tổng quát: "The more your tests resemble the way your software is used, the more confidence they can give you." Test từ outside-in (user perspective) → tests survive refactoring → tests có giá trị lâu dài.

### Câu 2: Điều gì xảy ra ở runtime khi Vitest chạy một test file, và tại sao mỗi test phải isolated (không share mutable state)?

**Đáp án:**

**Runtime execution của Vitest:**

Khi Vitest chạy, nó:

1. Tạo separate **worker thread** (hoặc process) cho mỗi test file → files run in parallel, không share memory
2. Import test file vào worker — module được evaluated một lần per worker
3. Collect `describe/it` callbacks — xây dựng test tree, chưa chạy
4. Chạy `beforeAll` hooks
5. Với mỗi test: chạy `beforeEach` → test callback → `afterEach`
6. Chạy `afterAll` hooks
7. Report results

```
Worker Thread 1: user.test.ts
  ├── Module evaluation (imports, module-level code)
  ├── beforeAll
  ├── Test 1: beforeEach → test → afterEach
  ├── Test 2: beforeEach → test → afterEach
  └── afterAll

Worker Thread 2: order.test.ts (chạy song song)
  ├── ...
```

**Tại sao shared mutable state phá vỡ tests:**

Tests trong cùng file chia sẻ module scope. Nếu state được declared ở module level và mutated trong tests:

```typescript
// ❌ Module-level mutable state — tests không isolated
const mockRepo = new InMemoryUserRepository();
// mockRepo được share giữa tất cả tests trong file

it('test A: creates user', async () => {
  await service.createUser({ email: 'alice@test.com' });
  expect(mockRepo.users).toHaveLength(1);
});

it('test B: finds no users initially', async () => {
  // Chạy AFTER test A → mockRepo.users đã có 1 user từ test A
  expect(mockRepo.users).toHaveLength(0); // FAIL — test B thấy state từ test A
});
```

Hậu quả của tests không isolated:

1. **Order-dependent:** Tests pass khi chạy theo thứ tự A→B, fail khi B→A
2. **Flaky when parallelized:** Vitest có thể run tests trong parallel, nếu test A và test B mutate cùng state concurrently → race condition
3. **Hard to debug:** Failure trong test B do test A gây ra → confusing
4. **False confidence:** Tất cả tests pass khi chạy cùng nhau, nhưng test B đơn lẻ fail → test suite không reliable

**Fix pattern:**

```typescript
// ✅ Fresh state cho mỗi test
describe('UserService', () => {
  let service: UserService;
  let repo: InMemoryUserRepository;

  beforeEach(() => {
    // Tạo mới trước mỗi test — isolated state
    repo = new InMemoryUserRepository();
    service = new UserService(repo, mockEmailService);
  });

  it('test A: creates user', async () => {
    await service.createUser({ email: 'alice@test.com' });
    expect(repo.users).toHaveLength(1);
  });

  it('test B: finds no users initially', async () => {
    // Fresh repo từ beforeEach → không bị ảnh hưởng bởi test A
    expect(repo.users).toHaveLength(0); // PASS
  });
});
```

**Vi.clearAllMocks() vs vi.resetAllMocks() vs vi.restoreAllMocks():**

```typescript
afterEach(() => {
  vi.clearAllMocks(); // Clear call history (calls, instances, results) — mock implementation giữ nguyên
  // vi.resetAllMocks(); // Clear history + reset implementation về vi.fn() empty
  // vi.restoreAllMocks(); // Restore original implementation (cho vi.spyOn)
});
```

### Câu 3: Đoạn test sau có những vấn đề gì, và làm sao sửa để test có giá trị hơn?

```typescript
describe('ProductCard', () => {
  it('renders correctly', () => {
    const product = {
      id: 'p1',
      name: 'Laptop',
      price: 999,
      imageUrl: '/laptop.jpg',
      stock: 5,
    };

    const { container } = render(<ProductCard product={product} />);
    expect(container).toMatchSnapshot();
  });

  it('add to cart works', async () => {
    const product = { id: 'p1', name: 'Laptop', price: 999, imageUrl: '/laptop.jpg', stock: 5 };
    const { getByTestId } = render(<ProductCard product={product} />);

    fireEvent.click(getByTestId('add-cart-btn'));

    expect(getByTestId('cart-count').textContent).toBe('1');
  });
});
```

**Đáp án:**

Code này có **5 vấn đề cụ thể:**

**Vấn đề 1: Snapshot test không có giá trị ở đây**

`renders correctly` với snapshot testing cho component nhỏ như `ProductCard` là anti-pattern. Snapshot sẽ include toàn bộ HTML structure. Mỗi lần đổi CSS class, thêm aria attribute, hay sửa text → snapshot fail → developer chỉ click "update" mà không thật sự review. Không verify được behavior nào cụ thể.

**Vấn đề 2: `getByTestId` thay vì accessible query**

`getByTestId('add-cart-btn')` query theo implementation artifact. Không biết button có accessible không, không tự động verify a11y.

**Vấn đề 3: `fireEvent.click` thay vì `userEvent.click`**

`fireEvent.click` chỉ dispatch một `click` event. Real user interaction là: `pointerover → pointerenter → mouseover → mouseenter → pointermove → mousemove → pointerdown → mousedown → focus → pointerup → mouseup → click`. `userEvent` simulate đầy đủ event sequence — bắt được bugs mà `fireEvent` bỏ qua.

**Vấn đề 4: Test implementation detail (cart-count)**

`getByTestId('cart-count')` test implementation. Nếu cart count được hiển thị ở header chứ không phải trong `ProductCard` → test fail dù behavior đúng. Test không phản ánh user flow thực tế.

**Vấn đề 5: Test name mơ hồ**

`'add to cart works'` không nói gì cụ thể. "Works" có nghĩa gì? Không readable như specification.

**Fix:**

```typescript
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { ProductCard } from './ProductCard';

describe('ProductCard', () => {
  const user = userEvent.setup();

  const mockProduct = {
    id: 'p1',
    name: 'Laptop Pro 15"',
    price: 999,
    imageUrl: '/laptop.jpg',
    stock: 5,
  };

  // ✅ Assert specific behavior thay vì snapshot
  it('should display product name and formatted price', () => {
    render(<ProductCard product={mockProduct} onAddToCart={vi.fn()} />);

    expect(screen.getByRole('heading', { name: /laptop pro 15"/i })).toBeInTheDocument();
    expect(screen.getByText('$999.00')).toBeInTheDocument();
    // Product image with alt text
    expect(screen.getByRole('img', { name: /laptop pro 15"/i })).toHaveAttribute('src', '/laptop.jpg');
  });

  // ✅ Test user behavior với accessible queries
  it('should call onAddToCart with product when "Add to Cart" is clicked', async () => {
    const onAddToCart = vi.fn();
    render(<ProductCard product={mockProduct} onAddToCart={onAddToCart} />);

    await user.click(screen.getByRole('button', { name: /add to cart/i }));

    expect(onAddToCart).toHaveBeenCalledOnce();
    expect(onAddToCart).toHaveBeenCalledWith(mockProduct);
  });

  // ✅ Test out-of-stock behavior — đây là business rule quan trọng
  it('should disable "Add to Cart" button when product is out of stock', () => {
    const outOfStockProduct = { ...mockProduct, stock: 0 };
    render(<ProductCard product={outOfStockProduct} onAddToCart={vi.fn()} />);

    expect(screen.getByRole('button', { name: /out of stock/i })).toBeDisabled();
  });

  // ✅ Test accessibility
  it('should have no accessibility violations', async () => {
    const { container } = render(<ProductCard product={mockProduct} onAddToCart={vi.fn()} />);
    const results = await axe(container);
    expect(results).toHaveNoViolations();
  });
});
```

Key differences:

- Specific assertions về visible user-facing content
- `onAddToCart` là prop — `ProductCard` không own cart state, hợp lý hơn về SRP
- Test business rules (out-of-stock) thay vì implementation (testid)
- Accessible queries verify a11y đồng thời
- Test names readable như specifications
