## Tại sao TypeScript là bắt buộc cho Senior+

TypeScript không phải "JavaScript với types". TypeScript là một **design tool** — nó encode business rules, constraints, và contracts vào type system. Khi types đúng, compiler làm việc thay cho code review.

> "Make impossible states impossible" — Richard Feldman

```typescript
// ❌ JavaScript: impossible states có thể xảy ra
function processOrder(order) {
  if (order.status === 'shipped') {
    sendTrackingEmail(order.trackingCode); // trackingCode có thể undefined — runtime crash
  }
}

// ✅ TypeScript: impossible states không compile được
type Order =
  | { status: 'pending'; trackingCode?: never }
  | { status: 'shipped'; trackingCode: string } // ← bắt buộc khi shipped
  | { status: 'delivered'; trackingCode: string };

function processOrder(order: Order) {
  if (order.status === 'shipped') {
    sendTrackingEmail(order.trackingCode); // TypeScript biết đây là string — safe
  }
}
// Compiler không cho phép create Order { status: 'shipped' } mà không có trackingCode
```

## 1. Type System Fundamentals — Structural Typing

### Cơ chế thật

TypeScript dùng **structural typing** (duck typing), không phải **nominal typing** (Java/C#). Hai types tương thích nếu chúng có cùng structure — không quan tâm tên.

```typescript
// Nominal typing (Java): phải khai báo explicitly
// class Dog extends Animal {} ← phải khai báo IS-A relationship

// Structural typing (TypeScript): shape-based compatibility
interface Point2D {
  x: number;
  y: number;
}
interface Coordinate {
  x: number;
  y: number;
}

// Hai interfaces khác tên nhưng CÙNG structure → compatible
const point: Point2D = { x: 1, y: 2 };
const coord: Coordinate = point; // ✓ — structural match

// Superset là compatible với subset
interface Point3D {
  x: number;
  y: number;
  z: number;
}

function distanceFrom2DOrigin(p: Point2D): number {
  return Math.sqrt(p.x ** 2 + p.y ** 2);
}

const p3: Point3D = { x: 1, y: 2, z: 3 };
distanceFrom2DOrigin(p3); // ✓ — Point3D có đủ { x, y } mà Point2D cần
// Point3D là subtype của Point2D trong structural type system
```

**Structural typing + excess property check — một exception quan trọng:**

```typescript
interface Config {
  host: string;
  port: number;
}

// ✓ Object literal assignment: TypeScript check excess properties
const config: Config = {
  host: 'localhost',
  port: 3000,
  timeout: 5000, // ← Error: 'timeout' does not exist in type 'Config'
};

// ✓ Variable assignment: NO excess property check (structural typing wins)
const options = { host: 'localhost', port: 3000, timeout: 5000 };
const config2: Config = options; // ✓ — no error, options có đủ {host, port}

// Tại sao: object literals có thể là typos ("timout" instead of "timeout")
// Variables có thể là intentional supersets
```

## 2. Generics — Cơ chế thật

### Cơ chế thật

Generic là **type parameter** — giống function parameter nhưng cho types. Thay vì hard-code `string` hay `number`, generic cho phép caller quyết định type.

**V8/compiler level:** Generic hoàn toàn bị erase ở runtime. TypeScript generics chỉ tồn tại ở compile time — không có overhead runtime.

```
Compile time:
  Array<string> → TypeScript kiểm tra type safety
         ↓
Runtime (JavaScript):
  Array → plain JavaScript array, không có generic info
```

```typescript
// Generic function — type parameter T
function identity<T>(value: T): T {
  return value;
}

// TypeScript infers T từ argument
const str = identity('hello'); // T inferred as 'hello' (literal type)
const num = identity(42); // T inferred as 42 (literal type)
const arr = identity([1, 2, 3]); // T inferred as number[]

// Explicit type argument khi inference không đủ
const result = identity<string>('hello'); // T explicitly = string

// Generic với constraints — T phải có property nhất định
function getLength<T extends { length: number }>(value: T): number {
  return value.length;
}

getLength('hello'); // ✓ — string có .length
getLength([1, 2, 3]); // ✓ — array có .length
getLength(42); // ✗ — number không có .length
```

### Generic với multiple type parameters

```typescript
// K extends keyof T — K phải là key của T
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
  // Return type T[K] = "indexed access type"
  // Nếu T = User và K = 'email', return type = string
  // Nếu T = User và K = 'age', return type = number
  // TypeScript tự động infer correct return type
}

interface User {
  id: string;
  name: string;
  age: number;
  isActive: boolean;
}

const user: User = { id: '1', name: 'John', age: 30, isActive: true };

const name = getProperty(user, 'name'); // type: string ✓
const age = getProperty(user, 'age'); // type: number ✓
const bad = getProperty(user, 'unknown'); // ✗ Error: Argument of type '"unknown"' is not assignable to parameter of type 'keyof User'
```

### Generic classes

```typescript
// Repository pattern với generics
interface Repository<T, ID = string> {
  findById(id: ID): Promise<T | null>;
  findAll(): Promise<T[]>;
  save(entity: T): Promise<T>;
  delete(id: ID): Promise<void>;
}

// Concrete implementation — specify T
class PrismaUserRepository implements Repository<User, string> {
  async findById(id: string): Promise<User | null> {
    return prisma.user.findUnique({ where: { id } });
  }
  async findAll(): Promise<User[]> {
    return prisma.user.findMany();
  }
  async save(user: User): Promise<User> {
    return prisma.user.upsert({
      where: { id: user.id },
      create: user,
      update: user,
    });
  }
  async delete(id: string): Promise<void> {
    await prisma.user.delete({ where: { id } });
  }
}

// Generic Result type — functional error handling
type Result<T, E = Error> =
  | { success: true; data: T }
  | { success: false; error: E };

function ok<T>(data: T): Result<T, never> {
  return { success: true, data };
}

function fail<E>(error: E): Result<never, E> {
  return { success: false, error };
}

// Generic helper functions
async function tryAsync<T>(fn: () => Promise<T>): Promise<Result<T>> {
  try {
    return ok(await fn());
  } catch (e) {
    return fail(e instanceof Error ? e : new Error(String(e)));
  }
}

// Usage — TypeScript infers T correctly
const result = await tryAsync(() => userRepo.findById('123'));
if (result.success) {
  console.log(result.data.name); // TypeScript biết data là User
} else {
  console.error(result.error.message);
}
```

### Default type parameters và Generic constraints

```typescript
// Default type parameter — giống default function parameter
interface ApiResponse<T = unknown> {
  data: T;
  status: number;
  message: string;
}

// Không specify T → T = unknown
const rawResponse: ApiResponse = {
  data: 'anything',
  status: 200,
  message: 'OK',
};

// Specify T → type-safe
const userResponse: ApiResponse<User> = {
  data: { id: '1', name: 'John' },
  status: 200,
  message: 'OK',
};

// Chaining constraints
function mergeObjects<T extends object, U extends object>(
  target: T,
  source: U,
): T & U {
  return { ...target, ...source };
}

const merged = mergeObjects({ name: 'John' }, { age: 30 });
// type: { name: string } & { age: number }
merged.name; // ✓
merged.age; // ✓
```

## 3. Utility Types — Production Usage

### Cơ chế thật

Utility types là **generic type transformations** built into TypeScript. Họ transform một type thành type khác. Dưới đây là cách họ được implement — hiểu implementation giúp bạn tự tạo custom utilities.

```typescript
// Partial<T> — tất cả properties trở thành optional
// Implementation:
type Partial<T> = {
  [P in keyof T]?: T[P];
  // [P in keyof T] — iterate mọi key của T
  // ? — make optional
  // T[P] — giữ nguyên value type
};

// Required<T> — tất cả properties trở thành required
type Required<T> = {
  [P in keyof T]-?: T[P]; // -? removes optional modifier
};

// Readonly<T> — tất cả properties trở thành readonly
type Readonly<T> = {
  readonly [P in keyof T]: T[P];
};

// Pick<T, K> — chỉ giữ lại K keys từ T
type Pick<T, K extends keyof T> = {
  [P in K]: T[P];
};

// Omit<T, K> — loại bỏ K keys khỏi T
type Omit<T, K extends keyof T> = Pick<T, Exclude<keyof T, K>>;

// Record<K, V> — object type với keys K và values V
type Record<K extends keyof any, V> = {
  [P in K]: V;
};

// Exclude<T, U> — loại bỏ U khỏi union T
type Exclude<T, U> = T extends U ? never : T;
// Exclude<'a' | 'b' | 'c', 'a'> = 'b' | 'c'

// Extract<T, U> — giữ lại phần T mà assign được cho U
type Extract<T, U> = T extends U ? T : never;
// Extract<'a' | 'b' | 'c', 'a' | 'd'> = 'a'

// NonNullable<T> — loại bỏ null và undefined
type NonNullable<T> = T extends null | undefined ? never : T;

// ReturnType<T> — lấy return type của function type T
type ReturnType<T extends (...args: any) => any> = T extends (
  ...args: any
) => infer R
  ? R
  : any;

// Parameters<T> — lấy parameter types của function T như tuple
type Parameters<T extends (...args: any) => any> = T extends (
  ...args: infer P
) => any
  ? P
  : never;
```

### Production patterns với Utility Types

```typescript
interface User {
  id: string; // readonly, không thể update
  email: string; // có thể update
  name: string; // có thể update
  role: 'admin' | 'user';
  createdAt: Date; // readonly
  passwordHash: string; // không bao giờ expose ra ngoài
}

// DTO types — không cần redefine, derive từ User

// CreateUserDTO: bỏ server-generated fields
type CreateUserDTO = Omit<
  User,
  'id' | 'createdAt' | 'role' | 'passwordHash'
> & {

};
// Result: { email: string; name: string; password: string }

// UpdateUserDTO: tất cả optional ngoại trừ id
type UpdateUserDTO = Partial<
  Omit<User, 'id' | 'createdAt' | 'passwordHash'>
> & {
  id: string;
};
// Result: { id: string; email?: string; name?: string; role?: 'admin' | 'user' }

// PublicUser: expose ra client, không có sensitive fields
type PublicUser = Omit<User, 'passwordHash'>;

// Readonly admin user — không thể mutate sau khi fetched
type ReadonlyUser = Readonly<User>;

// Specific role variants
type AdminUser = User & { role: 'admin' };

// Record for lookup tables
type UserLookup = Record<string, PublicUser>; // key: userId, value: PublicUser
const userCache: UserLookup = {};

// Function type utilities
async function createUser(dto: CreateUserDTO): Promise<User> {
  // ...
}

type CreateUserInput = Parameters<typeof createUser>[0]; // = CreateUserDTO
type CreateUserOutput = Awaited<ReturnType<typeof createUser>>; // = User
// Awaited<> unwraps Promise — Awaited<Promise<User>> = User
```

**Discriminated Union với utility types:**

```typescript
// State machine types
type RequestState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; error: Error };

// Extract specific states
type SuccessState<T> = Extract<RequestState<T>, { status: 'success' }>;
// Result: { status: 'success'; data: T }

type LoadingOrError = Extract<
  RequestState<unknown>,
  { status: 'loading' | 'error' }

// Result: { status: 'loading' } | { status: 'error'; error: Error }

// Utility: get data type từ success state

  : never;
type UserData = SuccessData<RequestState<User>>; // = User
```

## 4. Union Types, Intersection Types, Discriminated Unions

### Union Types — A | B

```typescript
// Union: value là một trong các types
type StringOrNumber = string | number;
type Status = 'active' | 'inactive' | 'suspended';

// Discriminated Union — union với literal type discriminant
// Đây là pattern quan trọng nhất trong TypeScript cho state modeling

type Shape =
  | { kind: 'circle'; radius: number }
  | { kind: 'rectangle'; width: number; height: number }
  | { kind: 'triangle'; base: number; height: number };

function calculateArea(shape: Shape): number {
  switch (shape.kind) {
    case 'circle':
      // TypeScript biết shape là { kind: 'circle'; radius: number }
      return Math.PI * shape.radius ** 2;

    case 'rectangle':
      // TypeScript biết shape là { kind: 'rectangle'; width: number; height: number }
      return shape.width * shape.height;

    case 'triangle':
      return 0.5 * shape.base * shape.height;
  }
  // TypeScript yêu cầu tất cả cases phải handle
  // Nếu thêm 'pentagon' vào Shape union → TypeScript error ở đây (exhaustiveness)
}

// Exhaustiveness check — đảm bảo mọi case được handle
function assertNever(x: never): never {
  throw new Error(`Unexpected value: ${JSON.stringify(x)}`);
}

function calculateArea2(shape: Shape): number {
  switch (shape.kind) {
    case 'circle':
      return Math.PI * shape.radius ** 2;
    case 'rectangle':
      return shape.width * shape.height;
    case 'triangle':
      return 0.5 * shape.base * shape.height;
    default:
      return assertNever(shape);
    // Nếu thêm 'pentagon' → TypeScript error: 'pentagon' is not assignable to 'never'
    // Compiler force bạn handle case mới
  }
}
```

### Intersection Types — A & B

```typescript
// Intersection: value phải satisfy cả hai types
type Timestamped = { createdAt: Date; updatedAt: Date };

type StringToNumber = (x: string) => number;

type Converter = StringToNumber & NumberToString;

// Real-world: mixin pattern với intersections

type WithMetrics = {
  track(event: string, data?: object): void;

type EnhancedService = UserService & WithLogging & WithMetrics;
```

## 5. Type Narrowing — Cơ chế thật

### Cơ chế thật

Type narrowing là cách TypeScript **thu hẹp** type của variable từ broad sang specific dựa trên control flow analysis. TypeScript theo dõi tất cả possible code paths.

**Trong compiler:** TypeScript xây dựng **Control Flow Graph** — directed graph của code paths. Tại mỗi node, TypeScript tính toán "narrowed type" dựa trên guards đã pass.

```
function process(value: string | number | null) {
                               ↑ Type = string | number | null

  if (value === null) {
    // Type = null (narrowed)
    return;
  }
                               ↑ Type = string | number (null eliminated)

  if (typeof value === 'string') {
    // Type = string (narrowed)
    console.log(value.toUpperCase()); // ✓
  } else {
    // Type = number (narrowed — only remaining)
    console.log(value.toFixed(2)); // ✓
  }
}
```

```typescript
// 6 loại type guards — mỗi loại narrow khác nhau

// 1. typeof — narrow primitives
function processInput(input: string | number | boolean) {
  if (typeof input === 'string') {
    // input: string
    return input.toLowerCase();
  }
  if (typeof input === 'number') {
    // input: number
    return input.toFixed(2);
  }
  // input: boolean
  return input ? 'yes' : 'no';
}

// 2. instanceof — narrow class instances
function formatDate(date: Date | string): string {
  if (date instanceof Date) {
    // date: Date
    return date.toISOString();
  }
  // date: string
  return new Date(date).toISOString();
}

// 3. in operator — narrow by property existence
type Dog = { bark(): void; fetch(): void };
type Cat = { meow(): void; purr(): void };
type Pet = Dog | Cat;

function interactWithPet(pet: Pet) {
  if ('bark' in pet) {
    // pet: Dog
    pet.bark();
  } else {
    // pet: Cat
    pet.meow();
  }
}

// 4. Literal type narrowing — equality check
type Direction = 'north' | 'south' | 'east' | 'west';

function getOpposite(dir: Direction): Direction {
  if (dir === 'north') return 'south';
  if (dir === 'south') return 'north';
  if (dir === 'east') return 'west';
  return 'east'; // TypeScript biết dir phải là 'west' ở đây
}

// 5. Discriminated union narrowing — switch trên literal property
type ApiEvent =
  | { type: 'USER_CREATED'; userId: string; email: string }
  | { type: 'ORDER_PLACED'; orderId: string; total: number }
  | { type: 'PAYMENT_FAILED'; orderId: string; reason: string };

function handleEvent(event: ApiEvent) {
  switch (event.type) {
    case 'USER_CREATED':
      // event: { type: 'USER_CREATED'; userId: string; email: string }
      sendWelcomeEmail(event.email);
      break;
    case 'ORDER_PLACED':
      // event: { type: 'ORDER_PLACED'; orderId: string; total: number }
      updateInventory(event.orderId);
      break;
    case 'PAYMENT_FAILED':
      // event: { type: 'PAYMENT_FAILED'; orderId: string; reason: string }
      notifySupport(event.orderId, event.reason);
      break;
  }
}

// 6. Custom type guard — user-defined narrowing
// Return type "value is SomeType" = type predicate
function isUser(value: unknown): value is User {
  return (
    typeof value === 'object' &&
    value !== null &&
    'id' in value &&
    'email' in value &&
    typeof (value as User).id === 'string' &&
    typeof (value as User).email === 'string'
  );
}

function processApiResponse(data: unknown) {
  if (isUser(data)) {
    // data: User — narrowed by custom guard
    console.log(data.email); // ✓
  }
}

// Assertion function — throw nếu guard fail
function assertIsUser(value: unknown): asserts value is User {
  if (!isUser(value)) {
    throw new TypeError('Expected User object');
  }
}

function mustProcessUser(data: unknown) {
  assertIsUser(data); // Throw nếu không phải User
  // data: User từ đây trở đi
  console.log(data.name);
}
```

## 🟡 6. Conditional Types

### Cơ chế thật

Conditional types là **ternary operator cho types**: `T extends U ? X : Y`. Đây là meta-programming — viết logic chạy tại compile time.

```typescript
// Basic conditional type
type IsString<T> = T extends string ? true : false;

type A = IsString<string>; // true
type B = IsString<number>; // false
type C = IsString<'hello'>; // true — 'hello' extends string

// Conditional types với union — distributes over union members
type ToArray<T> = T extends unknown ? T[] : never;

type StringOrNumberArray = ToArray<string | number>;
// TypeScript distributes: ToArray<string> | ToArray<number>
// = string[] | number[]
// NOT: (string | number)[]

// Để prevent distribution — wrap in tuple
type ToArrayNonDistributive<T> = [T] extends [unknown] ? T[] : never;
type Combined = ToArrayNonDistributive<string | number>;
// = (string | number)[]

// infer — extract type từ another type
// "Infer" = "deduce the type at this position"

// Extract return type (đây là cách ReturnType<T> được implement)
type GetReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

type FnReturn = GetReturnType<() => string>; // string
type AsyncReturn = GetReturnType<() => Promise<User>>; // Promise<User>
type NonFn = GetReturnType<string>; // never

// Unwrap Promise với infer
type Awaited<T> = T extends Promise<infer U> ? Awaited<U> : T;
// Recursive: Awaited<Promise<Promise<string>>> = string

// Extract element type từ array
type ElementType<T> = T extends (infer E)[] ? E : never;
type Elem = ElementType<User[]>; // User

// Extract first element của tuple
type Head<T extends any[]> = T extends [infer H, ...any[]] ? H : never;
type FirstArg = Head<[string, number, boolean]>; // string

// Extract rest của tuple
type Tail<T extends any[]> = T extends [any, ...infer R] ? R : never;
type RestArgs = Tail<[string, number, boolean]>; // [number, boolean]
```

**Conditional types trong production:**

```typescript
// Deep Readonly — recursive conditional type
type DeepReadonly<T> = {
  readonly [K in keyof T]: T[K] extends object ? DeepReadonly<T[K]> : T[K];
};

interface AppConfig {
  database: { host: string; port: number };
  cache: { ttl: number; maxSize: number };
  features: { darkMode: boolean; betaFeatures: string[] };
}

type ReadonlyConfig = DeepReadonly<AppConfig>;
// Tất cả nested objects đều readonly — không thể mutate config sau khi loaded

// Flatten nested object type
type Flatten<T> = T extends Array<infer U> ? U : T;

type NestedArray = number[][];
type Flat1 = Flatten<NestedArray>; // number[] — flatten 1 level
type Flat2 = Flatten<Flat1>; // number — flatten again

// Conditional function overloads
type Stringify<T> = T extends string
  ? T
  : T extends number
    ? `${T}`
    : T extends boolean
      ? `${T}`
      : T extends null
        ? 'null'
        : T extends undefined
          ? 'undefined'
          : string;

function stringify<T>(value: T): Stringify<T> {
  return String(value) as Stringify<T>;
}

const s1 = stringify('hello'); // type: 'hello' (literal preserved)
const s2 = stringify(42); // type: '42'
const s3 = stringify(true); // type: 'true'
```

## 🟡 7. Template Literal Types

### Cơ chế thật

Template literal types là **string manipulation ở type level** — TypeScript có thể concatenate, split, và transform string literal types tại compile time.

```typescript
// Basic template literal type
type Greeting = `Hello, ${string}!`;
const g1: Greeting = 'Hello, World!'; // ✓
const g2: Greeting = 'Hello, TypeScript!'; // ✓
const g3: Greeting = 'Hi, World!'; // ✗ — must start with "Hello, "

// Combine với union — Cartesian product
type Color = 'red' | 'green' | 'blue';
type Size = 'sm' | 'md' | 'lg';
type ButtonVariant = `${Color}-${Size}`;
// = 'red-sm' | 'red-md' | 'red-lg' | 'green-sm' | ... (9 variants)

// CSS utility type pattern
type CSSProperty =
  | `margin-${'top' | 'right' | 'bottom' | 'left'}`
  | `padding-${'top' | 'right' | 'bottom' | 'left'}`
  | `border-${'top' | 'right' | 'bottom' | 'left'}-${'width' | 'style' | 'color'}`;

// Event handler naming convention
type EventName = 'click' | 'focus' | 'blur' | 'change';
type HandlerName = `on${Capitalize<EventName>}`;
// = 'onClick' | 'onFocus' | 'onBlur' | 'onChange'

// Type-safe object property access với dot notation
type DeepKeyOf<T, Prefix extends string = ''> = {
  [K in keyof T & string]: T[K] extends object
    ? `${Prefix}${K}` | DeepKeyOf<T[K], `${Prefix}${K}.`>
    : `${Prefix}${K}`;
}[keyof T & string];

interface Config {
  database: { host: string; port: number };
  app: { name: string; debug: boolean };
}

type ConfigKeys = DeepKeyOf<Config>;
// = 'database' | 'database.host' | 'database.port' | 'app' | 'app.name' | 'app.debug'

// Get type at dot-notation path
type DeepGet<T, Path extends string> = Path extends `${infer Key}.${infer Rest}`
  ? Key extends keyof T
    ? DeepGet<T[Key], Rest>
    : never
  : Path extends keyof T
    ? T[Path]
    : never;

type DatabaseHost = DeepGet<Config, 'database.host'>; // string
type AppDebug = DeepGet<Config, 'app.debug'>; // boolean

// Type-safe route params extraction
type ExtractRouteParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? { [K in Param | keyof ExtractRouteParams<Rest>]: string }
    : T extends `${string}:${infer Param}`
      ? { [K in Param]: string }
      : {};

type UserRouteParams = ExtractRouteParams<'/users/:userId/posts/:postId'>;
// = { userId: string; postId: string }

function createRoute<T extends string>(
  path: T,
  params: ExtractRouteParams<T>,
): string {
  return Object.entries(params).reduce(
    (acc, [key, value]) => acc.replace(`:${key}`, value as string),
    path,
  );
}

const url = createRoute('/users/:userId/posts/:postId', {
  userId: '123', // ✓ — required
  postId: '456', // ✓ — required
  // extra: 'oops' ← TypeScript error: not in route params
});
```

## 🟡 8. Mapped Types

### Cơ chế thật

Mapped types transform mọi property của một type, tạo ra type mới. Chúng là nền tảng của utility types như `Partial`, `Required`, `Readonly`.

```typescript
// Basic mapped type syntax
type Mapped<T> = {
  [K in keyof T]: T[K]; // Iterate mọi key K của T, giữ nguyên value type
};

// Modifiers: add/remove optional và readonly
type AllOptional<T> = { [K in keyof T]?: T[K] }; // Add ?
type AllRequired<T> = { [K in keyof T]-?: T[K] }; // Remove ?
type AllReadonly<T> = { readonly [K in keyof T]: T[K] };
type AllMutable<T> = { -readonly [K in keyof T]: T[K] }; // Remove readonly

// Remap keys với `as` clause
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};

interface UserData {
  id: string;
  name: string;
  age: number;
}

type UserGetters = Getters<UserData>;
// = {
//   getId: () => string;
//   getName: () => string;
//   getAge: () => number;
// }

// Filter properties với `as never`
type OnlyStrings<T> = {
  [K in keyof T as T[K] extends string ? K : never]: T[K];
};

interface Mixed {
  name: string;
  age: number;
  email: string;
  isActive: boolean;
}

type StringFields = OnlyStrings<Mixed>;
// = { name: string; email: string }
// age và isActive bị filter ra vì không phải string

// Event map — type-safe event system
type EventHandlers<Events extends Record<string, unknown>> = {
  [K in keyof Events as `on${Capitalize<string & K>}`]?: (
    data: Events[K],
  ) => void;
};

type AppEvents = {
  login: { userId: string; timestamp: number };
  logout: { userId: string };
  error: { message: string; code: number };
};

type AppEventHandlers = EventHandlers<AppEvents>;
// = {
//   onLogin?: (data: { userId: string; timestamp: number }) => void;
//   onLogout?: (data: { userId: string }) => void;
//   onError?: (data: { message: string; code: number }) => void;
// }

// Selective mapped type — combine với conditional
type DeepPartial<T> = {
  [K in keyof T]?: T[K] extends object ? DeepPartial<T[K]> : T[K];
};

interface ComplexConfig {
  server: { host: string; port: number; ssl: boolean };
  database: { url: string; poolSize: number };

}

  base: ComplexConfig,
```

## 🟡 9. Variance — Covariance và Contravariance

### Cơ chế thật

Variance mô tả cách **subtyping relationship** của container type liên quan tới subtyping relationship của element type.

```
Covariant:     Animal ← Dog (Dog IS-A Animal, narrow)
               Array<Animal> ← Array<Dog>  (same direction)

Contravariant: Animal ← Dog
               Fn<Dog> ← Fn<Animal>  (opposite direction)

Invariant:     Fn<Animal, Dog> — không có safe substitution

Bivariant:     Both directions (unsafe, TypeScript method parameters)
```

**Tại sao quan trọng:** Hiểu variance giúp bạn thiết kế API types đúng và hiểu tại sao TypeScript báo lỗi ở một số chỗ không ngờ.

```typescript
// COVARIANCE — return types
// Nếu B extends A, thì () => B có thể assign cho () => A
// (Hàm trả B có thể dùng thay cho hàm trả A — vì B IS-A A)

class Animal {
  name: string = '';
}
class Dog extends Animal {
  breed: string = '';
}

type GetAnimal = () => Animal;
type GetDog = () => Dog;

const getDog: GetDog = () => new Dog();
const getAnimal: GetAnimal = getDog; // ✓ — Covariant! Dog IS-A Animal

// CONTRAVARIANCE — parameter types
// Nếu B extends A, thì (a: A) => void có thể assign cho (b: B) => void
// (Hàm nhận Animal rộng hơn → cũng nhận được Dog)

type HandleAnimal = (a: Animal) => void;
type HandleDog = (d: Dog) => void;

const handleAnimal: HandleAnimal = (a) => {
  console.log(a.name);
};

// TypeScript strict mode: function parameters are CONTRAVARIANT
// handleAnimal nhận Animal (rộng hơn Dog) → an toàn khi pass Dog vào
const handleDog: HandleDog = handleAnimal; // ✓ — Contravariant in strict mode

// Tại sao ngược chiều?
// handleDog được gọi với Dog objects
// handleAnimal chỉ dùng .name (Animal property) → an toàn với Dog
// Nếu ngược lại: HandleAnimal = handleDog → có thể pass Animal vào hàm expect Dog → crash

// PRACTICAL EXAMPLE: Callback types
interface EventEmitter<T> {
  on(handler: (data: T) => void): void;
  emit(data: T): void;
}

// Contravariance trong callback: handler nhận Dog event
const dogEmitter: EventEmitter<Dog> = createEmitter();

// Callback expect Animal (broader) có thể xử lý Dog events safely
const animalHandler: (animal: Animal) => void = (a) => console.log(a.name);
dogEmitter.on(animalHandler); // ✓ — Safe (contravariant parameter)
```

```typescript
// CHALLENGE 1: Type-safe EventEmitter
// Đảm bảo handler type match event type

type EventMap = {
  'user:login': { userId: string; timestamp: number };
  'user:logout': { userId: string };
  'order:created': { orderId: string; total: number };
};

class TypedEventEmitter<TEvents extends Record<string, unknown>> {
  private handlers = new Map<string, Set<Function>>();

  on<K extends keyof TEvents>(
    event: K,
    handler: (data: TEvents[K]) => void,
  ): () => void {
    if (!this.handlers.has(event as string)) {
      this.handlers.set(event as string, new Set());
    }
    this.handlers.get(event as string)!.add(handler);
    return () => this.off(event, handler);
  }

  off<K extends keyof TEvents>(
    event: K,
    handler: (data: TEvents[K]) => void,
  ): void {
    this.handlers.get(event as string)?.delete(handler);
  }

  emit<K extends keyof TEvents>(event: K, data: TEvents[K]): void {
    this.handlers.get(event as string)?.forEach((h) => h(data));
  }
}
```

## Câu hỏi ôn tập (NotebookLM)

### Câu 1: TypeScript structural typing là gì, tại sao nó khác nominal typing, và khi nào structural typing gây ra vấn đề ở production?

**Đáp án:**

**Structural typing** quyết định type compatibility dựa trên **shape** (structure) của type, không dựa trên **name** hay **explicit declaration**. Hai types tương thích nếu chúng có cùng properties với cùng types — bất kể tên của chúng là gì.

**Nominal typing** (Java, C#, Rust) quyết định compatibility dựa trên **explicit type identity**. Hai classes khác tên là **không tương thích** dù có cùng structure.

```typescript
// Structural typing (TypeScript):
interface UserId {
  value: string;
}
interface OrderId {
  value: string;
}

function processUser(id: UserId) {
  /* ... */
}

const orderId: OrderId = { value: 'order-123' };
processUser(orderId); // ✓ — cùng shape → compatible
// TypeScript không quan tâm tên "OrderId" vs "UserId"

// Nominal typing (hypothetical):
// processUser(orderId); // ✗ — khác type name → không compatible
```

**Khi structural typing gây vấn đề ở production:**

**Vấn đề 1: ID type mixing**

```typescript
// Tất cả IDs đều là string → structural typing cho phép mix
type UserId = string;
type OrderId = string;

async function getOrdersByUser(userId: UserId): Promise<Order[]> {
  return db.orders.findMany({ where: { userId } });
}

const orderId: OrderId = 'order-123';
getOrdersByUser(orderId); // ✓ TypeScript — ✗ Logic: wrong ID passed
// Runtime: SQL query với orderId thay vì userId → empty result hoặc wrong data
// Không có compile error, không có runtime error, chỉ có wrong behavior
```

**Vấn đề 2: Currency/unit mixing**

```typescript
type Euros = number;
type USD = number;
type Kilograms = number;
type Pounds = number;

function priceInEuros(amount: Euros): string {
  return `€${amount}`;
}

const priceInDollars: USD = 100;
priceInEuros(priceInDollars); // ✓ TypeScript — ✗ Logic
// Hiển thị €100 thay vì €91 (sau khi convert) — silent financial bug
```

**Vấn đề 3: DTO confusion**

```typescript
interface CreateUserDTO {
  email: string;
  name: string;
}
interface UpdateUserDTO {
  email: string;
  name: string;
} // Same shape

function createUser(dto: CreateUserDTO): Promise<User> {
  /* ... */
}

const updatePayload: UpdateUserDTO = {
  email: 'new@email.com',
  name: 'New Name',
};
createUser(updatePayload); // ✓ TypeScript — nhưng có thể là logic error
```

**Fix: Branded Types** (như đã trình bày) tạo nominal typing trong structural type system:

```typescript
type UserId = Brand<string, 'UserId'>;
type OrderId = Brand<string, 'OrderId'>;

function getOrdersByUser(userId: UserId): Promise<Order[]> {
  /* ... */
}

const orderId = OrderId('order-123');
getOrdersByUser(orderId); // ✗ TypeScript Error: OrderId không assign được cho UserId ✓
```

Branded types là pattern quan trọng trong domain-driven design với TypeScript — encode domain constraints vào type system.

### Câu 2: `infer` keyword trong conditional types hoạt động như thế nào, và tại sao nó mạnh mẽ trong việc extract type information?

**Đáp án:**

`infer` là một keyword chỉ dùng trong **conditional types** để **declare a type variable** ở vị trí mà TypeScript sẽ tự động "infer" (deduce) type đó khi condition được evaluated.

**Cơ chế:**

```typescript
type GetReturnType<T> = T extends (...args: any[]) => infer R ? R : never;
//                                                    ^^^^^^^^
//                      "Khi T là function, let R = whatever the return type is"
//                      R không được declare trước — TypeScript tự tìm ra
```

Khi TypeScript evaluate `GetReturnType<() => string>`:

1. Check: `() => string extends (...args: any[]) => infer R`? YES
2. TypeScript "captures" return type vào R: R = string
3. Condition true → return R = string

`infer` cho phép bạn "reach into" một type và extract một phần của nó mà không cần biết trước type đó là gì.

**Tại sao mạnh mẽ — 5 use cases production:**

**1. Extract từ function types:**

```typescript
// Parameters và ReturnType utility types dùng infer
type Parameters<T> = T extends (...args: infer P) => any ? P : never;
type ReturnType<T> = T extends (...args: any) => infer R ? R : never;

// Thực tế: type-safe wrapper functions
function withLogging<T extends (...args: any[]) => any>(fn: T) {
  return (...args: Parameters<T>): ReturnType<T> => {
    console.log('calling', fn.name, 'with', args);
    return fn(...args);
  };
}
// Parameters<T> và ReturnType<T> giữ exact types từ original function
```

**2. Unwrap nested types:**

```typescript
// Unwrap Promise — Awaited utility type
type Awaited<T> = T extends Promise<infer U> ? Awaited<U> : T;
// Awaited<Promise<Promise<string>>> → Awaited<Promise<string>> → Awaited<string> → string

// Unwrap array
type UnpackArray<T> = T extends Array<infer U> ? U : T;
type Num = UnpackArray<number[]>; // number
type Str = UnpackArray<string>; // string — unchanged (not an array)
```

**3. Tuple manipulation:**

```typescript
// First element
type Head<T extends any[]> = T extends [infer H, ...any[]] ? H : never;

// Last element
type Last<T extends any[]> = T extends [...any[], infer L] ? L : never;

// Remove first element
type Tail<T extends any[]> = T extends [any, ...infer R] ? R : [];

// Practical: function composition types
type Compose<Fns extends ((x: any) => any)[]> = Fns extends [
  (x: infer A) => infer B,
  ...infer Rest,
]
  ? Rest extends ((x: any) => any)[]
    ? (
        x: A,
      ) => Last<[B, ...{ [K in keyof Rest]: ReturnType<Rest[K] & Function> }]>
    : never
  : never;
```

**4. Extract from string patterns:**

```typescript
// Extract route parameters
type RouteParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? { [K in Param | keyof RouteParams<`/${Rest}`>]: string }
    : T extends `${string}:${infer Param}`
      ? { [K in Param]: string }
      : {};

type Params = RouteParams<'/users/:userId/posts/:postId'>;
// = { userId: string; postId: string }
```

**5. Conditional overloading:**

```typescript
// Return different types based on input type
type DeepValue<T, K extends string> = K extends `${infer Key}.${infer Rest}`
  ? Key extends keyof T
    ? DeepValue<T[Key], Rest>
    : never
  : K extends keyof T
    ? T[K]
    : never;

function get<T, K extends string>(obj: T, path: K): DeepValue<T, K> {
  return path.split('.').reduce((acc: any, key) => acc?.[key], obj);
}

const config = { db: { host: 'localhost', port: 5432 } };
const host = get(config, 'db.host'); // type: string ✓
const port = get(config, 'db.port'); // type: number ✓
const bad = get(config, 'db.xyz'); // type: never ✓ — path doesn't exist
```

`infer` essentially là **pattern matching cho types** — giống như destructuring nhưng ở type level, cho phép TypeScript "open up" một type và extract parts of it.

### Câu 3: Đoạn TypeScript sau có vấn đề gì về type safety, và làm sao thiết kế lại để compiler bắt được bugs?

```typescript
type UserRole = 'admin' | 'user' | 'moderator';

interface Permission {
  resource: string;
  action: string;
}

const rolePermissions: Record<string, Permission[]> = {
  admin: [{ resource: 'users', action: 'delete' }],
  user: [{ resource: 'profile', action: 'read' }],
  moderator: [{ resource: 'posts', action: 'delete' }],
};

function hasPermission(
  role: UserRole,
  resource: string,
  action: string,
): boolean {
  const permissions = rolePermissions[role];
  return (
    permissions?.some((p) => p.resource === resource && p.action === action) ??
    false
  );
}

// Caller
if (hasPermission(currentUser.role, 'users', 'delet')) {
  // Typo: 'delet' instead of 'delete' — no compile error, always returns false
  deleteUser(userId);
}
```

**Đáp án:**

Code này có **4 vấn đề type safety:**

**Vấn đề 1: `Record<string, Permission[]>` quá broad**

`string` key thay vì `UserRole` — rolePermissions có thể có keys không tồn tại trong `UserRole` hoặc thiếu keys của `UserRole`. TypeScript không enforce exhaustiveness.

**Vấn đề 2: `resource` và `action` là `string` — không bắt typos**

`'delet'` thay vì `'delete'` là valid string → không có compile error → silent bug: permission check luôn false → feature silently blocked.

**Vấn đề 3: Không có exhaustiveness check khi thêm role mới**

Thêm `'superadmin'` vào `UserRole` → `rolePermissions` không cần update → `hasPermission('superadmin', ...)` trả `false` cho mọi thứ → silent regression.

**Vấn đề 4: `Permission` type quá generic**

`resource: string; action: string` không encode relationship — `{ resource: 'users', action: 'read_profile' }` là valid type nhưng invalid business rule (action không thuộc resource).

**Fix — encode domain constraints vào type system:**

```typescript
// 1. Define specific resources và their allowed actions
type ResourceActions = {
  users: 'read' | 'create' | 'update' | 'delete';
  profile: 'read' | 'update';
  posts: 'read' | 'create' | 'update' | 'delete' | 'moderate';
  reports: 'read' | 'dismiss';
};

type Resource = keyof ResourceActions;
type ActionFor<R extends Resource> = ResourceActions[R];

// 2. Typed permission — resource/action pair is validated
type Permission<R extends Resource = Resource> = {
  resource: R;
  action: ActionFor<R>;
};

// Helper union of all valid permission pairs
type AnyPermission = {
  [R in Resource]: Permission<R>;
}[Resource];

// 3. Role permissions — Record keyed by UserRole (exhaustive)
type UserRole = 'admin' | 'user' | 'moderator';

// Record<UserRole, ...> enforces all roles must be present
const rolePermissions: Record<UserRole, AnyPermission[]> = {
  admin: [
    { resource: 'users', action: 'delete' }, // ✓
    { resource: 'users', action: 'create' }, // ✓
    { resource: 'posts', action: 'moderate' }, // ✓
  ],
  user: [
    { resource: 'profile', action: 'read' }, // ✓
    { resource: 'profile', action: 'update' }, // ✓
  ],
  moderator: [
    { resource: 'posts', action: 'moderate' }, // ✓
    { resource: 'reports', action: 'dismiss' }, // ✓
  ],
  // ← Nếu thêm 'superadmin' vào UserRole mà không add vào đây → TypeScript Error
};

// 4. Type-safe hasPermission
function hasPermission<R extends Resource>(
  role: UserRole,
  resource: R,
  action: ActionFor<R>, // ← Action phải match resource!
): boolean {
  const permissions = rolePermissions[role];
  return permissions.some(
    (p) => p.resource === resource && p.action === action,
  );
}

// Usage — compiler bắt tất cả bugs

hasPermission(currentUser.role, 'users', 'delete'); // ✓
hasPermission(currentUser.role, 'users', 'delet'); // ✗ Error: 'delet' not in 'read'|'create'|'update'|'delete'
hasPermission(currentUser.role, 'users', 'moderate'); // ✗ Error: 'moderate' not valid for 'users' resource
hasPermission(currentUser.role, 'unknown', 'read'); // ✗ Error: 'unknown' not in Resource

// Nếu add 'superadmin' vào UserRole mà không update rolePermissions:
// ✗ Error: Property 'superadmin' is missing in type Record<UserRole, ...>
```

Kết quả: tất cả 4 vấn đề được compiler bắt. `'delet'` typo → compile error. Missing role → compile error. Invalid action cho resource → compile error. Không cần runtime tests để catch những bugs này nữa.
