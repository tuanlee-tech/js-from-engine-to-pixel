# Phase 1.3 — Event Loop: Cơ chế thật từ Call Stack đến Microtask

---

## 1. Cơ chế thật

### Vấn đề gốc: JS là single-threaded

JavaScript chạy trên **một thread duy nhất**. Chỉ có một Call Stack. Chỉ một đoạn code chạy tại một thời điểm. Không có parallelism trong JS engine (trừ Workers).

Vậy tại sao `fetch()`, `setTimeout()`, event listeners không block toàn bộ chương trình?

**Câu trả lời:** Những operations đó không chạy trong JS engine — chúng chạy trong **browser/runtime APIs** trên các threads riêng. JS engine chỉ được thông báo khi chúng xong việc, thông qua **queues**. Event Loop là cơ chế điều phối thông báo đó.

---

### Kiến trúc đầy đủ — không phải "event loop đơn giản"

Hầu hết tài liệu vẽ event loop quá đơn giản. Đây là kiến trúc thật:

```
┌─────────────────────────────────────────────────────────────────┐
│                     BROWSER / NODE RUNTIME                      │
│                                                                 │

│   │   │              │    │   objects, closures, fns   │    │   │
│   │   │  [frame N]   │    │                            │    │   │
```

### Call Stack — cơ chế chi tiết

Call Stack chứa **Execution Contexts** (đã học ở 1.2). Quy tắc cứng:

- **Push** khi function được gọi
- **Pop** khi function return hoặc throw
- Nếu Stack **empty** → Event Loop có thể enqueue task tiếp theo

```javascript
console.log('A'); // Push Global EC, push console.log → pop → "A"

setTimeout(() => {
  // Push setTimeout → Web API nhận callback → pop
  console.log('B'); // Callback vào Macrotask Queue
}, 0);

console.log('C'); // "C"

// Stack empty → Event Loop: drain microtasks (none) → run one macrotask
// → callback pop ra từ queue → push lên stack → "B" → pop
// Output: A, C, B
```

**Tại sao `setTimeout(fn, 0)` không chạy ngay:** `0ms` chỉ là minimum delay hint. Browser/Node giao callback cho Web API timer. Timer fire ngay sau 0ms và đẩy callback vào **Macrotask Queue**. Nhưng callback chỉ được chạy khi Call Stack **empty** và Event Loop pick nó up. Nếu Call Stack đang bận → callback chờ.

---

### Macrotask Queue — "Task Queue" trong spec

Spec gọi đây là **Task Queue**. Mỗi lần Event Loop, chỉ **một task** được lấy ra và chạy đến hoàn thành. Không interrupt giữa chừng.

Nguồn tạo macrotasks:

- `setTimeout` / `setInterval` callbacks
- I/O callbacks (file read, network response trong Node)
- `MessageChannel.postMessage`
- `setImmediate` (Node.js only)
- User input events (click, keydown) — mỗi event là một task
- Script loading (initial script parse là một task)

**Quan trọng:** Browser có thể có **nhiều task queues** với độ ưu tiên khác nhau. User input events có thể được ưu tiên hơn setTimeout callbacks. Spec chỉ yêu cầu "chọn một task từ queue nào đó" mỗi iteration — để lại scheduling policy cho implementation. Đây là nơi Scheduler API can thiệp.

---

### Microtask Queue — "Job Queue" trong spec

Microtask Queue **khác hoàn toàn** với Macrotask Queue về cách Event Loop xử lý:

> **Sau mỗi task (hoặc khi Call Stack empty), Event Loop drain TOÀN BỘ Microtask Queue trước khi làm bất cứ điều gì khác.**

Nghĩa là: nếu microtask A thêm microtask B, và B thêm C... **tất cả đều chạy trước khi macrotask tiếp theo được xử lý**.

Nguồn tạo microtasks:

- `Promise.then()` / `.catch()` / `.finally()` callbacks
- `async/await` — mỗi `await` tạo microtask khi resume
- `queueMicrotask(fn)` — explicit enqueue
- `MutationObserver` callbacks

```javascript
console.log('1');

setTimeout(() => console.log('2'), 0); // Macrotask

Promise.resolve()
  .then(() => console.log('3')) // Microtask
  .then(() => console.log('4')); // Microtask (added by previous .then)

console.log('5');

// Output: 1, 5, 3, 4, 2
```

Trace từng bước:

```
[Synchronous phase]:
  "1" — in ngay
  setTimeout → Web API timer → callback vào Macrotask Queue
  Promise.resolve().then(→3) → microtask "→3" vào Microtask Queue
  "5" — in ngay

Call Stack empty. Event Loop nhìn vào:

[Microtask drain]:
  Lấy "→3" → run → console.log('3') → ".then(→4)" creates microtask "→4"
  Lấy "→4" → run → console.log('4') → không tạo thêm
  Microtask Queue empty.

[Render check] (browser only — nếu cần)

[Macrotask]: Lấy setTimeout callback → run → console.log('2')

Output: 1, 5, 3, 4, 2
```

### Tại sao Promise chạy trước setTimeout — cơ chế thật

Đây không phải "Promise nhanh hơn". Đây là **spec-defined priority** của hai queue khác nhau.

Khi Call Stack empty, Event Loop theo thứ tự:

```
1. Drain toàn bộ Microtask Queue (không giới hạn số lượng)
2. Run một Macrotask
3. Drain toàn bộ Microtask Queue lại
4. Render (browser)
5. Lặp lại
```

`Promise.then` callback vào Microtask Queue. `setTimeout` callback vào Macrotask Queue. Microtask Queue luôn được drain trước khi macrotask tiếp theo chạy — **không có exception**.

**Hệ quả nguy hiểm: Microtask starvation**

```javascript
// ❌ Microtask loop vô tận → browser freeze hoàn toàn

function infiniteMicrotasks() {
  Promise.resolve().then(infiniteMicrotasks);
  // Mỗi .then() tạo microtask mới
  // Event Loop mắc kẹt drain microtask queue mãi mãi
  // Macrotasks không bao giờ chạy
  // Render không bao giờ xảy ra
  // → UI hoàn toàn unresponsive
}

infiniteMicrotasks();

// setTimeout(() => console.log('never'), 0) — KHÔNG BAO GIỜ CHẠY
```

Đây là lý do tại sao recursive `Promise` chain không dùng được như "yield control" — bạn cần `setTimeout` hoặc Scheduler API để thực sự nhả CPU cho browser.

### `async/await` — desugaring thật sự

`async/await` là syntax sugar, nhưng biết desugar giúp predict behavior chính xác.

```javascript
async function fetchUserData(userId) {
  const response = await fetch(`/api/users/${userId}`);
  const data = await response.json();
  return data;
}
```

V8 desugars thành (conceptually):

```javascript
function fetchUserData(userId) {
  return new Promise((resolve, reject) => {
    // Phần trước await đầu tiên chạy đồng bộ:
    const fetchPromise = fetch(`/api/users/${userId}`);

    // await = .then() + suspend function execution:
    fetchPromise
      .then((response) => {
        // Resume sau await fetch — đây là microtask
        const jsonPromise = response.json();

        jsonPromise
          .then((data) => {
            // Resume sau await response.json() — microtask
            resolve(data);
          })
          .catch(reject);
      })
      .catch(reject);
  });
}
```

**Mỗi `await` tạo ít nhất một microtask** khi resume. Điều này có implications quan trọng:

```javascript
async function example() {
  console.log('A'); // Synchronous
  await Promise.resolve(); // Suspend → microtask scheduled
  console.log('B'); // Microtask resume
  await Promise.resolve(); // Suspend → microtask scheduled
  console.log('C'); // Microtask resume
}

console.log('1');
example();
console.log('2');

// Output: 1, A, 2, B, C
//
// Giải thích:
// "1" sync
// example() starts sync: "A" in, await → suspend
// "2" sync (example đã suspended)
// Call Stack empty → Microtask: resume example → "B" in, await → suspend
// Microtask: resume example → "C" in
```

### `requestAnimationFrame` — vị trí thật trong Event Loop

rAF không phải microtask, không phải macrotask thông thường. Nó là một **rendering step** đặc biệt.

Browser event loop thực tế có nhiều bước hơn mô hình đơn giản:

```
Một "frame" của browser event loop (khi render cần thiết):

1. [Macrotask] — chạy một task (setTimeout, I/O, event...)
2. [Microtasks] — drain toàn bộ microtask queue
3. [Rendering pipeline] — chỉ nếu browser quyết định cần render:
   a. requestAnimationFrame callbacks
   b. Intersection Observer callbacks
   c. ResizeObserver callbacks
   d. Style calculation (CSS)
   e. Layout
   f. Paint
   g. Composite
4. [Idle] — requestIdleCallback nếu có thời gian thừa
5. Lặp lại
```

**Quan trọng: Browser không render sau mỗi task.** Rendering được throttle theo refresh rate (thường 60fps = mỗi ~16.7ms). Nhiều macrotasks có thể chạy giữa hai frames. rAF chỉ chạy khi browser chuẩn bị render — tức là được **batch** theo refresh rate.

```javascript
// setTimeout vs rAF behavior:

// setTimeout(fn, 0): chạy ASAP, không sync với render
// → Có thể chạy giữa hai frames → visual update bị tearing
setTimeout(() => {
  element.style.transform = 'translateX(100px)';
}, 0);

// requestAnimationFrame: chạy NGAY TRƯỚC khi browser paint
// → Guaranteed sync với render cycle
// → Batched với tất cả visual updates trong cùng frame
requestAnimationFrame(() => {
  element.style.transform = 'translateX(100px)';
});
```

**Tại sao rAF không phải microtask:**

Nếu rAF là microtask, mọi `.then()` và `await` sẽ có thể trigger render → hàng nghìn renders mỗi giây → catastrophic. rAF được browser scheduling cẩn thận theo vsync signal từ display hardware.

**rAF callback timing:**

```javascript
// rAF nhận timestamp — thời điểm FRAME BẮT ĐẦU, không phải khi callback chạy
requestAnimationFrame((timestamp) => {
  // timestamp là DOMHighResTimeStamp
  // Tất cả rAF callbacks trong cùng frame nhận CÙNG timestamp
  // → Dễ sync animations với nhau
  const progress = (timestamp - startTime) / duration;
  updateAnimation(progress);
  if (progress < 1) requestAnimationFrame(updateAnimation);
});
```

### `queueMicrotask()` — khi nào dùng 🟡

```javascript
// queueMicrotask(fn) enqueue fn vào Microtask Queue trực tiếp
// Không cần tạo Promise

// Use case: khi muốn defer execution nhưng TRƯỚC macrotasks

function processBatch(items) {
  // Xử lý synchronously
  items.forEach(processItem);

  // Schedule cleanup AFTER current call stack clears
  // nhưng BEFORE any pending macrotasks (setTimeout, I/O)
  queueMicrotask(() => {
    flushPendingUpdates(); // Chạy trước bất kỳ setTimeout nào
  });
}

// So sánh timing:
queueMicrotask(() => console.log('microtask'));
setTimeout(() => console.log('macrotask'), 0);
Promise.resolve().then(() => console.log('promise'));

// Output: microtask, promise, macrotask
// (queueMicrotask và Promise.resolve().then đều là microtasks
//  → thứ tự giữa chúng = thứ tự enqueue)
```

**Khác với `Promise.resolve().then()`:**

- Không tạo Promise object → không có allocation overhead
- Không có `.catch()` error handling → error trong microtask unhandled → `unhandledrejection` hoặc crash
- Semantically rõ ràng hơn về intent

### Scheduler API — `isInputPending` 🟡

```javascript
// navigator.scheduling.isInputPending() — experimental, Chrome only hiện tại
// Cho phép JS kiểm tra "có user input đang chờ xử lý không?"

// Use case: Long synchronous work không muốn block user input

async function processLargeDataset(items) {
  const CHUNK_SIZE = 100;

  for (let i = 0; i < items.length; i += CHUNK_SIZE) {
    const chunk = items.slice(i, i + CHUNK_SIZE);
    processChunk(chunk);

    // Sau mỗi chunk, yield nếu có user input pending
    if (
      navigator.scheduling?.isInputPending({
        includeContinuous: true, // bao gồm mousemove, scroll
      })
    ) {
      // Yield control bằng setTimeout → macrotask
      // Cho phép input events được xử lý trước
      await new Promise((resolve) => setTimeout(resolve, 0));
    }
  }
}
// Tốt hơn: dùng scheduler.postTask() (Chrome 94+) với priority
```

### Node.js Event Loop — overview 🟢

Node.js có event loop riêng được build trên **libuv** thay vì browser engine. Khác biệt quan trọng nhất:

```
Node.js Event Loop phases (mỗi "tick" qua tất cả phases):

┌─────────────────────────────────────────┐
│           timers                        │ setTimeout, setInterval callbacks
│           (minimum delay đã qua)        │
├─────────────────────────────────────────┤
│           pending callbacks             │ I/O errors từ previous tick
├─────────────────────────────────────────┤
│           idle, prepare                 │ Internal
├─────────────────────────────────────────┤
│           poll                          │ ← QUAN TRỌNG NHẤT
│           (block ở đây nếu queue empty) │ Chờ I/O events
│           Execute I/O callbacks         │
├─────────────────────────────────────────┤
│           check                         │ setImmediate callbacks
├─────────────────────────────────────────┤
│           close callbacks               │ socket.on('close', ...)
└─────────────────────────────────────────┘

Giữa MỖI phase: drain microtask queue (process.nextTick + Promises)
process.nextTick() có HIGHER priority hơn Promise microtasks trong Node!
```

### Bug thực tế: React setState batching hiểu sai

```javascript
// ❌ Không hiểu event loop → confuse về khi nào state update

function SearchComponent() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);
  const [loading, setLoading] = useState(false);

  async function handleSearch(newQuery) {
    setLoading(true);
    // React 18: setLoading KHÔNG render ngay — được batched

    const data = await fetchResults(newQuery);
    // Sau await: microtask resume
    // React 18 automatic batching: cả hai setX bên dưới được batch

    setResults(data);
    setLoading(false);
    // Chỉ render MỘT LẦN cho cả hai setState trên
    // (React 18 batch automatic trong async functions)

    // Nhưng trong React 17:
    // Sau await, không còn trong React event handler
    // → setResults trigger một render
    // → setLoading trigger render khác
    // → 2 renders thay vì 1 → double flash
  }
}
```

```javascript
// ✅ Explicit batching (React 17 compat, hoặc khi cần control)
import { flushSync, unstable_batchedUpdates } from 'react-dom';

// Nếu cần 2 states update trong cùng 1 render sau async:
unstable_batchedUpdates(() => {
  setResults(data);
  setLoading(false);
});
```

```javascript
async function processInChunks(data, chunkSize = 1000) {
  const results = [];

  for (let i = 0; i < data.length; i += chunkSize) {
```

### Pattern: Microtask scheduling cho batch updates

```javascript
// Pattern thực tế: batch DOM updates trong microtask
// Dùng trong Vue 3, MobX, Solid.js

class Scheduler {
  constructor() {
    this.queue = new Set();
    this.flushing = false;
  }

  schedule(effect) {
    this.queue.add(effect);

    if (!this.flushing) {
      this.flushing = true;

      // Dùng microtask để flush CUỐI synchronous execution block
      // Nhiều schedule() calls trong cùng sync block → chỉ flush một lần
      Promise.resolve().then(() => {
        this.flush();
      });
      // queueMicrotask(() => this.flush()) cũng được — efficient hơn
    }
  }

  flush() {
    for (const effect of this.queue) {
      effect(); // Run tất cả queued effects
    }
    this.queue.clear();
    this.flushing = false;
  }
}

// Kết quả:
const scheduler = new Scheduler();
scheduler.schedule(updateA); // queue: [A], microtask scheduled
scheduler.schedule(updateB); // queue: [A, B], không schedule thêm
scheduler.schedule(updateA); // Set dedup: [A, B]
// Microtask runs: flush() → updateA(), updateB()
// Chỉ chạy 2 effects, không phải 3
```

### Câu 1: Tại sao `Promise.then()` callback luôn chạy trước `setTimeout(fn, 0)` callback, kể cả khi `setTimeout` được gọi trước?

**Đáp án:**

Đây không phải về thứ tự code hay timing — đây là **hai queue khác nhau** với priority khác nhau được định nghĩa trong HTML spec.

`setTimeout(fn, 0)` đăng ký `fn` vào **Macrotask Queue** (Task Queue). Dù delay là 0ms, callback chỉ được enqueue sau khi Web API timer fires, và được xử lý với tốc độ **một task mỗi event loop iteration**.

`Promise.then(fn)` enqueue `fn` vào **Microtask Queue** (Job Queue). Microtask Queue được drain **hoàn toàn** tại cuối mỗi event loop iteration, trước khi bất kỳ macrotask nào được run.

**Thứ tự xử lý của Event Loop:**

1. Chạy synchronous code đến khi Call Stack empty
2. **Drain toàn bộ Microtask Queue** — không giới hạn số lượng, kể cả microtasks tạo ra microtasks mới trong quá trình drain
3. Chạy một Macrotask
4. Drain Microtask Queue lại
5. Render (browser)
6. Lặp

Kết quả: Dù `setTimeout` được gọi trước `Promise.resolve()` trong code, callback của setTimeout nằm trong Macrotask Queue — nó chỉ được run sau khi **tất cả** microtasks (bao gồm Promise callbacks) đã chạy xong.

**Implication nguy hiểm:** Nếu microtasks tạo ra microtasks mới vô hạn (recursive `Promise.resolve().then(...)`), Event Loop sẽ bị kẹt vĩnh viễn trong microtask drain — macrotasks không bao giờ chạy, browser không render, UI freeze hoàn toàn.

### Câu 2: `requestAnimationFrame` là macrotask hay microtask? Tại sao nó được thiết kế như vậy, và khi nào nên dùng rAF thay vì `setTimeout`?

**Đáp án:**

`requestAnimationFrame` **không phải macrotask cũng không phải microtask** — nó là một phần của **rendering pipeline**, chạy trong một bước riêng biệt trong event loop.

**Vị trí thật trong browser event loop:**
Sau khi một macrotask chạy và microtask queue được drain, browser kiểm tra xem có cần render frame mới không. Nếu có (thường sau ~16.7ms), browser chạy:

1. rAF callbacks
2. IntersectionObserver, ResizeObserver callbacks
3. Style calculation → Layout → Paint → Composite

**Tại sao thiết kế như vậy:** Nếu rAF là microtask, mỗi Promise resolution có thể trigger rAF → hàng nghìn renders/giây → catastrophic. Nếu là macrotask thông thường, không có gì đảm bảo nó chạy đúng trước paint — có thể có setTimeout callbacks chen vào giữa. Bằng cách đặt rAF vào rendering step, browser đảm bảo:

- rAF chạy đúng trước paint (đồng bộ với vsync)
- Nhiều `rAF` calls trong cùng frame được batched
- Tất cả rAF callbacks nhận **cùng timestamp** → dễ sync animations

**Khi dùng rAF thay vì setTimeout:**

- **Visual animations:** rAF đảm bảo update được apply trước paint → không tearing
- **DOM measurements trước render:** Đọc layout properties (offsetHeight) trong rAF → layout đã ổn định
- **Throttle visual updates:** Nhiều state changes trong một frame → một rAF callback xử lý tất cả

**Khi KHÔNG dùng rAF:**

- Logic không liên quan đến rendering → setTimeout hoặc Promise
- Code cần chạy kể cả khi tab ẩn → rAF bị paused khi tab không visible

### Câu 3: Đoạn code sau output gì? Giải thích từng bước Event Loop. Sau đó chỉ ra bug nếu code này là production logic.

```javascript
console.log('init');

const p = new Promise((resolve) => {
  console.log('promise executor');
  resolve('resolved');
});

p.then((val) => {
  console.log('then 1:', val);
  return 'chained';
}).then((val) => {
  console.log('then 2:', val);
});

setTimeout(() => {
  console.log('timeout');
}, 0);

queueMicrotask(() => {
  console.log('queueMicrotask');
});

console.log('end');
```

**Đáp án:**

**Output:**

```
init
promise executor
end
then 1: resolved
queueMicrotask
then 2: chained
timeout
```

**Trace từng bước:**

1. **Sync:** `console.log('init')` → print `init`
2. **Sync:** `new Promise(executor)` → executor chạy **đồng bộ ngay lập tức** (không async!) → print `promise executor` → `resolve('resolved')` called → Promise state = fulfilled
3. **Sync:** `p.then(cb1)` → Promise đã fulfilled → `cb1` enqueue vào **Microtask Queue**: `[cb1]`
4. **Sync:** `.then(cb2)` → chưa enqueue (cb2 phụ thuộc cb1 chạy xong)
5. **Sync:** `setTimeout(cbTimeout, 0)` → Web API → **Macrotask Queue**: `[cbTimeout]`
6. **Sync:** `queueMicrotask(cbMicro)` → **Microtask Queue**: `[cb1, cbMicro]`
7. **Sync:** `console.log('end')` → print `end`
8. **Call Stack empty → Drain Microtasks:**
   - Run `cb1`: print `then 1: resolved` → returns `'chained'` → enqueue `cb2` → **Microtask Queue**: `[cbMicro, cb2]`
   - Run `cbMicro`: print `queueMicrotask`
   - Run `cb2`: print `then 2: chained`
   - Queue empty
9. **Run Macrotask:** Run `cbTimeout` → print `timeout`

**Bug trong production logic:**

`new Promise(executor)` executor chạy **synchronous**. Nhiều developer nhầm tưởng Promise constructor bất đồng bộ. Nếu `executor` chứa heavy computation hay blocking I/O (đọc file sync), nó sẽ **block Call Stack** ngay lập tức — không có gì async về executor. Chỉ callbacks (`.then`, `.catch`) là async.

```javascript
// BUG: executor block Call Stack
const p = new Promise((resolve) => {
  const data = fs.readFileSync('large-file.txt'); // BLOCKING — freeze event loop
  resolve(data);
});

// FIX: dùng async I/O thay vì sync
const p = new Promise((resolve) => {
  fs.readFile('large-file.txt', (err, data) => {
    resolve(data); // Callback từ I/O event — không block
  });
});
// Hoặc đơn giản hơn: dùng fs.promises.readFile()
```
