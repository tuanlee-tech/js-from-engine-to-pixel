## 1. Cơ chế thật

### GC Tuning — V8 flags và khi nào chúng quan trọng

V8 expose nhiều flags để control GC behavior. Hầu hết không cần touch trong normal apps — nhưng khi bạn gặp GC-related incidents, biết chúng giúp diagnose nhanh hơn nhiều.

**Flags quan trọng nhất:**

```bash
# Xem toàn bộ V8 flags:
node --v8-options | grep -i gc

# Flags thường dùng trong production tuning:
node \
  --max-old-space-size=4096 \      # Old Space limit (MB) — default ~1.5GB trên 64-bit
  --max-semi-space-size=64 \       # New Space semi-space size (MB) — default 16MB
  --optimize-for-size \            # Trade CPU cho memory — GC aggressive hơn
  --expose-gc \                    # Enable global.gc() — force GC từ JS code
  server.js
```

**`--max-old-space-size` — khi nào cần tăng:**

V8 default Old Space limit khoảng 1.4GB (64-bit). Khi heap gần limit, Major GC chạy thường xuyên hơn và aggressive hơn. Nếu app legitimately cần nhiều memory (in-memory cache, large datasets), tăng limit để giảm GC frequency:

```javascript
// Detect nếu đang gần limit — log để alert
const v8 = require('v8');

setInterval(() => {
  const stats = v8.getHeapStatistics();
  const usageRatio = stats.used_heap_size / stats.heap_size_limit;

  if (usageRatio > 0.85) {
    // 85% heap used → Major GC sắp chạy aggressively
    // Nếu đây là expected → tăng --max-old-space-size
    // Nếu không expected → có memory leak
    console.warn({
      usedMB: (stats.used_heap_size / 1e6).toFixed(1),
      limitMB: (stats.heap_size_limit / 1e6).toFixed(1),
      ratio: usageRatio.toFixed(3),
    });
  }
}, 10_000);
```

**`--max-semi-space-size` — khi nào quan trọng:**

New Space = 2 semi-spaces. Default 16MB → mỗi semi-space 8MB. Scavenger chạy khi semi-space đầy.

```
Semi-space 8MB  → app tạo nhiều short-lived objects nhanh
→ Scavenger chạy rất thường xuyên (mỗi vài giây)
→ Nhiều GC pause nhỏ (~1-2ms mỗi lần)

Tăng semi-space lên 64MB:
→ Scavenger ít chạy hơn
→ Mỗi Scavenge tốn nhiều thời gian hơn (~8-16ms)
→ Trade-off: ít pauses hơn nhưng mỗi pause dài hơn

Rule of thumb cho server apps:
- Latency-sensitive (p99 quan trọng): semi-space nhỏ → nhiều pauses ngắn
- Throughput-oriented (batch jobs): semi-space lớn → ít pauses dài
```

### Incremental Marking và Concurrent GC — tại sao V8 hiện đại ít pause hơn

V8 trước phiên bản ~6 (2017): Major GC là **stop-the-world** — JS execution dừng hoàn toàn trong khi GC mark và sweep. 200ms pause là bình thường.

V8 hiện đại dùng ba kỹ thuật để giảm pause:

**1. Incremental Marking:**

```
Thay vì:  [JS runs] → [STOP] → [Mark 200ms] → [Sweep] → [JS resumes]

Dùng:     [JS] [Mark 5ms] [JS] [Mark 5ms] [JS] [Mark 5ms] ... [Sweep]
               ↑                ↑               ↑
          incremental steps — xen kẽ với JS execution
```

V8 chia marking thành nhiều "increments" nhỏ (~1-5ms mỗi increment), xen kẽ với JS execution. Tổng marking time vẫn ~200ms nhưng không phải một pause liên tục.

**Vấn đề của Incremental Marking:** JS tiếp tục modify object graph trong khi GC đang mark. Object đã được marked có thể bị reference mới trỏ vào — những reference này phải được tracked. V8 dùng **write barriers** (đã học trong 1.2) để record những thay đổi này trong quá trình incremental marking.

**2. Concurrent Marking (V8 6.2+):**

```
Main thread:   [JS runs  ──────────────────────────── JS runs]
GC threads:         [Mark ─────────────────── Mark]
                    ↑ parallel trên background threads
                    ↑ không cần stop main thread để mark
```

Marking xảy ra trên **GC helper threads** song song với JS execution trên main thread. Main thread chỉ bị stop ngắn (~1-5ms) để finalize marking và start sweep.

**3. Concurrent Sweeping:**

Sweep phase (giải phóng dead objects) cũng chạy trên background threads. Main thread không cần chờ sweep hoàn thành.

**Kết quả thực tế:**

```
V8 6.0 (2017): Major GC pause ~200ms
V8 10.0 (2022): Major GC pause ~5-10ms (pause chính)
                + background work ~100ms (không affect JS)
```

### Heap Snapshot — cơ chế thật

Heap snapshot không phải "bức ảnh chụp heap" theo nghĩa đơn giản. V8 phải:

1. **Stop JS execution** (stop-the-world)
2. **Walk toàn bộ object graph** từ GC roots
3. **Serialize** mỗi object: type, size, references, constructor name
4. **Write** ra file `.heapsnapshot` (JSON format)
5. **Resume** JS execution

File `.heapsnapshot` có thể rất lớn — 1GB heap → ~2-4GB snapshot file. Vì vậy taking snapshot trong production phải cẩn thận: stop-the-world pause có thể dài hàng giây.

**Snapshot format — những gì bên trong:**

```json
{
  "snapshot": {
    "meta": { "node_fields": ["type","name","id","self_size","edge_count",...] },
    "node_count": 150000,
    "edge_count": 800000
  },
  "nodes": [1,23,1,48,5, 2,45,2,96,3, ...],
  "edges": [1,67,8, 2,89,15, ...],
  "strings": ["Object","Array","closure","system/Context",...]
}
```

Nodes và edges được encode flat thành integer arrays để compact. DevTools decode và render thành tree view.

### Heap Snapshot Workflow — quy trình debug memory leak thực tế

**Nguyên tắc 3-snapshot:**

```
Snapshot 1: Baseline (sau khi app start + warm up)
     ↓
[Perform action nghi ngờ gây leak N lần]
     ↓
Snapshot 2: After action
     ↓
[Perform action thêm N lần]
     ↓
Snapshot 3: After more actions

So sánh S1→S2→S3:
- Objects tăng đều → leak
- Objects tăng rồi ổn định → expected caching
- Objects tăng rồi giảm → transient, OK
```

**Tại sao cần 3 snapshots, không phải 2:**

Snapshot 2 vs 1 có thể show objects tăng do caching hợp lệ (first-time data load, JIT compilation artifacts). Nếu snapshot 3 vs 2 **tiếp tục tăng** với cùng rate → leak thật sự, không phải one-time initialization.

**Chrome DevTools workflow step-by-step:**

```
BƯỚC 1: Setup
─────────────
- Open DevTools → Memory tab
- Force GC trước (click garbage can icon)
  → Loại bỏ unreachable objects để baseline sạch
- Take Snapshot 1

BƯỚC 2: Reproduce
─────────────────
- Perform action N lần (navigate, click, submit form)
- Đây là action nghi ngờ gây leak
- Force GC
- Take Snapshot 2

BƯỚC 3: Lặp lại
────────────────
- Perform action N lần nữa
- Force GC
- Take Snapshot 3

BƯỚC 4: Compare
────────────────
- Chọn Snapshot 3 → dropdown "Comparison" → so sánh với Snapshot 1
- Sort by "# Delta" (số objects tăng) HOẶC "Size Delta" (bytes tăng)
- Focus vào constructors có positive delta cao

BƯỚC 5: Drill down
──────────────────
- Click vào constructor có leak (ví dụ "closure", "Array", "Object")
- Xem "Retainers" panel (bottom) → ai đang giữ object này alive?
- Trace retention path lên đến root

BƯỚC 6: Identify
─────────────────
- Retention path thường trông như:
  leaked_object ← array element ← listeners array ← EventEmitter
  ← module.exports ← <closure> ← (root)
  → EventEmitter không được cleanup → listeners accumulate
```

**Node.js — programmatic heap snapshot:**

```javascript
// Không muốn dùng DevTools (production server)?
// Trigger snapshot via HTTP endpoint hoặc signal

const v8 = require('v8');
const path = require('path');
const fs = require('fs');

// Option 1: HTTP endpoint (chỉ enable trong staging/debug mode)
app.get('/debug/heap-snapshot', (req, res) => {
  if (process.env.NODE_ENV === 'production') {
    return res.status(403).json({ error: 'Not available in production' });
  }

  const filename = path.join('/tmp', `heap-${Date.now()}.heapsnapshot`);

  // writeHeapSnapshot: stop-the-world trong khi write
  // Trên large heap (>500MB) có thể mất vài giây
  const snapshotPath = v8.writeHeapSnapshot(filename);
  res.json({ path: snapshotPath });
  // Download file → load vào Chrome DevTools Memory tab
});

// Option 2: Signal-based (safer cho production)
// kill -USR2 <pid> → trigger snapshot
process.on('SIGUSR2', () => {
  const filename = v8.writeHeapSnapshot();
  console.log(`Heap snapshot written to: ${filename}`);
});

// Option 3: heapdump package (more control)
// npm install heapdump
const heapdump = require('heapdump');
heapdump.writeSnapshot('/tmp/heap.heapsnapshot', (err, filename) => {
  console.log('Dump written to', filename);
});
```

**Đọc Heap Snapshot — key metrics:**

```
SHALLOW SIZE vs RETAINED SIZE
──────────────────────────────
Shallow Size: size của chính object
  - Number: 8 bytes
  - Pointer: 8 bytes
  - String "hello": 10 bytes (header + chars)
  - Object {}: 56 bytes (header + slots)

Retained Size: bytes được freed NẾU object này bị GC
  = Shallow + shallow sizes của tất cả objects
    chỉ reachable THÔNG QUA object này

Ví dụ:
  Cache object: shallow=56B, retained=50MB
  → Cache là root của 50MB object graph
  → Freeing cache → free 50MB

Tìm leak: Sort by Retained Size DESC
→ Object có retained size lớn đang giữ nhiều memory nhất
→ Đây là suspect đầu tiên
```

```
DISTANCE metric
───────────────
Distance = số edges từ GC root đến object

Distance thấp (1-3): Gần root → likely global state, cache
Distance cao (10+): Deep trong object graph

Detached DOM nodes: thường appear với constructor "(detached)"
→ Sort by "(detached)" constructor → xem retained size
→ Nếu > 0 → DOM leak
```

### Allocation Timeline — track objects khi chúng được tạo

Heap Snapshot là point-in-time. **Allocation Timeline** track objects theo thời gian — tốt hơn để tìm WHERE leak được tạo ra.

```
Chrome DevTools → Memory → Allocation instrumentation on timeline

Record:
  ████ ████ ████       ← blue bars = new allocations
       ██              ← gray bars = garbage collected

Interpretation:
  Blue bars remain blue after GC → objects survive → potential leak
  Blue bars turn gray → objects collected → normal

Click vào blue bar → xem stack trace tại điểm object được allocated
→ Tìm thấy function tạo ra leaked objects
```

### SpiderMonkey (Firefox) — khác V8 như thế nào

SpiderMonkey là JS engine của Firefox, develop bởi Mozilla. Đây không phải deep-dive — đủ để understand differences khi debug cross-browser issues.

**Architecture:**

```
SpiderMonkey Pipeline:
Source → Parser → AST
                   ↓
              Bytecode (đọc bởi Interpreter)
                   ↓
            [WarpBuilder] → MIR (Mid-level IR)
                   ↓
              [Ion Compiler] → Machine Code

Tương đương V8:
Parser → AST → Ignition bytecode → Maglev → TurboFan
```

SpiderMonkey có:

- **Interpreter:** Đọc bytecode (tương đương Ignition)
- **Warp/Ion:** Optimizing JIT (tương đương TurboFan)
- **Warp** là mid-tier (similar to Maglev, added 2020)

**Type inference khác V8:**

V8 dùng FeedbackVector (runtime-collected) để inform JIT. SpiderMonkey trước đây dùng **static type inference** — phân tích types tại compile time, không phụ thuộc vào runtime observations nhiều.

Sau nhiều năm, SpiderMonkey đã dịch chuyển gần về model của V8 hơn — runtime type feedback trở nên dominant. Nhưng implementation details vẫn khác.

**GC — Generational với Incremental:**

SpiderMonkey dùng **Generational GC** tương tự V8:

- Nursery (≈ New Space): small, collected thường xuyên
- Tenured Heap (≈ Old Space): mark-and-sweep, incremental

Khác biệt quan trọng: SpiderMonkey có **Compacting GC** chạy thường xuyên hơn V8 để combat fragmentation. V8 chỉ compact khi thật sự cần.

**Cross-browser implication cho code:**

```javascript
// Hidden class/shape pattern — V8 vs SpiderMonkey

// V8: property order tạo different hidden classes
// SpiderMonkey: dùng "shapes" (similar concept, different name)
// Behavior rất giống nhau — consistent property order vẫn là best practice

// Quan trọng hơn là: micro-optimizations bạn làm cho V8
// (như property order, monomorphic functions) thường cũng benefit SpiderMonkey
// vì root problem (type consistency, shape stability) là universal

// Exception: array hole behavior
const arr = [1, 2, , 4]; // Hole tại index 2

// V8: chuyển array sang "HOLEY_SMI_ELEMENTS" mode → slower
// SpiderMonkey: tương tự có different internal representations
// Behavior đều đúng, nhưng performance characteristics có thể differ
```

### JavaScriptCore (Safari/WebKit) — engine khác biệt nhất

JSC là JS engine của Safari, develop bởi Apple. Đây là engine **khác nhất** trong ba engine lớn — và thường là engine phát hiện bugs ẩn trong code.

**Architecture — 4 tầng JIT:**

```
JSC Pipeline (unique nhất trong các engines):

Source → Lexer → Parser → Bytecode
                               ↓
                     [LLInt - Low Level Interpreter]
                     Interpret bytecode trực tiếp
                     (Assembly interpreter, cực fast startup)
                               ↓ (warm)
                       [Baseline JIT]
                       Fast compile, minimal optimization
                       (tương đương Maglev nhưng đã có từ lâu hơn)
                               ↓ (hot)
                          [DFG JIT]
                          Data Flow Graph — medium optimization
                               ↓ (very hot)
                          [FTL JIT]
                          Faster Than Light — maximum optimization
                          Dùng B3 backend (Apple's compiler IR)
                          (tương đương TurboFan)
```

**4 tầng vs V8's 3 tầng:** JSC có nhiều tầng hơn, cho phép gradual optimization. Code escalate chậm hơn lên tầng cao nhất — JSC conservative hơn về khi nào apply expensive optimizations.

**Heap và Memory — JSC khác biệt:**

JSC dùng **bmalloc** (Apple's custom allocator) thay vì system malloc. bmalloc được thiết kế đặc biệt cho JavaScript object allocation patterns.

JSC GC cũng là generational nhưng implementation khác:

- **Eden space** (≈ Nursery/New Space): rất nhỏ, collect rất thường xuyên
- **Young space**: medium-lived objects
- **Old space**: long-lived

JSC collect Eden space cực kỳ thường xuyên (có thể mỗi vài milliseconds) với pause rất ngắn — khác với V8 collect New Space ít thường xuyên hơn nhưng thorough hơn mỗi lần.

**Strict mode enforcement:**

JSC nổi tiếng là engine **strict nhất** về spec compliance. Code chạy tốt trên V8 nhưng fail trên JSC thường là vì:

- Dựa vào `arguments.callee` (deprecated)
- Dựa vào property enumeration order (spec không guarantee)
- Dựa vào specific `toString()` output của functions
- Subtle timing-dependent bugs được expose bởi GC differences

```javascript
// Bug thường chỉ xuất hiện trên JSC:

// Property enumeration order — V8 thường giữ insertion order
// JSC trước đây không guarantee (giờ cũng giữ, nhưng behavior có thể differ
// ở edge cases với integer keys vs string keys)

const obj = {};
obj['2'] = 'two';
obj['1'] = 'one';
obj['b'] = 'bee';
obj['a'] = 'ay';

// V8 và JSC hiện đại: ['1', '2', 'b', 'a']
// Integer-like keys được sort lên trước theo spec
// Nhưng code dựa vào specific order (không phải spec) = time bomb
```

**Cross-engine performance — rules không thay đổi:**

Các optimization principles từ Phase 1 apply cho tất cả engines vì chúng address **root problems**, không phải V8-specific quirks:

```
1. Monomorphic functions → tất cả engines JIT optimize tốt hơn
2. Consistent object shapes → tất cả engines có shape/hidden-class concept
3. Avoid long synchronous blocks → tất cả engines single-threaded
4. Minimize allocations in hot loops → tất cả engines có GC pressure
5. Avoid memory leaks → tất cả engines dùng reachability-based GC
```

Các V8-specific tricks (như specific property order cho hidden classes) có thể không apply 1:1 cho JSC/SpiderMonkey, nhưng **direction luôn đúng**: consistent, predictable code là fast code trên mọi engine.

### Memory Leak Investigation — end-to-end workflow

```javascript
// Scenario: Express server với memory leak
// Memory tăng dần, restart fix tạm thời

// server.js — có leak ẩn
const express = require('express');
const EventEmitter = require('events');

const app = express();
const globalBus = new EventEmitter();
// BUG: globalBus là global, tồn tại suốt lifetime của app

const requestCache = new Map();

app.get('/api/data/:id', async (req, res) => {
  const { id } = req.params;

  // Mỗi request register một listener trên global bus
  // Listener này KHÔNG BAO GIỜ được remove
  // → listeners accumulate mỗi request
  globalBus.on(`data:${id}`, (data) => {
    // Closure giữ: req object, res object, id string
    // req/res objects giữ: headers, body, socket reference...
    requestCache.set(id, data);
  });

  const result = await fetchData(id);
  res.json(result);
  // res.json gửi xong → request "done"
  // Nhưng listener vẫn còn trên globalBus → giữ closure → giữ req/res
});

// Kiểm chứng:
// globalBus.listenerCount(`data:1`) sau 1000 requests = 1000
// Memory tăng tuyến tính với số requests
```

```javascript
// Investigation steps:

// Step 1: Confirm với metrics
setInterval(() => {
  const mem = process.memoryUsage();
  console.log({
    heapUsedMB: (mem.heapUsed / 1e6).toFixed(1),
    // Nếu con số này tăng đều → leak confirmed
    listenerCount: globalBus.eventNames().length,
    // Nếu tăng → listener leak confirmed
  });
}, 5000);

// Step 2: Heap snapshot comparison
// Before: 50MB
// After 1000 requests: 250MB
// Snapshot comparison → "closure" objects +2000 → point to handlers

// Step 3: Fix
app.get('/api/data/:id', async (req, res) => {
  const { id } = req.params;

  // Option A: once() thay vì on()
  // Listener tự remove sau lần đầu fire
  globalBus.once(`data:${id}`, (data) => {
    requestCache.set(id, data);
  });

  // Option B: manual cleanup
  const handler = (data) => {
    requestCache.set(id, data);
  };
  globalBus.on(`data:${id}`, handler);

  try {
    const result = await fetchData(id);
    res.json(result);
  } finally {
    // LUÔN cleanup dù success hay error
    globalBus.off(`data:${id}`, handler);
  }
});
```

### GC Tuning cho Node.js service thực tế

```javascript
// Monitoring script — attach vào production process
// Collect GC metrics mà không affect performance

const { PerformanceObserver } = require('perf_hooks');
const v8 = require('v8');

class GCMonitor {
  constructor(options = {}) {
    this.metrics = {
      scavengeCount: 0, // Minor GC (New Space)
      markSweepCount: 0, // Major GC (Old Space)
      totalScavengeMs: 0,
      totalMarkSweepMs: 0,
      maxMarkSweepMs: 0,
    };

    // PerformanceObserver: zero-overhead khi không có GC
    this.observer = new PerformanceObserver((list) => {
      for (const entry of list.getEntries()) {
        this._recordGC(entry);
      }
    });

    this.observer.observe({ entryTypes: ['gc'] });
  }

  _recordGC(entry) {
    const duration = entry.duration;
    const kind = entry.detail?.kind;
    // kind: 1=Scavenge, 2=MarkSweepCompact, 4=IncrementalMarking...

    if (kind === 1) {
      this.metrics.scavengeCount++;
      this.metrics.totalScavengeMs += duration;
    } else if (kind === 2) {
      this.metrics.markSweepCount++;
      this.metrics.totalMarkSweepMs += duration;
      this.metrics.maxMarkSweepMs = Math.max(
        this.metrics.maxMarkSweepMs,
        duration,
      );

      // Alert nếu Major GC quá lâu
      if (duration > 100) {
        console.warn(`[GC] Major GC took ${duration.toFixed(1)}ms`);
        // Trong production: send alert to PagerDuty/Datadog
      }
    }
  }

  report() {
    const heapStats = v8.getHeapStatistics();
    return {
      ...this.metrics,
      heapUsedMB: (heapStats.used_heap_size / 1e6).toFixed(1),
      heapLimitMB: (heapStats.heap_size_limit / 1e6).toFixed(1),
      externalMB: (heapStats.external_memory / 1e6).toFixed(1),
    };
  }
}

const gcMonitor = new GCMonitor();

// Report mỗi phút
setInterval(() => {
  console.log('[GC Report]', gcMonitor.report());
}, 60_000);
```

### Câu 1: Incremental Marking và Concurrent Marking giải quyết vấn đề gì của Stop-the-World GC? Trade-off là gì?

**Đáp án:**

**Vấn đề Stop-the-World:** Major GC (Mark-Sweep) phải scan toàn bộ heap để mark live objects. Trên heap lớn (hàng trăm MB), một GC pause có thể kéo dài 200-500ms. Trong thời gian đó, JS execution dừng hoàn toàn — user thấy freeze, server không handle requests, animations drop frames.

**Incremental Marking:** Chia marking thành nhiều increments nhỏ (~1-5ms), xen kẽ với JS execution. Tổng marking work không đổi nhưng không phải một pause dài liên tục.

**Trade-off của Incremental:** JS tiếp tục chạy và modify object graph trong khi GC đang mark. Object đã được marked có thể nhận reference mới từ object chưa được marked — **tri-color invariant** bị violated. V8 dùng **write barriers**: mỗi property assignment kiểm tra và record nếu tạo ra "old-to-new" hoặc "marked-to-unmarked" reference. Write barriers là overhead trên mọi property write — nhỏ nhưng tích lũy trong hot loops.

**Concurrent Marking:** Marking chạy trên background GC threads song song với JS thread. Main thread gần như không bị dừng — chỉ ~2-5ms để finalize (rasterize marking state từ background threads về main thread's view). Cho phép Major GC pause giảm từ ~200ms xuống ~5ms trong nhiều cases.

**Trade-off của Concurrent:** Cần hardware multi-core. Synchronization giữa GC threads và JS thread có overhead. Memory bandwidth shared giữa GC và JS — có thể affect throughput dù không affect latency.

### Câu 2: Tại sao cần 3 heap snapshots thay vì 2 để confirm memory leak? Retained Size và Shallow Size khác nhau như thế nào, và cái nào dùng để tìm leak?

**Đáp án:**

**Tại sao cần 3 snapshots:**

So sánh Snapshot 1 (baseline) và Snapshot 2 (sau action) có thể cho false positive vì:

- **One-time initialization:** Data được load lần đầu vào cache → objects tăng ở S2 nhưng stable ở S3
- **JIT compilation artifacts:** TurboFan tạo compiled code objects, FeedbackVectors — tăng ở S2 (first JIT), stable ở S3
- **Lazy initialization:** Modules, connection pools được khởi tạo lần đầu dùng

Snapshot 3 filter out tất cả one-time costs. Nếu S3 vs S2 tiếp tục tăng với **cùng rate** như S2 vs S1 → leak thật sự, không phải initialization artifact.

**Shallow Size vs Retained Size:**

**Shallow Size** = bytes của chính object đó — không tính bất kỳ object nào nó point tới. Một object với 3 properties: header (~32B) + 3 pointers (3×8B) = ~56B shallow size, dù các properties trỏ tới objects chiếm 100MB.

**Retained Size** = tổng bytes sẽ được freed nếu object này bị GC — bao gồm chính nó và tất cả objects **chỉ reachable thông qua nó** (không có path nào khác từ root). Nếu object A trỏ tới object B nhưng C cũng trỏ tới B, thì B không được tính vào retained size của A (vì B vẫn reachable qua C sau khi A bị freed).

**Dùng cái nào để tìm leak:**

**Sort by Retained Size DESC** để tìm leak. Một object với shallow=56B nhưng retained=50MB là root của một 50MB subgraph — đây là prime suspect. Shallow size misleading vì object nhỏ có thể giữ cả một cây object lớn alive thông qua references.

### Câu 3: Đoạn code Node.js sau có memory leak không? Viết investigation plan bằng heap snapshot methodology và propose fix.

```javascript
class MetricsCollector {
  constructor(reportingService) {
    this.reportingService = reportingService;
    this.buffer = [];
    this.flushInterval = setInterval(() => {
      this._flush();
    }, 5000);
  }

  record(metric) {
    this.buffer.push({
      name: metric.name,
      value: metric.value,
      timestamp: Date.now(),
      metadata: metric.metadata,
    });
  }

  _flush() {
    if (this.buffer.length === 0) return;
    this.reportingService.send(this.buffer);
    this.buffer = [];
  }

  destroy() {
    this._flush();
    clearInterval(this.flushInterval);
  }
}

// Usage trong request handler
app.post('/api/events', async (req, res) => {
  const collector = new MetricsCollector(globalReportingService);

  collector.record({ name: 'request.received', value: 1 });
  const result = await processEvent(req.body);
  collector.record({ name: 'request.processed', value: 1, metadata: req.body });

  res.json(result);
  // collector.destroy() không được gọi
});
```

**Đáp án:**

**Có hai leaks:**

**Leak 1 (Critical): `setInterval` không được cleared.**

`MetricsCollector` tạo `setInterval` trong constructor. `destroy()` clear nó — nhưng `destroy()` không bao giờ được gọi trong request handler. Mỗi request tạo một `MetricsCollector` mới với một interval mới. Sau 10,000 requests: 10,000 active intervals.

Mỗi interval closure giữ `this` (MetricsCollector instance) → giữ `reportingService` reference → giữ `buffer` array → giữ metric objects. Dù `collector` local variable ra khỏi scope sau request, interval vẫn giữ instance alive.

**Leak 2 (Subtle): `metadata: req.body` trong buffer.**

`req.body` là parsed request body — có thể chứa large payloads. Object này được reference bởi buffer entry. Dù `_flush()` send buffer, nếu `reportingService.send()` là async và giữ reference tới array, data có thể tồn tại lâu hơn expected.

**Investigation plan:**

```
1. Monitor:
   setInterval(() => {
     // Đếm số active intervals là không trivial
     // Nhưng có thể monitor memory:
     console.log(process.memoryUsage().heapUsed / 1e6);
   }, 5000);
   → Nếu tăng đều theo rate of requests → leak confirmed

2. 3-Snapshot methodology:
   S1: Baseline sau server start
   S2: Sau 1000 POST /api/events requests
   S3: Sau 2000 requests
   Compare S3 vs S1 → sort by Retained Size DESC
   → Expect: "MetricsCollector" constructor, "Timeout" objects, buffer arrays

3. Retention path:
   MetricsCollector instance
   ← closure trong setInterval callback
   ← Timeout object (active interval)
   ← libuv timer queue (GC root)
   → Không thể GC dù request đã xong
```

**Fix:**

```javascript
// Fix 1: Luôn destroy collector
app.post('/api/events', async (req, res) => {
  const collector = new MetricsCollector(globalReportingService);
  try {
    collector.record({ name: 'request.received', value: 1 });
    const result = await processEvent(req.body);
    // Chỉ record data cần thiết, không record toàn bộ req.body
    collector.record({ name: 'request.processed', value: 1 });
    res.json(result);
  } finally {
    collector.destroy(); // Luôn chạy dù error
  }
});

// Fix 2 (tốt hơn): Không tạo collector per-request
// Dùng singleton collector — share giữa tất cả requests
const collector = new MetricsCollector(globalReportingService);

app.post('/api/events', async (req, res) => {
  collector.record({ name: 'request.received', value: 1 });
  const result = await processEvent(req.body);
  collector.record({ name: 'request.processed', value: 1 });
  res.json(result);
});

// Singleton: một interval, không leak
// Tắt server: process.on('SIGTERM', () => collector.destroy())
```
