# Worker Event Loops (Web Workers, SharedArrayBuffer, Atomics)

> **Độ ưu tiên:** 🟢 — Staff/Principal territory. Cần hiểu khi design concurrent web apps, debug race conditions, hoặc offload CPU-heavy work.

---

## 1. Cơ chế thật

### Vấn đề Worker giải quyết

Main thread JS là single-threaded. Từ buổi Event Loop: long task > 50ms block render, input events không được xử lý, UI freeze. Với CPU-heavy work (image processing, crypto, ML inference, parsing large datasets), không có cách nào "yield đủ" từ main thread — bạn cần **thread thật sự**.

Web Workers cho phép chạy JS trên **OS thread riêng biệt**, parallel với main thread. Mỗi Worker có:

- V8 isolate riêng (không share heap với main thread)
- Event loop riêng
- Call stack riêng
- Không có DOM access

**Quan trọng nhất: Workers không share memory theo default.** Đây là thiết kế có chủ ý để tránh data races — cơ chế là **message passing**.

Buổi này focus vào **Dedicated Worker** và cơ chế event loop của nó. Service Worker deserves một buổi riêng trong Phase 3/5.

### Worker Event Loop — so sánh với Main Thread

Main thread event loop (đã học):

```
Task → Microtasks → Render → Task → Microtasks → Render...
```

Worker event loop — **không có Render step**:

```
Task → Microtasks → Task → Microtasks → Task...
```

Worker không có:

- `requestAnimationFrame`
- DOM APIs
- `document`, `window`
- `alert`, `confirm`

Worker có:

- `self` (thay cho `window`) — global scope của Worker
- `fetch`, `XMLHttpRequest`
- `setTimeout`, `setInterval`
- `Promise`, `async/await`
- `importScripts()` (classic workers) hoặc ESM imports
- `IndexedDB`
- `WebSockets`
- `Crypto API`
- `OffscreenCanvas` (Chrome)

### Message Passing — cơ chế thật của `postMessage`

```javascript
// Main thread
const worker = new Worker('./compute-worker.js');

worker.postMessage({ type: 'PROCESS', data: largeArray });

worker.onmessage = (event) => {
  console.log('Result:', event.data);
};
```

```javascript
// compute-worker.js
self.onmessage = (event) => {
  const { type, data } = event.data;
  if (type === 'PROCESS') {
    const result = heavyComputation(data);
    self.postMessage(result);
  }
};
```

**Điều thật sự xảy ra bên trong `postMessage`:**

Data được **serialized** (Structured Clone Algorithm) tại thời điểm `postMessage` được gọi, sau đó **deserialized** ở phía nhận. Đây không phải JSON — Structured Clone hỗ trợ nhiều types hơn (Date, RegExp, Map, Set, ArrayBuffer, Blob...) nhưng không hỗ trợ functions hay DOM nodes.

```
Main Thread                      Worker Thread
──────────                       ─────────────
postMessage(data)
  │
  ├── Structured Clone: serialize data
  │   (copy toàn bộ object graph)
  │   ~ O(n) với n = size of data
  │
  └── Enqueue message event
      vào Worker's message queue
                                   ──────────────────────────
                                   Worker Event Loop picks up
                                   message event
                                   → onmessage callback runs
                                   → heavyComputation(data)
                                   → postMessage(result)
                                      │
                                      ├── Serialize result
                                      └── Enqueue message event
                                          vào Main Thread's queue
Main Thread receives
message event → onmessage
```

**Performance implication của serialization:**

```javascript
// Serialize 10MB ArrayBuffer:
// Structured Clone COPY toàn bộ 10MB → 10MB copy overhead
// Main thread và Worker mỗi bên có bản riêng

const bigBuffer = new ArrayBuffer(10 * 1024 * 1024); // 10MB
worker.postMessage({ buffer: bigBuffer });
// bigBuffer vẫn accessible ở main thread (đã copy)
// Worker nhận bản copy — tốn thêm 10MB memory + copy time

// SOLUTION: Transferable Objects
worker.postMessage({ buffer: bigBuffer }, [bigBuffer]);
// bigBuffer được TRANSFER (zero-copy) sang Worker
// Main thread KHÔNG còn access được bigBuffer sau đây
// bigBuffer.byteLength === 0 ở main thread
// Zero memory overhead — chỉ transfer ownership
```

### Transferable Objects — zero-copy transfer

Các types hỗ trợ transfer (không phải copy):

- `ArrayBuffer`
- `MessagePort`
- `ImageBitmap`
- `OffscreenCanvas`
- `ReadableStream`, `WritableStream`, `TransformStream`

```javascript
// Pattern: Producer/Consumer với ArrayBuffer transfer

// Main thread
async function processImage(imageData) {
  const worker = new Worker('./image-worker.js');

  // imageData.buffer là ArrayBuffer — transferable
  worker.postMessage(
    { pixels: imageData.data.buffer, width: imageData.width },
    [imageData.data.buffer], // Transfer list
  );
  // imageData.data.buffer.byteLength === 0 bây giờ
  // (đã transfer sang worker)

  return new Promise((resolve) => {
    worker.onmessage = (e) => {
      // Worker transfer buffer về
      const resultBuffer = e.data.buffer;
      resolve(new Uint8ClampedArray(resultBuffer));
    };
  });
}

// image-worker.js
self.onmessage = (e) => {
  const { pixels, width } = e.data;
  const data = new Uint8ClampedArray(pixels);

  // Process pixels (heavy computation)
  for (let i = 0; i < data.length; i += 4) {
    // Grayscale conversion
    const avg = (data[i] + data[i + 1] + data[i + 2]) / 3;
    data[i] = data[i + 1] = data[i + 2] = avg;
  }

  // Transfer buffer back — zero-copy
  self.postMessage({ buffer: pixels }, [pixels]);
};
```

### SharedArrayBuffer — shared memory giữa threads

`SharedArrayBuffer` cho phép **cùng một vùng memory** được access từ nhiều threads đồng thời. Đây là exception lớn so với model "no shared memory" của Workers.

```javascript
// Main thread tạo shared buffer
const sharedBuffer = new SharedArrayBuffer(4); // 4 bytes
const sharedArray = new Int32Array(sharedBuffer);
sharedArray[0] = 0; // Initial value

// Gửi buffer sang worker (không copy — thật sự share)
worker.postMessage({ buffer: sharedBuffer });

// Main thread có thể đọc/write cùng lúc với worker!
console.log(sharedArray[0]); // Có thể thay đổi bất kỳ lúc nào bởi worker
```

```javascript
// worker.js
self.onmessage = (e) => {
  const { buffer } = e.data;
  const sharedArray = new Int32Array(buffer);
  // Đây là CÙNG memory với main thread — không phải copy
  sharedArray[0] = 42; // Main thread thấy thay đổi này ngay lập tức
};
```

**Vấn đề ngay lập tức: Race conditions**

```javascript
// Main thread
sharedArray[0] += 1;

// Worker (chạy đồng thời)
sharedArray[0] += 1;

// Expected: 2
// Actual: có thể là 1!
// Vì "read-modify-write" không phải atomic operation
// Cả hai đọc 0 → cả hai write 1 → result là 1 thay vì 2
```

### Atomics — giải quyết race conditions

`Atomics` là một tập operations đảm bảo **atomic execution** — không thể bị interrupt bởi thread khác ở giữa chừng.

```javascript
// Atomic increment — an toàn
Atomics.add(sharedArray, 0, 1);
// Equivalent to: sharedArray[0] += 1
// NHƯNG: operation này là indivisible
// Không thread nào có thể "chen vào" giữa read và write

// Các Atomics operations:
Atomics.add(array, index, value); // atomic +=
Atomics.sub(array, index, value); // atomic -=
Atomics.and(array, index, value); // atomic &=
Atomics.or(array, index, value); // atomic |=
Atomics.xor(array, index, value); // atomic ^=
Atomics.exchange(array, index, val); // atomic set, return old value

// Compare-and-swap — foundation của locks
Atomics.compareExchange(array, index, expectedValue, newValue);
// If array[index] === expectedValue → set to newValue, return expectedValue
// Else → don't change, return current value
// Entire operation is atomic
```

**Atomics.wait và Atomics.notify — thread synchronization:**

```javascript
// Worker: chờ signal từ main thread
const WAITING = 0;
const READY = 1;

// worker.js
self.onmessage = (e) => {
  const { controlBuffer, dataBuffer } = e.data;
  const control = new Int32Array(controlBuffer);
  const data = new Float64Array(dataBuffer);

  while (true) {
    // Block worker thread cho đến khi control[0] !== WAITING
    // (Chỉ dùng được trong Worker — KHÔNG dùng được ở main thread
    //  vì sẽ block main thread event loop)
    Atomics.wait(control, 0, WAITING);

    // Xử lý data
    processData(data);

    // Signal main thread: xong rồi
    Atomics.store(control, 0, WAITING); // Reset
    Atomics.notify(control, 0, 1); // Wake main thread nếu đang wait
  }
};

// Main thread:
// Atomics.wait KHÔNG được dùng ở main thread
// Dùng Atomics.waitAsync() thay thế (non-blocking)
const result = Atomics.waitAsync(control, 0, WAITING);
result.value.then(() => {
  console.log('Worker finished!');
});
```

**Tại sao `Atomics.wait` bị cấm ở main thread:**
Nó block thread hoàn toàn. Nếu main thread block → event loop dừng → browser unresponsive → browser sẽ kill tab.

### Worker Pool Pattern — production architecture

Tạo Worker mới cho mỗi task là expensive (bootstrap time ~5-50ms). Production pattern là dùng **pool**:

```javascript
// worker-pool.js — reusable Worker pool

class WorkerPool {
  constructor(workerScript, poolSize = navigator.hardwareConcurrency || 4) {
    this.workers = [];
    this.queue = []; // Pending tasks
    this.idle = []; // Available workers

    for (let i = 0; i < poolSize; i++) {
      const worker = new Worker(workerScript);
      worker.onmessage = (e) => this._handleResult(worker, e.data);
      worker.onerror = (e) => this._handleError(worker, e);
      this.workers.push(worker);
      this.idle.push(worker);
    }
  }

  run(payload) {
    return new Promise((resolve, reject) => {
      const task = { payload, resolve, reject };

      if (this.idle.length > 0) {
        this._dispatch(this.idle.pop(), task);
      } else {
        // Không có worker rảnh → queue task
        this.queue.push(task);
      }
    });
  }

  _dispatch(worker, task) {
    // Tag worker với current task để match khi response về
    worker._currentTask = task;
    worker.postMessage(task.payload);
  }

  _handleResult(worker, result) {
    const task = worker._currentTask;
    task.resolve(result);
    worker._currentTask = null;

    if (this.queue.length > 0) {
      // Có task đang chờ → dispatch ngay
      this._dispatch(worker, this.queue.shift());
    } else {
      // Không có task → trả worker về pool
      this.idle.push(worker);
    }
  }

  _handleError(worker, error) {
    const task = worker._currentTask;
    task.reject(error);
    worker._currentTask = null;
    // Worker vẫn usable sau error
    this.idle.push(worker);
  }

  terminate() {
    this.workers.forEach((w) => w.terminate());
  }
}

// Usage:
const pool = new WorkerPool('./compute.worker.js', 4);

// Submit nhiều tasks — pool tự schedule
const results = await Promise.all(
  chunks.map((chunk) => pool.run({ type: 'PROCESS', data: chunk })),
);
```

### Comlink — abstraction layer cho Workers

`Comlink` (Google Chrome Labs) wrap Workers trong một Proxy, cho phép gọi Worker functions như normal async functions:

```javascript
// math-worker.js
import { expose } from 'comlink';

const api = {
  async fibonacci(n) {
    // Heavy computation — chạy trên Worker thread
    if (n <= 1) return n;
    return api.fibonacci(n - 1) + api.fibonacci(n - 2);
  },

  processDataset(data) {
    return data.map(expensiveTransform);
  },
};

expose(api);

// main.js
import { wrap } from 'comlink';

const worker = new Worker('./math-worker.js', { type: 'module' });
const api = wrap(worker);

// Gọi Worker function như async function thông thường
// Không cần postMessage/onmessage boilerplate
const result = await api.fibonacci(40); // Runs on worker thread
const processed = await api.processDataset(largeArray);
```

Comlink dùng `MessageChannel` và `Proxy` để tạo illusion của direct function call — internally vẫn là `postMessage` + serialization.

### postMessage vs SharedArrayBuffer — data flow

```
postMessage (Structured Clone):
Main ──[serialize: copy 10MB]──► Queue ──[deserialize]──► Worker
      ↑ tốn 10MB + copy time                ↑ tốn thêm 10MB

postMessage với Transfer:
Main ──[transfer ownership]──► Worker
      ↑ zero copy, O(1)
      ↑ Main mất quyền access

SharedArrayBuffer:
Main ◄──[same physical memory]──► Worker
         ↑ zero copy
         ↑ cả hai đọc/write cùng lúc
         ↑ cần Atomics để synchronize
```

### Race condition và Atomics fix

```
WITHOUT Atomics:                  WITH Atomics:

Thread A: read  sharedArray[0]=0  Thread A: Atomics.add → indivisible
Thread B: read  sharedArray[0]=0  Thread B: phải chờ Thread A xong
Thread A: +1 → write 1            Thread A: write 1
Thread B: +1 → write 1            Thread B: read 1 → +1 → write 2
Result: 1  ← WRONG                Result: 2  ← CORRECT
```

```javascript
// ✅ FIX: Parse trên Worker thread

// parse-worker.js
self.onmessage = (e) => {
  const { text, id } = e.data;
  try {
    const data = JSON.parse(text); // Chạy trên Worker — không ảnh hưởng main thread
    self.postMessage({ id, data, error: null });
  } catch (err) {
    self.postMessage({ id, data: null, error: err.message });
  }
};

// main.js
class BackgroundParser {
  constructor() {
    this.worker = new Worker('./parse-worker.js');
    this.pending = new Map();
    this.nextId = 0;

    this.worker.onmessage = (e) => {
      const { id, data, error } = e.data;
      const { resolve, reject } = this.pending.get(id);
      this.pending.delete(id);
      error ? reject(new Error(error)) : resolve(data);
    };
  }

  parse(text) {
    return new Promise((resolve, reject) => {
      const id = this.nextId++;
      this.pending.set(id, { resolve, reject });
      this.worker.postMessage({ text, id });
      // Main thread không bị block — tiếp tục handle events
    });
  }
}

const parser = new BackgroundParser();

async function loadDashboardData() {
  const response = await fetch('/api/analytics?range=90days');
  const text = await response.text();

  // Main thread tự do trong lúc Worker parse
  // Animations continue, user input handled
  const data = await parser.parse(text);
  renderDashboard(data);
}
```

### SharedArrayBuffer + Atomics: Real-time progress tracking

```javascript
// Pattern: Worker báo cáo progress về main thread
// Không cần postMessage cho mỗi update — dùng shared memory

// main.js
async function runHeavyTask(inputData) {
  const worker = new Worker('./heavy-worker.js');

  // Shared control buffer: [progress(0-100), status(0=running,1=done,2=error)]
  const controlBuffer = new SharedArrayBuffer(8); // 2 x Int32
  const control = new Int32Array(controlBuffer);

  // Transfer input data (zero-copy)
  const inputBuffer = new ArrayBuffer(inputData.byteLength);
  new Uint8Array(inputBuffer).set(new Uint8Array(inputData));

  worker.postMessage({ controlBuffer, inputBuffer }, [inputBuffer]);

  // Poll progress via SharedArrayBuffer — no message overhead
  return new Promise((resolve, reject) => {
    const checkProgress = () => {
      const progress = Atomics.load(control, 0); // atomic read
      const status = Atomics.load(control, 1);

      updateProgressBar(progress); // Update UI

      if (status === 1) {
        resolve(); // Done
      } else if (status === 2) {
        reject(new Error('Worker error'));
      } else {
        requestAnimationFrame(checkProgress); // Check next frame
      }
    };
    requestAnimationFrame(checkProgress);
  });
}

// heavy-worker.js
self.onmessage = (e) => {
  const { controlBuffer, inputBuffer } = e.data;
  const control = new Int32Array(controlBuffer);
  const input = new Uint8Array(inputBuffer);

  try {
    for (let i = 0; i < input.length; i++) {
      processItem(input[i]);

      // Update progress atomically — main thread reads this without postMessage
      if (i % 1000 === 0) {
        const progress = Math.floor((i / input.length) * 100);
        Atomics.store(control, 0, progress); // atomic write
      }
    }
    Atomics.store(control, 1, 1); // Signal: done
  } catch (err) {
    Atomics.store(control, 1, 2); // Signal: error
  }
};
```

## 4. Ứng dụng thực tế

### Khi nào dùng Worker

```
CPU-bound > 50ms          → Dedicated Worker
────────────────────────────────────────────
JSON parse > 1MB          ✓ Worker
Image processing          ✓ Worker + OffscreenCanvas
Crypto operations         ✓ Worker
ML inference (ONNX.js)    ✓ Worker
Large array sorting       ✓ Worker
Markdown/code parsing     ✓ Worker (Shiki, Marked)

I/O-bound                 → KHÔNG cần Worker
────────────────────────────────────────────
fetch requests            ✗ async/await đủ rồi
IndexedDB                 ✗ async đủ
setTimeout delays         ✗ không cần

State sharing giữa tabs   → Shared Worker hoặc BroadcastChannel
Network intercept/cache   → Service Worker
```

### Real apps dùng Workers như thế nào

**Figma:** Rendering engine chạy hoàn toàn trong Worker với WebAssembly. Main thread chỉ handle UI interactions và gửi commands sang Worker.

**Google Docs:** Collaborative editing conflict resolution (OT/CRDT algorithms) chạy trong Worker — heavy computation không block typing.

**VS Code Web:** Language servers (TypeScript, ESLint) chạy trong Workers. Type-checking một file lớn không block editor.

**React (experimental):** React concurrent renderer thực nghiệm với Worker — reconciliation chạy off-main-thread.

### Bảo mật: Tại sao SharedArrayBuffer bị disable rồi re-enable

**2018 — Spectre vulnerability:**

`Atomics.wait` + high-resolution timer (`SharedArrayBuffer` làm được timer rất chính xác) cho phép side-channel attack đọc memory từ process khác. **Tất cả browsers disable `SharedArrayBuffer`** sau khi Spectre được công bố.

**2020 — Re-enable với điều kiện:**

`SharedArrayBuffer` được re-enable khi page có **cross-origin isolation**:

```javascript
// Server phải gửi 2 headers:
// Cross-Origin-Opener-Policy: same-origin
// Cross-Origin-Embedder-Policy: require-corp

// Kiểm tra trong code:
if (crossOriginIsolated) {
  // SharedArrayBuffer available
  const sab = new SharedArrayBuffer(1024);
} else {
  // Fallback: postMessage thay thế
}
```

Cross-origin isolation ngăn page load cross-origin resources không có explicit permission → giảm attack surface của Spectre.

### Câu 1: Worker Event Loop khác Main Thread Event Loop ở điểm gì? Tại sao `Atomics.wait()` bị cấm ở main thread?

**Đáp án:**

**Khác biệt cốt lõi — không có Render step:**

Main thread event loop: `Task → Microtasks → Render → Task → ...`

Worker event loop: `Task → Microtasks → Task → ...`

Worker không có rendering pipeline vì Worker không có DOM access, không có `window`, không thể paint pixels. `requestAnimationFrame` không hoạt động trong Worker thông thường (chỉ hoạt động với OffscreenCanvas). Worker cũng không có Web APIs liên quan đến UI: `document`, `alert`, `history`, `location`...

**Worker có đầy đủ:** `fetch`, `setTimeout`, `Promise`, `WebSocket`, `IndexedDB`, `Crypto`, `importScripts`.

**Tại sao `Atomics.wait()` bị cấm ở main thread:**

`Atomics.wait(array, index, value)` **block thread** cho đến khi `array[index] !== value`. Nếu chạy trên main thread, toàn bộ event loop dừng lại. Không có tasks được xử lý, không có microtasks, không có rendering. Browser coi đây là "unresponsive page" và có thể kill tab sau vài giây.

Thay thế cho main thread: `Atomics.waitAsync()` — non-blocking version, trả về Promise được resolve khi điều kiện thỏa mãn. Main thread event loop không bị block, Promise callback được enqueue vào microtask queue khi condition fires.

### Câu 2: Structured Clone Algorithm khác JSON.stringify/parse như thế nào? Khi nào nên dùng Transferable Objects thay vì clone?

**Đáp án:**

**Structured Clone hỗ trợ nhiều types hơn JSON:**

| Type          | JSON               | Structured Clone |
| ------------- | ------------------ | ---------------- |
| `Date`        | Mất (thành string) | ✓ Preserve       |
| `RegExp`      | Mất                | ✓ Preserve       |
| `Map`, `Set`  | Mất                | ✓ Preserve       |
| `ArrayBuffer` | Không hỗ trợ       | ✓ Copy buffer    |
| `undefined`   | Bị drop hoặc null  | ✓ Preserve       |
| Circular refs | Throw error        | ✓ Handle         |
| Functions     | Bị drop            | ✗ Không hỗ trợ   |
| DOM nodes     | Bị drop            | ✗ Không hỗ trợ   |
| Prototypes    | Mất                | ✗ Plain object   |

Structured Clone tạo **deep copy** hoàn toàn — `O(n)` với n là tổng size của data. Memory footprint tăng gấp đôi trong lúc transfer (bản gốc + bản copy).

**Dùng Transferable Objects khi:**

1. **Data lớn:** ArrayBuffer hàng MB → clone sẽ tốn thời gian và memory đáng kể. Transfer là `O(1)` — chỉ transfer ownership pointer.

2. **Không cần giữ bản gốc:** Sau transfer, bên gửi không còn access được data. Phù hợp với pipeline: main → worker process → worker trả về kết quả.

3. **Latency-sensitive:** Game loops, audio processing, real-time data — mỗi ms copy overhead có thể gây jitter.

**Không dùng Transfer khi:**

- Cần giữ reference ở cả hai bên (dùng `SharedArrayBuffer` thay thế)
- Data nhỏ (< vài KB) — overhead của Transfer setup lớn hơn clone cost
- Cần fallback compatibility (Transfer có thể fail silently trong một số environments)

### Câu 3: Đoạn code sau có race condition không? Nếu có, fix bằng Atomics như thế nào?

```javascript
// shared-counter.js (chạy trong 4 Workers đồng thời)
const sharedBuffer = new SharedArrayBuffer(4);
const counter = new Int32Array(sharedBuffer);

// Mỗi Worker chạy hàm này 10,000 lần
function incrementCounter() {
  const current = counter[0];
  counter[0] = current + 1;
}

for (let i = 0; i < 10000; i++) {
  incrementCounter();
}

// Main thread: sau khi tất cả Workers xong
console.log(counter[0]); // Expected: 40,000 — Actual: ???
```

**Đáp án:**

**Có race condition nghiêm trọng.** Expected 40,000 nhưng actual value sẽ nhỏ hơn — thường đâu đó 10,000–30,000, và mỗi lần chạy cho kết quả khác nhau.

**Tại sao:** `counter[0] = current + 1` là ba operations riêng biệt ở CPU level:

1. Load `counter[0]` vào CPU register
2. Add 1
3. Store kết quả về `counter[0]`

Giữa bất kỳ hai bước nào, OS scheduler có thể context-switch sang Worker khác. Ví dụ:

- Worker A reads `counter[0] = 5`
- Context switch → Worker B reads `counter[0] = 5`
- Worker B writes `counter[0] = 6`
- Worker A writes `counter[0] = 6` (ghi đè result của B)
- Hai increments chỉ tạo ra một unit tăng

**Fix với `Atomics.add`:**

```javascript
// Fixed: incrementCounter
function incrementCounter() {
  Atomics.add(counter, 0, 1);
  // Atomic: read-modify-write là một indivisible operation
  // Không có thread nào có thể đọc giá trị stale giữa read và write
}

// Kết quả: luôn đúng 40,000
// Trade-off: Atomic operations chậm hơn non-atomic (~5-10x)
// vì cần memory fence, cache coherency protocols giữa CPU cores
// Nhưng đúng > nhanh mà sai
```

**Bonus — check kết quả từ main thread:**

```javascript
// Chờ tất cả Workers xong (dùng Promise.all với Worker message)
// Sau đó:
const finalCount = Atomics.load(counter, 0);
// Atomics.load đảm bảo đọc giá trị "up-to-date" nhất
// (memory barrier — flush CPU cache)
console.log(finalCount); // 40,000
```
