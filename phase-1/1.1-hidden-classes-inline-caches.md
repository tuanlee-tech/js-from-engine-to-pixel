# Phase 1.1 — Hidden Classes & Inline Caches

---

## 1. Cơ chế thật

### Vấn đề V8 phải giải quyết

JavaScript là **dynamically typed**. Trong C++, compiler biết tại compile-time rằng `user.name` nằm ở offset 8 bytes từ đầu struct → một instruction `MOV rax, [rbx+8]` là xong.

Trong JS, không ai đảm bảo:

- `user` có property `name` không?
- `name` nằm ở vị trí nào trong memory?
- `user` có bị thêm/xóa property sau đó không?

Nếu V8 naive, mỗi property access phải **lookup trong một hash table** — tìm key `"name"`, resolve pointer. Expensive. Không thể viết hệ thống production với tốc độ đó.

**Hidden Classes (còn gọi là Maps trong V8 source code)** là giải pháp V8 dùng để đưa dynamic JS gần với static language performance.

---

### Hidden Class là gì — cơ chế thật

Mỗi khi bạn tạo một object, V8 gắn cho nó một **hidden class (Map)** — một internal metadata object mô tả **shape** của object đó: danh sách properties và **offset** của từng property trong memory.

```javascript
const point = {};
// V8: tạo object, gắn hidden class HC0 = { (empty) }
```

```
Object "point"           Hidden Class HC0
┌───────────┐            ┌─────────────────────┐
│ map ptr ──┼───────────►│ "shape": {}         │
│ (slots)   │            │ num properties: 0   │
└───────────┘            └─────────────────────┘
```

```javascript
point.x = 10;
// V8: HC0 chưa có "x"
// → Tạo HC1 = { x at offset 0 }
// → point.map = HC1
// → HC0 lưu "transition: nếu thêm 'x' → đi tới HC1"
```

```
Object "point"           Hidden Class HC1
┌───────────┐            ┌─────────────────────┐
│ map ptr ──┼───────────►│ "shape": { x: 0 }  │
│ slot[0]=10│            │ x → offset 0        │
└───────────┘            │ ← từ HC0 + "x"      │
                         └─────────────────────┘
                                   ▲
                         HC0 transition table:
                         "add x" → HC1
```

```javascript
point.y = 20;
// V8: HC1 chưa có "y"
// → Tạo HC2 = { x at offset 0, y at offset 1 }
// → point.map = HC2
// → HC1 lưu "transition: nếu thêm 'y' → đi tới HC2"
```

Kết quả cuối:

```
Object "point"           Hidden Class HC2
┌───────────┐            ┌──────────────────────┐
│ map ptr ──┼───────────►│ shape: { x:0, y:1 } │
│ slot[0]=10│            │ x → offset 0         │
│ slot[1]=20│            │ y → offset 1         │
└───────────┘            └──────────────────────┘
```

**Bây giờ `point.x` là một O(1) operation:** V8 nhìn vào hidden class → thấy "x ở offset 0" → `MOV rax, [object_ptr + 0]`. Đúng một instruction.

---

### Transition Tree — tại sao thứ tự quan trọng

Khi nhiều objects có **cùng thứ tự thêm properties**, chúng **share hidden classes**:

```javascript
const p1 = {};
p1.x = 1; // HC0 → HC1
p1.y = 2; // HC1 → HC2

const p2 = {};
p2.x = 3; // HC0 → HC1 (cùng transition!)
p2.y = 4; // HC1 → HC2 (cùng transition!)

// p1 và p2 SHARE hidden class HC2
// V8 chỉ cần lưu một HC2 cho cả hai
```

Nhưng nếu thứ tự khác nhau:

```javascript
const p3 = {};
p3.y = 5; // HC0 → HC3 (new! "add y first")
p3.x = 6; // HC3 → HC4 (new!)

// p3 có hidden class HC4 — khác hoàn toàn với HC2
// Dù p3 có cùng properties x, y như p1, p2
```

Transition tree trông như thế này:

```
           HC0 (empty)
          /           \
    "add x"           "add y"
        /                 \
      HC1{x}             HC3{y}
        |                   |
    "add y"             "add x"
        |                   |
      HC2{x,y}           HC4{y,x}
      ↑ p1,p2              ↑ p3
      SHARED             ISOLATED
```

**Hệ quả:** Code access `p1.x` và `p3.x` không thể share optimization vì chúng có hidden class khác nhau — dù cùng property name.

---

### Inline Caches (IC) — nơi tốc độ thực sự đến từ

Bây giờ hidden classes đã cho biết offset của property. Nhưng V8 không muốn lookup hidden class mỗi lần — nó muốn **cache kết quả lookup đó ngay tại call site**.

Đây là **Inline Cache**.

Mỗi property access trong bytecode có một **feedback slot** (đã đề cập ở buổi trước). Lần đầu chạy:

```
1. Load obj.name
2. Look up hidden class của obj → HC42
3. HC42 nói: "name ở offset 4"
4. Load value từ offset 4
5. Ghi vào feedback slot: { hidden_class: HC42, offset: 4 }
```

Lần thứ hai trở đi:

```
1. Load obj.name
2. Check feedback slot: "lần trước, HC42 → offset 4"
3. Check nhanh: obj.map == HC42? ← một instruction duy nhất
4. YES → Load từ offset 4 trực tiếp
   NO  → IC miss → fallback lookup
```

Check `obj.map == HC42` cực kỳ nhanh — đây là **pointer comparison**, một instruction. Nếu match, không cần lookup gì cả.

---

### IC States — Monomorphic, Polymorphic, Megamorphic

Đây là 3 trạng thái V8 track cho mỗi call site:

```
[Uninitialized] → lần đầu tiên thấy
      │
      ▼
[Monomorphic] → luôn thấy cùng 1 hidden class
  "HC42 → offset 4"
  Performance: ⚡⚡⚡ FASTEST
      │
      │ (thấy HC99 lần đầu)
      ▼
[Polymorphic] → thấy 2–4 hidden classes khác nhau
  "HC42 → offset 4, HC99 → offset 8, HC7 → offset 0"
  V8 lưu danh sách nhỏ, check từng cái
  Performance: ⚡⚡ OK
      │
      │ (thấy quá nhiều hidden classes)
      ▼
[Megamorphic] → 5+ hidden classes khác nhau
  V8 từ bỏ caching, fallback về global hash table lookup
  Performance: ⚡ SLOW — gần như unoptimized
```

**Megamorphic là trạng thái không thể recover.** Khi một call site đã megamorphic, V8 sẽ không bao giờ inline cache nó nữa, kể cả khi sau đó chỉ còn một type.

---

### Deoptimization từ Hidden Class violation

Khi TurboFan compile optimized machine code cho một function, nó **bakes in** hidden class assumptions:

```javascript
function getX(point) {
  return point.x; // IC feedback: "point luôn có HC2, x ở offset 0"
}

// TurboFan compile thành machine code giả định:
// ASSERT point.map == HC2
// MOV rax, [point + offset_0]
// (nếu assertion fail → deopt)
```

Nếu sau đó bạn truyền vào một object có hidden class khác:

```javascript
getX({ x: 1, y: 2 }); // HC2 — OK
getX({ x: 5, y: 8 }); // HC2 — OK
getX({ y: 2, x: 1 }); // HC4! — Deopt! (y trước x → hidden class khác)
```

Deoptimization pipeline:

```
[Machine Code]
     │
 ASSERT map == HC2 ← FAILS (got HC4)
     │
 [Deopt Handler]
     ├── Reconstruct interpreter frame
     ├── FeedbackVector: update IC → now Polymorphic [HC2, HC4]
     └── Return to Ignition bytecode
          │
          (sau nhiều lần mixed hidden classes)
          │
     [TurboFan re-compile] ← version mới với 2 cases
     HOẶC
     [Megamorphic → never optimize again]
```

### Object.defineProperty và sự khác biệt với property thường

Properties được thêm bằng assignment (`obj.x = 1`) khác với properties được define bằng `Object.defineProperty`. Cái sau tạo **accessor properties** hoặc thay đổi **property descriptors** — V8 phải dùng một hidden class riêng vì shape phức tạp hơn.

```javascript
// Fast property — V8 có thể inline cache hoàn toàn
const obj1 = { x: 1 };

// Accessor property — V8 phải gọi getter mỗi lần, không thể cache value
const obj2 = {};
Object.defineProperty(obj2, 'x', {
  get() {
    return 1;
  },
});

// Chúng có hidden classes khác nhau
// obj2.x không thể được tối ưu như obj1.x
```

### In-object slots vs Named properties store

V8 có hai nơi lưu property values:

**In-object slots:** Property được lưu trực tiếp trong object memory layout. Access là `[object_ptr + offset]` — cực nhanh.

**Named properties store (property backing store):** Khi object có quá nhiều properties, hoặc khi properties được thêm sau construction, V8 có thể chuyển sang lưu trong một separate array. Access chậm hơn vì thêm một pointer dereference.

```
Trường hợp thông thường (few props, defined at construction):

Object layout:
┌─────────────┐
│ map ptr     │ → HC
│ prop1 value │ ← in-object slot (fast)
│ prop2 value │ ← in-object slot (fast)
└─────────────┘

Trường hợp nhiều props hoặc dynamic:

Object layout:
┌─────────────────┐
│ map ptr         │ → HC
│ in-obj slot 1   │ ← fast
│ in-obj slot 2   │ ← fast
│ properties ptr  │──► [ prop3, prop4, prop5... ] ← slow
└─────────────────┘
```

**Tại sao quan trọng:** Objects được tạo bằng object literal `{ a, b, c }` V8 biết trước số properties → allocate in-object slots cho tất cả. Objects được build dần dần bằng assignments có thể overflow sang backing store.

### Pattern nâng cao: Object pools để maintain hidden class

```javascript
// Pattern dùng trong game engines, real-time data processing
// Khi cần tạo/hủy objects liên tục → GC pressure + hidden class churn

class ParticlePool {
  constructor(size) {
    // Pre-allocate pool với CÙNG hidden class
    // TẤT CẢ particles có shape {x, y, vx, vy, life, active}
    this.pool = Array.from({ length: size }, () => ({
      x: 0,
      y: 0,
      vx: 0,
      vy: 0,
      life: 0,
      active: false,
    }));
    this.nextFree = 0;
  }

  acquire() {
    // Reuse object từ pool — hidden class KHÔNG ĐỔI
    // Không tạo object mới → không có GC → không có hidden class transition
    const particle = this.pool[this.nextFree++ % this.pool.length];
    particle.active = true;
    return particle;
  }

  release(particle) {
    // Reset values, KHÔNG xóa properties
    // Xóa property sẽ tạo hidden class mới!
    particle.x = 0;
    particle.y = 0;
    particle.active = false;
    // particle.someNewProp = ... ← ĐỪng làm vậy
  }
}
```

## 4. Ứng dụng thực tế

### Performance impact thực sự

Hidden class miss không phải là catastrophic một lần — nó là **death by a thousand cuts**:

```
Monomorphic property access:  ~1 cycle (cached → direct MOV)
Polymorphic (2-4 classes):    ~3-5 cycles (check list)
Megamorphic:                  ~20-50 cycles (hash table lookup)
```

Một component React render 1000 items, mỗi item access 5 properties → **5000 property accesses per render**. Nếu megamorphic: 5000 × 50 cycles = **250,000 cycles per render**. Trên 1GHz effective throughput = 250 microseconds chỉ cho property access, trong một render 16ms budget.

**Không catastrophic một mình — nhưng combined với layout, paint, JS logic, nó là straw that breaks the camel's back.**

| Pattern                                         | Tại sao                                | Hệ quả                                |
| ----------------------------------------------- | -------------------------------------- | ------------------------------------- |
| Khởi tạo tất cả properties trong constructor    | Cùng hidden class cho mọi instance     | Monomorphic IC, TurboFan optimize tốt |
| Giữ thứ tự properties nhất quán                 | Transition tree chia sẻ hidden classes | Ít GC pressure (shared HCs)           |
| Không thêm/xóa properties sau construction      | Tránh hidden class transitions         | Giữ IC stable                         |
| Normalize API responses trước khi vào component | Tất cả objects cùng shape              | Component luôn monomorphic            |
| Dùng `null` thay vì absent cho optional fields  | Shape nhất quán                        | Không fork hidden class               |
| Object pools cho high-frequency allocation      | Reuse objects, giữ hidden class        | Zero GC pressure, no IC churn         |

### Câu 1: Hidden Class là gì và tại sao V8 cần nó? Giải thích cơ chế tạo và transition.

**Đáp án:**

**Vấn đề gốc:** JavaScript là dynamic — property có thể được thêm/xóa bất kỳ lúc nào, không có static type declaration. Nếu V8 naive, mỗi property access phải tra trong hash table (`obj["name"] → ?`), rất chậm so với static languages.

**Giải pháp — Hidden Class (Map):** V8 gắn cho mỗi object một **hidden class** — một internal metadata object mô tả shape của object: danh sách properties và **memory offset** của từng property. Với thông tin này, `obj.name` trở thành `[obj_ptr + offset_of_name]` — một instruction duy nhất.

**Cơ chế transition:** Hidden classes được tạo theo kiểu **incremental**. Khi object được tạo (`{}`), nó nhận HC0 (empty). Khi thêm property `x`, V8 tạo HC1 (`{x at offset 0}`) và lưu **transition edge** trong HC0: "nếu thêm property 'x' → đến HC1". Khi thêm `y`, V8 tạo HC2 (`{x:0, y:1}`) với edge trong HC1.

**Tại sao thứ tự quan trọng:** Hai objects với cùng properties nhưng **thêm theo thứ tự khác nhau** sẽ đi theo các paths khác nhau trong transition tree, dẫn đến different hidden classes. Objects chỉ share hidden classes khi chúng đi qua cùng sequence of transitions.

**Khi objects share hidden class:** V8 tối ưu memory (một hidden class object dùng chung) và quan trọng hơn, **Inline Caches** có thể assume cùng offset cho property — tất cả objects với cùng hidden class đều benefit từ một IC entry.

### Câu 2: Tại sao Megamorphic IC tệ hơn Polymorphic IC, và khi nào nên lo lắng về điều này?

**Đáp án:**

**IC States và mechanics:**

- **Monomorphic:** Một hidden class duy nhất. IC lưu `{HC → offset}`. Check là một pointer comparison (`obj.map == HC42?`). Nếu match → load trực tiếp. TurboFan có thể compile thành một `MOV` instruction.

- **Polymorphic (2–4 classes):** IC lưu một danh sách nhỏ. V8 check từng entry theo thứ tự. Chậm hơn monomorphic nhưng vẫn bounded — 2–4 comparisons.

- **Megamorphic (5+ classes):** V8 **từ bỏ inline cache hoàn toàn** cho call site này. Mỗi access fallback về global stub cache — essentially một hash table lookup. **Và trạng thái này không thể reverse** — kể cả khi sau đó chỉ còn một hidden class, call site vẫn megamorphic mãi.

**Tại sao megamorphic không thể recover:** V8 không có mechanism để "downgrade" một IC từ megamorphic về polymorphic/monomorphic trong một running program. Sẽ cần code được load lại từ đầu (page reload, worker restart).

**Khi nào cần lo:** Chỉ lo khi call site thực sự hot — một function được gọi hàng ngàn lần/giây với inputs có nhiều hidden classes. Ví dụ điển hình: render function trong React với 1000+ items, Node.js request handler với mixed request shapes, utility functions được dùng chung cho nhiều object types.

**Khi nào không cần lo:** Code chạy một vài lần, initialization code, event handlers ít trigger. Megamorphic IC overhead chỉ có ý nghĩa khi tích lũy qua hàng triệu calls.

### Câu 3: Đoạn code sau có vấn đề gì với Hidden Classes? Có bao nhiêu hidden classes được tạo ra và tại sao điều đó ảnh hưởng tới performance?

```javascript
function processApiResponse(items) {
  return items.map((item) => {
    const result = {
      id: item.id,
      value: item.value,
    };

    if (item.metadata) {
      result.metadata = item.metadata;
    }
    if (item.tags && item.tags.length > 0) {
      result.tags = item.tags;
      result.tagCount = item.tags.length;
    }

    return result;
  });
}
```

**Đáp án:**

**Số hidden classes được tạo:** Tối đa **4 hidden classes** cho `result`, tùy vào combination của conditions:

```
HC_base {id, value}                         ← metadata=false, tags=empty
HC_meta {id, value, metadata}               ← metadata=true, tags=empty
HC_tags {id, value, tags, tagCount}         ← metadata=false, tags=nonempty
HC_full {id, value, metadata, tags, tagCount} ← cả hai đều true
```

**Thứ tự thêm property cũng tạo ra hidden class khác nhau nếu cùng set properties nhưng order khác — nhưng trong case này, thứ tự consistent trong mỗi branch nên chủ yếu chỉ là 4 shapes.**

**Hệ quả:**

1. **Return values của `processApiResponse` có 4 different hidden classes.** Bất kỳ code nào consume array này và access properties sẽ thấy polymorphic hoặc megamorphic IC ngay lập tức.

2. **Không thể TurboFan-optimize effectively:** Function nhận `result` objects từ array này sẽ thấy multiple shapes — bị deopt hoặc compile generic version.

3. **GC pressure:** Không có gì đặc biệt về allocation, nhưng 4 different hidden class objects phải được tạo và maintained trong memory thay vì 1.

**Fix:**

```javascript
function processApiResponse(items) {
  return items.map((item) => ({
    id: item.id,
    value: item.value,
    metadata: item.metadata ?? null, // luôn present, null nếu không có
    tags: item.tags?.length > 0 ? item.tags : null, // luôn present
    tagCount: item.tags?.length ?? 0, // luôn present, 0 nếu không có
  }));
}
// Tất cả result objects có cùng hidden class {id, value, metadata, tags, tagCount}
// Dù giá trị là null hay real data — shape nhất quán
```
