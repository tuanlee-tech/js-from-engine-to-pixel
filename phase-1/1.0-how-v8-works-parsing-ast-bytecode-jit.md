> **Phạm vi buổi này:** Đủ lớn để chia 2 buổi, nhưng phần Parsing → Bytecode → JIT là một pipeline liên tục — không thể hiểu JIT nếu bỏ qua Parsing. Tôi sẽ dạy hết trong một buổi, nhưng **Hidden Classes & Inline Caches** (bullet thứ 3 của 1.1) sẽ là buổi riêng vì nó deserves chiều sâu riêng.

# Phase 1.1 — How V8 Works: Parsing → AST → Bytecode → JIT

---

## 1. Cơ chế thật

### Câu hỏi đầu tiên: V8 nhận gì và làm gì với nó?

V8 nhận **plain text** — một chuỗi UTF-16 đơn thuần. Không có magic. File `.js` của bạn là text. Nhiệm vụ của V8 là biến text đó thành thứ CPU có thể chạy, nhanh nhất có thể.

Pipeline đầy đủ:

```
Source Text (UTF-16)
      │
      ▼
  [Scanner]  — Tokenization
      │
      ▼
  [Parser]   — Produces AST
  (Pre-parser + Full Parser)
      │
      ▼
  [Ignition] — AST → Bytecode + Type Feedback
      │
      ├──(cold code)──► Bytecode chạy mãi
      │
      ▼
  [Maglev]   — Mid-tier JIT (V8 ≥ 2023, 🟢)
      │
      ▼
  [TurboFan] — Optimizing JIT → Machine Code
      │
      ▼
  [Deopt?]   — Bail out về Ignition nếu assumption sai
```

---

### Bước 1: Scanner (Tokenization)

Scanner đọc source text từ trái sang phải, nhóm các ký tự thành **tokens** — đơn vị có nghĩa nhỏ nhất.

```
const userAge = 25 + 1;
```

Scanner tạo ra stream tokens:

```
[KEYWORD: const]
[IDENTIFIER: userAge]
[PUNCTUATOR: =]
[NUMERIC_LITERAL: 25]
[PUNCTUATOR: +]
[NUMERIC_LITERAL: 1]
[PUNCTUATOR: ;]
```

Scanner không quan tâm ý nghĩa — nó chỉ nhóm ký tự. Sau đó **Parser** nhận token stream này.

**Điều V8 làm thông minh:** V8 scanner xử lý **UTF-16**, không phải UTF-8. JS spec yêu cầu UTF-16 vì lịch sử. Điều này ảnh hưởng tới emoji/surrogate pairs — topic khác.

---

### Bước 2: Parsing — Pre-parser và Full Parser

Đây là phần nhiều người không biết: **V8 có hai parser chạy song song**, phục vụ hai mục đích khác nhau.

#### Full Parser (Eager)

Parse code thành **AST đầy đủ**. Tốn CPU và memory. Chỉ chạy với code **sẽ chạy ngay**.

#### Pre-parser (Lazy) — cơ chế quan trọng nhất của bước này

Pre-parser chạy trước với code bên trong **function bodies chưa được gọi**. Nó **không** tạo AST đầy đủ — chỉ làm đủ để:

1. Kiểm tra syntax error
2. Thu thập variable declarations
3. Xác định scope structure

**Tại sao quan trọng:** Khi browser tải 500KB JS, nếu full-parse tất cả ngay lập tức, startup time sẽ rất chậm. Pre-parser cho phép V8 bỏ qua các function không cần thiết ngay lúc khởi động.

```javascript
// V8 EAGER PARSE cái này — nó chạy ngay khi file load
const config = { apiUrl: 'https://api.example.com' };

// V8 PRE-PARSE cái này — chỉ check syntax, không tạo AST
// AST chỉ được tạo khi handleClick() thực sự được gọi lần đầu
function handleClick(event) {
  const button = event.target;
  fetch(config.apiUrl + '/submit').then((res) => res.json());
}

// V8 EAGER PARSE cái này — IIFE chạy ngay
(function initApp() {
  document.addEventListener('DOMContentLoaded', handleClick);
})();
```

**Gotcha từ cơ chế này:** Pre-parser → Full parser có overhead. Nếu function được gọi ngay sau khi parse, V8 sẽ parse nó hai lần. Đây là lý do tại sao Parcel/Webpack có các `/* @__PURE__ */` và magic comments để hint cho bundler — không phải cho V8 trực tiếp.

---

### Bước 3: AST — Abstract Syntax Tree

Parser tạo ra một tree structure. Mỗi node trong tree là một **construct** của JavaScript.

Code: `function calculateTax(price, rate) { return price * rate; }`

```
FunctionDeclaration
├── id: Identifier (name: "calculateTax")
├── params:
│   ├── Identifier (name: "price")
│   └── Identifier (name: "rate")
└── body: BlockStatement
    └── ReturnStatement
        └── BinaryExpression (operator: "*")
            ├── left: Identifier (name: "price")
            └── right: Identifier (name: "rate")
```

**AST là representation ngữ nghĩa, không phải syntactic.** Khoảng trắng, comments, dấu chấm phẩy — đều bị bỏ đi.

**Bạn có thể tự inspect AST:** [astexplorer.net](https://astexplorer.net) — paste bất kỳ JS nào để xem AST thực tế. Đây là tool Babel, ESLint, TypeScript dùng khi transform code.

**Điều Ignition nhận:** Không phải AST raw — V8 có một bước trung gian là tạo ra **bytecode-friendly IR** từ AST trước khi đưa cho Ignition. Nhưng về mặt conceptual, Ignition bắt đầu từ AST.

---

### Bước 4: Ignition — Bytecode Interpreter

Ignition là một **register-based virtual machine**. Nó walk qua AST và tạo ra **bytecode** — một instruction set compact, platform-independent, được lưu dưới dạng `BytecodeArray` trên V8 heap.

**Tại sao không compile thẳng sang machine code?**

Trước V8 5.9 (2017), V8 dùng **full-codegen** — parse xong là compile thẳng sang machine code luôn. Vấn đề:

- Machine code chiếm nhiều memory hơn bytecode ~10x
- Không có phase để collect type information trước khi optimize
- Startup time chậm vì compile tất cả trước khi chạy

Ignition giải quyết cả ba vấn đề: bytecode nhỏ hơn, Ignition collect **type feedback** trong khi chạy, và code chỉ được JIT compile khi thực sự "hot".

**Bytecode thực tế cho `function add(a, b) { return a + b; }`:**

```
// Bạn có thể xem bytecode thật bằng:
// node --print-bytecode --print-bytecode-filter=add script.js

[generated bytecode for function: add]
Parameter count 3   // 'this' + a + b
Register count 0
Frame size 0
   0 : 25 02       Ldar a1        // Load register a1 (param 'a') vào accumulator
   2 : 34 03 00    Add a2, [0]    // accumulator += a2 (param 'b'), slot [0] là feedback slot
   5 : ab          Return         // Return giá trị trong accumulator
```

`Ldar` = Load Accumulator from Register. Ignition dùng **accumulator** (một register đặc biệt) cho kết quả trung gian.

**Feedback Vector — cơ chế quan trọng nhất của Ignition:**

Mỗi function có một **FeedbackVector** — một array slot được allocate song song với bytecode. Mỗi operation có feedback slots riêng:

```
FeedbackVector cho hàm add:
Slot [0] (Add operation):
  - Lần gọi 1: a=1 (Smi), b=2 (Smi) → ghi "SignedSmallInt + SignedSmallInt"
  - Lần gọi 2: a=5 (Smi), b=3 (Smi) → confirm "Smi + Smi"
  - Lần gọi 3: a="hello" (String) → feedback thay đổi! "polymorphic"
```

**Smi** (Small Integer) = số nguyên nhỏ vừa vặn trong một pointer. V8 tối ưu đặc biệt cho Smi vì rất phổ biến.

FeedbackVector là input quan trọng nhất cho TurboFan.

---

### Bước 5: TurboFan — Optimizing JIT Compiler

Khi V8 nhận thấy một function "hot" (được gọi đủ nhiều lần — threshold khoảng 1000-3000 lần tùy context), nó gửi function đó cho TurboFan.

TurboFan không nhận AST — nó nhận **bytecode + FeedbackVector**.

**TurboFan làm gì với thông tin đó?**

Với `add(a, b)` đã được gọi 2000 lần với Smi + Smi:

```
Speculative optimization:
"Tôi CỰC KỲ CHẮC rằng a và b luôn là Smi.
Tôi sẽ compile hàm này giả định đó là sự thật.
Tôi sẽ add một deopt guard: nếu assumption sai → bail out"
```

Machine code output (x64 pseudo-assembly):

```asm
; Optimized add(a, b) — assumes Smi inputs
; Deopt guard: check a is Smi
test rax, 1          ; Smi check (bit 0 = 0 for Smi in V8)
jnz [deopt_handler]  ; Jump to deopt nếu không phải Smi

; Actual operation — pure integer math, no boxing
mov rbx, rax         ; a
add rbx, rcx         ; b
jo  [overflow_deopt] ; Deopt nếu overflow (result không fit trong Smi)

ret
```

So sánh với interpreted bytecode: không có type checking overhead, không có boxing/unboxing, chạy trực tiếp trên CPU integer pipeline.

**Các optimization TurboFan thực hiện:**

```
1. Function Inlining     — copy body của callee vào caller, bỏ call overhead
2. Constant Folding      — 2 + 3 tính compile-time thành 5
3. Dead Code Elimination — code không bao giờ chạy → xóa khỏi machine code
4. Escape Analysis       — object không "thoát" ra ngoài function → stack allocate
5. Loop Optimization     — loop invariant code motion, strength reduction
```

**Escape Analysis — ít người biết, quan trọng:**

```javascript
function createPoint(x, y) {
  const point = { x, y }; // Tạo object trên heap?
  return point.x + point.y; // Kết quả là primitive — object không "escape"
}
```

TurboFan nhận ra `point` không escape ra khỏi function (không return object, không lưu vào biến ngoài). Nó **stack-allocates** `point` hoặc loại bỏ hoàn toàn — không có heap allocation, không có GC pressure.

---

### Deoptimization — Khi assumptions sai

```javascript
// Sau 2000 lần gọi với Smi, TurboFan đã optimize add()
// Code là machine code chạy ultra-fast

add(1, 2); // Machine code path — fast
add(3, 4); // Machine code path — fast
add('a', 1); // Deopt trigger!
```

Khi `"a"` đến, deopt guard fires:

1. V8 captures current machine state (registers, stack)
2. Reconstructs interpreter state (tương đương với bytecode frame)
3. Throws away optimized code cho function này
4. Tiếp tục từ bytecode, với FeedbackVector updated
5. Function này bị đánh dấu "polymorphic" — TurboFan sẽ compile lại với ít assumption hơn, hoặc không compile nữa

**Deopt là expensive** — không phải vì một lần deopt chậm, mà vì nó invalidates fast path cho function đó.

---

### Maglev — Mid-tier Compiler (V8 ≥ Chrome 117, 2023) 🟢

V8 nhận ra gap quá lớn giữa Ignition (chậm, flexible) và TurboFan (nhanh, đắt để compile). **Maglev** là tầng giữa:

```
Ignition    → Maglev    → TurboFan
(interpret)   (fast JIT)  (optimizing JIT)
~1x speed    ~5-10x       ~15-30x
~0ms compile ~few ms      ~10s+ ms
```

Maglev nhắm vào code "warm" — không đủ hot cho TurboFan, nhưng chạy đủ nhiều để deserving compilation. Đặc biệt hiệu quả cho React-style code với nhiều short-lived functions.

## 3. Ví dụ Code

### Bug phổ biến: Thay đổi type của argument phá vỡ optimization

```javascript
// ❌ BUG: Polymorphic function — TurboFan không thể optimize tốt

function processValue(value) {
  // Lần đầu: value là number → FeedbackVector ghi "Number"
  // Lần sau: value là string → FeedbackVector ghi "String"
  // TurboFan thấy "polymorphic" → compile version chung chung hơn
  // Hoặc không compile, chạy mãi ở Ignition
  return value * 2;
}

// Caller code gây ra vấn đề:
function renderPrices(items) {
  return items.map((item) => {
    // item.price đôi khi là number ("25"), đôi khi là string ("25")
    // Tùy data source — API trả về inconsistent types
    return processValue(item.price);
  });
}
```

```javascript
// ✅ FIX: Monomorphic function — một type duy nhất

function processPrice(price) {
  // Luôn nhận number → TurboFan optimize ultra-fast integer path
  // FeedbackVector: luôn "SignedSmallInt" hoặc "Double"
  return price * 2;
}

function renderPrices(items) {
  return items.map((item) => {
    // Normalize type tại entry point, trước khi vào hot function
    const price =
      typeof item.price === 'string'
        ? parseFloat(item.price) // ← type coercion một lần duy nhất
        : item.price;
    return processPrice(price); // luôn nhận number
  });
}
```

### Bug phổ biến 2: Object shape thay đổi sau construction phá vỡ Inline Cache

_(Liên quan đến Hidden Classes — sẽ dạy riêng, nhưng cần biết ngay)_

```javascript
// ❌ BUG: Thêm property sau construction → V8 tạo hidden class mới

function createUser(name, email) {
  const user = {}; // Hidden class HC0: {}
  user.name = name; // Hidden class HC1: { name }
  user.email = email; // Hidden class HC2: { name, email }

  if (Math.random() > 0.5) {
    user.role = 'admin'; // Một số object có HC3: { name, email, role }
    // Một số không có → shape khác nhau
  }

  return user;
  // V8 cố ghi nhớ "object ở vị trí này luôn có shape HC2"
  // nhưng đôi khi nó là HC3 → Inline Cache miss → chậm hơn
}
```

```javascript
// ✅ FIX: Declare tất cả properties tại construction, giữ consistent shape

function createUser(name, email, role = null) {
  // Tất cả users đều có cùng hidden class: { name, email, role }
  // Dù role là null hay 'admin', shape vẫn nhất quán
  return { name, email, role };
}
```

## 4. Ứng dụng thực tế

### Performance implications

**Parse time là bottleneck thực sự trên mobile:**

| Điều kiện         | Parse 1MB JS | Ghi chú               |
| ----------------- | ------------ | --------------------- |
| Desktop (M1 Mac)  | ~50ms        | Gần như không đáng kể |
| Mid-range Android | ~500-700ms   | Significant           |
| Low-end Android   | ~1500-2000ms | User thấy "lag"       |

Đây là lý do tại sao **bundle size < 170KB gzipped** là guideline phổ biến — không phải vì download time (network đã handle), mà vì **parse + compile time**.

**V8 Code Caching giảm nhẹ vấn đề này:**

Từ Chrome 66, V8 cache compiled bytecode vào disk sau lần chạy đầu. Lần sau load cùng script (same URL, same content), V8 bỏ qua parse+compile, dùng cache. Nhưng cache bị invalidate khi content thay đổi — nên mỗi deploy sẽ cold-parse lại.

### Hay gặp ở đâu

**React rendering:**

```javascript
// React re-render thường xuyên → render function phải monomorphic
// Nếu props shape thay đổi (đôi khi có field X, đôi khi không)
// → Hidden class miss, TurboFan deopt, render chậm hơn
function UserCard({ name, avatar, badge }) {
  // Luôn có cùng props shape → TurboFan happy
  // Dù badge là undefined, prop vẫn present → shape nhất quán
  return <div>{name}</div>;
}
```

**Node.js server — hot path optimization:**

```javascript
// Handler được gọi hàng nghìn lần/giây → TurboFan optimize
// Nhưng nếu request body shape inconsistent → deopt loop

// ❌ Bad: body có thể có nhiều shape khác nhau
app.post('/api/event', (req, res) => {
  processEvent(req.body); // body từ client A: {type, data}
  // body từ client B: {type, data, metadata}
  // → processEvent() thấy hai shapes → polymorphic
});

// ✅ Better: normalize shape trước khi vào hot function
app.post('/api/event', (req, res) => {
  const event = {
    // Luôn có cùng shape
    type: req.body.type,
    data: req.body.data,
    metadata: req.body.metadata ?? null,
  };
  processEvent(event); // processEvent() luôn thấy cùng shape
});
```

### Câu 1: Tại sao V8 có Pre-parser và Full Parser? Khi nào mỗi cái được dùng?

**Đáp án:**

V8 dùng hai parser vì tradeoff giữa **startup speed** và **execution completeness**.

**Full Parser** tạo ra AST đầy đủ — tốn CPU và memory. Nó chỉ chạy với code cần thực thi ngay: top-level statements, IIFEs, code trong `<script>` tags không defer.

**Pre-parser (Lazy Parser)** chạy với function bodies chưa được gọi. Nó chỉ:

- Check syntax errors
- Thu thập variable/function declarations (để scope resolution hoạt động đúng)
- Không tạo AST nodes đầy đủ

**Tại sao:** Khi browser tải một bundle 500KB, phần lớn code là function bodies sẽ chỉ chạy khi user tương tác (click handlers, lazy-loaded modules, v.v.). Nếu full-parse tất cả ngay lập tức, Chrome sẽ tốn hàng trăm milliseconds chỉ để parse code chưa cần thiết — user thấy blank screen.

**Gotcha:** Nếu một function bị pre-parsed nhưng được gọi ngay sau đó, V8 phải **parse lại lần thứ hai** (full parse). Đây là "double parse" penalty. Bundlers hiện đại (Parcel, esbuild) có thể detect và eager-parse các functions khả năng cao sẽ chạy ngay.

---

### Câu 2: FeedbackVector là gì và nó ảnh hưởng đến TurboFan như thế nào?

**Đáp án:**

**FeedbackVector** là một array các slots được allocate per-function-closure, song song với BytecodeArray. Mỗi operation trong bytecode có một hoặc nhiều feedback slots tương ứng.

Khi Ignition chạy bytecode, tại mỗi operation (phép cộng, property access, function call), nó **ghi lại type thông tin** vào feedback slot:

- Binary operation: "cả hai operands là Smi" hoặc "Number" hoặc "String"
- Property load: "luôn load từ object có hidden class HC42"
- Call: "luôn gọi cùng một function object"

Khi TurboFan compile một "hot" function, nó đọc FeedbackVector và đưa ra **speculative assumptions** — ví dụ: "a và b luôn là Smi, nên tôi compile integer-only path." Machine code kết quả không có type checks, không có boxing/unboxing — nhanh gần bằng C code.

Tuy nhiên, TurboFan cũng nhúng **deoptimization guards** vào machine code. Nếu assumption bị vi phạm (e.g., `a` đột ngột là `String`), guard fires, V8 bail out về Ignition, update FeedbackVector với type mới, và sau đó có thể re-compile với broader assumptions.

**Tóm lại:** FeedbackVector là "bộ nhớ lịch sử" — Ignition ghi, TurboFan đọc để optimize. Không có FeedbackVector, JIT compilation chỉ có thể tạo ra code generic, chậm hơn nhiều.

### Câu 3: Đoạn code sau có vấn đề gì từ góc độ V8 optimization? Nó sẽ gây ra hệ quả gì trong production?

```javascript
function formatCurrency(amount, currency) {
  if (typeof amount === 'string') {
    amount = parseFloat(amount);
  }
  return new Intl.NumberFormat('en-US', {
    style: 'currency',
    currency: currency,
  }).format(amount);
}

// Được gọi trong một loop render 60fps
items.forEach((item) => {
  labels.push(formatCurrency(item.rawPrice, item.currency));
});
```

**Đáp án:**

**Vấn đề 1 — Polymorphic parameter:**  
`amount` đôi khi là `string`, đôi khi là `number`. FeedbackVector slot cho `typeof amount` sẽ thấy hai types → function trở thành **polymorphic**. TurboFan không thể compile path tối ưu cho một type duy nhất. Trong loop 60fps, function này sẽ chạy ở tầng Ignition hoặc một compiled version rất generic.

**Vấn đề 2 — `new Intl.NumberFormat()` allocation trong hot loop:**  
`new Intl.NumberFormat(...)` allocates một object mới trên heap mỗi lần gọi. Object này expensive — Intl formatter load locale data, build internal structures. Nếu items là 1000 phần tử, 1000 formatter objects được tạo rồi ngay lập tức trở thành garbage. GC pressure tăng cao.

**Vấn đề 3 — Object không thể escape-analyzed:**  
Object config `{ style: 'currency', currency: currency }` được tạo mỗi lần gọi và truyền vào `Intl.NumberFormat`. V8 không thể biết `Intl.NumberFormat` làm gì với nó → không escape-analyze được → heap allocation mỗi lần.

**Fix:**

```javascript
// Cache formatter per currency (sống ngoài loop)
const formatterCache = new Map();

function getFormatter(currency) {
  if (!formatterCache.has(currency)) {
    formatterCache.set(
      currency,
      new Intl.NumberFormat('en-US', {
        style: 'currency',
        currency: currency,
      }),
    );
  }
  return formatterCache.get(currency);
}

function formatCurrency(amount, currency) {
  // Luôn nhận number → monomorphic → TurboFan optimize tốt
  return getFormatter(currency).format(amount);
}

// Normalize type tại entry point
items.forEach((item) => {
  const price =
    typeof item.rawPrice === 'string'
      ? parseFloat(item.rawPrice)
      : item.rawPrice;
  labels.push(formatCurrency(price, item.currency));
});
```

Kết quả: `formatCurrency` monomorphic, formatter được reuse, không có GC pressure từ object allocation trong hot loop.
