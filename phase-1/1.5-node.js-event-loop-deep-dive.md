# Node.js Event Loop: Deep Dive

> **Độ ưu tiên:** 🟢 — nhưng nếu bạn làm Node.js production, đây là kiến thức phân biệt Senior và Staff. Hiểu sai Node.js event loop = debug sai performance issues, viết sai async code order.

---

## 1. Cơ chế thật

### Node.js ≠ Browser: Hai runtime khác nhau hoàn toàn

Browser event loop được định nghĩa trong **HTML spec** — implement bởi browser engine (Blink, Gecko...).

Node.js event loop được implement bởi **libuv** — một C library cross-platform I/O. V8 chỉ là JS engine; libuv là thứ handle async I/O, threading, timers, network. Đây là lý do Node.js event loop có structure khác hẳn browser.

```
Browser:                          Node.js:
────────                          ────────
HTML spec → Blink/Gecko           libuv → C layer
V8 runs JS                        V8 runs JS
Render step tồn tại               Không có render
Web APIs (DOM, fetch...)          Node APIs (fs, net, crypto...)
Single macrotask queue            Multiple phases với queues riêng
```

---

### libuv Architecture — trước khi học event loop phases

libuv dùng **thread pool** (default 4 threads, tăng được qua `UV_THREADPOOL_SIZE`) cho các operations không có async OS API: file system, DNS lookup, crypto, compression.

```
┌─────────────────────────────────────────────────────────────┐
│  NODE.JS PROCESS                                            │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  V8 (JS Engine)          libuv Event Loop           │   │
│  │  ┌──────────┐            ┌─────────────────────┐    │   │
│  │  │ Call     │◄──────────►│  Event Loop Phases  │    │   │
│  │  │ Stack    │  callbacks │  (timers/poll/check) │    │   │
│  │  └──────────┘            └─────────────────────┘    │   │
│  └───────────────────────────────────┬─────────────────┘   │
│                                      │                      │
│  ┌───────────────────────────────────▼─────────────────┐   │
│  │  libuv Thread Pool (4 threads default)              │   │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌───────┐  │   │
│  │  │  fs ops  │ │  crypto  │ │   DNS    │ │  ...  │  │   │
│  │  └──────────┘ └──────────┘ └──────────┘ └───────┘  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  OS Async APIs (epoll/kqueue/IOCP)                  │   │
│  │  Network I/O, pipes, TTY — KHÔNG dùng thread pool   │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**Network I/O** (TCP, UDP, HTTP) **không** dùng thread pool — dùng OS-level async APIs (`epoll` trên Linux, `kqueue` trên macOS, `IOCP` trên Windows). Zero threads cho network. Đây là lý do Node.js handle hàng nghìn concurrent connections với ít threads.

**File system I/O** dùng thread pool vì hầu hết OS không có truly async file APIs (hoặc có nhưng complex/buggy). `fs.readFile()` thực ra chạy trên một thread pool thread, không block event loop thread.

### 6 Phases của Node.js Event Loop

Mỗi "tick" của libuv event loop đi qua 6 phases theo thứ tự cố định:

```
   ┌─────────────────────────────────┐
   │           timers                │  ← phase 1
   │  setTimeout, setInterval        │
   └─────────────┬───────────────────┘
                 │
   ┌─────────────▼───────────────────┐
   │       pending callbacks         │  ← phase 2
   │  I/O errors từ previous tick    │
   └─────────────┬───────────────────┘
                 │
   ┌─────────────▼───────────────────┐
   │       idle, prepare             │  ← phase 3
   │  Internal libuv use only        │
   └─────────────┬───────────────────┘
                 │
   ┌─────────────▼───────────────────┐
   │            poll                 │  ← phase 4 (QUAN TRỌNG NHẤT)
   │  Retrieve new I/O events        │
   │  Execute I/O callbacks          │
   │  Node blocks HERE if no tasks   │
   └─────────────┬───────────────────┘
                 │
   ┌─────────────▼───────────────────┐
   │           check                 │  ← phase 5
   │  setImmediate callbacks         │
   └─────────────┬───────────────────┘
                 │
   ┌─────────────▼───────────────────┐
   │       close callbacks           │  ← phase 6
   │  socket.on('close', ...)        │
   └─────────────┬───────────────────┘
                 │
   ┌─────────────▼───────────────────┐
   │  [nextTick + Promise microtasks]│  ← giữa MỖI phase transition
   │  Drain trước khi sang phase mới │
   └─────────────────────────────────┘
```

**Critical rule:** Giữa mỗi phase transition, Node.js drain hai queues theo thứ tự:

1. **`process.nextTick()` queue** — toàn bộ
2. **Promise microtask queue** — toàn bộ

Sau đó mới sang phase tiếp theo.

---

### Phase 1: Timers

Xử lý callbacks của `setTimeout` và `setInterval` có **threshold đã qua**.

```javascript
setTimeout(() => console.log('timer 1'), 100);
setTimeout(() => console.log('timer 2'), 100);
```

libuv kiểm tra: timestamp hiện tại ≥ timer threshold? Nếu có → callback được run trong phase này.

**Quan trọng: "minimum delay", không phải "exact delay".** `setTimeout(fn, 100)` nghĩa là "không trước 100ms" — không phải "đúng 100ms". Nếu event loop đang bận ở phase khác khi timer fires, callback phải chờ đến lần lượt tiếp theo của timers phase.

**System clock resolution:** OS timer resolution thường ~1ms. Các values nhỏ như `setTimeout(fn, 1)` hoặc `setTimeout(fn, 0)` trong Node.js thực ra được clamp lên `1ms` minimum.

---

### Phase 4: Poll — trái tim của event loop

Poll phase là nơi Node.js dành phần lớn thời gian. Hai việc:

1. **Calculate timeout:** bao lâu nữa timer gần nhất fires?
2. **Block và chờ I/O events** trong khoảng thời gian đó

```
Poll phase logic:
─────────────────────────────────────────────
if (poll queue has callbacks):
  Execute them one by one until:
    - Queue empty, OR
    - System-dependent hard limit reached

else if (setImmediate was called):
  End poll phase → go to check phase

else:
  Block here, waiting for:
    - New I/O event (network packet, file read complete...)
    - OR timer threshold reached
  When event arrives → execute callback → continue
```

**Đây là lý do Node.js không tốn CPU khi idle:** Process thật sự block trong `epoll_wait()` (Linux) — OS wakes process khi có I/O event hoặc timer. Không có busy-waiting.

```javascript
// Scenario: Node.js server đang chờ request
// 1. Poll phase: epoll_wait() — process sleeping
// 2. TCP packet arrives → OS wakes Node → epoll_wait() returns
// 3. libuv: new connection event → enqueue callback
// 4. Poll phase executes connection callback
// 5. Request handler runs
// 6. await db.query() → async I/O bắt đầu → callback registered
// 7. Return to poll phase → wait for db response
```

---

### Phase 5: Check — `setImmediate`

`setImmediate` callbacks chạy ở đây — **sau poll phase, trước timers phase của iteration tiếp theo**.

```javascript
const fs = require('fs');

fs.readFile('file.txt', () => {
  // Đang trong poll phase (I/O callback)

  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));

  // Output: immediate, timeout (GUARANTEED)
});
```

**Tại sao `setImmediate` guaranteed trước `setTimeout` trong I/O callback:**

Khi đang trong poll phase và gọi `setImmediate`, libuv biết check phase cần chạy. Sau khi poll callback xong, event loop đi thẳng đến check phase → `setImmediate` fires. `setTimeout(0)` phải chờ đến vòng lặp tiếp theo, timers phase.

**Ngoài I/O callback — KHÔNG guaranteed:**

```javascript
// Ở top level (ngoài I/O callback)
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));

// Output: không xác định — có thể timeout trước, có thể immediate trước
// Phụ thuộc vào: bao lâu để Node.js bootstrap xong và enter event loop,
// so với timer 1ms minimum threshold
```

### `process.nextTick()` — không phải "next tick" như tên gọi

Đây là API gây confusion nhiều nhất trong Node.js. `process.nextTick()` **không** chạy ở "next tick" (next iteration) của event loop — nó chạy **ngay sau current operation**, trước khi event loop tiếp tục sang phase tiếp theo.

`process.nextTick()` có **higher priority hơn Promise microtasks** trong Node.js:

```
Giữa mỗi phase:
1. Drain process.nextTick queue (toàn bộ)
2. Drain Promise microtask queue (toàn bộ)
→ Move to next phase
```

```javascript
Promise.resolve().then(() => console.log('promise'));
process.nextTick(() => console.log('nextTick'));
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));

// Output:
// nextTick      ← process.nextTick: highest priority
// promise       ← Promise microtask
// timeout       ← macrotask (timers phase)
// immediate     ← macrotask (check phase)
```

---

### `process.nextTick` vs `Promise` vs `setImmediate` vs `setTimeout` — bảng so sánh đầy đủ

```
API                  Queue              Priority    Khi nào chạy
──────────────────────────────────────────────────────────────────
process.nextTick()   nextTick queue     HIGHEST     Sau current op,
                                                    trước Promise
Promise.then()       microtask queue    HIGH        Sau nextTick queue
setImmediate()       check phase queue  MEDIUM      Sau poll phase
setTimeout(fn, 0)    timers phase       MEDIUM      Sau poll phase,
                                                    next timers phase
                                                    (thứ tự vs immediate
                                                     không guaranteed
                                                     ngoài I/O)
```

### Starvation — danger của `process.nextTick` và Promise

Giống browser, nếu nextTick queue hoặc microtask queue không bao giờ empty, event loop không thể tiến sang phase tiếp theo:

```javascript
// ❌ nextTick starvation — I/O callbacks không bao giờ chạy

function recursiveNextTick() {
  process.nextTick(recursiveNextTick);
}
recursiveNextTick();

// Mọi I/O: file reads, network, db queries → không bao giờ được xử lý
// Server trông như "hung" — nhận connections nhưng không respond

// Tương tự với Promise:
function recursivePromise() {
  Promise.resolve().then(recursivePromise);
}
```

**Real-world starvation pattern (tinh tế hơn):**

```javascript
// ❌ nextTick trong hot path — tích lũy thành starvation

class EventEmitter {
  emit(event, data) {
    const listeners = this.listeners[event] || [];
    listeners.forEach((listener) => {
      // Dev nghĩ: "async emit để không block"
      // Thực tế: tích lũy hàng nghìn nextTick callbacks
      process.nextTick(() => listener(data));
    });
  }
}

// Nếu emit() được gọi 10,000 lần/second với 5 listeners
// → 50,000 nextTick callbacks/second
// → Có thể outpace I/O callbacks trong certain patterns
```

### UV_THREADPOOL_SIZE — và khi nào nó quan trọng

```javascript
// File I/O dùng thread pool → 4 threads mặc định
// 4 concurrent fs operations là max cùng lúc

// ❌ Bottleneck không rõ ràng:
const files = Array.from({ length: 20 }, (_, i) => `file${i}.txt`);

// 20 concurrent reads → chỉ 4 chạy thật sự cùng lúc
// 16 phải queue trong thread pool → total time cao hơn expected
const contents = await Promise.all(files.map((f) => fs.promises.readFile(f)));
// Không phải network bottleneck, không phải JS bottleneck
// → Thread pool bottleneck

// ✅ Fix: Tăng thread pool size (trước khi require bất kỳ thứ gì)
// UV_THREADPOOL_SIZE=16 node server.js
// Hoặc:
process.env.UV_THREADPOOL_SIZE = 16; // Phải set TRƯỚC khi libuv init

// Note: Crypto operations (bcrypt, pbkdf2) cũng dùng thread pool
// bcrypt hash + nhiều file reads đồng thời = contention trên thread pool
// Tăng UV_THREADPOOL_SIZE giải quyết điều này
```

### Node.js Phases — trace ví dụ phức tạp

```javascript
const fs = require('fs');

console.log('1: start');

setTimeout(() => console.log('2: timeout 1'), 0);

fs.readFile(__filename, () => {
  // Đang trong POLL PHASE
  console.log('4: readFile callback');

  setTimeout(() => console.log('5: timeout in I/O'), 0);
  setImmediate(() => console.log('6: immediate in I/O'));

  process.nextTick(() => console.log('7: nextTick in I/O'));
  Promise.resolve().then(() => console.log('8: promise in I/O'));
});

process.nextTick(() => console.log('9: nextTick top-level'));
Promise.resolve().then(() => console.log('10: promise top-level'));

console.log('11: end');
```

**Output (guaranteed):**

```
1: start
11: end
9: nextTick top-level      ← nextTick queue drain (trước Promise)
10: promise top-level      ← Promise microtask drain
2: timeout 1               ← timers phase (không guaranteed vs immediate ở top level)
3: immediate 1             ← check phase (thứ tự 2 và 3 không guaranteed)
4: readFile callback       ← poll phase (I/O callback)
7: nextTick in I/O         ← nextTick drain SAU I/O callback
8: promise in I/O          ← Promise drain
6: immediate in I/O        ← check phase (GUARANTEED trước timeout khi trong I/O)
5: timeout in I/O          ← timers phase (next iteration)
```

**Trace từng bước:**

```

  "1: start", "11: end" in

[Between phases — drain queues]:
  nextTick: "9: nextTick top-level"
  Promise:  "10: promise top-level"

[Timers phase]:
  setTimeout(0) threshold passed? Depends on startup time
  → "2: timeout 1" (hoặc sau immediate nếu startup chậm)

[Poll phase]:
  fs.readFile complete → callback enqueued
  Execute: "4: readFile callback"

  [Between operations — drain queues immediately]:
  nextTick: "7: nextTick in I/O"
  Promise:  "8: promise in I/O"

[Check phase] (vì setImmediate đã được gọi trong I/O):
  "6: immediate in I/O"

[Next iteration — Timers phase]:
  "5: timeout in I/O"
```

### Bug: Sai thứ tự `nextTick` gây subtle ordering issues

```javascript
// ❌ BUG: API trả về kết quả synchronously đôi khi, async đôi khi
// "Zalgo" pattern — unpredictable behavior

class Cache {
  constructor() {
    this.store = new Map();
    this.loader = null;
  }

  get(key, callback) {
    if (this.store.has(key)) {
      // Cache hit: callback gọi SYNCHRONOUSLY
      callback(null, this.store.get(key));
      return;
    }
    // Cache miss: callback gọi ASYNCHRONOUSLY (sau I/O)
    this.loader(key, (err, value) => {
      if (!err) this.store.set(key, value);
      callback(err, value);
    });
  }
}

const cache = new Cache();
let result;

// Caller code không biết sync hay async
cache.get('userProfile', (err, data) => {
  result = data;
  // Nếu cache hit: result set TRƯỚC dòng dưới
  // Nếu cache miss: result set SAU dòng dưới
});

console.log(result); // undefined nếu async, data nếu sync
// Code sau đây không thể predict behavior
```

```javascript
// ✅ FIX: Luôn async — dùng process.nextTick để defer sync callback

class Cache {
  get(key, callback) {
    if (this.store.has(key)) {
      // Defer sang nextTick → LUÔN async về mặt caller
      // Nhưng nextTick = ngay sau current operation
      // → Rất nhanh, không tốn I/O round-trip
      process.nextTick(() => callback(null, this.store.get(key)));
      return;
    }
    this.loader(key, (err, value) => {
      if (!err) this.store.set(key, value);
      callback(err, value);
    });
  }
}

// Bây giờ caller có thể assume LUÔN async:
cache.get('userProfile', (err, data) => {
  result = data; // LUÔN chạy sau synchronous code bên dưới
});
console.log(result); // LUÔN undefined ở đây — behavior nhất quán
```

### Production pattern: CPU-heavy work không block I/O

```javascript
// ❌ BUG: Synchronous CPU work block event loop
// Server không accept new connections trong khi compute

app.get('/api/report', async (req, res) => {
  const rows = await db.query('SELECT * FROM events LIMIT 100000');

  // processRows là pure CPU work — 500ms
  // Trong 500ms: tất cả pending I/O (DB queries, network)
  // không được xử lý
  const report = processRows(rows); // BLOCKS EVENT LOOP

  res.json(report);
});
```

```javascript
// ✅ FIX: Chia nhỏ work với setImmediate
// setImmediate yields control về event loop sau mỗi chunk
// Poll phase có thể handle I/O giữa các chunks

async function processRowsAsync(rows, chunkSize = 500) {
  const results = [];

  for (let i = 0; i < rows.length; i += chunkSize) {
    const chunk = rows.slice(i, i + chunkSize);

    // Process chunk synchronously (fast)
    results.push(...chunk.map(transformRow));

    // Yield control về event loop
    // setImmediate: sau poll phase → I/O được xử lý
    // setTimeout(0): phải chờ đến timers phase — thường chậm hơn
    await new Promise((resolve) => setImmediate(resolve));
    // Trong khoảng này: pending DB callbacks, network events được handle
  }

  return results;
}

app.get('/api/report', async (req, res) => {

  // 200 chunks x 500 rows, yield sau mỗi chunk
  // Max blocking per chunk: ~2ms thay vì 500ms
  const report = await processRowsAsync(rows);
```

### Debug: Detect event loop lag trong production

```javascript
// Measure actual event loop delay
// Principle: schedule callback với delay 1000ms
// Measure actual delay — gap = event loop lag

class EventLoopMonitor {
  constructor(options = {}) {
    this.interval = options.interval || 1000;
    this.threshold = options.threshold || 100; // warn nếu lag > 100ms
    this.running = false;
  }

  start() {
    this.running = true;
    this._schedule();
  }

  _schedule() {
    if (!this.running) return;

    const expected = Date.now() + this.interval;

    setTimeout(() => {
      const actual = Date.now();
      const lag = actual - expected; // Actual delay - expected delay

      if (lag > this.threshold) {
        // Event loop bị block khoảng 'lag' ms
        // Tìm long task trong Performance profiler
        console.warn(`[EventLoop] Lag detected: ${lag}ms`);

        // Production: send to APM (Datadog, New Relic, etc.)
        // metrics.gauge('nodejs.event_loop.lag', lag);
      }

      this._schedule(); // Schedule next measurement
    }, this.interval);
  }

  stop() {
    this.running = false;
  }
}

const monitor = new EventLoopMonitor({ threshold: 50 });
monitor.start();

// Thư viện chuyên dụng: 'perf_hooks' module
const { monitorEventLoopDelay } = require('perf_hooks');

const histogram = monitorEventLoopDelay({ resolution: 10 }); // 10ms resolution
histogram.enable();

setInterval(() => {
  console.log({
    min: histogram.min / 1e6, // Convert nanoseconds → ms
    max: histogram.max / 1e6,
    mean: histogram.mean / 1e6,
    p99: histogram.percentile(99) / 1e6,
  });
  histogram.reset();
}, 5000);
```

### `async_hooks` — trace async context propagation

```javascript
// async_hooks: track lifecycle của async operations
// Dùng để implement: distributed tracing, AsyncLocalStorage, resource tracking

const async_hooks = require('async_hooks');
const fs = require('fs');

// Mỗi async operation được cấp một asyncId duy nhất
// triggerAsyncId: asyncId của operation TẠO RA operation này

const activeOps = new Map();

const hook = async_hooks.createHook({
  init(asyncId, type, triggerAsyncId) {
    // Async operation mới được tạo
    activeOps.set(asyncId, { type, triggerAsyncId, start: Date.now() });
    fs.writeSync(
      1,
      `INIT ${type} asyncId=${asyncId} triggeredBy=${triggerAsyncId}\n`,
    );
  },
  before(asyncId) {
    // Callback của async operation SẮP chạy
    fs.writeSync(1, `BEFORE asyncId=${asyncId}\n`);
  },
  after(asyncId) {
    // Callback của async operation VỪA XONG
    fs.writeSync(1, `AFTER asyncId=${asyncId}\n`);
  },
  destroy(asyncId) {
    // Async operation đã cleanup
    activeOps.delete(asyncId);
  },
});

hook.enable();

// Usage quan trọng hơn: AsyncLocalStorage
// Truyền context (request ID, user info) qua async boundaries
```

### Hiểu event loop → diagnose production incidents

**Incident pattern 1: "Server nhận request nhưng không respond"**

```
Symptoms:
- Health check timeout sau 30s
- CPU spike 100% trên một core
- Memory bình thường
- Không có error logs

Root cause: CPU-bound synchronous work block event loop
Diagnosis:
  node --prof server.js      # Record CPU profile
  # Tạo traffic
  node --prof-process isolate-*.log > profile.txt
  # Tìm function chiếm > 50% CPU time
  # Đó là thủ phạm block event loop
```

**Incident pattern 2: "Latency spike đột ngột mỗi vài phút"**

```
Symptoms:
- p99 latency 2000ms, p50 bình thường
- Periodic — xảy ra đều đặn
- Memory tăng dần rồi giảm đột ngột trước spike

Root cause: Major GC pause (Old Space full → Mark-Sweep-Compact)
Diagnosis:
  node --trace-gc server.js 2>&1 | grep "Mark-sweep"
  # Output: "[GC] Mark-sweep 500ms"
  # Tăng --max-old-space-size hoặc fix memory leak
```

**Incident pattern 3: "setTimeout(fn, 1000) fire sau 2-3 giây"**

```
Symptoms:
- Timer-based jobs delay hơn expected
- Không có CPU spike
- I/O load cao

Root cause: Poll phase block quá lâu xử lý I/O backlog
Diagnosis:
  Event loop lag monitor → lag = 1000-2000ms
  → Poll phase đang process quá nhiều I/O callbacks
  → Tăng số worker threads hoặc reduce I/O concurrency
```

### Câu 1: Tại sao `setImmediate` guaranteed chạy trước `setTimeout(fn, 0)` khi được gọi trong I/O callback, nhưng KHÔNG guaranteed ở top-level?

**Đáp án:**

**Khi trong I/O callback (guaranteed):**

I/O callbacks chạy trong **Poll phase**. Khi `setImmediate` được gọi trong poll phase, libuv đánh dấu "check phase cần chạy". Sau khi poll callbacks xong, event loop đi thẳng theo thứ tự: Poll → Check. `setImmediate` callback fires trong Check phase này.

`setTimeout(fn, 0)` callback phải chờ **Timers phase** — nhưng Timers phase đã qua trong iteration hiện tại. Nó phải đợi đến **iteration tiếp theo**, vòng lặp quay về Timers phase.

Thứ tự phases: `Poll → (drain queues) → Check → Close → [next iteration] → Timers`

Vì vậy: Check phase (setImmediate) LUÔN đến trước Timers phase (setTimeout) của iteration tiếp theo — guaranteed.

**Ở top-level (không guaranteed):**

Ở top-level, Node.js đang trong giai đoạn **bootstrap/initialization** — chưa vào event loop chính thức. Khi event loop bắt đầu iteration đầu tiên tại Timers phase:

- Nếu bootstrap mất > 1ms → `setTimeout(fn, 0)` threshold đã qua → fires trong Timers phase đầu tiên → trước setImmediate (Check phase)
- Nếu bootstrap nhanh < 1ms → setTimeout threshold chưa qua → Timers phase bỏ qua → Poll → Check → setImmediate fires trước → setTimeout fires iteration sau

Vì bootstrap time phụ thuộc vào system load, CPU speed, Node.js version → **không deterministic**.

### Câu 2: `process.nextTick()` và `Promise.resolve().then()` đều là microtasks. Chúng khác nhau như thế nào trong Node.js? Khi nào nên dùng cái nào?

**Đáp án:**

**Trong Node.js, chúng là hai queues riêng biệt với priority khác nhau:**

`process.nextTick()` callbacks được enqueue vào **nextTick queue** — được drain trước Promise microtask queue trong mỗi phase transition.

`Promise.resolve().then()` callbacks vào **Promise microtask queue** — drain sau nextTick queue.

```
Phase N done →
  drain nextTick queue (all) →
  drain Promise queue (all) →
Phase N+1 starts
```

**`process.nextTick` recursion nguy hiểm hơn:**

Nếu nextTick callback thêm nextTick mới → drain không bao giờ kết thúc → Promise callbacks bị starved → I/O bị delay. Node.js có `--next-tick-warn-with-count` để detect điều này.

**Khi dùng `process.nextTick`:**

- Cần callback chạy **trước bất kỳ I/O nào** trong tick hiện tại
- Fix "Zalgo" (đảm bảo API luôn async) — ví dụ cache hit nên defer thay vì gọi callback sync
- Error propagation sớm trước khi event loop tiếp tục
- Node.js internals, EventEmitter implementation

**Khi dùng `Promise.resolve().then()`:**

- Hầu hết use cases của async-but-soon
- Khi không cần guarantee "trước Promise callbacks khác"
- Khi code cần cross-platform (browser không có nextTick)
- Modern async code — cleaner, chainable, có error handling

**Rule of thumb:** Mặc định dùng `Promise`/`async-await`. Dùng `process.nextTick` khi có lý do cụ thể cần pre-empt Promise callbacks.

### Câu 3: Đoạn code Node.js server sau có vấn đề gì? Vấn đề sẽ biểu hiện ra sao trong production và fix như thế nào?

```javascript
const express = require('express');
const crypto = require('crypto');
const app = express();

app.post('/api/register', async (req, res) => {
  const { password, userData } = req.body;

  // Hash password với bcrypt-style (CPU intensive)
  const salt = crypto.randomBytes(16).toString('hex');
  let hash = password + salt;

  // 10,000 rounds của SHA-256
  for (let i = 0; i < 10000; i++) {
    hash = crypto.createHash('sha256').update(hash).digest('hex');
  }

  await db.save({ ...userData, passwordHash: hash });
  res.json({ success: true });
});

app.get('/api/health', (req, res) => {
  res.json({ status: 'ok' });
});

app.listen(3000);
```

**Đáp án:**

**Vấn đề: CPU-bound loop block event loop hoàn toàn.**

`crypto.createHash().update().digest()` là synchronous. Vòng lặp 10,000 iterations với SHA-256 mỗi lần tốn khoảng **50-200ms** tùy hardware.

**Biểu hiện trong production:**

Trong 50-200ms đó, **không request nào khác được xử lý** — kể cả `/api/health`. Load balancer health check timeout → instance được đánh dấu unhealthy → removed từ rotation → cascading failure nếu nhiều instances bị hit đồng thời.

Nếu 10 concurrent registration requests: 10 × 100ms = **1 second** event loop block. Tất cả health checks, DB callbacks, và mọi request khác bị delay ít nhất 1 giây.

**Fix 1: Dùng `crypto.pbkdf2` async (built-in, dùng thread pool):**

```javascript
const { promisify } = require('util');
const pbkdf2 = promisify(crypto.pbkdf2);

app.post('/api/register', async (req, res) => {
  const { password, userData } = req.body;

  // pbkdf2 chạy trên libuv thread pool — không block event loop
  const salt = crypto.randomBytes(16);
  const hash = await pbkdf2(password, salt, 10000, 64, 'sha256');

  await db.save({ ...userData, passwordHash: hash.toString('hex') });
  res.json({ success: true });
});
```

**Fix 2: Dùng `bcrypt` package (cũng dùng thread pool):**

```javascript
const bcrypt = require('bcrypt');

app.post('/api/register', async (req, res) => {
  const hash = await bcrypt.hash(req.body.password, 10);
  // bcrypt.hash là async — offload sang C++ layer với thread pool
  await db.save({ ...req.body.userData, passwordHash: hash });
  res.json({ success: true });
});
```

**Fix 3: Nếu bắt buộc phải dùng custom loop — offload sang Worker Thread:**

```javascript
const { Worker } = require('worker_threads');

function hashInWorker(password) {
  return new Promise((resolve, reject) => {
    const worker = new Worker(
      `
      const { workerData, parentPort } = require('worker_threads');
      const crypto = require('crypto');
      const { password, salt } = workerData;
      let hash = password + salt;
      for (let i = 0; i < 10000; i++) {
        hash = crypto.createHash('sha256').update(hash).digest('hex');
      }
      parentPort.postMessage(hash);
    `,
      {
        eval: true,
        workerData: { password, salt: crypto.randomBytes(16).toString('hex') },
      },
    );

    worker.on('message', resolve);
    worker.on('error', reject);
  });
}
```

**Kết luận:** Bất kỳ synchronous CPU work > ~5ms trong request handler đều là potential incident trong Node.js production. Sử dụng async built-ins, worker threads, hoặc child_process để offload.
