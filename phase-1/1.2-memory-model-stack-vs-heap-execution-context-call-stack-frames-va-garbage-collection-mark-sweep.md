## 1. Cơ chế thật

### Bức tranh tổng thể: V8 process memory

Khi Node.js hoặc Chrome chạy JS, V8 được cấp một vùng memory từ OS. V8 tổ chức vùng này thành nhiều **regions** với mục đích khác nhau:

```
V8 Process Memory (simplified)
┌────────────────────────────────────────────────┐
│  Code Space      — compiled machine code       │
│  Large Object Space — objects > 512KB          │
│                                                │
│  ┌──────────────────────────────────────────┐  │
│  │  HEAP (managed by GC)                   │  │
│  │  ┌─────────────────┐  ┌───────────────┐ │  │
│  │  │  New Space      │  │  Old Space    │ │  │
│  │  │  (Young Gen)    │  │  (Old Gen)    │ │  │
│  │  │  ~1-8 MB        │  │  ~unlimited   │ │  │
│  │  └─────────────────┘  └───────────────┘ │  │
│  └──────────────────────────────────────────┘  │
│                                                │
│  STACK (per thread, fixed size ~1MB default)   │
│  ┌──────────────────────────────────────────┐  │
│  │  Frame N   ← current function           │  │
│  │  Frame N-1                              │  │
│  │  Frame N-2                              │  │
│  │  ...                                    │  │
│  │  Frame 0   ← global/entry               │  │
│  └──────────────────────────────────────────┘  │
└────────────────────────────────────────────────┘
```

### Stack — cơ chế thật

Stack là vùng memory **LIFO** (Last In, First Out), grow/shrink tự động theo function calls. Mỗi khi một function được gọi, V8 **push** một **stack frame** (còn gọi là execution context frame). Khi function return, frame đó **pop** ra và memory được giải phóng **ngay lập tức** — không cần GC.

**Stack frame chứa gì:**

```
Stack Frame của một function call:
┌─────────────────────────────────────┐  ← frame pointer (FP)
│  Return address                     │  địa chỉ để về sau khi return
│  Previous frame pointer             │  pointer về frame cha
│  Local variables (primitives)       │  let x = 5 → 5 nằm ở đây
│  Function arguments (primitives)    │  giá trị, không phải reference
│  Temporary values                   │  kết quả trung gian của expressions
└─────────────────────────────────────┘  ← stack pointer (SP)
```

**Quan trọng:** Stack chỉ lưu **primitive values** và **references (pointers)**. Object bodies không bao giờ nằm trên stack trong V8 JS — chúng luôn trên heap.

```javascript
function calculateDiscount(price, rate) {
  // price = 100 (number primitive) → LƯU TRỰC TIẾP trên stack frame
  // rate = 0.1 (number primitive) → LƯU TRỰC TIẾP trên stack frame
  const discount = price * rate;
  // discount = 10 → LƯU TRỰC TIẾP trên stack frame
  return discount;
  // Frame bị pop → 100, 0.1, 10 đều biến mất ngay lập tức
}

function applyPromotion(order) {
  // order là OBJECT → Stack chỉ lưu POINTER đến object trên heap
  // Object body { items: [...], total: 200 } nằm trên HEAP
  const saved = calculateDiscount(order.total, 0.1);
  return saved;
}
```

### Execution Context — thứ V8 thực sự track

Khi JS engine chạy code, nó không chỉ track "đang ở dòng nào" — nó tạo ra **Execution Context** cho mỗi scope. Execution Context là một internal structure gồm 3 thành phần:

```
Execution Context
┌─────────────────────────────────────────────────┐
│ 1. Variable Environment (VE)                    │
│    — var declarations, function declarations    │
│    — initialized to undefined at creation       │
│                                                 │
│ 2. Lexical Environment (LE)                     │
│    — let, const declarations (TDZ until init)   │
│    — inner functions, block scope               │
│                                                 │
│ 3. ThisBinding                                  │
│    — giá trị của `this` trong context này       │
└─────────────────────────────────────────────────┘
```

**Mỗi Execution Context cũng có một outer reference** — pointer tới EC của scope bên ngoài. Đây là cơ chế tạo ra **scope chain**.

**Call Stack là stack của Execution Contexts:**

```javascript
// Khi code này chạy:

const TAX_RATE = 0.1; // Global EC: { TAX_RATE: 0.1 }

function computeTotal(price) {
  // Function EC: { price: 200, outer: GlobalEC }
  const tax = price * TAX_RATE;
  // 'TAX_RATE' không tìm thấy trong Function EC
  // → đi lên outer (Global EC) → tìm thấy 0.1
  return price + tax;
}

function processOrder(order) {
  // Function EC: { order: <HeapRef>, outer: GlobalEC }
  const total = computeTotal(order.price);
  return { ...order, total };
}

processOrder({ price: 200 });
```

Call Stack tại moment `TAX_RATE` được access:

```
Call Stack:
┌──────────────────────────────────────────────┐
│  computeTotal EC                             │  ← TOP (active)
│  { price: 200, tax: <uninitialized> }        │
│  outer → GlobalEC                            │
├──────────────────────────────────────────────┤
│  processOrder EC                             │
│  { order: <ptr:0x7f3a>, total: <uninit> }   │
│  outer → GlobalEC                            │
├──────────────────────────────────────────────┤
│  Global EC                                   │  ← BOTTOM
│  { TAX_RATE: 0.1, computeTotal: <fn>,       │
│    processOrder: <fn> }                      │
│  outer → null                                │
└──────────────────────────────────────────────┘
```

Khi `computeTotal` return: frame của nó bị pop, `price` và `tax` (primitives) biến mất. `processOrder` EC tiếp tục với `total = 220`.

### Heap — Object Allocation

Heap là nơi tất cả **objects, arrays, functions, closures** được allocate. Không có size limit cứng (bị giới hạn bởi available RAM), và không có cơ chế tự giải phóng — đó là việc của GC.

**Khi nào V8 allocate lên heap:**

```javascript
// STACK: primitives (value types)
let count = 0; // số nguyên → stack
let ratio = 0.75; // float → stack
let active = true; // boolean → stack

// HEAP: mọi thứ còn lại (reference types)
const user = {}; // object → heap
const items = [1, 2, 3]; // array → heap (array là object)
const fn = () => {}; // function → heap
const text = 'hello'; // string → heap (strings là objects trong V8)
// nhưng SHORT STRINGS có thể được interned

// Stack chỉ chứa POINTER:
// user → 0x7f3a (địa chỉ trên heap của { })
// items → 0x8b2c (địa chỉ trên heap của array)
```

**String interning — chi tiết ít biết:**

V8 intern (deduplicate) một số strings. String literals trong source code (`"hello"`) thường được intern — nhiều variables chứa `"hello"` có thể point đến cùng một heap object. Điều này tiết kiệm memory nhưng không ảnh hưởng semantics vì strings là immutable.

### Closure và Heap — tại sao closures giữ memory

Đây là nơi nhiều memory leaks ẩn náu. Khi một function **đóng** (close over) một biến từ outer scope, V8 phải đảm bảo biến đó còn sống sau khi outer function return.

**Cơ chế thật: Context object trên heap**

```javascript
function createCounter() {
  let count = 0; // Bình thường, count sẽ nằm trên stack frame của createCounter
  // Nhưng V8 phát hiện: count bị capture bởi inner function
  // → V8 CHUYỂN count lên HEAP, vào một "Context Object"

  return function increment() {
    count++; // increment giữ reference tới Context Object chứa count
    return count;
  };
}

const inc = createCounter();
// createCounter() đã return → stack frame của nó BỊ POP
// Nhưng Context Object { count: 0 } vẫn SỐNG TRÊN HEAP
// vì 'inc' (trên heap) còn giữ reference đến nó
```

Memory layout:

```
Heap:
┌──────────────────────────────────────────────┐
│  Function Object "increment"                 │
│  ├── code: <bytecode ref>                    │
│  └── [[Environment]]: ──────────────────┐   │
│                                         │   │
│  Context Object ◄────────────────────── ┘   │
│  ├── count: 0                               │
│  └── outer: GlobalContext                   │
└──────────────────────────────────────────────┘

Stack (sau khi createCounter() return):
[createCounter frame đã bị pop]
[Chỉ còn Global frame với: inc → pointer tới Function Object]
```

`count` sống mãi trên heap miễn là `inc` còn được reference từ bất kỳ đâu.

### Garbage Collection — Mark & Sweep

GC là cơ chế V8 dùng để giải phóng heap memory của objects không còn được dùng nữa. **GC không biết object nào "cần"** — nó chỉ biết object nào **có thể reach được** từ program.

**Root set — điểm xuất phát của GC:**

```
GC Roots (những gì luôn "alive"):
├── Global object (window / global)
├── Call Stack — tất cả local variables đang active
├── V8 internal references
└── Native stack (C++ code trong engine)
```

Bất kỳ object nào **không thể reach** từ root set → garbage → có thể collect.

**Mark phase:**

```
1. GC bắt đầu từ root set
2. DFS/BFS traverse tất cả references
3. "Mark" mỗi object được visit bằng một bit

Ví dụ:

Root ──► GlobalEC
             │
             ├──► userCache (Map object) ──► User{id:1} → MARKED ✓
             │                          └──► User{id:2} → MARKED ✓
             │
             └──► eventListeners[] ──► Handler{fn: callback} → MARKED ✓
                                              │
                                              └──► bigDataObject → MARKED ✓

[Unreachable objects — NOT MARKED]:
  OldUser{id:0} ← không ai point tới nữa
  TempBuffer{}  ← local variable đã bị pop
```

**Sweep phase:**

```
4. GC scan toàn bộ heap
5. Object không có mark bit → FREE (ghi vào free list)
6. Mark bits reset cho lần GC tiếp theo

Heap sau sweep:
┌────────────────────────────────────────────────┐
│  [LIVE - userCache]      │ [FREE - OldUser]    │
│  [LIVE - User{1}]        │ [FREE - TempBuffer] │
│  [LIVE - User{2}]        │                     │
│  [LIVE - bigDataObject]  │ [FREE SPACE]        │
└────────────────────────────────────────────────┘
```

**Compact phase (không phải lúc nào cũng chạy):**

Sau nhiều sweep cycles, heap bị **fragmented** — nhiều small free gaps xen kẽ với live objects. Compact phase di chuyển live objects lại gần nhau, tạo ra contiguous free space. Expensive — V8 chỉ compact khi fragmentation đủ tệ.

### Generational GC — tại sao cần hai "thế hệ"

Mark & Sweep thuần túy có một vấn đề: phải scan **toàn bộ heap** mỗi lần. Nếu heap 1GB, GC pause sẽ rất dài.

**Generational Hypothesis:** Trong hầu hết programs, **objects chết trẻ**. Temp variables, intermediate objects trong functional operations, React elements trong re-renders — chúng sống rất ngắn. Chỉ một số ít objects sống lâu (global state, caches, DOM nodes).

V8 tận dụng điều này:

```
┌────────────────────────────────────────────────────────┐
│  NEW SPACE (Young Generation) — ~1-8MB                 │
│  ┌──────────────────┐  ┌───────────────────────────┐  │
│  │   From-Space     │  │      To-Space              │  │
│  │  (active alloc)  │  │  (empty, reserved)         │  │
│  └──────────────────┘  └───────────────────────────┘  │
│  GC: Scavenger — runs frequently, very fast (~1ms)     │
│                                                        │
│  OLD SPACE (Old Generation) — ~hundreds MB to GB       │
│  ┌────────────────────────────────────────────────┐   │
│  │  Long-lived objects, promoted from New Space   │   │
│  └────────────────────────────────────────────────┘   │
│  GC: Major GC (Mark-Sweep-Compact) — runs rarely,      │
│      slow (~100ms+), pause-inducing                    │
└────────────────────────────────────────────────────────┘
```

**Scavenger (Minor GC) — cơ chế thật:**

New Space dùng **semi-space** layout: chia đôi thành From-Space và To-Space.

```
Allocation:  Objects mới luôn vào From-Space

Scavenge trigger: From-Space gần đầy

Scavenge process:
1. Traverse tất cả live objects trong From-Space
2. Copy mỗi live object sang To-Space
3. Update tất cả pointers trỏ vào object (pointer patching)
4. Flip: To-Space trở thành From-Space mới (swap labels)
5. Old From-Space là trống → all dead objects tự động freed

Không cần "mark dead" — chỉ copy live, còn lại là garbage.
```

```
Before Scavenge:
From-Space: [A(live)] [B(dead)] [C(live)] [D(dead)] [E(live)]
To-Space:   [empty]

After Scavenge:
From-Space: [empty - now the new allocation space]
To-Space → renamed From: [A'] [C'] [E']  ← compacted, no fragmentation
```

**Promotion:** Nếu một object survive đủ số lần Scavenge (thường 2 lần), nó được **promoted** lên Old Space. V8 coi nó là "long-lived" và không muốn copy nó nữa trong minor GC.

**Write Barriers — vấn đề khi generational GC gặp cross-generation pointers:**

Nếu một Old Space object có pointer tới New Space object, Scavenger phải biết về pointer đó — nếu không, nó sẽ collect New Space object dù đang được reference.

V8 dùng **write barrier**: mỗi khi code viết một pointer (`obj.child = newObj`), V8 kiểm tra nếu đây là old-to-new pointer và ghi vào **remembered set**. Scavenger check remembered set khi xác định live objects.

Write barriers là overhead ẩn trong mọi object property assignment — không đáng kể cho từng operation, nhưng tích lũy trong loop millions iterations.

### Memory Leak Patterns — cơ chế từng loại

Memory leak trong JS không phải "memory không được free" theo nghĩa C — GC sẽ free nếu không có reference. Leak xảy ra khi **object vô tình còn được reference** dù không cần thiết nữa.

#### Pattern 1: Detached DOM Nodes

```javascript
// ❌ LEAK: Element bị remove khỏi DOM nhưng JS vẫn giữ reference

let cachedButton; // Biến global

function setupButton() {
  const button = document.createElement('button');
  button.textContent = 'Submit';
  document.body.appendChild(button);

  cachedButton = button; // ← Lưu reference vào global
  // ...
}

function cleanup() {
  document.body.removeChild(cachedButton);
  // Button đã bị xóa khỏi DOM
  // Nhưng cachedButton vẫn hold reference tới DOM node
  // → Node là "detached": không trong DOM, nhưng không thể GC
  // → Tất cả children của node này cũng bị giữ theo
}
```

```
Memory:
  Global Scope → cachedButton → [Detached Button Node]
                                       │
                                       └──► children, event listeners,
                                            parent refs (circular!)
                                            → tất cả bị giữ
```

```javascript
// ✅ FIX
function cleanup() {
  document.body.removeChild(cachedButton);
  cachedButton = null; // Xóa reference → GC có thể collect
}
```

#### Pattern 2: Forgotten Timers và Event Listeners

```javascript
// ❌ LEAK: Timer giữ closure → closure giữ large data

class DataDashboard {
  constructor() {
    this.reportData = new Array(100000).fill({ value: Math.random() });
    // reportData là large array

    // Timer tạo closure — closure capture 'this'
    // 'this' reference tới DataDashboard instance
    // DataDashboard instance giữ reportData
    // → reportData không thể GC kể cả khi dashboard bị "destroy"
    this.intervalId = setInterval(() => {
      this.render(); // closure giữ 'this' alive
    }, 1000);
  }

  destroy() {
    // ❌ Thiếu: clearInterval(this.intervalId)
    // Timer vẫn chạy, closure vẫn giữ reference
    // GC không thể collect instance này
  }
}
```

```javascript
// ✅ FIX: Luôn cleanup timers và event listeners

// React pattern tương đương:
useEffect(() => {
  const id = setInterval(fetchData, 5000);
```

#### Pattern 3: Closure giữ large data vô tình

```javascript
// ❌ LEAK: Closure capture toàn bộ outer scope
// Đây là bug tinh tế nhất — thường xảy ra với event listeners

function attachHandler(element) {
  // largeData được tạo để xử lý setup ban đầu

    // Handler chỉ cần element.id

  });

  // largeData được dùng xong ở đây
  processConfig(largeData);
  // Sau dòng này, largeData "không cần nữa"
  // Nhưng handler closure vẫn giữ reference tới nó!
}
```

### WeakRef và FinalizationRegistry — khi nào dùng

`WeakRef` và `FinalizationRegistry` (ES2021) cho phép giữ reference "yếu" tới object — reference không ngăn GC collect object đó.

**WeakRef:**

```javascript
// Use case: Cache không ngăn GC

class ImageCache {
  constructor() {
    this.cache = new Map(); // Map thường → strong reference → images không bao giờ GC
  }

  set(url, imageData) {
    // WeakRef: GC có thể collect imageData khi memory pressure cao
    this.cache.set(url, new WeakRef(imageData));
  }

  get(url) {
    const ref = this.cache.get(url);
    if (!ref) return null;

    const imageData = ref.deref(); // Lấy object — có thể return undefined nếu đã bị GC
    if (!imageData) {
      this.cache.delete(url); // Cleanup dead entry
      return null;
    }
    return imageData;
  }
}

// Hành vi:
// - Memory pressure thấp: imageData tồn tại, cache hit
// - Memory pressure cao: GC collect imageData, cache miss → re-fetch
// Cache tự động adjust với memory availability
```

**Khi nào KHÔNG dùng WeakRef:**

- Business-critical data không được phép mất
- Khi pattern `WeakMap` đơn giản hơn đủ dùng
- WeakRef không đảm bảo timing — object có thể survive lâu hơn bạn nghĩ hoặc bị collect sớm hơn

**FinalizationRegistry:**

```javascript
// Use case: Cleanup side effects khi object bị GC

const registry = new FinalizationRegistry((heldValue) => {
  // Callback chạy SAU KHI object bị GC
  // heldValue là data bạn muốn dùng cho cleanup
  console.log(`Cleaning up resource: ${heldValue}`);
  releaseNativeResource(heldValue);
});

function createManagedResource(resourceId) {
  const resource = acquireNativeResource(resourceId);

  // Đăng ký: khi 'resource' bị GC, callback sẽ chạy với resourceId
  registry.register(resource, resourceId);

  return resource;
}

// Caveats quan trọng:
// - Callback KHÔNG đảm bảo chạy (GC có thể không collect trước khi process exit)
// - Callback chạy ở thời điểm không xác định
// - Không dùng cho cleanup QUAN TRỌNG — dùng explicit cleanup thay
// - Chỉ dùng như safety net, không phải primary cleanup mechanism
```

```
│ parseValue       │           ← được push khi gọi
│ str: 0x??? ──────┼─── (string "42" trên heap)
│ num: 42          │ ← primitive, sống trực tiếp trên stack
└──────────────────┘ ← TOP (active frame)

Khi parseValue() return:
  - Frame parseValue bị POP
  - num=42 biến mất khỏi stack ngay lập tức
  - Return value 42 được đặt vào processRequest frame

Khi processRequest() return:
  - { ...req, result: 42 } được tạo trên HEAP (0x1400)
  - Frame processRequest bị POP
  - req pointer biến mất
  - Return value là pointer 0x1400
```

### Generational GC — lifecycle của một object

```
ALLOCATION:
new Object() → New Space (From-Space)
┌──────────────────────────────────────┐
│ From-Space                           │
│ [ObjA] [ObjB] [ObjC] [ObjD] [ObjE] │
└──────────────────────────────────────┘

SCAVENGE 1 (ObjB, ObjD unreachable):
Copy live objects to To-Space:
┌──────────────────────────────────────┐
│ To-Space → renamed to From-Space     │
│ [ObjA*] [ObjC*] [ObjE*]             │ ← survived once (age=1)
└──────────────────────────────────────┘
Old From-Space wiped → all garbage freed instantly

SCAVENGE 2 (ObjC unreachable, ObjA/ObjE age=2):
┌──────────────────────────────────────┐
│ New From-Space                        │
│ [new objects...]                      │
└──────────────────────────────────────┘
    ↓ PROMOTE (survived 2 scavenges)
┌──────────────────────────────────────┐
│ Old Space                            │
│ [ObjA] [ObjE]  ← promoted, no more  │
│                   copy overhead      │
└──────────────────────────────────────┘

Major GC (runs when Old Space pressure):
Mark → Sweep → (optionally Compact)
[ObjA] freed if no longer reachable
[ObjE] kept if still referenced
```

### Memory Leak — closure reference chain

```
Leak scenario: EventListener giữ component

Window (root)
    │
    └──► document.body
              │
              └──► <button> ──► eventListeners[]
                                        │
                                        └──► handler() closure
                                                  │
                                    [[Environment]]│
                                                  │
                                                  ▼
                                         Context Object {
                                           largeData: [100MB array],
                                           component: ComponentInstance
                                         }
                                                  │
                                                  └──► ComponentInstance
                                                            │
                                                            └──► more refs...

GC trace từ root:
Window → body → button → handler → Context → largeData
→ largeData MARKED (reachable) → CANNOT COLLECT

Fix: button.removeEventListener('click', handler)
→ button không còn giữ handler
→ handler không còn được root reach
→ toàn bộ chain: handler → Context → largeData → COLLECTABLE
```

### Debug stack overflow — hiểu tại sao xảy ra

```javascript
// ❌ BUG: Infinite recursion → Stack Overflow

function flattenTree(node) {
  // Mỗi lần gọi push một frame lên stack
  // Stack size ~1MB (Node.js default) → ~10,000-15,000 frames
  // Tree quá sâu → "Maximum call stack size exceeded"
  if (!node) return [];
  return [node.value, ...flattenTree(node.left), ...flattenTree(node.right)];
}

// Tree với 20,000 levels sâu → stack overflow
```

```javascript
// ✅ FIX 1: Iterative approach dùng explicit stack (trên heap)

function flattenTree(root) {
  const result = [];
  const stack = [root]; // Stack của chúng ta trên HEAP — không giới hạn như call stack

  while (stack.length > 0) {
    const node = stack.pop();
    if (!node) continue;
    result.push(node.value);
    // Push children — heap có thể grow theo nhu cầu
    if (node.right) stack.push(node.right);
    if (node.left) stack.push(node.left);
  }
  return result;
}

// ✅ FIX 2: Trampoline pattern — biến đệ quy thành iterative
// (relevant khi cần giữ recursive style)

function trampoline(fn) {
  return function (...args) {
    let result = fn(...args);
    // Thay vì call recursively → return thunk function
    // Trampoline gọi thunk, không bao giờ stack sâu hơn 1 frame
    while (typeof result === 'function') {
      result = result();
    }
    return result;
  };
}
```

### Production-level: React component cleanup pattern

```javascript
// ❌ LEAK thực tế trong React component

function DataTable({ endpoint }) {
  const [data, setData] = useState([]);
  const [largeBuffer] = useState(
    () => new ArrayBuffer(1024 * 1024), // 1MB buffer
  );

  useEffect(() => {
    // AbortController không được cleanup → fetch tiếp tục dù unmount
    fetch(endpoint)
      .then((res) => res.json())
      .then((json) => {
        // Component đã unmount → state update trên unmounted component
        // React 18+ không crash nhưng vẫn là leak pattern
        setData(json);
      });

    // WebSocket không được close → connection & listeners còn sống
    const ws = new WebSocket('wss://realtime.example.com');
    ws.onmessage = (e) => {
      // closure giữ setData → giữ component fiber → giữ largeBuffer
      setData((prev) => [...prev, JSON.parse(e.data)]);
    };

    // Không có cleanup!
  }, [endpoint]);

  return <Table data={data} />;
}
```

```javascript
// ✅ FIX — Explicit cleanup cho mọi side effect

function DataTable({ endpoint }) {
  const [data, setData] = useState([]);

  useEffect(() => {
    // 1. AbortController để cancel fetch khi unmount
    const controller = new AbortController();
    let isMounted = true; // Guard cho state update

    fetch(endpoint, { signal: controller.signal })
      .then((res) => res.json())
      .then((json) => {
        if (isMounted) setData(json);
        // Chỉ update state nếu vẫn mounted
      })
      .catch((err) => {
        if (err.name !== 'AbortError') console.error(err);
        // AbortError là expected khi unmount — không phải real error
      });

    // 2. WebSocket với proper cleanup
    const ws = new WebSocket('wss://realtime.example.com');
    ws.onmessage = (e) => {
      if (isMounted) setData((prev) => [...prev, JSON.parse(e.data)]);
    };

    // Cleanup function — React chạy khi unmount hoặc deps thay đổi
    return () => {
      isMounted = false; // Ngăn state update
      controller.abort(); // Cancel in-flight fetch
      ws.close(); // Đóng WebSocket → GC event listeners
      // closure trong ws.onmessage không còn được reference
      // → có thể GC cùng ws object
    };
  }, [endpoint]);

  return <Table data={data} />;
}
```

### Performance implications của GC

**GC pause là kẻ thù của 60fps:**

```
Minor GC (Scavenger): ~0.5-2ms  → thường không gây jank
Major GC (Mark-Sweep): ~50-200ms → chắc chắn drop frames

Timeline animation với GC pauses:
Frame 1: 8ms render ✓
Frame 2: 9ms render ✓
Frame 3: 11ms render ✓
Frame 4: [Major GC: 150ms] ← spike → user thấy freeze/jank
Frame 5: 8ms render ✓
```

**Cách giảm GC pressure trong animation/game loops:**

```javascript
// ❌ Object allocation trong animation loop — GC heaven

function animate() {
  requestAnimationFrame(animate);

  // Mỗi frame tạo objects mới → New Space fill up nhanh → Scavenge liên tục
  const velocity = { x: dx, y: dy }; // new object mỗi frame
  const newPosition = add(position, velocity); // new object mỗi frame
  const transformed = transform(newPosition); // new object mỗi frame

  render(transformed);
}

// ✅ Object reuse — zero allocation trong hot loop

const velocity = { x: 0, y: 0 }; // Tạo một lần
const newPosition = { x: 0, y: 0 }; // Tạo một lần

function animate() {
  requestAnimationFrame(animate);

  // Mutate thay vì tạo mới
  velocity.x = dx;
  velocity.y = dy;
  newPosition.x = position.x + velocity.x;
  newPosition.y = position.y + velocity.y;

  render(newPosition); // Không tạo object mới → không có GC pressure
}
```

### DevTools — Memory profiling workflow thực tế

**Detect memory leak trong Chrome:**

```
1. Mở DevTools → Memory tab
2. Chọn "Allocation instrumentation on timeline"
3. Record → Perform action nghi ngờ leak (e.g., navigate nhiều lần)
4. Stop recording

Đọc kết quả:
- Màu xanh = objects còn alive
- Màu xám = objects đã bị GC
- Thanh xanh cao và không giảm → objects accumulate → potential leak

5. Take heap snapshot (trước và sau action)
6. Comparison view → xem objects nào được tạo nhưng không freed
   → Focus vào "(detached)" objects → detached DOM nodes
   → Focus vào closures với large retained size
```

**Retained Size vs Shallow Size:**

```
Shallow Size: size của chính object object (không tính objects nó point tới)
Retained Size: tổng size của TẤT CẢ objects sẽ được freed NẾU object này bị freed

Ví dụ:
  Cache object: Shallow=48 bytes, Retained=50MB
  → Cache giữ 50MB data. Nếu free Cache → free 50MB

Khi debug leak: Sort by Retained Size → tìm thứ gì đang giữ memory nhiều nhất
```

### Câu 1: Tại sao V8 cần Generational GC thay vì chỉ dùng Mark & Sweep thuần túy? Scavenger hoạt động khác Major GC như thế nào?

**Đáp án:**

**Vấn đề của Mark & Sweep thuần túy:** Cần scan toàn bộ heap mỗi lần GC. Khi heap lớn (hàng trăm MB đến GB), pause time tỷ lệ thuận với heap size — không acceptable cho interactive applications cần 60fps.

**Generational Hypothesis:** Quan sát thực nghiệm cho thấy hầu hết objects "chết trẻ" — intermediate values trong functional operations, temporary DOM elements, React render outputs. Chỉ một phần nhỏ objects thực sự long-lived. GC không cần scan long-lived objects liên tục.

**New Space + Scavenger (Minor GC):** New Space dùng semi-space layout — chia đôi thành From-Space (active) và To-Space (empty). Khi From-Space đầy, Scavenger **copy live objects sang To-Space** rồi swap labels. Dead objects không cần "freed" — chúng bị bỏ lại trong old From-Space và toàn bộ space được reclaim ngay. Scavenge nhanh (~0.5–2ms) vì chỉ xử lý ~1–8MB và work là proportional to live objects (ít), không phải dead objects (nhiều).

**Promotion:** Objects survive đủ 2 lần Scavenge được promoted lên Old Space — V8 coi chúng là "long-lived" và không muốn copy tiếp.

**Old Space + Major GC:** Dùng full Mark-Sweep-Compact. Chạy ít thường xuyên, có pause dài hơn (~50–200ms). V8 dùng incremental marking và concurrent sweeping để giảm pause time — mark được chia thành nhiều chunks nhỏ xen kẽ với JS execution.

### Câu 2: Execution Context gồm những thành phần gì? Scope chain được tạo ra như thế nào khi JS engine resolve một variable?

**Đáp án:**

**Execution Context (EC) gồm 3 thành phần:**

1. **Variable Environment:** Chứa `var` declarations và function declarations. Tất cả được khởi tạo lên `undefined` (hoặc function body) ngay khi EC được tạo — đây là cơ chế hoisting.

2. **Lexical Environment:** Chứa `let`, `const` declarations và block scopes. Biến tồn tại nhưng ở trạng thái **TDZ (Temporal Dead Zone)** cho đến khi dòng khai báo được execute — đó là tại sao `let x = x + 1` throw ReferenceError.

3. **ThisBinding:** Giá trị của `this` trong context này. Với function call thông thường → `undefined` (strict) hoặc `global`. Với method call → object. Với `new` → object mới được tạo.

Mỗi EC cũng có **outer reference** — pointer tới EC của scope bên ngoài tại thời điểm function được _định nghĩa_ (lexical scoping, không phải dynamic scoping).

**Scope chain resolution:** Khi engine gặp identifier (ví dụ `TAX_RATE`):

1. Lookup trong current EC's Lexical Environment → không tìm thấy
2. Follow outer reference lên EC cha → lookup → không thấy
3. Follow tiếp → ... → Global EC → tìm thấy `TAX_RATE: 0.1` → return
4. Nếu đến Global EC mà vẫn không thấy → `ReferenceError`

Outer reference được **đóng băng tại thời điểm function được tạo** — không phải lúc gọi. Đây là tại sao closures hoạt động: inner function giữ reference tới outer EC ngay cả khi outer function đã return.

### Câu 3: Đoạn code sau có memory leak không? Nếu có, giải thích tại sao GC không thể thu hồi memory và cách fix.

```javascript
class NotificationService {
  constructor(eventBus) {
    this.notifications = [];
    this.userPreferences = new Array(50000).fill({ muted: false });

    eventBus.on('new-notification', (data) => {
      const processed = this.processNotification(data);
      this.notifications.push(processed);
    });
  }

  processNotification(data) {
    return { ...data, timestamp: Date.now() };
  }

  destroy() {
    this.notifications = [];
  }
}

const bus = new EventEmitter();
let service = new NotificationService(bus);

// ... sau một lúc
service.destroy();
service = null;
```

**Đáp án:**

**Có leak.** Dù `service = null` được gán, object `NotificationService` vẫn không thể bị GC.

**Tại sao:**

`eventBus.on('new-notification', callback)` đăng ký một **arrow function closure** vào `eventBus`. Arrow function này:

1. Nằm trên heap như một Function object
2. Có `[[Environment]]` reference tới Context Object của `NotificationService` constructor
3. Context Object chứa `this` reference tới `NotificationService` instance
4. `NotificationService` instance chứa `userPreferences` (50,000 phần tử array)

**Reference chain từ root:**

```
EventEmitter (bus, còn live)
  → listeners['new-notification'][]
  → arrow function closure
  → [[Environment]] Context Object
  → 'this' = NotificationService instance
  → userPreferences (50K array, ~1.6MB)
```

`service = null` chỉ xóa reference từ local variable. `bus` vẫn giữ listener → listener giữ closure → closure giữ instance → **instance và toàn bộ data của nó không thể GC.**

Thêm nữa: `this.notifications.push(processed)` sẽ tiếp tục accumulate data vì listener vẫn active.

**Fix:**

```javascript
class NotificationService {
  constructor(eventBus) {
    this.eventBus = eventBus;
    this.notifications = [];
    this.userPreferences = new Array(50000).fill({ muted: false });

    // Lưu bound handler để có thể removeListener sau
    this._handler = (data) => {
      const processed = this.processNotification(data);
      this.notifications.push(processed);
    };
    eventBus.on('new-notification', this._handler);
  }

  processNotification(data) {
    return { ...data, timestamp: Date.now() };
  }

  destroy() {
    // Xóa listener → cắt reference chain
    this.eventBus.off('new-notification', this._handler);
    this._handler = null;
    this.notifications = [];
    this.userPreferences = null;
    this.eventBus = null;
  }
}
```

Sau `destroy()` và `service = null`: EventBus không còn listener reference → closure không còn được reach từ root → toàn bộ chain có thể GC.
