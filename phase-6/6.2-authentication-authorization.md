## Tổng quan threat model trước khi học

Trước khi học từng cơ chế, cần hiểu **mục tiêu của attacker** trong authentication/authorization:

```
Mục tiêu của attacker:
┌──────────────────────────────────────────────────────────┐
│ 1. Steal credentials (username/password)                  │
│ 2. Steal session/token (impersonate user sau khi login)  │
│ 3. Forge token (tạo token hợp lệ mà không cần login)    │
│ 4. Escalate privilege (regular user → admin)             │
│ 5. Reuse stolen token sau khi đã bị revoke               │
└──────────────────────────────────────────────────────────┘

Mỗi cơ chế auth phải defend against ít nhất một trong những goals này.
Khi học, luôn hỏi: "Cơ chế này defend against mục tiêu nào?"
```

## 1. Cookie-based Auth vs Token-based Auth

### 1.1 Cookie-based Auth — cơ chế thật

```
Flow:
1. User POST /login với credentials
2. Server verify → tạo session record trong DB/memory store
3. Server set-cookie: sessionId=abc123 (HttpOnly, Secure)
4. Browser lưu cookie, tự động gửi kèm mọi request tới domain đó
5. Server nhận request → đọc sessionId → lookup trong session store
6. Session store trả về user data → request được authenticated
```

```
SESSION STORE LOOKUP:

Browser Request:
Cookie: sessionId=abc123
         │
         ▼
Server middleware
         │
         ▼
Session Store (Redis/DB):
┌─────────────────────────────────────────┐
│ Key: "abc123"                           │
│ Value: {                                │
│   userId: 42,                           │
│   role: 'user',                         │
│   createdAt: 2024-01-15T10:00:00Z,     │
│   lastActive: 2024-01-15T12:30:00Z,    │
│   ipAddress: '1.2.3.4',               │
│   userAgent: 'Mozilla/5.0...'          │
│ }                                       │
│ TTL: 3600 (expires in 1 hour)          │
└─────────────────────────────────────────┘
         │
         ▼
req.user = { userId: 42, role: 'user' }
```

**Đặc điểm cốt lõi của cookie-based auth:**

- **Stateful** — server phải maintain session store
- **Revocable** — xóa key trong session store → session invalid ngay lập tức
- **Opaque** — client chỉ có sessionId (random string), không biết nội dung
- **Scalability challenge** — nhiều server instances phải share session store (Redis)

```javascript
// Production implementation — Express + Redis session store
const session = require('express-session');
const RedisStore = require('connect-redis').default;
const { createClient } = require('redis');
const crypto = require('crypto');

const redisClient = createClient({ url: process.env.REDIS_URL });
await redisClient.connect();

app.use(
  session({
    store: new RedisStore({ client: redisClient }),

    // Secret dùng để sign sessionId cookie (HMAC)
    // Kẻ tấn công không thể forge sessionId hợp lệ nếu không biết secret
    secret: process.env.SESSION_SECRET, // Min 32 bytes random string

    resave: false, // Không save session nếu không có thay đổi
    saveUninitialized: false, // Không tạo session cho unauthenticated requests

    cookie: {
      httpOnly: true, // JS không đọc được → chống XSS steal
      secure: true, // Chỉ HTTPS
      sameSite: 'strict', // Chống CSRF
      maxAge: 30 * 60 * 1000, // 30 phút
    },

    // Generate cryptographically random session IDs
    genid: () => crypto.randomBytes(32).toString('hex'),
  }),
);

// Login handler
app.post('/login', async (req, res) => {
  const { username, password } = req.body;

  const user = await db.users.findOne({ username });
  if (!user) {
    // Timing attack prevention: vẫn chạy bcrypt dù user không tồn tại
    // Nếu return ngay → attacker biết username không tồn tại qua timing difference
    await bcrypt.compare(password, '$2b$12$placeholder.hash.to.prevent.timing');
    return res.status(401).json({ error: 'Invalid credentials' });
  }

  const isValid = await bcrypt.compare(password, user.passwordHash);
  if (!isValid) {
    return res.status(401).json({ error: 'Invalid credentials' });
  }

  // Session fixation prevention:
  // Regenerate session ID sau khi login thành công
  // Nếu attacker đã biết sessionId trước khi login (e.g., via XSS)
  // và server không regenerate → attacker có session hợp lệ sau khi victim login
  req.session.regenerate((err) => {
    if (err) return res.status(500).send('Session error');

    req.session.userId = user.id;
    req.session.role = user.role;
    req.session.loginAt = Date.now();

    req.session.save((err) => {
      if (err) return res.status(500).send('Session error');
      res.json({ success: true });
    });
  });
});

// Logout — invalidate session
app.post('/logout', (req, res) => {
  req.session.destroy((err) => {
    res.clearCookie('connect.sid');
    res.json({ success: true });
  });
  // Session record bị xóa khỏi Redis ngay lập tức
  // sessionId cookie vẫn còn trong browser nhưng không valid
});
```

### 1.2 Token-based Auth (JWT) — cơ chế thật

JWT (JSON Web Token) là stateless authentication: **server không cần store gì cả**. Toàn bộ user data được encode trong token, và token được signed bằng secret key.

```
JWT Structure:
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOjQyLCJyb2xlIjoidXNlciIsImV4cCI6MTcwNTMyMTIwMH0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

─────────────────┬──────────────────────────────────────────────
Header (base64)  │ Payload (base64)                │ Signature
─────────────────┴──────────────────────────────────────────────

Decode:
Header:  { "alg": "HS256", "typ": "JWT" }
Payload: { "userId": 42, "role": "user", "exp": 1705321200, "iat": 1705317600 }
Signature: HMAC_SHA256(base64(header) + "." + base64(payload), SECRET_KEY)
```

**Cơ chế verification — điều thực sự xảy ra:**

```
Request đến server:
Authorization: Bearer eyJhbGci...

Server JWT middleware:

1. Split token thành 3 parts (by '.')
   header = parts[0]
   payload = parts[1]
   signature = parts[2]

2. Decode header → lấy algorithm (HS256)

3. Recompute signature:
   expectedSig = HMAC_SHA256(
     header + "." + payload,
     SERVER_SECRET_KEY
   )

4. Compare: expectedSig === signature?
   → Nếu khớp: token không bị tamper → trust payload
   → Nếu không khớp: token bị modify hoặc forge → reject

5. Decode payload → kiểm tra exp (expiration)
   → Nếu exp < now() → token expired → reject

6. req.user = payload data
   (KHÔNG query database — stateless)
```

```
Tại sao attacker không thể forge JWT?

Attacker muốn tạo JWT với { "userId": 42, "role": "admin" }:
1. Tạo header + payload → base64 encode → dễ
2. Cần compute signature: HMAC_SHA256(data, SECRET_KEY)
3. Attacker không biết SECRET_KEY → không compute được signature hợp lệ
4. Server verify: signature không khớp → reject

Attacker muốn modify payload của existing JWT:
1. Decode payload → thay đổi role: "admin"
2. Re-encode → nhưng signature vẫn là signature của payload cũ
3. Server recompute signature của payload mới → không khớp với signature cũ → reject
```

### 1.3 Cookie vs Token — trade-offs thật sự

```
                    COOKIE-BASED           JWT TOKEN-BASED
─────────────────────────────────────────────────────────────
State               Stateful (server       Stateless (no server
                    stores session)        storage needed)

Scalability         Cần shared session     Bất kỳ server nào
                    store (Redis)          verify được → easy scale

Revocation          Instant — xóa         Khó — phải maintain
                    session record         token blacklist
                    → invalid ngay         (mất stateless benefit)

Token size          ~50 bytes (ID only)   ~200-500 bytes
                                          (full payload)

Cross-domain        Khó — SameSite        Dễ — gửi qua header
                    block cross-site       Authorization: Bearer

XSS                 HttpOnly → JS         Nếu lưu trong
                    không đọc được        localStorage → JS đọc được
                    cookie → safer         → XSS steal token

CSRF                Cần SameSite +        Không dùng cookie →
                    CSRF token             không có CSRF risk

Mobile/API          Awkward với cookies    Natural — HTTP header

Payload visibility  Opaque (random ID)    Base64 decoded dễ dàng
                                          (không encrypt, chỉ sign)
```

### 1.4 JWT — những anti-patterns nguy hiểm

#### Anti-pattern 1: Algorithm confusion attack

```javascript
// DANGEROUS: Trust algorithm từ token header
const jwt = require('jsonwebtoken');

// VULNERABLE implementation:
function verifyToken(token) {
  const decoded = jwt.decode(token); // Decode không verify
  const algorithm = decoded.header.alg; // Lấy algo từ token header
  return jwt.verify(token, publicKey, { algorithms: [algorithm] }); // Dùng algo của token
}

// Attack:
// Server dùng RS256 (asymmetric): sign bằng private key, verify bằng public key
// Attacker biết public key (public!) → dùng public key như HMAC secret
// Attacker tạo token với alg: "HS256" → sign bằng HMAC với public key làm secret
// Server nhận token → đọc alg: "HS256" → verify bằng HMAC(data, publicKey)
// publicKey là known value → HMAC verify pass → token được accept
// → Attacker forge arbitrary JWT

// FIX: Luôn hardcode algorithm, không tin header
function verifyToken(token) {
  return jwt.verify(token, process.env.JWT_SECRET, {
    algorithms: ['HS256'], // Hardcode, không đọc từ token
  });
}
```

#### Anti-pattern 2: Lưu JWT trong localStorage

```javascript
// VULNERABLE: localStorage accessible bởi JS → XSS steal token
localStorage.setItem('token', jwtToken);
// Bất kỳ JS nào trên page đều có thể:
// document.cookie → không đọc được HttpOnly cookie
// localStorage.getItem('token') → đọc được ngay

// BETTER: Lưu JWT trong HttpOnly cookie
// Vẫn mất lợi thế "không dùng cookie" nhưng được security

// Tại sao nhiều người vẫn dùng localStorage?
// 1. SPA cần gửi token cross-domain → cookie SameSite block
// 2. Không muốn xử lý CSRF token
// 3. Mobile apps không có cookie jar

// Trade-off:
// localStorage: CSRF-safe, XSS-vulnerable
// HttpOnly cookie: XSS-safe, CSRF-vulnerable (cần SameSite + CSRF token)

// Production pattern tốt nhất cho SPA:
// - Access token: memory (JS variable, không persist)
// - Refresh token: HttpOnly cookie
// Chi tiết ở phần Refresh Token Rotation
```

#### Anti-pattern 3: JWT không có expiration

```javascript
// VULNERABLE: Token không expire → nếu bị steal, valid forever
const token = jwt.sign({ userId: 42 }, secret);
// Không có exp → token hợp lệ mãi mãi

// Nếu user bị compromised và đổi password:
// Cookie session: logout destroy session → done
// JWT: token cũ vẫn valid cho đến khi expire
// → Attacker tiếp tục access dù victim đổi password

// FIX: Luôn có exp, ngắn
const accessToken = jwt.sign(
  { userId: 42, role: 'user' },
  process.env.JWT_SECRET,
  { expiresIn: '15m' }, // 15 phút — đủ ngắn
);

// Claim chuẩn trong JWT:
// iss: issuer ("https://auth.example.com")
// sub: subject (userId)
// aud: audience ("https://api.example.com") — prevent token reuse cross-service
// exp: expiration timestamp
// iat: issued at timestamp
// jti: JWT ID (unique, dùng cho blacklist nếu cần)
```

#### Anti-pattern 4: Sensitive data trong JWT payload

```javascript
// JWT chỉ được SIGN, không ENCRYPT
// Base64 decode được ngay — không có secret needed
// btoa(atob("eyJ1c2VySWQiOjQyLCJwYXNzd29yZCI6InNlY3JldCJ9"))

// NEVER đặt vào JWT payload:
const token = jwt.sign(
  {
    userId: 42,
    password: 'hashed_password', // KHÔNG
    ssn: '123-45-6789', // KHÔNG
    creditCard: '4111111111111111', // KHÔNG
    internalNotes: 'VIP customer', // Cẩn thận
  },
  secret,
);

// OK để đặt trong JWT:
const token = jwt.sign(
  {
    sub: user.id, // User ID (public reference)
    role: user.role, // Role/permissions
    iss: 'auth.example.com',
    aud: 'api.example.com',
    exp: Math.floor(Date.now() / 1000) + 15 * 60,
    iat: Math.floor(Date.now() / 1000),
    jti: crypto.randomUUID(), // Unique token ID
  },
  secret,
);
```

## 2. OAuth 2.0 Flows

### 2.1 Tại sao OAuth 2.0 tồn tại

```
Vấn đề trước OAuth:
App muốn đọc Google Calendar của user
→ User phải cho App biết Google password
→ App có toàn quyền access Google account
→ User không thể revoke quyền riêng cho App này
→ Nếu App bị hack → Google password bị expose

OAuth 2.0 giải quyết:
→ User không bao giờ cho App biết password
→ Google authenticate user trực tiếp
→ Google cấp "access token" với limited scope
→ User có thể revoke token cho App này mà không ảnh hưởng gì khác
```

### 2.2 Authorization Code Flow + PKCE — cơ chế thật

Đây là flow DÙNG CHO SPA, mobile app, và bất kỳ public client nào (không có client secret).

```
PKCE = Proof Key for Code Exchange
Giải quyết: authorization code interception attack
```

```
FULL FLOW với PKCE:

1. App tạo code_verifier (random string, 43-128 chars)
   code_verifier = crypto.randomBytes(32).toString('base64url')
   → "dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk"

2. App compute code_challenge từ verifier:
   code_challenge = base64url(SHA256(code_verifier))
   → "E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM"

   code_challenge_method = "S256"
   (Lưu code_verifier trong memory/sessionStorage — không gửi đi)

3. App redirect user tới Authorization Server với:
   https://auth.google.com/oauth/authorize?
     client_id=your-app-id
     &redirect_uri=https://yourapp.com/callback
     &response_type=code
     &scope=calendar.read profile
     &state=random-state-value           ← CSRF protection
     &code_challenge=E9Melhoa2...        ← gửi challenge, giữ verifier
     &code_challenge_method=S256

4. Authorization Server:
   a. Authenticate user (Google login page)
   b. Show consent screen: "YourApp wants to: read your calendar"
   c. User approve
   d. Server redirect back:
      https://yourapp.com/callback?
        code=SplxlOBeZQQYbYS6WxSbIA  ← authorization code (short-lived, one-time)
        &state=random-state-value

5. App verify state === stored state (CSRF check)

6. App POST tới Token Endpoint:
   POST https://auth.google.com/oauth/token
   {
     grant_type: "authorization_code",
     code: "SplxlOBeZQQYbYS6WxSbIA",
     redirect_uri: "https://yourapp.com/callback",
     client_id: "your-app-id",
     code_verifier: "dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk"
     ← gửi verifier để prove ownership của challenge
   }

7. Authorization Server:
   SHA256(code_verifier) === stored code_challenge?
   → YES → code exchange thành công → trả về tokens:
   {
     access_token: "ya29.a0...",  ← dùng để call API
     refresh_token: "1//0G...",   ← dùng để lấy access token mới
     expires_in: 3600,
     token_type: "Bearer",
     scope: "calendar.read profile"
   }
```

```
Tại sao PKCE ngăn authorization code interception?

Không có PKCE:
Attacker intercept redirect: callback?code=SplxlOBeZQQ...
→ Attacker gửi code tới token endpoint với client_id
→ Nhận access token → compromised

Với PKCE:
Attacker intercept redirect: callback?code=SplxlOBeZQQ...
→ Attacker POST code tới token endpoint
→ Server yêu cầu code_verifier
→ Attacker không có code_verifier (ở trong app memory của victim)
→ SHA256(attacker's guess) ≠ stored code_challenge → reject
→ Code exchange fail
```

```javascript
// SPA Implementation (React example)

class PKCEAuth {
  async login() {
    // Step 1: Generate PKCE pair
    const codeVerifier = this.generateCodeVerifier();
    const codeChallenge = await this.generateCodeChallenge(codeVerifier);
    const state = crypto.randomUUID();

    // Store để dùng sau khi redirect về
    sessionStorage.setItem('pkce_verifier', codeVerifier);
    sessionStorage.setItem('oauth_state', state);

    // Step 2: Build authorization URL
    const params = new URLSearchParams({
      client_id: process.env.REACT_APP_CLIENT_ID,
      redirect_uri: `${window.location.origin}/callback`,
      response_type: 'code',
      scope: 'openid profile email',
      state,
      code_challenge: codeChallenge,
      code_challenge_method: 'S256',
    });

    window.location.href = `${AUTH_SERVER}/authorize?${params}`;
  }

  async handleCallback(code, returnedState) {
    // Step 5: CSRF verification
    const storedState = sessionStorage.getItem('oauth_state');
    if (returnedState !== storedState) {
      throw new Error('State mismatch — possible CSRF attack');
    }

    const codeVerifier = sessionStorage.getItem('pkce_verifier');

    // Cleanup — prevent reuse
    sessionStorage.removeItem('pkce_verifier');
    sessionStorage.removeItem('oauth_state');

    // Step 6: Exchange code for tokens
    const response = await fetch(`${AUTH_SERVER}/token`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
      body: new URLSearchParams({
        grant_type: 'authorization_code',
        code,
        redirect_uri: `${window.location.origin}/callback`,
        client_id: process.env.REACT_APP_CLIENT_ID,
        code_verifier: codeVerifier, // Prove ownership
      }),
    });

    const tokens = await response.json();
    // Store tokens securely — see section below
    this.storeTokens(tokens);
  }

  generateCodeVerifier() {
    const array = new Uint8Array(32);
    crypto.getRandomValues(array);
    return btoa(String.fromCharCode(...array))
      .replace(/\+/g, '-')
      .replace(/\//g, '_')
      .replace(/=/g, '');
  }

  async generateCodeChallenge(verifier) {
    const encoder = new TextEncoder();
    const data = encoder.encode(verifier);
    const hash = await crypto.subtle.digest('SHA-256', data);
    return btoa(String.fromCharCode(...new Uint8Array(hash)))
      .replace(/\+/g, '-')
      .replace(/\//g, '_')
      .replace(/=/g, '');
  }
}
```

### 2.3 Client Credentials Flow — khi nào dùng

```
Dùng cho: Machine-to-machine (M2M) communication
KHÔNG có user involvement

Use cases:
- Backend service A gọi Backend service B
- Cron job gọi internal API
- Microservice authentication

Flow:
Service A → POST /token {
  grant_type: "client_credentials",
  client_id: "service-a",
  client_secret: "secret-key"
}
→ Auth Server verify client credentials
→ Return access token (không có refresh token — không cần)
→ Service A dùng token để gọi Service B
```

```javascript
// Node.js — Client Credentials
class ServiceAuth {
  constructor() {
    this.accessToken = null;
    this.tokenExpiry = null;
  }

  async getToken() {
    // Reuse token nếu còn valid (với buffer 30s)
    if (this.accessToken && Date.now() < this.tokenExpiry - 30000) {
      return this.accessToken;
    }

    const response = await fetch(`${AUTH_SERVER}/token`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
      body: new URLSearchParams({
        grant_type: 'client_credentials',
        client_id: process.env.CLIENT_ID,
        client_secret: process.env.CLIENT_SECRET,
        scope: 'internal:read internal:write',
      }),
    });

    const { access_token, expires_in } = await response.json();
    this.accessToken = access_token;
    this.tokenExpiry = Date.now() + expires_in * 1000;

    return this.accessToken;
  }

  async callAPI(endpoint, options = {}) {
    const token = await this.getToken();
    return fetch(endpoint, {
      ...options,
      headers: {
        ...options.headers,
        Authorization: `Bearer ${token}`,
      },
    });
  }
}

// Singleton — share token across requests
const serviceAuth = new ServiceAuth();
module.exports = serviceAuth;
```

## 3. Secure Cookie Storage

### 3.1 HttpOnly + Secure + SameSite — tại sao cần cả ba

```
Mỗi attribute bảo vệ một attack vector khác nhau:

HttpOnly:
  Cookie header được gửi bởi browser
  JS KHÔNG đọc được qua document.cookie

  Bảo vệ: XSS steal cookie
  Attacker inject JS → document.cookie → undefined (HttpOnly cookies không xuất hiện)

  Không bảo vệ: CSRF (cookie vẫn được gửi tự động)
  Không bảo vệ: Network interception (vẫn sent qua HTTP nếu không có Secure)

Secure:
  Cookie CHỈ được gửi qua HTTPS
  HTTP request → browser không đính kèm cookie

  Bảo vệ: Network eavesdropping (attacker trên same network)
  Không bảo vệ: XSS (JS vẫn có thể đọc non-HttpOnly cookie)
  Không bảo vệ: CSRF

SameSite=Strict:
  Cookie chỉ gửi trong same-site context

  Bảo vệ: CSRF từ cross-site requests
  Không bảo vệ: XSS (JS trên same site vẫn có thể thực hiện requests)
  Không bảo vệ: Network interception
```

```javascript
// Tất cả attributes cần thiết cùng lúc:
res.cookie('sessionId', token, {
  httpOnly: true, // Chống XSS steal
  secure: true, // Chống network sniff
  sameSite: 'strict', // Chống CSRF

  // Domain và Path — giới hạn scope cookie
  domain: 'example.com', // Gửi cho tất cả subdomains
  path: '/', // Gửi cho tất cả paths

  // Expiry
  maxAge: 15 * 60 * 1000, // 15 phút (milliseconds) — cho Set-Cookie header
  // hoặc
  expires: new Date(Date.now() + 15 * 60 * 1000), // Absolute time

  // Không set maxAge/expires → session cookie (xóa khi browser close)
});

// Cookie Prefixes — thêm tầng bảo vệ:
// __Host- prefix: browser enforce Secure + path=/ + no domain
// __Secure- prefix: browser enforce Secure attribute
res.cookie('__Host-sessionId', token, {
  httpOnly: true,
  secure: true,
  sameSite: 'strict',
  path: '/',
  // Không set domain khi dùng __Host- prefix
});
```

## 4. Refresh Token Rotation

### 4.1 Vấn đề cần giải quyết

```
Access token expire nhanh (15m) → tốt cho security
Nhưng user phải login lại mỗi 15 phút → UX tệ

Giải pháp: Refresh tokens
Access token (short-lived, 15m): dùng để call API
Refresh token (long-lived, 7-30 days): dùng để lấy access token mới

Khi access token expire:
→ Dùng refresh token để lấy access token mới
→ User không cần login lại
→ Nhưng nếu refresh token bị steal → attacker access được 30 ngày

Refresh Token Rotation giải quyết:
→ Mỗi lần dùng refresh token → nhận refresh token MỚI
→ Token cũ bị invalidate
→ Nếu attacker steal và dùng token cũ → server detect duplicate use → revoke tất cả
```

```
REFRESH TOKEN ROTATION FLOW:

Lần 1 (Login):
Server → { access_token: AT1, refresh_token: RT1 }
DB: RT1 = { valid: true, family: "user-42-family-1" }

15 phút sau (AT1 expire):
Client → POST /token/refresh { refresh_token: RT1 }
Server:
  1. RT1 còn valid? YES
  2. Tạo AT2 mới
  3. Tạo RT2 mới
  4. Invalidate RT1
  5. DB: RT2 = { valid: true, family: "user-42-family-1" }
  6. Return { access_token: AT2, refresh_token: RT2 }

15 phút sau (AT2 expire):
Legitimate client → POST /token/refresh { refresh_token: RT2 }
→ Nhận AT3, RT3

ATTACK SCENARIO — Attacker steal RT1:
Legitimate client đã dùng RT1 → nhận RT2
Attacker dùng RT1 (cũ) → POST /token/refresh { refresh_token: RT1 }
Server: RT1 invalid! → Phát hiện reuse!
Server: Revoke toàn bộ "user-42-family-1" token family
→ Cả legitimate user và attacker đều bị logout
→ User phải login lại → safe
→ Attacker không còn access
```

```javascript
// Implementation — Node.js

const tokenService = {
  async createTokenPair(userId) {
    const family = crypto.randomUUID(); // Token family ID

    const accessToken = jwt.sign(
      { sub: userId, type: 'access' },
      process.env.ACCESS_TOKEN_SECRET,
      { expiresIn: '15m' },
    );

    const refreshToken = crypto.randomBytes(40).toString('hex');
    const hashedRefreshToken = await bcrypt.hash(refreshToken, 10);

    // Lưu hashed refresh token vào DB
    await db.refreshTokens.create({
      userId,
      tokenHash: hashedRefreshToken,
      family,
      expiresAt: new Date(Date.now() + 30 * 24 * 60 * 60 * 1000), // 30 days
      used: false,
    });

    return { accessToken, refreshToken, family };
  },

  async rotateRefreshToken(incomingRefreshToken) {
    // Tìm tất cả tokens để kiểm tra
    const tokens = await db.refreshTokens.findAll({
      where: { used: false, expiresAt: { [Op.gt]: new Date() } },
    });

    // Tìm matching token (compare hash)
    let matchedToken = null;
    for (const token of tokens) {
      const matches = await bcrypt.compare(
        incomingRefreshToken,
        token.tokenHash,
      );
      if (matches) {
        matchedToken = token;
        break;
      }
    }

    if (!matchedToken) {
      // Token không tồn tại hoặc đã hết hạn
      throw new Error('Invalid refresh token');
    }

    if (matchedToken.used) {
      // REUSE DETECTED: token đã được dùng rồi
      // Revoke toàn bộ family → force logout tất cả
      await db.refreshTokens.update(
        { used: true },
        { where: { family: matchedToken.family } },
      );
      throw new Error('Refresh token reuse detected — security event');
    }

    // Mark token hiện tại là đã dùng
    await db.refreshTokens.update(
      { used: true },
      { where: { id: matchedToken.id } },
    );

    // Tạo token pair mới (cùng family)
    const newAccessToken = jwt.sign(
      { sub: matchedToken.userId, type: 'access' },
      process.env.ACCESS_TOKEN_SECRET,
      { expiresIn: '15m' },
    );

    const newRefreshToken = crypto.randomBytes(40).toString('hex');
    const newHashedRefreshToken = await bcrypt.hash(newRefreshToken, 10);

    await db.refreshTokens.create({
      userId: matchedToken.userId,
      tokenHash: newHashedRefreshToken,
      family: matchedToken.family, // Giữ nguyên family
      expiresAt: new Date(Date.now() + 30 * 24 * 60 * 60 * 1000),
      used: false,
    });

    return { accessToken: newAccessToken, refreshToken: newRefreshToken };
  },
};

// Secure storage pattern cho SPA:
// Access token → JavaScript memory (biến, không localStorage)
// Refresh token → HttpOnly cookie

app.post('/token/refresh', async (req, res) => {
  // Refresh token từ HttpOnly cookie — không từ body (JS không đọc được)
  const refreshToken = req.cookies.refreshToken;

  try {
    const { accessToken, refreshToken: newRefreshToken } =
      await tokenService.rotateRefreshToken(refreshToken);

    // Set new refresh token cookie
    res.cookie('refreshToken', newRefreshToken, {
      httpOnly: true,
      secure: true,
      sameSite: 'strict',
      maxAge: 30 * 24 * 60 * 60 * 1000,
    });

    // Access token gửi trong response body
    // Client lưu trong memory (biến JS)
    res.json({ accessToken });
  } catch (error) {
    // Clear cookie nếu có security event
    res.clearCookie('refreshToken');
    res.status(401).json({ error: error.message });
  }
});
```

### 4.2 Silent Refresh Pattern — UX không gián đoạn

```javascript
// SPA Token Manager — access token trong memory, auto-refresh

class TokenManager {
  constructor() {
    this.accessToken = null;
    this.refreshPromise = null; // Prevent concurrent refresh calls
  }

  setAccessToken(token) {
    this.accessToken = token;

    // Schedule refresh TRƯỚC khi expire
    // Decode JWT để lấy exp (không verify — chỉ đọc payload)
    const payload = JSON.parse(atob(token.split('.')[1]));
    const expiresIn = payload.exp * 1000 - Date.now();
    const refreshAt = expiresIn - 60000; // Refresh 1 phút trước khi expire

    if (refreshAt > 0) {
      setTimeout(() => this.silentRefresh(), refreshAt);
    }
  }

  async silentRefresh() {
    // Prevent concurrent refresh — nếu đang refresh, return promise đó
    if (this.refreshPromise) return this.refreshPromise;

    this.refreshPromise = fetch('/token/refresh', {
      method: 'POST',
      credentials: 'include', // Include HttpOnly cookie
    })
      .then((res) => {
        if (!res.ok) {
          // Refresh failed → logout user
          this.accessToken = null;
          window.dispatchEvent(new Event('auth:logout'));
          throw new Error('Session expired');
        }
        return res.json();
      })
      .then(({ accessToken }) => {
        this.setAccessToken(accessToken);
        return accessToken;
      })
      .finally(() => {
        this.refreshPromise = null;
      });

    return this.refreshPromise;
  }

  async getValidToken() {
    if (!this.accessToken) {
      return this.silentRefresh();
    }

    // Check expiry
    const payload = JSON.parse(atob(this.accessToken.split('.')[1]));
    const isExpired = payload.exp * 1000 < Date.now();

    if (isExpired) {
      return this.silentRefresh();
    }

    return this.accessToken;
  }

  async fetch(url, options = {}) {
    const token = await this.getValidToken();
    return fetch(url, {
      ...options,
      headers: {
        ...options.headers,
        Authorization: `Bearer ${token}`,
      },
    });
  }
}

const auth = new TokenManager();
export default auth;

// Usage trong app:
// const response = await auth.fetch('/api/user/profile');
// Token tự động refresh nếu cần, user không bao giờ thấy
```

## 5. WebAuthn / Passkeys

### 5.1 Cơ chế thật — tại sao passkeys không thể bị phishing

```
Password auth — vulnerable to phishing:
1. Attacker tạo evil-bank.com trông giống bank.com
2. User nhập password vào evil-bank.com
3. Attacker có password → có thể dùng ở bank.com thật

WebAuthn — phishing-resistant by design:
1. Authentication gắn với origin (domain)
2. Credential được tạo cho "bank.com" cụ thể
3. evil-bank.com là origin khác → credential không apply
4. Browser tự động check origin — user không cần verify bằng mắt
```

```
WebAuthn Registration Flow:

1. Server tạo challenge (random bytes):
   { challenge: "u2f...random", rp: { name: "Bank", id: "bank.com" }, user: {...} }

2. Browser gọi:
   const credential = await navigator.credentials.create({ publicKey: options })

3. Authenticator (Touch ID / Face ID / Security key):
   - Tạo key pair (private key, public key)
   - Private key: lưu trong secure enclave (KHÔNG BAO GIỜ rời khỏi device)
   - Sign challenge với private key
   - Trả về public key + signature + attestation

4. Browser gửi về server:
   { id, rawId, response: { attestationObject, clientDataJSON }, type }

5. Server verify:
   - Verify origin trong clientDataJSON === server's expected origin
     (đây là phishing protection: origin baked vào signature)
   - Verify challenge match
   - Verify signature với public key
   - Lưu public key → đây là "passkey" của user

WebAuthn Authentication Flow:

1. Server gửi challenge mới (mỗi login)

2. Browser gọi:
   const assertion = await navigator.credentials.get({ publicKey: options })

3. Authenticator:
   - User verify (Touch ID / Face ID / PIN)
   - Sign challenge với private key (private key không rời device)
   - Trả về signature

4. Server verify signature với stored public key → authenticated
   - Không có password → không có gì để steal
   - Private key không bao giờ gửi đi → không thể intercept
   - Challenge unique mỗi lần → không thể replay
   - Origin baked vào signature → phishing impossible
```

```javascript
// Simplified WebAuthn Registration (server-side)
const {
  generateRegistrationOptions,
  verifyRegistrationResponse,
} = require('@simplewebauthn/server');

app.get('/webauthn/register/options', async (req, res) => {
  const user = req.user;

  const options = await generateRegistrationOptions({
    rpName: 'My App',
    rpID: 'myapp.com',
    userID: user.id,
    userName: user.email,

    // Authenticator types
    authenticatorSelection: {
      authenticatorAttachment: 'platform', // Touch ID / Face ID
      // hoặc 'cross-platform' cho YubiKey

      id: cred.credentialID,

});

    response: req.body,

    expectedRPID: 'myapp.com',

  if (verification.verified) {
    // Lưu public key vào DB
    await db.credentials.create({
      userId: req.user.id,
      credentialID: verification.registrationInfo.credentialID,
      credentialPublicKey: verification.registrationInfo.credentialPublicKey,
      counter: verification.registrationInfo.counter,
    });

    res.json({ verified: true });
  }
});
```

## Câu hỏi ôn tập

---

**Câu 1 — Tại sao JWT không thể "revoke" ngay lập tức như session cookie? Và production systems giải quyết bài toán này như thế nào?**

**Đáp án:**

JWT là **stateless** — server không lưu gì. Để verify JWT, server chỉ cần recompute signature và check exp. Không có lookup nào vào DB hay cache. Đây là nguồn gốc của vấn đề revocation: nếu server không tracking token nào tồn tại, server không biết token nào cần invalidate.

Khi user logout hoặc đổi password:

- Cookie session: xóa session record khỏi Redis → ngay lập tức invalid
- JWT: token vẫn valid cho đến khi `exp`. Không có cách nào để server nói "token này không còn được chấp nhận nữa" mà không thêm state.

Production solutions (theo thứ tự trade-off):

**Option 1: Short expiry (15m) + accept the window.** Nếu token bị steal, attacker chỉ có tối đa 15 phút. Acceptable cho nhiều use cases.

**Option 2: Token blacklist trong Redis.** Khi revoke, lưu `jti` (JWT ID) vào Redis với TTL bằng token expiry. Mỗi request check Redis xem `jti` có trong blacklist không. Nhược điểm: mất stateless benefit, thêm Redis lookup cho mọi request.

**Option 3: Token version trong DB.** Mỗi user có `tokenVersion` field. JWT chứa `tokenVersion` trong payload. Khi logout/change password, increment `tokenVersion` trong DB. Middleware verify: `jwt.tokenVersion === db.user.tokenVersion`. Nhược điểm: DB lookup mỗi request, nhưng rẻ hơn full blacklist (read một field thay vì check set).

**Option 4: Refresh token rotation như đã mô tả.** Access token ngắn (15m), refresh token có reuse detection. Revocation thực chất xảy ra ở tầng refresh token — khi user logout, invalidate refresh token trong DB. Access token còn sống tối đa 15m.

Không có perfect solution — đây là fundamental trade-off của stateless authentication.

---

**Câu 2 — Điều gì xảy ra nếu không implement PKCE trong Authorization Code Flow cho SPA?**

**Đáp án:**

Không có PKCE, SPA vulnerable với **authorization code interception attack**:

Authorization code được gửi qua URL redirect (`?code=SplxlOBeZQQ`). URL này:

1. Xuất hiện trong browser history
2. Được ghi vào server access logs của redirect_uri server
3. Có thể bị intercept nếu redirect_uri bị misconfigured
4. Có thể bị đọc bởi browser extensions, tracking pixels trong referrer header
5. Có thể bị intercept trong mobile apps qua custom URI schemes (khác app claim cùng URI scheme)

Không có PKCE: Attacker intercept code → gửi thẳng lên token endpoint với `client_id` (public, không secret vì SPA) → nhận access token + refresh token → full account access.

Với PKCE: Attacker intercept code nhưng không có `code_verifier` (nằm trong memory của legitimate SPA). Token endpoint yêu cầu `code_verifier` để verify `SHA256(verifier) === stored challenge`. Attacker không thể compute verifier từ challenge (SHA256 one-way). Exchange fail. Code trở nên vô dụng sau một lần attempt.

Lý do PKCE đặc biệt quan trọng cho SPA/mobile (public clients): không có `client_secret` để authenticate. Với confidential clients (server-side apps), `client_secret` đã là một barrier — attacker cần cả code lẫn secret. Public clients không có secret → PKCE là barrier duy nhất.

**Câu 3 — Đoạn code này có vấn đề bảo mật gì? Có thể bị exploit như thế nào?**

```javascript
app.post('/login', async (req, res) => {
  const { username, password } = req.body;
  const user = await db.users.findOne({ where: { username } });

  if (!user || !(await bcrypt.compare(password, user.passwordHash))) {
    return res.status(401).json({ error: 'Invalid credentials' });
  }

  const token = jwt.sign(
    { userId: user.id, role: user.role, email: user.email },
    process.env.JWT_SECRET,
  );

  res.json({ token });
});
```

**Đáp án:**

Có **4 vấn đề bảo mật** trong đoạn code này:

**Vấn đề 1: Timing attack để enumerate usernames.**

```javascript
if (!user || !(await bcrypt.compare(...)))
```

Nếu `user` không tồn tại, code return ngay mà không chạy `bcrypt.compare`. bcrypt.compare tốn ~100ms (intentionally slow). Response khi username không tồn tại: ~1ms. Response khi username tồn tại nhưng password sai: ~100ms. Attacker đo timing → biết username có tồn tại hay không.

Fix: Luôn chạy bcrypt dù user không tồn tại:

```javascript
const DUMMY_HASH = '$2b$12$placeholder...'; // pre-computed hash
if (!user) {
  await bcrypt.compare(password, DUMMY_HASH); // Waste time
  return res.status(401).json({ error: 'Invalid credentials' });
}
```

**Vấn đề 2: JWT không có expiration.**
`jwt.sign({...}, secret)` không có `expiresIn` → token valid forever. Nếu token bị steal, attacker có access không giới hạn thời gian.

Fix: `{ expiresIn: '15m' }`

**Vấn đề 3: Email trong JWT payload.**
`email: user.email` trong JWT — base64 decoded dễ dàng, visible to client và bất kỳ intermediate (proxy, log). Không cần thiết vì server có thể query email từ userId khi cần.

**Vấn đề 4: Response error message giống nhau che giấu timing nhưng cùng message** — đây thực ra là đúng (cùng message "Invalid credentials"). Nhưng timing vẫn leak. Message đồng nhất là necessary nhưng không sufficient.

**Bonus — Vấn đề 5 (nếu phát hiện): Không có rate limiting.** Endpoint này có thể bị brute-force. Cần: rate limiting per IP + per username (IP rotation không bypass per-username limit), account lockout sau N failed attempts, hoặc CAPTCHA.
